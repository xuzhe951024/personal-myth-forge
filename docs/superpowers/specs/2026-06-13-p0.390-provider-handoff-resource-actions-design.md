# P0.390 Provider Handoff Resource Actions Design

## Context

The final showcase readiness report is still blocked by external evidence:
iOS signing/device reachability, provider keys, and configured print handoff.
The resource reports already keep secrets in ignored local files and expose
validation-aware operator actions. The provider handoff report still gives a
rougher path for real 3D and NPC providers, especially when local demo providers
are selected.

## Goal

Make `provider_handoff_report` point operators to the same ignored resource
handoff path used by the rest of the final launch flow. This reduces ambiguity
without applying keys, touching Xcode signing, calling live providers, or writing
any global state.

## Behavior

- Keep `provider-handoff` read-only and sanitized.
- Keep the existing `core_real_ready` status contract.
- Put `make final-resource-apply-preview` at the front of `next_commands` when
  real provider readiness still needs local resources to be applied.
- Keep `make final-apply-resources` in `next_commands` as the explicit apply
  step after preview and user review.
- For missing backend provider secrets, expose operator actions in the same
  validation-aware form used elsewhere:
  `provide <KEY> in final-resources.env; rerun make final-resources-preflight`.
- When the blocker is provider selection rather than a direct missing env var,
  describe the real path: fill ignored final resources, preview, apply, then
  rerun provider handoff. Do not pretend `THREE_D_PROVIDER` or `NPC_PROVIDER`
  are edited directly in `final-resources.env`; `final-apply-resources` writes
  them to the ignored backend `.env`.
- Do not include raw key values, absolute local paths, raw private context,
  payment links, or provider call output.

## Data Shape

`provider_handoff_report.first_blocker` can include these additional sanitized
fields when blocked:

- `resources_file`: `services/backend/.local/final-resources.env`
- `apply_preview_command`: `make final-resource-apply-preview`
- `apply_command`: `make final-apply-resources`
- `handoff_command`: `make provider-handoff`
- `blocked_by`: ordered backend resource ids such as `MESHY_API_KEY` and
  `OPENAI_API_KEY` when applicable

Existing consumers can ignore these fields. Existing top-level fields remain.

## Testing

Focused tests cover:

- default local provider handoff surfaces apply preview before apply;
- local provider handoff operator actions include Meshy/OpenAI key actions with
  `final-resources-preflight` reruns;
- configured real providers with missing keys keep sanitized missing env actions;
- configured real providers with keys remain ready and do not leak secrets.

Full verification stays backend-only and does not call Meshy, OpenAI, Treatstock,
Xcode, Keychain, or signing.
