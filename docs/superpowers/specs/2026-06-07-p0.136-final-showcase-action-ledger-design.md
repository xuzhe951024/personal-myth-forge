# P0.136 Final Showcase Action Ledger Design

## Context

P0.133 through P0.135 made lower-level launch reports much more actionable:
final acceptance, final handoff, iOS device launch certificate, and iOS device
launch rehearsal now emit concrete `operator_actions`. The objective-level
`final_showcase_readiness_report` still reduces blocked or partial capabilities
to broad commands such as `make ios-device-launch-rehearsal` and
`make final-resource-apply-preview`.

That is too coarse for the final demo handoff. The Mac operator should be able
to open one report and see the immediate nested actions that unblock iOS launch,
resource handoff, live provider evidence, print fulfillment, functional
regression, and visual regression.

## Decision

Add a bounded objective-level action ledger to `final_showcase_readiness`:

- Keep capability rows unchanged so status classification stays stable.
- Build `operator_actions` from concrete downstream report actions first, then
  fall back to each blocked or partial capability command.
- Preserve existing live-provider consent wording when Meshy/OpenAI live evidence
  is still unproven.
- Keep the output bounded, deduped, sanitized, and read-only.
- Do not run live providers, Xcode, signing, keychain, global configuration, or
  print orders.

## Data Flow

`build_final_showcase_readiness_report` already constructs these reports before
creating the final ledger:

- `final_acceptance_readiness`
- `three_d_evaluation_readiness`
- `npc_agent_evaluation_readiness`
- `visual_regression_readiness`
- `live_provider_evidence`
- `print_fulfillment_readiness`
- `ios_device_launch_rehearsal_readiness`
- `final_resources_preflight`
- `final_resource_apply_preview`
- `resource_handoff`

P0.136 passes those report dictionaries into `_operator_actions`. The helper
extracts string actions from each report's `operator_actions` field in capability
order, then appends the existing capability command fallback and final rerun
commands. `_sanitize_report` remains the last boundary before the report leaves
the module.

## Action Ordering

The first actions should match the highest-level blocker and remain practical:

1. iOS deployable actions from `ios_device_launch_rehearsal_readiness`
2. live provider actions from `live_provider_evidence`
3. print fulfillment actions from `print_fulfillment_readiness`
4. provider/key handoff actions from `final_resource_apply_preview`,
   `final_resources_preflight`, and `resource_handoff`
5. functional and visual regression refresh actions
6. fallback capability commands and final rerun commands

The ledger should keep no more than sixteen actions. That is enough for the
current final showcase chain while preventing oversized mobile payloads.

## Tests

- Add a backend RED test that writes saved downstream reports with concrete
  actions and proves `final_showcase_readiness.operator_actions` includes those
  nested actions before broad fallback commands.
- Assert the list is bounded and deduped.
- Assert sensitive strings and local paths remain redacted by the existing
  sanitizer.
- Run focused `test_final_showcase_readiness.py`, then full backend and Swift
  verification before merging.

## Safety

This iteration reads local report files only. It does not create or use API
keys, call Meshy/OpenAI/Treatstock/Sculpteo, mutate `.env`, write iOS deploy
config, invoke Xcode, accept Apple licenses, touch keychains, or change global
machine state.

## Self-Review

- No placeholder requirements remain.
- Scope is one backend report-shaping helper and its tests.
- Existing capability statuses and evidence summaries stay compatible.
- The final report remains sanitized and bounded.
