# P0.377 iOS Completion Consent Design

## Context

P0.376 added `completion_requires_live_provider_consent` to backend final
showcase readiness capability rows. The field distinguishes two consent scopes:

- `requires_live_provider_consent`: the immediate action in `command` may call a
  live provider.
- `completion_requires_live_provider_consent`: the capability cannot become
  `ready` without a later live provider action, even if the immediate command is
  safe setup work.

The iOS client currently decodes final showcase capability rows without this new
field, so the Final Launch summary cannot show this distinction on-device.

## Decision

Extend the iOS final showcase capability model with:

- `completionRequiresLiveProviderConsent: Bool?`

The value remains optional for backward compatibility with older readiness
payloads. Final Launch summary rows will append a short consent hint when this
field is `true`.

## Scope

In scope:

- Decode `completion_requires_live_provider_consent` on
  `FinalShowcaseReadinessCapability`.
- Preserve older payload compatibility by defaulting the initializer argument to
  `nil`.
- Add the consent hint to `showcaseReadinessCapabilityRow`.
- Add focused Swift contract coverage for decoding and summary rendering.

Out of scope:

- Backend readiness changes.
- Running live Meshy/OpenAI providers.
- Xcode build/signing or device deployment.
- New UI layout work beyond the existing Final Launch summary row text.

## Safety

This phase only changes source models, summary text, docs, and contract tests. It
does not write secrets, create deployment config, call providers, run Xcode, or
touch Keychain/global signing state.
