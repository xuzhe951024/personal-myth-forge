# P0.31 Integrated Demo Acceptance Design

## Context

The project now has most of the pieces needed for a minimum showable iOS demo:

- mobile capture/upload contracts and guided scan photo-set intake
- Meshy-capable 3D generation behind `THREE_D_PROVIDER`
- OpenAI-capable NPC director/tick runtime behind `NPC_PROVIDER`
- backend-owned session history
- bounded NPC autonomy runs
- provider readiness and provider handoff reporting

P0.24 answers whether provider configuration is present, but it deliberately
does not execute the end-to-end demo loop. The final API/key handoff needs a
single command that can be run in two modes:

- local mode, with no provider keys and no external calls, to verify the
  integrated acceptance harness itself
- configured mode, after Meshy/OpenAI keys are supplied, to run the actual
  capture-to-3D-to-NPC-to-history path and write a sanitized report

## Goal

Add a backend CLI acceptance harness that runs one complete demo session,
advances a bounded NPC autonomy sequence, writes a sanitized JSON report, and
can be used unchanged when real Meshy/OpenAI keys are supplied later.

## Non-Goals

- No key creation or secret storage.
- No global environment mutation.
- No live provider calls by default.
- No iOS signing, simulator boot, or device install automation.
- No print fulfillment order placement.
- No raw photo, scan binary, data URI, local file path, or raw personal-source
  text in the report.
- No background NPC daemon.

## Recommended Approach

Add a new backend CLI command:

```bash
cd services/backend
uv run python -m myth_forge_api.cli demo-acceptance \
  --provider-mode local \
  --npc-steps 3 \
  --output /tmp/personal-myth-forge-demo-acceptance.json
```

`--provider-mode local` forces local 3D and local NPC providers in memory by
using `dataclasses.replace(settings, three_d_provider="local",
npc_provider="local")`. It does not mutate `.env` or process-wide environment
variables.

When provider keys are later supplied, the same command can run configured
providers:

```bash
cd services/backend
uv run python -m myth_forge_api.cli demo-acceptance \
  --provider-mode configured \
  --require-real-core \
  --npc-steps 3 \
  --output /tmp/personal-myth-forge-demo-acceptance.json
```

`--require-real-core` checks the same core provider kinds as P0.24:

- `three_d`
- `npc`
- `capture_storage`

If the selected configured providers are not real-provider-ready, the command
writes a report and exits `2` without calling Meshy/OpenAI. If providers are
ready, it runs the full acceptance flow and returns `0` on success.

## Alternatives Considered

### HTTP-Only Smoke Against `make backend-dev`

This would exercise routing through FastAPI, but it requires process lifecycle,
ports, and local server state before the acceptance report can run. Keeping the
first harness as CLI-first is easier to test and avoids leaving background
servers running.

### Extend `provider-handoff`

`provider-handoff` is intentionally configuration-only. Mixing live provider
calls into that command would make a safe status check unexpectedly spend
provider quota. A separate `demo-acceptance` command keeps the boundary clear.

### Mobile-Only Acceptance

Real iPhone acceptance is still required before the whole objective is complete,
but it is blocked on local signing/team config and Apple SDK license acceptance.
P0.31 prepares the backend side of that final run without requiring global Xcode
changes.

## Acceptance Flow

The command builds a fixed safe acceptance input:

- object label: `old brass key`
- materials: `metal`, `brass`
- source: `acceptance_harness`
- visual notes: `worn teeth, circular bow, warm reflections`
- current theme: `deadline pressure`
- desired tone: `tender, strange`

Then it:

1. loads settings and applies provider mode
2. builds provider readiness
3. exits `2` before provider calls when `--require-real-core` is set and core
   providers are not real-ready
4. creates a `MythSession` through `create_demo_myth_session`
5. saves it to a local myth session store
6. advances `npc_steps` stored ticks sequentially using the configured NPC tick
   runtime and the same recent-event rules as P0.29/P0.30
7. writes a compact report with timings, provider names, generated asset
   variants, NPC runtime names, tick count, and safety policy

`npc_steps` accepts `0...3` and defaults to `3`. Zero steps are allowed so the
operator can smoke-test 3D generation and initial NPC reactions without spending
extra NPC tick calls in configured mode.

## Report Shape

The report is JSON:

```json
{
  "kind": "demo_acceptance_report",
  "mode": "local",
  "status": "succeeded",
  "require_real_core": false,
  "core_real_ready": false,
  "provider_readiness": {
    "overall_demo_ready": true,
    "overall_real_ready": false,
    "providers": []
  },
  "selected_providers": {
    "three_d": "local",
    "npc": "local",
    "capture_storage": "local_filesystem"
  },
  "session": {
    "session_id": "myth_...",
    "title": "The Old Brass Key That Fell Through the Sky",
    "generated_asset_provider": "local_stub",
    "generated_asset_format": "glb",
    "generated_asset_uri": "local://generated-assets/myth_....glb",
    "scene_variant_uri": "local://generated-assets/myth_....usdz",
    "print_candidate_format": "3mf"
  },
  "npc": {
    "initial_runtime": "local_agent_runtime",
    "reaction_count": 3,
    "requested_steps": 3,
    "completed_steps": 3,
    "latest_tick_index": 3,
    "tick_runtime": "local_tick_runtime"
  },
  "timings": {
    "total_elapsed_seconds": 0.0123
  },
  "safety": {
    "mobile_secret_policy": "Provider secrets stay on the backend.",
    "raw_media_in_report": false,
    "raw_personal_source_in_report": false
  },
  "error": null
}
```

On failures, `status` becomes `failed`, `error` contains a sanitized message,
and the command returns `1`. Secret values, bearer tokens, data URIs, local file
paths, and `raw=` payload markers are redacted.

## Components

`services/backend/src/myth_forge_api/acceptance.py`

- Owns the acceptance flow and report construction.
- Uses existing provider factories, `create_demo_myth_session`,
  `LocalMythSessionStore`, and `NPCAgentTickRequest`.
- Sanitizes errors through a local helper that matches the CLI redaction rules.

`services/backend/src/myth_forge_api/cli.py`

- Adds the `demo-acceptance` parser.
- Delegates to `run_demo_acceptance`.
- Writes JSON to `--output` or stdout.
- Returns `2` when `--require-real-core` fails readiness.

Docs and evidence:

- README and mobile README describe final acceptance commands.
- Roadmap adds P0.31.
- Static visual evidence shows local versus configured acceptance modes, report
  contents, and the backend-only secret boundary.

## Testing

Backend tests will cover:

- local acceptance writes a succeeded report with session, asset, NPC, tick, and
  safety summaries
- `--npc-steps 0` skips tick generation while still creating the session report
- configured strict mode without Meshy/OpenAI keys writes a report and exits `2`
  before provider calls
- provider failures are sanitized and return `1`
- stdout mode prints valid JSON
- invalid `--npc-steps` values fail argparse validation

Regression will include backend lint/test, SwiftPM project checks, Swift core
contract tests, app compile check, static visual evidence at 390x844, and the
known local deploy/Xcode license gates.

## Acceptance Criteria

- `demo-acceptance --provider-mode local --npc-steps 3` completes without
  external keys and writes a sanitized report.
- `demo-acceptance --provider-mode configured --require-real-core` exits `2`
  without provider calls until Meshy/OpenAI keys are supplied.
- The command can be rerun later with real providers and the same output schema.
- Reports contain generated asset and NPC runtime evidence without provider key,
  raw media, local path, or raw personal-source leakage.
- Existing provider handoff, 3D generation, NPC tick, mobile, and deploy gates
  remain intact.
