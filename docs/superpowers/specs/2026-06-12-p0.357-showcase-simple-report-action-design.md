# P0.357 Showcase Simple Report Action Design

## Goal

Make blocked simple report capabilities in `final-showcase-readiness` expose their concrete child `next_action` at the capability row level.

## Context

After refreshing local evidence, the stale local reports are cleared. `functional_regression` now has a real blocker from final acceptance readiness:

```text
mobile_deploy_preflight -> provide DEVELOPMENT_TEAM in Deployment.local.xcconfig
validation: make mobile-deploy-preflight
```

The final showcase row already includes that action in `detail`, but the row-level `command` still points to the generic report command:

```text
make final-acceptance-local
```

That makes the capability table less actionable than the child readiness report.

## Design

For `_simple_report_capability`:

- Ready reports keep the existing ready command.
- Missing reports keep the existing report-generation command.
- Blocked reports keep the existing command unless their first blocker has `next_action.command`.
- If a blocked report has `next_action.command`, the capability row uses that command.
- If the child next action has `validation_command`, the capability row includes `validation_command`.
- Detail formatting remains unchanged and continues to include the child action and diagnostic detail.

This is intentionally generic because `functional_regression` and `visual_regression` both use the same simple report capability path. It does not change the child reports themselves.

## Scope

In scope:

- Update `services/backend/src/myth_forge_api/final_showcase_readiness.py`.
- Extend focused tests in `services/backend/tests/test_final_showcase_readiness.py`.
- Add this spec, an implementation plan, and a roadmap entry.
- Refresh safe ignored `.local` evidence after verification.

Out of scope:

- Running Xcode signing, Keychain writes, real provider calls, Treatstock live quote calls, or paid actions.
- Changing final acceptance readiness generation.
- Changing iOS deploy preflight behavior.

## Acceptance Criteria

- A blocked `functional_regression` capability with child `next_action.command` exposes that command at row level.
- The same row exposes child `validation_command`.
- Existing detail assertions still pass.
- Ready `functional_regression` remains ready and keeps the existing generic report command.
- Focused backend tests, lint, and diff checks pass.
