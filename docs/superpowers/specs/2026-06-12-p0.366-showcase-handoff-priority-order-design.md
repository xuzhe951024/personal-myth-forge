# P0.366 Final Showcase Handoff Priority Order Design

## Context

After refreshing local evidence, `final_showcase_readiness.operator_actions` is current and sanitized, but the two core external handoff actions still appear near the end:

- `make provider-handoff; rerun make live-provider-evidence`
- `after explicit Treatstock cost consent ... /v1/print-quotes; rerun make print-fulfillment-readiness`

The final demo launch report promotes these actions at the top level, and the iOS summary now filters them correctly. A reader or client that consumes `final_showcase_readiness` directly still has to scan through resource and report-maintenance actions before seeing the live 3D/NPC and print fulfillment handoffs that matter most for the final goal.

## Decision

Reorder final showcase top-level operator actions so critical external handoffs are visible immediately after the first concrete blocker action:

1. Keep the first action unchanged.
2. Promote provider/live handoff actions.
3. Promote Treatstock/print quote handoff actions.
4. Append the existing remaining actions in their prior relative order.
5. Keep existing dedupe, validation-aware normalization, and action cap behavior.

## Scope

In scope:

- Add a backend regression test proving provider and print handoff actions appear before generic resource/final report actions in `final_showcase_readiness.operator_actions`.
- Update final showcase action ordering only.
- Add a P0.366 roadmap entry.

Out of scope:

- Changing mobile summary logic.

## Safety

This phase only edits tracked source, tests, and docs. Any evidence refresh writes ignored `.local` reports only. It does not write secrets, call providers, place print orders, mutate global state, touch Keychain, or perform signing.
