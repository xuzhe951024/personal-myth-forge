# P0.575 iOS Rehearsal Command Normalization Design

## Context

`final-showcase-readiness.json`, `final-demo-launch-local.json`, and
`final-launch-closure-packet.json` can currently expose command strings like:

```text
PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence; rerun make ios-device-launch-rehearsal; rerun make ios-device-launch-rehearsal;
```

The command is safe because it still requires explicit live provider consent, but
it is noisy and copy-hostile: the same validation target is appended twice and a
trailing semicolon remains.

## Goal

Normalize operator action command chains so repeated `; rerun <target>` segments
are emitted once, preserving order and preserving detail suffixes after ` | `.

## Approach

Implement the behavior in the shared
`myth_forge_api.operator_actions.normalize_operator_action()` path. That keeps
the fix available to final showcase, final demo launch, iOS rehearsal readiness,
closure packets, and any future report that already uses shared operator action
normalization.

The normalizer will:

- trim trailing semicolons from the executable command portion;
- split command text on `; rerun `;
- keep the first root command unchanged;
- keep later validation commands in first-seen order;
- drop duplicate validation commands after normalizing whitespace and trailing
  semicolons;
- reattach any existing ` | ` detail suffix unchanged.

## Non-Goals

- Do not run real Meshy/OpenAI providers.
- Do not change provider consent policy.
- Do not change which capability is blocked or partial.
- Do not perform Xcode signing, Keychain, device deployment, or paid print calls.

- Run focused tests for shared operator actions plus final readiness aggregators.
