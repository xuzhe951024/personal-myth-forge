# P0.90 NPC Agent Evaluation Design

## Purpose

P0.90 adds a repeatable NPC Agent evaluation suite so the project can validate
that NPC behavior is genuinely agent-shaped before the final iPhone showcase.
The suite is analogous to the existing 3D evaluation harness: it can run locally
without keys today, and it can later run through the configured OpenAI NPC
director and tick runtime after the operator provides `OPENAI_API_KEY`.

This iteration does not call OpenAI by default, does not create or read provider
keys outside the existing backend settings path, and does not mutate global
machine state.

## Scope

In scope:

- A backend `evaluate-npc` CLI command.
- A canonical local NPC evaluation suite with multiple object/context cases.
- A report that records initial NPC reactions, agent traces, one or more tick
  steps, world-arbitration outcomes, elapsed time, coverage, and manual review
  placeholders.
- Secret and raw-data redaction in all report content.
- Documentation, source acceptance coverage, and static visual evidence for the
  report shape.

Out of scope:

- Live OpenAI calls during default tests or final acceptance.
- New mobile UI controls.
- Long-term NPC memory beyond the existing session/tick history contract.
- Voice, animation, Unity import, or background autonomous loops.

## Product Behavior

The operator can run:

```bash
cd services/backend
uv run python -m myth_forge_api.cli evaluate-npc \
  --provider local \
  --suite default-v0 \
  --tick-steps 2 \
  --output .local/npc-evaluation-local.json
```

The report answers five practical questions:

- Did every case produce exactly Mara, Ior, and Senn?
- Did every NPC include an explicit belief, intention, proposed action,
  rationale, and confidence?
- Did the proposed action appear in the visible plan for that NPC?
- Did the world arbitrator accept or reject concrete actions for each tick?
- Did the report avoid raw private content, provider secrets, local paths, and
  media payloads?

When `--provider openai` is selected later, the CLI reuses the existing backend
provider factory and settings. Missing `OPENAI_API_KEY` returns exit code `1`
without writing a misleading report.

## Architecture

Add `services/backend/src/myth_forge_api/evaluation/npc.py`.

The module owns:

- `NPCAgentEvaluationCase`: object observation, context capsule, expected NPCs,
  review focus, and recent events.
- `DEFAULT_NPC_AGENT_EVALUATION_SUITE`: a small canonical suite that exercises
  ordinary object worship, emotional context, conflict, privacy boundaries, and
  follow-up ticks.
- `run_npc_agent_evaluation`: builds a myth-session-shaped deterministic input
  per case, calls an injected `NPCDirector` and `NPCTickRuntime`, and returns a
  sanitized JSON-compatible report.

The CLI owns only parsing, provider selection, configuration validation, and
writing JSON. This matches the existing `evaluate-3d` split.

## Report Shape

Top-level fields:

- `kind`: `npc_agent_evaluation_report`
- `suite`
- `provider`
- `tick_steps`
- `total_cases`
- `succeeded`
- `failed`
- `coverage`
- `review_rubric`
- `safety`
- `cases`

Each case includes:

- identity and prompt context: `case_id`, `category`, `object_label`,
  `review_focus`
- execution status and elapsed time
- initial provider/runtime metadata
- sanitized reaction and trace summaries
- tick summaries with world-resolution accepted/rejected counts
- manual review placeholders
- sanitized error, if any

The report intentionally stores approved capsule summaries and object labels
because they are the evaluation input. It must not store raw email, chat,
calendar, file bodies, provider bearer tokens, local file paths, or media data
URIs.

## Safety

The evaluation module sanitizes recursively before returning the report.
Redaction patterns match the existing 3D evaluation harness and add common
private-context leak markers such as `raw_email:`, `raw_calendar:`, and
`private_message:`.

Tests must cover a provider failure containing a fake bearer token and fake raw
private text, and must prove the written report does not contain those strings.

## Testing

Backend TDD coverage:

- `evaluate-npc --suite default-v0 --provider local` writes a rich report.
- The default suite records expected NPC ids, trace counts, tick counts, action
  coverage, world-arbitration coverage, and manual review placeholders.
- Invalid CLI usage rejects missing or conflicting suite/custom-case input.
- Provider configuration failure returns exit code `1` and does not write a
  stale report.
- Per-case provider errors are sanitized.
- Source acceptance checks prove the CLI, evaluation module, docs, and final
  acceptance source gates mention `evaluate-npc`.

Visual regression:

- A static 390x844 HTML snapshot shows the NPC Agent Evaluation receipt,
  coverage, safety, and first case summary. It is documentation evidence only;
  it does not run providers.

## Acceptance Criteria

- Local `evaluate-npc` runs without external keys and produces a safe report.
- The report gives the final operator a concrete review rubric for NPC Agent
  behavior after OpenAI resources are supplied.
- No default test, acceptance command, or visual artifact performs live OpenAI
  calls.
- Existing backend and Swift checks remain green.
