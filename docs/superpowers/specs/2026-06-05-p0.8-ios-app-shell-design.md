# P0.8 iOS App Shell Design

Date: 2026-06-05

## Context

P0.7 created a SwiftPM mobile core and source-only SwiftUI scaffold. That moved
the client contract forward, but it is still not an iOS app a developer can open
in Xcode. The current machine still reports:

```text
xcode-select: error: tool 'xcodebuild' requires Xcode, but active developer
directory '/Library/Developer/CommandLineTools' is a command line tools instance
```

Changing the active developer directory with `xcode-select` would alter global
machine state, so P0.8 must not do that.

## Goal

Create a checked-in iOS app project shell for Personal Myth Forge that is ready
to open in a full Xcode environment, while keeping local verification possible
with the current SwiftPM-only environment.

## Non-Goals

- No simulator, device, archive, or App Store build on this machine.
- No global Xcode configuration changes.
- No real camera, LiDAR, ARKit, photogrammetry, Unity, or provider API calls.
- No API keys or provider secrets in the mobile app bundle.

## Chosen Approach

P0.8 adds a minimal `PersonalMythForge.xcodeproj` under `apps/mobile/ios`. The
project compiles the existing SwiftUI app files as the native iOS target and
links `PersonalMythForgeMobileCore` through the local Swift package product. The
project is intentionally simple: one app target, one `Info.plist`, one local
Swift package dependency, no CocoaPods, no package registry dependencies, and no
local developer signing identity committed.

Because local `xcodebuild` cannot run, P0.8 also adds a SwiftPM executable
project inspector. The inspector validates the project shell as data:

- the Xcode project exists and declares a `PersonalMythForge` iOS app target
- the target references app Swift files and links the local core package product
- the Swift package manifest declares both iOS and macOS platforms
- `Info.plist` declares camera/photo usage descriptions
- backend base URL is configurable from the app bundle
- no mobile file contains committed API keys or provider secrets
- docs clearly state full Xcode is required for simulator/device deployment

This gives the next iteration a concrete Xcode artifact without pretending local
deployment has already been proven.

## Project Layout

```text
apps/mobile/ios/
  PersonalMythForge.xcodeproj/

  App/

    PersonalMythForgeMobileProjectChecks/
      main.swift
```

## App Configuration

The app exposes backend configuration from `Info.plist`:

- `PMFBackendBaseURL`: defaults in source to `http://127.0.0.1:8080` if absent.

Provider keys stay backend-side:

- `OPENAI_API_KEY`
- `MESHY_API_KEY`
- future print provider keys

The mobile app may show provider results, but it must not store provider secrets.

## iOS Permissions

The app shell declares permission strings now so future camera/photo work has a
clear deployment contract:

- `NSCameraUsageDescription`
- `NSPhotoLibraryUsageDescription`
- `NSPhotoLibraryAddUsageDescription`

P0.8 does not request these permissions at runtime. That belongs to the next
capture UI iteration.

## Testing

Local verification uses SwiftPM only:

```bash
swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests
swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks
make backend-lint
make backend-test
```

The project check executable fails if the package manifest, Xcode shell, plist,
or mobile source secret boundary drift out of shape. Documentation coverage is
verified separately with grep checks in the implementation plan.

## Acceptance Criteria

- `apps/mobile/ios/PersonalMythForge.xcodeproj/project.pbxproj` exists.
- `apps/mobile/ios/App/Info.plist` exists and has required iOS permissions.
- `apps/mobile/ios/App/AppConfiguration.swift` exposes backend base URL lookup.
- `apps/mobile/ios/Package.swift` declares `.iOS(.v17)` and `.macOS(.v13)`.
- SwiftPM contract tests still pass.
- SwiftPM project checks pass.
- Backend lint and tests still pass.
- Mobile docs include exact open/build guidance and the full-Xcode limitation.
- No mobile source or plist file contains committed API keys or provider secrets.
