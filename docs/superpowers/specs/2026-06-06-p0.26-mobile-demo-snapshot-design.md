# P0.26 Mobile Demo Snapshot Design

## Context

The mobile app can already upload object media, create a myth session, show a
generated 3D artifact handoff, display initial NPC agent traces, and request
stateless NPC ticks. P0.25 added project-local iOS deploy configuration.

The next demo risk is continuity: after a successful forge, the app only keeps
the current `MythSession` and latest NPC tick in memory. If the app is relaunched
or the backend is temporarily unreachable during a live iPhone demo, the player
loses the generated relic, NPC interpretation, and visible world changes.

## Decision

Add a mobile-local demo snapshot that stores the latest showable myth session
and NPC tick history. This is a client-side continuity layer, not server-side
memory and not a fulfillment/order system.

Chosen approach:
- Store a versioned `DemoRunSnapshot` in the Swift mobile core.
- Persist only backend-returned structured session/tick metadata.
- Do not persist raw capture media bytes, PhotosPicker selections, file URLs,
  provider keys, bearer tokens, local capture payloads, or raw personal source
  documents.
- Restore the snapshot on app launch into the existing ready-state UI.
- Append NPC ticks to a short local history and show latest tick plus count in
  `NPCTickView`.

Alternatives considered:
- Server-side session/tick history: closer to eventual autonomous runtime, but
  requires new backend storage and session retrieval contracts. It is a better
  later step once the local demo surface is stable.
- Print fulfillment handoff: important for the final product, but it depends on
  provider/API credentials and does less to improve the next iPhone demo loop.

## Data Model

Create `DemoRunSnapshot` in the mobile core:

- `schemaVersion: Int`
- `savedAt: String`
- `session: MythSession`
- `npcTicks: [NPCAgentTick]`

Rules:
- `currentSchemaVersion` is `1`.
- `npcTicks` is ordered by ascending `tickIndex`.
- Only ticks whose `sessionId` matches `session.sessionId` are retained.
- Persist at most `12` ticks to keep the file bounded.
- `latestTick` returns the last retained tick.

Create `DemoRunSnapshotStore`:

- `snapshotURL: URL`
- `load() throws -> DemoRunSnapshot?`
- `save(_ snapshot: DemoRunSnapshot) throws`
- `clear() throws`

Use atomic file replacement through `Data.write(options: [.atomic])`. The
default app store writes under Application Support, not Caches, because the
snapshot is a user-visible continuation artifact rather than disposable asset
cache.

## App Integration

`ForgeRootView` will own:

- `demoSnapshotStore`
- `restoredSnapshot`
- `npcTickHistory: [NPCAgentTick]`
- `snapshotStatusText`

On launch:
- Load the latest snapshot.
- If valid, set `state.phase = .ready(snapshot.session)`.
- Set `npcTickHistory = snapshot.npcTicks`.
- Show a compact restored-session status above the forge form.

On forge:
- Clear in-memory tick history before starting.
- Once a new session is ready, save a snapshot with an empty tick list.
- If saving fails, show a non-blocking status message; the forge result remains
  visible.

On NPC tick:
- Append the returned tick to `npcTickHistory`, retaining only ticks matching
  the active session.
- Save the updated snapshot.
- `recentEvents` for the next tick continue to use the latest visible changes.

## UI

Keep the UI dense and operational:

- Add a small `DemoSnapshotStatusView` above the capture form.
- It shows whether a previous run was restored, how many ticks are available,
  and a clear action.
- `NPCTickView` shows the latest tick and a history count.

No new landing page, marketing copy, or decorative layout is added.

## Testing

Swift core contract tests:
- `DemoRunSnapshot` encodes/decodes with snake_case JSON through `PMFJSON`.
- Snapshot construction filters ticks for the active session and keeps the last
  12 ticks sorted by `tickIndex`.
- File store can save, load, overwrite, and clear a snapshot in a temporary
  directory.
- Snapshot JSON does not contain sample raw media bytes or bearer/API-key text.

Project checks:
- Ensure Xcode references the new app view file.
- Ensure the app source calls snapshot load/save/clear paths.
- Ensure `NPCTickView` exposes history count.

Visual regression:
- Add a 390x844 static evidence page showing restored session, tick history,
  local-only storage, and no raw media/provider-secret persistence.

Full regression:
- Swift project checks.
- Swift mobile core contract tests.
- Swift app compile gate.
- Backend lint/test.
- Expected deploy preflight gate until local config exists.
- Expected Xcode license gate on this machine.

## Non-Goals

- No server-side session database.
- No autonomous background NPC loop.
- No raw media persistence.
- No provider key or personal source storage in the app.
- No print ordering or fulfillment.
- No simulator/device installation until the external Apple SDK license and
  signing inputs are provided.
