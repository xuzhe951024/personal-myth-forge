# P0.543 iOS Rehearsal Fill Guide Actions Design

## Goal

Make `ios-device-launch-rehearsal-readiness` expose the same provider/key handoff path as the final readiness reports:

```text
make final-resource-fill-guide; rerun make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence
```

This keeps the iPhone rehearsal evidence aligned with the global launch handoff and avoids asking the operator to provide Meshy/OpenAI keys directly from the readiness report.

## Current Gap

`final-showcase-readiness`, `final-local-report-refresh`, `final-external-action-ledger`, and `final-launch-closure-packet` already prefer the fill-guide handoff chain. The lower-level `ios-device-launch-rehearsal-readiness` report can still surface:

- `make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence`
- `make provider-handoff; rerun make live-provider-evidence`
- `provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight`
- `provide OPENAI_API_KEY in final-resources.env; rerun make final-resources-preflight`

That makes device rehearsal instructions less executable and less consistent with the resource/key handoff model.

## Intended Behavior

- Top-level readiness `operator_actions` should prefer the canonical fill-guide provider handoff action whenever provider handoff or provider resource actions are present.
- Top-level readiness `operator_actions` should suppress bare provider secret fill actions for Meshy/OpenAI/Treatstock/Sculpteo after the canonical provider action is emitted.
- Per-sequence bounded `operator_actions` should receive the same normalization so embedded action bundles do not reintroduce the older key-first handoff.
- Device deploy actions, backend-device-demo actions, Xcode/license actions, and guarded print quote actions remain unchanged.
- No external provider calls, Xcode signing, Keychain writes, backend env writes, or paid print actions happen during this phase.

## Acceptance

- A focused test proves `build_ios_device_launch_rehearsal_readiness_report()` converts mixed provider/key actions into one canonical fill-guide provider action.
- A focused test proves sequence-level bounded actions use the same canonical provider action and do not leak top-level bare provider key actions.
- `make ios-device-launch-rehearsal-readiness` writes blocked local evidence without failing Make.
- Existing readiness and backend tests remain green.
