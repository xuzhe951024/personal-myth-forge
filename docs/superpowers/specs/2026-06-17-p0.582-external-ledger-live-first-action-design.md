# P0.582 External Ledger Live First Action Design

## Context

After local iOS/device evidence is ready, `final-external-action-ledger.json` correctly reports the first blocker as live Meshy/OpenAI evidence:

```text
PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence
```

However, its top-level `operator_actions` can still begin with the configured evidence bundle gate. That makes the final external action handoff less direct than `first_blocker` and `next_action`.

## Decision

Keep the existing grouped action audit intact. Only change the top-level operator action ordering:

- If `first_blocker` is a live provider blocker, promote its normalized command to the first top-level operator action.
- Keep the configured evidence bundle validation action after it.
- Do not alter resource, device, print, or global action grouping.

## Acceptance

- A live provider first blocker yields the guarded consent action as `operator_actions[0]`.
- `next_action.command` and `operator_actions[0]` agree.
- Existing configured evidence bundle gate behavior still remains in the queue as validation follow-up.
- No live provider calls are run by report generation.
