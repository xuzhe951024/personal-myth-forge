# P0.370 Final Handoff Priority Actions Design

## Goal

Make `final-handoff-index` expose the same provider and Treatstock print handoff actions that already appear in `final-demo-launch-local` and `ios-device-launch-rehearsal`.

## Current Evidence

- `final-demo-launch-local.json` contains the live provider handoff and Treatstock configured quote handoff near the top.
- `ios-device-launch-rehearsal.json` now preserves those actions.
- `final-handoff-index.json` still shows only local rehearsal, configured preflight, configured launch, backend device, and live acceptance actions. It drops the provider and print handoff actions even though its `source_reports.final_demo_launch_local.operator_actions` contains them.

## Behavior

`final_handoff_index` should:

- Keep the concrete local rehearsal / iOS deploy action first.
- Extract provider and print handoff actions from the saved `final_demo_launch_local` source report.
- Promote those actions immediately after the first local rehearsal action and before generic configured-preflight / configured-launch actions.
- Keep existing source-prefix stripping, normalization, secret redaction, and six-action cap behavior.

## Non-Goals

- Do not run live Meshy/OpenAI/Treatstock calls.
- Do not write API keys, Xcode config, or provider resources.
- Do not change first blocker or next action semantics.
- Do not include every final-demo action; only provider and print handoff actions should be promoted.

## Acceptance

- A focused RED/GREEN test proves `final-handoff-index` includes provider and Treatstock handoff actions from saved `final-demo-launch-local`.
- Existing final handoff index tests remain green.
- Adjacent iOS launch rehearsal/certificate tests remain green.
- Refreshed `final-handoff-index.json` shows provider and print handoff actions near the top while remaining blocked on external evidence.
