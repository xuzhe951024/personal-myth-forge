# P0.99 Forge Progress Receipt Design

## Context

P0.98 added a `Capture-to-3D` receipt after a capture has been uploaded and a
myth session has returned generation provenance. The next gap is the in-between
moment on iPhone: after the operator taps `Forge Myth`, real Meshy/OpenAI calls
may be slow or may fail, but the mobile UI only exposes coarse phase text.

For the final iPhone showcase, the operator needs a compact, privacy-safe receipt
that explains what the app is doing while the forge request is in flight and what
has completed after it returns.

## Goal

Add a mobile `Forge Progress` receipt that turns the current `ForgeFlowState`
into visible stage rows for upload, session creation, 3D generation, and NPC
interpretation. It should make local and configured-provider runs explainable on
phone without changing backend APIs or starting background polling.

## Non-Goals

- No backend job queue or long-running generation polling.
- No live provider calls during tests or source acceptance.
- No display of raw media URIs, local filesystem paths, checkout URLs, provider
  keys, raw personal context, or raw provider errors.
- No changes to the Forge button gate or capture upload payload contract.

## Recommended Approach

Use a mobile-core summary builder, `ForgeProgressReceiptBuilder`, that accepts
the current `ForgeFlowState` plus `CaptureGenerationReadiness` and returns a
small value model for SwiftUI.

This is preferable to a backend progress endpoint because the current backend
creates a myth session synchronously. A client-side receipt gives immediate demo
value now, stays accurate for the existing synchronous flow, and leaves a clean
surface for future async job IDs if the backend later moves to polling.

## User Experience

The iPhone app will show a `Forge Progress` card under the forge action area and
above `Capture-to-3D`.

The card status is:

- `WAIT` before object/context inputs are ready.
- `RUN` while capture upload or myth session creation is active.
- `READY` once a myth session has returned.
- `CHECK` when the forge request fails.

Rows should be stable and compact:

- `Capture upload`: waiting, running, uploaded, or blocked.
- `Myth session`: waiting, running, ready, or blocked.
- `3D generation`: route and provider when known, otherwise pending.
- `NPC Agent`: local/openai runtime when known, otherwise pending.

When ready, the card should mention the generated asset provider and route if
generation provenance is present. When failed, it should show a sanitized failure
message and keep any sensitive strings withheld.

## Data Model

Create:

- `ForgeProgressReceiptStatus`: `waiting`, `running`, `ready`,
  `needsAttention`.
- `ForgeProgressReceiptRow`: `label`, `status`, `detail`.
- `ForgeProgressReceipt`: `status`, `title`, `detail`, `rows`,
  `privacyNotes`.
- `ForgeProgressReceiptBuilder.build(state:captureGenerationReadiness:)`.

The builder should rely only on existing mobile models:

- `ForgeFlowState`
- `ForgeFlowPhase`
- `ObjectCapture`
- `MythSession`
- `GeneratedAssetProvenance`
- `CaptureGenerationReadiness`

## Privacy And Safety

All generated strings must pass through a local sanitizer. The sanitizer must
withhold:

- OpenAI/Meshy-style key prefixes such as `sk-...`
- `Bearer ...`
- `api_key=...`
- `local-capture://...`
- `file://...`
- `/Users/...`
- `/tmp/...`
- checkout or payment links
- raw context labels such as `private_message:` or `raw_context:`

The receipt privacy notes should explicitly say that provider keys, raw capture
media, and local paths stay off the receipt.

## Integration

`ForgeRootView` should derive:

```swift
private var forgeProgressReceipt: ForgeProgressReceipt {
    ForgeProgressReceiptBuilder.build(
        state: state,
        captureGenerationReadiness: captureGenerationReadiness
    )
}
```

`CaptureFormView` already owns the forge action and status area. Add a small
`ForgeProgressReceiptView` near that area so the operator sees it before the
post-session artifact cards.

The Xcode project must explicitly include the new SwiftUI file because this repo
uses manual PBX file references for app sources.

## Testing

Use TDD in mobile-core contract tests:

- idle/editing state waits with capture route context
- uploading state shows `RUN` and capture upload running
- capture-uploaded state shows myth session running
- ready session shows generated asset provider, route, and NPC runtime
- failed state redacts secrets, local paths, capture URIs, and payment links

Add source acceptance guards:

- Swift ProjectChecks must require the core builder, SwiftUI view, root wiring,
  Xcode source membership, and the new contract tests.
- Backend `ios_showcase_acceptance` must add a `forge_progress_receipt` feature.

Add a 390x844 static visual regression artifact showing the running/ready card
with readable rows and privacy notes.
