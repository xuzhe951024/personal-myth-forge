# P0.158 Local Showcase Smoke Design

## Context

The project already has safe local implementations for object capture upload,
capture-to-3D generation, local generated asset serving, NPC Agent ticks, print
quotes, iOS source gates, and final launch readiness. Most acceptance checks
exercise those pieces separately or through module-level helpers. The remaining
gap before final key/device handoff is an HTTP-boundary smoke proof that uses
the same REST calls the iPhone app will use during a no-key local demo.

The current final closure packet is correctly blocked on user-supplied resources
and physical-device evidence. P0.158 should not try to fake those blockers.
Instead, it should prove the local showcase loop is internally runnable without
Meshy, OpenAI, print-provider, Xcode, signing, keychain, or global machine
actions.

## Recommended Approach

Add a backend `local_showcase_smoke_report` builder and CLI/Make target that
drives the FastAPI app through `TestClient`:

1. `POST /v1/object-captures` with a guided-scan style multi-image upload.
2. `POST /v1/myth-sessions/from-capture` with an approved context capsule.
3. `GET` the served local game GLB and iOS scene DAE URLs returned in the
   generated asset.
4. `POST /v1/myth-sessions/{session_id}/autonomy-runs` for a bounded NPC Agent
   loop.
5. `POST /v1/print-quotes` using the session print candidate.
6. `GET` the served local print 3MF URL.

The smoke runner should use temporary capture and myth-session storage by
temporarily setting process-local environment variables while the TestClient
requests run. This avoids persistent raw media, keeps the repo clean, and does
not touch global state.

## Alternatives Considered

### Extend `demo-acceptance`

This would reuse an existing command, but `demo-acceptance` mostly drives domain
providers directly. It does not prove the iPhone-facing REST path, multipart
capture upload, served generated asset URLs, or print candidate download.

### Add an iOS UI-only Smoke

A Swift-only summary would make the phone panel richer, but it would not prove
the backend API loop actually works. The first missing proof is backend HTTP
behavior.

### Generate a Built-In Sample Session

A canned session would be easy to display, but it would weaken the product
direction by skipping capture upload, generation source routing, and NPC
runtime execution. It also conflicts with the existing project checks that avoid
sample-media fallbacks.

## CLI And Make

Add:

```bash
make local-showcase-smoke
```

The target runs:

```bash
cd services/backend && uv run python -m myth_forge_api.cli local-showcase-smoke --output .local/local-showcase-smoke.json
```

The CLI writes the report when `--output` is provided and returns:

- `0` when all required checks pass
- `1` when the HTTP smoke runs but a required check fails

## Final Launch Integration

Embed the report in `/v1/final-demo-launch` as `local_showcase_smoke`. This gives
the iPhone Final Launch panel and operator packet one authoritative local loop
proof. It remains independent from configured live-provider evidence.

Add compact mobile rows to `FinalLaunchMobileSummary` and render a `Local Smoke`
section in `FinalLaunchStatusView`:

- ready row: `Local showcase smoke ready: HTTP 6, NPC ticks 2, downloads 3`
- blocked row: `Local showcase smoke failed: ...`
- safety row: `provider_calls=false global=false temp_storage=true`

## Visual Regression

Add a static 390x844 visual regression artifact for the new mobile `Local Smoke`
section. Required text:

- `Local Smoke`
- `Local showcase smoke ready`
- `HTTP 6`
- `NPC ticks 2`
- `downloads 3`
- `provider_calls=false global=false temp_storage=true`

The artifact must not contain provider keys, raw media, local filesystem paths,
private LAN URLs, checkout/payment URLs, or data URIs.

## Safety

P0.158 must not:

- call Meshy, OpenAI, Treatstock, Sculpteo, or payment providers
- start a real backend server
- run Xcode, SwiftPM, signing, keychain, or Apple SDK commands
- write `.env` or iOS deployment config
- persist raw media in repo-local `.local` storage
- expose raw uploaded bytes, data URIs, temp paths, provider secrets, or payment
  links in reports or mobile rows

The only write from the Make target is the sanitized JSON report at
`services/backend/.local/local-showcase-smoke.json`.

## Testing

Backend tests:

- report succeeds through the full TestClient path
- report fails safely when a required API response fails
- report sanitizes provider secrets, raw media markers, data URIs, temp paths,
  and payment links
- CLI writes the report and returns the report exit code
- Make dry-run includes the exact CLI command
- final demo launch embeds the smoke report

Swift tests:

- `FinalDemoLaunchReport` decodes `local_showcase_smoke`
- `FinalLaunchMobileSummary` emits ready and failed smoke rows
- unsafe smoke details are redacted

Acceptance/source gates:

- mobile final launch readiness checks the new model, summary, UI section, and
  contract tests
- iOS showcase acceptance checks backend builder, CLI, Make target, final launch
  embed, Swift models, UI, and visual artifact registration

Verification:

```bash
make backend-lint
make backend-test
swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests
swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks
swift build --package-path apps/mobile/ios
make visual-regression
git diff --check
```

## Self-Review

No placeholders remain. The scope is one focused local HTTP smoke proof plus
mobile surfacing and visual regression. It does not solve resource provisioning,
live provider evidence, or physical iPhone deployment; those remain explicit
final closure blockers until the user supplies keys and device values.
