# P0.244 Handoff Local Lane Detail Design

## Context

`ios-device-evidence-bundle` and `ios-device-launch-rehearsal` now surface concrete device blockers such as missing `DEVELOPMENT_TEAM` and non-iPhone-reachable `PMF_BACKEND_BASE_URL`. The higher-level `final-handoff-index` still collapses the local lane to generic text:

```text
run make final-rehearsal-local
```

The final handoff index is the operator-facing map that ties local rehearsal, configured preflight, device deploy, and live acceptance together. If its first blocker is generic, the final handoff still requires drilling into lower reports before the operator knows the immediate device issue.

## Goal

Make `final-handoff-index` local rehearsal lane surface saved local report blocker detail when the local lane is blocked.

## Non-Goals

- Do not change lane IDs, lane ordering, commands, or status semantics.
- Do not add Swift UI changes.
- Do not run mobile deploy preflight, Xcode signing, Keychain, live providers, or paid provider calls.
- Do not make stale report handling less strict.

## Approaches Considered

1. Carry compact `detail` and `operator_actions` from saved local source reports into `final-handoff-index` and use them for the local lane.
   This is selected. It keeps the report shape stable and reuses evidence already produced by safe local reports.

2. Depend directly on `ios-device-evidence-bundle.json`.
   This would be tempting because it has strong device detail, but it is not currently one of the handoff index's local source reports. Adding another source would broaden freshness and ordering semantics for this small phase.

3. Only update `operator_actions`, leaving `first_blocker.detail` generic.
   This is incomplete because `first_blocker` is the field consumed by several readiness summaries.

## Design

`_source_report` will extract compact, sanitized blocker detail from saved reports:

- Prefer `payload["first_blocker"]["detail"]` when present.
- Otherwise prefer the first saved `operator_actions` entry.
- Keep the value bounded and sanitize through the existing whole-report sanitizer.

`_lanes` will attach saved local source detail to the `local_rehearsal` lane as `detail`. It prefers `final_demo_launch_local` because that report already consolidates the operator-facing launch blocker, including mobile deploy preflight evidence. If that detail is absent, it falls back to the first blocked or missing local source with detail or operator actions. The local lane status still comes from the combined local source statuses.

`_lane_blocker_detail` and `_operator_actions` will prefer `lane["detail"]` for `local_rehearsal`. The resulting top-level first blocker should read like:

```text
final_demo_launch_local: physical iPhone backend health gate | Checks local config and backend /health; does not build or sign. | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

## Testing

Add a backend RED/GREEN test:

- Arrange ready local 3D/NPC/visual reports.
- Arrange blocked `final-acceptance-local.json` so the local lane is blocked.
- Arrange partial `final-demo-launch-local.json` with a `first_blocker.detail` containing missing `DEVELOPMENT_TEAM` and non-iPhone-reachable backend URL.
- Arrange other required local reports as ready.
- Build `final-handoff-index`.
- Assert `lanes_by_id["local_rehearsal"]["detail"]`, top-level `first_blocker.detail`, and `operator_actions[0]` include the saved blocker detail.
- Assert the report text does not leak `MESHY_API_KEY`.

Focused verification:

```bash
cd services/backend
uv run pytest tests/test_final_handoff_index.py::test_final_handoff_index_local_lane_uses_saved_blocker_detail
uv run pytest tests/test_final_handoff_index.py
```

Final verification:

```bash
make backend-lint
make backend-test
git diff --check
make final-handoff-index
```

`make final-handoff-index` is expected to exit 2 while external device/provider values are still missing, but it must write `services/backend/.local/final-handoff-index.json` with a detailed local rehearsal first blocker.
