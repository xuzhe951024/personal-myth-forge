# P0.161 Final Showcase Provider Handoff Digest Design

## Context

After P0.160, the Final Showcase summary exposes the current iOS deploy blocker as a first-class `ios_deploy` stage. The latest `final_showcase_readiness` report now has these remaining statuses:

- Ready: capture scanning, local showcase smoke, visual regression, privacy safety.
- Partial: live Meshy 3D generation, live OpenAI NPC Agent, print fulfillment.
- Blocked: iOS deployable, provider/key handoff, functional regression.

The mobile Final Showcase already has a generic `Resources` stage based on backend provider readiness. That stage is useful for local-vs-real provider mode, but it does not summarize the final handoff path the operator must complete before configured acceptance: final resource fill guide, apply preview, resource handoff, live provider evidence, and configured evidence plan.

## Options

1. Add a dedicated `provider_handoff` Final Showcase stage. This is the recommended path. It gives the top mobile digest a direct view of the required external key/resource blocker without changing backend readiness semantics or writing any config.
2. Expand the existing `Resources` stage to include final handoff rows. This would avoid one more row, but it would mix provider readiness with final resource application state and make the existing local-demo resource status harder to read.
3. Add only another static visual artifact. This would document the blocker, but the actual mobile summary model would still hide it.

## Decision

Add a dedicated `provider_handoff` stage to `FinalShowcaseSummaryBuilder` after `resources` and before `local_smoke`. The stage will be present only when `finalLaunchSummary` is available, matching the other final launch digest stages.

The stage consumes existing sanitized rows from `FinalLaunchMobileSummary` in this priority order:

1. `liveProviderEvidenceRows`, when present and not ready.
2. `applyPreviewRows`, when live provider evidence is absent or ready but resource application is still blocked.
3. `resourceFillGuideRows`, when apply preview is absent or ready.
4. `resourceHandoffRows`, as a fallback for existing backend/iOS handoff status.

The stage status is:

- `ready` when the first selected row starts with a ready/pass phrase such as `Live evidence ready`, `Apply preview ready`, `Fill guide ready`, or `Resource handoff ready`.
- `waiting` when rows are missing, have not loaded, or ask the operator to run/refresh a report.
- `needsAttention` when rows include blocked, missing, failed, partial, consent, or required-input language.

For blocked detail, the stage should show the first selected status row plus the first concrete actionable row from the combined provider handoff rows. Actionable rows are rows that contain `provide`, `fill`, `make`, `run`, `consent`, `missing`, `blocked`, or `required`, excluding freshness-only lines. This lets a missing live evidence row still surface a specific next input such as `provide MESHY_API_KEY`. All output must pass through the existing Final Showcase sanitizer so provider keys, local file paths, checkout/payment links, and raw private context never appear.

## Data Flow

`FinalDemoLaunchReport` already includes final resource fill guide, apply preview, resource handoff, configured evidence plan, live provider evidence, and print fulfillment readiness. `FinalLaunchMobileSummaryBuilder` already converts these reports into compact row arrays. P0.161 will not add backend fields. It will only teach `FinalShowcaseSummaryBuilder` to pick the highest-priority provider-handoff rows and render a compact stage.

The overall readiness calculation must treat `provider_handoff` as a final launch stage. A ready local demo with configured final launch data should require this stage to be ready, while a blocked provider handoff should make the Final Showcase overall status `needsAttention`.

## Tests

Swift contract tests will cover:

- Ready provider handoff is inserted after `resources` and before `local_smoke`.
- Missing keys/apply preview rows produce `.needsAttention` and include a concrete action such as `provide MESHY_API_KEY`.
- Unsafe provider-handoff detail is redacted from encoded JSON.

Backend source acceptance will require the new stage function, id, label, and Swift test names.

Visual regression will add a 390x844 artifact showing `Final Showcase`, `Provider Handoff`, missing provider key detail, and `provide MESHY_API_KEY` so the final mobile digest proves the blocker is visible.

## Safety

This is a read-only mobile summary change. It does not write `.env`, iOS deploy config, final resources, key material, or provider evidence. It does not run Xcode, signing, backend servers, live Meshy/OpenAI/Treatstock calls, checkout/payment flows, or global machine commands. It consumes rows that are already sanitized and sanitizes the final detail again before display.
