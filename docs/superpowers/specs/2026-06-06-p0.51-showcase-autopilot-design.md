# P0.51 Showcase Autopilot Design

## Product Intent

The iOS showcase can now capture, forge, preview a 3D artifact, run NPC
autonomy, show NPCs around the relic, and request a print quote. The remaining
demo risk is operator sequencing: the presenter still needs to remember which
button to tap next. P0.51 adds a small, state-driven autopilot control that
advances the demo one safe step at a time.

This is not a background automation engine. It is an operator aid for the final
iPhone demo.

## Product Design Brief

- Product: Personal Myth Forge iOS app.
- Screen/component: existing `Demo Script` panel near the top of `ForgeRootView`.
- Visual source: current compact SwiftUI app style; no external design context
  exists.
- Interactivity: fully working source path compiled by SwiftPM.

## Approaches Considered

1. **One-button sequential autopilot, recommended.**
   Add a pure Swift planner that decides the next safe action from current app
   state, then wire one button to execute that action. This keeps the UI small,
   testable, and reversible.

2. **Fully automatic background pipeline.**
   Start forging, autonomy, and quote generation as soon as prerequisites are
   available. This is too risky for a live demo because provider failures and
   network delays would feel hidden.

3. **Manual checklist only.**
   Keep the current P0.49 script without executable assistance. This is safe but
   does not reduce final demo operation risk.

P0.51 uses approach 1.

## Scope

Add a Swift core planner:

- `ShowcaseAutopilotAction`
- `ShowcaseAutopilotPlan`
- `ShowcaseAutopilotPlanner`

Inputs:

- `DemoScript`
- `ForgeFlowPhase`
- optional ready `MythSession`
- current NPC tick count
- optional `PrintQuote`
- provider readiness or readiness error
- operation flags for forge/autonomy/quote loading

Outputs:

- next action enum
- button title
- status detail
- whether the action is executable
- sanitized stop reason when blocked

Wire the plan into the iOS app:

- `ForgeRootView` computes a single plan from current state.
- `DemoScriptView` accepts the plan and a `runAutopilot` closure.
- The button is disabled while any relevant operation is already running.
- Running the button calls exactly one existing action:
  - `forgeMyth()`
  - `runAutonomy()`
  - `requestPrintQuote()`

## Behavior

The planner picks the first safe next action:

1. If capture media is not ready, action is `blocked` with "Capture object first."
2. If forge is already running, action is `waiting`.
3. If no session exists and capture is ready, action is `forge`.
4. If a session exists but the 3D scene is not ready, action is `waiting`.
5. If NPC tick count is below 3 and session id is backend-owned, action is
   `runAutonomy`.
6. If NPC tick count is below 3 but the session id is not backend-owned, action
   is `advanceNPC`.
7. If print quote is missing after NPC completion, action is `requestQuote`.
8. If provider readiness is blocked or unreachable, action is `blocked`.
9. Otherwise action is `complete`.

Unsafe strings from provider errors or script details are replaced with
`[withheld]`.

## Non-Goals

- No new backend endpoint.
- No live provider calls in tests.
- No auto-start on app launch.
- No background retry loop.
- No signing, Xcode license, simulator, or iPhone deployment changes.
- No global environment mutation.

## Tests

Swift core contract tests must prove:

- capture missing blocks autopilot
- capture-ready/no-session plans `forge`
- ready session with backend id and fewer than three ticks plans `runAutonomy`
- legacy session id plans `advanceNPC`
- NPC-complete/no-quote plans `requestQuote`
- quote-ready and provider-ready plans `complete`
- loading flags disable executable actions
- unsafe text is redacted

Swift project/source checks must prove:

- `DemoScriptView` renders an autopilot button
- `ForgeRootView` builds `ShowcaseAutopilotPlanner.plan`
- `ForgeRootView` has a `runShowcaseAutopilot` action
- the SwiftPM app compile target still builds

Backend source acceptance must prove:

- `ios_showcase_acceptance` includes `showcase_autopilot`
- final acceptance stubs use `ios_showcase_acceptance` internal
  `passed=12`, `failed=0`
