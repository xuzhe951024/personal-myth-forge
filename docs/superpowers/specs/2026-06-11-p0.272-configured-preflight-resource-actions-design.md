# P0.272 Configured Preflight Resource Actions Design

## Context

Configured preflight is the operator-facing gate for proving the final provider,
resource, configured launch, and iOS deploy inputs before a real demo. After
P0.271, final demo launch actions are validation-aware, but
`final-configured-preflight.json` still surfaces bare top-level
`final-resources.env` actions from its embedded preflight and handoff reports.

## Goal

Top-level configured preflight operator actions should tell the operator to
rerun `make final-resources-preflight` after providing any known
`final-resources.env` value.

## Non-Goals

- Do not alter embedded child reports or their raw blocker fields.
- Do not apply resources, call live providers, invoke Xcode/signing, touch
  Keychain, or place print orders.
- Do not change configured preflight status calculation or command ordering.

## Design

`final_configured_preflight` should reuse the shared
`add_final_resource_validation_command()` helper when assembling top-level
operator actions. The helper already canonicalizes known final-resource actions,
including actions that arrive with an unrelated rerun suffix.

The report can still embed child reports with their original blocker commands;
only the top-level `operator_actions` field is normalized for operator handoff.

## Acceptance

- A RED/GREEN test proves top-level configured preflight actions include
  `provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight`
  and no known bare final-resource actions.
- Existing configured preflight and final demo launch tests still pass.
- Refreshed `final-configured-preflight.json` top-level `operator_actions` has
  no known bare final-resource action and no final-resource action rerunning a
  non-resource command.
- Safety flags remain read-only: no provider calls, env writes, iOS config
  writes, global mutation, Xcode/signing, or Keychain writes.
