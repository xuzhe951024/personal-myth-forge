# P0.288 Resource Apply Unblock Normalization Design

## Goal

Normalize the remaining vague final resource apply-chain operator actions into concrete commands for the final iPhone showcase handoff.

## Context

The project is still blocked by external requirements: provider keys, Treatstock configuration, iOS deploy settings, an iPhone-reachable backend URL, and physical iPhone/Xcode evidence. Local code can still improve the final operator path by ensuring every reported next action is executable.

After P0.287, current final showcase readiness still includes:

```text
unblock final_resource_apply_preview after final_resource_fill_guide
unblock final_apply_resources after final_resource_apply_preview
```

These come from the configured evidence plan dependency chain. They are accurate as status descriptions, but weak as operator actions because the project already has explicit make targets:

```text
make final-resource-apply-preview
run make final-apply-resources to apply the filled resource bundle
```

## Design

Extend `myth_forge_api.operator_actions.normalize_operator_action()` with exact mappings for these two known unblock phrases:

- `unblock final_resource_apply_preview after final_resource_fill_guide` becomes `make final-resource-apply-preview`
- `unblock final_apply_resources after final_resource_apply_preview` becomes `run make final-apply-resources to apply the filled resource bundle`

Keep the mapping in the existing `UNBLOCK_ACTION_REPLACEMENTS` table so bare, prefixed, and detail-suffixed report actions share one normalization path.

## Scope

In scope:

- public operator action normalization for the two resource apply-chain phrases
- focused tests for direct normalization and final-showcase aggregation
- roadmap entry for P0.288
- refreshed local report verification

Out of scope:

- running `make final-apply-resources`
- writing ignored real resource files
- provider calls, live cost actions, Xcode signing, Keychain, or device actions
- changing the configured evidence plan dependency model

## Verification

This iteration is valid when:

- focused RED tests show the resource apply unblock phrases currently leak through
- focused GREEN tests show they normalize to concrete commands
- refreshed `.local` final showcase readiness no longer exposes those vague phrases
- backend lint and backend tests pass
