# P0.255 Final Demo Launch Concrete Next Action Design

## Goal

Make the local final demo launch report surface the same concrete mobile deploy handoff that final showcase readiness already exposes. The iPhone Final Launch panel should show the operator the actual next edit, such as providing `DEVELOPMENT_TEAM`, plus the validation command to rerun afterward.

## Current Gap

`final-showcase-readiness.json` now promotes the saved mobile preflight child action, but `final-demo-launch-local.json` still derives `next_action` directly from the `mobile_deploy_preflight` phase. That leaves the iPhone-facing launch receipt with the generic command `make mobile-deploy-preflight` instead of the concrete blocker-specific action.

## Requirements

- Local final demo launch keeps `first_blocker.id == "mobile_deploy_preflight"` for the iOS deployment lane.
- When saved mobile preflight/final showcase evidence contains a concrete next action, `final-demo-launch-local.json.next_action.command` uses that action.
- The next action preserves `validation_command`, so the operator can rerun the right safe check after making the local ignored edit.
- iOS mobile-core models decode the optional validation command for `FinalDemoLaunchNextAction`.
- The iPhone Final Launch receipt includes the validation command when present.

## Non-Goals

- No real provider calls.
- No Xcode signing, Keychain writes, or global macOS mutation.
- No change to readiness status semantics.
