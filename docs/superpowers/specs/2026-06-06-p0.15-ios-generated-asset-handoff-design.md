# P0.15 iOS Generated Asset Handoff Design

## Goal

Move the iOS client from a proxy-only 3D preview toward a real generated-asset
handoff by adding a SwiftPM-tested mobile core service that downloads remote
generated assets, stores them in an app-local cache, and exposes a local
SceneKit-ready URL when the format can be loaded directly.

## Context

P0.14 added `ArtifactPreviewState` and a SceneKit preview surface, but the app
still renders proxy geometry even when the backend returns a generated asset
URI. The current backend already keeps provider API keys server-side and returns
`GeneratedAsset` metadata. P0.15 preserves that boundary: iOS never receives
Meshy, OpenAI, or print-provider keys. It only prepares the asset URI that the
backend already approved.

## Approaches Considered

1. **App-only download inside `Artifact3DPreviewView`**: fastest, but hard to
   test and would mix networking, caching, status classification, and SceneKit
   view code.
2. **Core asset preparation service plus app wiring**: adds a small tested
   service boundary. The app triggers it and uses the returned local reference.
   This is the chosen approach because it keeps file/network behavior testable
   without simulator deployment.
3. **Backend converts every generated asset to USDZ before iOS sees it**: cleaner
   eventual product path, but it depends on provider and conversion work that is
   not needed to make the mobile client ready for a generated-asset handoff.

## Design

Add `ArtifactAssetPreparation.swift` to `PersonalMythForgeMobileCore`.

The core service exposes:

- `ArtifactAssetDownloader`: downloads bytes for a remote URL.
- `ArtifactAssetCache`: stores bytes under a deterministic safe filename and
  returns a local file URL.
- `PreparedArtifactAsset`: immutable result with preview state, source URI,
  cached URL, optional SceneKit scene URL, status, status title, and detail.
- `ArtifactAssetPreparer`: classifies the session asset, downloads HTTP/HTTPS
  assets, stores them locally, and returns whether SceneKit can load the cached
  file directly.

Supported direct SceneKit formats remain `dae`, `scn`, `usd`, and `usdz`.
Remote GLB/GLTF assets are downloaded and cached, but return no `sceneURL`; their
status explicitly says conversion/import is still required. That keeps the
current Meshy GLB path honest while making later conversion work a small
extension of the same handoff boundary.

## App Wiring

`Artifact3DPreviewView` receives a default live `ArtifactAssetPreparer`, starts
preparation when a ready session appears, and displays the prepared status. When
`sceneURL` is available, it attempts `SCNScene(url:options:)`; otherwise it
falls back to the proxy geometry from P0.14.

The app cache root uses the user/app caches directory at runtime. This is not a
global environment mutation and is safe for an iOS app sandbox. Local verification
does not run the app binary or write cache files.

## Out Of Scope

- Meshy API key handling on mobile.
- Provider-side USDZ export or GLB-to-USDZ conversion.
- Runtime GLB import.
- Xcode simulator/device deployment or signing.
- ARKit mesh capture runtime.
- Unity village import.

## Testing And Evidence

- Core contract tests cover local SceneKit-ready assets, remote USDZ download and
  cache, remote GLB download/cache with conversion-required status, and invalid
  URI handling.
- Project source checks require the new core file and app wiring strings,
  including `ArtifactAssetPreparer`, `.task(id:)`, and `SCNScene(url:`.
- `PersonalMythForgeMobileAppCompileCheck` compiles the updated SwiftUI/SceneKit
  app source under SwiftPM.
- Static Browser evidence shows the mobile asset handoff state and the GLB
  conversion caveat in a 390x844 viewport.
