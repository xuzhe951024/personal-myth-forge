# P0.425 Final Resource Requirements Action Bundle Design

## Goal

Expose a structured action bundle from `final_resource_requirements` so the final local refresh and mobile launch evidence can show resource/key handoff steps as typed actions instead of only string operator commands.

## Current State

`final_resource_requirements` already reports sanitized requirement rows, a first blocker, next action, operator actions, validation commands, and source reports. Missing API keys and iOS deploy values are currently visible, but they are not available as a `device_action_bundle`, while downstream final reports already preserve that field generically.

## Approach

Add `device_action_bundle` to the final resource requirements report. The bundle will contain one action per blocking requirement, derived only from already-sanitized requirement metadata. Each action records the command, validation command, destination, domain, unblocked capabilities, and safety flags. The bundle never includes resource values, absolute local paths, live provider calls, writes, Xcode signing, or Keychain mutations.

## Safety

This is report shaping only. The implementation reads the existing ignored resource handoff state through the current preflight path and emits sanitized instructions. It does not write `.env`, write iOS deploy config, call Meshy/OpenAI/Treatstock, run Xcode, sign builds, or place orders.

## Testing

Add a focused contract test for missing resources. It should assert the new bundle includes missing Meshy/OpenAI/iOS resource actions, preserves validation commands and unblocked capability IDs, reports no provider/global/Xcode actions, and does not leak synthetic secrets or local temp paths.
