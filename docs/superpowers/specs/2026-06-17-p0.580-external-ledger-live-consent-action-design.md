# P0.580 External Ledger Live Consent Action Design

## Context

The final showcase and live-provider evidence reports now point operators to the
guarded live-provider command:

```text
PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence
```

`final-external-action-ledger.json` still promotes the live-provider group first
blocker as:

```text
make live-provider-evidence
```

That command is a validation/report refresh step, not the cost-consented action
that creates the missing Meshy/OpenAI evidence. The ledger should keep the
validation command, but its top-level next action should be the guarded live
provider handoff.

## Goal

When the external action ledger's first blocker is a live-cost action and that
action has an `operator_action`, the promoted blocker/next action should use the
operator action as its command.

## Approach

- Keep live action rows unchanged, including their diagnostic `command`.
- In `_blocker_command_for_action()`, prefer `operator_action` for live actions
  that require cost consent.
- Preserve `validation_command` derived from the row command, so the report still
  tells the operator to rerun `make live-provider-evidence`.

## Non-Goals

- Do not call Meshy/OpenAI/Treatstock or any live provider.
- Do not change provider evaluation ordering.
- Do not change configured evidence bundle routing in `operator_actions`.
- Do not write ignored env files or iOS deploy config.

## Verification

- Add a RED test for ready resources + ready iOS runbook where the first blocker
  is live-provider evidence.
- Verify first blocker and next action use the guarded consent command and keep
  `validation_command: make live-provider-evidence`.
- Run focused external ledger tests, lint, and `git diff --check`.
- Refresh safe reports and inspect `.local/final-external-action-ledger.json`.
