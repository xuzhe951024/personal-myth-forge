# P0.578 Final Refresh Nonblocking Resource Handoff Partial Design

## Context

After P0.577 and a successful local device rehearsal refresh, the final
showcase readiness first blocker correctly moved to live provider evidence.
However, `final-local-report-refresh.json` still marks `resource_handoff` as a
blocked step when the saved `resource-handoff.json` is:

```json
{
  "status": "partial",
  "summary": {"ready": 11, "missing": 0, "blocked": 0, "manual": 0, "optional": 2},
  "first_blocker": null,
  "next_action": null,
  "operator_actions": []
}
```

That report means optional resource handoff items remain deferred, not that the
local refresh sequence is blocked. Promoting it to a blocker creates noisy
operator actions such as `review refreshed resource_handoff report` and obscures
the real remaining live-provider consent gate.

## Goal

Treat `resource_handoff` partial reports with no blocker and no actionable
missing/blocked/manual summary counts as nonblocking in final-local refresh.

## Approach

Keep the existing final-local refresh report shape. Add a narrow effective
status helper in `final_local_report_refresh.py`:

- For `resource_handoff` only, when raw status is `partial`, `first_blocker` and
  `next_action` are absent, `operator_actions` is empty, and summary `missing`,
  `blocked`, and `manual` are all zero, report the refresh step as `ready`.
- Preserve `raw_status: partial` and the saved summary for operator visibility.
- Leave all other partial/blocked reports unchanged.

## Non-Goals

- Do not mark live provider evidence ready.
- Do not run Meshy/OpenAI/Treatstock or any live provider.
- Do not change resource handoff's own status.
- Do not change iOS/Xcode signing behavior.

## Verification

- Add a RED test proving a nonblocking partial `resource_handoff` step is not a
  final-local-refresh blocker and does not emit `review refreshed
  resource_handoff report`.
- Keep existing blocked resource handoff coverage for missing/blocked resources.
- Run focused final-local refresh and closure/readiness tests.
- Run lint and `git diff --check`.
- Refresh safe reports and verify the first blocker remains the live provider
  consent path rather than resource handoff noise.
