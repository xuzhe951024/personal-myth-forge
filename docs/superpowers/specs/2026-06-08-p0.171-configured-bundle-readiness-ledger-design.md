# P0.171 Configured Bundle Readiness Ledger Design

## Decision

Extend the backend final showcase readiness ledger so it consumes the configured
live evidence bundle. P0.170 made the iPhone `Provider Handoff` and
`Live Provider Consent` surfaces block on `configuredLiveEvidenceBundle`; the
backend `final_showcase_readiness_report` must enforce the same gate.

## Current Gap

`final_showcase_readiness.py` currently builds and summarizes
`live_provider_evidence`, `final_resources`, `final_resource_apply_preview`, and
`resource_handoff`. It does not build `configured_live_evidence_bundle`, include
it in `evidence`, add its operator actions, or require it for
`provider_key_handoff`.

That leaves a mismatch:

- iOS can block configured acceptance because the configured bundle is blocked.
- The backend final showcase readiness ledger can still describe provider handoff
  only in terms of live provider evidence, without the canonical configured
  bundle state.

## Considered Approaches

1. Add a new `configured_evidence_bundle` capability.
   This is explicit, but it expands the final showcase capability list and
   duplicates the existing provider/key handoff responsibility.
2. Fold the configured bundle into `provider_key_handoff`.
   This keeps the capability model stable and makes provider/key handoff mean
   "all resource, handoff, live evidence, and configured bundle evidence is
   ready."
3. Only add the bundle to the `evidence` object.
   This improves visibility but does not prevent a misleading ready handoff
   status.

Use approach 2 with the visibility from approach 3.

## Scope

- `build_final_showcase_readiness_report` builds
  `build_configured_live_evidence_bundle_report`.
- The report includes
  `evidence.configured_live_evidence_bundle`.
- `provider_key_handoff` includes
  `configured_live_evidence_bundle:<status>` in its evidence list.
- `provider_key_handoff` is ready only when final resources, apply preview,
  resource handoff, live provider evidence, and configured live evidence bundle
  are all ready.
- If local resource gates are ready but the configured bundle is blocked,
  `provider_key_handoff` is `partial`, its command is
  `make configured-live-evidence-bundle`, and operator actions include the
  configured bundle's next action.
- Privacy/safety checks include the configured bundle safety flags.
- Commands include `make configured-live-evidence-bundle`.

## Non-Goals

- Do not run live provider calls.
- Do not write backend `.env`, iOS deploy config, keychain, signing, or Xcode
  state.
- Do not add new iOS UI.
- Do not change `configured_live_evidence_bundle.py` semantics.
- Do not change the capability count or order.

## Tests

Use TDD:

- Add a failing test showing a blocked configured bundle keeps
  `provider_key_handoff` partial and exposes its evidence/action.
- Add a failing test showing a ready configured bundle is included in report
  evidence and lets provider handoff be ready when the existing gates are ready.
- Add a safety/redaction assertion that unsafe configured bundle detail does not
  leak secrets or local paths through final showcase readiness.

Verification:

- `cd services/backend && uv run pytest tests/test_final_showcase_readiness.py -q`
- `make backend-lint`
- `cd services/backend && uv run pytest tests/test_final_demo_launch.py tests/test_final_launch_closure_packet.py -q`

The last command verifies downstream reports that consume final showcase
readiness still tolerate the expanded evidence and action set.
