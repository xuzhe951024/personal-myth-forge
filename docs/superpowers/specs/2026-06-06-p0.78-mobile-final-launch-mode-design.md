# P0.78 Mobile Final Launch Mode Design

## Context

P0.77 made the final operator handoff visible on iPhone, but the app still
loads `/v1/final-demo-launch` with `mode=local` only. That is useful for no-key
demo operation, but it hides the configured Meshy/OpenAI/Treatstock readiness
view that matters when final API keys are supplied.

The project already has the backend `local` and `configured` final launch
profiles. The missing piece is a safe mobile mode handoff: the final resource
bundle should be able to set the default mobile final launch mode, and the app
should allow the operator to switch between `local` and `configured` without
running Mac commands or provider calls.

## Decision

Add `PMF_FINAL_LAUNCH_MODE` as a project-local iOS deployment value. It is
validated by final resource preflight, written by unified final resource apply,
exposed through `Info.plist`, and read by `AppConfiguration.finalLaunchMode`.

Add a mobile final launch mode control to the iPhone app. The operator can
switch between `local` and `configured`; the app re-fetches
`/v1/final-demo-launch?mode=<mode>` and renders the existing read-only final
launch panel. This is a mode selection for report visibility only. It does not
call live providers, run final acceptance, start Xcode, deploy to a device, or
write any local files.

## Backend And Resource Shape

Add `PMF_FINAL_LAUNCH_MODE` to:

- `services/backend/final-resources.env.example`
- final resources accepted keys
- final resources preflight
- unified final resource apply
- iOS deployment config writer if needed
- resource template/source acceptance

Allowed values are `local` and `configured`. Missing value defaults to `local`
for backwards compatibility. Any other non-empty value is `blocked` with
classification `unsupported_value`.

`Deployment.xcconfig` gets:

```text
PMF_FINAL_LAUNCH_MODE = local
```

`Deployment.local.xcconfig.example` documents the same key. Final resource
apply writes it to ignored `Deployment.local.xcconfig`.

## iOS Shape

Add a small mobile-core enum:

- `FinalLaunchMode.local`
- `FinalLaunchMode.configured`
- `safe(rawValue:)` fallback to `.local`
- `displayLabel` for segmented UI

`Info.plist` exposes `PMFFinalLaunchMode = $(PMF_FINAL_LAUNCH_MODE)`.

`AppConfiguration.finalLaunchMode` reads and sanitizes the plist value.

`ForgeRootView` owns:

- `@State private var finalLaunchMode = AppConfiguration.finalLaunchMode`

`loadFinalDemoLaunch(mode:)` calls:

```swift
apiClient.getFinalDemoLaunch(mode: mode.rawValue)
```

The view renders a segmented picker for `local` and `configured`, plus the
existing `FinalLaunchStatusView`. Changing the segment reloads the report.

## Testing

Backend TDD:

- final resources preflight accepts missing mode as local-compatible
- final resources preflight blocks unsupported mode
- final resource apply writes `PMF_FINAL_LAUNCH_MODE`
- resource template/source acceptance requires the new key
- iOS showcase acceptance requires mobile mode source wiring

iOS TDD:

- `FinalLaunchMode.safe(rawValue:)` maps `configured`, `local`, invalid, and
  unsafe strings correctly
- final launch API still sends `mode=configured`
- project checks require plist, xcconfig, app config, segmented picker, and
  reload on mode change

Visual evidence:

- Static 390x844 screenshot showing a `Final Launch Mode` segmented control,
  local/configured options, and a configured handoff panel.

## Non-Goals

- Do not call live providers from the iPhone when switching modes.
- Do not run final acceptance from the iPhone.
- Do not run Xcode, signing, simulator, device deployment, archive, or Apple
  SDK license commands.
- Do not store provider keys or raw local resource files in the app.
