# P0.102 3D Generation Input Review Design

## Goal

Make the iPhone demo show a clear pre-forge receipt for the current capture input and the 3D provider route it will drive. The operator should know whether the next forge request is waiting, locally demo-ready, or ready for a real Meshy route before tapping Forge.

## Product Behavior

- Add a mobile-core `ThreeDGenerationInputReview` derived from:
  - current `CaptureMediaSelection`
  - existing `CaptureGenerationReadiness`
  - backend provider readiness and provider readiness error
- Show the receipt in `CaptureFormView` for every capture mode, not only guided scan and ARKit scan.
- The receipt summarizes:
  - capture mode
  - source media counts
  - provider route label (`text_prompt`, `single_image`, `multi_image`, `scan_asset`, or `waiting`)
  - how many source images/assets the provider will use
  - whether the route is local demo-only or real-provider ready
- Guided/photo-set routes must explicitly show provider image selection, for example `selected 4/9 provider images`.
- ARKit scan package routes must show scan asset count and reference image count.
- Single-photo routes must show `image-to-3D`.
- Text-prompt fallback stays visible as `waiting` until capture media is selected; this app remains capture-first.

## Privacy And Safety

- The receipt must not display raw filenames, local filesystem paths, local capture URIs, data URIs, checkout URLs, raw personal context, or provider keys.
- It may show counts, normalized content-type families, route labels, and backend-only missing env names such as `MESHY_API_KEY`.
- Provider readiness errors are sanitized before display.
- The receipt does not call Meshy, OpenAI, Treatstock, or any external provider.

## UI Placement

`CaptureFormView` renders the receipt directly under the capture input controls and before context approval. Existing forge readiness and forge progress panels stay in place. The receipt is compact enough for the mobile demo but includes enough rows for scan/photo/provider review.

## Acceptance

- Swift contract tests cover waiting, guided multi-image selection, ARKit scan package review, real Meshy readiness, and unsafe text redaction.
- Swift ProjectChecks require the mobile-core builder, SwiftUI view, root wiring, capture form handoff, Xcode membership, and contract test names.
- Backend iOS showcase acceptance adds a required `mobile_3d_generation_input_review` feature.
- A 390x844 visual regression image shows waiting, guided scan, ARKit scan, and Meshy-ready review states without text overlap or clipping.
