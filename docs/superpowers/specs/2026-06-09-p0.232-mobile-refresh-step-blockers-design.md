# P0.232 Mobile Refresh Step Blockers Design

## Context

P0.231 made `final-local-report-refresh` step rows expose top-level
`classification`, `command`, and `detail` fields. The iOS final launch summary
already reads `ios_device_launch_rehearsal_readiness.sequence`, but the mobile
model only preserves `classification` and `command`. That means the iPhone
status screen can show which safe refresh step is blocked, but not the operator
detail that explains why.

## Goal

Let the mobile final launch summary decode and display launch rehearsal sequence
row `detail` values. A blocked sequence row should surface:

- step id and status
- command
- classification
- detail

This keeps the safe local refresh blocker visible on-device without exposing
provider keys, raw media, local capture URLs, payment links, or bearer tokens.

## Scope

This is an iOS model and summary-display change only. It does not change backend
report generation, provider calls, resource application, Xcode behavior, signing
configuration, or iOS deployment commands.

## Architecture

Add optional `detail` to `IOSDeviceLaunchRehearsalSequenceRow`. Update
`launchRehearsalSequenceRow` to append the detail when present. Keep the
existing sanitizer at the final row boundary so new detail text is redacted the
same way as commands and operator actions.

## Testing

Use TDD:

1. Add a contract test proving `detail` decodes from the final demo launch
   payload.
2. Add a summary test proving blocked rehearsal rows include detail.
3. Extend the existing redaction test so unsafe detail content is withheld.
4. Implement the model and summary change.
5. Verify focused Swift contracts, project checks, and diff hygiene.
