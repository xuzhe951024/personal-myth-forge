# P0.520 Mobile Preflight Concrete Next Action Design

## Context

The final launch reports now promote the iOS device blocker as a complete command chain:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

The source report, `mobile-deploy-preflight-evidence`, still exposes:

```text
next_action.command = DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto
next_action.validation_command = make mobile-deploy-preflight
operator_actions[0] = DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

That forces downstream reports to repair the command for human handoff. The source report should be copy-pasteable on its own.

## Goal

Make `mobile-deploy-preflight-evidence.next_action.command` validation-aware for blocked preflight actions, while keeping `validation_command` as structured metadata.

## Non-Goals

- Do not make mobile deploy preflight pass without real `DEVELOPMENT_TEAM`, iPhone-reachable `PMF_BACKEND_BASE_URL`, and backend health evidence.
- Do not write `Deployment.local.xcconfig`, Keychain entries, signing data, backend env files, provider keys, or any other local resource.
- Do not run Xcode, live providers, or paid actions.
- Do not change the command used to execute the preflight itself: it remains `make mobile-deploy-preflight`.

## Behavior

When a blocked preflight has a raw remediation action, the report should expose a validation-aware top-level `next_action.command`:

```text
<raw action>; rerun make mobile-deploy-preflight
```

Examples:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
provide DEVELOPMENT_TEAM in Deployment.local.xcconfig; rerun make mobile-deploy-preflight
start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight
```

`validation_command` remains `make mobile-deploy-preflight`. Ready reports still have no first blocker, no next action, and no operator actions.

## Testing

Add a focused RED test in `services/backend/tests/test_mobile_deploy_preflight_evidence.py` that asserts the auto-writer `next_action.command` is the same validation-aware command already emitted as `operator_actions[0]`.

Verification commands:

```bash
cd services/backend && uv run pytest tests/test_mobile_deploy_preflight_evidence.py::test_preflight_evidence_next_action_promotes_validation_aware_writer_command -q
cd services/backend && uv run pytest tests/test_mobile_deploy_preflight_evidence.py -q
make backend-lint
make backend-test
make mobile-deploy-preflight-evidence
```

The refreshed `.local/mobile-deploy-preflight-evidence.json` should remain `blocked`, but `next_action.command` should be:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```
