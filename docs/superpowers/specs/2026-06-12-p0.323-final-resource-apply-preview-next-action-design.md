# P0.323 Final Resource Apply Preview Next Action Design

## Context

`final-resource-apply-preview.json` shows which ignored local resource values would be written into backend and iOS config files, but blocked reports currently expose only `first_blocker`. The top-level `next_action` is `null`, unlike the adjacent final resource preflight, requirements, and fill guide reports.

That leaves the iPhone Final Launch apply-preview section with a blocker row but no explicit next-action row.

## Decision

Add `next_action` to the final resource apply preview report:

- blocked or missing reports copy `first_blocker` and add `source: "first_blocker"`
- ready reports set `next_action` to `null`
- the action keeps `command`, `detail`, `destination`, `writer`, `blocked_by`, and `validation_command`

This is report-only. It does not apply resources, write `.env`, write `Deployment.local.xcconfig`, invoke Xcode, call providers, or mutate global state.

## iOS Handoff

`FinalResourceApplyPreviewReport` will decode optional `next_action` with the same shape as `first_blocker`, extended by optional `source`. `FinalLaunchMobileSummary` will show a `Next action:` row before the existing first blocker row.

The summary continues to sanitize displayed text.

## Acceptance

- Backend tests prove blocked/missing apply preview reports include `next_action` with `source: "first_blocker"`.
- Backend tests prove ready apply preview reports set `next_action` to `null`.
- Swift contract tests prove final launch payloads decode `final_resource_apply_preview.next_action`.
- Swift summary tests prove the iPhone Final Launch apply-preview rows show the next-action command and validation command.
- `make final-resource-apply-preview` still exits `2` in the current missing-resource state and only writes the ignored JSON report.
