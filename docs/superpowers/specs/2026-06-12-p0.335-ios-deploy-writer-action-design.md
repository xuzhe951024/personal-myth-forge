# P0.335 iOS Deploy Writer Action Design

## Context

The physical-iPhone launch path is still blocked by local deploy config values.
P0.334 made `mobile-write-deploy-config` reuse the committed bundle identifier,
so the common device-demo case now needs only an Apple Team ID and an
iPhone-reachable backend URL.

The readiness reports still surface separate manual actions:

- `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig`
- `set PMF_BACKEND_BASE_URL to an iPhone-reachable LAN URL`

Those actions are correct, but they hide the safer project-local writer command
that avoids hand-editing the ignored xcconfig.

## Decision

When `mobile-deploy-preflight-evidence` sees both of these blockers and does
not also see a missing bundle identifier, it should promote this command as the
first action:

```bash
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto
```

The report still keeps `validation_command: "make mobile-deploy-preflight"`.
The action normalizer treats the writer command as an iOS deploy action and
appends `; rerun make mobile-deploy-preflight` when it appears in
`operator_actions`.

## Non-Goals

- Do not write `Deployment.local.xcconfig` during report generation.
- Do not run Xcode, accept licenses, touch signing, write Keychain, or call any
  provider.
- Do not use real Apple Team IDs or secrets in tracked files.

## Acceptance

- A blocked preflight report with missing `DEVELOPMENT_TEAM` plus loopback or
  missing backend URL uses the writer command as `next_action.command`.
- `operator_actions[0]` appends the mobile deploy validation command.
- Existing backend-health and missing-bundle behaviors remain unchanged.
- Final local report refresh remains safely blocked without provider, Xcode, or
  config writes.
