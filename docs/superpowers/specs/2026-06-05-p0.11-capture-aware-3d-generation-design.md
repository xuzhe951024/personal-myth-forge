# P0.11 Capture-Aware 3D Generation Design

## Context

P0.10 connects the iOS capture draft flow to the backend from-capture session
loop. The backend still generates 3D assets from a myth prompt only. It preserves
capture media refs in the object card, but the 3D provider does not receive the
uploaded reference image or scan bundle as structured input.

For the final iOS demo, real-world media needs to shape the generated
`game_asset`. P0.11 adds the backend data path that passes capture media into the
3D provider. This does not require a real Meshy key in tests.

## Provider Fact

Meshy official API docs describe `POST /openapi/v1/image-to-3d` with an
`image_url` field that accepts either a publicly accessible image URL or a
base64-encoded data URI, and task polling at `GET /openapi/v1/image-to-3d/:id`.
P0.11 uses that contract for the adapter shape while keeping all tests on
`httpx.MockTransport`.

Source: https://docs.meshy.ai/en/api/image-to-3d

## Goal

Let `POST /v1/myth-sessions/from-capture` build a capture-aware 3D generation
request that includes local reference image bytes as data URIs, then route Meshy
generation through Image-to-3D when such an image source exists.

## Non-Goals

- Exposing local capture files through public URLs or object storage.
- Calling Meshy with a real key in tests.
- Multi-image-to-3D, scan-asset remeshing, or print repair.
- iOS model rendering or ARKit capture runtime.

## Architecture

Add backend 3D generation request types:

```python
@dataclass(frozen=True)
class ThreeDSourceImage:
    uri: str
    content_type: str
    data_uri: str

@dataclass(frozen=True)
class ThreeDSourceAsset:
    uri: str
    content_type: str

@dataclass(frozen=True)
class ThreeDGenerationRequest:
    session_id: str
    prompt: str
    source_images: tuple[ThreeDSourceImage, ...] = ()
    source_assets: tuple[ThreeDSourceAsset, ...] = ()
```

`ThreeDProvider.generate_game_asset` accepts a `ThreeDGenerationRequest`.

`LocalCaptureStore` gains `read_media(capture_id, media_id)` so routes can read
stored bytes without exposing paths or raw data in API responses. The method
validates capture ids, manifest membership, and generated media filenames.

The from-capture route resolves capture media into a generation request:

- `reference_image` with supported image bytes becomes a `ThreeDSourceImage`
  with a base64 data URI.
- `scan_asset` becomes a `ThreeDSourceAsset` with its local URI and content type.
- The route still returns only the normal `MythSession`; raw bytes and data URIs
  are not exposed to clients.

`MeshyThreeDProvider` chooses:

- Image-to-3D when `request.source_images` is non-empty.
- Existing Text-to-3D preview/refine path when no image source exists.

The image path posts:

```json
{
  "image_url": "<data URI>",
  "enable_pbr": true,
  "should_remesh": true,
  "target_formats": ["glb"],
  "should_texture": true
}
```

Then it polls `GET /openapi/v1/image-to-3d/:id` and reads `model_urls.glb`.

## Testing

Backend tests prove:

- `LocalCaptureStore.read_media` returns stored bytes for a known media id and
  rejects traversal or unknown media ids.
- `create_demo_myth_session` passes source images/assets to an injected 3D
  provider.
- `POST /v1/myth-sessions/from-capture` passes a base64 image data URI and scan
  asset metadata into an injected 3D provider.
- `MeshyThreeDProvider` sends image-to-3D payloads when a source image exists.
- Text-to-3D Meshy tests remain valid for prompt-only generation.

## Visual Evidence

Static storyboard evidence should show the backend path:

```text
iOS capture upload -> local capture store -> data URI source image -> Meshy image-to-3D -> GLB game_asset
```

It must say this is backend/provider-contract evidence, not a real Meshy run or
iOS 3D renderer.

## Acceptance Criteria

- Backend tests pass with no real external keys.
- Meshy adapter supports both prompt-only and image-source flows.
- From-capture sessions can create a ready `MythSession` using capture media as
  3D generation input.
- API responses still do not leak raw media bytes, filesystem paths, API keys,
  or data URIs.
- Docs identify the remaining gap: public/storage-backed media delivery,
  multi-image/scan remesh, and iOS 3D rendering.
