# P0.135 Launch Rehearsal Action Visibility Design

## Context

P0.133 and P0.134 made iOS launch rehearsal actions more concrete at the source
reports. The saved `ios-device-launch-rehearsal.json` now contains scoped actions
for `final_acceptance_local`, `final_handoff_index`, and
`ios_device_launch_certificate`.

The final demo launch payload embeds the rehearsal through
`ios_device_launch_rehearsal_readiness`, but that readiness report currently
keeps only five operator actions. The mobile summary also shows only the first
operator action when blocked sequence rows are present. As a result, the iOS
final launch status can miss the downstream handoff/certificate actions that matter
for device deployment.

## Decision

Make the rehearsal action chain visible without changing command execution:

- Backend readiness preserves up to twenty deduped operator actions from the
  saved rehearsal report. This keeps the list bounded while covering the current
  final acceptance, configured preflight, final handoff, and certificate action
  groups. The real saved launch rehearsal currently emits eighteen scoped
  actions, with certificate actions appearing after final handoff actions.
- Stale readiness still prepends the rerun action and then preserves bounded
  saved actions.
- iOS `FinalLaunchMobileSummary` appends up to three operator actions in
  blocked/partial rehearsal summaries after the sequence attention rows. This
  keeps the UI compact but makes the Mac operator's next actions visible.
- Existing sanitization remains the boundary for unsafe details.

## Alternatives Considered

1. Only raise the backend readiness cap. This helps API payloads but not the
   mobile summary because the UI still appends only one action when sequence
   rows exist.
2. Only change the mobile summary. This still cannot show actions that the
   readiness report has already dropped.
3. Add a new dedicated action ledger section to the mobile UI. This can be useful
   once the UI needs a full operator console, but it is broader than needed for this iteration.

## Data Flow

`ios_device_launch_rehearsal_readiness._operator_actions` receives the saved
rehearsal actions and returns a bounded sanitized list. The final demo launch
payload embeds that readiness report. `FinalLaunchMobileSummaryBuilder` turns
the readiness into `launchRehearsalRows`; for blocked/partial states, it keeps
the sequence context and appends the first three operator actions.

## Safety

This iteration does not run live providers, Xcode, signing, Apple SDK license
commands, keychain operations, or global configuration. It only changes local
report shaping and iOS summary rendering.
