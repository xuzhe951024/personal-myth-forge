# P0.524 Dedupe Readiness Detail Validation Design

## Context

After refreshing local evidence for the current `main`, `final-showcase-readiness` no longer reports stale local acceptance or visual evidence. The remaining final acceptance capability still carries a detail string like:

```text
Next action: DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight | rerun make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

The command is correct, but the detail text repeats the same validation rerun once from the command and once from `validation_command`.

## Goal

Make readiness detail strings copy-readable by avoiding duplicate validation rerun fragments when the child next action command already includes the same rerun.

## Scope

- Update final showcase readiness blocker-detail formatting.
- Preserve the current command and validation metadata.
- Preserve distinct validation commands when a command does not already include them.
- Add regression coverage for functional regression capability detail.

## Non-Goals

- Do not change iOS deploy, Xcode signing, live provider, print provider, or resource writing behavior.
- Do not remove validation commands from structured fields.
- Do not run real Meshy/OpenAI/Treatstock calls or any global Mac/Xcode action.

## Acceptance Criteria

- Functional regression detail contains one `rerun make mobile-deploy-preflight` fragment when the next action command already includes it.
- The capability still exposes `validation_command: make mobile-deploy-preflight`.
- The capability command remains the concrete deploy writer command plus validation rerun.
- Focused final showcase readiness tests pass.
- Backend lint and tests pass before merge.
