# P0.122 Final Showcase Readiness Ledger Design

## Purpose

Add one objective-level readiness ledger for the C-path final showcase goal: an iOS-deployable minimal demo with mobile capture/scanning, 3D generation, AI Agent NPCs, provider/key handoff, functional regression, and visual regression.

The project already has many focused reports. The remaining risk is operator ambiguity: final review requires reading several JSON files and deciding whether the overall product objective is truly proven. P0.122 adds a read-only ledger that maps each product requirement to existing evidence, shows the first blocker, and keeps final live-provider and iPhone/Xcode work explicit.

## Chosen Approach

Use a backend report builder named `final_showcase_readiness` and expose it through:

- `myth_forge_api.cli final-showcase-readiness`
- `make final-showcase-readiness`
- embedded `final_showcase_readiness` inside `/v1/final-demo-launch`
- iPhone `Final Launch Status` rows labeled `Showcase Readiness`

This reuses existing readiness reports and source acceptance gates instead of running commands or calling providers. The report is an auditor, not an orchestrator.

## Alternatives Considered

1. Add another rehearsal wrapper that runs every local command.
   This would be useful later, but it duplicates `final-rehearsal-local` and `ios-device-launch-rehearsal`. It also risks hiding which specific objective still lacks proof.

2. Add only a README checklist.
   This is cheaper, but not testable and not consumable by the iPhone final launch panel.

3. Add a structured ledger.
   This is the chosen path because it is machine-testable, iPhone-visible, and aligns with the existing report architecture.

## Report Shape

The report has:

- `kind`: `final_showcase_readiness_report`
- `status`: `ready`, `partial`, or `blocked`
- `summary`: counts by status
- `capabilities`: ordered rows for each final objective area
- `first_blocker`: the first non-ready required capability
- `operator_actions`: bounded next actions
- `commands`: safe commands to refresh evidence
- `evidence`: selected embedded report summaries
- `safety`: booleans proving the ledger is read-only and sanitized

Initial capabilities:

- `ios_deployable`
- `capture_scanning`
- `game_asset_3d_generation`
- `ai_agent_npc`
- `provider_key_handoff`
- `functional_regression`
- `visual_regression`
- `privacy_safety`

Each capability row includes `id`, `label`, `status`, `classification`, `required`, `evidence`, `command`, and `detail`.

## Status Rules

`ready` means current local evidence proves the capability without unhandled blockers.

`partial` means the local proof exists but final live-provider or iPhone/Xcode proof remains intentionally manual or consent-gated.

`blocked` means required evidence is missing, stale, failed, or unsafe.

The ledger exits `0` only when every required capability is `ready`. It exits `2` for `partial` or `blocked`, because this command answers whether the final product objective is fully proven, not whether the local rehearsal can run.

## Data Sources

The builder uses read-only sources:

- `run_ios_showcase_acceptance(repo_root)` for source-level iOS feature coverage
- `build_final_acceptance_readiness_report(repo_root)`
- `build_three_d_evaluation_readiness_report(repo_root)`
- `build_npc_agent_evaluation_readiness_report(repo_root)`
- `build_visual_regression_readiness_report(repo_root)`
- `build_live_provider_evidence_report(repo_root)`
- `build_ios_deploy_runbook_report(mode="local", repo_root=repo_root)`
- `build_ios_device_launch_rehearsal_readiness_report(repo_root)`
- `build_final_resources_preflight_report(repo_root)`
- `build_resource_handoff_report(settings, repo_root)`

The ledger does not call `build_final_demo_launch_report`; `final_demo_launch` will embed the ledger, so the dependency direction stays acyclic.

## iPhone Surface

Add optional decoding models for `FinalShowcaseReadinessReport` and a `showcaseReadinessRows` array to `FinalLaunchMobileSummary`. The iPhone row builder should show:

- one summary row with overall status and counts
- first blocker row when present
- up to two operator actions

Rows must be sanitized with the existing summary sanitizer.

## Safety

The ledger must not:

- run shell commands
- call live providers
- write backend env files
- write iOS deployment config
- invoke Xcode, signing, keychain, or device install steps
- include provider secrets, raw media, private context, payment links, or absolute local paths

## Test Strategy

Use TDD:

- backend unit tests for missing, partial, ready-ish, and sanitizer behavior
- CLI test for `final-showcase-readiness --output`
- final launch test proving `final_showcase_readiness` is embedded
- Swift contract tests proving decode and iPhone rows
- iOS showcase source acceptance requirement for the new final launch surface
- existing lint, backend tests, Swift contract/project/compile checks, visual regression, rehearsal checks, and `git diff --check`

## Out of Scope

P0.122 does not perform live Meshy/OpenAI/Treatstock calls, real device install, Apple signing setup, Xcode license acceptance, or Unity integration.
