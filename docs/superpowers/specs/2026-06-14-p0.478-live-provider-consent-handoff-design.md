# P0.478 Live Provider Consent Handoff Design

## Context

P0.477 added an execution-layer guard for configured Meshy/OpenAI commands. The remaining gap is operator guidance: `live-provider-evidence` can still emit live provider actions such as `make backend-evaluate-3d-configured` without showing the required local consent environment variable in the action itself.

## Goal

Make live provider handoff reports mirror the execution guard by rendering consent-required operator actions as `PMF_ALLOW_LIVE_PROVIDER_CALLS=1 <make target>; rerun make live-provider-evidence`.

## Scope

- Update report-only command text for live provider evidence.
- Preserve safe local provider handoff actions that do not require live provider consent.
- Keep commands in the canonical `commands` list unchanged so source gates still reference Make targets directly.
- Do not run Meshy, OpenAI, Treatstock, Xcode, signing, Keychain, or global environment actions.

## Behavior

When the first blocker is a slot with `requires_live_provider_consent: true`, `operator_actions` must prefix the slot command with `PMF_ALLOW_LIVE_PROVIDER_CALLS=1 `. The `next_action` continues to expose the canonical command and `requires_live_provider_consent: true` for structured clients.

When the first blocker is provider handoff or another safe local step, `operator_actions` must not include the consent prefix.

## Testing

- Add a failing unit test for a blocked configured 3D evaluation report that expects the consent-prefixed operator action.
- Keep existing provider handoff tests proving safe local actions are not prefixed.
- Add a source acceptance gate that requires the consent prefix to be present in the live provider evidence source.
- Run focused Python tests, Swift ProjectChecks, lint, and diff checks.

## Safety

This phase only changes generated report text and tests. It must not set `PMF_ALLOW_LIVE_PROVIDER_CALLS`, read provider secrets, write repo-tracked secrets, or invoke live provider targets.
