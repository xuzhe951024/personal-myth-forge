# P0.242 Rehearsal Preflight Detail Design

## Context

`final-demo-launch-local`, `final-showcase-readiness`, and `ios-device-evidence-bundle` now surface concrete saved mobile deploy preflight details from `services/backend/.local/mobile-deploy-preflight-evidence.json`. The final iOS wrapper report, `ios-device-launch-rehearsal`, still collapses the first blocked local rehearsal action to generic text such as "start backend-device-demo and rerun mobile deploy preflight".

`ios-device-launch-rehearsal` already reads `mobile-deploy-preflight-evidence.json` as one of its local rehearsal sources. The gap is that `_source_report` only exposes `operator_actions`, while the more specific details live in the saved evidence report's `checks[].detail` fields.

## Goal

Make `ios-device-launch-rehearsal` prefer saved mobile preflight check details when reporting blocked local rehearsal actions, without changing the public report shape.

## Non-Goals

- Do not add new iOS model fields or Swift UI rows.
- Do not run mobile deploy preflight, Xcode, signing, Keychain, or any global Mac mutation.
- Do not call Meshy, OpenAI, Treatstock, Sculpteo, or any live provider.
- Do not change final acceptance, final handoff, or certificate report generation.

## Approaches Considered

1. Extract actionable detail from `mobile-deploy-preflight-evidence.json` inside `ios-device-launch-rehearsal`.
   This is selected. It keeps the existing report shape and improves the first blocker with the evidence the builder already reads.

2. Depend on `ios-device-evidence-bundle.json` from the rehearsal wrapper.
   This would duplicate higher-level report dependencies and risks circular handoff ordering. The rehearsal report already has the source evidence needed.

3. Change `final-acceptance-local.json` so it stores preflight check details.
   This is broader and mixes final acceptance concerns with device evidence. The saved preflight evidence is the more direct source.

## Design

When `_source_report` reads a saved `mobile_deploy_preflight_evidence_report`, it will derive an internal `detail` from blocked or missing `checks[].detail` values. If multiple checks are blocked, details are joined with `; `. If no check details exist, the builder keeps the existing `operator_actions` fallback.

`_local_rehearsal_operator_actions` will prefer a blocked source's `detail` before nested operator actions. It will also replace generic upstream actions that merely say to rerun mobile deploy preflight when saved blocked preflight detail exists. This handles the real current ordering where `final_acceptance_local` appears before `mobile_deploy_preflight_evidence` in `LOCAL_REPORT_SOURCES`.

This makes `final_rehearsal_local.operator_actions[0]` and the top-level `first_blocker.detail` include concrete device requirements such as:

```text
final_rehearsal_local: mobile_deploy_preflight_evidence: Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

Missing report behavior and non-mobile report behavior remain unchanged.
