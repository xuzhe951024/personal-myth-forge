# P0.35 3D Provider Evaluation Suite Design

## Context

P0.34 made generated asset provenance explicit, so a run can prove whether the
3D provider used text, single-image, or multi-image inputs. The remaining gap
for the final key handoff is repeatability: once Meshy and OpenAI keys are
provided, the project needs a fixed 3D evaluation suite that can be run without
inventing prompts by hand, then reviewed for quality, latency, failure rate,
route coverage, asset variant coverage, and safety.

The current `evaluate-3d` CLI accepts a plain prompt file and writes a basic
report with success/failure and elapsed time. It does not define a canonical
20-case suite, does not preserve case categories, does not include provenance
or asset variants, and does not provide a review rubric for manual inspection.

## Approach Options

### Option A: Extend `evaluate-3d` with a Canonical Suite

Add a first-class evaluation module with a built-in `default-v0` suite of 20
text-driven cases. The existing CLI gains `--suite default-v0`, optional
`--prompts-file`, and richer report output. This keeps the workflow close to
the existing provider adapter and requires no new external system.

Trade-off: the first suite focuses on text/provider quality and only reserves
schema for future media-backed cases. It does not itself validate real iPhone
image inputs.

### Option B: Put the Suite Inside Final Acceptance

Run a 20-case suite as part of `final-acceptance --profile full`. This makes the
last handoff command stronger, but it also turns a long/costly live Meshy run
into a broad acceptance gate that should not run accidentally during normal
local regression.

### Option C: Build a Review Dashboard First

Create a richer HTML dashboard for scoring generated models. This would help
manual review later, but it adds UI surface before the report schema is stable.

## Decision

Use Option A. P0.35 extends `evaluate-3d` into a canonical, repeatable suite
runner while keeping live provider calls opt-in. Final acceptance can reference
the generated report later, but it should not automatically spend provider
credits yet.

## Feature Summary

P0.35 adds:

- a backend evaluation module for 3D provider cases and reports
- a built-in `default-v0` suite with 20 prompt cases
- richer `evaluate-3d` report fields:
  - `kind`
  - `suite`
  - `provider`
  - `total_cases`
  - success/failure counts
  - route/input-mode coverage
  - asset variant coverage
  - per-case category and expected input mode
  - generated asset provider, URI, format, variants, and provenance
  - per-case elapsed time
  - sanitized errors
  - empty manual review scoring fields
  - safety summary
- documentation and static mobile-width evidence showing how to run and review
  the suite

## Built-In Suite

The `default-v0` suite has exactly 20 text cases across five categories:

- `object_description`: direct physical object descriptions
- `object_plus_context`: real object plus personal capsule-style context
- `mythic_prompt`: backend-style myth seed prompts
- `material_stress`: difficult materials, transparency, fabric, patina, and
  mixed surfaces
- `printability_probe`: small relics that should later be assessed for print
  feasibility

Each case has:

- stable `case_id`
- `category`
- `prompt`
- `expected_input_mode`, initially `text_prompt`
- `review_focus`, a short phrase that tells a human reviewer what to judge

The suite intentionally contains no private user data and no raw media.

## CLI Contract

Existing prompt-file behavior remains valid:

```bash
cd services/backend
uv run python -m myth_forge_api.cli evaluate-3d \
  --provider local \
  --prompts-file /tmp/prompts.txt \
  --output /tmp/personal-myth-forge-3d-eval.json
```

New canonical suite behavior:

```bash
cd services/backend
uv run python -m myth_forge_api.cli evaluate-3d \
  --provider local \
  --suite default-v0 \
  --output /tmp/personal-myth-forge-3d-eval.json
```

With Meshy:

```bash
export MESHY_API_KEY=...
cd services/backend
uv run python -m myth_forge_api.cli evaluate-3d \
  --provider meshy \
  --suite default-v0 \
  --output /tmp/personal-myth-forge-3d-eval-meshy.json
```

If both `--suite` and `--prompts-file` are omitted, the CLI exits with a clear
argument error. If both are provided, the CLI exits with a clear argument error
so every report has exactly one case source.

## Privacy and Safety

The report can include prompts because the built-in suite is non-private. It
must not include provider keys, bearer tokens, raw media data URIs, or local
filesystem paths. Per-prompt errors continue to use the existing sanitizer, but
P0.35 should extend sanitizer coverage to data URI and file URL patterns if
needed.

## Testing Strategy

Backend tests:

- `evaluate-3d --suite default-v0` writes a report with 20 cases.
- report top-level `kind`, suite name, coverage, rubric, and safety fields are
  present.
- per-case rows include category, expected input mode, review focus, variants,
  scene-loadable status, and generation provenance.
- prompt-file mode still works and reports `suite == "custom-prompts"`.
- passing both `--suite` and `--prompts-file` returns a CLI argument error.
- provider setup failures still exit `1` without writing a success report.
- per-case provider errors are sanitized and do not leak secrets, data URIs, or
  local paths.

Regression:

- `make backend-lint`
- `make backend-test`
- SwiftPM checks should remain green because this is backend-only.
- `evaluate-3d --provider local --suite default-v0` should write a 20-case
  local report.
- final acceptance quick should retain its known blocker classification.

Visual evidence:

- static 390x844 evidence page summarizing suite coverage, sample cases, and
  known live-provider gates
- screenshot rendered with installed Chrome through Playwright MCP

## Non-Goals

- No live Meshy run without `MESHY_API_KEY`.
- No automatic provider-credit spending inside final acceptance.
- No visual model scoring automation. Empty human review scoring fields are
  enough for this iteration.
- No media-backed source image evaluation cases yet. The report schema should
  make room for later image/guided-scan cases, but P0.35 only implements the
  canonical text suite.
