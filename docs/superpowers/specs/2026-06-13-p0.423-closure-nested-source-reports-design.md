# P0.423 Closure Nested Source Reports Design

## Goal

Preserve nested source report summaries inside `final_launch_closure_packet.source_reports` so the final closure packet can trace an iOS device blocker from the top-level launch package down through `ios_device_evidence_bundle` to the mobile deploy and Xcode evidence action bundles.

## Current State

`ios_device_evidence_bundle_report` now keeps `mobile_deploy_preflight_evidence` and `mobile_xcode_build_evidence` source summaries, including their `device_action_bundle` payloads. `final_launch_closure_packet_report` includes an `ios_device_evidence_bundle` source summary, but that summary currently keeps only `kind`, `status`, `summary`, and a direct `device_action_bundle`. The nested `source_reports` are dropped at the closure packet boundary.

## Approach

Extend `final_launch_closure_packet._source_summary()` to copy a source report's existing `source_reports` field when it is a dictionary. This mirrors the existing `device_action_bundle` preservation and keeps source-report propagation generic for any future nested report that already exposes safe summaries.

## Safety

This change only copies already-built report summaries into a higher-level ignored/local report. It does not run Xcode, modify signing, write environment files, invoke live providers, or place print orders. The final closure packet still passes through the existing recursive sanitizer before returning or writing JSON.

## Testing

Add a focused contract test that builds a closure packet from synthetic saved mobile evidence reports. The test asserts that `source_reports.ios_device_evidence_bundle.source_reports.mobile_deploy_preflight_evidence.device_action_bundle` and the matching Xcode path survive into the final closure packet, and that synthetic secrets and repo paths are redacted.
