# P0.160 Final Showcase iOS Deploy Digest Design

## Context

After P0.159, `final_showcase_readiness` now shows `capture_scanning` and `local_showcase_smoke` as ready. The first remaining blocker is `ios_deployable`, backed by the iOS deploy runbook and iOS device launch rehearsal. The mobile Final Showcase summary currently shows local smoke, 3D evaluation, NPC evaluation, handoff, and launch, but it does not expose iOS deploy evidence as its own stage.

## Options

1. Add a dedicated `ios_deploy` Final Showcase stage. This is the recommended path because it surfaces the current first blocker directly without changing backend readiness semantics.
2. Fold deploy evidence into the existing `final_launch` stage. This keeps the stage count smaller, but it hides the real source of the blocker behind a generic launch label.
3. Add a separate Device Preflight-only visual update. This would help the device screen, but not the Final Showcase summary where the current readiness digest is already collected.

## Decision

Add a dedicated `ios_deploy` stage to `FinalShowcaseSummaryBuilder` when `finalLaunchSummary` is available. The stage will appear after `local_smoke` and before `three_d_evaluation`, mirroring the actual final readiness order: local proof first, then deploy evidence, then evaluation/launch closeout.

## Behavior

The `ios_deploy` stage reads existing sanitized rows from `FinalLaunchMobileSummary`:

- Prefer `launchRehearsalRows` because final readiness first requires the rehearsal evidence.
- Fall back to `deployRunbookRows` when rehearsal rows are absent.
- Ready when the first chosen row starts with `iOS launch rehearsal ready:` or `iOS deploy runbook ready.`
- Waiting when rows are empty or contain `has not loaded`, `Run iOS device launch rehearsal`, or similar waiting copy.
- Needs attention when rows contain blocked, failed, missing, stale, or partial evidence.

For blocked states, the detail should include the first actionable row and the next row when available, so the mobile summary can show both the blocker and the command hint. All detail text must pass through the existing Final Showcase sanitizer.

## Acceptance

Swift contract tests will cover:

- A ready iOS deploy digest appears as `ios_deploy`, label `iOS Deploy`, status `.ready`, with rehearsal ready detail.
- A blocked digest surfaces `iOS launch rehearsal blocked` and the relevant command/action text.
- Unsafe text in deploy/rehearsal rows is redacted.
- Existing Final Showcase ready stage ordering now includes `local_smoke`, `ios_deploy`, `three_d_evaluation`, `npc_evaluation`, `operator_handoff`, and `final_launch`.

Backend source acceptance will require the new stage function, id, label, and Swift tests. Visual regression will add a 390x844 artifact showing `Final Showcase`, `iOS Deploy`, `iOS deploy blocked`, and `make ios-device-launch-rehearsal`.

## Non-Goals

- Do not start backend servers.
- Do not write `Deployment.local.xcconfig`.
- Do not run Xcode signing or device deployment.
- Do not treat missing real provider keys or missing physical device evidence as solved.
- Do not change `final_showcase_readiness` status semantics.

## Safety

The change is read-only and mobile-summary-only. It consumes already sanitized rows from final launch reports and applies the Final Showcase sanitizer again before display. No provider calls, raw capture media, payment links, local file paths, backend secrets, signing credentials, or global machine state are introduced.
