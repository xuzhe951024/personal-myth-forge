# P0.408 iOS Certificate Device Bundle Design

## Context

The final iPhone showcase remains blocked on deployable-device evidence, not on local backend behavior. Recent iterations added read-only `device_action_bundle` sections to final showcase readiness, the iOS deploy runbook, and the iOS device evidence bundle. `ios-device-launch-certificate.json` is the final physical-iPhone operator certificate, but it still lacks its own top-level action bundle even though it already computes `device_gates`, `first_blocker`, `next_action`, and `operator_actions`.

## Goal

Expose a sanitized `device_action_bundle` directly in the iOS device launch certificate report so the final certificate carries its own ordered device actions, first action, summary, and safety metadata.

## Scope

- Add `device_action_bundle` to `build_ios_device_launch_certificate_report`.
- Derive actions from existing `device_gates`; do not introduce new checks.
- Preserve existing `status`, `first_blocker`, `next_action`, `operator_actions`, and ready-report behavior.
- Add focused blocked and ready tests.
- Update the roadmap with P0.408.

## Non-Goals

- Do not run backend, Xcode, signing, Keychain, provider, iPhone, or paid-print commands.
- Do not write iOS deploy config or backend env files.
- Do not change final showcase readiness classification.
- Do not alter mobile app models in this iteration.
- Do not embed secrets, local absolute paths, raw media, or payment links.

## Design

The report will include a top-level object:

- `id`: `ios_device_launch_certificate_actions`
- `label`: `iOS Device Launch Certificate Actions`
- `status`: the same overall certificate status derived from `device_gates`
- `actions`: one action per existing device gate
- `first_action`: the first required blocked or missing action, matching the current `first_blocker` command
- `summary`: counts for actions, ready, missing, blocked, manual, live, provider calls, global actions, and Xcode/signing actions
- `safety`: read-only flags matching the report's no-mutation behavior

Each action will mirror gate metadata: `id`, `label`, `status`, `classification`, `command`, `detail`, `required`, `requires_consent`, `manual`, `provider_calls: false`, `global_action`, `xcode_or_signing`, `validation_command`, and `next_action` for actionable blocked/missing/manual/live rows. This keeps the certificate self-contained while leaving existing gate selection as the source of truth.
