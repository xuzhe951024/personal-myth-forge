# P0.92 Demo Script NPC Readiness Design

## Context

P0.90 added a local NPC Agent evaluation suite. P0.91 surfaced the saved
evaluation readiness through `/v1/final-demo-launch` and the iPhone Final Launch
Status panel. The remaining gap is the operator demo path: `DemoScriptBuilder`
and `ShowcaseAutopilotPlanner` can still treat the final loop as ready based on
the broad final launch step, without making the AI Agent NPC evaluation a first
class demo checkpoint.

This iteration turns NPC Agent evaluation readiness into an explicit iPhone demo
script step. It does not create a new provider call, run evaluation from the app,
or change the backend readiness contract.

## Options Considered

1. Add a separate `npc_evaluation` step to the iPhone demo script.
   This is the chosen approach because it makes AI Agent readiness visible in the
   operator's step-by-step flow and gives Autopilot a precise gate.

2. Keep only the Final Launch Status panel and change its copy.
   This is too passive. It improves visibility but does not prevent the demo
   script from completing while the saved NPC evaluation is missing or blocked.

3. Add a backend `/v1/npc-evaluation-readiness` endpoint.
   This is unnecessary for the current app because `/v1/final-demo-launch`
   already carries the sanitized readiness report.

## Design

`DemoScriptBuilder` will derive a new step from
`FinalLaunchMobileSummary.npcEvaluationRows` when a final launch summary is
available. The step id is `npc_evaluation`, the label is `NPC Evaluation`, and
the detail is the first sanitized NPC evaluation row. Status mapping:

- `complete` when the first row starts with `NPC Agent evaluation ready:`
- `blocked` when any row contains `blocked`, `failed`, or
  `npc_agent_evaluation_failed`
- `waiting` when rows are missing, not loaded, or instruct the operator to run
  local evaluation

The existing `final_launch` step remains as the broad launch readiness checkpoint.
The new NPC evaluation step is inserted immediately before `final_launch`, so the
demo script reads as capture -> forge -> 3D -> NPC autonomy -> print -> resources
-> NPC evaluation -> final launch.

`ShowcaseAutopilotPlanner` will inspect `script.step(id: "npc_evaluation")`
after print quote and provider resources are ready, and before `final_launch`.
If the step is complete, planning continues. If it is blocked or waiting,
Autopilot returns a non-executable plan with button title `Check NPC Eval`.

## Error Handling and Privacy

No app code runs `evaluate-npc`. The app only consumes sanitized rows already
provided by `FinalLaunchMobileSummary`. The demo script and Autopilot keep their
existing conservative redaction behavior; unsafe text collapses to `[withheld]`.

## Tests

Swift contract tests will be written first:

- ready final launch with ready NPC evaluation adds a complete
  `npc_evaluation` step before `final_launch`
- missing NPC evaluation adds a waiting step and next action asks to load/run NPC
  evaluation
- blocked NPC evaluation adds a blocked step and redacts unsafe detail
- Autopilot blocks on waiting/blocked `npc_evaluation` before final launch
- Autopilot completes only when both `npc_evaluation` and `final_launch` are ready

Source acceptance and project checks will require the new step id, UI label, and
contract tests. A 390x844 static visual regression page will show the Demo Script
with the `NPC Evaluation` checkpoint.

## Non-Goals

- No new backend endpoint
- No OpenAI/Meshy/Treatstock calls
- No Xcode, signing, deployment, keychain, or global configuration changes
- No evaluation execution from iOS
