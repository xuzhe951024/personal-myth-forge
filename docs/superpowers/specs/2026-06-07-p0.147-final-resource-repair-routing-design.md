# P0.147 Final Resource Repair Routing Design

## Problem

P0.146 added a safe `final-resource-repair` workflow, but the main operator paths still primarily surface generic resource actions when stale placeholder values remain in `services/backend/.local/final-resources.env`.

This creates a handoff gap:

- `final-resource-apply-preview` knows a slot is blocked by `placeholder_value`, but does not compose the repair report.
- `final-external-action-ledger` lists safe local writes, but does not show the repair step before apply preview.
- The iOS runbook inherits apply preview status but not the repair-specific action.

## Goal

Route repairable final resource placeholders into the primary final-resource and launch handoff reports so operators see the safe repair command before filling final keys or applying resources.

## Non-Goals

- Do not auto-run repair from preview reports.
- Do not fill API keys, Team IDs, bundle IDs, or backend URLs.
- Do not write backend `.env`, iOS local config, Xcode state, signing state, or provider evidence.
- Do not call live providers or network services.

## Design

`final_resource_apply_preview` will compose the read-only `build_final_resource_repair_report(..., apply=False)` report.

When repair status is `repairable`:

- `source_reports.final_resource_repair` is included.
- `commands` includes `make final-resource-repair-preview` and `make final-resource-repair` before `make final-resource-apply-preview`.
- `operator_actions` begins with `run make final-resource-repair`.
- status remains blocked until the operator runs repair and reruns preview.

`final_external_action_ledger` will read the same repair report and add a safe local action:

- `repair_final_resources`
- command `make final-resource-repair`
- status `blocked` when repairable, `ready` when no repair is needed, `missing` when resources file is missing
- marked as a safe repo-local write, not a backend/iOS config write

`ios_deploy_runbook` already includes apply preview operator actions through the final resource reports; no new command sequence step is required unless tests show the action is not propagated.

## Safety

All repair routing is read-only except the existing explicit `make final-resource-repair` command. Reports keep these invariants:

- `provider_secrets_in_report=False`
- `local_paths_in_report=False`
- no live provider calls
- no global mutation
- no Xcode/signing/keychain changes

## Testing

Tests must prove:

- Apply preview reports repair source status and repair actions when stale placeholders exist.
- Apply preview does not include repair commands/actions when resources only have missing values.
- External action ledger exposes `repair_final_resources` with correct status and safe-local-write flags.
- Final iOS rehearsal/final showcase action promotion can surface `run make final-resource-repair`.

Full backend and Swift regression remain required before merge.
