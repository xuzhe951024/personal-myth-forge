# P0.54 Deploy Health Preflight Design

## Product Intent

P0.53 added an in-app backend `/health` check, but the command-line deploy
preflight still only validates that `PMF_BACKEND_BASE_URL` is present and not
loopback. P0.54 upgrades `make mobile-deploy-preflight` so an operator can catch
a missing backend process, wrong LAN URL, firewall issue, or non-ok backend
health response before opening Xcode or connecting a physical iPhone.

This moves the project closer to real iOS deployment without mutating global
developer settings, accepting the Apple SDK license, signing an app, running a
simulator/device, or calling Meshy/OpenAI/print providers.

## Behavior

`apps/mobile/ios/scripts/deploy_preflight.sh` keeps the existing checks:

- `Deployment.xcconfig` exists.
- ignored `Deployment.local.xcconfig` is not tracked.
- `DEVELOPMENT_TEAM` is present.
- `PRODUCT_BUNDLE_IDENTIFIER` is present.
- `PMF_BACKEND_BASE_URL` is present.
- backend URL is not loopback for a physical iPhone demo.

When those configuration checks pass, the script calls:

```text
GET $PMF_BACKEND_BASE_URL/health
```

The backend health step passes only when the HTTP response body contains
`"status":"ok"` or `"status": "ok"`. It intentionally uses command-line tools
available on macOS by default:

1. `python3` with `urllib.request` when available.
2. `curl` fallback when Python is unavailable.

The health request has a short timeout and prints only safe, bounded text. It
must not print provider secrets, raw media, local file paths, checkout URLs, or
full response bodies.

Exit codes:

- `0`: deploy config and backend health passed.
- `1`: repository/script structural error.
- `2`: local deploy config blocker or backend health blocker.

## Acceptance

Add a backend pytest file that copies the iOS config/script into a temp git repo
and runs `deploy_preflight.sh` with controlled local config:

- missing local deployment values still exits `2`.
- loopback backend URL still exits `2`.
- tracked `Deployment.local.xcconfig` still exits `1`.
- configured LAN backend with a test HTTP server returning `{"status":"ok"}`
  exits `0` and prints `Backend health: ok`.
- configured LAN backend with a test HTTP server returning `{"status":"down"}`
  exits `2` without leaking the response body.

Update project checks to prove the script source contains the health request,
`/health`, timeout, Python/curl fallback, and no global mutation commands.

Update final acceptance test stubs to recognize a backend health failure as
`blocked_by_local_ios_backend_health` when the mobile deploy preflight command
exits `2` with health-related stderr. Existing missing config classification
remains `blocked_by_local_ios_deploy_config`.

## Docs and Evidence

README and mobile README explain that `make mobile-deploy-preflight` now checks
both local iOS deployment config and backend `/health`.

Static visual evidence shows the command path:

- local config values
- non-loopback LAN backend URL
- `/health` check
- final acceptance still blocked by local signing/Xcode license when those
  machine-specific values are absent

## Non-Goals

- No global `xcode-select`, `sudo`, keychain, signing, or license mutation.
- No simulator/device execution.
- No live Meshy/OpenAI/print provider calls.
- No changes to backend `/health`; reuse the existing endpoint.
- No persistent local config generation; P0.54 only validates.
