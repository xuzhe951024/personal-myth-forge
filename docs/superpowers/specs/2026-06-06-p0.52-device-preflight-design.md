# P0.52 Device Preflight Design

## Product Intent

The final iPhone demo still has local deployment risks: the app must point at a
Mac LAN backend URL, provider readiness must load, and the demo loop must be in
a usable state. P0.52 adds a compact device-facing preflight panel inside the
iOS app so an operator can detect those issues before starting the showcase.

This is a source-verified app feature. It does not claim that a real iPhone has
already been installed or run.

## Product Design Brief

- Product: Personal Myth Forge iOS app.
- Screen/component: a new compact panel near the top of `ForgeRootView`, after
  `Final Showcase`.
- Visual source: current SwiftUI `.thinMaterial` panels and dense mobile
  operator UI.
- Interactivity: fully functional SwiftPM-compiled source path.

## Approaches Considered

1. **Pure preflight summary, recommended.**
   Add a pure Swift builder that classifies the backend URL, provider readiness,
   and local demo state. Render it in a small panel. This is deterministic,
   testable, and does not require a real device.

2. **Live health polling.**
   Continuously call `/health` and provider readiness from the panel. Useful
   later, but it adds timing and networking behavior that can distract from the
   current final acceptance gates.

3. **Shell-only preflight.**
   Keep the current `make mobile-deploy-preflight` path only. This helps the
   developer but not the person holding the iPhone.

P0.52 uses approach 1.

## Scope

Add Swift mobile-core types:

- `DevicePreflightStatus`
- `DevicePreflightItem`
- `DevicePreflightSummary`
- `DevicePreflightSummaryBuilder`

Inputs:

- backend base URL
- provider readiness or provider readiness error
- final showcase summary
- restored snapshot tick count

Outputs:

- overall status: `blocked`, `waiting`, or `ready`
- rows for backend URL, provider readiness, local demo state, and saved NPC
  history
- privacy/safety notes

Add `DevicePreflightView` to the iOS app:

- display backend URL classification
- warn when the backend URL is loopback, because iPhone cannot reach
  `127.0.0.1` on the Mac
- show provider readiness state using existing backend readiness data
- show whether current final showcase state is ready for local demo
- show saved NPC tick count as continuity evidence

Add `device_preflight` to `ios_showcase_acceptance` so final acceptance proves
the checked-in mobile source includes the device preflight path.

## Behavior

- `http://127.0.0.1:*`, `http://localhost:*`, and `::1` classify as blocked for
  device use.
- private LAN hosts such as `192.168.*`, `10.*`, and `172.16-31.*` classify as
  ready for device use.
- public or unknown hosts classify as waiting, not blocked.
- provider readiness errors classify as blocked with sanitized detail.
- missing provider readiness classifies as waiting.
- `overallDemoReady` true classifies provider state as ready.
- final showcase `readyForLocalDemo` classifies local demo state as ready.
- saved NPC tick count of at least one is ready; zero is waiting.
- unsafe text is replaced with `[withheld]`.

## Non-Goals

- No live health polling.
- No new backend endpoint.
- No Xcode signing, Apple SDK license, simulator, or device install action.
- No global environment mutation.
- No provider key creation or live provider call.

## Tests

Swift core contract tests must prove:

- loopback backend URL blocks device preflight
- LAN backend URL is device-ready
- provider readiness errors are blocked and redacted
- missing provider readiness waits
- final showcase ready state contributes a ready row
- saved NPC ticks contribute a ready row

Swift project/source checks must prove:

- `DevicePreflightView.swift` exists in app source and Xcode project
- `ForgeRootView` builds `DevicePreflightSummaryBuilder.build`
- `DevicePreflightView` renders backend URL and checklist rows
- SwiftPM app compile still builds

Backend source acceptance must prove:

- `ios_showcase_acceptance` includes `device_preflight`
- final acceptance stubs use `ios_showcase_acceptance` internal
  `passed=13`, `failed=0`
