# P0.21 HEIC Meshy Inputs Design

## Context

P0.19 made `guided_scan` a first-class iOS capture path and allowed JPEG, PNG,
HEIC, and HEIF uploads. P0.20 added provider readiness. The remaining 3D
generation gap is that Meshy Image-to-3D only receives JPEG or PNG source
images, while real iPhone camera and Object Capture workflows commonly produce
HEIC/HEIF photos. Today, a HEIC-only guided scan reaches the backend but Meshy
falls back to text-to-3D because no supported source image is selected.

## Approaches Considered

### Recommended: backend input preparation before provider dispatch

Keep the capture manifest unchanged, read media during
`/v1/myth-sessions/from-capture`, and convert only HEIC/HEIF reference images
into JPEG data URIs for `ThreeDSourceImage`. This preserves the user's original
uploads, keeps provider keys backend-only, and makes Meshy receive an
Image-to-3D-compatible input without changing the mobile upload contract.

### Alternative: transcode at upload time

The upload endpoint could write JPEG derivatives next to the original files.
This would make later generation cheaper, but it changes storage semantics and
requires derivative lifecycle rules before they are needed.

### Alternative: mobile-side JPEG export

The iOS app could convert selected photos before upload. This keeps backend
simpler, but it leaks provider-specific constraints into mobile code and makes
Object Capture import behavior harder to keep consistent across entry points.

## Design

Add a small backend image-preparation module responsible for provider-ready
source images. It accepts capture media bytes and content type, returns a
`PreparedSourceImage` with provider-facing content type and bytes, and preserves
the original capture URI separately. JPEG and PNG pass through unchanged.
HEIC and HEIF decode through Pillow plus `pillow-heif`, convert to RGB JPEG,
and return `image/jpeg` bytes.

`main._capture_generation_sources()` will use that module before constructing
`ThreeDSourceImage`. For HEIC/HEIF captures, the resulting
`ThreeDSourceImage.uri` remains the original `local-capture://...heic` or
`.heif` URI, while `content_type` becomes `image/jpeg` and `data_uri` becomes
`data:image/jpeg;base64,...`. Response payloads still never include data URIs,
raw media bytes, local filesystem paths, or provider secrets.

The Meshy adapter remains simple: it still chooses the first source image whose
content type is in `{"image/jpeg", "image/png"}`. The behavior change is that
valid iOS HEIC/HEIF media becomes eligible before the Meshy adapter sees it.
If decoding fails, the backend should raise a sanitized provider-preparation
error and return a non-secret HTTP error instead of silently falling back to
text-to-3D for a capture that was intended to be image-guided.

## Testing

Backend TDD coverage will include:

- unit tests for JPEG/PNG passthrough and HEIC/HEIF-to-JPEG preparation
- a regression test proving `/v1/myth-sessions/from-capture` sends HEIC/HEIF
  capture media to the injected 3D provider as `image/jpeg` data URIs while
  preserving original capture URIs
- a Meshy regression test proving a HEIC-only guided scan, after preparation,
  uses Image-to-3D instead of text-to-3D
- error-redaction coverage for failed image preparation

Full verification remains the existing P0 gate set:

- backend lint and pytest
- SwiftPM mobile project checks, core contract tests, and app compile check
- Xcode build gate, expected to stop at Apple SDK license on this machine until
  the user accepts it
- `git diff --check HEAD`

## Non-Goals

P0.21 does not perform a live Meshy run, add API keys, change mobile upload
contracts, add storage-backed public media URLs, implement ARKit reconstruction,
or solve Xcode signing/device deployment.
