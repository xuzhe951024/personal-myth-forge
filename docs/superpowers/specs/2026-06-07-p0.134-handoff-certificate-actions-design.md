# P0.134 Handoff Certificate Actions Design

## Context

P0.133 made `final_acceptance_report` publish concrete `operator_actions`.
The iOS device launch rehearsal now starts with actionable final acceptance
steps, but the tail still contains coarse fallbacks:

- `review final_handoff_index: make final-handoff-index`
- `review ios_device_launch_certificate: make ios-device-launch-certificate`

Both source reports already contain structured lane/gate rows and command
sequences. They only need top-level actions so downstream rehearsal routing can
preserve and prefix them.

## Decision

Add bounded top-level `operator_actions` to:

- `final_handoff_index_report`
- `ios_device_launch_certificate_report`

`final_handoff_index_report` derives actions from blocked or missing lanes:

- `local_rehearsal`: `run make final-rehearsal-local`
- `configured_preflight`: `run make final-configured-preflight`
- `configured_launch`: use the lane command.
- `device_deploy`: `run make mobile-deploy-preflight after backend is running`
- `live_acceptance`: `run configured final acceptance after live provider cost review`

`ios_device_launch_certificate_report` derives actions from blocked, missing,
or manual device gates:

- `final_handoff_index`: `run make final-handoff-index`
- `ios_deploy_config`: `provide iOS deploy config and rerun mobile deploy preflight`
- `ios_deploy_runbook`: `run make ios-deploy-runbook-local`
- `final_demo_launch`: `run make final-demo-launch`
- `backend_device_server`: `start backend-device-demo`
- `mobile_deploy_preflight`: `run make mobile-deploy-preflight after backend is running`
- `xcode_build_gate`: `resolve Xcode build gate outside the app`
- `configured_final_acceptance`: `run configured final acceptance after live provider cost review`

Actions are deduped, capped to six items per report, and sanitized by the
existing report sanitizers. Ready reports still include action hints for manual
or live follow-up gates where those gates remain in the report.

## Alternatives Considered

1. Add fallback-specific mapping inside `ios_device_launch_rehearsal.py`.
   This would improve only one consumer and keep the source reports less useful.
2. Add a shared action mapping module. The two reports use different row ids and
   user-facing wording, so a shared helper would obscure rather than simplify.
3. Publish actions in the two source reports. This is the smallest durable fix
   because downstream readers already preserve nested actions.

## Data Flow

`build_final_handoff_index_report` builds lanes, computes status and summary,
then attaches `_operator_actions(lanes)`. `build_ios_device_launch_certificate_report`
builds device gates, computes status and summary, then attaches
`_operator_actions(device_gates)`. `ios_device_launch_rehearsal._source_report`
already preserves bounded `operator_actions`, so the top-level rehearsal report
will replace `review ...` fallbacks with prefixed concrete actions.

## Safety

This iteration only changes local JSON report generation and tests. It does not
run provider calls, Xcode signing, Apple SDK license commands, keychain writes,
or global configuration changes.
