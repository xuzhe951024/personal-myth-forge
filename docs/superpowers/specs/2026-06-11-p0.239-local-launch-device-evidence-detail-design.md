# P0.239 Local Launch Device Evidence Detail Design

## Context

The final local launch report now correctly prioritizes the physical iPhone path over continuable provider-key setup. The current top-level `first_blocker` for `final-demo-launch-local` is `mobile_deploy_preflight`, but its detail only says this is the physical iPhone backend health gate. The actionable evidence already exists in `final_showcase_readiness.device_action_bundle`, where `run_mobile_deploy_preflight` carries the concrete blocked checks from `services/backend/.local/mobile-deploy-preflight-evidence.json`, such as missing `DEVELOPMENT_TEAM` or an iPhone-unreachable backend URL.

## Goal

Make the local final demo launch top-level `first_blocker` and derived `next_action` show the saved mobile deploy preflight evidence detail when the blocker is `mobile_deploy_preflight`.

## Non-Goals

- Do not run Xcode, sign an app, touch Keychain, or mutate global Mac state.
- Do not call Meshy, OpenAI, Treatstock, or any live provider.
- Do not change configured-mode provider blocker behavior.
- Do not write secrets or local absolute paths into committed reports.

## Approaches Considered

1. Reuse `final_showcase_readiness.device_action_bundle`.
   This is the selected approach. `build_final_demo_launch_report` already builds `final_showcase_readiness`, and that report already normalizes and sanitizes mobile preflight evidence. Reusing it keeps the behavior consistent with the iPhone Final Launch panel.

2. Load `mobile-deploy-preflight-evidence.json` directly in `final_demo_launch.py`.
   This would work, but it duplicates evidence parsing and fallback logic that already exists in `final_showcase_readiness.py`.

3. Read detail from `ios_device_evidence_bundle`.
   This is less precise today because the bundle can fall back to broader rehearsal or final acceptance blockers instead of the direct mobile preflight checks.

## Design

`build_final_demo_launch_report` will pass the already-built `final_showcase_readiness` report into the first-blocker selection path. When local mode selects a phase blocker with id `mobile_deploy_preflight`, the final demo launch builder will look for the `run_mobile_deploy_preflight` action in `final_showcase_readiness.device_action_bundle.actions`. If that action has `evidence_detail`, the phase blocker detail will append it after the existing phase detail.

The enrichment must be local-mode only and mobile-preflight only. It must not append provider-key evidence such as `MESHY_API_KEY` to the local device blocker. The same enriched detail should flow into `next_action` because `next_action` is copied from `first_blocker`.

## Testing

Add a focused failing backend test in `services/backend/tests/test_final_demo_launch.py`:

- Arrange a temp repo with missing `DEVELOPMENT_TEAM` and iPhone-unreachable `PMF_BACKEND_BASE_URL`.
- Write a saved `mobile-deploy-preflight-evidence.json` with those two blocked checks.
- Build `final-demo-launch` in local mode.
- Assert the first blocker remains `mobile_deploy_preflight`.
- Assert the blocker detail and next action detail include both concrete mobile evidence strings.
- Assert the detail does not include `MESHY_API_KEY`.

Focused verification:

```bash
cd services/backend
uv run pytest tests/test_final_demo_launch.py::test_local_final_demo_launch_mobile_preflight_blocker_includes_saved_evidence_detail
uv run pytest tests/test_final_demo_launch.py
```

Final verification:

```bash

git diff --check
make final-local-report-refresh-local
```

`make final-local-report-refresh-local` is expected to exit blocked in the current no-key/no-device environment; the evidence to read is that it refreshes saved reports and keeps the accepted local blockers explicit.
