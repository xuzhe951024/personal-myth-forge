# P0.32 Final Acceptance Report Design

## Context

P0.31 added a backend `demo-acceptance` command that proves the core
capture-to-session-to-3D-to-NPC-ticks path can run with local providers, and can
later run configured Meshy/OpenAI providers once API keys are supplied.

The remaining operator-facing gap is that final demo acceptance is now spread
across several commands:

- backend provider handoff
- backend demo acceptance
- backend lint and tests
- Swift package project and contract checks
- iOS deploy preflight
- Xcode build gate

Some of these commands are expected to fail on this machine until local signing,
LAN backend URL, provider keys, or Apple SDK license state are supplied. The next
iteration should make those states visible in one sanitized report instead of
requiring the operator to remember which failures are expected gates.

## Goal

Add a single CLI command that runs the final no-global-mutation acceptance
orchestrator, writes a sanitized JSON report, and classifies each check as
passed, failed, or blocked by local/provider configuration.

## Non-Goals

- No API key creation, storage, or printing.
- No global environment changes.
- No `sudo`, `xcode-select`, Apple license acceptance, signing setup, simulator
  boot, or device install automation.
- No live Meshy/OpenAI calls by default.
- No print fulfillment order placement.
- No raw media, raw personal source text, local file paths, bearer tokens, or API
  key values in the report.
- No background dev server lifecycle management.

## Recommended Approach

Add a backend CLI command:

```bash
cd services/backend
uv run python -m myth_forge_api.cli final-acceptance \
  --profile quick \
  --provider-mode local \
  --output /tmp/personal-myth-forge-final-acceptance.json
```

`quick` is the default profile for handoff and local acceptance. It should run:

1. provider handoff in-process
2. demo acceptance in-process
3. `make mobile-deploy-preflight`
4. `make mobile-xcode-build`

`full` adds regression-grade checks:

1. `make backend-lint`
2. `make backend-test`
3. `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks`
4. `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests`
5. `swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck`

The command runs from the backend package but accepts `--repo-root`, defaulting
to the repository root derived from the installed source path. Backend acceptance
logic stays in Python for fast tests; shell commands are isolated behind a small
runner abstraction.

## Provider Modes

`--provider-mode local` forces local 3D and NPC providers in memory by reusing
the P0.31 `demo-acceptance` behavior. It does not mutate `.env` or process-wide
environment.

`--provider-mode configured --require-real-core` is the final provider-key handoff
mode. If core providers are not real-provider-ready, the provider and demo checks
are classified as `blocked_by_provider_configuration`, and no Meshy/OpenAI calls
are made.

## Check Classification

Each check has:

- `id`
- `label`
- `kind`: `in_process` or `command`
- `status`: `passed`, `failed`, `blocked`, or `skipped`
- `classification`: a stable machine-readable reason
- `exit_code`
- `elapsed_seconds`
- optional sanitized `stdout_tail`, `stderr_tail`, and `error`

Classification rules:

- provider handoff exit `2` with strict real core missing is
  `blocked_by_provider_configuration`
- demo acceptance exit `2` with strict real core missing is
  `blocked_by_provider_configuration`
- mobile deploy preflight exit `2` is `blocked_by_local_ios_deploy_config`
- Xcode build gate output containing Apple SDK license failure or exit `69` is
  `blocked_by_apple_sdk_license`
- Xcode missing at the expected path is `blocked_by_xcode_installation`
- any other non-zero command result is `command_failed`
- an unexpected in-process exception is `runner_failed`

Overall status:

- `passed` when all required checks pass
- `blocked` when at least one check is blocked and no check failed
- `failed` when any check failed

The CLI should return:

- `0` for overall `passed`
- `2` for overall `blocked`
- `1` for overall `failed`

## Report Shape

The report is JSON:

```json
{
  "kind": "final_acceptance_report",
  "profile": "quick",
  "provider_mode": "local",
  "require_real_core": false,
  "overall_status": "blocked",
  "summary": {
    "passed": 2,
    "blocked": 2,
    "failed": 0,
    "skipped": 0
  },
  "checks": [
    {
      "id": "provider_handoff",
      "label": "Provider handoff readiness",
      "kind": "in_process",
      "status": "passed",
      "classification": "ready",
      "exit_code": 0
    },
    {
      "id": "mobile_deploy_preflight",
      "label": "iOS deploy preflight",
      "kind": "command",
      "status": "blocked",
      "classification": "blocked_by_local_ios_deploy_config",
      "exit_code": 2
    }
  ],
  "timings": {
    "total_elapsed_seconds": 12.3456
  },
  "safety": {
    "global_mutation": false,
    "live_provider_calls_by_default": false,
    "mobile_secret_policy": "Provider secrets stay on the backend.",
    "raw_media_in_report": false,
    "raw_personal_source_in_report": false
  }
}
```

The report may include compact nested provider/demo reports under the relevant
checks, but they must be sanitized through the same redaction rules used by
P0.31.

## Components

`services/backend/src/myth_forge_api/final_acceptance.py`

- Owns the orchestrator, check definitions, command runner abstraction,
  classification rules, summary, exit-code mapping, and report sanitization.
- Calls existing provider handoff and demo acceptance functions in process.
- Runs shell gates with bounded output capture and sanitized tails.

`services/backend/src/myth_forge_api/cli.py`

- Adds `final-acceptance`.
- Accepts `--profile quick|full`, `--provider-mode local|configured`,
  `--require-real-core`, `--repo-root`, and `--output`.
- Writes JSON to `--output` or stdout and returns the orchestrator exit code.

Docs and evidence:

- README describes the one-command final acceptance flow.
- Roadmap marks P0.32.
- Verification evidence records quick/full behavior and known local blockers.

## Testing

Backend tests will cover:

- quick profile returns blocked, not failed, when mobile deploy and Xcode gates
  hit their known local blockers
- full profile includes backend and Swift regression checks
- provider strict mode classifies missing real providers as
  `blocked_by_provider_configuration` without running live providers
- command stdout/stderr and nested reports redact API keys, bearer tokens, raw
  payloads, data URIs, and local file paths
- CLI writes the final report and returns the mapped exit code
- invalid profiles or provider modes are rejected by argparse

## Acceptance Criteria

- `final-acceptance --profile quick --provider-mode local` writes one report that
  includes provider handoff, demo acceptance, iOS deploy preflight, and Xcode
  build gate results.
- Known local environment blockers are classified as blocked with exit code `2`,
  not as product failures.
- `final-acceptance --profile full --provider-mode local` also runs backend and
  Swift regression checks.
- `final-acceptance --provider-mode configured --require-real-core` exits `2`
  before provider calls until Meshy/OpenAI keys are present.
- Reports are sanitized and contain no provider secrets, raw media, local paths,
  or raw personal source content.
