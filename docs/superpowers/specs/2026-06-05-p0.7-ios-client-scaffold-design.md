# P0.7 iOS Client Scaffold Design

Date: 2026-06-05

## Context

P0.6 added a hardened mobile capture contract:

- `POST /v1/object-captures`
- `GET /v1/object-captures/{capture_id}`
- `POST /v1/myth-sessions/from-capture`

The next product step is a real iOS-first client foundation that can exercise this
contract and become the deployable mobile surface. Unity is still a good later
candidate for the village scene and GLB runtime, but the current local environment
does not have a usable Unity CLI. The Swift toolchain is available, while
`xcodebuild` currently points at Command Line Tools instead of a full Xcode
developer directory. Switching `xcode-select` would be a global machine change, so
P0.7 must not depend on that.

## Goal

Build a testable iOS client scaffold that proves the mobile capture-to-myth API
loop in Swift without requiring global Xcode configuration changes.

## Non-Goals

- No real App Store archive or device deployment in P0.7.
- No Unity project generation.
- No camera permission flow automation.
- No real GLB/USDZ rendering of remote provider assets yet.
- No provider keys embedded in the app.

## Recommended Approach

Create `apps/mobile/ios/` as a Swift Package with a pure Swift core library and
tests. Keep SwiftUI app source in the same directory as compile-ready source files
for a future Xcode target, but do not require `xcodebuild` for P0.7 verification.

This gives us a real typed client contract now, avoids global toolchain changes,
and keeps the future iOS app path straightforward when full Xcode is selected.

## Architecture

### Mobile Core Library

Package: `apps/mobile/ios/Package.swift`

Module: `PersonalMythForgeMobileCore`

Responsibilities:

- Codable models matching the P0.6 backend contract.
- `PersonalMythForgeAPIClient` for direct session, capture upload, capture lookup,
  and from-capture session creation.
- `MultipartFormDataBuilder` that produces deterministic multipart bodies for
  tests and native `URLRequest` upload.
- `ForgeFlowReducer` that models the client state machine independent of SwiftUI.

The core module must not import SwiftUI, UIKit, ARKit, RealityKit, or SceneKit.
That keeps it testable with `swift test` under Command Line Tools.

### SwiftUI App Source

Directory: `apps/mobile/ios/App/`

Source-only scaffold:

- `PersonalMythForgeApp.swift`
- `ForgeRootView.swift`
- `CaptureFormView.swift`
- `ArtifactSummaryView.swift`
- `NPCReactionsView.swift`

These files document the intended iOS surface and can be copied or attached to an
Xcode iOS app target later. They should use the core reducer and models, but P0.7
does not require compiling them until full Xcode is available.

### Fixtures

Directory: `apps/mobile/ios/Tests/PersonalMythForgeMobileCoreTests/Fixtures/`

Fixtures mirror backend examples:

- `object-capture-metadata.json`
- `object-capture-response.json`
- `myth-session-response.json`

They should stay small and deterministic.

## Data Flow

1. User chooses a local image in the future iOS UI.
2. Client builds `ObjectCaptureMetadata`.
3. Client uploads metadata plus binary media to `POST /v1/object-captures`.
4. Backend returns `ObjectCapture`.
5. Client submits `capture_id` plus `ContextCapsule` to
   `POST /v1/myth-sessions/from-capture`.
6. Client renders:
   - capture status
   - generated asset provider/URI
   - world state deltas
   - visible changes
   - NPC reactions and accepted/rejected actions
   - print candidate review text

## API Client Behavior

Base URL is injectable and defaults only in examples, never in code that would ship
with production settings.

Required methods:

```swift
public func uploadObjectCapture(
    metadata: ObjectCaptureMetadata,
    media: [CaptureUpload]
) async throws -> ObjectCapture

public func getObjectCapture(captureId: String) async throws -> ObjectCapture

public func createMythSessionFromCapture(
    captureId: String,
    context: ContextCapsule
) async throws -> MythSession

public func createDirectMythSession(
    object: ObjectObservation,
    context: ContextCapsule
) async throws -> MythSession
```

`CaptureUpload` includes generated client-side filename, content type, and data.
The client must validate capture ids with `^cap_[0-9a-f]{16}$` before lookup or
from-capture requests.

## State Machine

`ForgeFlowState` should include:

- `idle`
- `editingObject`
- `uploadingCapture`
- `creatingSession`
- `ready(MythSession)`
- `failed(ForgeFlowError)`

`ForgeFlowReducer` handles deterministic events:

- `setObjectMetadata`
- `setContextCapsule`
- `beginUpload`
- `captureUploaded(ObjectCapture)`
- `sessionCreated(MythSession)`
- `requestFailed(ForgeFlowError)`
- `reset`

The reducer should be pure and unit-tested. Networking remains in
`PersonalMythForgeAPIClient`.

## Error Handling

Map backend and transport failures into `ForgeFlowError`:

- invalid base URL
- invalid capture id
- encoding failure
- HTTP status with sanitized response body
- transport error
- decoding error

Do not log request bodies containing personal context or raw image bytes in tests
or app code.

## Testing

P0.7 verification uses only local, non-global commands:

```bash
swift test --package-path apps/mobile/ios
make backend-lint
make backend-test
```

Core tests:

- Codable decoding for `ObjectCapture` and `MythSession`.
- Multipart request contains `metadata_json`, one media part, correct content type,
  and no absolute local paths.
- API client builds correct routes and HTTP methods using a custom `URLProtocol`
  test transport.
- Invalid capture ids fail before network requests.
- Reducer transitions through upload, session creation, ready, failure, and reset.

## Visual Verification

Because simulator/Xcode build is not available without a global developer directory
change, P0.7 visual regression should be documentation-backed:

- Add a static mobile flow storyboard under
  `docs/superpowers/verification/assets/p0.7-ios-flow-390x844.png`.
- The storyboard must show the first-screen shape of the future SwiftUI app:
  capture form, upload/session status, artifact summary, NPC reaction list, and
  print review area.
- This is not a substitute for simulator screenshots. A later Xcode-enabled
  iteration must add real iOS screenshot tests or simulator evidence.

## Security And Privacy

- No API keys in app source, fixtures, screenshots, or committed config.
- Base URL is injectable through local config examples.
- Fixtures do not contain real personal data.
- Client tests assert multipart bodies do not include local absolute paths.
- Personal context remains user-supplied summary capsule only.

## Acceptance Criteria

- `apps/mobile/ios` contains a SwiftPM package that passes `swift test`.
- Core Swift models can decode backend capture and myth session fixtures.
- Swift API client can build and execute capture upload and from-capture requests
  against mocked responses.
- Invalid capture ids are rejected client-side.
- SwiftUI app scaffold files describe the intended first mobile screen and consume
  core models/reducer types.
- Mobile README documents the P0.7 local verification path and the Xcode limitation.
- Backend lint/test still pass after adding mobile artifacts.
