# P0.157 Final Launch Closure Packet Design

## Context

The final demo launch payload now exposes resource requirements, the fill guide,
the external action ledger, configured evidence planning, showcase readiness, and
iOS device evidence. These reports are accurate but still require the operator to
mentally assemble the last acceptance path across several sections.

The next useful iteration is not another provider call or Xcode/device action. It
is a read-only closure packet that answers one final-acceptance question:

What exactly remains before the final iPhone demo can be accepted, in which order,
and which steps require secrets, cost consent, safe local writes, or global
machine/Xcode action?

## Decision

Add a read-only `final_launch_closure_packet_report` and include it in
`final_demo_launch`. The report composes existing evidence instead of introducing a
new source of truth:

- `final_resource_fill_guide`: what the user must fill at the final resource file.
- `final_external_action_ledger`: external, live-cost, safe-write, and global
  actions.
- `ios_device_evidence_bundle`: backend LAN, deploy preflight, Xcode gate, and
  device rehearsal evidence.
- `final_showcase_readiness`: product capability readiness and first blocker.
- `final_acceptance_readiness`: current local acceptance blocker state.

The packet is ordered as a final-acceptance checklist:

1. `resource_inputs`: fill required backend/iOS/provider resources.
2. `safe_local_writes`: preview/apply ignored local resource files.
3. `device_evidence`: refresh backend LAN, deploy preflight, Xcode, and rehearsal
   proof.
4. `live_provider_consent`: run Meshy/OpenAI/print evidence only after cost
   consent.
5. `final_acceptance`: rerun local/configured final acceptance and showcase
   readiness after the preceding sections are ready or explicitly approved.

## Backend Shape

Create `services/backend/src/myth_forge_api/final_launch_closure_packet.py` with:

- `build_final_launch_closure_packet_report(settings=None, repo_root=None)`.
- `FinalLaunchClosurePacketResult(exit_code, report)`.
- `report.kind = "final_launch_closure_packet_report"`.
- `report.status` is:
  - `blocked` when required inputs, safe local resources, device evidence, or
    final acceptance/showcase readiness are blocked or missing.
  - `partial` when only manual/global or live-cost sections remain.
  - `ready` when every required section is ready.
- `report.summary` counts sections, actions, ready/missing/blocked/manual/live,
  required inputs, secret actions, cost-consent actions, global actions, and safe
  local writes.
- `report.sections` is the bounded ordered list above.
- `report.sections_by_id` maps section id to section.
- `report.operator_actions` dedupes the next user-visible actions across the
  section order.
- `report.commands` includes the read/report command order, not commands executed
  by the builder.
- `report.safety` states the builder ran no commands, wrote no env/config files,
  called no providers, and only describes global/Xcode/signing actions.

Each section has:

- `id`, `label`, `status`, `command`, `detail`
- `required`
- `actions`
- `first_action`
- `blocked_by`
- booleans for `requires_user_input`, `requires_user_confirmation`,
  `requires_cost_consent`, `global_action`, `xcode_or_signing`,
  `live_provider_call`, and `safe_local_write`

Each action is a compact sanitized row derived from the source reports. It keeps
`id`, `label`, `status`, `command`, `detail`, `required`, `secret`, and the same
consent/safety booleans.

## CLI And Make Target

Add:

```bash
cd services/backend && uv run python -m myth_forge_api.cli final-launch-closure-packet --repo-root ../.. --output .local/final-launch-closure-packet.json
make final-launch-closure-packet
```

The target writes only the JSON report file under ignored `.local`. It does not
apply resources or run acceptance.

## Final Launch Integration

`build_final_demo_launch_report()` should embed the report as
`final_launch_closure_packet`. This makes the final mobile panel a single API
handoff for:

- what to fill
- what is safe to write locally
- what must be run on device/Xcode
- what needs live provider cost consent
- what final acceptance command closes the loop

## iOS Mobile Summary

Add Swift Codable models for the closure packet and a new
`FinalLaunchMobileSummary.closurePacketRows` section. The rows should be compact:

- status/count summary
- first blocked/manual/live section row
- first actionable command
- consent/safety summary

`FinalLaunchStatusView` should render the rows under `Closure Packet`, before the
lower-level external action and resource sections.

## Visual Regression

Add a static 390x844 visual artifact showing `Final Launch Status` with the new
`Closure Packet` section. Required text:

- `Closure Packet`
- `Final closure blocked`
- `resource_inputs`
- `provide MESHY_API_KEY`
- `cost consent`
- `commands_run=false global=false live_calls=false`

## Safety

This iteration is read-only. It must not:

- start the backend server
- run mobile deploy preflight
- run Xcode build gates
- write backend `.env`
- write iOS deploy config
- apply final resources
- call Meshy/OpenAI/print providers
- mutate keychain, signing, Apple SDK license, or device trust state
- include provider secrets, raw media, private context, absolute local paths, LAN
  URLs, or payment links in reports or visual artifacts

## Acceptance

The iteration is complete when:

- backend tests prove the closure packet status, ordering, sanitization, CLI, and
  final demo launch integration
- Swift contract tests decode the embedded packet and produce mobile rows
- source acceptance requires the backend builder, CLI/Make target, Swift models,
  summary rows, UI section, and contract tests
- visual regression includes the new artifact and all default artifacts pass
- backend lint/tests, Swift contract/project checks/build, and diff whitespace
  checks pass
