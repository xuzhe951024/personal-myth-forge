# P0.460 Local iOS Runbook Device-First Design

## Context

The refreshed local readiness evidence shows `final-showcase-readiness` points at the iOS deploy blocker, but `ios-deploy-runbook-local.json` still reports `MESHY_API_KEY` as its first blocker. That makes the local iPhone deployment lane appear blocked by live provider resources.

For the iOS-first demo, local device deployment must be independently actionable with local providers. Meshy/OpenAI/Treatstock resources still matter for final configured/live evidence, but they should not be the first gate for `mode=local`.

## Decision

Make `ios_deploy_runbook` device-first in `mode="local"`:

- `final_resources_env`, `final_resource_apply_preview`, and `apply_final_resources` remain visible in the runbook.
- In local mode, final resource slots are advisory, not required gates.
- In local mode, the resource command steps are `partial` when resources are missing or blocked, so they do not become the first blocker.
- In configured mode, resource slots and command steps remain blocking.
- The first local blocker should be the iOS deploy config/backend health path, such as `development_team` or `backend_base_url`.

## Alternatives Considered

1. Keep local runbook blocked by provider keys.
   This preserves strict final configured readiness, but prevents a local iPhone demo from advancing without Meshy/OpenAI keys.

2. Remove resource steps from the local runbook.
   This hides important final handoff context and makes local/configured lanes harder to compare.

3. Keep resource steps visible but advisory in local mode.
   This keeps handoff context while letting the device deployment lane focus on device-specific blockers. This is the chosen approach.

## Acceptance Criteria

- A local runbook with an existing incomplete `final-resources.env` reports `development_team` as the first blocker instead of `final_resources_env`.
- The local resource slots still appear and still expose provider/resource operator actions.
- Configured mode still reports resource blockers first when required provider resources are missing.
- Existing device action bundle behavior remains intact.

## Safety

This change only changes read-only report classification and operator ordering. It does not write env files, deploy config, Xcode state, Keychain, provider resources, or payment/order state.
