# P0.287 Unblock Action Normalization Design

## Goal

Replace vague `unblock ...` operator actions in final readiness reports with concrete commands that the final iPhone demo operator can actually run or validate.

## Context

The final showcase remains intentionally blocked on external resources and device proof: provider keys, iOS deploy values, backend reachability from iPhone, Xcode/license state, and physical device evidence. Local code can still improve the handoff by making every saved next action executable and validation-aware.

Current saved reports still expose vague actions such as:

```text
unblock final_resource_fill_guide after MESHY_API_KEY
unblock provider_handoff before configured evidence bundle
```

These are weaker than the existing concrete actions already present elsewhere:

```text
provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight
make provider-handoff
```

The vague variants add noise to `operator_actions` and make the final operator sequence harder to follow.

## Design

Keep the fix centralized in `myth_forge_api.operator_actions.normalize_operator_action()`.

Add a small mapping for known unblock phrases:

- `unblock final_resource_fill_guide after MESHY_API_KEY` becomes `provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight`
- `unblock provider_handoff before configured evidence bundle` becomes `make provider-handoff`

The mapping should preserve existing prefix and detail handling. For example:

```text
final_showcase_readiness: unblock provider_handoff before configured evidence bundle
```

should become:

```text
final_showcase_readiness: make provider-handoff
```

Existing dedupe logic will then collapse these actions when the concrete command is already present.

## Scope

In scope:

- shared operator action normalization for the two known vague unblock phrases
- focused tests for direct normalization
- focused report aggregation tests for final local and final showcase actions
- roadmap entry for P0.287

Out of scope:

- changing resource requirements, provider key files, or ignored real configuration
- running live Meshy/OpenAI/Treatstock calls
- starting backend-device-demo
- Xcode signing, Keychain, or global machine state
- normalizing every possible future English phrase

## Verification

The iteration is valid when:

- focused RED tests prove the vague unblock phrases currently leak through
- focused GREEN tests prove they normalize to concrete commands
- refreshed `.local` readiness no longer exposes those vague phrases in top-level operator actions
- backend lint and backend tests pass
