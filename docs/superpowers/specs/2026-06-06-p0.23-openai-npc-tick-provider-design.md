# P0.23 OpenAI NPC Tick Provider Design

## Context

P0.22 added a stateless `POST /v1/npc-ticks` endpoint and a mobile
`Advance Village` panel, but the endpoint always uses `LocalNPCTickRuntime`.
The project already has an OpenAI structured-output path for the initial NPC
director behind `NPC_PROVIDER=openai`, plus provider readiness reporting for
`OPENAI_API_KEY`. P0.23 closes the gap between initial NPC interpretation and
follow-up NPC actions by letting the same provider setting drive the tick loop.

## Goal

Add an OpenAI-backed NPC tick runtime that can generate the next NPC actions
through structured output when `NPC_PROVIDER=openai`, while preserving the local
deterministic runtime for no-key demos and tests.

## Approaches Considered

### Recommended: Reuse `NPC_PROVIDER` For Both Initial Reactions And Ticks

`build_npc_tick_runtime(settings)` returns `LocalNPCTickRuntime` for
`NPC_PROVIDER=local` and `OpenAINPCTickRuntime` for `NPC_PROVIDER=openai`.
This keeps one NPC provider switch, one OpenAI key, and one readiness surface.
It also matches the product direction: NPCs are a subsystem, not two separate
provider categories.

Tradeoff: teams cannot choose OpenAI for initial reactions while forcing local
ticks without changing `NPC_PROVIDER`. That is acceptable for the demo because
mixed-provider NPC behavior would be harder to reason about during key handoff.

### Alternative: Add `NPC_TICK_PROVIDER`

This would provide more granular control, but it introduces another key handoff
surface and readiness state before the demo needs it. It also creates ambiguous
states such as OpenAI initial reactions with local follow-up ticks.

### Alternative: Fold Ticks Into `OpenAINPCDirector`

This avoids a new provider class, but the director and tick runtime have
different inputs and outputs. Keeping them separate gives each file a clearer
contract while sharing validation helpers from `providers/npc.py`.

## Backend Design

Add `OpenAINPCTickOutput` and `OpenAINPCTickRuntime` to
`services/backend/src/myth_forge_api/providers/npc_ticks.py`.

`OpenAINPCTickRuntime` accepts:

- `api_key`
- `model`
- optional `api_base_url`
- optional injected OpenAI-compatible client for tests
- optional `WorldArbitrator`

`generate_tick(request)` will:

1. Validate that `OPENAI_API_KEY` is present before any client call.
2. Call `client.responses.parse(...)` with `text_format=OpenAINPCTickOutput`.
3. Require exactly three NPC ids: `mara`, `ior`, and `senn`.
4. Require exactly three agent traces and three reactions.
5. Validate/synthesize traces so each proposed action is also present in that
   NPC reaction plan.
6. Run the returned reactions through the existing `LocalWorldArbitrator`.
7. Return `NPCAgentTick` with `agent_runtime="openai_tick_structured_runtime"`.

The prompt will include only approved session fields and a sanitized recent
event summary. It must not include raw `recent_events`, data URIs, local capture
paths, bearer tokens, or provider keys.

## Factory And API Design

Add `build_npc_tick_runtime(settings)` to
`services/backend/src/myth_forge_api/providers/factory.py`.

`POST /v1/npc-ticks` will call:

```python
build_npc_tick_runtime().generate_tick(request)
```

It will catch `OpenAINPCProviderError` and `ValueError`, then return sanitized
HTTP errors through the existing `_safe_provider_error` path.

No new environment variables are introduced. Existing variables apply:

- `NPC_PROVIDER=local | openai`
- `OPENAI_API_KEY`
- `OPENAI_NPC_MODEL`
- `OPENAI_API_BASE_URL`

## Readiness Design

The provider readiness response keeps one `kind="npc"` row. Its OpenAI notes and
capabilities will explicitly say that the configured provider covers both:

- initial structured NPC reactions
- stateless structured NPC ticks
- world arbitration handoff

The response still never emits secret values.

## Mobile Design

No mobile API shape changes are required. `NPCTickView` already displays the
runtime string returned by the backend. When the backend uses OpenAI, the same
panel will show `openai_tick_structured_runtime`.

The project checks will require that the mobile contract remains free of
provider keys and that tick rendering still exposes `agentRuntime`.

## Privacy And Safety

- OpenAI tick prompts use approved `MythSession` fields and aggregate recent
  events by count.
- Raw `recent_events` text is not sent to the provider in P0.23.
- Provider exceptions are sanitized before HTTP return.
- OpenAI-produced actions are not directly applied. The existing world
  arbitrator still accepts/rejects every proposed action.
- Print-order and purchase-like actions remain blocked by arbitration.

## Tests

Backend TDD coverage:

- `OpenAINPCTickRuntime` refuses to run without `OPENAI_API_KEY`.
- It parses a structured OpenAI tick response into `NPCAgentTick`.
- It synthesizes traces when missing.
- It replaces trace proposed actions that are not in the reaction plan.
- It rejects wrong NPC ids.
- It sanitizes provider errors.
- It does not send raw recent event payloads to the OpenAI prompt.
- Factory returns the OpenAI tick runtime when `NPC_PROVIDER=openai`.
- `/v1/npc-ticks` routes through the factory and sanitizes OpenAI errors.
- Readiness reports OpenAI NPC capabilities that include structured ticks.

Regression coverage:

- Existing backend full test suite.
- Existing Swift project checks, core contract tests, and app compile check.
- `make mobile-xcode-build` remains gated by local Apple SDK license state.

## Non-Goals

- No live OpenAI API call in automated tests.
- No server-side tick persistence.
- No background autonomous loop.
- No Unity/SceneKit NPC movement execution.
- No new API key creation or secret storage.
- No iOS-side provider key handling.
