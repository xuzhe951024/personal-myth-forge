# P0.103 Generation Result Receipt Design

## Product Intent

After the user taps Forge, the iPhone demo needs a compact proof that one myth session produced the required demo artifacts: a game 3D asset, an iOS scene-loadable asset, a printable derivative candidate, and AI Agent NPC trace output. Existing preview and handoff panels show parts of this, but there is no single receipt that a demo operator can scan to verify the forge result is complete.

## Scope

P0.103 adds a post-forge `3D Generation Result` receipt to the mobile app. It is derived entirely from the already returned `MythSession`; it does not call Meshy, OpenAI, print providers, or Xcode. It must support local deterministic sessions and configured-provider sessions equally.

The receipt has three states:

- `waiting`: no session has been forged yet.
- `ready`: the session has a game asset, an iOS scene-loadable variant, a print candidate, and at least one NPC Agent trace.
- `needsAttention`: a session exists but one or more required result pieces are missing or unsafe to display.

## Data Model

Add `GenerationResultReceipt` in mobile core with rows:

- `game_asset`: provider, format, and source route for the generated game asset.
- `ios_scene`: scene-loadable iOS variant status.
- `print_candidate`: printable derivative format and approval requirement.
- `npc_agent`: runtime plus trace count.

The builder sanitizes provider text, URIs, prompts, approval reasons, and NPC runtime text. Receipt display uses short labels and bounded detail strings so it fits in the existing mobile form.

## UI Placement

Render `GenerationResultReceiptView` in `ArtifactSummaryView` after the existing `Artifact3DPreviewView` and before the provenance card. This keeps the receipt close to the generated artifact without blocking the pre-forge capture form.

## Acceptance

Swift contract tests prove waiting, ready, missing iOS scene variant, and redaction behavior. Swift ProjectChecks and backend iOS showcase acceptance assert the new core builder, view, wiring, Xcode membership, and contract tests. A static 390x844 visual regression shows waiting, ready, and needs-attention receipts without clipping or overlapping text.

## Non-Goals

P0.103 does not implement new provider calls, evaluate Meshy quality, convert GLB to USDZ/DAE, place print orders, persist new backend fields, or resolve local Apple SDK/iOS deploy blockers.
