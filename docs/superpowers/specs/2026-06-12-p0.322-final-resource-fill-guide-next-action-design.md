# P0.322 Final Resource Fill Guide Next Action Design

## Context

`final-resource-fill-guide.json` already identifies a `first_blocker`, but it leaves top-level `next_action` absent. Other final handoff reports expose both fields, which lets the goal loop and iPhone Final Launch summary show an immediately executable action plus the validation command.

The current local report proves the gap: the first blocker is `MESHY_API_KEY`, while `next_action` is `null`.

## Decision

Add `next_action` to the final resource fill guide report:

- when `first_blocker` exists, `next_action` is a copy of it with `source: "first_blocker"`
- when the guide is ready, `next_action` is `null`
- the action keeps `command`, `detail`, `input_source`, `write_destination`, and `validation_command`

This is a read-only report contract change. It must not write `.env`, iOS deploy config, Keychain, global state, or call Meshy/OpenAI/Treatstock.

## iOS Handoff

`FinalResourceFillGuideReport` will decode optional `next_action` using the existing first-blocker shape with an optional `source` field. `FinalLaunchMobileSummary` will show a `Next action:` row before the first blocker row so the iPhone launch panel mirrors the Mac-side operator handoff.

All displayed text remains sanitized through the existing summary sanitizer.

## Acceptance

- Backend tests prove blocked fill guide reports include `next_action` with `source: "first_blocker"`.
- Backend tests prove ready fill guide reports have `next_action: null`.
- Swift contract tests prove the final launch payload decodes `final_resource_fill_guide.next_action`.
- Swift summary tests prove the resource fill guide rows show the next-action command and validation command.
- `make final-resource-fill-guide` or the CLI equivalent still exits `2` in the current missing-resource state and performs no writes except the ignored report output.
