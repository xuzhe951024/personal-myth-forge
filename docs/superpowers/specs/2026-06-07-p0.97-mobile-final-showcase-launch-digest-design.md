# P0.97 Mobile Final Showcase Launch Digest Design

## Context

The iPhone app now has a detailed Final Launch panel, a Demo Script step for
Final Launch, NPC Agent evaluation readiness rows, an iOS deploy runbook, and a
resource handoff overview. The top `Final Showcase` card, however, still only
summarizes the local demo loop:

- capture
- 3D asset
- NPC Agent
- print
- resources

That means the first phone-screen answer can say the local loop is ready while
the final launch report below is still waiting, blocked, stale, or missing the
NPC evaluation handoff. For physical-device demo review, the top card should
answer both questions:

- Is the live mobile myth loop ready?
- Is the final launch handoff ready enough to accept?

## Design

Extend `FinalShowcaseSummaryBuilder.build(...)` with an optional
`finalLaunchSummary: FinalLaunchMobileSummary?`.

When absent, behavior stays unchanged so lightweight local preview tests and
call sites keep the existing local-demo semantics. When present, append three
launch-facing stages after `resources`:

- `npc_evaluation`: summarizes `FinalLaunchMobileSummary.npcEvaluationRows`
- `operator_handoff`: summarizes `FinalLaunchMobileSummary.handoffRows`
- `final_launch`: summarizes `FinalLaunchMobileSummary.overallStatus`

`ForgeRootView.finalShowcaseSummary` will pass the already derived
`finalLaunchMobileSummary`, so the first card is consistent with the detailed
Final Launch panel and Demo Script.

## Stage Rules

`npc_evaluation`:

- `ready`: first row starts with `NPC Agent evaluation ready:`
- `needsAttention`: row text mentions blocked, failed, or failed eval marker
- `waiting`: missing or not-loaded rows

`operator_handoff`:

- `ready`: first row is `Final operator handoff ready.`
- `needsAttention`: row text contains blocked, missing, failed, stale, or a
  next action
- `waiting`: handoff has not loaded

`final_launch`:

- `ready`: `FinalLaunchMobileSummary.overallStatus == .ready`
- `needsAttention`: `.blocked`
- `waiting`: `.waiting`

The overall top-card status remains:

- `needsAttention` if any stage needs attention
- `readyForLocalDemo` only if the required local loop stages are ready and any
  present launch-facing stages are ready
- `waiting` otherwise

## Privacy and Safety

Reuse `FinalShowcaseSummary` sanitization for every new launch-facing detail.
The sanitizer must cover provider secrets, bearer tokens, absolute local paths,
local capture URIs, raw context/private message markers, and checkout/payment
links. The iPhone card remains read-only and must not run commands or write
provider/deploy config.

## Tests

Swift contract tests first:

- blocked Final Launch downgrades the top card to `needsAttention`
- ready Final Launch adds ready `npc_evaluation`, `operator_handoff`, and
  `final_launch` stages
- missing NPC Agent evaluation keeps the top card waiting even when local loop
  and final launch are otherwise ready
- unsafe launch detail is redacted from top-card stage details

Project/source guards will require the new root wiring, builder parameter,
stage ids, and contract tests. Backend iOS showcase acceptance will require the
same source markers.

## Visual Regression

Add a 390x844 static visual showing the top `Final Showcase` card with:

- local capture/3D/NPC/print/resource rows
- launch-facing `NPC Eval`, `Handoff`, and `Launch` rows
- a `CHECK` state for the known local iOS deploy/Xcode blockers

## Non-Goals

- No backend endpoint or payload shape changes
- No changes to provider adapters
- No Unity/SceneKit rendering changes
- No writes to `.env`, deploy xcconfig, keychain, signing, or global Xcode
  state
- No live Meshy, OpenAI, Treatstock, Sculpteo, or payment calls
