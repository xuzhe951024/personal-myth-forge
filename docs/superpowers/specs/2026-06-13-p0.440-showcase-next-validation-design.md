# P0.440 Showcase Next Validation Design

## Context

The final showcase readiness report already exposes a top-level
`next_action.validation_command`. The current iPhone Final Launch summary shows
the showcase next action command and detail, but drops the validation command.
That makes the phone handoff less copy-pasteable than the `.local` readiness
JSON, especially when the next action is an iOS deploy preflight handoff that
must be rerun after the operator fills `DEVELOPMENT_TEAM` and backend URL data.

## Design

Extend the iPhone Final Launch showcase readiness next-action row so it includes
the validation command when present. The row should keep the current command and
detail text, then append a rerun hint:

```text
Next action: ios_deployable blocked ios_deploy_evidence | DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto | ... | rerun make mobile-deploy-preflight
```

If the backend already provides a validation command prefixed with `rerun `, the
mobile summary should not duplicate the prefix. This mirrors the existing print
fulfillment handoff formatting.

## Scope

- Modify only the Swift mobile summary and contract tests.
- Do not change backend readiness logic or `.local` report generation.
- Do not write provider keys, iOS deploy config, Keychain entries, or global
  Xcode state.
- Do not run live provider calls or third-party print quote/order actions.

## Acceptance

- A Swift contract test fails before implementation because the showcase
  readiness next-action row omits `rerun make mobile-deploy-preflight`.
- The same test passes after implementation.
- Existing mobile project checks and Swift build continue to pass.
- Backend lint still passes because no backend behavior changes are expected.
