# P0.119 Visual Regression Handoff Design

## Goal

P0.119 makes the P0.118 visual regression gate part of the final launch handoff path. A Mac operator should not have to remember a separate visual command: the local final rehearsal writes a saved visual regression report, the final handoff index treats that report as required local evidence, and the iPhone final launch summary shows whether visual evidence is ready, missing, stale, or blocked.

## Scope

In scope:

- Add a saved local visual regression report at `services/backend/.local/visual-regression-local.json`.
- Add a `visual-regression-local` Make target and include it in `final-rehearsal-local`.
- Add visual regression to final handoff local source indexing and freshness accounting.
- Add visual regression to the iOS device launch rehearsal local report set.
- Add final launch readiness data for visual regression.
- Add iPhone-visible summary rows for visual regression readiness.
- Update source acceptance checks and docs so the contract is enforced.

Out of scope:

- Adding new live screenshot capture automation.
- Adding new visual artifacts to the default manifest beyond the existing P0.118 proof.
- Calling any provider APIs, writing provider keys, mutating Xcode signing, or running iOS build/sign commands from the app.

## Backend Design

The existing `visual_regression.py` remains the source of truth for artifact validation. P0.119 adds a readiness wrapper that reads the saved local report, validates its shape, maps `passed` to `ready`, maps `failed` to `blocked`, and applies the same git-HEAD freshness guard pattern used by final acceptance and evaluation readiness reports.

`final_handoff_index.py` adds `visual_regression` to `LOCAL_REPORT_SOURCES`. This makes `local_rehearsal` blocked when the visual report is missing, stale, unreadable, invalid, or failed. The saved source appears in `source_reports` and contributes to `freshness_summary`.

`final_demo_launch.py` adds a top-level `visual_regression_readiness` object. The report is read-only: it never runs screenshot or provider commands. It simply explains whether `.local/visual-regression-local.json` is acceptable for the current git revision and gives the operator a rerun action when it is not.

`ios_device_launch_rehearsal.py` also adds `visual_regression` to its local rehearsal report set so the saved iOS launch rehearsal evidence shows visual readiness alongside 3D, NPC, final acceptance, final launch, and iOS deploy runbook reports.

## Mobile Design

Swift adds a small `VisualRegressionReadinessReport` model matching the backend readiness payload. `FinalDemoLaunchReport` decodes `visual_regression_readiness`.

`FinalLaunchMobileSummary` adds `visualRegressionRows`. The rows are intentionally compact:

- Ready: `Visual regression ready: 1 artifact passed, 0 failed.`
- Missing: `Run local visual regression to create the readiness report.`
- Blocked/failed: show up to three sanitized blockers or operator actions.
- Stale: show a freshness rerun row before blocker details.

The summary continues to sanitize secrets, local paths, raw media references, checkout/payment links, and unsafe capture URIs.

## Commands

`make visual-regression` remains the direct console check and prints JSON.

`make visual-regression-local` writes:

```bash
services/backend/.local/visual-regression-local.json
```

`make final-rehearsal-local` depends on `visual-regression-local`, so a final rehearsal refreshes visual evidence automatically.

`make ios-device-launch-rehearsal` indirectly refreshes visual evidence because it already invokes `make final-rehearsal-local`.

## Testing

Backend tests cover:

- Saved visual regression readiness: missing, ready, failed, stale, unreadable, and redaction.
- CLI output and Makefile target for `visual-regression-local`.
- Final handoff index blocks when visual regression evidence is missing or stale.
- iOS launch rehearsal includes visual regression in local report sets.
- Final demo launch includes `visual_regression_readiness`.

Swift tests cover:

- Decoding `visual_regression_readiness` from final launch JSON.
- iPhone summary rows for ready, missing, blocked, stale, and sanitized unsafe text.

Source acceptance/project checks cover:

- Makefile target and `.local/visual-regression-local.json` output path.
- Backend final launch includes `visual_regression_readiness`.
- Swift model and summary include the visual regression rows.

## Verification

Final verification on merged `main` must include:

```bash
make visual-regression
make visual-regression-local
make backend-lint
make backend-test
swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests
swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks
swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck
make ios-device-launch-rehearsal
```

The expected local-state result is that commands which only inspect local readiness may exit with accepted status `2` when external iOS/configured resources are absent, but they still must write sanitized JSON reports and must not call live providers by default.
