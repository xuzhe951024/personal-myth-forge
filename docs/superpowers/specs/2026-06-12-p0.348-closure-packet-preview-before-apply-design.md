# P0.348 Closure Packet Preview Before Apply Design

## Context

`final-launch-closure-packet.json` is one of the final operator handoff packets.
Its safe local writes section already uses the read-only preview command:

```text
make final-resource-apply-preview
```

However, top-level `operator_actions` can still include direct writer actions
from nested source reports:

```text
make final-apply-resources
```

The writer is valid only after preview review. It should not appear as a
top-level closure packet action while the preview action is present.

## Goal

Make `final-launch-closure-packet` prefer the read-only final resource apply
preview before direct final resource writer actions in top-level
`operator_actions`.

## Non-Goals

- Do not change final external action ledger source data.
- Do not change safe local writer commands or resource writer scripts.
- Do not execute live provider calls, Xcode signing, Keychain writes, resource
  writers, backend `.env` writes, iOS deploy config writes, or print orders.
- Do not remove detailed nested action records that document writer existence.

## Design

Add closure-packet constants for:

```text
make final-resource-apply-preview
make final-apply-resources
```

After existing operator-action normalization and deduplication, filter the
top-level actions: if `make final-resource-apply-preview` is present, drop exact
roots equal to `make final-apply-resources`.

This keeps the closure packet aligned with the safety boundary used by the
configured preflight, demo launch, resource handoff, and configured evidence
reports.

## Acceptance Criteria

- Missing-resource closure packet includes `make final-resource-apply-preview`
  in top-level `operator_actions`.
- The same top-level action list does not include direct
  `make final-apply-resources` while preview is present.
- The safe local writes section still uses `make final-resource-apply-preview`.
- Existing first blocker, next action, safety flags, and command list behavior
  remain unchanged.
- Focused closure packet tests, operator action tests, lint, diff checks, and
  backend tests pass.
