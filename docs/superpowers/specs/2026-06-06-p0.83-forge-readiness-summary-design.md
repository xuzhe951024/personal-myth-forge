# P0.83 Forge Readiness Summary Design

## Goal

Add a single iPhone-visible Forge readiness summary near the `Forge Myth` action. The player or operator should be able to see, before forging, whether capture media, context capsule approval, 3D generation routing, and NPC Agent provider setup are ready.

## Scope

This iteration adds a mobile summary and UI panel only. It does not change provider selection, live-call consent, Meshy/OpenAI calls, capture upload, myth session creation, printing, or Xcode deploy behavior.

## User Experience

The capture form gains a compact `Forge Readiness` panel immediately before the `Forge Myth` button. It shows:

- overall state: waiting, ready, or needs attention
- route label: local demo, configured providers, or waiting
- short rows for capture, context, 3D provider, and NPC Agent provider
- privacy notes explaining that provider keys and raw source data stay backend-only

The button remains disabled when capture media is not ready or the context capsule is not approved. The panel makes the same rule explicit and also surfaces provider readiness before the user starts the forge flow.

## Architecture

Create `ForgeReadinessSummary` in `PersonalMythForgeMobileCore`. The builder combines existing summaries:

- `CaptureGenerationReadiness`
- `ContextCapsuleReview`
- `ProviderReadinessResponse`
- `NPCAgentModeSummary`

`ForgeRootView` builds the summary from existing computed properties and passes it to `CaptureFormView`. `CaptureFormView` renders the panel with fixed compact rows and privacy notes.

## Data Rules

The summary may include provider kind, selected provider name, missing env variable names, route labels, and status text. It must not include API key values, Authorization tokens, local file paths, local capture URLs, checkout links, or raw personal source text.

## Testing

Add Swift contract tests for:

- waiting when capture media is not ready
- waiting when context capsule is not approved
- ready local demo route when capture, context, 3D, and NPC Agent are ready
- needs attention when backend provider readiness errors
- needs attention when configured providers are missing Meshy/OpenAI env values
- redaction of unsafe provider/error text

Update Swift project checks and backend iOS showcase acceptance so this panel remains part of the final mobile contract.

## Acceptance

The iteration is accepted when focused Swift and backend source gates pass, the Swift app compile target builds, full backend lint/test pass, quick final acceptance has zero failed checks, and the 390x844 visual evidence shows the new panel without obvious clipping or overlap.

## Self-Review

No placeholders remain. The design is one bounded mobile entry-readiness feature, uses existing summaries instead of duplicating provider logic, and preserves backend-only secret handling.
