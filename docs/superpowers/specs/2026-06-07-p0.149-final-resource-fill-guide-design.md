# P0.149 Final Resource Fill Guide Design

## Problem

The final resource handoff has become executable, but the resource reports still
mix two different ideas under one field:

- where the operator fills the value now
- where `make final-apply-resources` eventually writes the value

For backend provider keys those paths are often the same source file, but for
iOS deploy values the operator still fills
`services/backend/.local/final-resources.env` while the apply step writes
`apps/mobile/ios/Config/Deployment.local.xcconfig`.

This ambiguity matters at final acceptance time because the user plans to provide
all API/key resources at once.

## Goal

Make the final resource contract unambiguous and add a read-only fill guide that
lists exactly which values the user must provide, where to put them, where they
will be applied, and which commands validate the handoff.

## Non-Goals

- Do not write or modify `final-resources.env`.
- Do not write backend `.env`.
- Do not write iOS deploy config.
- Do not call live providers.
- Do not run Xcode, signing, keychain, or device install steps.
- Do not expose existing secret values in reports or Markdown.

## Report Shape

Every `final_resource_requirements_report.requirements[]` row will keep the
existing `destination` field for backward compatibility. It will also add:

- `input_source`: the operator-filled source file, always
  `services/backend/.local/final-resources.env` for final resources.
- `write_destination`: the eventual applied destination, such as
  `services/backend/.env` or
  `apps/mobile/ios/Config/Deployment.local.xcconfig`.
- `apply_command`: `make final-apply-resources`.
- `fill_action`: a concise operator instruction, such as
  `fill DEVELOPMENT_TEAM in services/backend/.local/final-resources.env`.

For compatibility, `destination` remains what it means today: the final
destination concept used by existing mobile decoding and action ledger tests.

## Fill Guide

Add `final_resource_fill_guide.py` with
`build_final_resource_fill_guide_report(repo_root=None, resources_file=None)`.

The report includes:

- `kind`: `final_resource_fill_guide_report`
- `status`: `ready` when required values are filled, else `blocked`
- `required_inputs`: missing or blocked required rows
- `optional_inputs`: optional rows
- `configured_inputs`: ready rows
- `commands`: ordered validation/apply commands
- `markdown`: a sanitized Markdown guide
- `source_reports.final_resource_requirements`
- `safety`

The Markdown guide is intentionally human-readable and safe to share inside the
repo-local `.local` directory. It must:

- list required missing values first
- show `input_source` and `write_destination`
- mark secrets as `<secret: fill locally>`
- not include actual secret values, raw IP addresses from configured resources,
  local absolute paths, or payment links
- include the exact command sequence:
  1. `make final-resource-requirements`
  2. `make final-resource-apply-preview`
  3. `make final-apply-resources`
  4. `make final-local-report-refresh`

## CLI and Make

Add:

- CLI: `final-resource-fill-guide --repo-root ../.. --output .local/final-resource-fill-guide.json --markdown-output .local/final-resource-fill-guide.md`
- Make: `make final-resource-fill-guide`

The command is read-only except for writing the two ignored `.local` report files.

## Integration

Add the fill guide to `final-local-report-refresh` so the guide stays current
when the operator refreshes final evidence.

## Testing

Use TDD:

- Requirements report test proves required rows expose `input_source`,
  `write_destination`, `apply_command`, and `fill_action`.
- Fill guide test proves missing resources produce a blocked report, Markdown
  lists required inputs, and secrets/paths are redacted.
- Fill guide ready test proves valid resources produce `ready` without leaking
  secret values or backend LAN URL.
- CLI test proves JSON and Markdown outputs are written.
- Make target dry-run test proves the target calls the CLI with both outputs.
- Final local report refresh test proves `final-resource-fill-guide.json` and
  `.md` are refreshed.
