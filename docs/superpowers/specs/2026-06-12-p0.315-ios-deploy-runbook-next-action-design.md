# P0.315 iOS Deploy Runbook Next Action Design

## Goal

Expose the iOS deploy runbook's immediate blocker and next action as structured fields so final launch handoff and iOS summary screens do not need to infer them from input slots and operator action arrays.

## Context

The final iOS showcase is still blocked by external local conditions: provider resources, Apple Team ID, iPhone-reachable backend URL, Xcode license/build state, and real device evidence. The runbook already lists input slots, command steps, and operator actions, but it does not have top-level `first_blocker` or `next_action`.

## Scope

This iteration only changes report structure, Swift decoding, and summary text. It does not run Xcode, write signing config, touch Keychain, call live providers, place orders, or write secrets.

## Behavior

The backend runbook derives `first_blocker` in this order:

1. first required `input_slot` whose status is not `ready`
2. first `command_sequence` step whose status is not `ready`
3. `null` when no blocker remains

For input slots, `first_blocker.command` is the validated operator action for that slot and `input_source` records the original slot source. For command steps, `first_blocker.command` is the step command and `detail` is the joined notes.

`next_action` is a copy of `first_blocker` with `source: "first_blocker"`. When there is no first blocker, `next_action` is `null`.

## iOS Contract

Swift adds `IOSDeployRunbookFirstBlocker` and `IOSDeployRunbookNextAction` models and optional fields on `IOSDeployRunbookReport`. `FinalLaunchMobileSummary` shows compact, redacted rows for first blocker and next action.

## Testing

Backend tests cover missing input blockers and the all-inputs-ready manual Xcode gate. Swift contract tests cover decoding and summary text.
