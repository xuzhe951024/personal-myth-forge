# P0.123 Print Fulfillment Readiness Design

## Goal

P0.123 adds a read-only print fulfillment readiness surface for the final demo. It answers whether the "real-world relic" path has local quote evidence, iPhone approval UX evidence, visual regression coverage, resource/key handoff coverage, and live Treatstock evidence readiness without placing orders or calling providers.

## Product Boundary

This iteration does not implement order placement, payment, shipping forms, address capture, fulfillment tracking, or automatic checkout. The demo claim remains: a generated `print_asset` can become a third-party print quote handoff, and user approval is required before any external fulfillment step.

## Approach

Add a backend `print_fulfillment_readiness_report` that aggregates existing evidence:

- `print_quote_acceptance`: deterministic local quote from a printable candidate.
- `ios_showcase_acceptance`: iPhone source gate for `PrintFulfillmentReceipt` and final launch display wiring.
- `visual_regression_readiness`: static artifact coverage for `p0.101_print_fulfillment_receipt`.
- `resource_handoff` and `final_resources_preflight`: backend-only `PRINT_PROVIDER` and `TREATSTOCK_API_KEY` readiness.
- `live_provider_evidence`: whether live Treatstock proof has been supplied after operator consent.

The report returns `blocked` when local print proof or required UI/source evidence is missing, `partial` when local proof is ready but live Treatstock evidence is not ready, and `ready` only when local proof, UI evidence, resource handoff, and live Treatstock evidence are all ready.

## Integration Points

- Add a CLI command and Make target: `print-fulfillment-readiness`.
- Embed `print_fulfillment_readiness` into `final_demo_launch`.
- Add a `print_fulfillment` capability to `final_showcase_readiness`, ordered after AI Agent NPC and before provider/key handoff.
- Expose the new report through iPhone final launch models and render it in `FinalLaunchStatusView` under `Print Fulfillment`.
- Extend source acceptance so final demo readiness requires the backend report, CLI/Make entry, final launch embedding, Swift model, summary rows, and mobile display section.

## Safety

The report must be read-only. It must not call Meshy, OpenAI, Treatstock, Apple tooling, Xcode, signing, Keychain, or payment endpoints. It must not write `.env`, deployment config, provider keys, raw media, raw private context, checkout/payment links, or absolute local paths into the report.

## Testing

Use TDD for each surface:

- Backend unit tests for missing, partial, ready, visual/source blocker, and sanitization behavior.
- CLI tests for JSON output and exit code propagation.
- Final demo launch and final showcase readiness tests for embedding/capability behavior.
- iOS source acceptance tests for the new required feature.
- Swift contract tests for decoding and redacted display rows.
- Full verification remains backend lint/test, Swift contract/project/build checks, visual regression, final rehearsal, iOS launch rehearsal, and final showcase readiness.
