# P0.521 Device Bundle Concrete Next Action Design

## Context

P0.520 made `mobile-deploy-preflight-evidence.next_action.command` validation-aware:

```bash
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

Top-level final reports now promote that closed-loop command, but copied device action bundles in `final-demo-launch-local` and `final-local-report-refresh` can still expose the older raw writer command in `device_action_bundle.first_action.command`.

## Goal

When a source device action carries `saved_next_action.command`, downstream final handoff reports should use that command for the copied first action. The visible first device action should tell the operator both what to do and which validation command to rerun.

## Scope

- Promote `saved_next_action.command` into copied `first_action.command` for final demo launch and final local report refresh device bundles.
- Preserve `validation_command` on the first action.
- Preserve the nested `saved_next_action` payload for provenance.
- Keep report generation read-only and local-only.

## Non-Goals

- Do not run Xcode, signing, Keychain, real devices, live providers, or print providers.
- Do not change the actual mobile deploy preflight command runner.
- Do not write API keys or provider credentials.

## Acceptance Criteria

- `final-demo-launch-local.device_action_bundle.first_action.command` uses the validation-aware saved command when present.
- `final-local-report-refresh.device_action_bundle.first_action.command` uses the validation-aware saved command when present.
- Existing top-level blocker command behavior remains unchanged.
- Focused tests cover both downstream reports.
- Full backend lint and tests pass before merge.
