# P0.438 Final Acceptance Next Validation Design

## Context

P0.437 made the iPhone Final Launch acceptance section decode and display
`final_acceptance_readiness.blockers[].next_action`. The current real local
report includes:

```json
{
  "command": "DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto",
  "detail": "Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable",
  "validation_command": "make mobile-deploy-preflight"
}
```

The mobile row currently shows the next command and detail, but not the
validation command. That leaves the iPhone operator with the config writer but
not the explicit rerun command that proves the device gate.

## Decision

Append `validation_command` to the rendered final acceptance next action when it
is present, using `rerun <validation_command>` so the handoff is explicit. Keep
the blocker command and detail unchanged, and do not synthesize new backend
actions. The row format becomes:

```text
... | next DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable | rerun make mobile-deploy-preflight
```

## Scope

Included:

- Add Swift contract coverage using a realistic final acceptance next action
  detail that does not already contain the rerun command.
- Update the Final Launch acceptance row formatter to include
  `nextAction.validationCommand`.
- Update the roadmap.

Not included:

- No backend behavior change.
- No Xcode signing, Keychain writes, global Apple SDK license actions, provider
  calls, or paid print-provider actions.
- No change to final acceptance blocker ordering.

## Testing

Use TDD:

1. Change the Swift fixture detail to match the real local report.
2. Assert the acceptance row still contains `make mobile-deploy-preflight`
   through the nested next action validation command.
3. Confirm RED.
4. Add the minimal formatter change.
5. Run Swift contract, project checks, Swift build, backend lint, and diff
   checks before merge.
