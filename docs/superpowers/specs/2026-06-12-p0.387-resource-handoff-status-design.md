# P0.387 Resource Handoff Status Design

## Context

The final readiness reports mostly expose a top-level `status` field that can
be decoded uniformly by CLI summaries, iPhone final launch panels, and saved
`.local` report scanners. `resource-handoff.json` currently exposes
`overall_status` but not `status`:

```text
resource-handoff.json status = None
resource-handoff.json overall_status = blocked
```

The report already has correct blocker detail and next-action routing for
`MESHY_API_KEY`, `OPENAI_API_KEY`, iOS deploy config, and live-provider handoff.
The gap is contract shape, not readiness logic.

## Decision

Add a top-level `status` field to `build_resource_handoff_report()` and keep it
equal to `overall_status`.

- Preserve the existing `overall_status` field for current consumers.
- Do not change summary calculation, first blocker selection, next action,
  operator action ordering, provider selection, iOS deploy config reading, or
  secret redaction.
- Keep the saved `.local/resource-handoff.json` blocked until the real required
  resources are provided by the operator.
- Do not write any real keys, call Meshy/OpenAI/Treatstock, touch Xcode signing,
  or mutate global machine state.

## Testing

Extend `tests/test_resource_handoff.py` so the default blocked report asserts:

- `report["status"] == "blocked"`
- `report["status"] == report["overall_status"]`

Also extend the fully configured ready case so it asserts:

- `report["status"] == "partial"`
- `report["status"] == report["overall_status"]`

The ready case remains `partial` because optional/manual items such as
`SCULPTEO_API_KEY` and `APPLE_SDK_LICENSE` are still not final-complete.
