# P0.133 Final Acceptance Actions Design

## Context

The iOS device launch rehearsal now expands nested `operator_actions` from saved
reports. The remaining first local blocker is still coarse:

`final_rehearsal_local: review final_acceptance_local: make final-acceptance-local`

The saved final acceptance report already contains structured blocked checks such
as `mobile_deploy_preflight` and `mobile_xcode_build`, but it does not publish a
top-level `operator_actions` list. Downstream reports can only ask the operator
to review the whole final acceptance report.

## Decision

Add bounded top-level `operator_actions` to `final_acceptance_report`.

The actions are derived from the sanitized check list after summary/status
calculation:

- `mobile_deploy_preflight` with `blocked_by_local_ios_backend_health` becomes
  `start backend-device-demo and rerun mobile deploy preflight`.
- Other `mobile_deploy_preflight` blockers become
  `provide iOS deploy config and rerun mobile deploy preflight`.
- `mobile_xcode_build` blockers become
  `resolve Xcode build gate outside the app`.
- Provider/demo configuration blockers become
  `provide backend provider keys in services/backend/.local/final-resources.env and rerun final acceptance`.
- Other failed checks become `fix <check_id>: <label>`.
- Other blocked command checks become `unblock <check_id>: <command>`.
- Other blocked in-process checks become `unblock <check_id>: <label>`.

Actions are deduped, capped to six items, and sanitized by the existing final
acceptance report sanitizer. A passing report includes an empty action list.

## Alternatives Considered

1. Derive final acceptance actions only inside
   `ios_device_launch_rehearsal.py`. This would fix the current rehearsal output
   but leave `final_acceptance_report` less useful for other handoff surfaces.
2. Create a shared action-mapping module for final acceptance and final
   acceptance readiness. This may become useful once there is another caller, but it is premature for a
   single source report change.
3. Add `operator_actions` directly to `final_acceptance_report`. This is the
   smallest durable fix because all downstream readers already know how to
   preserve and route nested actions.

## Data Flow

`run_final_acceptance` executes inline checks and command gates, computes
`summary` and `overall_status`, then attaches `operator_actions` before
sanitization. `ios_device_launch_rehearsal._source_report` preserves that list,
`_local_rehearsal_operator_actions` prefixes it with `final_acceptance_local`,
and the top-level rehearsal report prefixes it with `final_rehearsal_local`.

## Safety

This iteration does not run live providers, Xcode signing, Apple SDK license
commands, keychain operations, or global environment changes. It only changes
local report generation and tests.
