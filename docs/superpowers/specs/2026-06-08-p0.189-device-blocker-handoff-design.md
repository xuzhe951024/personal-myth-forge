# P0.189 Device Blocker Handoff Design

## Problem

The final showcase readiness ledger correctly reports that iOS deployability is still blocked, but two intermediate reports are too hard to consume:

- `final_handoff_index_report` has lanes and operator actions but no top-level `first_blocker`.
- `ios_device_launch_rehearsal_report` has a sequence and operator actions but no top-level `first_blocker`.

The visual regression index also lacks a dedicated phone-sized proof that summarizes the current device-evidence blocker chain: missing deploy config, backend LAN preflight, and Xcode license/build evidence.

## Scope

P0.189 adds a structured first-blocker handoff without changing readiness semantics:

- Add `first_blocker` to `final_handoff_index_report`.
- Add `first_blocker` to `ios_device_launch_rehearsal_report`.
- Preserve all existing statuses, summaries, safety flags, and operator actions.
- Add one checked-in 390x844 visual artifact for the device blocker handoff.
- Update source/project gates so the new report fields and visual artifact remain covered.

## Non-Goals

- Do not mark iOS deployability ready.
- Do not run Xcode, accept Apple licenses, install to a device, start servers, or call providers.
- Do not write deploy config or provider keys.
- Do not change provider/key handoff requirements.

## Data Shape

`first_blocker` is either `null` or a sanitized object:

```json
{
  "id": "local_rehearsal",
  "label": "Local rehearsal",
  "status": "missing",
  "classification": "lane_missing",
  "command": "make final-rehearsal-local",
  "detail": "run make final-rehearsal-local"
}
```

For device launch rehearsal, the object is derived from the first `sequence` item whose status is `missing` or `blocked`. Its detail uses the first nested operator action when available, otherwise a review/refresh action.

## Visual Artifact

## Acceptance Criteria

- Missing `final_handoff_index` inputs produce a `first_blocker` for `local_rehearsal`.
- Ready `final_handoff_index` inputs produce `first_blocker: null`.
- Blocked `ios_device_launch_rehearsal` inputs produce a `first_blocker` with the first actionable nested detail.
- Ready/partial device launch rehearsal inputs produce `first_blocker: null`.
- `check_visual_artifacts` passes with 21 artifacts, including `p0.189_device_blocker_handoff`.
- README and Swift project checks mention 21 static visual artifacts and the device blocker handoff.
- Existing safety flags continue to prove no device/Xcode/live/global action is executed by report builders.
