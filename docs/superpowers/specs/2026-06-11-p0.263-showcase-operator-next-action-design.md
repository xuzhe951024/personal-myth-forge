# P0.263 Final Showcase Operator Next Action Design

## Goal

Make `final-showcase-readiness` expose its own concrete `next_action` as the first top-level `operator_actions` entry.

## Context

`final-showcase-readiness.json` already computes a concrete `next_action` for the first final showcase blocker. In the current local iOS blocker case that action is:

`provide DEVELOPMENT_TEAM in Deployment.local.xcconfig`

with validation:

`make mobile-deploy-preflight`

The report's `operator_actions` list is still assembled only from nested reports and capability commands. That means the first top-level action can be a diagnostic detail from a child report even though the same report already knows the exact next command.

## Options Considered

1. **Prepend `next_action` to `operator_actions` in `final_showcase_readiness`.** Preferred because it fixes the report closest to the final showcase ledger and keeps mobile consumption simple.
2. **Only rely on `final-demo-launch-local.operator_actions`.** This helps launch reports but leaves the final showcase ledger inconsistent.
3. **Teach Swift to synthesize the action from `next_action`.** This duplicates backend command formatting and does not help CLI handoff reports.

## Design

`build_final_showcase_readiness_report()` will compute `first_blocker` and `next_action` once, then pass the concrete action into `_operator_actions()`.

Rules:

- If `next_action.command` is present, prepend it to `operator_actions`.
- If `next_action.validation_command` is present, format the entry as `<command>; rerun <validation_command>`.
- Preserve all existing nested report actions after the top-level action.
- Deduplicate while preserving order.
- Keep the existing action limit.
- Do not change readiness status, blockers, provider calls, resource application, Xcode signing, or global state.

## Test Plan

Extend `test_final_showcase_readiness_next_action_uses_preflight_child_action()` to assert:

`operator_actions[0] == "provide DEVELOPMENT_TEAM in Deployment.local.xcconfig; rerun make mobile-deploy-preflight"`

The same test already builds the exact local device blocker and verifies the concrete `next_action`, so it is the narrowest regression point.

## Self-Review

- No placeholders.
- Scope is one report contract improvement.
- The change moves the iOS showcase handoff closer to the final deployable demo without touching external resources.
