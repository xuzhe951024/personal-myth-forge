# P0.317 Configured Evidence Next Action Design

## Context

The final showcase goal requires configured live evidence for Meshy 3D generation, OpenAI NPC Agent behavior, configured final acceptance, and final launch. The current `configured_live_evidence_bundle_report` already aggregates command sequence, evidence files, safety flags, live-call policy, and a `current_blocker`.

The missing piece is consistency with newer final launch reports: operators and the iPhone Final Launch summary should not have to infer the next action from `current_blocker`, `operator_actions`, or the command sequence. The report should expose a top-level `first_blocker` and `next_action`.

## Scope

Add `first_blocker` and `next_action` to `configured_live_evidence_bundle_report`.

The fields will:

- Mirror the existing sanitized `current_blocker` when the bundle is blocked or consent-required.
- Be `null` when the bundle is `ready` or `ready_to_run` and no current blocker exists.
- Add `"source": "first_blocker"` to `next_action`.
- Preserve command, classification, blocked-by, and detail fields already exposed on the current blocker.

Update Swift models and the iPhone Final Launch summary to decode and display the new `next_action` row.

## Out Of Scope

- No real Meshy call.
- No OpenAI live call.
- No configured final acceptance run.
- No Treatstock call or payment action.
- No Xcode signing, Keychain writes, or global environment mutation.
- No change to how command sequence status is calculated.

## Data Flow

```text
configured plan + live provider evidence
  -> command_sequence + current_blocker
  -> first_blocker
  -> next_action
  -> FinalDemoLaunchReport.configuredLiveEvidenceBundle
  -> iPhone Final Launch configured bundle rows
```

## Safety Behavior

The existing `_sanitize_report()` still sanitizes the entire report after adding the new fields. This keeps API keys, bearer tokens, local paths, payment links, and raw media references out of the JSON and iPhone summary.

Swift decoding remains backward-compatible because `firstAction` and `nextAction` are optional.

## Acceptance Criteria

- Backend configured bundle reports include `first_blocker` and `next_action` when blocked by missing resources.
- Backend configured bundle reports include the same fields when blocked by live cost consent.
- Backend configured bundle reports set both fields to `None` when ready or ready-to-run.
- Swift contract tests decode `nextAction` from final launch payload fixtures.
- Final Launch summary rows show `Next action: ...` for configured evidence bundle readiness.
- Existing redaction tests cover the new summary row.
- Focused backend tests, Swift contract tests, mobile compile checks, lint, diff check, and full backend tests pass.

## Self Review

- The design is limited to configured evidence handoff visibility.
- It reuses existing blocker selection instead of inventing a new status order.
- It does not trigger or enable live provider work without explicit operator consent.
- It keeps all resource/key material in ignored local handoff files.
