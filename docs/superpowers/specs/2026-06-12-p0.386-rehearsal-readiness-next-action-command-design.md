# P0.386 Rehearsal Readiness Next Action Command Design

## Context

P0.385 made the top-level `ios-device-launch-rehearsal` next action command
copy-pasteable by stripping ` | ...` detail suffixes from the structured
`next_action.command` field. The saved readiness wrapper still has the same
shape problem when it embeds the saved rehearsal report:

```text
ios_device_launch_rehearsal_readiness.next_action.command =
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

This nested report is consumed by final demo launch, configured preflight, and
final showcase readiness. Its structured command should be executable, while
human explanation remains available in `detail` and `operator_actions`.

## Decision

Normalize only the structured `next_action.command` produced by
`ios_device_launch_rehearsal_readiness`.

- If the command comes from `operator_actions[0]`, split off the trailing
  ` | ...` suffix before writing `next_action.command`.
- Keep `operator_actions` unchanged so text reports still show missing
  `DEVELOPMENT_TEAM` and backend URL detail.
- Keep `first_blocker.detail` unchanged.
- Do not change live provider, Xcode, signing, key, or print fulfillment
  behavior.

## Testing

Extend the existing iOS rehearsal readiness tests so a saved rehearsal report
with a detail-suffixed operator action produces:

- `next_action.command` without the suffix.
- `operator_actions[0]` still containing the suffix.
- `next_action.detail` still explaining the missing iOS deploy values.
