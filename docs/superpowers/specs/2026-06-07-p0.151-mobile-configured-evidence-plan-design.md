# P0.151 Mobile Configured Evidence Plan Design

## Problem

P0.150 added a safe backend `final_configured_evidence_plan` report, but the
configured evidence sequence is still not part of the final launch payload that
the iPhone app decodes. The iPhone final launch panel can show live provider
evidence, resource handoff, apply preview, and deploy runbook status, but it
does not show the ordered configured evidence plan: which steps are blocked,
which steps write repo-local config, and which steps require live provider cost
consent.

## Goal

Make the configured evidence plan visible in the final launch handoff surface:
backend `final-demo-launch` embeds the plan, Swift decodes it, the mobile summary
turns it into concise rows, and `FinalLaunchStatusView` renders those rows.

## Non-Goals

- Do not execute configured evidence commands.
- Do not call Meshy, OpenAI, Treatstock, or any live provider.
- Do not write backend `.env`.
- Do not write iOS deploy config.
- Do not run Xcode, signing, keychain, or device install commands.
- Do not expose provider secrets, raw media, raw private context, payment links,
  absolute local paths, or private LAN URLs.

## Backend Changes

`build_final_demo_launch_report` will call
`build_final_configured_evidence_plan_report(repo_root=selected_repo_root,
settings=selected_settings)` and include it as
`final_configured_evidence_plan`.

The report remains read-only and uses `allow_live_provider_calls=False`.
The final launch `safety` block remains unchanged because the plan does not run
commands or provider calls.

## Swift Model Changes

Add focused Codable models in `PMFModels.swift`:

- `FinalConfiguredEvidencePlanSummary`
- `FinalConfiguredEvidencePlanStep`
- `FinalConfiguredEvidencePlanLiveCallPolicy`
- `FinalConfiguredEvidencePlanSafety`
- `FinalConfiguredEvidencePlanReport`

Add `finalConfiguredEvidencePlan: FinalConfiguredEvidencePlanReport?` to
`FinalDemoLaunchReport`.

The model maps snake_case through the existing `PMFJSON.decoder` strategy.
Defaults are used only where existing API conventions already allow optional
fields. Required plan fields should decode strictly enough to catch broken
backend payloads.

## Mobile Summary and UI

Add `configuredEvidencePlanRows` to `FinalLaunchMobileSummary`.

Rows should include:

- status/count summary: steps, blocked, consent-required, live/cost steps
- first blocked or consent-required step with command
- live call policy: default live calls false and consent flag
- safety: commands and live provider calls are false

`FinalLaunchStatusView` renders this under a new `Configured Evidence` group.
Rows are sanitized through existing summary sanitization.

## Acceptance and Project Checks

Extend source gates so the mobile readiness acceptance requires:

- backend payload field `final_configured_evidence_plan`
- Swift model type `FinalConfiguredEvidencePlanReport`
- Swift report field `finalConfiguredEvidencePlan`
- summary rows `configuredEvidencePlanRows`
- UI group `Configured Evidence`
- contract test coverage for decode and summary rows

## Testing

Use TDD:

- Backend test proves final demo launch embeds the configured evidence plan and
  does not run commands or live provider calls.
- Swift contract test proves final launch payload decodes the new plan.
- Swift contract test proves mobile summary shows configured evidence plan rows.
- Swift contract test proves unsafe configured evidence plan strings are
  redacted.
- Backend mobile readiness acceptance/source gate test proves the mobile source
  requires the new configured evidence plan model, rows, UI group, and contract
  tests.
- Project checks and Swift compile checks prove the new UI/model surface stays
  wired.
