# P0.352 Final Showcase iOS Blocker Command Design

## Goal

Make `final-showcase-readiness` expose the concrete iOS deploy handoff command on its top-level `first_blocker`, not only on `next_action`, so the final showcase readiness report points directly at the next operator input needed for the physical-iPhone path.

## Context

The final goal is blocked by iOS deployment evidence and local provider resources. Current saved readiness already gives a concrete `next_action`:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto
```

with `make mobile-deploy-preflight` as its validation command. The same report's `first_blocker.command` still says `make ios-device-launch-rehearsal`, which is a later wrapper command that remains blocked until local deploy config and backend URL evidence exist.

The final demo and closure packet reports have recently been tightened so their top-level actions identify the concrete handoff. The showcase readiness report should follow that same contract because it is the main capability ledger for the whole objective.

## Design

When the first blocked capability is `ios_deployable`, `final-showcase-readiness` already computes a `device_action_bundle`. The first blocker should use the preferred device action source to project a concrete command:

- If the preferred device action has a child `next_action`, use that child command.
- Otherwise use the preferred device action's command when its evidence is present and blocked.
- Preserve the existing blocker detail, including the `Next device action: ...` hint and sanitized preflight detail.
- Copy a child `validation_command` onto `first_blocker` when one exists.
- Let `next_action` mirror `first_blocker.command` and `first_blocker.validation_command`.

This keeps missing-evidence behavior unchanged. If no concrete device evidence exists yet, the blocker can still point at the existing rehearsal command because there is no safer child action to promote.

## Safety

This phase only changes a read-only report builder and tests. It does not write deploy config, edit secrets, call Meshy/OpenAI/Treatstock, run Xcode, sign an app, touch Keychain, or place orders. Local verification may refresh ignored JSON reports, which are sanitized summaries and remain blocked without real resources.
