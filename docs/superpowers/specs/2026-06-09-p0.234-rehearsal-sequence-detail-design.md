# P0.234 Rehearsal Sequence Detail Design

## Context

P0.232 lets iOS display launch rehearsal sequence row `detail`, and P0.233
preserves that field in backend readiness normalization. The original
`ios-device-launch-rehearsal` report still emits sequence rows without
`detail`, so the real saved report cannot yet feed the new mobile display unless
another process injects the field.

## Goal

Add `detail` to each missing or blocked `ios-device-launch-rehearsal` sequence
row. The detail should match the explanation already used by `first_blocker`,
so downstream readiness and mobile summary views all show the same operator
guidance.

## Scope

This only changes the backend rehearsal report shape. It does not change status
calculation, command selection, final demo launch aggregation, provider calls,
Xcode behavior, or mobile UI code.

## Architecture

After building the rehearsal `sequence`, normalize rows through a helper that
adds `detail` only for `missing` and `blocked` statuses. Reuse the existing
`_step_blocker_detail` function so sequence rows and `first_blocker.detail`
cannot diverge.

## Testing

Use TDD:

1. Add a failing test requiring missing sequence rows to include the same
   `detail` guidance as `first_blocker`.
2. Add a failing test requiring blocked sequence rows with nested operator
   actions to include the normalized first nested action.
3. Implement the smallest helper to add details before the report is sanitized.
4. Verify focused backend tests, full backend tests, mobile contract
   compatibility, and diff hygiene.

- Safety remains sanitizer-backed.
