# P0.530 Print Request Aggregator Design

## Context

P0.529 made `print-fulfillment-readiness` check the ignored configured quote request file before the guarded Treatstock quote command. `final-showcase-readiness` and local demo launch use that request preflight, but higher-level aggregators can still flatten their child reports into a generic print readiness command or a guarded quote command.

That weakens the final operator handoff because the closure packet and external action ledger should not ask for a provider quote before the local request payload exists.

## Decision

Make `final-external-action-ledger` and `final-launch-closure-packet` preserve concrete print child actions.

When `print-fulfillment-readiness` reports:

- missing/invalid configured quote request: top-level aggregators surface `prepare services/backend/.local/print-quote-request-configured.json; rerun make print-fulfillment-readiness`;
- configured request ready but quote missing: top-level aggregators surface the guarded `PMF_ALLOW_PRINT_PROVIDER_CALLS=1 make print-quote-configured; rerun make print-fulfillment-readiness`;
- unrelated print source or visual evidence missing: top-level aggregators keep the relevant readiness validation action.

## Non-Goals

- Do not create the request file.
- Do not call Treatstock.
- Do not change quote/order approval behavior.
- Do not add keys or write global state.

## Acceptance

- External action ledger and closure packet tests prove request-preflight wins over guarded provider quote when both appear.
- Current local `.local` reports show the same print handoff action across final showcase, external ledger, and closure packet.
- Full backend tests still pass.
