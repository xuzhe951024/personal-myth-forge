# P0.509 iOS Certificate Backend Priority Design

## Goal

Make the iOS device launch certificate tell an operator to start `backend-device-demo` immediately after the deploy-config writer, before provider, print, Xcode, or backend URL follow-up actions.

## Current Behavior

`services/backend/.local/ios-device-launch-certificate.json` is blocked as expected, but its `operator_actions` currently place provider and print handoff commands before the backend LAN server action:

1. `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight`
2. provider handoff
3. print handoff
4. `start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight`

That ordering is less useful for the physical iPhone path. Once the deploy config writer is the first blocker, the next actionable local step is to start the backend device server so the subsequent mobile preflight can test an iPhone-reachable `/health`.

## Desired Behavior

When certificate operator actions begin with a concrete mobile deploy handoff, the ordered actions should be:

1. deploy writer or deploy preflight handoff
2. validated `backend-device-demo` handoff
3. iPhone-reachable backend URL handoff, when present
4. provider and print handoffs
5. remaining actions such as Xcode or configured evidence

If the first action is not a mobile deploy handoff, preserve the existing defensive ordering so missing-report or unrelated blocked states do not jump ahead to backend startup.

## Scope

In scope:

- Add a regression test to `services/backend/tests/test_ios_device_launch_certificate.py`.
- Update `services/backend/src/myth_forge_api/ios_device_launch_certificate.py` sorting logic.
- Add the P0.509 roadmap entry.

Out of scope:

- Real Xcode signing, Keychain writes, provider calls, print quotes, or paid actions.
- Changing `ios-device-launch-rehearsal`; that can be a separate small phase if its ordering remains stale.
- Changing the actual first blocker, which remains `DEVELOPMENT_TEAM`.

## Safety

The report remains read-only. It must not run commands, write secrets, call live providers, mutate global Xcode state, or place print orders.
