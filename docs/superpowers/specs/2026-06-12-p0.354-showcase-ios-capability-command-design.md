# P0.354 Final Showcase iOS Capability Command Design

## Goal

Make the `ios_deployable` capability row in `final-showcase-readiness` expose the same concrete iOS deploy handoff command that the report already exposes through `first_blocker` and `next_action`.

## Context

The final showcase readiness report now surfaces this concrete top-level action when saved mobile preflight evidence exists:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto
```

with `make mobile-deploy-preflight` as the validation command. However, the `capabilities_by_id.ios_deployable.command` row still shows `make ios-device-launch-rehearsal`. That command is useful later, but it is not the next operator action when local iOS deploy config is missing.

The iPhone Final Launch panel reads capability rows as well as top-level actions. Keeping the row command generic makes the mobile status list less actionable than the top-level receipt.

## Design

After the existing `device_action_bundle` is built, decorate the capabilities list for report display:

- If the capability id is not `ios_deployable`, leave it unchanged.
- If the device action bundle exposes a preferred concrete command, copy that command to the `ios_deployable` row.
- If the preferred concrete action has a validation command, copy it to the row as `validation_command`.
- Do not change capability status, classification, evidence, or readiness summary.
- Keep first blocker and next action behavior unchanged; they can continue using the existing detail-enrichment path.

This keeps the raw readiness calculation stable while making mobile-visible capability rows match the current operator handoff.

## Safety

This phase only changes read-only report shaping and tests. It does not write deploy config, write secrets, call Meshy/OpenAI/Treatstock, run Xcode, sign builds, touch Keychain, or place orders. Ignored `.local` report refreshes remain safe and are expected to stay blocked without real local resources.

  - `capabilities_by_id.ios_deployable.validation_command == "make mobile-deploy-preflight"`
  - `first_blocker.command` and `next_action.command` remain the same concrete handoff.
