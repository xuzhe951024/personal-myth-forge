# P0.28 Mobile Backend History Sync Design

## Context

P0.26 added a local iOS demo snapshot so the last showable myth run survives app
relaunch. P0.27 added backend-owned local session history and Swift client
methods:

- `GET /v1/myth-sessions/{session_id}`
- `GET /v1/myth-sessions/{session_id}/history`
- `PersonalMythForgeAPIClient.getMythSessionHistory(sessionId:)`

The iOS app still does not call the backend history endpoints from the visible
demo flow. That means backend-owned NPC continuity exists, but the mobile demo
surface continues to rely only on client-local snapshot state.

## Decision

Add a non-blocking backend history sync path to the iOS app.

Chosen approach:
- Keep the P0.26 local snapshot as the fastest offline restore path.
- After restoring a local snapshot, call backend history for the restored
  `session_id`.
- If backend history is available, replace the visible ready session and tick
  history with the backend record, then overwrite the local snapshot with that
  normalized state.
- If backend history is unavailable, invalid, or returns an error, keep the
  local snapshot visible and show a compact non-blocking status.

Alternatives considered:
- Backend-first restore with no local snapshot fallback: closer to server
  authority, but worse for live demos when the backend is offline on launch.
- New history screen: useful later, but it adds UI surface before the main demo
  flow actually consumes backend continuity.
- Background autonomous tick loop: more product-like, but it changes runtime
  behavior and scheduling. P0.28 should first make existing history visible and
  recoverable.

## Mobile Core

Add a small public `MythSessionID` validator:

- Accepts only `^myth_[0-9a-f]{16}$`.
- Used by the API client instead of a private inline regex.
- Gives project checks and app code a shared definition of backend-session id
  validity.

Add a `DemoRunSnapshot` conversion helper:

- `init(history:savedAt:)`
- It stores `history.session` and the bounded normalized `history.npcTicks`.
- It keeps the existing local snapshot cap of 12 ticks.

The backend may retain 24 ticks, but the local app snapshot intentionally keeps
12 to stay compact. The visible `npcTickHistory` can show the normalized local
set after sync.

## App Integration

`ForgeRootView` adds:

- `isSyncingBackendHistory`
- `backendHistoryStatusText`
- `syncBackendHistory(for:)`
- `applyBackendHistory(_:)`

Launch flow:

1. Load local `DemoRunSnapshot`.
2. Immediately show the local restored run if present.
3. If the session id matches `MythSessionID`, call
   `apiClient.getMythSessionHistory(sessionId:)` in a task.
4. On success, set `.ready(history.session)`, set `npcTickHistory` from the
   normalized snapshot, save that snapshot, and show a synced status.
5. On failure, leave the local state untouched and show a fallback status.

Forge/tick flow:

- New forge still clears old tick history and saves an empty local snapshot when
  ready.
- NPC tick still appends locally after the backend returns a tick. The backend
  already persists that tick in P0.27, so P0.28 does not add another sync call
  on every tick.

## UI

Update `DemoSnapshotStatusView` only:

- Show existing restored/local status.
- Show a second compact backend-history status line.
- Show a small progress indicator while backend history sync is running.

No new screen, modal, navigation, or marketing copy is added.

## Testing

Mobile core contract tests:

- `MythSessionID.isValid` accepts backend ids and rejects traversal/legacy ids.
- `DemoRunSnapshot(history:savedAt:)` converts backend history into a local
  bounded snapshot, retaining only matching sorted ticks.
- Existing `getMythSessionHistory` request tests remain valid.

Project checks:

- `ForgeRootView` calls `getMythSessionHistory`.
- `ForgeRootView` has `syncBackendHistory` and fallback status text.
- `DemoSnapshotStatusView` renders backend sync state and does not display local
  file URLs or provider secrets.

Visual regression:

- Add a 390x844 static evidence page showing local restore, backend sync,
  backend failure fallback, tick count, and privacy exclusions.

Full regression:

- Swift project checks.
- Swift mobile core contract tests.
- Swift app compile gate.
- Backend lint/test, because the mobile contract depends on the P0.27 backend.
- Expected deploy preflight gate until local signing/backend values exist.
- Expected Xcode license gate on this machine.

## Non-Goals

- No production account sync.
- No background autonomous NPC loop.
- No server-authoritative conflict resolution beyond the P0.27 stored history.
- No new provider-key handling in the mobile app.
- No raw media, provider secret, local file URL, or raw personal source storage.
- No print fulfillment.
- No simulator/device installation until local Apple signing/license inputs are
  provided.
