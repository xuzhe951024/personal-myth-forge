# P0.137 Mobile Showcase Action Ledger Design

## Context

P0.136 made the backend `final_showcase_readiness_report.operator_actions`
actionable. It now carries a bounded objective-level ledger with final handoff,
iOS device launch certificate, live-provider, final-resource, and final-showcase
actions.

The iOS `FinalLaunchMobileSummaryBuilder.showcaseReadinessRows` still appends
only `operatorActions.prefix(2)` after the first blocker row. That can hide the
certificate, live-provider, and resource actions the final demo operator needs
on the phone.

## Decision

Keep the existing `FinalLaunchStatusView` layout and improve the summary rows:

- Preserve the current status row and first blocker/capability row.
- Replace `readiness.operatorActions.prefix(2)` with a small action selection
  helper that returns at most six sanitized rows.
- Preserve input order, but always include key final-showcase action families
  when present:
  - `final_handoff_index:`
  - `ios_device_launch_certificate:`
  - `live-provider` / `live_provider`
  - `final-resource` / `final_resource`
  - `final-showcase` / `final_showcase`
- Deduplicate after priority selection.
- Keep all existing redaction behavior through the shared `sanitize` path.

## Alternatives Considered

1. Show all backend actions. This risks turning the compact mobile status card
   into a long console dump.
2. Keep the first two actions. This preserves compactness but fails the P0.136
   product goal because later high-value actions can remain invisible.
3. Add a new UI component for action sections. Useful later, but broader than
   this iteration because the existing grouped lines already support readable
   operator text.

## Data Flow

`FinalDemoLaunchReport.finalShowcaseReadiness.operatorActions` is decoded by
`PMFModels`. `FinalLaunchMobileSummaryBuilder` converts the report into
`showcaseReadinessRows`. `FinalLaunchStatusView` renders those rows under the
existing "Showcase Readiness" section.

P0.137 only changes the builder. The API payload shape, Swift model types, and
SwiftUI layout remain unchanged.

## Tests

- Add a Swift RED contract test with a final-showcase readiness payload that has
  early low-value actions and later handoff/certificate/live/resource/final
  actions.
- The test should prove the mobile summary includes the priority actions and
  stays bounded.
- Existing redaction test must continue to prove provider secrets and local paths
  are withheld.
- Run Swift contract tests, project checks, app compile check, backend lint, and
  backend tests before merging.

## Safety

This iteration does not call providers, write resource files, run Xcode signing,
change deployment config, touch keychains, or mutate global machine state. It is
a deterministic mobile summary rendering change.

## Self-Review

- No placeholder requirements remain.
- Scope is limited to `FinalLaunchMobileSummary` and its contract tests.
- Existing payload compatibility is preserved.
- Text output remains bounded and sanitized.
