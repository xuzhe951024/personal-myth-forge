# P0.124 Final Resource Requirements Design

## Problem

The final launch path already has several readiness reports:

- `final_resources_preflight` validates the current final resource file.
- `resource_handoff` explains backend and iOS handoff slots.
- `final_configured_preflight` composes configured launch checks.
- `final_showcase_readiness` summarizes remaining blockers.

These reports are state-oriented. The operator still lacks one compact manifest that says which resource is needed, who owns it, where it must be placed, which final capability it unblocks, whether it is secret, and which command validates it.

## Goal

Add a read-only `final_resource_requirements_report` that can be generated locally and embedded into Final Launch. It must make the last resource handoff explicit without exposing secrets or mutating local or global state.

## Non-Goals

- Do not write `.env`, `final-resources.env`, Xcode config, keychain, or signing settings.
- Do not call Meshy, OpenAI, Treatstock, Sculpteo, Apple tooling, or a real iPhone.
- Do not replace `final_resources_preflight`; the new report references it and adds operator-facing requirement metadata.
- Do not add another user-editable config format.

## Report Contract

The backend report has:

- `kind`: `final_resource_requirements_report`
- `status`: `ready`, `partial`, or `blocked`
- `summary`: counts for total, ready, missing, blocked, required, optional, secret, backend, ios, print, and validation commands
- `requirements`: ordered requirement rows
- `requirements_by_id`: sanitized lookup by id
- `operator_actions`: ordered next actions
- `validation_commands`: commands the operator can run after filling resources
- `source_reports`: compact statuses from `final_resources_preflight` and related resource reports
- `safety`: read-only and secret-redaction assertions

Each requirement row includes:

- `id`
- `label`
- `domain`: `backend_provider`, `print_provider`, `ios_deploy`, `final_launch`, or `storage`
- `destination`
- `source_template`
- `required`
- `secret`
- `configured`
- `status`
- `classification`
- `unblocks`
- `validation_command`
- `notes`

Secret values are never returned. Local absolute paths are never returned; destination and template fields are repo-relative labels.

## Requirement Set

Always required for final launch:

- `MESHY_API_KEY`: real game asset generation, backend secret
- `OPENAI_API_KEY`: AI NPC Agent runtime, backend secret
- `DEVELOPMENT_TEAM`: iOS device signing, iOS config
- `PRODUCT_BUNDLE_IDENTIFIER`: iOS device install, iOS config
- `PMF_BACKEND_BASE_URL`: iPhone-reachable backend URL

Conditional or recommended:

- `PRINT_PROVIDER`: selects `local` or `treatstock`
- `TREATSTOCK_API_KEY`: required only when `PRINT_PROVIDER=treatstock`
- `TREATSTOCK_API_BASE_URL`: optional override for Treatstock integration testing
- `OPENAI_API_BASE_URL`: optional OpenAI-compatible endpoint override
- `SCULPTEO_API_KEY`: optional backup print provider secret
- `PMF_FINAL_LAUNCH_MODE`: optional local/configured mode switch
- `CAPTURE_STORAGE_DIR`: optional capture storage override
- `MYTH_SESSION_STORAGE_DIR`: optional session storage override

## Status Rules

- `blocked` if any required requirement is missing or blocked.
- `partial` if all required requirements are usable but optional/recommended requirements remain missing or the configured print path is not real.
- `ready` if required requirements are ready and no configured conditional requirement is missing.

The report maps current state from `final_resources_preflight.items` when available. If the final resource file is missing, requirement rows still appear with missing required rows.

## iPhone Surface

`FinalDemoLaunchReport` decodes an optional `final_resource_requirements` object. `FinalLaunchMobileSummary` derives a `Resource Requirements` section with:

- overall status
- required/missing/secret summary
- first blocker detail
- first validation command
- at most a few sanitized requirement rows

The app must not render raw values, absolute paths, payment data, or provider response payloads.

## Acceptance

- Backend tests prove missing, ready, loopback, and Treatstock conditional states.
- CLI test proves `final-resource-requirements` writes a sanitized report and returns exit code 2 when blocked.
- Final Demo Launch test proves the report is embedded.
- Swift contract tests prove decode and summary rows.
- Source acceptance includes the new backend file, CLI command, Make target, iOS models, and Final Launch rows.
- Full local regression remains deterministic and makes no provider calls.
