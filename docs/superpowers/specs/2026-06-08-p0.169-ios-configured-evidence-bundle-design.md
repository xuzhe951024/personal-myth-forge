# P0.169 iOS Configured Evidence Bundle Design

## Context

P0.168 added `configured_live_evidence_bundle`, a backend read-only report that packages configured acceptance commands, evidence files, current blocker, live-call policy, and safety metadata. The report is available through:

```bash
make configured-live-evidence-bundle
```

The iPhone final launch screen already decodes and summarizes `live_provider_evidence` and `final_configured_evidence_plan`, but it does not yet receive or show the P0.168 bundle inside `final-demo-launch`. That leaves the newest configured acceptance handoff invisible in the mobile operator surface.

## Goal

Embed `configured_live_evidence_bundle` in `final_demo_launch_report`, decode it on iOS, and show concise rows in the existing Final Launch Status grouped panel.

## Non-Goals

- No new iOS screen, visual style, navigation, or interaction model.
- No live Meshy, OpenAI, Treatstock, Sculpteo, Apple, Xcode, or keychain calls.
- No writes to backend env, iOS deploy config, provider keys, payment links, or global state.
- No attempt to make configured acceptance ready without real evidence files.

## Approach

### Option A: Only Embed Backend Payload

Add the bundle to `final_demo_launch_report` and leave iOS unchanged. This helps API consumers but does not help the mobile-first final acceptance workflow.

### Option B: Recompute Bundle-Like Rows On iOS

Use existing live evidence and configured plan reports to synthesize bundle rows on device. This avoids a new model but duplicates backend logic and can drift from the canonical P0.168 bundle.

### Option C: Embed And Decode Canonical Bundle

Add `configured_live_evidence_bundle` to `final_demo_launch_report`, add Swift models for the canonical report, and render a small `Configured Evidence Bundle` group in the existing Final Launch Status page.

This is the selected approach. It keeps backend as the source of truth and follows the existing iOS pattern for live evidence, configured evidence plan, print fulfillment, and showcase readiness.

## Backend Contract

`build_final_demo_launch_report(...)` should include:

```json
"configured_live_evidence_bundle": {
  "kind": "configured_live_evidence_bundle_report",
  "status": "blocked",
  "summary": {},
  "current_blocker": {},
  "evidence_files": [],
  "command_sequence": [],
  "operator_actions": [],
  "live_call_policy": {},
  "safety": {}
}
```

The embedded report must be sanitized by the existing final-demo-launch sanitization pass. It remains read-only and must not call providers.

## iOS Contract

Add Swift models for:

- `ConfiguredLiveEvidenceBundleReport`
- `ConfiguredLiveEvidenceBundleSummary`
- `ConfiguredLiveEvidenceBundleBlocker`
- `ConfiguredLiveEvidenceBundleEvidenceFile`
- `ConfiguredLiveEvidenceBundleCommand`
- `ConfiguredLiveEvidenceBundleLiveCallPolicy`
- `ConfiguredLiveEvidenceBundleSafety`

`FinalDemoLaunchReport` gets an optional `configuredLiveEvidenceBundle`.

`FinalLaunchMobileSummary` gets `configuredEvidenceBundleRows`. The existing `FinalLaunchStatusView` renders those rows in a grouped section titled `Configured Evidence Bundle`.

Rows should include:

- Overall bundle status and evidence counts.
- Command readiness and consent counts.
- Current blocker or first non-ready command/evidence row.
- First operator action when not ready.
- Live call policy and safety flags.

All rendered text must pass through the existing sanitizer.

## Testing

Use TDD:

1. Backend test fails until `final_demo_launch_report` embeds `configured_live_evidence_bundle`.
2. Swift contract test fails until the payload decodes into `FinalDemoLaunchReport.configuredLiveEvidenceBundle`.
3. Swift summary test fails until `configuredEvidenceBundleRows` shows status, blocker, command/action, live-call policy, and safety.
4. Source acceptance test fails until mobile final launch readiness includes the new model, field, summary rows, and contract tests.
5. Run backend lint/tests, Swift contract tests, and source acceptance.

## Acceptance

This iteration is accepted when:

- `GET /v1/final-demo-launch?mode=configured` payload includes `configured_live_evidence_bundle`.
- iOS decodes that payload without losing bundle evidence files or command sequence rows.
- Final Launch Status shows the bundle using the existing grouped-line design.
- Unsafe provider secrets, local paths, payment links, and private network URLs are redacted in mobile summary text.
- Automated backend and Swift tests cover the contract.
