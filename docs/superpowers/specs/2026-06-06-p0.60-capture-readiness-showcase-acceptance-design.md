# P0.60 Capture Readiness Showcase Acceptance Design

## Purpose

P0.59 added the mobile `CaptureGenerationReadinessBuilder` and wired it into
the capture form, but quick final acceptance still reports the iOS showcase as
15 source features and does not explicitly prove that this capture-to-3D
readiness layer exists. P0.60 closes that acceptance gap.

## Scope

This is a source-only acceptance iteration. It does not change mobile runtime
behavior, call Meshy/OpenAI, run Xcode, install on an iPhone, or create provider
keys.

In scope:

- Add `capture_generation_readiness` to `ios_showcase_acceptance`.
- Require evidence from the mobile-core readiness builder, root wiring, capture
  form display, and Swift contract tests.
- Update final acceptance test fixtures from 15 to 16 nested iOS showcase
  features.
- Update docs, roadmap, and visual regression evidence.

Out of scope:

- Changing the P0.59 readiness rules.
- Adding new UI panels.
- Running real provider generation.
- Resolving local iOS deployment or Apple SDK license blockers.

## Requirements

`ios_showcase_acceptance` must require these source facts:

- `apps/mobile/ios/Sources/PersonalMythForgeMobileCore/CaptureGenerationReadiness.swift`
  contains `CaptureGenerationReadinessBuilder`, `maximumProviderSourceImages = 4`,
  `CaptureGenerationRoute`, and `displayLabel`.
- `apps/mobile/ios/App/ForgeRootView.swift` contains
  `CaptureGenerationReadinessBuilder.build` and
  `captureGenerationReadiness.route.displayLabel`.
- `apps/mobile/ios/App/CaptureFormView.swift` contains
  `generationReadinessTitle`, `generationReadinessRouteLabel`, and
  `generationReadinessDetail`.
- `apps/mobile/ios/Sources/PersonalMythForgeMobileCoreContractTests/main.swift`
  contains `testCaptureGenerationReadinessMarksGuidedScanMultiImageRoute` and
  `testCaptureGenerationReadinessMarksARKitScanAssetRoute`.

When all source evidence is present, the iOS showcase report summary must become
`passed=16, failed=0`. If the readiness builder file is missing, the report must
fail only `capture_generation_readiness`, return `passed=15, failed=1`, and not
leak absolute local paths, provider secrets, or raw media data.

## Architecture

The existing `ios_showcase_acceptance` module already models source-only
features as `FeatureRequirement` entries. P0.60 adds one new entry after
`arkit_scan_package`, because readiness describes the next decision point after
capture media exists and before the backend generation route is exercised.

`final_acceptance` remains unchanged as an orchestrator: it still treats
`ios_showcase_acceptance` as one top-level quick check. Only test fixtures that
model a passing nested iOS showcase report need their summary updated to 16.

## Testing

Use TDD:

1. Update `test_ios_showcase_acceptance_passes_complete_fixture` to expect the
   new feature id and `passed=16`.
2. Add a missing-readiness-builder failure test that expects
   `passed=15, failed=1` and sanitized output.
3. Update final acceptance passing iOS showcase stubs to report `passed=16`.
4. Run the focused backend tests until green.
5. Run backend, SwiftPM, and quick final acceptance regression gates.

## Acceptance

P0.60 is complete when:

- `cd services/backend && uv run pytest tests/test_ios_showcase_acceptance.py tests/test_final_acceptance.py -q` passes.
- `make backend-test` passes.
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks` passes.
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests` passes.
- `swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck` passes.
- Quick final acceptance still exits `2` with known external blockers and
  `failed=0`, while nested iOS showcase summary is `passed=16, failed=0`.
