# P0.130 Source Gate Linebreak Design

## Context

P0.129 added the mobile external action ledger to the Final Launch surface.
The Swift implementation formats the builder call across two lines:

```swift
externalActionLedgerRows: externalActionLedgerRows(
    from: report.finalExternalActionLedger
)
```

The real iOS showcase source acceptance gate currently expects the exact string
`externalActionLedgerRows(from:` in the same file. That makes
`run_ios_showcase_acceptance(repo_root=repo)` fail on the real checkout even
though the Swift code, Swift contract tests, and fixture-only acceptance tests
are valid.

## Decision

Keep the Swift formatting and fix the source gate. The gate should support
requirements that prove several tokens exist in the same file without requiring
them to be adjacent. For P0.130, the mobile external action ledger requirement
will require:

- `externalActionLedgerRows`
- `from: report.finalExternalActionLedger`

This proves the summary property and the actual payload handoff are both wired
while remaining insensitive to harmless line breaks.

## Alternatives Considered

1. Rewrite Swift to include `externalActionLedgerRows(from:` on one line.
   This is the smallest text change, but it makes production formatting serve a
   brittle test instead of improving the gate.

2. Replace all source checks with AST or parser checks.
   This is stronger, but too broad for the current defect. The acceptance gate
   intentionally remains a lightweight source scanner.

3. Add multi-token requirements to the existing source scanner.
   This is the selected path. It is narrow, keeps existing report shape, and can
   be covered by a focused regression test.

## Behavior

`SourceRequirement` will gain an optional `all_contains` tuple. A requirement
passes when:

- `contains` is present in the file, for the existing single-token mode, or
- every string in `all_contains` is present in the file, for the new multi-token
  mode.

Missing report entries will continue to use the existing shape with `file` and
`contains`. For multi-token requirements, `contains` will be a compact readable
description formed by joining the required tokens with ` && `. This preserves
sanitized, relative-path reports and avoids changing downstream report readers.

## Tests

The RED test will create a complete fixture whose
`FinalLaunchMobileSummary.swift` contains the real Swift linebreak style and
assert that `run_ios_showcase_acceptance` succeeds. Before implementation, this
fixture fails because the exact `externalActionLedgerRows(from:` string is
missing.

The GREEN verification will run:

```bash
cd services/backend && uv run pytest tests/test_ios_showcase_acceptance.py -q
cd services/backend && uv run python - <<'PY'
from pathlib import Path
from myth_forge_api.ios_showcase_acceptance import run_ios_showcase_acceptance
r = run_ios_showcase_acceptance(repo_root=Path('../..'))
print(r.exit_code)
print(r.report['summary'])
PY
```

The final rehearsal smoke check will run `make final-rehearsal-local`. Exit `0`
is expected after the source gate no longer fails hard; later readiness reports
may still classify device, key, or live-provider work as blocked or partial.
