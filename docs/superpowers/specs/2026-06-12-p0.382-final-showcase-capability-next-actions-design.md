# P0.382 Final Showcase Capability Next Actions Design

## Context

`final_showcase_readiness` is the operator-facing summary for the iPhone final
launch view. It already exposes a top-level `next_action` and an iOS device
action bundle, but each capability row only carries `command`,
`validation_command`, and consent flags.

That makes the iPhone Final Launch view less precise when a user scans the
remaining capabilities. The top-level blocker is clear, but secondary blocked
or partial capabilities such as provider handoff, live Meshy/OpenAI evidence,
or Treatstock quote handoff do not expose the same structured "do this, then
rerun this validation" shape.

## Decision

Add an optional `next_action` field to non-ready final showcase capability rows.

- Backend derives the field in `_capability()` so all capability builders share
  the behavior.
- The derived action mirrors the capability `id`, `label`, `status`,
  `classification`, `command`, and `detail`.
- It includes `source: capability`, plus `validation_command`,
  `requires_cost_consent`, `requires_live_provider_consent`, and
  `completion_requires_live_provider_consent` when those fields exist.
- Ready capabilities keep `next_action: null` by omission.
- iOS decodes the optional nested action using the existing
  `FinalShowcaseReadinessNextAction` model and renders it inside capability
  rows.

## Data Flow

```text
final showcase child evidence
  -> final_showcase_readiness.capabilities[].next_action
  -> FinalDemoLaunchReport.finalShowcaseReadiness
  -> FinalLaunchMobileSummary.showcaseReadinessRows
```

## Safety

This phase only restructures already sanitized readiness data. It does not run
provider calls, write secrets, start Xcode, sign an app, mutate Keychain, or
place print orders.

## Testing

Backend tests verify blocked, partial, consent-gated, and ready capability rows.
Swift contract tests verify decode and rendering in the Final Launch mobile
summary.
