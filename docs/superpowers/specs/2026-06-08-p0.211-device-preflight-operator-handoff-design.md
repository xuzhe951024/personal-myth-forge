# P0.211 Device Preflight Final Operator Handoff Design

## Purpose

`FinalDemoLaunchReport` already embeds `final_operator_handoff`, and the iPhone Final Launch panel renders its `Next` rows. The iPhone Device Preflight panel still does not show the final operator handoff state, so an operator can see launch evidence and closure readiness without seeing the concrete next Mac-side action in the compact preflight checklist.

## Scope

Add a read-only Device Preflight item with id `final_operator_handoff` and label `Operator Handoff`. Place it after `Final Closure` and add it to the required-ready set. The item reads only `FinalOperatorHandoffReport` already decoded from the final launch report; it does not execute commands, call providers, run Xcode, write config, or mutate global state.

## Behavior

- Missing final launch report or missing operator handoff returns `waiting` with `Final operator handoff has not loaded.`
- `status == "ready"` returns `ready` and shows ready/missing/blocked/manual/live counts plus safety flags.
- `status == "blocked"`, `partial`, `missing`, or `live` returns `blocked` except `missing` returns `waiting`; details include summary, first next action, first non-ready step, consent marker when required, and safety flags.
- Unknown statuses return `waiting` with summary and safety details.
- All output passes through the existing Device Preflight sanitizer.

## Acceptance

- Swift contract tests cover missing, blocked, ready, live-consent, and redaction behavior.
- Backend mobile final launch readiness source requirements check the new item, label, source field, required-ready set entry, and Swift contract tests.
- `PersonalMythForgeMobileProjectChecks` checks the same source-level contract.
- Existing final rehearsal and app compile gates continue to pass.
