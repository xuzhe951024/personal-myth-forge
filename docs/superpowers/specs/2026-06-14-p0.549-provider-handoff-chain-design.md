# P0.549 Provider Handoff Chain Design

## Goal

Make the direct provider handoff report point operators to the same local resource handoff chain used by final readiness reports.

## Current Gap

`final-showcase-readiness.json` already tells the operator to run:

```text
make final-resource-fill-guide; rerun make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence
```

But `provider-handoff.json` still reports the default local-stub provider blocker with:

```text
make final-resource-apply-preview
```

That skips the fill guide in the primary blocker, even though the blocker detail says to fill ignored final resources first. The operator action list does include the fill guide, but the `first_blocker` and `next_action` should be equally executable when someone opens the provider handoff report directly.

## Intended Behavior

- For local-stub core providers such as `three_d` and `npc`, `first_blocker.command` uses:

```text
make final-resource-fill-guide; rerun make final-resource-apply-preview; rerun make provider-handoff
```

- `next_action.command` mirrors the blocker command.
- `live-provider-evidence` appends its own validation command without duplicating an already-present `rerun make provider-handoff`.
- Missing explicit env values still use the more specific `provide <KEY> in final-resources.env` command.
- `apply_preview_command`, `apply_command`, `handoff_command`, safety flags, and no-provider-call behavior remain unchanged.

## Acceptance

- `test_provider_handoff_blocks_default_local_core_provider_with_next_action` fails before implementation because it expects the blocker command to start with the fill guide chain.
- The same test passes after implementation and confirms `next_action.command` matches.
- `test_live_provider_evidence_projects_provider_handoff_child_next_action` confirms the full provider handoff chain is not duplicated before `rerun make live-provider-evidence`.
- Focused provider/live/final readiness tests and full backend tests pass.
