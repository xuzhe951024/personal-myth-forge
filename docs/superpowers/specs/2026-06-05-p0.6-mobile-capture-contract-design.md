# P0.6 Mobile Capture Contract Design

## Purpose

P0.6 turns the mobile object-capture part of the final iOS MVP into a real backend contract. P0.5 can generate a myth session from typed object metadata, but the iOS-first product needs a phone-native path where a client can submit object imagery or future scan artifacts, receive a capture manifest, and create a myth session from that capture without exposing raw files in session responses.

This is still not the final Unity/iOS app. It is the bridge that lets an iOS Unity client, a future Swift capture wrapper, or the current web demo all exercise the same capture-to-myth flow.

## Current State

- `POST /v1/myth-sessions` accepts `ObjectObservation` JSON and returns a complete local or provider-backed myth session.
- `/demo` is mobile-first, but it only submits text fields.
- `apps/mobile/README.md` says Unity owns capture/upload but no API contract or client fixture exists yet.
- Xcode is available locally. Unity CLI is not available in this environment, so P0.6 must not require a Unity build step to pass.

## Non-Goals

- Real ARKit reconstruction, LiDAR meshing, or Apple Object Capture photogrammetry.
- A committed Unity project generated without Unity installed.
- Cloud object storage, signed URLs, CDN delivery, or permanent media retention.
- Vision model object recognition. P0.6 accepts user/client-provided capture metadata and leaves AI object understanding for a later provider-backed iteration.
- Reading personal raw data. Context remains the approved `ContextCapsule` summary only.

## Approaches Considered

### A. Metadata-Only Capture

Only add `capture_id` fields to `ObjectObservation`, without uploading files.

This is simple but does not move the product toward real phone capture. It would still leave the mobile client without a tested upload path.

### B. Multipart Capture Manifest

Add `POST /v1/object-captures` as multipart form data: one `metadata_json` field plus one or more media files. The backend writes files to a local ignored directory and returns a bounded manifest with file roles, sizes, content types, and local URIs.

This is the chosen approach because it is realistic for iOS/Unity, works locally without external keys, and can be tested with FastAPI `TestClient`.

### C. Direct-To-Cloud Upload

Issue pre-signed upload URLs and have mobile upload directly to object storage.

This is the right production shape later, but it would add storage provider setup before the project has a proven iOS capture contract. P0.6 should keep provider interfaces open without depending on cloud credentials.

## Chosen Scope

P0.6 adds:

- `ObjectCapture` domain models.
- `CaptureStore` provider boundary.
- `LocalCaptureStore` writing to `services/backend/.local/captures` by default.
- `POST /v1/object-captures` for multipart upload.
- `GET /v1/object-captures/{capture_id}` for manifest lookup.
- `POST /v1/myth-sessions/from-capture` to create a myth session from a stored capture and a `ContextCapsule`.
- `/demo` support for optional image upload using the same capture contract.
- Mobile contract docs under `apps/mobile/`.

Existing `POST /v1/myth-sessions` remains intact for typed/manual sessions.

## API Contract

### `POST /v1/object-captures`

Request is `multipart/form-data`:

- `metadata_json`: JSON object.
- `files`: one or more uploaded files.

Metadata shape:

```json
{
  "label": "old brass key",
  "materials": ["metal", "brass"],
  "source": "phone_capture",
  "capture_mode": "single_photo",
  "visual_notes": "worn teeth, circular bow, warm reflections"
}
```

Allowed `capture_mode` values:

- `single_photo`
- `photo_set`
- `manual_upload`
- `arkit_scan`

File constraints for local P0.6:

- At least 1 file.
- At most 12 files.
- Each file must be at most 10 MB.
- Accepted content types: `image/jpeg`, `image/png`, `image/heic`, `image/heif`, `model/gltf-binary`, `model/vnd.usdz+zip`, `application/octet-stream`.
- `application/octet-stream` is accepted only for future scan artifacts and is marked `needs_review`.

Response shape:

```json
{
  "capture_id": "cap_0123456789abcdef",
  "status": "ready",
  "source": "phone_capture",
  "capture_mode": "single_photo",
  "object_observation": {
    "label": "old brass key",
    "materials": ["metal", "brass"],
    "source": "phone_capture",
    "visual_notes": "worn teeth, circular bow, warm reflections"
  },
  "media_items": [
    {
      "media_id": "media_0",
      "role": "reference_image",
      "content_type": "image/jpeg",
      "byte_size": 1240,
      "uri": "local-capture://cap_0123456789abcdef/media_0.jpg",
      "moderation_status": "needs_review"
    }
  ],
  "created_at": "2026-06-05T00:00:00Z"
}
```

The response never returns raw file bytes, base64 data, local absolute paths, or request headers.

### `GET /v1/object-captures/{capture_id}`

Returns the same `ObjectCapture` manifest if present.

Missing captures return `404`.

### `POST /v1/myth-sessions/from-capture`

Request:

```json
{
  "capture_id": "cap_0123456789abcdef",
  "context_capsule": {
    "current_theme": "deadline pressure",
    "desired_tone": "tender, strange",
    "recent_milestone": "finished a difficult project draft"
  }
}
```

The backend loads the capture manifest, enriches the `ObjectObservation` with `capture_id` and media URI references, then calls the existing myth pipeline. The response is the existing `MythSession` shape.

## Storage Design

`CaptureStore` is an internal provider boundary with:

- `save_capture(metadata, files) -> ObjectCapture`
- `get_capture(capture_id) -> ObjectCapture | None`

`LocalCaptureStore` writes to `CAPTURE_STORAGE_DIR`, defaulting to `services/backend/.local/captures`. This directory must be ignored by git. Stored filenames are generated from capture id and media id, never from raw user filenames.

The store writes a `manifest.json` next to media files so local sessions can survive process restarts during demo development.

Future provider slots:

- `OBJECT_STORAGE_BUCKET`
- `OBJECT_STORAGE_ENDPOINT`
- `OBJECT_STORAGE_ACCESS_KEY`
- `OBJECT_STORAGE_SECRET_KEY`

These already exist in `.env.example`; P0.6 should not require them.

## Demo Flow

The `/demo` form gets an optional object image input. If a file is selected:

1. Submit `POST /v1/object-captures`.
2. Render capture status and media count.
3. Submit `POST /v1/myth-sessions/from-capture`.
4. Render the same artifact, NPC, world, and print sections as P0.5.

If no file is selected, the demo keeps the current direct `POST /v1/myth-sessions` path.

Browser visual regression must cover:

- 390x844 mobile: no horizontal overflow, file input visible, capture status visible after submit.
- 1280x720 desktop: no horizontal overflow, two-column layout remains readable, capture status visible after submit.
- Console errors are empty.

## Mobile Client Contract

P0.6 adds mobile-facing docs and fixtures:

- `apps/mobile/contracts/capture-api.md`
- `apps/mobile/fixtures/object-capture-metadata.json`
- `apps/mobile/fixtures/myth-session-from-capture.json`

The intended iOS/Unity flow:

1. Unity or a Swift wrapper captures one image or a future scan bundle.
2. Client posts files and metadata to `POST /v1/object-captures`.
3. Client posts `capture_id` plus approved `ContextCapsule` to `POST /v1/myth-sessions/from-capture`.
4. Unity loads `generated_asset.uri` if remote GLB; local stub remains a placeholder.
5. Unity maps `world_resolution` and `npc_reactions` to staging, text, and animation states.

## Error Handling

- Missing upload file: `422`.
- Too many files: `413`.
- Oversized file: `413`.
- Unsupported content type: `415`.
- Missing capture id on myth-from-capture: `404`.
- Provider errors from 3D or NPC generation remain sanitized HTTP `502`.

Error bodies must not include raw file bytes, absolute storage paths, API keys, authorization headers, or personal raw data beyond approved request metadata.

## Testing And Regression

Required automated coverage:

- `ObjectCapture` models serialize to JSON without bytes or local absolute paths.
- `LocalCaptureStore` writes generated filenames and persists `manifest.json`.
- Upload endpoint accepts a small JPEG/PNG fixture and returns an `ObjectCapture`.
- Upload endpoint rejects too many files, oversized files, and unsupported content types.
- `GET /v1/object-captures/{capture_id}` returns the stored manifest.
- `POST /v1/myth-sessions/from-capture` returns a `MythSession` and includes capture references in `object_card` or observation-derived fields.
- Existing direct myth session endpoint remains unchanged.
- Demo HTML includes upload and capture status mount points.
- Browser visual regression evidence is refreshed after demo changes.

Existing verification remains:

- `make backend-lint`
- `make backend-test`
- local CLI `evaluate-3d`

## Acceptance Criteria

- A local backend can receive at least one uploaded object image and return a capture manifest.
- A myth session can be generated from `capture_id` without external API keys.
- The current browser demo can exercise both direct metadata and file-backed capture flows.
- Mobile docs provide enough concrete request/response examples for a Unity/iOS client implementation.
- No global machine state is changed. Local media is stored under an ignored project-local directory.
- All tests pass and visual regression evidence is committed.
