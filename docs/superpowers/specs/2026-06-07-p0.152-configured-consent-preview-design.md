# P0.152 Configured Consent Preview Design

## Problem

The configured evidence plan already marks three steps as live-provider and
cost-risk steps: Meshy 3D evaluation, OpenAI NPC evaluation, and configured final
acceptance. When resource inputs are still blocked, however, every downstream
step is also blocked, so the existing `summary.consent_required` count is `0`.
That field is a current status count, not a preview of future consent needs.

This makes the iPhone Final Launch panel understate the final operator work: it
shows live/cost steps but no consent count until resources are ready.

## Goal

Expose a stable planned-consent count for configured evidence. The backend
should report how many steps will need live-provider consent once upstream
configuration is unblocked, and the iPhone summary should render that count
alongside the current consent-required status count.

## Non-Goals

- Do not run Meshy, OpenAI, Treatstock, or any live provider.
- Do not execute configured evidence commands.
- Do not write backend `.env` or iOS deploy config files.
- Do not change the meaning of `summary.consent_required`; it remains the number
  of steps currently in `consent_required` status.
- Do not add a new UI panel; extend the existing `Configured Evidence` rows.

## Backend Design

Add `planned_consent_steps` to the `final_configured_evidence_plan.summary`
object. It counts all configured evidence steps where
`requires_live_provider_consent` is true and the step is not already `ready`.

This means:

- Missing resource bundle: `consent_required == 0`, `planned_consent_steps == 3`
- Ready resources without live-call consent: `consent_required == 3`,
  `planned_consent_steps == 3`
- Ready resources with `--allow-live-provider-calls`: `consent_required == 0`,
  `planned_consent_steps == 3`
- Saved configured evidence ready: `planned_consent_steps == 0`

The safety block remains unchanged because no commands or provider calls run.

## Mobile Design

Add `plannedConsentSteps` to `FinalConfiguredEvidencePlanSummary`.

Update `FinalLaunchMobileSummaryBuilder.configuredEvidencePlanRows(from:)` so the
primary row says:

`Configured evidence blocked: steps 10, ready 0, blocked 10, consent now 0, planned 3.`

The existing live/cost/repo-write row and consent flag row stay in place. All
text still passes through existing redaction.

## Testing

Use TDD:

- Backend red test proves missing resources still expose planned consent steps.
- Backend green implementation adds `planned_consent_steps`.
- Swift red test proves payload decode and summary text require
  `plannedConsentSteps`.
- Swift green implementation updates model and summary rows.
- Source gates require the new model field and visible summary copy.
