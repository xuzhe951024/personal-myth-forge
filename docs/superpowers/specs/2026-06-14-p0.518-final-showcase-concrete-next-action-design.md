# P0.518 Final Showcase Concrete Next Action Design

## Context

The final iOS showcase remains blocked by external device setup: the operator must provide an Apple `DEVELOPMENT_TEAM`, use an iPhone-reachable `PMF_BACKEND_BASE_URL`, then rerun mobile deploy preflight. Recent phases made the final handoff index and iOS device rehearsal expose a complete copy-pasteable command chain:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

The top-level final showcase readiness report still exposes the writer command and the validation command as separate fields. That is structured, but less direct than the downstream handoff artifacts and easier to misread during final device rehearsal.

## Goal

Promote the final showcase readiness top-level iOS deploy blocker to the same concrete command chain used by downstream handoff reports, while keeping `validation_command` as a machine-readable field.

## Non-Goals

- Do not make iOS deploy readiness pass without real external evidence.
- Do not write real signing data, Keychain entries, provider keys, or local user resources.
- Do not call Meshy, OpenAI, Treatstock, Xcode build, or any paid/provider action.
- Do not change provider handoff semantics.

## Behavior

When `final-showcase-readiness` is blocked by `ios_deployable` and the preferred mobile preflight child action has both:

- `command = DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto`
- `validation_command = make mobile-deploy-preflight`

then both `first_blocker.command` and `next_action.command` should be:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

`validation_command` remains:

```text
make mobile-deploy-preflight
```

If the child action command already includes the validation suffix, the report must not duplicate it. Downstream consumers such as `final-demo-launch-local` and `final-local-report-refresh` should preserve the concrete command chain without appending a second identical rerun suffix.

## Testing

Add a focused regression test in `services/backend/tests/test_final_showcase_readiness.py` that builds a blocked mobile deploy preflight evidence fixture with a child next action. The test must fail before implementation because the current top-level command lacks the `; rerun make mobile-deploy-preflight` suffix.

Verification commands:

```bash
cd services/backend && uv run pytest tests/test_final_showcase_readiness.py::test_final_showcase_readiness_top_level_ios_blocker_promotes_validation_aware_command -q
cd services/backend && uv run pytest tests/test_final_showcase_readiness.py -q
cd services/backend && uv run pytest tests/test_final_demo_launch.py tests/test_final_local_report_refresh.py -q
make backend-lint
make backend-test
make final-showcase-readiness
```

The refreshed `.local/final-showcase-readiness.json` should remain `blocked`, but its top-level first/next command should be the complete validation-aware command.
