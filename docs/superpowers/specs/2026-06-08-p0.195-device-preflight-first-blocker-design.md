# P0.195 Device Preflight First Blocker Design

## Purpose

The final launch receipt now uses backend `final_demo_launch_report.first_blocker`, but the iOS device preflight panel still derives its final launch detail from `operatorChecklist` or launch phases. That can make the first deployment screen disagree with the canonical backend blocker. This iteration makes Device Preflight prefer the same top-level blocker.

## Scope

- Update `DevicePreflightSummaryBuilder.finalLaunchDetail` to prefer `FinalDemoLaunchReport.firstBlocker`.
- Keep current fallback behavior for older backend payloads without `firstBlocker`.
- Add a Swift contract test that builds a blocked final launch report with top-level `first_blocker` and asserts the preflight final launch row shows that blocker.
- Add ProjectChecks coverage so this preference remains source-gated.

## Acceptance

- Device Preflight final launch row includes blocker id, status/classification, command, and detail when `report.firstBlocker` exists.
- Existing redaction still applies through DevicePreflight item sanitization.
- Existing fallback tests continue to pass for payloads without `firstBlocker`.
