# P0.180 Mobile App Compile Check Design

## Context

`PersonalMythForgeMobileAppCompileCheck` is intended to be a fast automated
gate that proves the SwiftUI app sources still compile against
`PersonalMythForgeMobileCore`.

The current SwiftPM target points at `apps/mobile/ios/App` and includes
`PersonalMythForgeApp.swift`. That file contains the real SwiftUI `@main` app
entry point:

```swift
@main
struct PersonalMythForgeApp: App {
    var body: some Scene {
        WindowGroup {
            ForgeRootView()
        }
    }
}
```

`swift build --package-path apps/mobile/ios --product
PersonalMythForgeMobileAppCompileCheck` still compiles and exits, but
`swift run --package-path apps/mobile/ios
PersonalMythForgeMobileAppCompileCheck` launches the SwiftUI app executable and
does not naturally exit. That makes the target unsafe as a run-style acceptance
gate.

## Goal

Make `swift run --package-path apps/mobile/ios
PersonalMythForgeMobileAppCompileCheck` compile the app source and exit
successfully without launching the real SwiftUI app runtime.

## Design

Keep the SwiftPM target pointed at `apps/mobile/ios/App` so it continues to
compile the app views. Exclude `PersonalMythForgeApp.swift` from the compile
check target so the real app `@main` is not used by SwiftPM.

Add `apps/mobile/ios/App/AppCompileCheckMain.swift` with a compile-check-only
entry point guarded by `#if PMF_APP_COMPILE_CHECK`. The SwiftPM target defines
that flag with `.define("PMF_APP_COMPILE_CHECK")`. The real Xcode app target
does not define that flag, so the file contributes no `@main` to the real iOS
application.

The compile-check entry point references `ForgeRootView.self` and prints a
stable success line:

```text
PersonalMythForgeMobileAppCompileCheck passed
```

This gives automation a command that both compiles the SwiftUI app surface and
terminates predictably.

## Project Gate

Update `PersonalMythForgeMobileProjectChecks` to require:

- `AppCompileCheckMain.swift` exists.
- the compile-check entry prints `PersonalMythForgeMobileAppCompileCheck passed`.
- the compile-check entry references `ForgeRootView.self`.
- `Package.swift` excludes `PersonalMythForgeApp.swift` from the SwiftPM
  compile-check target.
- `Package.swift` defines `PMF_APP_COMPILE_CHECK`.

These checks catch the regression class before anyone waits on a hanging
`swift run`.

## Alternatives Considered

1. Keep using only `swift build`.
   Rejected because this target is an executable target and existing run-style
   checks in the repository make `swift run` a natural acceptance command.

2. Move app views into a separate SwiftPM library target.
   Rejected for this slice because it would be a larger app structure change.
   The current target can stay source-compatible by using a compile-check-only
   entry point.

3. Add a shell timeout wrapper around `swift run`.
   Rejected because the root cause is the wrong executable entry point. A
   timeout would hide the issue and make success depend on timing.

## Acceptance

- `swift run --package-path apps/mobile/ios
  PersonalMythForgeMobileProjectChecks` exits `0`.
- Before the implementation, ProjectChecks fails on the missing compile-check
  entry contract.
- `swift run --package-path apps/mobile/ios
  PersonalMythForgeMobileAppCompileCheck` exits `0` and prints
  `PersonalMythForgeMobileAppCompileCheck passed`.
- `swift run --package-path apps/mobile/ios
  PersonalMythForgeMobileCoreContractTests` exits `0`.
- `make backend-lint` passes.
- `make backend-test` passes with only the known FastAPI/Starlette `TestClient`
  deprecation warning.
- No provider calls, key reads, Xcode signing, keychain mutations, resource
  writers, backend runtime behavior, or mobile UI behavior changes.

## Test Strategy

Use TDD:

1. Add static ProjectChecks requirements for the compile-check-only entry point
   and SwiftPM target settings.
2. Run ProjectChecks and watch it fail because the entry file and package
   settings do not exist yet.
3. Add the compile-check-only entry and package target settings.
4. Run ProjectChecks again and watch it pass.
5. Run the repaired `swift run ... PersonalMythForgeMobileAppCompileCheck`
   command and verify it exits.
6. Run Swift core contract tests, backend lint, backend tests, and repeat key
   verification after merging to `main`.

## Root Cause

The SwiftPM executable target reused the real app source directory without a
separate test/check entry point. As a result, `swift run` compiled successfully
and then executed the real SwiftUI app `@main`, which is correct for an app but
incorrect for an automated compile gate.

- Scope check: one SwiftPM target, one compile-check entry file, and one static
