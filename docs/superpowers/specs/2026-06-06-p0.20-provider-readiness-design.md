# P0.20 Provider Readiness Design

## Goal

Add a safe provider readiness layer that tells the mobile demo and operator
which external integrations are demo-ready, real-provider-ready, missing
configuration, or still local-only. This prepares the final key/API handoff
without putting secrets in the mobile app or returning secret values from the
backend.

## Product Behavior

P0.20 introduces `GET /v1/provider-readiness`. The response summarizes the 3D
provider, NPC provider, print provider, and capture storage path. Each row
reports the selected provider, readiness status, demo readiness, real-provider
readiness, missing environment variable names, capabilities, and operational
notes.

Local providers count as demo-ready but not real-provider-ready. Meshy becomes
real-provider-ready only when `THREE_D_PROVIDER=meshy` and `MESHY_API_KEY` is
present. OpenAI NPC becomes real-provider-ready only when `NPC_PROVIDER=openai`
and `OPENAI_API_KEY` is present. Print fulfillment remains a local stub in this
iteration and is explicitly reported as not real-provider-ready.

## Backend Contract

New models:

- `ProviderReadinessItem`
- `ProviderReadinessResponse`

Response shape:

```json
{
  "overall_demo_ready": true,
  "overall_real_ready": false,
  "providers": [
    {
      "kind": "three_d",
      "selected_provider": "local",
      "status": "local_stub",
      "is_demo_ready": true,
      "is_real_provider_ready": false,
      "missing_env": [],
      "capabilities": ["text_to_3d_stub", "asset_variant_stub"],
      "notes": ["Use THREE_D_PROVIDER=meshy and MESHY_API_KEY for real 3D generation."]
    }
  ]
}
```

The endpoint must never include raw key values, Authorization headers, local
absolute capture paths, or environment dumps. Missing configuration is reported
only by environment variable name.

## Mobile Contract

The Swift core adds matching Codable models and a
`PersonalMythForgeAPIClient.getProviderReadiness()` method. `ForgeRootView`
loads readiness on first display and renders a compact `ProviderReadinessView`
above the capture form. If the backend is unavailable, the app shows a short
non-blocking status and the forge flow remains usable once the backend returns.

## Testing

Backend tests cover:

- default local readiness
- Meshy selected without key returns `missing_configuration` with
  `MESHY_API_KEY`
- Meshy selected with key returns real-provider-ready without leaking the key
- OpenAI NPC selected without key returns `OPENAI_API_KEY`
- endpoint response contains no secret values

Swift contract tests cover:

- decoding readiness responses
- API client sends `GET /v1/provider-readiness`
- app source/project checks include the readiness view and non-secret display

Static evidence mirrors earlier P0 evidence: an HTML page and a `390x844`
screenshot showing the readiness strip and key handoff boundary.

## Non-Goals

P0.20 does not create provider keys, store secrets, call Meshy/OpenAI live,
implement Treatstock/Sculpteo fulfillment, bypass the Apple SDK license gate, or
claim simulator/device deployment.
