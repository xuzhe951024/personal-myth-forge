# P0.131 iOS Rehearsal Action Routing Design

## Context

`make ios-device-launch-rehearsal` is the safe local command that refreshes the
final iPhone launch evidence set. It runs no live providers, no Xcode build, no
signing, and no global machine mutation. After it runs, the generated
`ios_device_launch_rehearsal_report` can still be blocked because the operator
has not supplied iOS deploy config, API keys, live provider consent, or manual
device/Xcode steps.

The current report always starts `operator_actions` with
`run make ios-device-launch-rehearsal`. That is correct when the report is
missing or stale, but misleading immediately after a successful rehearsal
refresh. In that state, repeating the same command does not resolve the blocker.
The operator needs the next concrete blocked report command or nested action.

## Decision

Make `operator_actions` state-aware:

- If a required report is missing, keep a top-level action to run
  `make ios-device-launch-rehearsal` and refresh the missing report command.
- If reports exist but are blocked, do not put
  `run make ios-device-launch-rehearsal` first. Instead, list actionable
  blockers from the blocked reports in sequence order.
- Prefer nested `operator_actions` from a blocked report when they are present,
  because they usually contain the concrete command such as
  `make mobile-write-deploy-config` or `make final-resource-apply-preview`.
- Fall back to `review <step id>: <step command>` when a blocked report has no
  nested actions.
- Keep the existing ready/partial path that continues with
  `make backend-device-demo` and `make mobile-deploy-preflight`.

## Alternatives Considered

1. Add `make ios-device-launch-rehearsal` to every final readiness command.
   This refreshes more files but does not fix the misleading self-referential
   action in the report.

2. Change the iPhone UI to hide the first action.
   That would mask the symptom only on mobile and leave the backend handoff JSON
   misleading for CLI operators.

3. Make the backend report generate specific next actions.
   This is selected because the backend owns the handoff report and every
   consumer benefits from a more accurate contract.

## Data Flow

`build_ios_device_launch_rehearsal_report` already reads the saved local
rehearsal set, configured preflight, final handoff index, and iOS device launch
certificate. P0.131 will enrich the compact source report with bounded nested
`operator_actions` from each saved report, then route the top-level
`operator_actions` through those nested actions.

The report remains sanitized:

- no absolute local paths
- no provider secrets
- no raw media
- no checkout-specific temporary directories

## Tests

The RED test will create ready local evidence plus a blocked
`final-configured-preflight.json` containing nested `operator_actions`. It will
assert that the generated `operator_actions` does not start with
`run make ios-device-launch-rehearsal` and does include the nested concrete
action. Before implementation, the first assertion fails.

The GREEN checks will run:

```bash
cd services/backend && uv run pytest tests/test_ios_device_launch_rehearsal.py -q
make ios-device-launch-rehearsal
make final-demo-launch
```

`make ios-device-launch-rehearsal` should still exit `0`, with inner blocked
reports accepted as expected. The generated final demo launch report should
continue embedding the rehearsal readiness for mobile display.
