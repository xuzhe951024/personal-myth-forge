# P0.185 Mobile Configured Launch Payload Parity Design

## Goal

The Swift mobile contract tests must prove that a configured final demo launch payload shaped like the backend report carries `make final-acceptance-configured` through decoding and into the iPhone Final Launch summary.

## Current Behavior

Backend `build_final_demo_launch_report(mode="configured")` already includes `make final-acceptance-configured` in the top-level `commands` list. P0.184 updated `FinalLaunchMobileSummaryBuilder` so the command is prioritized if it is present, but the Swift configured payload fixture still defaults to the local one-command list unless a test manually injects commands.

That means mobile tests cover the helper behavior, but they do not cover realistic backend configured command ordering.

## Scope

- Add a Swift contract test that decodes `finalDemoLaunchPayload(mode: "configured")`.
- Make the Swift fixture emit a realistic configured command list by default.
- Assert the decoded report includes `make final-acceptance-configured` after the early setup commands.
- Assert `FinalLaunchMobileSummaryBuilder` promotes that decoded configured command to the first visible command row.
- Preserve the existing local payload default command list.
- Do not change backend runtime behavior or execute live provider commands.

## Acceptance

- A red Swift contract test fails before the fixture parity change because configured payload defaults to `["make backend-device-demo"]`.
- After the fixture update, Swift core contract tests pass.
- ProjectChecks contain a source gate for the configured payload parity test.
- Backend tests continue to pass because backend behavior is not changed.
