# P0.18 Xcode iOS Build Gate Design

## Problem

P0.13 added a SwiftPM compile gate for the SwiftUI app source. That catches many
source-level regressions, but it still does not prove that the checked-in Xcode
project can build the iOS application target. The v0.1 demo is iOS-first, so the
next deployment-readiness step is a repeatable Xcode build command that uses the
real `PersonalMythForge.xcodeproj` without changing global Xcode settings.

The current machine has `/Applications/Xcode.app`, but `xcodebuild -list` is
blocked by the Apple SDK license agreement until the user accepts it in Terminal.
P0.18 should therefore add the build gate and make license-gated failure
actionable, while not pretending that a blocked local build succeeded.

## Design

Add a repository script at `apps/mobile/ios/scripts/xcode_build_gate.sh`. The
script resolves Xcode from `$DEVELOPER_DIR` when provided, otherwise from
`/Applications/Xcode.app/Contents/Developer`. It runs `xcodebuild` with:

- the checked-in `apps/mobile/ios/PersonalMythForge.xcodeproj`
- the `PersonalMythForge` scheme
- `generic/platform=iOS`
- a repo-local derived data path under `apps/mobile/ios/.build/xcode-derived-data`
- `CODE_SIGNING_ALLOWED=NO`
- `CODE_SIGNING_REQUIRED=NO`

The script must never run `xcode-select`, mutate global Xcode selection, open a
simulator, install to a device, archive, or require signing credentials.

Add a shared scheme at
`apps/mobile/ios/PersonalMythForge.xcodeproj/xcshareddata/xcschemes/PersonalMythForge.xcscheme`
so CLI `xcodebuild` has a stable scheme name independent of a developer's local
Xcode user data.

Extend `PersonalMythForgeMobileProjectChecks` to enforce the script, shared
scheme, build destination, local derived data path, and disabled signing flags.
This gives us deterministic source-level verification even when local Xcode is
blocked by license acceptance.

Add `make mobile-xcode-build` as the human-facing command. It should call the
script directly and inherit `DEVELOPER_DIR` if the user sets it.

## Error Handling

If Xcode is missing, the script exits non-zero with a clear message telling the
user to install Xcode or set `DEVELOPER_DIR`.

If `xcodebuild` is present but blocked by license acceptance, the script exits
non-zero with the original `xcodebuild` message intact. Documentation and
evidence should state that the command becomes the real acceptance gate after
the user runs `sudo xcodebuild -license` in Terminal.

## Verification

P0.18 source and regression verification:

```bash
swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks
swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests
swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck
make backend-lint
make backend-test
git diff --check HEAD
```

P0.18 Xcode gate command:

```bash
make mobile-xcode-build
```

On the current machine before license acceptance, `make mobile-xcode-build` is
expected to fail with the Apple SDK license message. After the user accepts the
license, the same command should perform the real unsigned generic iOS build.

## Non-Goals

- Do not run `xcode-select` or change global developer directory state.
- Do not accept the Apple SDK license on the user's behalf.
- Do not build for a named simulator, launch a simulator, install to a device,
  archive, notarize, sign, or upload to TestFlight.
- Do not add ARKit runtime capture or Unity integration in this iteration.
- Do not move provider API keys into the mobile app.
