# P0.199 Final Launch Resource Blocker Hint Design

## Purpose

`final_demo_launch_report.first_blocker` correctly identifies the blocked `apply_final_resources` phase, but its detail only says to run the apply command. P0.198 added a structured `final_resource_requirements.first_blocker`, which contains the actual missing resource action such as `provide MESHY_API_KEY in final-resources.env`. The iPhone final launch rows should show that actionable hint without changing the phase-level blocker identity.

## Scope

- Keep `final_demo_launch_report.first_blocker.id == "apply_final_resources"` for the blocked phase.
- Include the first nested resource blocker command/detail in the phase blocker detail.
- Add `final_resource_requirements` and apply/fill-guide reports to the nested blocker scan before other final launch readiness reports.
- Preserve existing nested fallback behavior when no failed/blocked/missing phase exists.
- Add backend tests and source gates.

## Acceptance

- Default configured final launch first blocker detail contains `provide MESHY_API_KEY in final-resources.env`.
- The blocker still has source `final_demo_launch_phase` and command `make final-apply-resources`.
- Nested fallback test for ready phase blockers still resolves to `final_showcase_readiness`.
- No secret values or local absolute paths leak.
