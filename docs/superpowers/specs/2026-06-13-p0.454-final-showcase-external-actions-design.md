# P0.454 Final Showcase External Actions Design

## Context

The iPhone Final Launch panel and Device Preflight panel now both surface the final external action ledger, including a hidden `backend-device-demo` handoff. The top-level Final Showcase Summary still has no stage for this ledger.

That creates a visibility gap: the first summary a demo operator sees can show capture, 3D, NPC, print, resources, provider handoff, local smoke, iOS deploy, evaluations, operator handoff, and launch state, while omitting the consolidated external action list for provider keys, Treatstock cost consent, Xcode/global actions, and backend-device-demo.

## Goal

Add a compact `External Actions` stage to `FinalShowcaseSummaryBuilder` whenever `FinalLaunchMobileSummary.externalActionLedgerRows` is available.

## Approaches Considered

1. Add a dedicated `external_actions` stage after `operator_handoff`.
   - Pro: keeps the top-level summary scannable and mirrors Device Preflight/Final Launch terminology.
   - Con: adds one more stage to the already dense summary.

2. Fold external action text into the existing `operator_handoff` stage.
   - Pro: no new stage.
   - Con: operator handoff and external ledger have different meanings; combining them can hide cost/global/device blockers.

3. Fold external action text into `final_launch`.
   - Pro: fewer rows.
   - Con: final launch already prioritizes first blocker receipts and should stay concise.

Recommended approach: option 1. The external ledger is a first-class readiness artifact and should be visible as a first-class top-level stage.

## Design

`FinalShowcaseSummaryBuilder.build(...)` will append `externalActionStage(finalLaunchSummary)` after `operatorHandoffStage(finalLaunchSummary)` and before `finalLaunchStage(finalLaunchSummary)`, but only when `finalLaunchSummary.externalActionLedgerRows` is non-empty.

Status mapping will reuse the existing row text:

- first row starting with `External actions ready` -> `.ready`
- rows containing `blocked`, `missing`, `manual`, `live`, `cost consent`, `global confirmation`, or `backend-device-demo` -> `.needsAttention`
- rows containing `has not loaded` or no rows -> `.waiting`

Detail selection:

- ready or single-row state: first row
- blocked state: first row plus the first explicit operator action, falling back to the first actionable row
- if a backend-device-demo row exists, include it after the first actionable row unless it is already selected
- sanitize the final detail through the existing summary sanitizer

Do not change backend reports, mobile models, Final Launch summary behavior, Device Preflight behavior, SwiftUI layout, provider calls, key files, Xcode settings, or any paid service flow.

## Testing

Add a MobileCore contract test that builds a final launch summary with external ledger rows containing:

- blocked summary row
- resource next action
- resource group row
- first operator action
- backend-device-demo action

Before implementation, `summary.stage(id: "external_actions")` should be missing. After implementation, the stage should be `.needsAttention` and include both the first operator action and the backend-device-demo handoff.

Also update the ready final showcase digest test so a ready external ledger row appears in the expected stage list and maps to `.ready`.
