# P0.607 Final Demo Live Consent Next Action

## Context

P0.606 preserved live-provider consent metadata on `final-showcase-readiness`.
`final-demo-launch-local` embeds that report and promotes its first blocker into the
top-level final demo handoff, but compacting nested blockers can drop the same
metadata. The iPhone final launch surface can then show the correct command without
machine-readable proof that the command needs live provider consent.

## Decision

When `final-demo-launch` compacts nested report blockers, it should preserve explicit
validation and live/cost-consent fields from the nested blocker. The final demo
top-level `next_action` already starts from the compact blocker, so preserving the
fields there keeps both top-level payloads aligned.

## Acceptance

- A nested `final_showcase_readiness` blocker for `game_asset_3d_generation` keeps
  `requires_live_provider_consent: true`.
- The same blocker keeps `completion_requires_live_provider_consent: true`.
- The promoted final-demo top-level `next_action` keeps those fields and the
  validation command.
- No Meshy, OpenAI, print-provider, Xcode signing, or global machine action is run.
