# P0.64 Mobile Final Launch Readiness Design

## Product Intent

P0.63 created a CLI final demo launch profile, but the iPhone app still only
shows provider readiness, backend health, local demo state, and saved NPC
history. For the final on-device showcase, the operator should also see the
same launch-lane status inside the app: whether the local/configured launch path
is ready, partial, or blocked, and whether live provider acceptance remains
explicitly gated.

P0.64 exposes the sanitized final demo launch report through the backend and
adds mobile models/client/preflight integration. The endpoint and client do not
write resources, call live providers, run Xcode, or mutate global state.

## Chosen Approach

Add a read-only backend endpoint:

```http
GET /v1/final-demo-launch?mode=local
GET /v1/final-demo-launch?mode=configured
```

The endpoint returns the same sanitized report shape as the P0.63 CLI command by
calling `build_final_demo_launch_report()`. It accepts only `local` or
`configured`; invalid modes return FastAPI validation errors. For the mobile
app, local mode is the default because it is no-key and safe during device
preflight.

Add Swift mobile core models:

- `FinalDemoLaunchReport`
- `FinalDemoLaunchSummary`
- `FinalDemoLaunchPhase`
- `FinalDemoLaunchLiveCallPolicy`
- `FinalDemoLaunchSafety`

Add API client method:

```swift
getFinalDemoLaunch(mode: String = "local") async throws -> FinalDemoLaunchReport
```

Add device preflight integration:

- `DevicePreflightSummaryBuilder.build(...)` accepts optional
  `finalDemoLaunch` and `finalDemoLaunchError`.
- The builder adds a `final_launch` item.
- The app loads `/v1/final-demo-launch?mode=local` alongside provider readiness.
- The device preflight panel naturally shows this new item through its existing
  item list.

## Status Mapping

- `overall_status == "ready"` -> preflight item `.ready`
- `overall_status == "partial"` -> preflight item `.waiting`
- `overall_status == "blocked"` -> preflight item `.blocked`
- missing report -> `.waiting`
- fetch error -> `.blocked`

Details are sanitized before display. The mobile app never displays provider key
values, raw media, local capture URIs, checkout URLs, or local absolute paths.

## Safety

The backend endpoint and mobile client may expose:

- provider/resource names
- statuses
- missing env variable names
- safe operator commands
- live call policy flags

They must not expose:

- API key values
- bearer tokens
- raw media/base64 data
- checkout/payment URLs
- local absolute paths

They must not execute live provider calls or global machine operations.

## Testing

Backend tests cover:

- `GET /v1/final-demo-launch?mode=local` returns a sanitized report.
- invalid mode returns `422`.
- no secret, raw media, local path, or checkout data appears in the response.

Swift contract tests cover:

- decoding a final launch report payload.
- API client builds `GET /v1/final-demo-launch?mode=local`.
- invalid launch mode is rejected before network.
- device preflight adds a `final_launch` item and maps ready/partial/blocked.
- unsafe launch report details are redacted.

Swift project checks cover source-level wiring:

- `ForgeRootView` fetches final demo launch readiness.
- `PersonalMythForgeAPIClient` has `getFinalDemoLaunch`.
- `DevicePreflightView` renders the new item through existing item rows.

Regression remains:

- `make backend-lint`
- `make backend-test`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests`
- quick final acceptance, expected blocked only by local iOS deploy/Xcode
  environment where applicable

## Non-Goals

- Do not start backend servers.
- Do not write `.env` or iOS deploy config.
- Do not call Meshy, OpenAI, Treatstock, or Sculpteo.
- Do not run Xcode, signing, keychain, Apple license, or global developer setup.
- Do not implement a new mobile screen; the existing preflight list is enough for
  this slice.
