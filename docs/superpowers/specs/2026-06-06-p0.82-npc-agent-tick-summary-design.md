# P0.82 NPC Agent Tick Summary Design

## Goal

Make option C concrete by turning the NPC Agent loop into an iPhone-readable, testable status surface. The app should show whether NPCs are waiting, using initial session traces, running a tick or autonomy batch, or showing the latest resolved Agent tick.

## Scope

This iteration adds a mobile summary for NPC Agent ticks. It does not change the backend Agent runtime, provider selection, OpenAI calls, Meshy calls, printing, or Unity integration.

## User Experience

The existing `Next Village Tick` section gains a compact summary header. Before forging, it explains that a myth session is needed. After forging, it shows the initial Agent traces and invites the user to advance the village. During single-tick or autonomy runs, it shows an in-progress status. After a tick resolves, it summarizes runtime, tick number, accepted/rejected actions, visible changes, and saved tick history.

## Architecture

Create `NPCAgentTickSummary` in `PersonalMythForgeMobileCore`. The builder accepts the current session, latest tick, saved tick count, loading flags, and sanitized error text. `ForgeRootView` builds the summary from existing state and passes it into `NPCTickView`. `NPCTickView` renders the summary while keeping the detailed per-NPC trace cards unchanged.

## Data Rules

The summary uses only data already present in `MythSession` and `NPCAgentTick`: runtime label, trace counts, world-resolution counts, and visible change counts. It must not include raw media, local capture URLs, absolute file paths, provider secrets, checkout links, or raw private context.

## Testing

Add Swift contract tests for:

- waiting before a myth session exists
- initial session traces after forge but before the first tick
- latest tick resolution counts and runtime
- autonomy or tick in-progress state
- redaction of secrets, local paths, capture URLs, and checkout links

Update the Swift project source gate and backend iOS showcase acceptance gate so this feature remains part of the mobile demo contract.

## Acceptance

The feature is accepted when the mobile app compiles, the Swift core contract tests pass, project checks pass, backend iOS showcase acceptance passes, full backend lint/test pass, and final quick acceptance still reports no failures. Any remaining iOS deploy blockers must be unchanged local-environment blockers.

## Self-Review

No placeholders remain. The scope is limited to one mobile summary feature. The design keeps provider calls out of the app and preserves the privacy rule that only approved summaries and session-level Agent traces are shown.
