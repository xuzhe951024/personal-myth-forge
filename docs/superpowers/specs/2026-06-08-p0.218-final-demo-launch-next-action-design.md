# P0.218 Final Demo Launch Next Action Design

## Goal

Expose a structured `next_action` on the top-level final demo launch report so
the iPhone Final Launch receipt can show the next operator move without parsing
the longer first-blocker detail.

## Scope

- Backend `final_demo_launch_report.next_action`.
- Swift Codable model and launch receipt rendering.
- Source gates and static 390x844 iPhone visual evidence.
- No provider calls, no Xcode/signing, no secret writes, no global mutation.

## Contract

When launch is blocked or partial:

```json
{
  "id": "apply_final_resources",
  "label": "Apply final resources",
  "status": "blocked",
  "classification": "final_demo_launch_phase",
  "command": "make final-apply-resources",
  "detail": "one-file backend and iOS final demo handoff ...",
  "source": "first_blocker",
  "source_id": "apply_final_resources"
}
```

When launch is ready, `next_action` is `null`.

## Mobile Rendering

`FinalLaunchMobileSummary.launchReceiptRows` inserts a sanitized row before the
existing first-blocker row:

```text
Next action: apply_final_resources blocked final_demo_launch_phase | make final-apply-resources | one-file backend and iOS final demo handoff ...
```

The existing first blocker row remains intact for backward compatibility and
audit detail.
