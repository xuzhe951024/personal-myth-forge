# P0.567 iOS Device Provider Split Design

## Context

`final-showcase-readiness` currently reports `ios_deployable` as the first blocker while the nested iOS device launch rehearsal next action can be `PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence`. That mixes two different final-goal lanes:

- iOS deployability: local deploy config, backend LAN health, Xcode build gate, launch certificate, and saved device rehearsal evidence.
- Live provider proof: Meshy/OpenAI configured evidence, explicit live-call consent, and provider cost review.

The product goal needs both lanes, but presenting live provider consent as an iOS deploy blocker sends the operator to the wrong next action.

## Design

Keep live provider evidence requirements in `game_asset_3d_generation`, `ai_agent_npc`, and `provider_key_handoff`. For iOS device reports, provider/live actions should remain visible as contextual operator actions, but they must not become the primary device gate action when a device-specific action exists.

The smallest change is to make iOS device launch certificate and rehearsal readiness choose device-first actions for `first_blocker`, `next_action`, and device action bundles:

- Prefer iOS deploy config, backend-device-demo/mobile-preflight, and Xcode/build actions before provider/live actions.
- Ignore provider-only nested actions when deciding the command for the `final_handoff_index` device gate.
- Preserve provider/live actions elsewhere so configured provider evidence is still visible and validated by its own capability.

## Testing

Add focused regression tests that construct saved reports where the first nested action is live-provider evidence and a later nested action is a device action. The expected behavior is that iOS device reports choose the device action first, while provider readiness remains a separate final-showcase capability.

No live provider calls, Xcode signing, Keychain writes, or global state changes are part of this phase.
