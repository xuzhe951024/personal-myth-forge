# P0.146 Final Resource Repair Design

## Problem

`services/backend/final-resources.env.example` now keeps user-specific iOS values empty, but existing ignored local copies can still contain older example values:

- `PRODUCT_BUNDLE_IDENTIFIER=com.example.personalmythforge`
- `PMF_BACKEND_BASE_URL=http://192.168.1.10:8080`

P0.145 correctly blocks those placeholders, but the operator path is still manual: inspect the ignored file, identify stale examples, clear them, then rerun preview. This creates avoidable friction in the final API/key handoff.

## Goal

Add a repo-local final resource repair workflow that clears known stale placeholders from `services/backend/.local/final-resources.env` while preserving already filled provider secrets and other user values.

## Non-Goals

- Do not invent or fill API keys, Apple Team IDs, bundle identifiers, backend URLs, or provider endpoints.
- Do not write `services/backend/.env`.
- Do not write `apps/mobile/ios/Config/Deployment.local.xcconfig`.
- Do not call Meshy, OpenAI, Treatstock, Xcode, signing, keychain, or network services.
- Do not mutate global machine state.

## Design

Add a focused Python report module, `myth_forge_api.final_resource_repair`, with a single builder:

```python
build_final_resource_repair_report(repo_root=None, resources_file=None, apply=False)
```

The builder parses the final resources file line by line without shell evaluation. It detects repairable placeholder values for these keys only:

- `PRODUCT_BUNDLE_IDENTIFIER` values under `com.example` or `com.example.*`
- `PMF_BACKEND_BASE_URL` values matching the old example LAN URL, with or without trailing slash

In preview mode, it returns `status="repairable"` and does not write files. In apply mode, it rewrites only the affected assignment lines to empty values, preserving comments, ordering, blank lines, unrelated keys, and already filled secrets. Reports never include raw secret values.

## CLI And Make Targets

Add two CLI paths:

- `final-resource-repair-preview`: read-only, writes `.local/final-resource-repair-preview.json`
- `final-resource-repair`: applies the safe local repair and writes `.local/final-resource-repair.json`

Add matching Make targets:

- `make final-resource-repair-preview`
- `make final-resource-repair`

`make final-resource-init` should keep refusing to overwrite an existing file, but its message should point operators to `make final-resource-repair-preview` when the existing file may be from an older template.

## Safety

The repair command only writes the ignored final resource intake file when `apply=True`. It refuses to touch the file if git tracks `services/backend/.local/final-resources.env`. The report includes explicit safety flags:

- `writes_final_resources`
- `writes_backend_env=False`
- `writes_ios_deploy_config=False`
- `provider_calls=False`
- `global_mutation=False`
- `provider_secrets_in_report=False`

## Testing

Tests must prove:

- Missing resources report routes to `make final-resource-init`.
- Preview detects stale placeholders and does not write.
- Apply clears only stale placeholders and preserves secrets.
- Apply refuses tracked destination.
- CLI writes preview/apply reports without leaking secret values.
- Makefile exposes both targets.
- `final-resource-init` points existing-file operators to repair preview.

Full backend and Swift regression remain required before merge.
