# P0.5 AI NPC And 3D Loop Design

## Product Goal

Upgrade the current local browser demo from a deterministic walkthrough into a convincing proof of the core product loop: a real-world object and approved context capsule become a myth seed, optionally generate a real Meshy GLB, drive AI-authored NPC interpretation and action plans, pass through a lightweight world arbitrator, and render visible world-state consequences in the demo.

P0.5 is still backend-first and web-demo-first. It is designed to reduce the largest risks before building the deployable iOS Unity client: 3D generation quality, NPC autonomy, world-state arbitration, API-key integration, and regression coverage.

## Current Baseline

The repository already has:

- FastAPI backend with `POST /v1/myth-sessions` and `/demo`.
- Pydantic session models for object observations, context capsules, myth seeds, NPC reactions, generated assets, print candidates, and myth sessions.
- `ThreeDProvider` abstraction with `LocalThreeDProvider` and `MeshyThreeDProvider`.
- CLI for local or Meshy 3D prompt generation.
- Deterministic hard-coded NPC reactions in `domain.pipeline._create_npc_reactions`.
- A mobile-first web demo that renders object input, a procedural artifact placeholder, NPC cards, and print candidate state.

P0.5 replaces only the deterministic NPC generation path and extends the session contract. It preserves local deterministic fallbacks so tests, demos, and development do not require external API keys.

## Scope

Included:

- Add `NPCDirector` provider abstraction.
- Add `LocalNPCDirector` that preserves deterministic behavior.
- Add `OpenAINPCDirector` using OpenAI Responses API structured outputs to produce `NPCReaction` data.
- Add `WorldArbitrator` abstraction and a local rule-based arbitrator.
- Extend `MythSession` with a `world_resolution` field.
- Extend `/demo` to show provider badges, NPC debate, accepted/rejected actions, and world-state changes.
- Add config keys for OpenAI model/provider selection without requiring actual keys during local demo.
- Add CLI/report path for 20-prompt 3D evaluation using existing `ThreeDProvider`.
- Add unit, API contract, CLI, and browser visual regression tests.
- Keep all raw private context out of scope: only user-approved capsule fields are accepted.

Excluded:

- Full Unity/iOS implementation.
- Real device 3D scanning.
- Persistent NPC memory across sessions.
- Long-running multi-agent tool handoff through Agents SDK.
- Print provider quoting or fulfillment.
- Voice NPCs.

These exclusions remain explicit because the active long-term goal requires iOS deployment, 3D scanning, 3D generation, and AI Agent NPCs. P0.5 is a necessary intermediate step, not the final goal.

## Architecture

```text
ObjectObservation + ContextCapsule
  -> ObjectCard
  -> MythSeed
  -> ThreeDProvider
       local | meshy
  -> NPCDirector
       local | openai
  -> WorldArbitrator
       local_rule_based
  -> LocalPrintProvider
  -> MythSession + WorldResolution
  -> /demo rendering
```

### Provider Boundaries

`ThreeDProvider` remains responsible only for generating or describing a game asset.

`NPCDirector` accepts:

- `session_id`
- `object_card`
- `myth_seed`
- `context_capsule`
- `generated_asset`

It returns a list of `NPCReaction` values and a provider name. The local provider returns the existing three NPCs. The OpenAI provider calls the Responses API with a structured Pydantic output model matching the backend schema.

`WorldArbitrator` accepts the same session context plus NPC reactions and returns `WorldResolution`. It does not call a model in P0.5. It applies deterministic rules so the demo has stable behavior and tests can prove the action validity layer.

## Data Model

Add:

```python
class NPCDirectorResult(BaseModel):
    provider: str
    reactions: list[NPCReaction]


class ResolvedNPCAction(BaseModel):
    npc_id: str
    action: str
    status: Literal["accepted", "rejected"]
    reason: str


class WorldResolution(BaseModel):
    arbitrator: str
    summary: str
    accepted_actions: list[ResolvedNPCAction]
    rejected_actions: list[ResolvedNPCAction]
    world_state_delta: dict[str, int | str | bool]
    visible_changes: list[str]
```

Extend:

```python
class MythSession(BaseModel):
    ...
    npc_director: str
    world_resolution: WorldResolution
```

Backward compatibility is not required for old demo payloads because this is a pre-production prototype, but tests should assert that the new response shape is JSON-safe.

## OpenAI NPC Director

Use official OpenAI API guidance for structured outputs. The OpenAI docs describe Structured Outputs as a way to make model responses adhere to a supplied schema, and the Responses API as the current response-generation surface with text/image inputs, structured outputs, tools, and stateful workflows. P0.5 uses this surface through the Python SDK when `NPC_PROVIDER=openai`.

Default configuration:

- `NPC_PROVIDER=local`
- `OPENAI_API_KEY=` empty
- `OPENAI_NPC_MODEL=gpt-4.1-mini` as the default configured model string
- `OPENAI_API_BASE_URL=` optional override for tests or compatible gateways

Runtime behavior:

- If `NPC_PROVIDER=local`, no OpenAI dependency or key is required.
- If `NPC_PROVIDER=openai` and no key exists, raise a typed configuration error before any network call.
- The OpenAI provider requests exactly three NPC reactions for Mara, Ior, and Senn for P0.5, preserving stable character identities while allowing generated interpretation and plans.
- The prompt forbids raw private data access and states that only the provided capsule may be used.
- The response must parse into a Pydantic schema. Invalid output becomes a provider error rather than silently falling back.

P0.5 deliberately avoids full Agents SDK orchestration. The NPC Director is an agent-like runtime with structured model output, but it does not yet use tool handoff or persistent traces. This keeps the first AI NPC milestone small enough to test well.

Sources:

- OpenAI Structured Outputs: `https://platform.openai.com/docs/guides/structured-outputs`
- OpenAI Responses API: `https://platform.openai.com/docs/api-reference/responses/create`
- OpenAI Agents SDK overview: `https://platform.openai.com/docs/guides/agents-sdk/`

## World Arbitration

The local arbitrator provides guardrails between autonomous NPC text and executable world changes. It should:

- Accept harmless actions such as approaching, observing, naming, debating, guarding, sketching, kneeling, or inviting.
- Reject unsafe, private-data, destructive, impossible, or fulfillment-related actions.
- Convert accepted actions into compact `world_state_delta` values such as `faith`, `curiosity`, `suspicion`, `debate_intensity`, and `artifact_renown`.
- Produce visible changes for the demo, such as `"Mara kneels near the relic"` or `"The village debate grows louder"`.

The arbitrator is deterministic in P0.5. A model-backed arbitrator can be added after tests prove the contract.

## 3D Evaluation CLI

Extend the CLI with:

```bash
uv run python -m myth_forge_api.cli evaluate-3d \
  --provider local|meshy \
  --prompts-file prompts.txt \
  --output reports/p0.5-3d-eval.json
```

The report includes one row per prompt:

- prompt
- provider
- status
- asset URI when successful
- elapsed seconds
- error message when failed

The CLI should not compute subjective visual quality automatically. It should produce a structured evidence log that supports later manual review of 20 prompts.

## Demo UX

The `/demo` page should remain the first screen and remain mobile-first. Add:

- Provider badges: 3D provider and NPC provider.
- A world state strip with compact values for faith, suspicion, curiosity, debate intensity, and artifact renown.
- NPC debate grouping: each NPC card shows interpretation, plan, and whether its actions were accepted or rejected.
- Visible changes list below the artifact stage.
- GLB viewer remains enabled for remote Meshy URLs; local assets keep the procedural artifact placeholder.

The page should still be useful with no external keys. When external providers are unavailable, it should communicate provider state without exposing stack traces.

## API And Error Handling

`POST /v1/myth-sessions` keeps the same request shape.

Errors:

- Missing Meshy key with `THREE_D_PROVIDER=meshy` returns an HTTP 502 provider error for API calls and a non-zero CLI exit for CLI paths.
- Missing OpenAI key with `NPC_PROVIDER=openai` returns an HTTP 502 provider error for API calls and a non-zero CLI exit for CLI paths.
- Local provider paths must never require external keys.
- Provider error messages must not include raw API keys, request headers, or private capsule data beyond fields already submitted by the user.

## Testing And Regression

Required test coverage:

- `NPCDirector` local provider returns three stable NPCs.
- OpenAI provider builds the expected structured-output request using a fake client.
- OpenAI provider rejects missing API key before network calls.
- OpenAI provider parses a valid fake structured response into `NPCReaction`.
- World arbitrator accepts allowed actions and rejects unsafe actions.
- Pipeline injects custom `NPCDirector` and `WorldArbitrator`.
- API response includes `npc_director` and `world_resolution`.
- CLI `evaluate-3d` emits a JSON report with success and failure rows.
- `/demo` renders world resolution fields.
- Browser visual regression covers mobile and desktop, including no horizontal overflow and visible world-state panel.

Existing verification remains:

- `make backend-lint`
- `make backend-test`
- local CLI command paths
- browser smoke test for `/demo`

## Acceptance Criteria

- Local demo works with no external keys.
- Setting `NPC_PROVIDER=openai` routes NPC generation through the OpenAI provider when `OPENAI_API_KEY` is present.
- Setting `THREE_D_PROVIDER=meshy` routes game asset generation through Meshy when `MESHY_API_KEY` is present.
- `POST /v1/myth-sessions` returns generated asset metadata, AI/local NPC reactions, and world arbitration output.
- Demo makes the NPC autonomy and world-state consequence visible.
- CLI can run a 20-prompt 3D evaluation report without changing global machine state.
- Tests prove all new local behavior and fake-provider behavior without making paid external API calls.
- No implementation reads raw email, calendar, chat, or private documents.

## Long-Term Fit

P0.5 should leave clean seams for the final iOS minimum showable model:

- Unity can call the same `POST /v1/myth-sessions` endpoint.
- Unity can render `generated_asset.uri` when it is a GLB.
- Unity can map `WorldResolution.visible_changes` and `ResolvedNPCAction` to animation and staging.
- Future 3D scan upload can extend `ObjectObservation` with asset/image references without changing NPC/world contracts.
- Future print quote providers can use the existing `PrintCandidate` path.
