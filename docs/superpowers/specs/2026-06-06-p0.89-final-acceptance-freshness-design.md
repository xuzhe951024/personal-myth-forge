# P0.89 Final Acceptance Freshness Design

## Context

The final launch flow now reads
`services/backend/.local/final-acceptance-local.json` and exposes that saved
acceptance report through `/v1/final-demo-launch`. P0.88 made the mobile
`Launch Receipt` very direct, but that creates one remaining risk: a saved
acceptance report can be older than the current checked-out code and still look
ready or blocked for the wrong revision.

During final iPhone handoff, stale acceptance evidence should never be treated
as current readiness.

## Approaches

### A. Backend freshness guard using git HEAD commit time

The backend `final_acceptance_readiness` report can compare the saved acceptance
file mtime with the current repository HEAD commit timestamp. If the saved
report is older than HEAD, readiness becomes `blocked` with a
`final_acceptance_freshness` blocker and an operator action to regenerate the
report.

This is the recommended approach. It is deterministic, read-only, does not need
new files, and protects both the backend final launch report and the iPhone
display.

### B. Embed git revision in every final acceptance report

This is stronger long term, but it requires changing the final acceptance report
schema and every producer/fixture. It is useful later for comparing exact
revision hashes, but it is larger than needed for the current local handoff.

### C. Mobile-only warning

The iPhone app could show a generic warning that reports may be stale. This does
not prevent stale backend readiness from becoming `ready`, so it does not solve
the safety issue.

## Chosen Design

Use approach A. Add a `freshness` object to the backend
`final_acceptance_readiness_report`:

```json
{
  "status": "fresh | stale | unknown",
  "classification": "fresh_report | stale_report | git_unavailable | source_missing",
  "checked_against": "git_head",
  "source_modified_at": "2026-06-06T12:00:00Z",
  "current_revision": "abcdef1",
  "current_revision_committed_at": "2026-06-06T12:05:00Z"
}
```

When the acceptance file is missing, freshness is `unknown` with
`source_missing`. When git metadata is unavailable, freshness is `unknown`, not
blocked, because source-gate tests and non-git temp fixtures should still work.
When freshness is `stale`, readiness status is forced to `blocked`.

The iPhone model decodes `freshness` and `FinalLaunchMobileSummary` shows it in
both the `Launch Receipt` and `Acceptance` rows.

## Data Flow

```text
services/backend/.local/final-acceptance-local.json
  -> final_acceptance_readiness.freshness
  -> final_demo_launch.final_acceptance_readiness
  -> FinalAcceptanceFreshness
  -> FinalLaunchMobileSummary launch receipt + acceptance rows
```

## Safety

- No command is executed to regenerate final acceptance.
- Git metadata reads are local and read-only.
- The report exposes only a short revision, timestamps, and classifications.
- Absolute paths, secrets, checkout links, media URIs, and payment URLs remain
  sanitized by the existing readiness sanitizer and mobile `sanitize`.
- If git metadata is unavailable, the backend reports `unknown` freshness rather
  than inventing a ready/stale verdict.

## Acceptance Criteria

- Backend tests cover:
  - saved report newer than HEAD remains fresh
  - saved report older than HEAD becomes blocked with
    `final_acceptance_freshness`
  - missing report has `source_missing` freshness
  - no-git temp fixture has `git_unavailable` freshness without crashing
- `/v1/final-demo-launch` embeds the freshness object.
- Swift model decodes freshness from final launch payload.
- Mobile summary receipt and acceptance rows display stale/freshness status.
- Project checks and backend iOS showcase source acceptance require the freshness
  model and tests.
- A 390x844 visual evidence page shows a stale final acceptance report on the
  iPhone status panel without overlap, clipping, or secret values.
