# P0.388 Evaluation Report Status Design

## Context

The final showcase readiness ledger uses local 3D and NPC evaluation reports as
evidence for two core product capabilities:

- `game_asset_3d_generation`
- `ai_agent_npc`

The readiness wrappers already infer status from `failed` counts, but the raw
saved reports under `services/backend/.local/3d-evaluation-local.json` and
`services/backend/.local/npc-evaluation-local.json` do not expose a top-level
`status`. That makes these core proof artifacts less consistent with the rest
of the final handoff reports.

## Decision

Add a top-level `status` field to both raw evaluation report builders:

- `three_d_evaluation_report.status`
- `npc_agent_evaluation_report.status`

The status value is:

- `succeeded` when `failed == 0`
- `failed` when `failed > 0`

This keeps the existing `succeeded` and `failed` counters as the authoritative
case summary while making report-level consumers able to decode the result
directly.

## Non-Goals

- Do not change CLI exit code behavior.
- Do not change provider selection, prompts, evaluation suites, review rubric,
  generated asset shape, NPC trace/tick behavior, or safety redaction.
- Do not call real Meshy/OpenAI providers or write any API keys.
- Do not change the final showcase readiness classification from partial to
  ready; live Meshy/OpenAI evidence still requires operator-provided resources
  and consent.

## Testing

Add RED/GREEN coverage for:

- default local 3D evaluation reports `status == "succeeded"`
- per-case 3D provider failures produce `status == "failed"`
- default local NPC evaluation reports `status == "succeeded"`
- NPC provider failures produce `status == "failed"`

After implementation, regenerate ignored local evaluation reports and confirm
their top-level status fields are present.
