# P0.458 Visual Readiness Freshness Design

## Context

P0.457 added a new checked-in visual regression artifact for the Demo Script external actions handoff. The current saved local readiness files can still contain an older `visual-regression-local.json` with fewer passed artifacts, while `final-showcase-readiness` reports visual regression as ready.

The final iPhone showcase needs visual regression evidence to prove the current checked-in demo surface, not just any older passing report.

## Decision

Add an artifact inventory freshness guard to `visual_regression_readiness`.

The readiness report will compare the saved visual regression report against the current `DEFAULT_VISUAL_ARTIFACTS` ids:

- If every current artifact id is present and passed, readiness can remain ready.
- If any current artifact id is missing, readiness becomes blocked.
- If the saved summary passed count is lower than the current artifact count, readiness becomes blocked.
- The blocker command is `make visual-regression-local`.
- The report remains read-only and does not run browser, Xcode, providers, or global commands.

## Alternatives Considered

1. Keep only the current mtime/git freshness guard.
   This misses checked-in artifact additions when the saved JSON file is newer than the product commit or comes from a copied fixture.

2. Automatically regenerate visual regression during final showcase readiness.
   This would make the readiness command slower and would violate the current read-only report pattern.

3. Store a separate schema version in `.local`.
   This still needs a current source of truth. The checked-in artifact id list is already the canonical source.

## Acceptance Criteria

- `build_visual_regression_readiness_report()` blocks a saved passing report that lacks any current visual artifact id.
- The blocker detail names at least one missing artifact id.
- The operator action tells the operator to rerun `make visual-regression-local`.
- Existing stale-by-git freshness behavior remains intact.
- Final showcase readiness inherits the blocked visual capability through the existing readiness dependency.

## Safety

This change only reads checked-in source files and ignored local JSON reports. It does not write provider env files, iOS deploy config, Keychain, Xcode state, or external provider resources.
