# P0.376 Live Provider Completion Consent Design

## Context

The final showcase readiness report marks `game_asset_3d_generation` and
`ai_agent_npc` as `partial` when local proof exists but live Meshy/OpenAI
evidence is still missing. These rows currently propagate
`requires_live_provider_consent` from the live evidence report's immediate
`next_action`.

That immediate action can be a safe setup action such as `make provider-handoff`,
which does not call live providers and correctly has
`requires_live_provider_consent: false`. The capability still cannot become
`ready` without later live Meshy/OpenAI calls that require explicit consent.

## Decision

Add a separate completion-level flag to final showcase capability rows:

- `completion_requires_live_provider_consent: true`

This flag means the capability's path from `partial` to `ready` still requires
live provider consent somewhere downstream. It does not replace
`requires_live_provider_consent`, which continues to describe the immediate
action in `command`.

## Scope

In scope:

- Add `completion_requires_live_provider_consent` to
  `game_asset_3d_generation` when local 3D proof is ready and live provider
  evidence is not ready.
- Add `completion_requires_live_provider_consent` to `ai_agent_npc` when local
  NPC Agent proof is ready and live provider evidence is not ready.
- Preserve existing immediate-action consent semantics.
- Add focused regression coverage for the case where immediate provider handoff
  is safe but completion still requires live provider consent.

Out of scope:

- Running live Meshy/OpenAI providers.
- Changing provider key handoff behavior.
- Changing iOS UI decoding in this phase.
- Changing print fulfillment cost consent semantics.

## Safety

The report remains read-only and local-only. The change does not write secrets,
does not run provider commands, does not trigger Xcode, and does not mutate any
global machine state.
