# P0.508 Handoff Index Backend Priority Design

## Context

The final iOS launch handoff remains blocked by physical device evidence and local resource/provider configuration. Recent phases aligned `final-showcase-readiness`, `final-demo-launch-local`, `final-local-report-refresh`, `final-external-action-ledger`, and `final-launch-closure-packet` so their top-level operator actions start with:

1. write or repair iOS deploy config,
2. start `backend-device-demo`,
3. continue provider, print, backend URL, and Xcode handoffs.

`final-handoff-index` is the unified local/configured handoff index used by downstream iOS launch reports. Its current top-level actions still place the iPhone-reachable backend URL action before `backend-device-demo`. That is less executable for the current first blocker because the operator should start the LAN backend before rerunning iPhone device checks.

## Decision

Change `final_handoff_index._prioritize_final_handoff_operator_actions` so backend-device-demo is the first promoted device action after the first action only when that first action is already a mobile deploy handoff. iPhone-reachable backend URL actions remain promoted, but they follow backend-device-demo in that physical-device flow.

The intended top-level order is:

1. current first action, usually `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight`,
2. `start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight`,
3. iPhone-reachable backend URL handoff,
4. provider handoff,
5. guarded print quote,
6. remaining configured preflight or live consent actions.

## Alternatives Considered

1. Reorder upstream `final-demo-launch-local` actions again. Those reports are already correct; the regression is in `final-handoff-index`'s own priority pass.
2. Special-case only `MOBILE_WRITE_DEPLOY_CONFIG_AUTO`. This would miss manual deploy config flows where the first action is `provide DEVELOPMENT_TEAM...`.
3. Reorder the existing device priority bucket in `final_handoff_index`. This is narrow, deterministic, and keeps lanes/source reports untouched.

## Behavior

- Existing first action stays first.
- If the first action is a mobile deploy handoff and backend-device-demo is present in remaining actions, backend-device-demo is promoted immediately after the first action.
- If the first action is not a mobile deploy handoff, existing missing-report behavior is preserved.
- iPhone-reachable backend URL actions stay promoted, but after backend-device-demo.
- Provider and print actions still stay before generic configured preflight/liveness actions.
- The report remains read-only and safe: no provider calls, print calls, Xcode signing, Keychain actions, shell writer execution, or secret writes are introduced.

## Tests

Add a regression test in `services/backend/tests/test_final_handoff_index.py` using the existing final demo launch fixture. It should assert:

- deploy writer remains the first operator action,
- backend-device-demo is second,
- backend-device-demo appears before the iPhone-reachable backend URL action,
- backend-device-demo appears before provider and print handoffs.

## Self-Review

- Scope is limited to one priority order in the flat final handoff index action list.
- Structured lane data, device action bundles, source freshness, and safety fields are unchanged.
- Requirements are covered by local deterministic tests and safe ignored evidence refresh.
