# P0.330 Mobile Xcode Build Evidence Next Action Design

## Context

The final iPhone demo cannot be considered deployable until the Mac-side Xcode
build gate has evidence. The current `mobile_xcode_build_evidence_report`
already records status, classification, checks, output lines, and
`operator_actions`, but it lacks top-level `first_blocker` and `next_action`.

Other final handoff reports now expose those fields. Leaving this report as a
list-only handoff makes the final device lane less consistent and harder for
parent reports or mobile summaries to consume.

## Decision

Add top-level `first_blocker` and `next_action` to
`mobile_xcode_build_evidence_report`.

- Ready reports return `null` for both.
- Blocked reports promote the first check into `first_blocker`.
- `command` is the same concrete operator action already emitted for the
  classification.
- `validation_command` is always `make mobile-xcode-build-evidence`.
- `next_action` mirrors `first_blocker` with `source: "first_blocker"`.

The final-local-report-refresh synthetic Xcode snapshot gets the same shape, but
keeps `commands_run: false` and `xcode_or_signing: false` so safe refresh still
does not touch global Apple SDK/signing state.

## Safety

This phase does not accept the Xcode license, run a real Xcode build from safe
refresh, write signing state, use Keychain, or call live providers.

## Acceptance

- Focused tests prove blocked mobile Xcode evidence exposes concrete
  `first_blocker` and `next_action`.
- Ready mobile Xcode evidence exposes `null` action fields.
- Final local report refresh synthetic Xcode evidence includes the same action
  shape without running Xcode.
- Safe local report refresh writes an ignored Xcode evidence snapshot with
  top-level action fields and without running Xcode/signing commands.
- `make mobile-xcode-build-evidence` remains the validation command for a human
  operator after Xcode/license state is explicitly handled.
