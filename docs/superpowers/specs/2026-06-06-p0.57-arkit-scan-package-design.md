# P0.57 ARKit Scan Package Design

## Product Intent

The iOS app already has several capture paths:

- camera single photo
- photo set
- guided Object Capture photo capture
- manual file import
- `arkit_scan` payload validation that requires at least one scan asset

The gap is the bridge between a real ARKit/Object Capture scan export and the
existing upload contract. Today `arkit_scan` can accept a manually selected GLB,
USDZ, or binary file, but there is no named mobile-core builder that packages a
scan export plus optional reference photos as a single scan capture payload.

P0.57 adds that bridge. It does not attempt to run ARKit or reconstruct meshes
inside local tests; those are iOS device/runtime concerns. Instead, it makes the
post-scan handoff deterministic and SwiftPM-verifiable: a scan export file plus
reference images becomes a `CaptureMediaSelection(mode: .arkitScan, ...)` that
the existing upload and myth-session flow already understands.

## Behavior

Add `ARKitScanPackageBuilder` to `PersonalMythForgeMobileCore`.

Inputs:

- `ARKitScanExportFile`
  - filename
  - content type
  - data
- `[ARKitScanReferenceImageFile]`
  - filename
  - content type
  - data

Output:

- `CaptureMediaSelection(mode: .arkitScan, media: [scanAsset] + referenceImages)`

Rules:

- scan export content types:
  - `model/gltf-binary`
  - `model/vnd.usdz+zip`
  - `application/octet-stream`
- reference image content types:
  - `image/heic`
  - `image/heif`
  - `image/jpeg`
  - `image/png`
- content types are trimmed and lowercased.
- scan export is always first in the output media array.
- reference images are sorted by localized filename.
- at most 11 reference images are kept so the resulting `arkit_scan` package
  fits the existing 12-media upload cap.
- files larger than `CaptureDraft.maxFileBytes` fail.
- unsupported scan or reference content types fail.

The builder returns a ready `CaptureMediaSelection`, not only an array of media,
so app code and tests can directly assert `selection.isReadyForUpload`.

## App Integration

Keep the existing app UI behavior:

- `arkit_scan` can still import a scan file via fileImporter.
- reference photos can still be selected through PhotosPicker.
- the guided Object Capture photo session remains unchanged.

P0.57 does not introduce a new visible UI surface. It adds a source-level bridge
that a future full-device iteration can call after ARKit/Object Capture exports
the final model. The app source and project checks should reference the builder
so this handoff remains part of the no-simulator acceptance gates.

## Acceptance

Add SwiftPM mobile core contract tests:

- builds a ready `arkit_scan` selection from one GLB export and two reference
  images.
- normalizes scan and reference content types.
- sorts references by filename while keeping the scan asset first.
- truncates references to 11 images.
- rejects unsupported scan content type.
- rejects unsupported reference content type.
- rejects oversized scan export.
- rejects oversized reference image.

Extend `PersonalMythForgeMobileProjectChecks` to prove:

- `ARKitScanPackageBuilder.swift` exists in mobile core.
- `Package.swift` still exposes the file through the core target.
- contract tests call `testARKitScanPackage...` cases.
- `ForgeRootView` or the mobile app source references the ARKit scan package
  bridge.
- no mobile source contains provider secret markers.

## Non-Goals

- No live ARKit device run.
- No mesh reconstruction in local tests.
- No Unity integration.
- No backend API contract change.
- No Meshy/OpenAI/provider call.
- No new visible UI flow.
- No Xcode signing, device install, or Apple SDK license mutation.
