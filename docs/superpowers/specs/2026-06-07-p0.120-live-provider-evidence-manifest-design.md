# P0.120 Live Provider Evidence Manifest Design

## Context

P0.119 wires local visual regression evidence into the final handoff and iPhone launch readiness. The project now has strong no-key rehearsal gates, configured resource preflight, provider readiness checks, 3D/NPC evaluation suites, and consent-gated configured final acceptance.

The remaining final-demo gap is operational: after the operator supplies Meshy, OpenAI, Treatstock, and iOS deployment values, there is no single sanitized manifest that tells the operator which live evidence files must exist, which commands create them, which commands may spend credits, and whether those files prove the configured 3D/NPC/print/iPhone path is ready enough for final review.

## Goal

Add a read-only live provider evidence manifest that consolidates the configured evidence expected after keys are supplied, without calling live providers, reading secrets, running Xcode, starting servers, or mutating global state.

## Options Considered

### Option A: Fold More Checks Into `final-configured-preflight`

This keeps fewer report files, but it makes the configured preflight too broad. Preflight currently answers "are resources and commands ready enough to proceed?" Live evidence answers "did the explicit live runs produce usable proof?" Keeping those separate makes operator intent clearer and avoids making preflight appear to spend credits.

### Option B: Add A New `live-provider-evidence` Manifest

This adds one report and one Make target. It can be referenced by final launch reports and iPhone readiness without changing the safety semantics of existing preflight commands. This is the recommended approach.

### Option C: Run Live Evidence Automatically During Rehearsal

This would be closest to a one-command final pass, but it violates the project safety policy. Meshy/OpenAI/Treatstock calls may spend credits and must remain behind explicit consent and key provisioning.

## Chosen Design

Implement Option B.

The backend adds `build_live_provider_evidence_report()` and a CLI command:

```bash
cd services/backend
uv run python -m myth_forge_api.cli live-provider-evidence \
  --repo-root ../.. \
  --output .local/live-provider-evidence.json
```

The project root adds:

```bash
make live-provider-evidence
```

The command is read-only. It reads known ignored local JSON reports when present, classifies each evidence slot, sanitizes output, and returns exit `0` only when required live evidence is ready. Exit `2` means a usable blocked or partial manifest was written.

## Evidence Slots

The first version tracks these slots:

- `provider_handoff`: `services/backend/.local/provider-handoff.json`
  - Command: `cd services/backend && uv run python -m myth_forge_api.cli provider-handoff --require-core-real --output .local/provider-handoff.json`
  - Proves: Meshy/OpenAI core provider config is real-provider-ready without calling providers.
  - Live calls: no.

- `three_d_evaluation_configured`: `services/backend/.local/3d-evaluation-configured.json`
  - Command: `cd services/backend && uv run python -m myth_forge_api.cli evaluate-3d --provider meshy --suite default-v0 --output .local/3d-evaluation-configured.json`
  - Proves: Meshy generated the canonical 20-case 3D suite and produced scene handoff coverage for review.
  - Live calls: yes, may spend Meshy credits.

- `npc_evaluation_configured`: `services/backend/.local/npc-evaluation-configured.json`
  - Command: `cd services/backend && uv run python -m myth_forge_api.cli evaluate-npc --provider openai --suite default-v0 --tick-steps 2 --output .local/npc-evaluation-configured.json`
  - Proves: OpenAI NPC Agent structured-output ticks can complete the canonical NPC evaluation.
  - Live calls: yes, may spend OpenAI credits.

- `final_acceptance_configured`: `services/backend/.local/final-acceptance-configured.json`
  - Command: `cd services/backend && uv run python -m myth_forge_api.cli final-acceptance --profile quick --provider-mode configured --require-real-core --allow-live-provider-calls --repo-root ../.. --output .local/final-acceptance-configured.json`
  - Proves: the configured quick acceptance loop can run with real core providers and explicit live-call consent.
  - Live calls: yes, may spend Meshy/OpenAI/print provider credits.

- `final_demo_launch_configured`: `services/backend/.local/final-demo-launch-configured.json`
  - Command: `cd services/backend && uv run python -m myth_forge_api.cli final-demo-launch --mode configured --repo-root ../.. --output .local/final-demo-launch-configured.json`
  - Proves: the configured launch packet can be regenerated after live evidence is present.
  - Live calls: no.

## Status Rules

Each slot has:

- `ready`: required file exists, parses as JSON, has the expected `kind`, and the expected success field is present.
- `blocked`: required file exists but reports failure, blocked status, invalid JSON, unsafe text, or unexpected kind.
- `missing`: required file is absent.
- `partial`: the file is parseable but represents a report-written state that is not final-ready while still useful for review.

The manifest status is:

- `ready` when all required slots are `ready`.
- `blocked` when any required slot is `blocked`.
- `missing` when at least one required slot is missing and none is blocked.
- `partial` when no slot is blocked or missing but at least one required slot is partial.

The CLI returns `0` only for `ready`, otherwise `2`.

## Integration

`final-demo-launch` embeds `live_provider_evidence` for both local and configured modes. Local mode treats it as optional visibility. Configured mode surfaces missing or blocked evidence in the operator checklist and launch phase summary.

The iPhone `Final Launch` panel decodes the embedded report and shows concise rows:

- live evidence overall status
- first missing/blocked slot
- required vs ready evidence count
- whether any listed command may call live providers

The iPhone must not receive provider keys, provider request payloads, raw capture media, local absolute paths, payment links, or signed order URLs.

## Safety

The new report must declare:

```json
{
  "commands_run": false,
  "provider_calls": false,
  "writes_backend_env": false,
  "writes_ios_deploy_config": false,
  "xcode_or_signing": false,
  "global_mutation": false,
  "provider_secrets_in_report": false,
  "raw_media_in_report": false,
  "payment_links_in_report": false,
  "local_paths_in_report": false
}
```

The report may include command strings that would call live providers when the operator explicitly runs them. Those slots must set `requires_live_provider_consent: true`.

## Testing

Backend tests cover:

- missing reports return exit `2` and status `missing`.
- ready fixture reports return exit `0`, summary counts, and redacted safe output.
- invalid JSON and failed report statuses become `blocked`.
- Makefile and CLI route write `.local/live-provider-evidence.json`.
- `final-demo-launch` embeds the evidence report and configured mode adds the missing evidence action.
- sanitizer removes repo paths, `sk-` keys, bearer tokens, data URIs, payment URLs, and local file URLs.

iOS tests cover:

- Swift models decode the embedded live evidence report.
- `FinalLaunchMobileSummaryBuilder` renders ready, missing, blocked, and live-consent rows.
- unsafe strings are redacted before visible summary rows are produced.

Visual/source acceptance covers:

- iPhone final launch source contains the live evidence section.
- mobile final launch readiness acceptance requires the new embedded report and iPhone rows.

## Out Of Scope

- Running Meshy/OpenAI/Treatstock automatically.
- Creating provider keys.
- Placing an order with a print provider.
- Running Xcode, signing, installing on a device, or accepting Apple SDK licenses.
- Verifying subjective 3D quality automatically.
- Replacing the existing local/no-key rehearsal lane.

## Acceptance

P0.120 is accepted when:

- `make live-provider-evidence` writes a sanitized ignored report and exits `2` on the default no-key checkout.
- Fixture-ready configured evidence returns a `ready` report.
- `/v1/final-demo-launch?mode=configured` includes `live_provider_evidence`.
- The iPhone final launch summary shows live evidence readiness.
- Existing local final rehearsal and iPhone launch rehearsal remain no-key and do not run live providers.
- Backend, Swift, final rehearsal, iPhone rehearsal, and visual regression verification all pass.
