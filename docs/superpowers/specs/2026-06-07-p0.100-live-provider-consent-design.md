# P0.100 Live Provider Consent Interface Design

## Goal

Make the iPhone demo explicitly answer whether the app is ready to switch from local stubs to configured live providers before any real Meshy, OpenAI NPC, or print quote calls are attempted.

## User-Facing Behavior

- Add a mobile-first "Live Provider Consent" panel near provider readiness and final launch status.
- The panel has one compact verdict:
  - `WAITING` when backend provider/final launch data has not loaded.
  - `BLOCKED` when required real providers or final resource handoff items are missing.
  - `READY` when real provider readiness, final resources, and resource handoff are all ready.
- The panel exposes four rows:
  - provider readiness for `three_d`, `npc`, and print provider setup
  - live-call policy, including the consent flag when configured acceptance requires it
  - final resources state
  - resource handoff state
- The panel must state that provider keys remain backend-only and that the iPhone does not trigger live provider acceptance by default.

## Privacy And Safety

- Do not show raw key values, bearer tokens, local filesystem paths, capture media URIs, checkout/payment URLs, or raw personal context.
- Treat the iPhone panel as read-only operator guidance; live calls still require the existing backend/CLI consent flag.
- Keep all provider secrets in backend `.env`/resource files, never in app state.

## Non-Goals

- Do not call live Meshy/OpenAI/Treatstock providers from the phone.
- Do not add voice NPC behavior.
- Do not change final acceptance semantics or remove the existing no-live-calls-by-default gate.

## Acceptance

- Swift contract tests cover waiting, blocked, ready, and redaction cases.
- SwiftUI compile gate includes the new panel.
- Backend iOS showcase acceptance requires the new source artifacts.
- A 390px-wide visual regression image shows the panel without overlap or clipped text.
