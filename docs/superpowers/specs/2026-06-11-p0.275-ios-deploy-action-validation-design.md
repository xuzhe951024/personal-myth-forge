# P0.275 iOS Deploy Action Validation Design

## Context

The final showcase is blocked first by physical iPhone deploy evidence. Recent
phases made final-resource actions validation-aware, but several iOS deploy
configuration actions still appear as bare instructions:

- `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig`
- `provide PRODUCT_BUNDLE_IDENTIFIER in Deployment.local.xcconfig`
- `set PMF_BACKEND_BASE_URL to an iPhone-reachable LAN URL`
- `set PMF_FINAL_LAUNCH_MODE to local or configured`

These actions are correct, but they are incomplete handoff steps. After the
operator changes `Deployment.local.xcconfig`, the next safe validation command
is `make mobile-deploy-preflight`.

## Goal

Source and key aggregate reports should emit validation-aware iOS deploy config
operator actions with `; rerun make mobile-deploy-preflight`.

## Non-Goals

- Do not write `Deployment.local.xcconfig`.
- Do not run Xcode, signing, Keychain, or real-device operations.
- Do not call Meshy, OpenAI, Treatstock, Sculpteo, or any live provider.
- Do not change readiness status or blocker calculation.
- Do not change `next_action.command`; `next_action.validation_command` already
  carries machine-readable validation.
- Do not normalize generic backend-health actions such as
  `start backend-device-demo and rerun mobile deploy preflight`.

## Design

Add `add_mobile_deploy_validation_command()` to
`myth_forge_api.operator_actions`. The helper mirrors the final-resource helper:
it recognizes known iOS deploy configuration action roots, preserves existing
`; rerun ...` actions, and appends `; rerun make mobile-deploy-preflight` only
to the known roots.

Apply the helper to the top-level operator actions emitted by:

- `resource_handoff`, because it is the source report for backend/iOS resource
  configuration.
- `mobile_deploy_preflight_evidence`, because it is the source report for the
  physical-device preflight failure.
- `ios_deploy_runbook`, because it assembles deploy slots directly.
- `final_configured_preflight`, because it aggregates configured handoff
  actions for the real-provider iPhone path.

Existing final-resource validation remains unchanged and runs alongside the new
iOS deploy validation where both action families are present.

## Acceptance

- RED/GREEN tests prove the shared helper appends
  `; rerun make mobile-deploy-preflight` to known iOS deploy config actions.
- RED/GREEN tests prove `resource_handoff`, `mobile_deploy_preflight_evidence`,
  `ios_deploy_runbook`, and `final_configured_preflight` expose no known bare
  top-level iOS deploy config actions.
- Existing resource validation tests still pass.
- Refreshed local evidence remains blocked by missing external/device inputs,
  but top-level iOS deploy config actions include the mobile preflight rerun.
- Safety flags remain read-only with no provider calls, env writes, iOS config
  writes, global mutation, Xcode/signing, or Keychain writes.
