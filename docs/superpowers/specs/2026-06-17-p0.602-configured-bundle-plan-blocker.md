# P0.602 Configured Bundle Plan Blocker

## Context

P0.601 makes `final-configured-evidence-plan` preserve concrete child blockers from
`final-configured-preflight`, such as `mobile_deploy_preflight`. The configured live
evidence bundle can still promote a live provider report blocker first, causing the
handoff to ask for paid Meshy/OpenAI evidence while the local iPhone/backend preflight
is still blocked.

## Decision

When `final-configured-evidence-plan` is blocked, `configured-live-evidence-bundle`
should use that plan blocker as its current/first blocker before considering stale or
missing live provider evidence.

## Acceptance

- If the configured plan is blocked by a child `mobile_deploy_preflight` blocker, the
  bundle current blocker preserves the configured plan id, source blocker id, command,
  detail, and validation command.
- The bundle first operator action starts with the local child blocker command instead
  of a `PMF_ALLOW_LIVE_PROVIDER_CALLS=1` command.
- Existing resource-missing, live consent, and ready evidence bundle behavior remains
  unchanged.
- No live provider calls, API key writes, print orders, or Xcode/global actions are
  introduced.
