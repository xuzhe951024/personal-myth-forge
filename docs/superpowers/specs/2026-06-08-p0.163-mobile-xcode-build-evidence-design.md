# P0.163 Mobile Xcode Build Evidence Design

## Context

P0.162 made `make mobile-deploy-preflight-evidence` save a sanitized, repeatable
report for the iPhone-reachable backend and deploy configuration gate. The next
remaining local iOS gate is `make mobile-xcode-build`. Today it is represented in
final acceptance and device evidence reports as an operator action, but its
latest local output is not saved as a first-class evidence artifact.

The current machine blocks at the Apple SDK/Xcode license step:

```text
You have not agreed to the Xcode license agreements.
```

This iteration records that blocker precisely without changing global developer
state. It does not run `sudo xcodebuild -license`, switch `xcode-select`, write
signing credentials, or mutate keychains.

## Goal

Add an explicit Xcode build evidence command that runs the existing safe build
gate, writes a sanitized `.local` report, and lets downstream acceptance reports
surface the exact Xcode blocker when the report exists.

## Non-Goals

- Do not accept the Xcode license or alter `/Library/Developer`.
- Do not sign the iOS app or touch keychains.
- Do not automatically run Xcode from `final-local-report-refresh`.
- Do not change the mobile app project, scheme, or deploy configuration.

## Options Considered

### Option A: Keep the existing manual Xcode action only

This avoids any new code, but it leaves final acceptance unable to distinguish
between missing Xcode, unaccepted license, and a real project build failure
unless the operator reads terminal output manually.

### Option B: Run `make mobile-xcode-build` inside final refresh

This would produce fresh evidence by default, but it makes the safe local refresh
invoke Xcode every time. That is heavier, can trigger local Apple tooling prompts,
and blurs the line between passive report refresh and host build checks.

### Option C: Add an explicit saved Xcode evidence target

This adds a dedicated command:

```bash
make mobile-xcode-build-evidence
```

The command invokes the existing `apps/mobile/ios/scripts/xcode_build_gate.sh`,
saves `services/backend/.local/mobile-xcode-build-evidence.json`, and exits `0`
when the build gate passes or `2` when it is externally blocked. This is the
recommended approach because it preserves operator intent and gives the iOS
device evidence bundle structured, sanitized evidence when the operator has run
the gate.

## Architecture

Add `myth_forge_api.mobile_xcode_build_evidence` with the same shape as
`mobile_deploy_preflight_evidence`:

- `run_mobile_xcode_build_evidence(repo_root=None, command_runner=None)`
- `build_mobile_xcode_build_evidence_report(repo_root, returncode, stdout, stderr)`
- A default runner that executes `sh apps/mobile/ios/scripts/xcode_build_gate.sh`

The report classifies output into these statuses:

- `ready` when the script exits `0`
- `blocked_by_xcode_installation` when `xcodebuild` is missing or exit code is
  `127`
- `blocked_by_apple_sdk_license` when exit code is `69` or output mentions the
  Xcode license agreement
- `xcode_project_build_failed` for other non-zero Xcode build failures
- `xcode_build_gate_failed` for other unexpected command failures

The report stores bounded, sanitized stdout/stderr lines and never includes local
absolute paths, bearer tokens, API keys, raw personal context, or checkout URLs.

## CLI And Make

Add CLI support:

```bash
cd services/backend
uv run python -m myth_forge_api.cli mobile-xcode-build-evidence \
  --repo-root ../.. \
  --output .local/mobile-xcode-build-evidence.json
```

Add Make support:

```bash
make mobile-xcode-build-evidence
```

The Make target is intentionally separate from `final-local-report-refresh`.

## Evidence Bundle Integration

`ios_device_evidence_bundle` will look for
`services/backend/.local/mobile-xcode-build-evidence.json`.

When present and valid, the `xcode_build_gate` slot uses that saved report's
status, classification, detail, and evidence source. The bundle itself remains a
read-only report builder: its safety section still says `commands_run: false`,
`xcode_or_signing: false`, and `describes_global_actions: true`.

When the saved report is absent, the existing final-acceptance-derived behavior
stays unchanged.

## Safety Contract

The new Xcode evidence report declares:

```json
{
  "commands_run": true,
  "provider_calls": false,
  "live_provider_calls": false,
  "writes_backend_env": false,
  "writes_ios_deploy_config": false,
  "global_mutation": false,
  "xcode_or_signing": true,
  "code_signing_allowed": false,
  "keychain_writes": false,
  "provider_secrets_in_report": false,
  "local_paths_in_report": false,
  "writes_derived_data": true,
  "derived_data_path": "apps/mobile/ios/.build/xcode-derived-data"
}
```

The existing build gate already passes `CODE_SIGNING_ALLOWED=NO` and
`CODE_SIGNING_REQUIRED=NO`.

## Acceptance

- `make mobile-xcode-build-evidence` writes a report in `.local`.
- In the current unaccepted-license environment, it exits `2` and classifies the
  blocker as `blocked_by_apple_sdk_license`.
- The report does not leak `/Users/...`, `file://`, bearer tokens, API keys, or
  checkout URLs.
- The iOS device evidence bundle uses the saved Xcode report when it exists.
- Existing backend lint, focused backend tests, Swift package checks, visual
  regression, and final readiness commands retain their expected statuses.
