# P0.77 Final Operator Handoff Design

## Context

P0.76 shows saved final acceptance blockers inside the mobile Final Launch
Status. The remaining gap before an iPhone demo is operational: the Mac operator
still needs to know the exact sequence for resource setup, backend startup,
local final acceptance, iOS deploy preflight, Xcode build gate, and optional
live-provider acceptance.

That sequence already exists across `final_demo_launch`, resource handoff,
final resources preflight, and final acceptance readiness. It is not yet exposed
as one safe handoff report that can be reviewed on the backend and summarized on
the iPhone.

## Decision

Add a read-only `final_operator_handoff` report and embed it in
`/v1/final-demo-launch`.

The report consolidates the final Mac-side operation order into ordered steps
and next actions. It is an instruction and readiness surface only. It never runs
commands, never triggers Xcode, never deploys to a device, never mutates global
machine state, and never calls live providers.

The operator handoff should answer:

- What should the Mac operator do next?
- Which final-demo steps are ready, missing, blocked, manual, or live-only?
- Which command should be run manually on the Mac for each step?
- Is the report safe to show on the iPhone without leaking secrets, file URLs,
  payment links, raw media, or local machine paths?

## Backend Shape

Create `myth_forge_api.final_operator_handoff`.

The report includes:

- `kind`: `final_operator_handoff_report`
- `status`: `ready`, `partial`, or `blocked`
- `mode`: `local` or `configured`
- `summary`: counts for ready, missing, blocked, manual, optional, and live
- `steps`: ordered rows with `id`, `label`, `status`, `command`, `required_for`,
  `source`, and `notes`
- `next_actions`: compact operator actions derived from resource preflight,
  final acceptance readiness, launch phases, and live-provider policy
- `safety`: booleans proving commands were not run, provider calls were not
  made, global mutation did not happen, and unsafe strings were redacted

Step order:

1. `final_resources_preflight`: `make final-resources-preflight`
2. `apply_final_resources`: `make final-apply-resources`
3. `backend_device_server`: `make backend-device-demo`
4. `local_final_acceptance`: run quick local final acceptance and write
   `.local/final-acceptance-local.json`
5. `mobile_deploy_preflight`: `make mobile-deploy-preflight`
6. `xcode_build_gate`: `make mobile-xcode-build`
7. `configured_final_acceptance`: run configured quick final acceptance with
   explicit live-provider consent

`local` mode treats configured provider acceptance as optional/live. `configured`
mode requires real Meshy and OpenAI configuration before the configured
acceptance step is considered ready.

## iOS Shape

Extend mobile models with:

- `FinalOperatorHandoffReport`
- `FinalOperatorHandoffSummary`
- `FinalOperatorHandoffStep`

Extend `FinalDemoLaunchReport` with optional `finalOperatorHandoff`.

Extend `FinalLaunchMobileSummary` with `handoffRows: [String]`. The builder
maps handoff as:

- missing report -> handoff waiting row
- blocked/partial report -> first few `next_actions`, then the first blocked
  or manual step if no actions exist
- ready report -> final operator handoff ready row

`FinalLaunchStatusView` renders a `Next` section below `Acceptance` and above
`Commands`. The section is read-only and contains no executable buttons.

## Testing

Backend TDD:

- missing final resources and missing final acceptance produce blocked handoff
  rows and safe next actions
- blocked saved final acceptance propagates deploy/Xcode next actions
- configured mode marks live configured acceptance as live and consent-gated
- unsafe strings in upstream reports are redacted
- `/v1/final-demo-launch` embeds `final_operator_handoff`

iOS TDD:

- decode `final_operator_handoff` from final launch payload
- mobile summary shows handoff waiting, blocked, and ready rows
- unsafe command/detail text is redacted before display
- SwiftUI source acceptance requires the `Next` section

Acceptance and visual evidence:

- iOS showcase source acceptance requires backend handoff builder, final launch
  embedding, mobile decode, mobile summary rows, and SwiftUI rendering
- static 390x844 visual evidence shows the new `Next` section with operator
  actions without unsafe strings

## Non-Goals

- Do not execute handoff commands from an API request or from the iPhone app.
- Do not run `xcodebuild`, signing, simulator, device deployment, archive, or
  Apple SDK license commands.
- Do not call Meshy, OpenAI, Treatstock, or any live provider.
- Do not expose provider keys, raw capture media, checkout/payment URLs, file
  URLs, or absolute local paths.
