# P0.465 Closure Packet Device-First Actions Design

## Context

The final launch closure packet now promotes the final showcase iPhone blocker into `first_blocker` and `next_action`. Its top-level `operator_actions` still expands the resource-input section first, so the first visible action asks for Meshy/OpenAI keys even when the closure packet says the next blocker is physical iPhone deploy evidence.

## Decision

Promote the closure packet's current `next_action` command to the front of top-level `operator_actions` when that action is an iPhone device handoff. Keep the existing section order and nested resource action groups unchanged so the packet still inventories all external actions.

This keeps the operator reading path consistent:

1. `first_blocker`
2. `next_action`
3. first top-level `operator_actions` row

All three should point to the same iPhone deploy path in local blocked mode.

## Scope

- Add a regression test for closure packet operator action ordering.
- Add a small promotion helper in `final_launch_closure_packet.py`.
- Preserve provider key actions after the promoted device action.
- Preserve validation suffixes and existing normalization.

## Out Of Scope

- No changes to final external action ledger semantics.
- No live provider calls.
- No Xcode signing, Keychain, or global state mutation.
- No real Treatstock quote/order action.

## Acceptance

- The new test fails before implementation and passes after implementation.
- Existing closure packet tests pass.
- Focused final local refresh/demo launch/showcase readiness tests still pass.
- `make backend-lint` and `git diff --check` pass.
