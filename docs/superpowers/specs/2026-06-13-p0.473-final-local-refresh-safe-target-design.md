# P0.473 Final Local Refresh Safe Target Design

## Context

`make final-local-report-refresh` is part of the final local readiness handoff, but the canonical target still calls the CLI directly. When the report correctly returns blocked status for missing external iPhone/provider evidence, the CLI exits `2`; Make then fails even though `services/backend/.local/final-local-report-refresh.json` is usable.

The repo already has `services/backend/scripts/write_final_local_report_refresh.sh`, which accepts exit codes `0` and `2`, writes the same local JSON report, and syncs `final-demo-launch-local.json`. The safer wrapper is only wired to `final-local-report-refresh-local`.

## Decision

Promote the existing wrapper to the canonical `final-local-report-refresh` Make target while keeping `final-local-report-refresh-local` as a compatibility alias to the same script.

## Scope

- Update `Makefile` so both `final-local-report-refresh` and `final-local-report-refresh-local` invoke `services/backend/scripts/write_final_local_report_refresh.sh`.
- Update focused tests to expect the canonical target to use the wrapper, not the raw CLI.
- Update source acceptance gates so final iOS/showcase checks require the canonical target and alias to use the safe wrapper.
- Update `docs/iteration-roadmap.md` with P0.473.

## Non-Goals

- Do not run real provider calls.
- Do not modify iOS signing, Keychain, Xcode global settings, or device state.
- Do not change final readiness semantics; missing `DEVELOPMENT_TEAM`, provider keys, or iPhone evidence must remain blocked.

## Verification

- RED: focused Make dry-run/source-gate tests fail while the canonical target still uses the raw CLI.
- GREEN: focused tests pass after Makefile and source gates use the wrapper.
- Runtime: `make final-local-report-refresh` returns `0` while writing `.local/final-local-report-refresh.json` with blocked status when external resources are absent.
