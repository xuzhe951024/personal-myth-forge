# P0.504 Final Demo Launch Backend Priority Design

## Context

`final-showcase-readiness.json` now preserves the local backend device-demo handoff near the top of its operator actions. The local final demo launch report still surfaces provider and print handoffs before the same device-demo action:

`start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight`

For the iPhone-first final showcase, the local demo launch report should guide the operator through physical-device setup before provider/key work. Provider and print handoffs remain important, but they do not unblock the immediate iPhone LAN/backend health path.

## Scope

This iteration only changes local final demo launch operator-action ordering. It does not start the backend server, write iOS deploy config, accept Xcode licenses, call Meshy/OpenAI/Treatstock, or place print orders.

## Design

`final_demo_launch._operator_actions()` should keep the concrete first blocker first, then prioritize the canonical backend-device-demo handoff before provider and print actions imported from final showcase readiness. Existing dedupe, provider chain, print quote guard, deploy writer, LAN backend URL, and Xcode evidence behaviors must remain intact.

The canonical action remains:

`start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight`

## Testing

Add a regression test that monkeypatches final showcase readiness to expose provider and print handoff actions while the local final demo launch checklist includes the backend-device-demo action. The test must fail before implementation because provider/print actions currently appear before backend-device-demo.
