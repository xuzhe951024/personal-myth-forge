# P0.569 Device Evidence Rehearsal Action Design

## Context

`final-showcase-readiness` now promotes the saved iOS device launch rehearsal readiness child action to the top-level next action. The separate `ios-device-evidence-bundle` still reads the raw saved rehearsal report first, so it can describe the same iOS deploy blocker as a live-provider action. That creates conflicting handoff evidence for the final iPhone path.

## Design

The iOS device evidence bundle should prefer the readiness report's concrete device next action for the launch rehearsal slot:

- Keep the raw saved `ios-device-launch-rehearsal.json` as the source artifact.
- Use `ios-device-launch-rehearsal-readiness.json` as the interpretation layer for next action, detail, and device action bundle metadata.
- When readiness exposes a concrete `next_action` or `device_action_bundle.first_action.next_action`, set the bundle slot command and top-level next action to that command.
- Preserve `make ios-device-launch-rehearsal` as the validation command so the operator still refreshes the full saved evidence after taking the concrete action.
- Do not execute live provider calls, Xcode, signing, Keychain writes, or global state changes.

## Testing

Add a regression test where the raw launch rehearsal report contains a live-provider operator action, but the readiness report exposes a backend/mobile-preflight device action. The bundle should pick the device action, not the live provider action.
