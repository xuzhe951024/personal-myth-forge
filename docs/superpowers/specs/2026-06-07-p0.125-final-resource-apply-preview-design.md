# P0.125 Final Resource Apply Preview Design

## Problem

P0.124 tells the operator which final resources are required. The actual apply
path is still a shell script:

```bash
make final-apply-resources
```

That script validates `services/backend/.local/final-resources.env`, then writes
ignored backend and iOS config files. Before running the write step, the
operator needs a structured dry-run report that says exactly which destinations
would be written, which key slots are ready/missing/blocked, which command would
perform the write, and whether any safety boundary would be crossed.

## Goal

Add a read-only `final_resource_apply_preview_report`. It previews the effect of
`make final-apply-resources` without writing files, running shell writers,
calling providers, touching Xcode, changing signing state, or exposing secret
values.

## Non-Goals

- Do not change `apply_final_resources.sh` behavior.
- Do not write `services/backend/.env`.
- Do not write `apps/mobile/ios/Config/Deployment.local.xcconfig`.
- Do not run Xcode, open an iPhone connection, or call Meshy/OpenAI/Treatstock.
- Do not include raw API key values, raw LAN URLs, local absolute paths, payment
  links, or raw media data in the report.

## Report Contract

The report has:

- `kind`: `final_resource_apply_preview_report`
- `status`: `ready`, `blocked`, or `missing`
- `summary`: counts for ready/missing/blocked/secret/backend/ios/print/write_targets
- `resources_file`: sanitized resource file status
- `write_targets`: backend and iOS destination groups
- `write_targets_by_id`: lookup by `backend_env` and `ios_deploy_config`
- `operator_actions`: safe next actions
- `commands`: `make final-resource-apply-preview`, `make final-resources-preflight`, and `make final-apply-resources`
- `source_reports`: compact statuses from `final_resources_preflight` and
  `final_resource_requirements`
- `safety`: all write/live/global mutation fields are false

Each write target includes:

- `id`
- `label`
- `destination`
- `writer`
- `status`
- `command`
- `slots`
- `blocked_by`
- `notes`

Each slot includes:

- `id`
- `status`
- `required`
- `secret`
- `configured`
- `classification`
- `redacted`
- `writes`

`writes` is a list of output config keys, not values.

## Status Rules

- Missing final resources file -> report `missing`; no write targets are ready.
- Any missing required resource or blocked preflight item -> report `blocked`.
- Valid local resources with required values -> report `ready`.
- `PRINT_PROVIDER=treatstock` makes `TREATSTOCK_API_KEY` required; otherwise it
  remains optional.

## iPhone Surface

`FinalDemoLaunchReport` decodes optional `final_resource_apply_preview`. The
iPhone Final Launch panel renders `Apply Preview` rows:

- overall preview status and target count
- first blocked or missing slot
- backend/iOS destination labels
- safe command row

The app still does not run the command or receive secrets.

## Acceptance

- Backend tests prove missing, blocked loopback, Treatstock conditional, ready,
  no writes, and no secret leaks.
- CLI/Make tests prove `final-resource-apply-preview` writes JSON and returns
  exit 2 for missing/blocked resources.
- Final Demo Launch embeds the preview report.
- Swift contract tests decode the report and render summary rows.
- Source acceptance gains `final_resource_apply_preview` so the feature stays
  wired through backend, CLI, Make, Final Launch, and iPhone UI.
