# P0.138 Source-Scoped Freshness Design

## Context

The final showcase readiness report currently marks saved local evidence stale
when the evidence file modification time is older than the latest git `HEAD`
commit. That is conservative, but it creates noisy blockers after documentation
or planning commits that do not change the runnable backend, iOS app, scripts,
or tests.

P0.138 makes freshness compare saved evidence against the latest relevant
product-source commit instead of every commit. This keeps the final showcase
handoff honest when code changes, while reducing false stale blockers from
docs-only bookkeeping commits.

## Scope

Apply source-scoped freshness to the saved evidence reports that directly feed
the final showcase readiness path:

- final acceptance readiness
- full-showcase visual regression readiness
- iOS device launch rehearsal readiness

Do not change provider behavior, run live Meshy/OpenAI/Treatstock calls, write
global settings, accept Xcode licenses, edit signing, or write secrets.

## Product Source Scope

Use one shared backend helper that treats these repository paths as freshness
inputs:

- `Makefile`
- `services/backend/pyproject.toml`
- `services/backend/src`
- `services/backend/tests`
- `services/backend/scripts`
- `apps/mobile/ios`
- `packages`

The helper asks git for the latest commit touching that path set. A docs-only
commit after the saved evidence timestamp should not stale the evidence. A
commit touching any product-source path after the evidence timestamp should
still stale it.

If git is unavailable, keep the existing `unknown` / `git_unavailable` behavior.
If the evidence file is missing, keep `unknown` / `source_missing`.

## Data Contract

Preserve the existing freshness shape used by mobile and backend reports:

- `status`
- `classification`
- `checked_against`
- `source_modified_at`
- `current_revision`
- `current_revision_committed_at`

Set `checked_against` to `git_product_sources` when git metadata is available
or attempted for existing evidence. Keep missing-source reports explicit with
`source_missing`.

Add a bounded `source_scope` object:

```json
{
  "kind": "git_product_sources",
  "paths": ["Makefile", "services/backend/pyproject.toml"]
}
```

Do not include absolute local paths, raw media, secrets, payment links, or user
private context in the report.

## Architecture

Create `myth_forge_api.source_freshness` with:

- `PRODUCT_SOURCE_PATHS`
- `git_product_source_metadata(repo_root)`
- `freshness_payload(...)`

The three readiness modules will call the helper instead of duplicating
`git log HEAD` logic. Each module remains responsible for its report-specific
blocker ID, command text, and operator action.

The helper will not run tests or providers. It only reads git metadata through
bounded subprocess calls.

## Testing

Use TDD with focused RED tests first:

1. Final acceptance readiness remains fresh when a docs-only commit is newer
   than the saved evidence, but the latest product-source commit is older.
2. Visual regression readiness remains fresh under the same docs-only commit
   condition.
3. iOS device launch rehearsal readiness remains fresh under the same docs-only
   commit condition.
4. At least one stale test still proves that a product-source commit newer than
   saved evidence blocks the readiness report.

Then run the focused backend readiness tests, backend lint/test, Swift contract
checks, compile check, and `git diff --check` before merge.
