# P0.241 iOS Device Evidence Preflight Detail Design

## Context

`final-demo-launch-local` and `final-showcase-readiness` now surface concrete mobile deploy preflight blockers from `services/backend/.local/mobile-deploy-preflight-evidence.json`. The lower-level `ios-device-evidence-bundle` report still derives backend and mobile preflight slot details from `final-acceptance-local.json`, which can collapse the issue to a generic message such as "start backend-device-demo and rerun mobile deploy preflight".

The iPhone Device Preflight UI already renders `ios_device_evidence_bundle.evidence_slots`. If those slots use the saved mobile preflight evidence directly, the existing UI can show the concrete blocked checks without adding mobile model fields.

## Goal

Make `ios-device-evidence-bundle` prefer saved `mobile-deploy-preflight-evidence.json` for the `backend_device_server` and `mobile_deploy_preflight` slots when that report exists.

## Non-Goals

- Do not add a new iOS model field for `first_action`.
- Do not run mobile deploy preflight, Xcode, signing, Keychain, or any global Mac mutation.
- Do not call Meshy, OpenAI, Treatstock, Sculpteo, or any live provider.
- Do not change Xcode build evidence behavior.

## Approaches Considered

1. Prefer saved mobile preflight evidence inside the existing backend/preflight slots.
   This is selected. It keeps the report shape stable and makes current iPhone UI rows more actionable.

2. Add `first_action` to `ios_device_evidence_bundle`.
   This would be explicit but requires Swift model and UI work. The current UI already renders the first blocked slot, so a shape change is unnecessary for this phase.

3. Change final acceptance refresh behavior so final acceptance always embeds direct preflight evidence.
   This is broader and risks mixing report responsibilities. `ios-device-evidence-bundle` is the right place to consolidate device evidence.

## Design

`build_ios_device_evidence_bundle_report` will load `services/backend/.local/mobile-deploy-preflight-evidence.json` with the same safe optional-report pattern used for Xcode evidence. The report will include a `mobile_deploy_preflight_evidence` source summary when present.

For `backend_device_server`:

- If saved mobile preflight evidence exists and is not ready, status is blocked.
- Detail prefers the `backend_health` or `backend_base_url` check detail.
- Evidence source becomes `services/backend/.local/mobile-deploy-preflight-evidence.json`.

For `mobile_deploy_preflight`:

- If saved mobile preflight evidence exists, status follows that evidence status.
- Detail summarizes non-ready check details, such as `Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable`.
- Evidence source becomes `services/backend/.local/mobile-deploy-preflight-evidence.json`.

If saved preflight evidence is missing or malformed, existing final-acceptance fallback behavior remains unchanged.

## Testing

Add a backend RED/GREEN test:

- Arrange blocked final acceptance with only a generic `mobile_deploy_preflight` blocker.
- Write saved blocked `mobile-deploy-preflight-evidence.json` with `development_team` and `backend_base_url` checks.
- Build `ios-device-evidence-bundle`.
- Assert backend slot detail is `PMF_BACKEND_BASE_URL must be iPhone-reachable`.
- Assert mobile preflight slot detail summarizes both blocked checks.
- Assert both slots use `services/backend/.local/mobile-deploy-preflight-evidence.json`.
- Assert source reports include `mobile_deploy_preflight_evidence`.
- Assert report text does not leak `MESHY_API_KEY`.
