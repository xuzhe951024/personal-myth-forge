# P0.613 Showcase Demo Live Confirmation

## Problem

`final-showcase-readiness` and `final-demo-launch` expose the final live Meshy/OpenAI blocker to the operator and iPhone-facing launch state, but their top-level rows only carried `requires_live_provider_consent`. Other final handoff reports already carry explicit user-confirmation, cost-consent, and live-provider-call flags.

## Scope

- For final showcase live provider capability rows, derive `requires_cost_consent`, `live_provider_call`, and `requires_user_confirmation` whenever `requires_live_provider_consent` is true.
- Preserve those fields into capability `next_action`, top-level `first_blocker`, and top-level `next_action`.
- Preserve `requires_user_confirmation` when final demo launch compacts nested showcase blockers.
- Do not run live provider commands or any external paid/provider action.

## Acceptance

- Focused RED tests prove the showcase and demo launch rows lack the explicit confirmation fields.
- GREEN implementation preserves the fields through the report layers.
- Full showcase/demo launch tests and backend regression gates remain green.
