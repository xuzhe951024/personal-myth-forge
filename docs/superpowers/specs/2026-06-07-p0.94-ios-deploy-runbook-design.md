# P0.94 iOS Deploy Runbook Design

## Context

P0.25 through P0.93 created the project-local pieces needed for a physical iPhone
demo: ignored iOS deploy config, LAN backend preflight, Xcode build gate, final
resource apply, final acceptance readiness, NPC Agent evaluation readiness, and a
Mac-side final operator handoff.

The remaining operator gap is that the deploy sequence is still spread across
several reports, scripts, and README sections. Before the user provides final
keys and signing values, the repo should be able to produce a single sanitized
runbook that says exactly what is configured, what is missing, which local files
are the handoff inputs, and which commands to run on the Mac. The runbook must
not execute those commands.

## Design

Add a backend read-only report builder named `ios_deploy_runbook` with a CLI
command and Make target:

```bash
make ios-deploy-runbook
cd services/backend && uv run python -m myth_forge_api.cli ios-deploy-runbook \
  --mode local \
  --repo-root ../.. \
  --output .local/ios-deploy-runbook-local.json
```

The report will include:

- `kind`: `ios_deploy_runbook_report`
- `mode`: `local` or `configured`
- `status`: `ready`, `blocked`, or `partial`
- `input_slots`: required local inputs for Team ID, bundle id, backend URL, final
  launch mode, backend provider env, final acceptance, and NPC evaluation
- `command_sequence`: ordered Mac-side commands, starting with final resource
  preflight/apply and ending with deploy preflight and Xcode build gate
- `operator_actions`: deduplicated next actions from final resources preflight,
  final acceptance readiness, NPC evaluation readiness, deploy preflight status,
  and Xcode build gate
- `safety`: no command execution, no provider calls, no global mutation, no
  provider secrets, no raw media, no payment links, and no local absolute paths

Status rules:

- `ready` only when required input slots are ready and the command sequence has
  no blocked or missing non-manual command.
- `blocked` when any required input slot or command is missing/blocked.
- `partial` when the only remaining gate is manual Xcode build or configured
  live-provider acceptance consent.

## Integration

`build_final_demo_launch_report` will embed the runbook as
`ios_deploy_runbook`. `build_final_operator_handoff_report` will receive that
runbook and add a `ios_deploy_runbook` step before `mobile_deploy_preflight`.
When the runbook is blocked or missing, its sanitized operator actions are
included in final handoff next actions.

The iPhone app does not execute the runbook. Existing mobile summary code can
render the handoff step and command rows as plain text. The Swift project guard
will assert that the backend exposes the runbook and that the Make target exists.

## Tests

Backend tests will be written first:

- blocked runbook reports missing ignored `final-resources.env`, iOS deploy
  config values, final acceptance, and NPC evaluation without leaking paths or
  provider keys
- ready local inputs produce an ordered command sequence including
  `make final-resources-preflight`, `make final-apply-resources`,
  `make backend-device-demo`, `make mobile-deploy-preflight`, and
  `make mobile-xcode-build`
- final demo launch embeds `ios_deploy_runbook`
- final operator handoff includes an `ios_deploy_runbook` step before
  `mobile_deploy_preflight`
- CLI writes the report to the requested output path
- iOS showcase source acceptance and Swift project checks require the source
  wires and Make target

## Non-Goals

- No Xcode invocation
- No simulator or device deployment
- No signing, keychain, Apple ID, or provisioning profile changes
- No `xcode-select`, license acceptance, firewall, or network configuration
- No Meshy, OpenAI, Treatstock, or other live provider calls
- No secrets or raw private context in reports
