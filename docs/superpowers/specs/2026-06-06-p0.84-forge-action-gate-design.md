# P0.84 Forge Action Gate Design

Date: 2026-06-06

## Problem

P0.83 added `ForgeReadinessSummary.canForge`, but the `Forge Myth` button still only checks media readiness and context approval. That leaves a mismatch: the panel can say backend/provider/NPC setup needs attention while the primary action remains tappable. For a final iPhone demo, the primary action should follow the same readiness contract the operator sees.

## Goal

Add a mobile-core Forge action gate that controls the `Forge Myth` button and explains the next required action without adding provider calls, changing backend behavior, or blocking the local no-key demo when the readiness summary says local demo is ready.

## Non-Goals

- Do not add new Meshy, OpenAI, Treatstock, Xcode, signing, or device calls.
- Do not change the backend forge pipeline.
- Do not require configured real providers for local demo mode.
- Do not remove the existing flow phase status line.

## Approach

Create a small `ForgeActionGate` model and `ForgeActionGateBuilder` in the mobile core. The builder consumes:

- `isMediaReadyForUpload`
- `ContextCapsuleReview`
- `ForgeReadinessSummary`

The gate returns:

- `isEnabled`: whether the primary Forge button is enabled
- `title`: current button label
- `detail`: short visible next-action copy
- `disabledReason`: machine-readable reason for tests, future analytics, and source acceptance

Priority order is intentionally user-action oriented:

1. Missing capture blocks first.
2. Unapproved context capsule blocks second.
3. Forge readiness blocks third, including provider preflight errors, missing configured env, or NPC setup failures.
4. Ready state enables the button and labels the active route as `local_demo` or `configured`.

All user-visible gate copy must be redacted with the same privacy stance as readiness summaries: no provider secrets, raw local paths, local capture URIs, payment URLs, or authorization text.

## UI Behavior

`CaptureFormView` receives `ForgeActionGate` from `ForgeRootView`. The `Forge Myth` button uses `forgeActionGate.title` and is disabled when `!forgeActionGate.isEnabled`. A compact caption below the button shows `forgeActionGate.detail`; the existing flow phase status remains below it for upload/session progress.

The Forge readiness panel stays informational. The action gate is the single source for primary button enablement, which removes the current mismatch between the panel and the button.

## Tests

Swift core contract tests cover:

- capture missing disables the action
- context unapproved disables the action
- readiness `canForge == false` disables the action
- local demo readiness enables the action
- unsafe detail text is redacted

Source acceptance gates cover:

- `ForgeActionGateBuilder` exists
- `CaptureFormView` uses `forgeActionGate.isEnabled`
- `CaptureFormView` renders `forgeActionGate.detail`
- `ForgeRootView` builds and passes the gate

## Visual Regression

Add a static 390x844 verification page showing:

- Forge Readiness in a needs-attention provider state
- `Forge Myth` disabled
- a concise reason such as missing `MESHY_API_KEY`
- no text overlap or right-edge clipping

## Safety

This iteration remains source/local only. It must not run global Xcode commands, mutate signing state, call live providers, expose provider secrets, or include raw media in reports.
