# P0.200 Mobile Resource Requirements First Blocker Design

## Purpose

The backend now emits `final_resource_requirements.first_blocker`, but the iOS model ignores it and the Resource Requirements section still derives blockers from requirement rows. Decoding the structured blocker gives the iPhone the same actionable command/detail that upstream final launch receipts now use.

## Scope

- Add `FinalResourceRequirementsFirstBlocker` to Swift models.
- Add optional `firstBlocker` to `FinalResourceRequirementsReport`.
- Make `FinalLaunchMobileSummary.resourceRequirementRows` prefer that blocker when present.
- Add Swift contract tests and ProjectChecks source gates.

## Acceptance

- Final launch payload decoding exposes `finalResourceRequirements.firstBlocker`.
- Resource Requirements rows include `MESHY_API_KEY`, `provide MESHY_API_KEY in final-resources.env`, and the blocker detail.
- Ready/legacy payloads without `first_blocker` continue to render via existing fallback.
- Redaction remains covered by existing summary sanitization.
