# P0.238 Local Launch Device First Blocker Design

## Context

The final goal still requires real provider evidence and physical iPhone evidence. The current local launch report preserves those blockers, but `final-demo-launch-local` currently chooses `apply_final_resources` as the top-level first blocker when Meshy/OpenAI/iOS values are missing.

That ordering is useful for configured/live acceptance, but it is misleading for the no-key local iPhone showcase path. Local launch can still demonstrate capture, local 3D handoff, local NPC runtime, print review, and readiness panels without provider keys. The next concrete blocker for putting that local showcase on an iPhone is the device path: local deploy config, LAN backend health, iOS preflight, and rehearsal.

## Goal

For `mode="local"`, make the final demo launch first blocker point to the first local device/showcase blocker instead of the provider-key apply phase when local no-key launch can continue.

## Requirements

- Configured mode remains unchanged: missing final resources still block configured launch first.
- Local mode still embeds final resource requirements, fill guide, apply preview, and provider-key blockers.
- Local mode still keeps `apply_final_resources` in `launch_phases` and command lists.
- Local mode `first_blocker` should skip `apply_final_resources` when it is missing, blocked, or partial because local no-key acceptance can continue.
- Local mode should then surface the next local device blocker, currently `mobile_deploy_preflight` when deploy config or backend health is not ready.
- Local mode device blocker detail must not append provider-key nested hints such as `MESHY_API_KEY`.
- Reports remain sanitized and read-only.

## Non-Goals

- Do not mark `ios_deployable`, provider readiness, or configured live evidence ready.
- Do not write `.env`, `Deployment.local.xcconfig`, Keychain, Xcode settings, or provider credentials.
- Do not run live Meshy/OpenAI/Treatstock calls or place print orders.

## Acceptance

- New backend test proves local launch first blocker is `mobile_deploy_preflight` when no-key local launch can continue and provider keys are missing.
- New backend test proves the local device blocker detail does not include provider-key nested hints.
- Existing configured-mode test still proves configured launch first blocker is `apply_final_resources`.
- Focused final demo launch tests pass.
- Safe local report refresh still writes blocked reports while external resources and physical-device evidence remain missing.
