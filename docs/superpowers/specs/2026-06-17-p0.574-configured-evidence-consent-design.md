# P0.574 Configured Evidence Consent Design

## Context

The final showcase readiness, live provider evidence, and closure packet now
route configured Meshy/OpenAI work through an explicit consent command:

```text
PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence
```

`configured-live-evidence-bundle.json` still exposes a weaker current blocker
for the same condition:

```text
make final-acceptance-configured
```

That report is part of the final handoff path. It should not surface a bare
command that may call Meshy or OpenAI.

## Design

When the configured bundle is blocked by the live provider evidence report, it
should prefer `live_provider_evidence.next_action` over the raw
`first_blocker.command`.

The configured bundle should:

- keep the blocker id and label from `live_provider_evidence.first_blocker`;
- use the consent-aware command from `live_provider_evidence.next_action`;
- preserve `validation_command` when present;
- surface the same command in `current_blocker`, `first_blocker`, `next_action`,
  and `operator_actions`;
- keep all ready and ready-to-run behavior unchanged.

## Safety

This phase only changes report text and action metadata. It does not run live
providers, write secrets, invoke Xcode, change signing state, or place print
orders.

## Testing

Add a regression test that stubs live provider evidence with a bare
`first_blocker.command` and a consent-aware `next_action.command`. The configured
bundle should choose the consent-aware command everywhere an operator would see
the next step.
