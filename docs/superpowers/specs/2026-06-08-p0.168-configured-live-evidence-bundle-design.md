# P0.168 Configured Live Evidence Bundle Design

## Context

The configured acceptance path already has three separate reports:

- `final_configured_evidence_plan`: ordered commands, consent gates, and current step status.
- `live_provider_evidence`: saved evidence slots for provider handoff, Meshy 3D evaluation, OpenAI NPC evaluation, configured final acceptance, and configured launch packet.
- `final_demo_launch`: the broader launch packet that embeds both reports.

These reports answer the right questions, but the operator still has to combine them mentally when preparing the final key-backed acceptance run. P0.168 adds a read-only bundle report that packages the configured evidence plan, evidence file manifest, risk summary, current blocker, and next commands into one handoff artifact.

## Goal

Add `configured_live_evidence_bundle`, a backend-only report and CLI/Make target that shows exactly what to run after final resources are filled, which `.local` evidence files should exist, which steps may call live providers, and whether the bundle is ready for final configured acceptance.

## Non-Goals

- Do not call Meshy, OpenAI, Treatstock, Sculpteo, Apple, Xcode, keychain, or any live provider.
- Do not write backend `.env`, iOS deploy config, provider keys, media files, payment links, or global state.
- Do not change iOS UI in this iteration.
- Do not make configured acceptance pass without real provider evidence.

## Approach

### Option A: Extend `final_configured_evidence_plan`

Add bundle fields directly to the existing plan report. This is compact, but the plan would mix step planning with final artifact packaging and make the report harder to read.

### Option B: Extend `live_provider_evidence`

Add plan and command sequence fields directly to live evidence. This keeps evidence files central, but live evidence should remain a validator of saved reports rather than an operator runbook.

### Option C: Add A Dedicated Bundle Report

Create `configured_live_evidence_bundle.py` that composes the two existing reports. It keeps the plan and evidence validators focused while adding a top-level artifact for final acceptance handoff.

This is the selected approach. It is small, testable, and matches the existing report-builder pattern.

## Report Contract

`build_configured_live_evidence_bundle_report(repo_root, settings, allow_live_provider_calls=False)` returns:

- `kind`: `configured_live_evidence_bundle_report`
- `status`: `ready`, `ready_to_run`, `consent_required`, or `blocked`
- `summary`: counts for evidence files, blocked steps, live-provider steps, cost-risk steps, and commands run
- `current_blocker`: first blocking or consent step from the composed reports
- `evidence_files`: ordered rows with `id`, `label`, `path`, `status`, `expected_kind`, `command`, `requires_live_provider_consent`, and `detail`
- `command_sequence`: ordered rows copied from the configured evidence plan with `id`, `command`, `status`, and risk/write flags
- `operator_actions`: concise next actions with the first action tied to the current blocker
- `source_reports`: summaries for the configured evidence plan and live provider evidence reports
- `live_call_policy`: explicit confirmation that this bundle does not call live providers and that `--allow-live-provider-calls` only changes planning status
- `safety`: redaction and mutation guarantees

All text must be sanitized using the same style as the existing reports: no provider secrets, raw media, payment links, local filesystem paths, or LAN/private URLs should leak.

## CLI And Make Target

Add:

```bash
cd services/backend && uv run python -m myth_forge_api.cli configured-live-evidence-bundle --repo-root ../.. --output .local/configured-live-evidence-bundle.json
make configured-live-evidence-bundle
```

The command is read-only except for writing the requested JSON output. It should exit `0` only when the bundle status is `ready`; otherwise it should exit `2`, matching the existing readiness report convention.

## Testing

Use TDD:

1. Add backend unit tests proving missing resources produce a blocked bundle without running commands.
2. Add tests for ready resource configuration showing consent-required or ready-to-run command rows without provider calls.
3. Add tests for fully ready configured evidence producing `status=ready`.
4. Add CLI output and Make dry-run coverage.
5. Run targeted tests, `make backend-lint`, and `make backend-test`.

## Acceptance

This iteration is accepted when:

- `make configured-live-evidence-bundle` exists and writes `.local/configured-live-evidence-bundle.json`.
- The bundle report includes evidence file manifest, command sequence, current blocker, operator actions, source report summaries, live call policy, and safety metadata.
- Missing or unsafe saved reports are redacted.
- Automated tests prove the report does not execute live provider commands.
