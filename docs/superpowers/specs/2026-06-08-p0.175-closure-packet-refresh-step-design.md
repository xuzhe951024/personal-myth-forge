# P0.175 Closure Packet Refresh Step Design

## Context

`final-local-report-refresh` is the safe local command that refreshes ignored
evidence files before final showcase review. It already writes final resource
reports, local 3D/NPC evaluation reports, iOS device evidence, configured live
evidence bundle, print fulfillment readiness, external action ledger, and final
showcase readiness.

The final launch closure packet is still outside that refresh chain. The Make
target exists, but a saved `services/backend/.local/final-launch-closure-packet.json`
can remain stale after backend or mobile contract changes. After P0.173/P0.174,
that stale file can still show the old five-section closure packet even though
the code and mobile contract require the `configured_evidence_bundle` section.

## Goal

Make `final-local-report-refresh` regenerate the final launch closure packet as
part of the safe local evidence bundle, so the final operator handoff cannot
accidentally rely on stale closure evidence.

## Design

`final_local_report_refresh.py` will add a `final_launch_closure_packet` step.
The step will call `build_final_launch_closure_packet_report(repo_root=repo_root)`
and write `services/backend/.local/final-launch-closure-packet.json`.

The step will run after `configured_live_evidence_bundle`,
`print_fulfillment_readiness`, `final_external_action_ledger`, and
`final_showcase_readiness`. This gives the closure packet access to the current
saved configured bundle and lets the refresh report record the closure packet as
the final operator-facing packet after the readiness ledger.

The closure packet builder remains read-only with respect to external systems.
It does not call live providers, run Xcode, mutate global developer settings,
write resource env files, write iOS deploy config, or call shell commands. The
refresh command only writes the ignored `.local` JSON report.

`ios_showcase_acceptance.py` will also require that
`final_local_report_refresh.py` references the closure packet builder and output
path, so source acceptance protects the refresh chain.

## Alternatives Considered

1. Tell operators to run `make final-launch-closure-packet` separately.
   Rejected because `final-local-report-refresh` is already the safe evidence
   bundling command. Splitting the final closure packet out allows stale
   operator evidence.

2. Build the closure packet before final showcase readiness.
   Rejected because the closure packet is the final operator-facing packet; it
   should appear after the readiness ledger in the refresh sequence.

3. Make `final-showcase-readiness` write the closure packet as a side effect.
   Rejected because readiness builders should stay report builders. The refresh
   orchestrator is the right place to write ignored evidence files.

## Acceptance

- `final-local-report-refresh` writes
  `services/backend/.local/final-launch-closure-packet.json`.
- The refresh report includes a `final_launch_closure_packet` step.
- The closure packet step runs after configured live evidence bundle, print
  fulfillment readiness, final external action ledger, and final showcase
  readiness.
- The saved closure packet has kind `final_launch_closure_packet_report`,
  includes `sections_by_id.configured_evidence_bundle`, and includes command
  `make configured-live-evidence-bundle`.
- The refresh report safety remains unchanged: no live provider calls, no global
  mutation, no Xcode/signing, no keychain writes, no backend env or iOS deploy
  config writes.
- iOS showcase source acceptance requires the refresh orchestrator to include
  the closure packet builder and `.local/final-launch-closure-packet.json`
  output path.

## Test Strategy

Use TDD in `services/backend/tests/test_final_local_report_refresh.py`:

1. Extend the safe reports test to require the closure packet output file.
2. Extend the order/freshness test to assert step order and saved closure packet
   contents.
3. Run the focused test and watch it fail because the step is absent.
4. Add source acceptance requirements and watch the focused iOS showcase
   acceptance test fail until source strings are implemented.
5. Add the smallest refresh step and source requirement changes.
6. Re-run focused tests, backend lint, and backend test suite before merging.

## Self-Review

- Placeholder scan: no TBD/TODO/open placeholders.
- Scope check: one safe local report refresh step plus source acceptance; no live
  provider, mobile UI, Xcode, signing, or resource writer behavior changes.
- Safety check: the step only writes an ignored `.local` report and reuses a
  read-only report builder.
- Ambiguity check: step id, output file, ordering, report kind, and source
  acceptance strings are explicit.
