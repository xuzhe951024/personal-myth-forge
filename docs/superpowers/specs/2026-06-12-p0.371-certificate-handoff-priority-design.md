# P0.371 iOS Certificate Handoff Priority Design

## Goal

Make the iOS device launch certificate preserve provider and Treatstock print handoff actions from the final handoff index, so every final physical-iPhone handoff artifact exposes the same remaining external lanes.

## Current Evidence

- `final-handoff-index.json` now exposes iOS deploy, provider handoff, and Treatstock print quote handoff near the top.
- `ios-device-launch-rehearsal.json` also exposes those actions.
- `ios-device-launch-certificate.json` still shows only final-handoff-index refresh, iOS deploy config, runbook, backend server, mobile preflight, and Xcode actions.

## Behavior

`ios_device_launch_certificate` should:

- Keep `run make final-handoff-index` first when that gate is blocked.
- Copy only provider and print handoff actions from `final_handoff_index.operator_actions` into the certificate gate.
- Promote copied provider/print handoff actions immediately after the first action and before generic iOS deploy/runbook/server actions.
- Preserve existing blocker, next-action, safety, and sanitization behavior.

## Non-Goals

- Do not call live providers or Treatstock.
- Do not run Xcode, signing, or backend server commands.
- Do not mark certificate readiness as complete without external iPhone evidence.
- Do not copy every final handoff index action; only provider and print handoff actions.

## Acceptance

- A focused RED/GREEN test proves certificate top-level `operator_actions` includes provider and Treatstock handoff from final handoff index.
- Existing certificate tests remain green.
- Adjacent final handoff index and rehearsal tests remain green.
- Refreshed `ios-device-launch-certificate.json` shows provider/print handoff near the top while remaining blocked on external evidence.
