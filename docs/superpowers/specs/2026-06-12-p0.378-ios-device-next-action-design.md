# P0.378 iOS Device Next Action Design

## Context

The backend final showcase readiness report already embeds a child
`next_action` inside blocked iOS device actions. For example, the
`start_backend_device_demo` action can carry the concrete deploy writer command
from `mobile-deploy-preflight-evidence`:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto
```

The iOS model currently decodes device action evidence fields but drops this
child `next_action`. The Final Launch summary can therefore show the evidence
status and validation command, but not the most specific next command for the
blocked device action.

## Decision

Decode `FinalShowcaseDeviceAction.nextAction` as an optional
`FinalShowcaseReadinessNextAction`. Render it in the existing device action row
as:

```text
next <command>
```

The row keeps existing evidence and validation text. The new text is additive
and sanitized through the existing row sanitizer.

## Scope

In scope:

- Add optional `nextAction` to `FinalShowcaseDeviceAction`.
- Add test fixture JSON for the existing backend child `next_action` field.
- Render child next action command/detail in Final Launch showcase device rows.
- Add focused Swift contract coverage.

Out of scope:

- Backend report generation changes.
- Running Xcode signing/build gates.
- Writing deploy config.
- Device launch or provider calls.

## Safety

This phase only changes Swift decoding, summary text, docs, and contract tests.
It does not write secrets, run provider calls, mutate Xcode/global state, or
create deploy configuration.
