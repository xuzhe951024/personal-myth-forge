# P0.551 Live Provider Evidence Next Action Design

## Context

The final objective still needs configured Meshy/OpenAI/print provider evidence before the iPhone showcase can be called complete. `live-provider-evidence` already emits an `operator_actions` entry that reruns itself after provider handoff, but its `next_action.command` keeps the raw child command. Reports and mobile rows that surface `next_action` can therefore show an incomplete handoff such as:

```text
make final-resource-fill-guide; rerun make final-resource-apply-preview; rerun make provider-handoff
```

That command updates provider handoff but does not refresh the live evidence report that the final gate actually checks.

## Decision

Keep `first_blocker.command` as the raw blocker command for provenance, and make `next_action.command` use the same validation-aware command produced for `operator_actions`.

Examples:

```text
make provider-handoff; rerun make live-provider-evidence
```

```text
make final-resource-fill-guide; rerun make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence
```

Configured live provider calls still receive the explicit consent prefix:

```text
PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make backend-evaluate-3d-configured; rerun make live-provider-evidence
```

## Scope

This phase only changes local report command shaping. It does not call Meshy, OpenAI, Treatstock, Xcode, Keychain, or any paid provider. It does not write secrets or mutate global state.

## Acceptance

- Missing provider handoff reports expose a `next_action.command` that reruns `make live-provider-evidence`.
- Provider handoff child blockers expose a full final-resource-to-provider-to-live-evidence chain.
- Configured provider evaluation blockers keep live-call consent in `next_action.command`.
- The refreshed local `live-provider-evidence.json` contains a validation-aware `next_action.command`.
- Focused tests, lint, diff hygiene, and backend tests pass.
