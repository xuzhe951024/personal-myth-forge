# P0.115 Rehearsal Freshness Design

## Context

P0.114 made `make ios-device-launch-rehearsal` rewrite
`services/backend/.local/final-demo-launch-local.json` after writing
`services/backend/.local/ios-device-launch-rehearsal.json`. That keeps the
iPhone-facing final launch payload synchronized when the wrapper is run.

The remaining gap is stale saved evidence. If code changes after the rehearsal
JSON is written, `/v1/final-demo-launch` can still embed an old
`ios_device_launch_rehearsal_readiness` block. Final acceptance readiness
already guards against this by comparing the saved report modification time to
the current git HEAD commit time. P0.115 applies the same pattern to the iOS
device launch rehearsal readiness report.

## Approaches Considered

1. Add no backend freshness and rely on the operator to rerun
   `make ios-device-launch-rehearsal`. This keeps the schema stable, but the
   phone cannot distinguish current evidence from stale evidence.

2. Add a freshness block directly to
   `ios_device_launch_rehearsal_readiness`. This reuses the proven
   `final_acceptance_readiness` pattern, keeps the endpoint read-only, and makes
   the iPhone panel show an actionable stale-evidence message.

3. Add a separate top-level final launch freshness report that checks every
   saved report. This remains useful after P0.115, but it is broader than the current
   freshness bug and would add another mobile surface before the rehearsal lane
   itself is guarded.

P0.115 uses approach 2.

## Design

`services/backend/src/myth_forge_api/ios_device_launch_rehearsal_readiness.py`
will add a `freshness` payload with the same public shape as final acceptance
freshness:

```json
{
  "status": "fresh",
  "classification": "fresh_report",
  "checked_against": "git_head",
  "source_modified_at": "2026-06-07T10:15:00Z",
  "current_revision": "abc1234",
  "current_revision_committed_at": "2026-06-07T10:00:00Z"
}
```

Freshness statuses:

- `fresh`: the saved rehearsal JSON modification time is equal to or newer than
  current git HEAD commit time.
- `stale`: the saved rehearsal JSON modification time is older than current git
  HEAD commit time.
- `unknown`: the report is missing or git metadata cannot be read.

If an otherwise valid saved rehearsal report is stale, readiness status becomes
`blocked`, exit code becomes `2`, and the first blocker is:

- `id`: `ios_device_launch_rehearsal_freshness`
- `classification`: `stale_report`
- `command`: `make ios-device-launch-rehearsal`
- `detail`: saved rehearsal report is older than the current git revision

Operator actions should start with a rerun instruction for the wrapper:

```text
rerun make ios-device-launch-rehearsal to regenerate services/backend/.local/ios-device-launch-rehearsal.json for the current git revision
```

The iOS model will decode the new `freshness` block. The mobile summary will add
one row to `Launch Rehearsal`:

- `Freshness: fresh_report` for fresh evidence
- `Freshness: stale_report; rerun iOS device launch rehearsal.` for stale
  evidence
- `Freshness: source_missing` or `Freshness: git_unavailable` for unknown
  evidence

The UI section already renders `launchRehearsalRows`, so no new SwiftUI layout
is needed.

## Safety

The freshness check may run local read-only git commands:

- `git rev-parse --short HEAD`
- `git log -1 --format=%ct HEAD`

It must not run rehearsal commands, write reports, call Meshy/OpenAI/Treatstock,
start servers, run deploy preflight, run Xcode, write backend/iOS config, touch
signing/keychain, accept Apple licenses, or mutate global machine state.

The report sanitizer must continue to redact absolute paths, provider secrets,
raw media, payment links, and bearer tokens.

## Tests

Backend tests should prove:

- missing rehearsal report has `freshness.status == "unknown"` and
  `classification == "source_missing"`
- fresh saved rehearsal report remains ready or partial according to its saved
  status
- stale saved rehearsal report is blocked with
  `ios_device_launch_rehearsal_freshness` as the first blocker
- final demo launch embeds the new freshness block
- source gates require the freshness field, stale blocker id, and rerun action

Swift tests should prove:

- `FinalDemoLaunchReport` decodes rehearsal `freshness`
- `FinalLaunchMobileSummary.launchRehearsalRows` displays fresh freshness
- stale freshness adds the rerun row
- unsafe strings inside freshness fields are redacted from encoded mobile
  summary output

Visual evidence should show a 390px-wide iPhone final launch card where
`Launch Rehearsal` includes stale/fresh freshness rows and the safe rerun action.

## Non-Goals

- Do not add a new endpoint.
- Do not change `make ios-device-launch-rehearsal` behavior.
- Do not broaden freshness checks to every final launch saved report.
- Do not call live providers.
- Do not run or configure Xcode.
- Do not write provider keys, backend `.env`, or iOS deploy config.
