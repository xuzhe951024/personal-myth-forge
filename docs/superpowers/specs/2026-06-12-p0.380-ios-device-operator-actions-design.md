# P0.380 iOS Device Operator Actions Design

## Context

The final showcase readiness report already includes per-device-action
`operator_actions` for manual iOS/Xcode steps. The iOS Final Launch summary
decodes and displays each device action's command, evidence, validation command,
detail, and child `next_action`, but it does not decode or display
`operator_actions`.

For the current `ios_deployable` blocker, the Xcode/signing device row can carry
a concrete operator action such as accepting the Xcode license outside Codex and
rerunning `make mobile-xcode-build-evidence`. Without this field on iPhone, the
operator sees a more generic row and has to infer the next manual step from
other reports.

## Decision

Add optional `operatorActions` support to `FinalShowcaseDeviceAction`:

- Decode `operator_actions` as `[String]` with a default empty array.
- Render up to two operator actions in `showcaseDeviceActionRow`.
- Keep the field non-breaking for older saved launch reports that lack it.
- Keep existing command, evidence, validation command, child next action, and
detail rendering unchanged.

No provider calls, Xcode signing, Keychain writes, or global machine mutations
are part of this phase.

## Data Flow

```text
backend final_showcase_readiness.device_action_bundle.actions[].operator_actions
  -> FinalShowcaseDeviceAction.operatorActions
  -> FinalLaunchMobileSummary.showcaseDeviceActionRow
```

## Testing

Extend the existing final demo launch fixture's Xcode device action with
`operator_actions`. Add contract assertions that the field decodes and that the
Final Launch showcase readiness rows display the concrete manual action. Verify
RED before adding model/rendering support, then rerun focused Swift checks.
