# P0.332 iOS Launch Report Actions Design

## Context

The final showcase readiness ledger now points at concrete iOS deploy blockers,
but two reports in the physical iPhone launch chain still have incomplete
top-level action fields:

- `ios_device_launch_certificate_report` can be `blocked` while
  `first_blocker` and `next_action` are `null`.
- `ios_device_launch_rehearsal_report` can have `first_blocker` while
  `next_action` is `null`.

These reports are read by final-local-report-refresh, final handoff rows, and
mobile launch summaries. Missing top-level actions force downstream code to
infer from `operator_actions`, which makes the operator handoff less direct.

## Decision

Add explicit top-level action payloads to both reports:

- The launch certificate promotes the first required blocked/missing device gate
  into `first_blocker`.
- Certificate `next_action` mirrors that blocker with `source:
  "first_blocker"`.
- The certificate action command comes from the existing gate-specific operator
  action, so the report does not invent a new sequence.
- The launch rehearsal keeps its existing `first_blocker` logic and adds
  `next_action`.
- Rehearsal `next_action.command` uses the first useful top-level
  `operator_actions` entry when present, otherwise falls back to the blocker
  command.
- Ready/partial reports without hard blockers expose `null` for both fields.

## Safety

This phase is report-only. It does not run Xcode, sign an app, accept Apple SDK
licenses, write deploy config, mutate Keychain/global state, call providers, or
touch live print fulfillment.

## Acceptance

- Blocked launch certificate reports expose `first_blocker` and `next_action`.
- Ready launch certificate reports expose `null` action fields.
- Blocked launch rehearsal reports expose `next_action` derived from the current
  concrete operator action.
- Missing-report launch rehearsal cases expose a useful next action instead of a
  self-refresh action.
- Existing operator action ordering and safety fields remain unchanged.
