# P0.522 iOS Runbook Concrete Device Actions Design

## Context

The iOS deploy runbook already promotes top-level blockers to validation-aware commands such as:

```bash
provide DEVELOPMENT_TEAM in Deployment.local.xcconfig; rerun make mobile-deploy-preflight
```

Its `device_action_bundle` still exposes raw manual config commands in `action.command` and `action.next_action.command`. Downstream reports embed that source bundle, so raw commands can reappear inside final external, closure, handoff index, and local refresh evidence.

## Goal

Make `ios_deploy_runbook.device_action_bundle` manual config actions validation-aware at the source. When an operator sees a device bundle action for Apple Team ID, bundle id, or iPhone backend URL, the command should include the validation rerun needed to prove the gate moved.

## Scope

- Apply mobile deploy validation chaining to runbook device slot actions.
- Update both `action.command` and nested `action.next_action.command`.
- Keep `validation_command` as a separate structured field.
- Preserve report safety: no command execution, no Xcode, no signing, no Keychain, no provider calls.

## Non-Goals

- Do not execute `make mobile-deploy-preflight`.
- Do not write `Deployment.local.xcconfig`.
- Do not resolve real Apple Team ID, iPhone backend URL, Xcode license, or provider credentials.
- Do not change final demo, final local refresh, or final external ledger behavior except through refreshed source evidence.

## Acceptance Criteria

- Missing-config iOS deploy runbook tests assert validation-aware device slot commands.
- Focused iOS deploy runbook tests pass.
- Refreshed `ios-deploy-runbook-local.json` reports a validation-aware device bundle first action.
- Backend lint and full backend tests pass before merge.
