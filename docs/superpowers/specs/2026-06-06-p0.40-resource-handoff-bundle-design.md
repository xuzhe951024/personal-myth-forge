# P0.40 Resource Handoff Bundle Design

## Context

The project now has separate readiness and acceptance surfaces:

- `GET /v1/provider-readiness`
- `provider-handoff`
- `demo-acceptance`
- `final-acceptance`
- `make mobile-deploy-preflight`
- `make mobile-xcode-build`
- backend `.env.example`
- iOS `Deployment.local.xcconfig.example`

This is useful during development, but the final handoff asks the user to provide
all API keys and local device deployment values at once. Those resources are
currently discoverable only by reading multiple documents and command outputs.
P0.40 adds a single sanitized resource handoff report that tells the operator
what to provide, where it belongs, whether it appears configured, and which
commands to run next.

## Goal

Add a `resource-handoff` CLI report that consolidates backend provider keys,
print provider key slots, iOS deployment config, Apple SDK license/build gate,
and final acceptance commands into one machine-readable JSON document.

## Non-Goals

- Do not create, request, validate, or store real provider keys.
- Do not print secret values or local absolute paths.
- Do not call Meshy, OpenAI, Treatstock, Sculpteo, or Apple services.
- Do not mutate `.env`, Xcode settings, global developer directory, signing
  identities, keychain, or Apple license state.
- Do not mark Treatstock/Sculpteo live quote adapters real-ready.

## Alternatives Considered

### Option A: Extend `provider-handoff`

Add iOS deploy and operator-resource sections to the existing report. This keeps
one command, but it changes the meaning of a provider-specific command that is
already used by final acceptance.

### Option B: Add A New `resource-handoff` Command

Keep `provider-handoff` as a backend provider readiness report and add a broader
operator-facing command for final resource provisioning. This avoids changing
existing final acceptance semantics and gives the final handoff a dedicated
shape.

This is the recommended option.

### Option C: Only Document The Checklist

Update README with a checklist. That helps humans, but it is not testable and
does not give final acceptance a structured artifact to inspect later.

## Chosen Design

Use Option B.

P0.40 adds:

- `myth_forge_api.resource_handoff`
- CLI command:

```bash
cd services/backend
uv run python -m myth_forge_api.cli resource-handoff \
  --repo-root ../.. \
  --output /tmp/personal-myth-forge-resource-handoff.json
```

The report is configuration-only. It reads current backend settings through
`load_settings()` and checks project-local iOS deployment files. It does not run
provider calls, Xcode builds, signing commands, or global setup commands.

## Report Shape

Top-level fields:

- `kind: "resource_handoff_report"`
- `overall_status`: `ready`, `blocked`, or `partial`
- `summary`: counts by `ready`, `missing`, `blocked`, and `manual`
- `backend`: backend provider/resource items
- `ios`: iOS deployment/resource items
- `operator_actions`: concise list of values the user must provide
- `commands`: next commands to run after resources are supplied
- `safety`: secret/path leak flags

Backend resource items:

- `THREE_D_PROVIDER`
- `MESHY_API_KEY`
- `NPC_PROVIDER`
- `OPENAI_API_KEY`
- `PRINT_PROVIDER`
- `TREATSTOCK_API_KEY`
- `SCULPTEO_API_KEY`
- `CAPTURE_STORAGE_DIR`

iOS resource items:

- `DEVELOPMENT_TEAM`
- `PRODUCT_BUNDLE_IDENTIFIER`
- `PMF_BACKEND_BASE_URL`
- Apple SDK license/build gate

Each item includes:

- `id`
- `label`
- `destination`
- `required_for`
- `status`
- `configured`
- `missing`
- `notes`

Allowed statuses:

- `ready`: configured enough for its stated use.
- `missing`: required value is absent.
- `blocked`: local external state blocks final acceptance.
- `manual`: human/operator action is required and cannot be verified without
  global or external state.
- `optional`: optional future resource slot is documented but not required for
  the current core demo.

## Readiness Rules

Core real-provider demo requires:

- `THREE_D_PROVIDER=meshy`
- `MESHY_API_KEY`
- `NPC_PROVIDER=openai`
- `OPENAI_API_KEY`

Print live-provider handoff is documented but not required for the current core
iOS/AI/3D demo because Treatstock/Sculpteo adapters are not implemented yet.
`PRINT_PROVIDER=local` is acceptable for the current local quote review.

iOS deploy handoff requires:

- ignored `apps/mobile/ios/Config/Deployment.local.xcconfig`
- non-empty `DEVELOPMENT_TEAM`
- non-empty `PRODUCT_BUNDLE_IDENTIFIER`
- `PMF_BACKEND_BASE_URL` that is not loopback

Apple SDK license/build state remains `manual` or `blocked` until the user
accepts the license outside this repo. P0.40 must not run `sudo xcodebuild
-license` or `xcode-select`.

## Safety

The report may include environment variable names, destination files, provider
names, and whether a value is configured. It must not include:

- key values
- bearer tokens
- `sk-...`
- `api_key=...`
- absolute local paths
- checkout/payment URLs

`.env.example` and `Deployment.local.xcconfig.example` remain sample files only.

## Testing

Backend tests cover:

- `resource-handoff` writes JSON with the expected shape.
- Local default settings report missing Meshy/OpenAI core real-provider
  resources.
- Configured Meshy/OpenAI settings mark core backend resources ready without
  leaking secret values.
- Treatstock/Sculpteo configured keys appear as configured slots but
  `not_implemented`/optional live print handoff remains clear.
- iOS deploy local config absent or loopback backend URL reports missing/blocked.
- CLI stdout works when `--output` is omitted.
- Report does not contain secret values or absolute paths.

Existing full regression remains:

- `make backend-lint`
- `make backend-test`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests`
- `swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck`
- `final-acceptance --profile quick --provider-mode local`
- `make mobile-deploy-preflight` expected exit `2` until local signing/LAN URL
  exist
- `make mobile-xcode-build` expected Apple SDK license gate on this machine
