# P0.233 Rehearsal Readiness Step Detail Design

## Context

P0.232 taught the iOS final launch summary to decode and display
`ios_device_launch_rehearsal_readiness.sequence[].detail`. The backend readiness
adapter still drops that field when it normalizes the saved
`ios-device-launch-rehearsal.json` report. As a result, actual final demo launch
payloads can still lose the blocker explanation before the mobile app sees it.

## Goal

Preserve sequence row `detail` values in
`ios_device_launch_rehearsal_readiness` reports. The readiness report should
include the saved detail when it is present, and sanitize it through the existing
report sanitizer before returning or writing JSON.

## Scope

This only changes the backend readiness normalization layer for saved iOS device
launch rehearsal reports. It does not change how rehearsal reports are produced,
which commands run, final launch aggregation, provider calls, Xcode behavior, or
mobile UI code.

## Architecture

Extend `_sequence` in `ios_device_launch_rehearsal_readiness.py` so it copies a
non-empty string `detail` from each saved step row into the normalized row. The
existing `_sanitize_report` call remains the boundary that redacts secrets,
absolute local paths, file URLs, capture URLs, checkout URLs, bearer tokens,
and local network URLs.

## Testing

Use TDD:

1. Add a failing backend test that writes a saved rehearsal readiness report with
   `sequence[0].detail` and asserts the readiness report preserves it.
2. Add a failing backend test that writes unsafe content in `detail` and asserts
   sanitized output contains `[withheld]` but no unsafe substrings.
3. Implement the minimal `_sequence` field copy.
4. Verify focused backend tests, mobile contract compatibility, and diff hygiene.

## Self-Review

- No placeholders remain.
- Scope is backend report normalization only.
- Safety posture stays read-only and sanitizer-backed.
