# P0.205 Device Preflight Apply Preview Design

## Purpose

The backend final launch report already includes `final_resource_apply_preview`, and the mobile final launch summary renders its first blocker. `DevicePreflightSummaryBuilder` still stops at the fill guide, so the iOS device preflight does not show the next resource handoff step: previewing and applying the final resources into backend and iOS config files.

## Scope

- Add a `final_resource_apply_preview` device preflight item labeled `Apply Preview`.
- Show waiting when the final launch report or apply preview report is missing.
- Show blocked when `FinalResourceApplyPreviewReport.firstBlocker` is present, including the command, detail, blocked input IDs, destination, writer, and validation command.
- Show ready when the apply preview report status is `ready`.
- Include the item in the required ready set so device preflight cannot become ready before final resources are apply-ready.
- Reuse the existing device preflight sanitizer; do not add file writes, shell execution, or provider calls.
- Add mobile readiness source gates and Swift contract coverage, including unsafe first blocker redaction.

## Acceptance

- `DevicePreflightSummaryBuilder` returns a `final_resource_apply_preview` item.
- Blocked reports surface `make final-apply-resources`, blocked input IDs, and `make final-resources-preflight`.
- Ready reports mark the item ready and include ready/missing/write-target counts.
- Encoded preflight output redacts provider keys, bearer tokens, local paths, local capture URLs, checkout text, and `api_key` snippets from apply preview first blocker fields.
- Mobile final launch readiness requires the item, label, source, contract tests, and required-ready set entry.
