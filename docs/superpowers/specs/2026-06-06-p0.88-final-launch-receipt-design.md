# P0.88 Final Launch Receipt Design

## Context

P0.87 made backend `final_resources_preflight.items` visible in the iPhone
`Final Launch Status` panel. The phone now shows individual missing or blocked
API/key/deploy slots, plus existing mode, resource, acceptance, handoff, command,
and note sections.

The remaining operator gap is the first-read verdict. During final iPhone
handoff, the operator should not need to infer the launch state by scanning
several sections. The top of the panel should say whether the showcase is ready,
which single blocker currently prevents launch, and whether live-provider consent
is still required.

## Approaches

### A. Add a mobile-only Launch Receipt section

Derive short `launchReceiptRows` inside `FinalLaunchMobileSummary` from the
already-sanitized final launch report pieces. Render the rows near the top of
`FinalLaunchStatusView`.

This is the recommended approach. It changes no backend contract, runs no
commands, keeps the iPhone app read-only, and gives the operator a compact
receipt for the final handoff.

### B. Add a backend `launch_receipt` field

The backend could emit a dedicated receipt object in `/v1/final-demo-launch`.
That is more canonical, but it creates another API field and duplicates logic
that the mobile summary already owns for display ordering and redaction.

### C. Replace existing sections with a single condensed receipt

The UI would become shorter, but the operator would lose useful detail:
resource checklist rows, acceptance blockers, next actions, and command rows.
This would make troubleshooting slower.

## Chosen Design

Use approach A. Add `launchReceiptRows: [String]` to
`FinalLaunchMobileSummary`, produced by `FinalLaunchMobileSummaryBuilder`.
Render `Launch Receipt` before `Mode`.

The receipt rows are display text, not executable actions. They must pass through
the existing `sanitize` function before reaching the UI.

## Receipt Rules

When the final launch report is missing:

- row 1: `Receipt: waiting for final launch report.`

When the report exists:

- row 1 describes the final launch mode and overall state:
  - `Receipt: local launch waiting.`
  - `Receipt: configured launch ready.`
  - `Receipt: configured launch blocked.`
- row 2 summarizes final acceptance when present:
  - `Acceptance: 12 passed, 2 blocked, 0 failed.`
  - If final acceptance readiness is missing:
    `Acceptance: readiness report not loaded.`
- row 3 identifies the first actionable blocker:
  - prefer first final acceptance blocker
  - else first final resource item with required missing status or blocked status
  - else first operator handoff next action
  - else ready message
- row 4 reports live-provider policy:
  - `Live providers: consent required for configured acceptance.`
  - `Live providers: no live calls by default.`

Rows are capped at four.

## Data Flow

```text
/v1/final-demo-launch
  -> FinalDemoLaunchReport
  -> FinalLaunchMobileSummaryBuilder
  -> FinalLaunchMobileSummary.launchReceiptRows
  -> FinalLaunchStatusView "Launch Receipt"
```

## Safety

- The app remains read-only for final launch operations.
- No deploy, Xcode, provider, resource apply, or shell command is executed.
- Provider secrets, absolute paths, local media URIs, checkout links, and bearer
  tokens remain redacted through `sanitize`.
- The receipt uses only data already present in the final launch report.

## Acceptance Criteria

- Swift contract tests cover:
  - missing final launch report receipt
  - blocked receipt with final acceptance blocker priority
  - blocked receipt with resource item fallback
  - ready configured receipt
  - unsafe text redaction in receipt rows
- `FinalLaunchStatusView` renders a `Launch Receipt` section.
- Swift project checks and backend iOS showcase source acceptance require the new
  receipt model/UI/test handoff.
- A 390x844 visual evidence page shows the receipt on a blocked final launch
  state without overlap, clipping, or secret values.
