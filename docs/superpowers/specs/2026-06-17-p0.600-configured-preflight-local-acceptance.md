# P0.600 Configured Preflight Local Acceptance

## Context

The configured preflight path embeds the configured iOS deploy runbook. When local
final acceptance is blocked or stale, the runbook currently exposes a generic slot
action such as `run make final-acceptance-local`, even when the final acceptance
readiness report already knows a more concrete child blocker.

That makes configured handoff less actionable and can hide the real next step needed
before live Meshy/OpenAI evidence.

## Decision

Let the iOS deploy runbook copy child blocker metadata from local final acceptance
readiness into the `local_final_acceptance` slot. The configured preflight first
blocker should then surface the child blocker command, detail, and validation command.

## Acceptance

- `local_final_acceptance` slot records `source_blocker_id`, `blocker_detail`, and
  `validation_command` when final acceptance readiness has a blocker.
- Configured preflight first blocker can surface the concrete child command, not only
  the generic final acceptance writer action.
- No live provider calls, Xcode signing, global mutation, or secret writes are added.
