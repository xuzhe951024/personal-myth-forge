# P0.432 Rehearsal Readiness Operator Actions Design

## Context

The final iPhone showcase remains blocked by real device evidence: `DEVELOPMENT_TEAM`, an iPhone-reachable `PMF_BACKEND_BASE_URL`, Xcode/device proof, and provider credentials. P0.431 made `make ios-device-launch-rehearsal-readiness` refresh the iPhone-visible final launch report, but the readiness report drops saved rehearsal sequence row `operator_actions` while converting the saved report into a bounded readiness payload.

When the saved rehearsal report is stale, this leaves the device action bundle with the correct stale rerun command, but the row no longer exposes the structured next command that will be needed after rerun, such as `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto`.

## Decision

Preserve sanitized, bounded `operator_actions` from each saved rehearsal sequence row in `ios_device_launch_rehearsal_readiness.sequence`. The device action bundle will keep the stale rerun command as the primary command, and add a `saved_next_action` field for the first saved row action when present.

## Behavior

- `ios-device-launch-rehearsal-readiness` remains `blocked` when the saved rehearsal file is stale.
- The top-level `first_blocker` and `next_action` still tell the operator to rerun `make ios-device-launch-rehearsal`.
- Each sequence row may include up to four sanitized `operator_actions` copied from the saved rehearsal report.
- Each device action may include `saved_next_action` with the first preserved row action, structured as `{id, label, status, command, detail, source, validation_command}`.
- `saved_next_action.command` is the executable command portion only; detail suffixes stay in `saved_next_action.detail`.
- Reports remain sanitized and do not include local absolute paths, secrets, raw media, raw personal context, payment links, or live provider calls.

## Safety

This change only changes read-only readiness report shape. It does not write `.env`, iOS deploy config, Xcode project settings, Keychain entries, or provider resources. It does not call live Meshy/OpenAI/Treatstock providers and does not make stale evidence count as ready.

## Tests

Add a regression test for a stale saved rehearsal report whose sequence row has a deploy writer action. The test asserts readiness stays blocked, the stale rerun remains the primary action, and the device action exposes the deploy writer as `saved_next_action`.
