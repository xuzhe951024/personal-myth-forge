# P0.174 Mobile Closure Configured Bundle Design

## Context

P0.173 added a required `configured_evidence_bundle` section to the backend
final launch closure packet. The mobile final launch summary already renders
closure packet sections generically and also has a separate configured evidence
bundle panel, but its Swift contract fixture still describes the closure packet
as five sections and does not assert that the new closure section exists.

That leaves the iOS handoff weaker than the backend handoff. A future fixture or
model change could drop the configured bundle section from the closure packet
while the generic summary tests continue passing against older sections.

## Goal

Make the mobile final launch contract explicitly require the closure packet's
`configured_evidence_bundle` section, its action id, command, visible summary
row, and redaction behavior.

## Design

The Swift final launch fixture will be updated to match the backend closure
packet shape from P0.173: six sections, with `configured_evidence_bundle` after
`live_provider_consent` and before `final_acceptance`.

`PersonalMythForgeMobileCoreContractTests` will assert the decoded
`FinalLaunchClosurePacketReport` exposes:

- `summary.sections == 6`
- `sectionsById["configured_evidence_bundle"]`
- `firstAction.id == "configured_live_evidence_bundle"`
- command `make configured-live-evidence-bundle`
- cost-consent/live-provider metadata preserved through the model

The mobile summary test will assert `closurePacketRows` contains
`configured_evidence_bundle`, `configured_live_evidence_bundle`, and
`make configured-live-evidence-bundle`. To make that deterministic, the fixture
will place the configured bundle section as the first non-ready section for this
test path, or the summary test will use a fixture whose selected closure section
is the configured bundle.

The redaction test will feed unsafe configured bundle action text through the
same fixture path and assert the mobile summary removes provider secrets, local
paths, file URLs, checkout fields, bearer tokens, and private-message markers.

The backend mobile readiness acceptance source requirements will be tightened so
the iOS showcase gate requires the new contract test names and source strings.
No provider adapters, Make targets, live API calls, iOS view layout, or backend
closure packet generation behavior changes are in scope.

## Alternatives Considered

1. Rely on generic `closurePacketRows` rendering only.
   Rejected because generic rendering proves the UI can show one selected
   section, not that the configured bundle closure section is part of the mobile
   contract.

2. Add a dedicated `closureConfiguredEvidenceBundleRows` UI panel.
   Rejected for now because the existing closure packet panel already has the
   right user-facing location. This iteration only closes the contract gap.

3. Update backend tests only.
   Rejected because P0.173 already covers backend behavior. The open risk is the
   mobile decode/render/acceptance layer.

## Acceptance

- Swift final launch fixture includes closure section
  `configured_evidence_bundle` with first action
  `configured_live_evidence_bundle`.
- Swift decode contract asserts six closure packet sections and the configured
  bundle section's command/action metadata.
- Swift mobile summary contract shows the configured bundle closure row.
- Swift redaction contract proves unsafe configured bundle closure details are
  sanitized before display.
- Backend mobile readiness acceptance requires the new Swift contract coverage
  and explicit configured bundle closure packet source strings.
- Focused Swift contract tests and backend mobile readiness acceptance pass.

## Test Strategy

Use TDD:

1. Add Swift contract assertions for the configured bundle closure section and
   run `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests`
   to see the expected failure against the existing five-section fixture.
2. Add backend mobile readiness acceptance requirements and run
   `cd services/backend && uv run pytest tests/test_mobile_final_launch_readiness_acceptance.py -q`
   to see the expected missing-source failure.
3. Update the Swift fixture and backend requirement source strings with the
   smallest changes needed.
4. Re-run focused Swift/backend checks, then run `make backend-lint` and the
   relevant backend/mobile acceptance tests before merging.

## Self-Review

- Placeholder scan: no TBD/TODO/open placeholders.
- Scope check: one mobile contract closure gap; no provider, CLI, live call, or
  visual layout changes.
- Safety check: fixture-only unsafe strings stay inside tests and are sanitized
  before display assertions.
- Ambiguity check: section id, action id, command, ordering, and test commands
  are explicit.
