# P0.186 Configured Acceptance Command Visual Design

## Goal

The default visual regression gate must include an iPhone-sized proof that configured final launch mode exposes `make final-acceptance-configured` as the first visible command row in the Final Launch Status panel.

## Current Behavior

P0.184 and P0.185 proved the data path in Swift contract tests: configured launch payloads can carry the wrapper command and the mobile summary prioritizes it. The checked-in visual regression manifest, however, does not yet include a static visual artifact for that specific configured `Commands` row. The final rehearsal can therefore pass visual regression without proving that the operator-facing command is visible on the iPhone handoff surface.

## Scope

- Add one default visual regression artifact:
  - `p0.186_configured_acceptance_command_visual`
  - HTML under `docs/superpowers/verification/`
  - PNG under `docs/superpowers/verification/assets/`
  - notes under `docs/superpowers/verification/`
- Required text must include:
  - `Final Launch Status`
  - `Commands`
  - `make final-acceptance-configured`
  - `Live providers: consent required for configured acceptance.`
  - `--allow-live-provider-calls`
  - `commands_run=false live_calls=false global=false`
- Keep the proof static and read-only. It must not execute configured final acceptance or call live providers.

## Acceptance

- A red test first proves the default visual artifact list lacks the new proof.
- `check_visual_artifacts()` passes with the new checked-in HTML and PNG.
- `make visual-regression-local` includes the new artifact in the saved report.
- Existing backend, Swift, and local rehearsal checks still pass.
