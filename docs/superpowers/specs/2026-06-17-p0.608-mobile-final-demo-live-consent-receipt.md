# P0.608 Mobile Final Demo Live Consent Receipt

## Context

P0.607 preserved `requires_live_provider_consent` and
`completion_requires_live_provider_consent` on the backend
`final-demo-launch-local` first blocker and next action. The iPhone final launch
receipt still needs to decode and display those fields so the operator can see
that the next configured evidence command is a live-provider/cost-sensitive handoff
before running it.

## Decision

Extend the mobile `FinalDemoLaunchFirstBlocker` and `FinalDemoLaunchNextAction`
models with optional consent metadata fields. The final launch receipt rows should
append short consent markers when those fields are true.

This is a source-only mobile handoff improvement. It does not call Meshy, OpenAI,
print providers, Xcode signing, Keychain, or any global machine action.

## Acceptance

- Swift decodes top-level final demo `first_blocker` consent metadata.
- Swift decodes top-level final demo `next_action` consent metadata.
- The iPhone final launch receipt includes `live provider consent required` when
  `requires_live_provider_consent` is true.
- The receipt includes `completion live provider consent required` when
  `completion_requires_live_provider_consent` is true.
- Existing launch receipt rows continue to include command, detail, and validation
  command text.
