# P0.43 iOS Showcase Acceptance Design

## Goal

Add an in-process acceptance check that proves the checked-in iOS source contains
the complete minimal showcase surface: camera capture, guided scan, capture
upload, 3D preview, NPC agent actions, print quote review, provider readiness,
final showcase summary, and safe deployment configuration. Include this check in
`final-acceptance --profile quick` so the default final report represents the
mobile showcase source contract, not only backend and deployment blockers.

## Context

P0.42 added the missing phone-native single-photo camera bridge. The repository
already has a SwiftPM `PersonalMythForgeMobileProjectChecks` executable, but the
quick final acceptance profile does not run Swift commands. Quick acceptance
currently reports four passing in-process checks plus two blocked iOS deploy
gates:

- provider handoff
- demo acceptance
- capture-to-3D acceptance
- print quote acceptance
- iOS deploy preflight
- Xcode build gate

This leaves a gap: a default final acceptance report can say nothing about
whether the mobile showcase source still includes the full visible loop unless
the caller chooses the slower `full` profile.

## Recommended Approach

Create a lightweight backend-side `ios_showcase_acceptance` module that reads
source files and project files directly, then returns a sanitized report with
feature flags and missing requirements.

Why this approach:

- It is fast enough for quick acceptance.
- It does not invoke Xcode, SwiftPM, simulators, devices, signing, or provider
  APIs.
- It is complementary to the existing Swift project check, not a replacement.
- It gives final acceptance a durable, machine-readable summary of the mobile
  showcase surface.

Rejected alternatives:

- Move all Swift project checks into quick profile: stronger compile proof, but
  slower and more prone to local SwiftPM/Xcode toolchain noise.
- Leave this only in documentation: easy, but weak evidence for final acceptance.
- Add a CLI-only visual screenshot gate: useful for docs, but less precise than
  source/project contract checks.

## Acceptance Report Shape

The new in-process report should look like:

```json
{
  "kind": "ios_showcase_acceptance_report",
  "status": "succeeded",
  "required_features": [
    {
      "id": "camera_capture",
      "status": "passed",
      "evidence": ["CameraCaptureView.swift", "Take Photo"]
    }
  ],
  "summary": {
    "passed": 9,
    "failed": 0
  },
  "safety": {
    "global_mutation": false,
    "runs_xcode": false,
    "runs_swiftpm": false,
    "provider_secrets_in_report": false,
    "raw_media_in_report": false
  }
}
```

If any required feature is missing, the report exits `1`, marks the missing
feature as `failed`, and includes only safe file-relative evidence names. It must
not include absolute local paths.

## Required Feature Checks

The first version checks these feature IDs:

- `camera_capture`: `CameraCaptureView.swift`, `Take Photo`, and
  `CameraCaptureMediaBuilder.singlePhotoSelection`.
- `guided_scan`: `GuidedScanCaptureView.swift`, `ObjectCaptureSession`,
  `GuidedScanPhotoSetBuilder.mediaDrafts`.
- `capture_upload`: `ForgeFlowService`, `uploadObjectCapture`, and
  `createMythSessionFromCapture`.
- `three_d_preview`: `Artifact3DPreviewView`, `ArtifactAssetPreparer`, and
  `ios_scene_asset`.
- `npc_agent`: `NPCTickView`, `Run Autonomy`, and `runMythSessionAutonomy`.
- `print_quote`: `PrintQuoteReviewView`, `createPrintQuote`, and `Get Quote`.
- `provider_readiness`: `ProviderReadinessView`, `getProviderReadiness`, and
  `missingEnv`.
- `final_showcase`: `FinalShowcaseSummaryView`, `FinalShowcaseSummaryBuilder`,
  and `Final Showcase`.
- `deploy_config`: `Deployment.xcconfig`,
  `Deployment.local.xcconfig.example`, `PMF_BACKEND_BASE_URL`, and
  `NSCameraUsageDescription`.

## Final Acceptance Integration

Add the in-process check after `print_quote_acceptance` and before command-based
deployment gates. Quick local final acceptance should become:

- passed: `5`
- blocked: `2`
- failed: `0`
- skipped: `0`

The two blocked checks remain:

- `mobile_deploy_preflight`
- `mobile_xcode_build`

Full profile should include the same in-process iOS showcase acceptance plus the
existing backend and Swift command checks.

## Safety

The check only reads files under the supplied repository root. It must not:

- execute shell commands
- call Xcode, SwiftPM, providers, or network APIs
- read ignored local deployment values
- include absolute filesystem paths in the report
- include provider secrets, raw media bytes, data URIs, checkout/payment URLs,
  or personal source content

## Testing

Use TDD:

- add a unit test that a complete fixture repo passes all required feature
  checks
- add a unit test that a missing camera bridge fails with safe relative evidence
- update final acceptance tests to include `ios_showcase_acceptance`
- run CLI final acceptance and assert the new quick summary is
  `passed=5`, `blocked=2`, `failed=0`

## Non-Goals

- Do not replace SwiftPM compile checks.
- Do not run Xcode, simulators, or device installs.
- Do not validate real camera runtime behavior.
- Do not create provider keys or mutate local deployment config.

## Acceptance

P0.43 is accepted when:

- the new iOS showcase acceptance module has red/green tests
- `final-acceptance --profile quick` includes `ios_showcase_acceptance`
- quick local final acceptance reports `passed=5`, `blocked=2`, `failed=0`
- full regression remains green
- documentation and visual evidence explain the new final acceptance line item
