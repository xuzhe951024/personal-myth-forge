# P0.68 iOS Local Network Readiness Design

## Context

The final iPhone demo expects the mobile app to call a backend running on the
same LAN, for example `http://192.168.1.10:8080`. P0.45-P0.66 already added a
backend device-server command, ignored iOS deployment config, preflight health
checks, and final resource apply. The checked-in app also allows local
networking through App Transport Security.

One deploy-facing gap remains in the source gates: the app does not explicitly
declare an iOS local network usage purpose string. Without that source-level
contract, a physical-device demo can look correct in backend config but still be
underprepared for the OS privacy prompt that appears when the app reaches a
LAN-hosted backend.

## Goal

Add source-verified iOS local network readiness for the final iPhone demo, while
keeping all provider keys backend-only and avoiding any global Xcode/signing
mutation.

## Non-Goals

- Do not run Xcode builds, accept Apple licenses, modify `xcode-select`, sign
  apps, touch keychain, or install to a device.
- Do not add Bonjour discovery or network scanning.
- Do not store backend URLs or provider secrets in committed mobile source.
- Do not change the backend deployment command or final resource apply file.

## Approaches Considered

### A. Add only `NSLocalNetworkUsageDescription`

This is the smallest source fix and directly addresses iOS's privacy prompt.
However, if it is not included in the source acceptance gates, a later refactor
could drop it without final acceptance noticing.

### B. Add the plist key plus SwiftPM project/source gates

This keeps the change small and testable. `PersonalMythForgeMobileProjectChecks`
will parse `Info.plist` and require a non-empty local network usage string.
`ios_showcase_acceptance` will include it under deploy config, so quick final
acceptance also proves the source handoff.

This is the selected approach.

### C. Add Bonjour service declarations

This is premature. The current device handoff uses an explicit
`PMF_BACKEND_BASE_URL` rather than Bonjour discovery. Adding Bonjour service
types now would make the permissions surface larger without improving the
current demo path.

## Design

Add `NSLocalNetworkUsageDescription` to `apps/mobile/ios/App/Info.plist` with a
plain explanation that the app connects to the local Personal Myth Forge backend
for the live demo.

Extend the source gates:

- `PersonalMythForgeMobileProjectChecks` requires the plist key.
- `ios_showcase_acceptance` `deploy_config` feature requires the plist key.
- `tests/test_ios_showcase_acceptance.py` fixture and expected summary update
  from 16 to 17 only if a new feature is introduced. For P0.68, keep the feature
  count at 16 and strengthen `deploy_config`.

The report remains source-only. It does not run Xcode, call the backend, call
providers, or expose absolute paths/secrets.

## Testing

Use TDD:

1. RED: update `PersonalMythForgeMobileProjectChecks` to require
   `NSLocalNetworkUsageDescription`; run the SwiftPM project check and confirm it
   fails against the current `Info.plist`.
2. GREEN: add the plist key and rerun the project check.
3. RED: update `ios_showcase_acceptance` tests to require the plist key in the
   deploy config feature and confirm focused backend tests fail.
4. GREEN: update the fixture/acceptance source requirement and rerun focused
   backend tests.
5. Verify with backend lint/test, Swift project/core checks, quick final
   acceptance, and a 390x844 static evidence screenshot.

## Acceptance Criteria

- `Info.plist` contains a non-empty `NSLocalNetworkUsageDescription`.
- SwiftPM project checks fail without the key and pass with it.
- `ios_showcase_acceptance` fails fixture/source checks when the key is absent.
- Quick final acceptance still reports only the known local iOS deploy and Apple
  SDK license blockers.
- Reports and evidence contain no raw media, provider keys, payment links, or
  absolute local paths.
