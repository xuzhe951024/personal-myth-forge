# P0.392 Showcase Provider Child Actions Design

## Context

The final showcase currently reports `game_asset_3d_generation` and
`ai_agent_npc` as partial when live Meshy/OpenAI evidence is not proven. Those
rows receive their command from `live-provider-evidence`, which still points at
the broad `make provider-handoff` entry even after `provider-handoff.json`
contains a more concrete child action such as
`make final-resource-apply-preview`.

That makes the final showcase less actionable than the lower-level provider
handoff report.

## Goal

Make live provider evidence project the concrete provider handoff child action
into its first blocker and next action. Final showcase rows should then inherit
the same actionable command for the live 3D and NPC capability rows.

## Behavior

- If `provider-handoff.json` is missing, unreadable, or does not include a
  child blocker/next action, keep the existing `make provider-handoff` action.
- If `provider-handoff.json` is present and blocked, prefer its `next_action`
  command over the generic provider handoff command.
- Preserve the live evidence slot id `provider_handoff` so existing grouping and
  capability mapping stay stable.
- Add sanitized source blocker fields such as `source_blocker_id`,
  `source_blocker_label`, `source_blocker_command`, and
  `source_blocker_validation_command` for traceability.
- Keep `make live-provider-evidence` as the live evidence validation command,
  because the live evidence report is the report that consumers must refresh
  after resources are supplied.
- Do not write secrets, apply resources, call providers, run signing, touch
  Keychain, or perform paid external actions.

## Testing

Add a focused live provider evidence regression where saved provider handoff
data has a child `next_action`. The live evidence first blocker and next action
should keep id `provider_handoff` while exposing the child command.

Update the final showcase regression for live provider rows so the 3D and NPC
capability rows inherit the child command and still require completion-level
live provider consent.
