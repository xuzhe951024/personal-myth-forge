# P0.260 Final Refresh Operator Actions Design

## Context

`final-local-report-refresh.json` is the safest aggregate evidence file for the final iPhone showcase. It now exposes:

- `next_action`: the resource-first blocker action, such as `provide MESHY_API_KEY in final-resources.env`
- `showcase_next_action`: the current iPhone showcase action, such as `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig`

The top-level `operator_actions` list still contains generic entries like `review refreshed final_resource_requirements report`. That is less useful for Goal-mode continuation and for an operator checking the aggregate file.

## Goal

Make `final-local-report-refresh.operator_actions` start with concrete actions from the aggregate blockers while preserving all existing report status, step ordering, and child report data.

## Selected Approach

Build top-level operator actions from structured aggregate fields:

1. Add `next_action.command` when it is non-empty.
2. Add `showcase_next_action.command` when it is non-empty and distinct.
3. Add concrete blocked step commands from step-level promoted hints.
4. Fall back to the existing generic `review refreshed <step> report` strings only when a blocked step has no command.

This preserves the existing resource-first `next_action` contract while making the iPhone showcase action visible in the same compact action queue.

## Behavior

On the default local checkout, the aggregate report should include operator actions beginning with:

```json
[
  "provide MESHY_API_KEY in final-resources.env",
  "provide DEVELOPMENT_TEAM in Deployment.local.xcconfig"
]
```

The list remains bounded to 12 strings and sanitized by the existing report sanitizer. Ready reports have no operator actions. Failed steps still place `fix final-local-report-refresh step <id>` first.

## Safety

This does not run Meshy, OpenAI, Treatstock, Xcode, signing, Keychain, real ordering, or global mutations. It only changes a sanitized JSON field produced by the safe local refresh command.
