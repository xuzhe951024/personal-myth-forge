# P0.572 Closure Configured Consent Design

## Context

The final showcase readiness report now correctly marks iOS deploy evidence as
ready and points the remaining first blocker at live Meshy/OpenAI evidence:

```text
PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence
```

The final launch closure packet still exposes a weaker configured-evidence
blocker:

```text
make final-acceptance-configured
```

That command is not the safest operator handoff because configured final
acceptance may call Meshy and OpenAI. It must remain gated by explicit live
provider consent and should route the operator through the same evidence refresh
chain used by final showcase readiness.

## Design

When `configured-live-evidence-bundle.json` is missing or not ready, the closure
packet should treat the configured evidence bundle as a live-provider-consent
handoff, not as a bare local command.

The configured evidence bundle action should:

- Keep the action id `configured_live_evidence_bundle`.
- Keep the section id `configured_evidence_bundle`.
- Use `PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence` as the operator command when no more specific saved blocker command exists.
- Mark the action as requiring cost consent and live provider calls.
- Surface the same command in `first_blocker`, `next_action`, and `operator_actions`.
- Keep ready configured evidence bundles unchanged.

## Safety

This phase only changes report text and action metadata. It does not call
Meshy/OpenAI, does not write secrets, does not change Xcode signing, and does
not place print orders.

## Testing

Add a regression test around the configured-evidence section builder and closure
packet first-blocker selection. When configured live evidence is missing or
blocked, the configured evidence action, first blocker, next action, and
operator action list should all use the explicit live provider consent chain.
