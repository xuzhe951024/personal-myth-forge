# P0.24 Provider Handoff Smoke Design

## Context

P0.20 added `GET /v1/provider-readiness`, P0.21 made iPhone HEIC/HEIF photos
eligible for Meshy Image-to-3D, and P0.23 routed NPC ticks through the same
OpenAI provider switch as initial NPC reactions. The project now has the
backend interfaces needed for final API/key handoff, but the operator still
lacks a single non-secret report that answers:

- Which environment variables must be provided?
- Which providers are selected?
- Which selected providers are demo-ready?
- Which selected core providers are real-provider-ready?
- What command should be run after keys are supplied?

## Goal

Add a backend CLI smoke/handoff report that can be run before and after API keys
are supplied. It must not call Meshy or OpenAI by default, must never print
secret values, and must exit non-zero when asked to require the core real
provider set and that set is not ready.

## Recommended Approach

Add a new CLI command:

```bash
uv run python -m myth_forge_api.cli provider-handoff --output /tmp/handoff.json
```

The command loads settings, reuses `build_provider_readiness`, and writes a
structured JSON report. A stricter mode:

```bash
uv run python -m myth_forge_api.cli provider-handoff \
  --require-core-real \
  --output /tmp/handoff.json
```

returns exit code `2` when any core provider is not real-provider-ready.

Core provider kinds are:

- `three_d`
- `npc`
- `capture_storage`

`print` is included in the report but not part of `core_real_ready` for the iOS
deployable AI/3D minimum demo. Print fulfillment remains a later provider
iteration.

## Alternatives Considered

### Endpoint Only

The existing readiness endpoint is useful for mobile display, but it does not
write a persistent artifact for handoff review and does not provide a stricter
operator exit code. Keeping this as CLI-first is simpler and safer.

### Live Smoke By Default

Calling Meshy/OpenAI by default would prove more but could spend quota or create
assets accidentally. P0.24 deliberately avoids live calls. Later work can add an
explicit `provider-live-smoke` command with prompt budgets and confirmation.

### Treat Print As Core

The full product includes print fulfillment, but the current active goal is the
minimum iOS-deployable demo with 3D scanning, 3D generation, and AI-agent NPCs.
Blocking core readiness on Treatstock/Sculpteo would slow that goal without
improving the first mobile demo.

## Report Shape

The JSON report will include:

```json
{
  "kind": "provider_handoff_report",
  "mode": "configuration",
  "overall_demo_ready": true,
  "overall_real_ready": false,
  "core_real_ready": false,
  "core_provider_kinds": ["three_d", "npc", "capture_storage"],
  "missing_env": ["MESHY_API_KEY", "OPENAI_API_KEY"],
  "backend_only_env": [
    "MESHY_API_KEY",
    "OPENAI_API_KEY",
    "TREATSTOCK_API_KEY",
    "SCULPTEO_API_KEY"
  ],
  "mobile_secret_policy": "Provider secrets stay on the backend; mobile clients only see readiness metadata.",
  "providers": [],
  "next_commands": [
    "make backend-dev",
    "curl http://127.0.0.1:8080/v1/provider-readiness"
  ]
}
```

`providers` uses the existing `ProviderReadinessItem` JSON shape. Missing env
names are deduplicated and sorted. Secret values are not included.

## CLI Behavior

- `provider-handoff --output PATH` writes the report and returns `0`.
- `provider-handoff` without `--output` prints the report to stdout and returns
  `0`.
- `provider-handoff --require-core-real --output PATH` writes the report and
  returns `2` when `core_real_ready` is false.
- Provider configuration errors are sanitized through the existing CLI error
  redaction helper.
- The command does not instantiate live Meshy/OpenAI network calls.

## Documentation And Evidence

Docs will show a final handoff flow:

1. Copy `.env.example` to `.env`.
2. Set `THREE_D_PROVIDER=meshy`.
3. Set `NPC_PROVIDER=openai`.
4. Add `MESHY_API_KEY` and `OPENAI_API_KEY`.
5. Run `provider-handoff --require-core-real --output ...`.
6. Run backend/mobile checks.

Static evidence will show a 390px operator/mobile handoff card with selected
providers, missing env names, and backend-only secret policy.

## Tests

Backend tests will cover:

- local-default report returns exit code `0` and writes JSON.
- missing Meshy/OpenAI keys in `--require-core-real` mode returns exit code `2`
  while still writing a sanitized report.
- configured Meshy/OpenAI keys make `core_real_ready` true without leaking
  secret values.
- stdout mode prints valid JSON.
- `backend_only_env`, `next_commands`, and `mobile_secret_policy` are present.

Regression tests will include full backend pytest, backend lint, SwiftPM mobile
project checks, Swift contract tests, app compile check, and the known Xcode
license gate result.

## Non-Goals

- No live Meshy generation.
- No live OpenAI generation.
- No provider key creation.
- No global environment modification.
- No print fulfillment adapter.
- No iOS signing or App Store/TestFlight automation.
