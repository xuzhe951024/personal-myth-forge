# P0.535 Final Acceptance Mobile Evidence Design

## Context

The final readiness stack already has a concrete mobile deploy next action:

```bash
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

`final-showcase-readiness` and `final_acceptance_readiness` surface that action from `services/backend/.local/mobile-deploy-preflight-evidence.json`. The saved `final-acceptance-local.json` report can still show the older generic backend-device-demo wording when `make final-local-report-refresh` runs the mobile preflight in skipped backend-health mode.

This creates an operator handoff split: the aggregate report points to the deploy-config writer, while the saved final acceptance report points to backend-device-demo.

## Design

`run_final_acceptance()` should read the optional saved mobile deploy preflight evidence from:

```text
services/backend/.local/mobile-deploy-preflight-evidence.json
```

When the first blocked check is `mobile_deploy_preflight` and the saved evidence has a valid `next_action.command`, the final acceptance report should:

- attach that saved `next_action` to the `mobile_deploy_preflight` check,
- use the saved command in `first_blocker.command`,
- include `validation_command` from the saved evidence,
- emit `next_action` from the same enriched blocker,
- emit `operator_actions[0]` as `<saved command>; rerun <validation command>`.

If the evidence file is missing, invalid JSON, or does not have the expected kind, `run_final_acceptance()` keeps current behavior.

`final-local-report-refresh` should also run the `mobile_deploy_preflight_evidence` step before `final_acceptance_local`. That ensures a fresh local refresh writes the evidence before `run_final_acceptance()` reads it, instead of requiring a previous refresh run.

## Safety

This is read-only report shaping. It does not run `mobile-write-deploy-config-auto`, does not write `Deployment.local.xcconfig`, does not start a backend, does not call Xcode, and does not call Meshy/OpenAI/Treatstock.

The saved evidence is already sanitized by `mobile_deploy_preflight_evidence`; final acceptance still runs its existing report sanitizer before returning.

## Tests

Add a focused RED test in `services/backend/tests/test_final_acceptance.py` that writes a blocked mobile preflight evidence file and asserts `run_final_acceptance()` uses that concrete saved command in:

- `first_blocker.command`
- `next_action.command`
- `operator_actions[0]`

Update `services/backend/tests/test_final_local_report_refresh.py` so the safe refresh fixture expects `mobile_deploy_preflight_evidence` before `final_acceptance_local` and expects the written `final-acceptance-local.json` command to use the concrete writer handoff.

Then implement the smallest helper set in `services/backend/src/myth_forge_api/final_acceptance.py` and run focused final acceptance tests plus lint.
