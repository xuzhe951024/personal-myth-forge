# P0.385 iOS Rehearsal Next Action Command Design

## Context

`ios-device-launch-rehearsal` is one of the core Mac-side reports used to get
the final iPhone demo onto a physical device. It exposes both plain
`operator_actions` and a structured top-level `next_action` consumed by other
reports and mobile launch panels.

The current report can produce a top-level `next_action.command` that includes
a human detail suffix:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

That shape is acceptable for plain operator action lists, but the structured
`next_action.command` should be copy-pasteable. The missing-value explanation
belongs in `next_action.detail`, which already preserves the richer context.

## Decision

When `ios-device-launch-rehearsal` builds its structured `next_action` from
`operator_actions`, strip a trailing ` | ...` detail suffix from the command.

- Keep `operator_actions` unchanged so text-only handoff still carries detail.
- Keep `first_blocker.detail` unchanged for diagnosis.
- Preserve `validation_command: make ios-device-launch-rehearsal`.
- Do not run Xcode, write signing configuration, call providers, or mutate any
  ignored resource handoff file.

## Testing

Update the existing mobile-preflight-detail test so it proves:

- `operator_actions[0]` can retain the detail suffix.
- `next_action.command` is the executable command without the suffix.
- `next_action.detail` still contains the missing `DEVELOPMENT_TEAM` and
  iPhone-reachable backend URL details.
