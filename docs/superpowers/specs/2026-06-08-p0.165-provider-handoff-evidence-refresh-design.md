# P0.165 Provider Handoff Evidence Refresh Design

## Context

`live-provider-evidence` expects a saved provider handoff report at:

```text
services/backend/.local/provider-handoff.json
```

The CLI command already exists:

```bash
cd services/backend
uv run python -m myth_forge_api.cli provider-handoff --require-core-real \
  --output .local/provider-handoff.json
```

But there is no Make target and `final-local-report-refresh` does not write this
report. As a result, the local final evidence set can still say provider handoff
is missing even though the project has enough code to generate the readiness
report safely.

## Goal

Make provider handoff a first-class saved evidence artifact for final resource
acceptance by adding a Make target, status-bearing report output, final refresh
integration, and clearer live evidence detail when core real providers are not
ready.

## Non-Goals

- Do not call Meshy, OpenAI, Treatstock, or any other live provider.
- Do not validate API key values against external services.
- Do not write backend `.env`, final resources, iOS deploy config, signing, or
  keychain state.
- Do not require provider keys for the local command to write a report.

## Options Considered

### Option A: Only document the existing CLI command

This keeps the code unchanged but leaves `provider-handoff.json` absent from the
normal local evidence refresh. The operator still has to remember a long CLI
command before live evidence becomes meaningful.

### Option B: Add only a Make target

This improves the manual workflow, but `final-local-report-refresh` would still
not produce a complete local evidence set.

### Option C: Add Make target plus refresh integration

This is the selected approach. Add:

```bash
make provider-handoff
```

and include a `provider_handoff` step in `final-local-report-refresh`, writing:

```text
services/backend/.local/provider-handoff.json
```

The report is configuration-only. It may be `blocked` until
`THREE_D_PROVIDER=meshy`, `MESHY_API_KEY`, `NPC_PROVIDER=openai`, and
`OPENAI_API_KEY` are configured, but it is no longer missing.

## Architecture

Update the existing provider handoff report to include:

- `status`: `ready` when `core_real_ready` is true, otherwise `blocked`
- `classification`: `core_real_providers_ready` or `core_real_providers_not_ready`
- `summary`: compact provider counts for local refresh display

The CLI exit behavior stays compatible:

- `provider-handoff` without `--require-core-real` exits `0` after writing the
  report, even if status is `blocked`.
- `provider-handoff --require-core-real` exits `2` when `core_real_ready` is
  false.

`final-local-report-refresh` calls the report builder directly and therefore
marks the provider handoff step `blocked` until real providers are configured,
without running external commands.

`live-provider-evidence` keeps using the saved file but returns a clearer detail
for local stubs: "Core real providers are not ready for live evidence."

## Safety

The new Make target and refresh step must declare or preserve:

- no live provider calls
- no global mutation
- no backend env writes
- no iOS deploy config writes
- no provider secret values in reports
- no local absolute paths in reports

## Acceptance

- `make provider-handoff` writes `services/backend/.local/provider-handoff.json`.
- In the current local environment, `make provider-handoff` exits `2` by design
  because core real providers are not configured.
- The saved report includes status/classification/summary and does not leak
  provider secrets.
- `make final-local-report-refresh` writes `provider-handoff.json` and includes a
  `provider_handoff` step.
- `make live-provider-evidence` sees provider handoff as `blocked` rather than
  `missing` after the refresh.
- Existing backend tests, Swift checks, and visual regression remain valid.
