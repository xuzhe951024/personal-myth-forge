# P0.251 Final Demo Launch Status Design

## Context

`final-demo-launch-local.json` and `final-demo-launch-configured.json` are central handoff reports for the iPhone showcase. They already include `overall_status`, `first_blocker`, and `next_action`, but they do not expose the common top-level `status` field used by adjacent readiness reports. As a result, generic consumers that read `{status, first_blocker, next_action}` see `status: null` even when the launch report has an authoritative `overall_status`.

Current readiness remains blocked by external/device evidence: provider keys, Treatstock configuration, Apple Team ID, an iPhone-reachable backend URL, Xcode state, physical device execution, and live provider proof. This iteration does not perform those actions.

## Goal

Expose a top-level `status` in final demo launch reports, equal to the existing `overall_status`, so operator tooling and iPhone-facing readiness panels can consume the same status contract across final reports.

## Design

`build_final_demo_launch_report` will continue computing `overall_status` from launch phases. The returned report will add:

```json
"status": "<same value as overall_status>"
```

The existing `overall_status` field remains in place for backward compatibility. Exit code behavior does not change:

- configured mode exits `2` when required configured resources are missing or blocked
- local mode can still exit `0` while reporting `partial` if only manual/device work remains

## Testing

Add focused regression coverage for both configured and local launch reports:

- configured mode with missing resources has `status == overall_status == "blocked"`
- local mode has `status == overall_status == "partial"` for the current no-key/device-handoff path
- the report still includes `first_blocker` and `next_action`

Then run the final demo launch test file, lint, backend tests, and regenerate `final-local-report-refresh` to verify saved `.local/final-demo-launch-local.json` now has non-null top-level `status`.

## Safety

The change is report-shape only. It does not write secrets, call live providers, run Xcode, alter signing, or execute physical-device commands.

## Out of Scope

This iteration does not satisfy the remaining external requirements and does not mark the full goal complete.
