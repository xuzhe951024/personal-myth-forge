# P0.41 Final Showcase Summary Design

## Context

The mobile app now has the major vertical-slice pieces, but they are displayed
as separate sections:

- provider readiness
- capture form and media selection
- generated artifact summary and 3D preview
- world arbitration
- NPC reactions and stored NPC ticks/autonomy controls
- print quote review
- local demo snapshot restore

For the final demo, the operator needs one compact mobile-first summary that
answers: did this run cover scan/capture, 3D generation, AI Agent NPC behavior,
print review, and resource handoff boundaries? P0.41 adds that summary without
changing the backend contract or calling external services.

## Goal

Add a mobile `Final Showcase` summary panel backed by a testable Swift model.
The panel should make the current run's demo readiness visible at a glance:
capture status, 3D generated asset/provenance, NPC agent/autonomy status, print
quote status, provider readiness, and privacy/resource boundaries.

## Non-Goals

- Do not add a new backend endpoint.
- Do not call Meshy, OpenAI, Treatstock, Sculpteo, Apple, or payment services.
- Do not store raw capture media, provider keys, checkout URLs, or local file
  paths in the summary model.
- Do not replace the existing detailed sections. The summary sits above them.
- Do not attempt simulator/device installation, signing, Apple license
  acceptance, or global Xcode configuration.

## Alternatives Considered

### Option A: Backend Summary Endpoint

Add a new backend endpoint that returns a complete showcase report. This could
serve web/mobile consumers equally, but it duplicates state that the app already
has and adds another contract before the real-device handoff.

### Option B: Mobile-Computed Summary

Compute the summary from the existing mobile state: `CaptureMediaSelection`,
`MythSession`, `ProviderReadinessResponse`, `NPCAgentTick` history, and
`PrintQuote`. This is the recommended option because it is local, fast,
testable with SwiftPM, and makes the app's already-loaded state immediately
reviewable.

### Option C: Documentation-Only Run Checklist

Add a README checklist for operators. This helps, but it is not visible in the
mobile app and cannot be exercised through the existing Swift contract checks.

## Chosen Design

Use Option B.

P0.41 adds a Foundation-only summary model to the mobile core:

- `FinalShowcaseSummary`
- `FinalShowcaseSummaryStage`
- `FinalShowcaseSummaryBuilder`

The app adds:

- `FinalShowcaseSummaryView`
- root-view wiring that passes current state into the builder
- project-check assertions that the view is included in the Xcode target and
  rendered before the detailed artifact/NPC/print sections

## Data Model

`FinalShowcaseSummary` includes:

- `overallStatus`: `waiting`, `readyForLocalDemo`, or `needsAttention`
- `title`: derived from the myth seed when a session exists
- `stages`: ordered rows for `capture`, `three_d`, `npc_agent`, `print`, and
  `resources`
- `privacyNotes`: fixed notes that no raw media, provider keys, checkout URLs,
  or local file paths are summarized

Each `FinalShowcaseSummaryStage` includes:

- `id`
- `label`
- `status`: `waiting`, `ready`, `needsAttention`, or `optional`
- `detail`

The builder takes:

- optional `CaptureMediaSelection`
- optional `MythSession`
- `npcTickHistoryCount`
- optional `PrintQuote`
- optional `ProviderReadinessResponse`
- optional provider readiness error string

## Readiness Rules

Capture:

- `ready` when the current media selection is ready for upload
- `waiting` when no usable media is selected

3D:

- `ready` when a session exists with a generated asset URI and a primary format
- detail includes provider, format, and provenance input mode when available
- `waiting` before session creation

NPC Agent:

- `ready` when a session exists and has three NPC reactions or agent traces
- detail includes initial runtime and saved tick count
- `needsAttention` only if the session exists without NPC output

Print:

- `ready` when a quote exists
- `optional` when a session has a print candidate but no quote yet
- `waiting` before session creation

Resources:

- `ready` when provider readiness reports `overallDemoReady`
- `needsAttention` when provider readiness reports demo not ready or the
  backend preflight has an error
- `waiting` before readiness response or error

Overall:

- `waiting` when no session exists
- `needsAttention` if any non-optional stage needs attention
- `readyForLocalDemo` when a session exists and capture, 3D, NPC, and resources
  are ready; print can be `ready` or `optional`

## UI

`FinalShowcaseSummaryView` is compact and utilitarian:

- title row: `Final Showcase`
- status text such as `Ready for local demo` or `Waiting for first myth session`
- five stage rows with small status labels
- short privacy/resource boundary line

It should render near the top of `ForgeRootView`, after snapshot restore status
and before provider readiness/capture form. Detailed panels remain below it.

## Testing

Swift contract tests cover:

- waiting summary before session creation
- local-demo-ready summary with capture media, generated asset provenance,
  provider readiness, NPC output, and no print quote
- print stage changing from optional to ready once a quote exists
- provider readiness error turning resources into `needsAttention`
- summary text does not include obvious secret strings, checkout URLs, or local
  filesystem paths from source data

Project checks cover:

- `FinalShowcaseSummaryView.swift` exists
- Xcode source membership includes the view
- `ForgeRootView` renders `FinalShowcaseSummaryView` before
  `ProviderReadinessView`
- app source contains `FinalShowcaseSummaryBuilder`

Full regression remains:

- `make backend-lint`
- `make backend-test`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests`
- `swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck`
- `resource-handoff` local blocked/safe report
- `final-acceptance --profile quick --provider-mode local`
- `make mobile-deploy-preflight` expected local config blocker
- `make mobile-xcode-build` expected Apple SDK license gate
