# P0.65 Mobile Final Launch Readiness Acceptance Design

## Goal

Add a formal backend final acceptance gate for the mobile Final Launch readiness path introduced in P0.64.

The gate must prove that:

- `GET /v1/final-demo-launch?mode=local` returns a sanitized, read-only launch report.
- invalid launch modes are rejected by the API contract.
- the iOS source contains the model, API client, root view load path, device preflight item, and contract-test coverage for the launch report.
- the gate does not run provider calls, mutate global machine state, expose provider secrets, expose raw media, expose checkout/payment links, or expose absolute local paths.

## Acceptance Surface

The new runner will live at:

- `services/backend/src/myth_forge_api/mobile_final_launch_readiness_acceptance.py`

It will be wired into:

- `services/backend/src/myth_forge_api/final_acceptance.py`

The runner is intentionally source-and-contract based. It should not start servers, call Meshy/OpenAI/Treatstock, run SwiftPM, run Xcode, sign apps, or write local config.

## Report Shape

The runner returns:

```json
{
  "kind": "mobile_final_launch_readiness_acceptance_report",
  "status": "succeeded",
  "summary": {"passed": 4, "failed": 0},
  "checks": [],
  "endpoint": {},
  "mobile_source": {},
  "safety": {}
}
```

Final acceptance should count this as an in-process check named:

- `mobile_final_launch_readiness_acceptance`

Expected local quick acceptance after this gate:

- `11 passed, 2 blocked, 0 failed`

The two blocked gates remain local machine blockers:

- `mobile_deploy_preflight`
- `mobile_xcode_build`

## Non-Goals

- No Unity or device deploy integration.
- No live provider calls.
- No generated 3D quality evaluation.
- No printing fulfillment request.
- No changes to the mobile UI behavior beyond verifying the existing P0.64 path.
