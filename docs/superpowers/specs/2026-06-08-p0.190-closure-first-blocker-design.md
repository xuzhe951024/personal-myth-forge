# P0.190 Closure First Blocker Design

## Problem

`final_launch_closure_packet_report` is the final operator handoff bundle, but when it is blocked the top-level `first_blocker` is currently `null`. The packet already contains enough data in `sections[].first_action`, but operators and downstream readiness ledgers must inspect nested sections to find the first actionable item.

## Scope

P0.190 adds a structured top-level `first_blocker` to the closure packet:

- Derive it from the first required section whose status is `blocked` or `missing`.
- Use that section's existing `first_action` as the blocker action.
- Preserve existing status, summary, sections, actions, operator actions, source reports, and safety semantics.
- Update the existing P0.157 closure packet visual artifact to show `first_blocker`.
- Update source gates so future closure packet changes cannot remove the field silently.

## Non-Goals

- Do not mark the closure packet ready.
- Do not reorder closure sections.
- Do not run any commands, provider calls, Xcode, signing, keychain, payment, or global actions.
- Do not add or apply provider keys.

## Data Shape

When blocked:

```json
{
  "id": "resource_inputs",
  "label": "Resource inputs",
  "status": "blocked",
  "classification": "missing_required_value",
  "command": "make final-resources-preflight",
  "detail": "Backend-only secret for live Meshy 3D generation.",
  "section_id": "resource_inputs",
  "action_id": "provide_MESHY_API_KEY"
}
```

When no required section is blocked or missing, `first_blocker` is `null`.

## Acceptance Criteria

- Missing/blocked closure packet inputs produce `first_blocker` for the first required blocked section/action.
- Ready resource and device sections keep `first_blocker` absent unless another required section is blocked.
- `first_blocker` is sanitized by the existing report sanitizer.
- The P0.157 visual artifact requires and contains `first_blocker`.
- iOS showcase source acceptance requires `first_blocker` in `final_launch_closure_packet.py`.
- Existing safety flags continue to prove the report builder is read-only and does not execute the listed actions.
