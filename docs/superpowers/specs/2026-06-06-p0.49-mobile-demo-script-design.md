# P0.49 Mobile Demo Script Design

## Product Intent

The iOS app already has the building blocks for the local final showcase:
capture, forge, 3D preview, NPC ticks/autonomy, print quote, provider readiness,
and a final summary. What is still weak for a live iPhone demo is operator
guidance. A first-time reviewer can see the modules, but the app does not give a
clear ordered script for what to do next.

P0.49 adds an in-app `Demo Script` panel. It turns the existing local state into
a short, ordered checklist that highlights the next action in the demo loop.
This improves the final handoff without changing providers, keys, backend
contracts, or global machine configuration.

## Scope

Add a mobile-core model and builder:

- `DemoScript`
- `DemoScriptStep`
- `DemoScriptStepStatus`
- `DemoScriptBuilder`

The builder consumes existing app state:

- `CaptureMediaSelection?`
- `MythSession?`
- NPC tick history count
- `PrintQuote?`
- `ProviderReadinessResponse?`
- provider readiness error text

It returns ordered steps:

1. Capture Object
2. Forge Myth
3. Load 3D Scene
4. Run NPC Autonomy
5. Request Print Quote
6. Check Resources

Exactly one step should be marked `current` when the demo is in progress. Steps
that are complete stay `complete`. The print quote step may be `optional` after
NPC autonomy if no quote has been requested. Resource problems are `blocked`.

Add a compact SwiftUI `DemoScriptView` below the existing final summary. It must
use short text that fits mobile screens and no new external dependencies.

Add `demo_script` to the source-only `ios_showcase_acceptance` gate so final
acceptance proves the checked-in app contains the script panel and builder.

## Behavior

Before any media is selected:

- `Capture Object` is `current`.
- Remaining steps are `waiting`.
- Next action: "Capture or import an object."

After media is ready but before a session exists:

- `Capture Object` is `complete`.
- `Forge Myth` is `current`.
- Next action: "Forge the myth session."

After a session exists:

- `Forge Myth` is `complete`.
- `Load 3D Scene` is `complete` when the generated asset has a loadable
  `ios_scene_asset` variant.
- `Run NPC Autonomy` is `current` until at least three NPC ticks are saved.

After three or more ticks:

- `Run NPC Autonomy` is `complete`.
- `Request Print Quote` is `optional` until a quote is available.
- `Check Resources` is `complete` when provider readiness says local demo is
  ready, `blocked` on readiness error or missing demo readiness, otherwise
  `waiting`.

## Safety

The script must not expose raw media, local file paths, checkout URLs, API keys,
or provider secrets. It should only render short derived statuses from existing
typed state.

## Tests

Swift core contract tests should cover:

- initial script state
- media-ready state
- full local demo state with 3 ticks and loadable DAE scene asset
- provider readiness blocked state
- unsafe text redaction

Backend tests should cover:

- `ios_showcase_acceptance` includes the `demo_script` feature
- the feature fails when the app view or builder evidence is missing

## Non-Goals

- No Meshy/OpenAI/Treatstock/Sculpteo live calls.
- No provider key creation.
- No Apple SDK license acceptance or signing configuration.
- No changes to the backend session schema.
- No Unity runtime integration in this iteration.
