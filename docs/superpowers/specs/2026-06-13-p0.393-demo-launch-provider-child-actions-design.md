# P0.393 Demo Launch Provider Child Actions Design

## Context

`final-showcase-readiness` now exposes concrete live provider child actions for
3D and NPC readiness, for example:

```text
make final-resource-apply-preview; rerun make live-provider-evidence
```

However, `final-demo-launch-local` can still surface both that concrete action
and the older broad action:

```text
make provider-handoff; rerun make live-provider-evidence
```

This creates an avoidable fork in the final operator handoff. The concrete
child action is the better next step because it points at the ignored final
resource handoff path without implying a provider call or secret write.

## Goal

Make final demo launch prefer the final showcase live provider child action
over the generic provider handoff action when both are present.

## Behavior

- Keep broad `make provider-handoff; rerun make live-provider-evidence` when it
  is the only live provider handoff action available.
- If a concrete live provider child action is also present, keep the child
  action and drop the broad provider handoff duplicate.
- Do not drop print handoff actions, iOS deploy actions, resource preflight
  actions, or unrelated final showcase actions.
- Keep command validation intact. The child action should still rerun
  `make live-provider-evidence`.
- This report remains read-only. It must not apply resources, write secrets,
  call Meshy/OpenAI, run Xcode signing, or create print orders.

## Testing

Add a final demo launch regression where final showcase operator actions include
both the broad provider handoff and the concrete child action. The final demo
launch operator actions should contain only the child action.

Existing tests should continue proving the broad provider handoff is preserved
when no child action is present.
