# P0.25 iOS Deploy Config Design

## Context

The iOS app has a checked-in Xcode project, shared scheme, SwiftPM compile
gate, and an `xcode_build_gate.sh` command that builds without code signing.
This is enough for local source-level verification, but it is not enough for a
real iPhone deployment handoff:

- `DEVELOPMENT_TEAM` is empty in the project.
- `PMFBackendBaseURL` is fixed to `http://127.0.0.1:8080`, which points to the
  phone itself on a physical device.
- There is no project-local place to enter bundle/team/backend deployment
  values without editing global Xcode settings or committing local secrets.
- There is no preflight command that tells the operator why device deployment is
  not ready.

## Goal

Add a project-local iOS deployment configuration layer and preflight check. The
check should make iPhone deployment inputs explicit without mutating global Xcode
state and without requiring real Apple Team ID values in the repository.

## Recommended Approach

Add:

- `apps/mobile/ios/Config/Deployment.xcconfig`
- `apps/mobile/ios/Config/Deployment.local.xcconfig.example`
- ignored `apps/mobile/ios/Config/Deployment.local.xcconfig`
- `apps/mobile/ios/scripts/deploy_preflight.sh`
- `make mobile-deploy-preflight`

`Deployment.xcconfig` is checked in and referenced by both app target build
configurations. It provides safe defaults:

- `PRODUCT_BUNDLE_IDENTIFIER = com.personalmythforge.app`
- `DEVELOPMENT_TEAM =`
- `CODE_SIGN_STYLE = Automatic`
- `PMF_BACKEND_BASE_URL = http://127.0.0.1:8080`

It optionally includes `Deployment.local.xcconfig` so the operator can keep
local values untracked:

```xcconfig
#include? "Deployment.local.xcconfig"
```

`Info.plist` changes `PMFBackendBaseURL` to `$(PMF_BACKEND_BASE_URL)`.

The preflight script reads `Deployment.xcconfig` plus the optional local file and
checks:

- `DEVELOPMENT_TEAM` is non-empty.
- `PRODUCT_BUNDLE_IDENTIFIER` is non-empty.
- `PMF_BACKEND_BASE_URL` is non-empty.
- `PMF_BACKEND_BASE_URL` is not loopback for device deployment.
- The local override file is not committed.
- No global Xcode developer directory mutation is performed.

The script returns:

- `0` when deployment inputs are ready.
- `2` when required local deployment inputs are missing or device-hostile.
- `1` for malformed config or unexpected script errors.

## Alternatives Considered

### Edit Xcode Project Build Settings Directly Only

Direct project settings are already present, but they encourage committing
machine-specific values. A project-local optional xcconfig keeps shared defaults
clean and local values ignored.

### Use Environment Variables Only

Environment-only deployment inputs are hard to discover inside Xcode and easy to
forget. Xcode already understands `.xcconfig`, so it is the right handoff format.

### Run A Signed Xcode Build Now

This environment still stops at the Apple SDK license gate. Also, running a
signed build would require real Apple team credentials. P0.25 should make the
inputs explicit, not pretend signing can be proven here.

## Project Changes

Xcode project:

- Add `Deployment.xcconfig` as a project file reference.
- Set `baseConfigurationReference` for the app target Debug and Release build
  configurations.
- Keep source and package wiring unchanged.

Info.plist:

- Replace hardcoded backend URL with `$(PMF_BACKEND_BASE_URL)`.

Git ignore:

- Ignore `apps/mobile/ios/Config/Deployment.local.xcconfig`.

Makefile:

- Add `.PHONY: mobile-deploy-preflight`.
- Add `mobile-deploy-preflight` target that calls
  `apps/mobile/ios/scripts/deploy_preflight.sh`.

## Tests

Swift project checks will assert:

- `Deployment.xcconfig` exists.
- `Deployment.local.xcconfig.example` exists.
- `.gitignore` ignores `Deployment.local.xcconfig`.
- `Info.plist` uses `$(PMF_BACKEND_BASE_URL)`.
- Xcode target Debug/Release configs reference the deployment xcconfig.
- `deploy_preflight.sh` exists and avoids `sudo`, `xcode-select`, and
  `xcodebuild -license`.
- `Makefile` has `mobile-deploy-preflight`.

Shell-level verification will run:

- `apps/mobile/ios/scripts/deploy_preflight.sh` with no local config and expect
  exit code `2`.
- A temporary copied local config outside git-tracked state and expect exit code
  `0`.

## Documentation And Evidence

Docs will explain:

1. Copy `Deployment.local.xcconfig.example` to `Deployment.local.xcconfig`.
2. Set Apple Team ID.
3. Set a unique bundle id.
4. Set backend URL to the Mac/LAN host reachable by iPhone, not loopback.
5. Run `make mobile-deploy-preflight`.

Static evidence will show the deploy preflight states and the local/ignored
configuration boundary.

## Non-Goals

- No real Apple Team ID in the repo.
- No signed device build in this environment.
- No `sudo`, `xcode-select`, or Apple SDK license acceptance.
- No TestFlight/archive automation.
- No global Xcode settings mutation.
