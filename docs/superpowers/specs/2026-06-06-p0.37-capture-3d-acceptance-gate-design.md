# P0.37 Capture 3D Acceptance Gate Design

## Context

P0.36 added a media-backed `guided-scan-smoke-v0` evaluation suite. That proves
the evaluation runner can send source images to the 3D provider contract, but
the final acceptance harness still creates its myth session from a direct text
observation. As a result, `final-acceptance` can pass local demo checks without
proving the iOS capture path that matters most for the mobile demo:

1. mobile uploads or imports guided scan photos,
2. backend reads stored capture media,
3. backend prepares provider-safe source images,
4. the 3D provider receives single-image or multi-image inputs,
5. reports expose only safe counts/provenance.

P0.37 closes that verification gap without spending provider credits by
default.

## Goal

Add a final-acceptance check that proves the capture-to-3D source-image route
works locally and is reportable without raw media, local paths, or secrets.

## Non-Goals

- Do not run live Meshy or OpenAI calls by default.
- Do not mutate global Xcode state, accept licenses, or install to a device.
- Do not create real provider keys or write secrets into repository files.
- Do not replace the existing API capture tests or the P0.36 evaluation suite.

## Approaches Considered

### A. Add only another CLI evaluation suite

This is simple, but P0.36 already covers evaluation-runner source-image routing.
It still does not prove final acceptance can report the actual capture flow.

### B. Add a full FastAPI TestClient flow inside final acceptance

This is closest to the app path, but it couples final acceptance to multipart
request mechanics and global app dependency monkeypatching. It is also more
fragile than needed because API contract tests already cover upload endpoints.

### C. Add an in-process capture acceptance runner

The runner creates a temporary local capture store, saves a synthetic
`guided_scan` capture with 2-4 tiny PNG/JPEG files, reads those media items back
through the same preparation function used by `POST /v1/myth-sessions/from-capture`,
and calls `create_demo_myth_session` with a recording/local 3D provider.

This is the selected approach. It proves the source-image preparation and
provider request handoff while staying deterministic, fast, and no-key by
default.

## Design

Create a backend acceptance module for capture-backed generation:

- `Capture3DAcceptanceResult` carries `exit_code` and sanitized `report`.
- `run_capture_3d_acceptance(...)` creates a synthetic guided scan capture in a
  temporary local store, builds provider source images from stored media, runs
  the normal myth session pipeline, and reports the result.
- The local/default check uses a recording wrapper around `LocalThreeDProvider`
  so the report can prove provider request counts without changing production
  provider behavior.

The report shape should include:

- `kind: "capture_3d_acceptance_report"`
- `status: "succeeded" | "failed" | "not_ready"`
- `capture_mode: "guided_scan"`
- `source_image_count`
- `source_asset_count`
- `provider_request_source_image_count`
- `generation_provenance.input_mode`
- `generation_provenance.source_image_count`
- `generation_provenance.selected_source_image_count`
- `scene_loadable_variant`
- `safety.raw_media_in_report: false`
- `safety.local_paths_in_report: false`
- `safety.provider_secrets_in_report: false`

The report must be sanitized with the same principles as final acceptance:
redact `data:*;base64,...`, `local-capture://...`, filesystem paths, bearer
tokens, and API-key-looking strings.

## Final Acceptance Integration

Add one new in-process check to `final-acceptance`:

- `id: "capture_3d_acceptance"`
- label: `Capture-to-3D acceptance`
- default behavior: run locally in all profiles after `demo_acceptance` and
  before mobile/Xcode gates.

For P0.37, this check should always use local 3D/NPC providers. Live Meshy
credit-spending remains explicit through existing `evaluate-3d --provider meshy`
or later dedicated live-provider flags. This preserves the existing safety
invariant that final acceptance does not make live provider calls by default.

The quick profile will then prove:

1. provider handoff readiness,
2. local demo acceptance with NPC ticks,
3. local capture-to-3D source-image route,
4. iOS deploy preflight classification,
5. Xcode build gate classification.

## Testing

Backend tests should be TDD-first:

- RED: `run_final_acceptance(profile="quick", provider_mode="local")` includes a
  `capture_3d_acceptance` check with safe report fields.
- RED: the capture runner passes 2-4 source images to its 3D provider and
  reports `multi_image` provenance.
- RED: reports do not contain raw `data:image`, `local-capture://`, `/tmp`,
  repository paths, bearer tokens, or provider keys.

GREEN should add the runner and final acceptance integration with minimal code.

Full regression remains:

- `make backend-lint`
- `make backend-test`
- SwiftPM project checks
- SwiftPM mobile core contract tests
- SwiftPM app compile check
- local final acceptance quick
- `make mobile-deploy-preflight` expected exit `2` until local signing values
- `make mobile-xcode-build` expected Apple SDK license gate on this machine
- `git diff --check HEAD`

## Acceptance Criteria

- `final-acceptance --profile quick --provider-mode local` reports a
  `capture_3d_acceptance` check.
- The new check succeeds locally without provider keys.
- The new check proves source images were read from capture storage and passed
  to the 3D provider.
- The new check reports multi-image provenance and a SceneKit-loadable variant
  in local mode.
- Acceptance reports remain sanitized and contain no raw media, local paths, or
  provider secrets.
- Existing API capture tests, P0.36 evaluation tests, and mobile SwiftPM gates
  remain green.
