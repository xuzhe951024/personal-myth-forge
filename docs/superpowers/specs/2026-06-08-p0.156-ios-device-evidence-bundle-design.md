# P0.156 iOS Device Evidence Bundle Design

## Context

The final local report is now blocked by expected final-device gates rather than
missing product code: resource inputs, the backend LAN server handoff, mobile
deploy preflight, Xcode/signing, and iOS device launch rehearsal. Existing reports
surface these pieces in several places, but the final demo launch payload does not
provide one compact device-evidence checklist that can be read from the phone
before the final operator runs device commands.

## Decision

Build a read-only `ios_device_evidence_bundle_report` and include it in the final
demo launch payload. The report describes evidence slots for:

- `backend_device_server`: `make backend-device-demo`, expected LAN `/health` proof.
- `mobile_deploy_preflight`: `make mobile-deploy-preflight`, expected local deploy
  preflight output/evidence.
- `xcode_build_gate`: `make mobile-xcode-build`, expected local Xcode build-gate
  result.
- `ios_device_launch_rehearsal`: `make ios-device-launch-rehearsal`, expected saved
  rehearsal report.

The report does not run commands, start servers, write env/config files, invoke
Xcode, touch signing/keychain state, or call providers. It reads existing readiness
reports and saved local report files, then produces sanitized operator actions and
safety flags.

## Backend Shape

Create `services/backend/src/myth_forge_api/ios_device_evidence_bundle.py` with:

- `build_ios_device_evidence_bundle_report(repo_root=None)`.
- `IOSDeviceEvidenceBundleResult(exit_code, report)`.
- `report.kind = "ios_device_evidence_bundle_report"`.
- `report.status` is `ready` only when every required evidence slot is ready;
  otherwise `blocked`.
- `report.summary` counts `ready`, `missing`, `blocked`, `manual`, `required`, and
  `global_actions`.
- `report.evidence_slots` is a bounded ordered list.
- `report.operator_actions` provides the first actionable commands.
- `report.safety` explicitly states no commands ran and global/Xcode/signing
  actions are only described.

The report should derive status from existing reports where possible:

- Final acceptance readiness indicates current mobile deploy/Xcode blockers.
- iOS deploy runbook tells whether deploy config inputs are ready.
- iOS device launch rehearsal readiness tells whether rehearsal evidence exists.

## Final Launch Integration

`build_final_demo_launch_report()` should include the report as
`ios_device_evidence_bundle`. This keeps the API handoff single-call for the phone:
`GET /v1/final-demo-launch?mode=local|configured` returns all device evidence
requirements alongside existing resource, deploy runbook, and rehearsal sections.

## iOS Mobile Summary

Add Swift Codable models for the evidence bundle and a new
`FinalLaunchMobileSummary.deviceEvidenceRows` section. The rows should be compact:

- summary row with status and counts
- first three slot rows with id/status/command
- safety row: `commands_run=false xcode=false global=false`

`FinalLaunchStatusView` should render the rows under a `Device Evidence` section.
`DevicePreflight` remains focused on fast launch blockers; the full detail belongs
in Final Launch Status.

## Visual Regression

Add one static 390x844 visual artifact showing `Final Launch Status` with the new
`Device Evidence` section. Required text:

- `Device Evidence`
- `iOS device evidence blocked`
- `mobile_deploy_preflight`
- `make mobile-deploy-preflight`
- `commands_run=false xcode=false global=false`

## Safety

This iteration is read-only. It must not:

- start the backend server
- run `mobile-deploy-preflight`
- run `mobile-xcode-build`
- mutate Xcode, signing, keychain, or Apple license state
- write backend `.env` or iOS deploy config
- call Meshy/OpenAI/print providers
- include provider secrets, raw media, absolute local paths, or payment links in
  reports or visual artifacts

## Acceptance

The iteration is complete when:

- backend unit tests prove the new report shape and final demo launch integration
- Swift contract tests decode the new payload and render mobile rows
- mobile final launch readiness acceptance checks the new source requirements
- visual regression includes the new artifact and reports all default artifacts
  passing
- full backend tests and Swift mobile compile checks pass
