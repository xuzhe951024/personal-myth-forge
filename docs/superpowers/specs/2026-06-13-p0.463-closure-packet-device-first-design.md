# P0.463 Closure Packet Device-First Design

## Context

The final showcase, local demo launch, and local report refresh reports now point their top-level blocker at the local iPhone deployment path: write `DEVELOPMENT_TEAM`, set an iPhone-reachable backend URL, and rerun `make mobile-deploy-preflight`.

The final launch closure packet still chooses `resource_inputs` as its `first_blocker`, usually `provide MESHY_API_KEY in final-resources.env`. That is useful for configured/live provider completion, but it is not the most actionable first step for the local iOS showcase lane.

## Decision

Keep the closure packet sections and resource input actions intact, but promote the final showcase readiness device action to the packet-level `first_blocker` and `next_action` when it is blocked on iOS deployment.

The packet should still expose:

- `resource_inputs` as a blocked section;
- Meshy/OpenAI/Treatstock/iOS resource actions in `operator_actions`;
- configured evidence and live provider consent sections;
- failed or malformed report sections as blocking evidence.

This keeps the final closure packet aligned with the local iPhone demo path while preserving the configured/live handoff inventory.

## Acceptance Criteria

- The standard missing-resource closure packet reports `final_showcase_readiness` as its `first_blocker`.
- The promoted blocker command is the current final showcase iOS device action, such as `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto` or `make ios-device-launch-rehearsal`.
- `next_action` mirrors the promoted device blocker and carries a validation command when the source blocker provides one.
- The `resource_inputs` section remains blocked and still exposes `provide_MESHY_API_KEY`.
- If the final showcase blocker is not an iOS deploy/device action, the existing section-order behavior remains.

## Safety

This phase only changes read-only report prioritization. It does not write `.env`, iOS deploy config, Xcode state, Keychain data, live provider evidence, or print orders.
