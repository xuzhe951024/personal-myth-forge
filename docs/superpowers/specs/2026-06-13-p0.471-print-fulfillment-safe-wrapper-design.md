# P0.471 Print Fulfillment Safe Wrapper Design

## Context

`make print-fulfillment-readiness` still calls the backend CLI directly. In the current no-key/no-device state, the CLI writes a usable blocked report to `services/backend/.local/print-fulfillment-readiness.json` and exits `2`, which makes Make fail. Recent final handoff phases already moved similar blocked-report targets behind safe wrappers so operators can refresh evidence without treating expected blockers as command failures.

## Goal

Make print fulfillment readiness match the rest of the final handoff evidence path: the Make target should write the ignored local JSON report, accept exit code `0` or `2`, and return `0` for usable reports while preserving real failures.

## Scope

- Add `services/backend/scripts/write_print_fulfillment_readiness.sh`.
- Change the `print-fulfillment-readiness` Make target to call that wrapper.
- Extend wrapper/source acceptance tests so the final showcase source gate recognizes the wrapper and output path.
- Keep the underlying `print_fulfillment_readiness_report` semantics unchanged.

## Non-Goals

- Do not run Treatstock, Sculpteo, Meshy, OpenAI, Xcode, signing, Keychain, or payment actions.
- Do not generate or approve a real print quote.
- Do not refresh visual regression artifacts beyond the normal safe report command.
- Do not mark print fulfillment ready when visual or configured quote evidence is still blocked, stale, or missing.

## Design

The new wrapper follows the existing safe report wrapper pattern:

```sh
mkdir -p services/backend/.local
cd services/backend
uv run python -m myth_forge_api.cli print-fulfillment-readiness \
  --repo-root ../.. \
  --output .local/print-fulfillment-readiness.json
```

If the CLI exits `0` or `2`, the wrapper prints:

- `accepted print fulfillment readiness exit code $status`
- `wrote services/backend/.local/print-fulfillment-readiness.json`

and exits `0`. Any other exit code is forwarded after a stderr message.

`ios_showcase_acceptance` should require:

- the wrapper script exists,
- the Makefile calls it,
- the wrapper contains the print fulfillment CLI command,
- the wrapper writes `.local/print-fulfillment-readiness.json`,
- the wrapper accepts blocked exit code `2`.

## Verification

- RED: focused tests fail before the wrapper exists and before the Makefile/source gate use it.
- GREEN: focused tests pass after adding the wrapper and source gate.
- `make print-fulfillment-readiness` exits `0` in the current blocked state and writes `.local/print-fulfillment-readiness.json`.
- `make backend-lint`, `git diff --check`, and focused backend tests pass.
