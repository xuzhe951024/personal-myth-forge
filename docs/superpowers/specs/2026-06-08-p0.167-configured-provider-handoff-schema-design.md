# P0.167 Configured Provider Handoff Schema Design

## Context

P0.165 introduced `services/backend/src/myth_forge_api/provider_handoff.py` as
the canonical provider handoff report builder. The CLI `provider-handoff` command
and `final-local-report-refresh` now use that builder, so their reports include
`status`, `classification`, and `summary`.

`final-configured-preflight` still has a private `_provider_handoff_report`
implementation. It produces the same broad content, but not the same schema. In
configured mode, this creates two provider handoff contracts:

- `.local/provider-handoff.json` from the canonical builder.
- `final-configured-preflight.provider_handoff` from a local copy.

That drift matters for final acceptance because the user will provide provider
keys only once near the end. Every configured evidence packet should evaluate
those resources with the same status and classification vocabulary.

## Goal

Make `final-configured-preflight.provider_handoff` use the canonical
`build_provider_handoff_report(settings)` output exactly enough that downstream
configured handoff consumers can rely on:

- `kind == "provider_handoff_report"`
- `status`
- `classification`
- `summary`
- `core_real_ready`
- `missing_env`
- `next_commands`

No live provider calls, env writes, iOS config writes, Xcode actions, keychain
writes, or global mutations are allowed.

## Approaches Considered

1. **Recommended: import and call `build_provider_handoff_report`.**
   This removes duplicated schema logic and keeps all provider handoff reports
   aligned through one builder.

2. **Keep the private function but add missing fields.**
   This is a smaller diff today, but it preserves schema drift and repeats logic
   that already exists in `provider_handoff.py`.

3. **Change configured preflight to read `.local/provider-handoff.json`.**
   That would make configured preflight depend on a saved report that may be
   stale or missing. For preflight, current settings should be evaluated
   directly and safely.

## Design

`final_configured_preflight.py` will import `build_provider_handoff_report` from
`myth_forge_api.provider_handoff` and use it in
`build_final_configured_preflight_report`.

The private `_provider_handoff_report`, duplicated constants, and direct
`build_provider_readiness` import will be removed from the module. Existing
status, summary, and operator action logic can continue to read
`core_real_ready`, `missing_env`, and `next_commands` from the returned report.

Tests will assert both blocked and ready configured preflight reports include
the canonical provider handoff fields. This protects the final resource
handoff path that the operator will run after filling keys.

## Data Flow

1. Operator fills ignored final resource file.
2. `final-apply-resources` writes backend env and iOS local config.
3. `final-configured-preflight` loads current settings.
4. It calls `build_provider_handoff_report(settings)`.
5. The configured preflight report embeds the canonical handoff report.

## Error Handling

- Missing provider resources produce `status: "blocked"` and
  `classification: "core_real_providers_not_ready"`.
- Ready core providers produce `status: "ready"` and
  `classification: "core_real_providers_ready"`.
- Secret values must remain absent from serialized reports.
- Existing final configured preflight exit code behavior remains unchanged:
  `0` for ready, `2` for blocked.

## Testing

- Add failing tests to `tests/test_final_configured_preflight.py` for blocked
  and ready provider handoff schema.
- Run the focused test file red, implement the shared builder, then run green.
- Run `make backend-lint`, relevant backend pytest, and `git diff --check`.

## Non-Goals

- No provider API calls.
- No live evidence generation.
- No changes to the mobile app.
- No changes to resource application scripts.
