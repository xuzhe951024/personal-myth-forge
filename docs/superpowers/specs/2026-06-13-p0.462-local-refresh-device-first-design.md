# P0.462 Local Refresh Device-First Design

## Context

The final showcase readiness report already prioritizes the physical iPhone blocker for the local demo: write `DEVELOPMENT_TEAM`, set an iPhone-reachable backend URL, and rerun `make mobile-deploy-preflight`. The local final demo launch report follows the same device-first path.

`final-local-report-refresh` still exposes `final_resource_requirements` as its top-level `first_blocker`, which sends the operator back to `MESHY_API_KEY` before the local iPhone path. That is correct for configured/live completion, but it is the wrong top-level action for the local iOS showcase lane.

## Decision

Keep all provider resource steps visible and blocked, but promote the final showcase iOS deploy action to the top-level `first_blocker` and `next_action` when:

- there is no failed refresh step;
- `final_showcase_readiness` is blocked; and
- the showcase action is an iOS deploy/device action.

Failed steps still win over device prioritization. Provider/key blockers remain in `steps_by_id` and `operator_actions`, preserving configured/live handoff visibility.

## Acceptance Criteria

- `run_final_local_report_refresh()` reports `final_showcase_readiness` as its top-level `first_blocker` for the standard local fixture.
- `next_action` mirrors that device-first blocker and keeps `validation_command: make mobile-deploy-preflight`.
- Provider resource blockers remain available under `steps_by_id["final_resource_requirements"]` and the operator action list.
- Existing failed-step behavior still returns the failed step as the first blocker.

## Safety

This change only changes report ordering/classification for local operator guidance. It does not write secrets, run live providers, run Xcode, modify signing, mutate Keychain, or place print orders.
