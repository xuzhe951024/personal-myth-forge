# P0.210 Device Preflight Final Closure Packet Design

## Purpose

`FinalDemoLaunchReport` already embeds `final_launch_closure_packet`, and the iPhone Final Launch panel renders it under `Closure Packet`. The iPhone Device Preflight panel still stops at deploy/device rehearsal evidence, so the operator cannot see the final closure packet verdict in the compact preflight checklist.

## Scope

Add a read-only Device Preflight item with id `final_launch_closure_packet` and label `Final Closure`. The item is placed after `Launch Rehearsal`, uses the existing `FinalLaunchClosurePacketReport`, and is included in the required-ready set. This does not call backend commands, run Xcode, write resource files, call live providers, or mutate global machine state.

## Behavior

- Missing final launch report or missing closure packet returns `waiting` with `Final launch closure packet has not loaded.`
- `status == "ready"` returns `ready` and shows section/action counts, consent/global counts, and safety flags.
- `status == "blocked"` or `status == "partial"` returns `blocked`, showing the first blocker when present, first non-ready section/action, first operator action, consent/global counts, and safety flags.
- Unknown statuses return `waiting` with summary, consent, and safety details.
- All output passes through the existing Device Preflight sanitizer.

## Acceptance

- Swift contract tests cover missing, blocked, ready, configured evidence section, and redaction behavior.
- Backend mobile final launch readiness source requirements check the new item, label, source field, required-ready set entry, and Swift contract tests.
- `PersonalMythForgeMobileProjectChecks` checks the same source-level contract.
- Existing final rehearsal and app compile gates continue to pass.
