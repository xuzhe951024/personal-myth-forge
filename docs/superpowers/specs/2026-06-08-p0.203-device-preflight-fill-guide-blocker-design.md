# P0.203 Device Preflight Fill Guide Blocker Design

## Purpose

P0.202 added `final_resource_fill_guide.first_blocker`, but `DevicePreflightSummaryBuilder` still derives the fill guide blocker from `requiredInputs.first`. That can drift from backend ordering or future blocker selection rules.

## Scope

- Use `FinalResourceFillGuideReport.firstBlocker` in device preflight when present.
- Keep `requiredInputs.first` as a fallback for older payloads.
- Include blocker command/detail/validation command in the preflight detail.
- Preserve existing redaction and ready/missing behavior.
- Add source gates so mobile readiness protects the preflight first blocker source.

## Acceptance

- Device preflight contract test shows `First blocker: MESHY_API_KEY`, the fill command, backend detail, and validation command.
- Unsafe legacy required-input fallback still redacts secrets and paths.
- Mobile final launch readiness requires `guide.firstBlocker` in `DevicePreflight.swift`.
