# P0.381 iOS Rehearsal Readiness Next Action Design

## Context

`ios-device-launch-rehearsal` writes a full rehearsal report with structured
blocked sequence rows and operator actions. The wrapper
`ios_device_launch_rehearsal_readiness_report` currently preserves the sequence
and operator actions, but does not expose top-level `first_blocker` or
`next_action`.

That creates a gap in the iPhone Final Launch view: the rehearsal section is
blocked, but the most important concrete step is mixed into sequence rows and
free-form operator actions instead of being promoted as a structured next
action.

## Decision

Add `first_blocker` and `next_action` to the rehearsal readiness report:

- `first_blocker` is the first explicit blocker when blockers exist.
- Otherwise, it is the first non-ready sequence row.
- `next_action` mirrors `first_blocker` with `source: first_blocker`.
- If operator actions exist, `next_action.command` uses the first operator
  action because it is the most concrete copy-pasteable step.
- Ready reports expose both fields as `null`.

On iOS, decode these optional fields and show `next_action` before sequence
rows in `launchRehearsalRows`.

## Data Flow

```text
saved ios-device-launch-rehearsal report
  -> ios_device_launch_rehearsal_readiness.first_blocker / next_action
  -> FinalDemoLaunchReport.iosDeviceLaunchRehearsalReadiness
  -> FinalLaunchMobileSummary.launchRehearsalRows
```

## Safety

This phase only reads saved local JSON evidence and renders sanitized strings.
It does not call live providers, write provider secrets, run Xcode, sign an app,
or mutate global machine state.

## Testing

Backend tests cover blocked saved reports and missing/ready reports. Swift
contract tests cover JSON decode and Final Launch summary rendering. The work
uses RED/GREEN verification before implementation.
