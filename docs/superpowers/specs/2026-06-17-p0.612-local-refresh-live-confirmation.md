# P0.612 Local Refresh Live Confirmation

## Problem

`final-local-report-refresh` can promote configured live provider evidence commands into step-level or top-level next actions, but the promoted rows did not consistently preserve live/cost/user-confirmation metadata. That leaves the final operator refresh report less explicit than the showcase, external ledger, and closure packet.

## Scope

- Preserve `requires_live_provider_consent`, `may_call_live_provider`, `cost_risk`, `requires_cost_consent`, `live_provider_call`, and `requires_user_confirmation` when final configured evidence consent actions are promoted.
- Carry the same metadata through step `next_action`, top-level `first_blocker`, and top-level `next_action`.
- Do not run live Meshy/OpenAI/Treatstock calls, Xcode signing, Keychain writes, or global mutations.

## Acceptance

- A focused RED test proves configured evidence consent metadata is absent from `final-local-report-refresh`.
- GREEN implementation preserves metadata on the refreshed step and step next action.
- A helper-level test proves top-level blocker and next action preserve the same metadata when the live consent step is first.
- Adjacent final report refresh and handoff tests remain green.
