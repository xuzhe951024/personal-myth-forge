# P0.237 Mobile Device Action Evidence Detail Design

## Context

P0.236 made `final_showcase_readiness_report.device_action_bundle.actions[].evidence_detail` more precise. The backend report now distinguishes backend URL/health blockers from general mobile deploy preflight blockers.

The iPhone Final Launch summary decodes `evidence_detail`, but its device action rows only show evidence status and validation command. On the phone, the operator can see that device actions are blocked, but cannot see the concrete saved evidence detail such as `PMF_BACKEND_BASE_URL must be iPhone-reachable`.

## Goal

Surface device action evidence details in the mobile Final Launch summary so the iPhone display carries the exact next blocker before the physical-device handoff.

## Requirements

- Show `FinalShowcaseDeviceAction.evidenceDetail` in each device action row when present.
- Keep existing evidence status and validation command text.
- Keep sanitization on all row text.
- Preserve row count and action selection behavior.
- Do not change backend readiness classification, provider calls, Xcode/signing behavior, or local config writes.

## Non-Goals

- Do not make `ios_deployable` ready without actual device evidence.
- Do not run `backend-device-demo`, Xcode, signing, Keychain, Meshy, OpenAI, or Treatstock.
- Do not add new UI sections or change layout.

## Acceptance

- Swift contract test fails before implementation when the summary omits device action evidence details.
- Swift contract test passes after implementation and includes both backend/preflight details.
- Swift project checks still pass.
- Safe local final report refresh still ends blocked while external resources and physical-device evidence remain absent.
