# P0.93 Operator NPC Evaluation Handoff Design

## Context

P0.90 introduced `evaluate-npc`. P0.91 made the saved evaluation readiness
visible through `/v1/final-demo-launch`. P0.92 made the iPhone Demo Script block
on missing or failed NPC evaluation readiness.

The remaining integration gap is the Mac-side operator handoff. The final handoff
currently tells the operator how to apply resources, start the LAN backend, run
local acceptance, run deploy preflight, and run Xcode/configured acceptance. It
does not include the explicit `evaluate-npc` step that produces the readiness
file now required by the iPhone demo script.

## Design

`build_final_operator_handoff_report` will accept
`npc_agent_evaluation_readiness` and add a `npc_agent_evaluation` step after
`local_final_acceptance` and before iOS deploy preflight. The step uses the
existing local no-key command:

```bash
cd services/backend && uv run python -m myth_forge_api.cli evaluate-npc \
  --provider local \
  --suite default-v0 \
  --tick-steps 2 \
  --output .local/npc-evaluation-local.json
```

Status mapping:

- `ready` when readiness status is `ready`
- `missing` when readiness status is `missing`
- `blocked` when readiness status is `blocked` or `failed`
- any other status maps to `blocked` with an `unknown_npc_evaluation_status` note

`final_demo_launch` will pass the already-built readiness report into final
operator handoff. The endpoint still does not run evaluation or call providers.

## Operator Actions

When the NPC evaluation step is missing or blocked, `final_operator_handoff`
will include the readiness report's sanitized operator actions. The step detail
will include the first blocker classification/detail when present, without raw
private context, provider secrets, local absolute paths, media payloads, or
payment links.

## Tests

Backend tests will be written first:

- final operator handoff includes `npc_agent_evaluation` in the expected order
- missing readiness produces a missing step and next action to run `evaluate-npc`
- ready readiness produces a ready step with no extra next action
- blocked readiness preserves sanitized classification/detail and leaks no secret
- final demo launch passes embedded NPC readiness into final operator handoff

Swift contract coverage is not expanded for a new model because the existing
`FinalOperatorHandoffReport` and `FinalLaunchMobileSummary.handoffRows` already
decode arbitrary handoff steps and next actions.

## Non-Goals

- No new endpoint
- No iOS command execution
- No OpenAI/Meshy/Treatstock calls
- No Xcode, signing, keychain, license, deployment, network, or global config
  mutation
