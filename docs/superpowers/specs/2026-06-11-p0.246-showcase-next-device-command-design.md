# P0.246 Showcase Next Device Command Design

## Context

`final-showcase-readiness` now surfaces the complete iOS preflight blocker in the top-level detail:

```text
Next device action: make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

However, `next_action.command` still comes from the high-level `ios_deployable` capability and remains `make ios-device-launch-rehearsal`. That is technically the final rehearsal gate, but it is not the immediate operator command while mobile preflight is blocked.

## Goal

Make `final-showcase-readiness.next_action.command` point to the preferred concrete device action command when the first blocker is `ios_deployable` and a preferred blocked device action has evidence detail.

## Non-Goals

- Do not change capability ordering, status, or classification.
- Do not change the device action bundle order.
- Do not run Xcode signing, Keychain, live providers, or paid provider calls.
- Do not make `ios_deployable` ready without real iPhone/Xcode evidence.

## Approaches Considered

1. Override only `next_action.command` while keeping `first_blocker.command` unchanged.
   This is selected because `first_blocker` remains a capability summary, while `next_action` should be the immediate operator step.

2. Override both `first_blocker.command` and `next_action.command`.
   This would make the blocker less faithful to the capability's canonical gate.

3. Add a new `next_device_action` field.
   This adds report shape without solving the confusing existing command.

## Design

Add a helper that extracts the preferred concrete device action command from `device_action_bundle`. It will use the same action source as the existing top-level detail hint:

- Prefer blocked `run_mobile_deploy_preflight` when it has non-missing evidence detail.
- Fall back to the first device action when it has usable detail.
- Return an empty string when no safe concrete action exists.

`_next_action` will copy the first blocker as before, then replace the returned `command` with the preferred concrete device action command only for `ios_deployable`.

Expected report shape:

```json
{
  "first_blocker": {
    "id": "ios_deployable",
    "command": "make ios-device-launch-rehearsal"
  },
  "next_action": {
    "id": "ios_deployable",
    "command": "make mobile-deploy-preflight",
    "detail": "iOS deploy runbook and device launch rehearsal must both be ready. | Next device action: make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable"
  }
}
```

## Testing

Add a RED/GREEN backend test that builds blocked mobile deploy preflight evidence and asserts:

- `first_blocker.command` remains `make ios-device-launch-rehearsal`.
- `next_action.command` is `make mobile-deploy-preflight`.
- `next_action.detail` still includes `Missing DEVELOPMENT_TEAM` and the iPhone-reachable backend URL blocker.
- No provider secret names such as `MESHY_API_KEY` leak into the iOS next action.

Focused verification:

```bash
cd services/backend
uv run pytest tests/test_final_showcase_readiness.py::test_final_showcase_readiness_next_action_command_uses_preferred_device_action
uv run pytest tests/test_final_showcase_readiness.py
```

Final verification:

```bash
make backend-lint
make backend-test
git diff --check
make final-local-report-refresh
```

`make final-local-report-refresh` is expected to exit 2 while external provider and iPhone conditions remain missing, but the saved `final-showcase-readiness.json` must show `next_action.command` as `make mobile-deploy-preflight`.

## Self-Review

- No placeholders or deferred behavior.
- Scope is one report command selection rule and one focused regression test.
- The design does not trigger external provider calls or global Mac state changes.
