# P0.179 Mobile Project Checks Refresh Wrapper Design

## Context

After P0.178, the canonical local rehearsal command uses
`final-local-report-refresh-local`, a wrapper target that accepts the safe
blocked refresh exit code `2`. The direct `final-local-report-refresh` target
keeps exit code `2` for automation that needs to distinguish blocked from ready.

The Swift project-level checker still has two stale assumptions:

1. It checks the `final-rehearsal-local` dependency only up to
   `ios-deploy-runbook-local`, so it can pass even if the final refresh wrapper
   is missing.
2. It still expects `final_configured_preflight.py` to contain
   `build_provider_readiness`, but P0.167 intentionally changed that composition
   to `build_provider_handoff_report`.

The current failure is reproducible with:

```bash
swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks
# Project check failed: Missing configured handoff preflight provider readiness composition: build_provider_readiness
```

Backend source acceptance already checks the newer provider handoff and final
refresh wrapper contracts. The mobile project checker needs the same contract so
iOS-side validation cannot drift from the backend launch pipeline.

## Goal

Make `PersonalMythForgeMobileProjectChecks` pass against the current backend
architecture while explicitly protecting the final rehearsal refresh wrapper and
configured provider handoff composition.

## Design

Update `apps/mobile/ios/Sources/PersonalMythForgeMobileProjectChecks/main.swift`
only.

The Makefile checks will add:

- `final-local-report-refresh-local:`
- `services/backend/scripts/write_final_local_report_refresh.sh`
- the full final rehearsal dependency string ending in
  `final-local-report-refresh-local`

The configured preflight composition check will replace the stale
`build_provider_readiness` requirement with `build_provider_handoff_report`.

This keeps the Swift checker read-only. It does not run provider calls, Xcode
signing, keychain commands, resource writers, or live API calls. It only scans
source files and fails when the source no longer matches the intended launch
contract.

## Alternatives Considered

1. Reintroduce `build_provider_readiness` into `final_configured_preflight.py`.
   Rejected because P0.167 deliberately centralized configured provider evidence
   in `build_provider_handoff_report`, which includes the richer handoff schema
   used by downstream reports.

2. Leave Swift ProjectChecks with the loose final rehearsal prefix check.
   Rejected because it would miss the P0.178 regression class: a rehearsal target
   can look locally valid while still depending on a direct exit-`2` refresh
   target.

3. Remove the Swift configured preflight check.
   Rejected because mobile-side project validation is useful as a fast iOS
   deployment guard. The check should be updated to the current source of truth,
   not deleted.

## Acceptance

- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks`
  exits `0`.
- The Swift checker requires `final-local-report-refresh-local:` and
  `write_final_local_report_refresh.sh`.
- The Swift checker requires the full final rehearsal dependency ending in
  `final-local-report-refresh-local`.
- The Swift checker requires `build_provider_handoff_report` in
  `final_configured_preflight.py`.
- Backend iOS showcase source acceptance requires these Swift checker strings so
  future drift is caught by backend tests too.
- No mobile runtime UI, backend report builder, provider adapter, resource
  writer, live provider, Xcode/signing, keychain, or global configuration
  behavior changes.

## Test Strategy

Use TDD:

1. Add backend source acceptance requirements that
   `PersonalMythForgeMobileProjectChecks/main.swift` contains the new wrapper
   and provider handoff strings.
2. Run the focused backend source acceptance test and watch it fail because the
   Swift checker still contains the old strings.
3. Update the Swift checker.
4. Run Swift ProjectChecks and focused backend source acceptance.
5. Run Swift core contract tests, backend lint, and backend tests before merging.

## Root Cause

The failing Swift check is stale. It was written when final configured preflight
directly depended on provider readiness. The backend now composes the provider
handoff report instead, and P0.178 added a wrapper target for refresh exit-code
semantics. Backend source gates evolved, but Swift project-level source gates did
not.
