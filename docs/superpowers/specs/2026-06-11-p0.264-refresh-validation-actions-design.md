# P0.264 Refresh Validation Actions Design

## Goal

Preserve validation commands when `final-local-report-refresh` promotes concrete `next_action` entries into top-level `operator_actions`.

## Context

The aggregate refresh report already exposes two structured actions:

- `next_action`: the resource blocker, usually `provide MESHY_API_KEY in final-resources.env`
- `showcase_next_action`: the iOS blocker, currently `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig` plus `validation_command: make mobile-deploy-preflight`

The report currently converts both actions with `_action_command()`, which only keeps `command`. This makes the second operator action less useful than the source `final-showcase-readiness` action, which already formats the same step as:

`provide DEVELOPMENT_TEAM in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`

## Design

Update the refresh report action formatting helper:

- If an action has both `command` and `validation_command`, format it as `<command>; rerun <validation_command>`.
- If it only has `command`, keep the current behavior.
- Preserve ordering: resource `next_action` first, showcase `showcase_next_action` second, then step actions.
- Keep existing de-duplication and action limit.
- Do not change readiness statuses, step execution, external providers, resource writes, Xcode state, or ignored file layout.

## Test Plan

Update `test_final_local_report_refresh_operator_actions_use_concrete_next_actions()` so the expected second action includes `; rerun make mobile-deploy-preflight`. The test should fail before the helper change and pass after it.

## Self-Review

- No placeholders.
- Scope is one formatting helper and its focused regression test.
- The change improves the iOS operator handoff while keeping all external actions manual.
