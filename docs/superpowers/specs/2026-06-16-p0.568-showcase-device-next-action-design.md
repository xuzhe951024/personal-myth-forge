# P0.568 Final Showcase Device Next Action Design

## Context

After P0.567, the saved `ios-device-launch-rehearsal-readiness.json` can identify a concrete next device action, such as starting `backend-device-demo` and rerunning mobile preflight. The top-level `final-showcase-readiness.json` still reports the iOS deploy blocker with the broader `make ios-device-launch-rehearsal` command.

For the final iPhone showcase handoff, the operator should see the most concrete safe device action at the top level. Requiring the operator to open nested JSON to find the actionable command slows the final path and can send them through a wrapper that only re-emits the same blocker.

## Design

Thread the saved iOS device launch rehearsal readiness action into the final showcase device action bundle.

- Load the existing `ios_device_launch_rehearsal_readiness` report already built by `final_showcase_readiness`.
- If that report has a blocked `next_action` or `device_action_bundle.first_action.next_action`, expose it as metadata on the `run_ios_device_launch_rehearsal` action.
- Allow `_preferred_device_action_hint_source()` to choose `run_ios_device_launch_rehearsal` when that action has a child next action, even if mobile preflight evidence is already ready.
- Keep `make ios-device-launch-rehearsal` as the validation command so the full wrapper still refreshes the saved report after the operator acts.

This does not execute Xcode, call live providers, write secrets, or change final-goal completion criteria.

## Testing

Add a regression test where mobile preflight and Xcode evidence are ready, but saved iOS launch rehearsal readiness is blocked with a concrete backend/mobile-preflight next action. The final showcase `first_blocker`, `next_action`, and `ios_deployable` capability should all surface that concrete command while retaining `make ios-device-launch-rehearsal` as validation.
