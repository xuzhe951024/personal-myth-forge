# P0.379 iOS First Blocker Validation Design

## Context

`final-demo-launch-local.json` exposes both `first_blocker.validation_command`
and `next_action.validation_command`. The iOS `FinalDemoLaunchNextAction`
model decodes and renders its validation command in the Final Launch receipt,
but `FinalDemoLaunchFirstBlocker` does not decode the same field. This leaves
the top-level first blocker receipt less actionable on iPhone when the report
has no separate next action row or when the operator reads the first blocker
row directly.

## Decision

Add optional validation-command support to the iOS final demo first blocker
contract:

- Decode `validation_command` into `FinalDemoLaunchFirstBlocker.validationCommand`.
- Render the validation command in `finalDemoLaunchFirstBlockerReceiptRow`.
- Keep the field optional so older saved launch reports still decode.
- Keep all behavior local to the iOS Swift core model and summary builder.

No backend schema change is needed. The backend already emits the field in the
current local readiness report.

## Data Flow

```text
backend final-demo-launch first_blocker.validation_command
  -> FinalDemoLaunchFirstBlocker.validationCommand
  -> FinalLaunchMobileSummary.launchReceiptRows first blocker row
```

## Safety

The field contains a command string already present elsewhere in the report.
It is sanitized by the existing summary row sanitation path. This phase does
not write secrets, does not call live providers, does not run Xcode signing,
and does not mutate global machine state.

## Testing

Use the existing Swift contract fixture for the final demo launch top-level
first blocker. Add `validation_command` to the fixture, assert the model
decodes it, and assert the first blocker receipt row displays it. Verify RED
before implementation, then rerun the focused Swift contract tests for GREEN.
