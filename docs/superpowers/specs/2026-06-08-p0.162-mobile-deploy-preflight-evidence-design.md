# P0.162 Mobile Deploy Preflight Evidence Design

## Context

The final showcase is still blocked by iOS deploy evidence. The current reports correctly say the operator must start `backend-device-demo` and run `make mobile-deploy-preflight`, but there is no durable sanitized evidence file for the preflight itself. That makes final acceptance harder to inspect because the last device-facing check only exists as terminal output.

## Goal

Add a safe local evidence report for `make mobile-deploy-preflight` so final launch rehearsal can show whether the iOS deployment preflight has actually been run, why it is blocked, and what to do next.

## Scope

The iteration adds a backend-side report builder and CLI command:

- `mobile-deploy-preflight-evidence` runs the existing `apps/mobile/ios/scripts/deploy_preflight.sh`.
- The command captures return code, bounded stdout/stderr lines, recognized checks, operator actions, and safety metadata.
- A new Make target writes `services/backend/.local/mobile-deploy-preflight-evidence.json`.
- `final-local-report-refresh` runs the evidence command as part of its safe local refresh set.
- `ios-device-launch-rehearsal` consumes the saved evidence file in its local rehearsal report set.

Out of scope:

- No iOS config writes.
- No backend `.env` writes.
- No Xcode build or signing commands.
- No Keychain writes.
- No live provider calls.
- No automatic backend server start. The report should tell the operator to start `make backend-device-demo` if backend health is unavailable.

## Report Contract

The report shape is intentionally compact:

```json
{
  "kind": "mobile_deploy_preflight_evidence_report",
  "status": "ready | blocked",
  "command": "make mobile-deploy-preflight",
  "script": "apps/mobile/ios/scripts/deploy_preflight.sh",
  "exit_code": 0,
  "checks": [
    {
      "id": "backend_health",
      "label": "Backend health",
      "status": "ready",
      "detail": "Backend health: ok"
    }
  ],
  "stdout_lines": ["iOS deploy preflight passed."],
  "stderr_lines": [],
  "operator_actions": [],
  "safety": {
    "commands_run": true,
    "provider_calls": false,
    "live_provider_calls": false,
    "writes_backend_env": false,
    "writes_ios_deploy_config": false,
    "global_mutation": false,
    "xcode_or_signing": false,
    "keychain_writes": false,
    "provider_secrets_in_report": false,
    "local_paths_in_report": false
  }
}
```

Blocked reports use the script exit code and stderr to classify common cases:

- missing Apple Team ID or bundle identifier
- missing or loopback backend URL
- invalid final launch mode
- backend health unavailable
- tracked `Deployment.local.xcconfig`

Unknown failures remain blocked with a generic action to review `make mobile-deploy-preflight`.

## Integration

`final-local-report-refresh` should run the evidence builder after the iOS deploy runbook and before the iOS device launch rehearsal. This order lets the rehearsal consume fresh saved evidence in the same refresh pass.

`ios-device-launch-rehearsal` should include `mobile_deploy_preflight_evidence` in `LOCAL_REPORT_SOURCES`. A ready evidence report moves that local source to ready; a blocked report routes its operator actions into the rehearsal action list. This keeps the existing final readiness semantics intact while making the device preflight gate inspectable.

## Safety And Privacy

All command output is sanitized before being written:

- provider keys and bearer tokens become `[redacted]`
- repository/home paths become `[repo-root]` or `[home]`
- `file://`, checkout/payment URLs, raw media, and raw private context markers are redacted
- stdout/stderr lines are bounded so a noisy subprocess cannot bloat the report

The evidence command is allowed to run the existing preflight script because the script is read-only except for its network health request. It must not start services, mutate global state, write config, or invoke Xcode.

## Testing

Backend tests cover:

- ready report with a fake healthy backend
- blocked missing config report with actionable diagnostics
- blocked backend health report without leaking response body, tokens, or paths
- CLI writes the report and returns `0` for ready, `2` for blocked
- `final-local-report-refresh` writes the evidence report
- `ios-device-launch-rehearsal` consumes and routes the evidence report

Existing Swift contract tests remain unchanged unless the mobile model already decodes the new saved source. Visual regression remains unchanged for this iteration because this is a backend evidence/reporting improvement rather than a new screen state.

## Self-Review

- No placeholders or deferred requirements.
- The design preserves existing deploy scripts instead of replacing them.
- The report runs only a read-only preflight and writes only ignored `.local` evidence.
- The integration improves iOS deploy evidence without pretending that signing, device installation, or live provider keys are ready.
