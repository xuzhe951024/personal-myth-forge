# P0.453 Device Preflight External Actions Design

## Context

The final external action ledger is already part of `FinalDemoLaunchReport` and the iPhone Final Launch summary can now surface a hidden `backend-device-demo` handoff from `operator_actions`.

The iPhone Device Preflight panel is the operator-facing place to check whether an actual iPhone demo can proceed. It currently shows final launch, resources, iOS deploy runbook, device evidence, rehearsal readiness, certificate, closure packet, and operator handoff rows, but it does not expose the external action ledger directly. That can hide the consolidated list of remaining non-code actions:

- backend-only provider keys such as `MESHY_API_KEY` and `OPENAI_API_KEY`
- Treatstock quote/cost consent handoff
- Xcode/global Mac actions
- `backend-device-demo` before physical-device checks

## Goal

Add a compact read-only `External Actions` item to `DevicePreflightSummaryBuilder` so the iPhone preflight surface shows the final external action ledger status and the backend-device-demo handoff when present.

## Approaches Considered

1. Add a dedicated Device Preflight row for `FinalExternalActionLedgerReport`.
   - Pro: mirrors existing preflight rows, keeps Final Launch summary unchanged, and gives the physical-device operator one place to inspect external blockers.
   - Con: adds another row to an already dense preflight list.

2. Fold the ledger text into the existing `Final Launch` row.
   - Pro: no extra row.
   - Con: the existing row already prioritizes the first blocker; adding ledger content would make it long and harder to scan.

3. Show only the ledger first blocker.
   - Pro: smallest implementation.
   - Con: repeats current first-blocker behavior and can still hide `backend-device-demo`.

Recommended approach: option 1. The preflight panel is already a checklist, so a dedicated row is clearer and avoids overloading the Final Launch row.

## Design

Add `externalActionLedgerItem(report:)` to `DevicePreflightSummaryBuilder` and place it after the `Final Launch` row.

Status mapping:

- `ready` -> `.ready`
- `blocked`, `missing`, `manual`, or `live` -> `.blocked`
- any other loaded status -> `.waiting`
- missing `final_external_action_ledger` -> `.waiting`

Detail text should be compact and sanitized through the existing `item(...)` helper:

- summary: ledger status, group count, blocked, missing, live, manual
- next action if present: id, status, command, validation command
- first blocker if present and different in content from next action
- up to three non-ready action group summaries
- operator action rows: first non-empty operator action plus first non-empty `backend-device-demo` action, deduped
- consent/safety: global confirmation, live cost consent, commands run, global mutation, live calls

Do not change backend JSON contracts, model decoding, SwiftUI layout, or Final Launch summary behavior.

## Testing

Add a MobileCore contract test that builds a final demo launch report whose external ledger operator actions include:

1. a resource/apply preview action
2. the backend-device-demo handoff
3. a live provider cost action

Before implementation, `DevicePreflightSummaryBuilder` should fail because `summary.item(id: "external_actions")` is missing. After implementation, the detail should include the ledger status, first operator action, and backend-device-demo action.

Focused verification:

```bash
swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests
swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks
git diff --check
```

This phase does not build/sign with Xcode, write Keychain state, start the backend, call live providers, write secrets, or place print orders.
