# P0.236 Device Action Evidence Detail Design

## Context

The final showcase readiness report is still blocked by `ios_deployable`, which is expected without real iPhone/Xcode/provider evidence. The existing `device_action_bundle` already splits the iOS device path into:

- start `backend-device-demo`
- run mobile deploy preflight
- resolve the Xcode build gate
- rerun iOS device launch rehearsal

However, the first two rows currently reuse the same first mobile preflight check detail. When saved evidence contains multiple blockers, such as missing `DEVELOPMENT_TEAM` and a loopback backend URL, both rows can show the signing blocker even though the backend server row should point to backend reachability and the preflight row should summarize all deploy preflight blockers.

## Goal

Make final showcase readiness more actionable before the physical iPhone handoff by giving each device action a step-specific evidence detail.

## Requirements

- `start_backend_device_demo` should prefer backend-health/backend-URL evidence when present, so the row explains the LAN backend blocker instead of an unrelated signing blocker.
- `run_mobile_deploy_preflight` should summarize all saved preflight check details, capped to a short readable sentence.
- Ready evidence should keep the current concise success detail.
- Missing/unreadable evidence should keep the existing fallback behavior.
- The report remains read-only and must not run Xcode, signing, Keychain, providers, or live printing.
- Sanitization must continue to redact secrets, local paths, payment links, and private context.

## Non-Goals

- Do not make `ios_deployable` ready without real device evidence.
- Do not start `backend-device-demo`.
- Do not write or repair ignored deploy config.
- Do not alter provider key requirements.

## Acceptance

- Focused backend test proves backend action evidence prefers backend URL/health blockers.
- Focused backend test proves preflight action evidence summarizes multiple blockers.
- Existing readiness tests continue to pass.
- `final-showcase-readiness` still exits blocked on this machine, but the saved report gives clearer device action details.
