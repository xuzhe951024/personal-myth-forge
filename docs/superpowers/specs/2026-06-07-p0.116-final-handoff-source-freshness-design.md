# P0.116 Final Handoff Source Freshness Design

## Context

P0.115 blocks stale `ios-device-launch-rehearsal.json` evidence by comparing
that saved report's modification time with the current git HEAD commit time.
This protects the top-level rehearsal receipt, but the rehearsal still depends
on a `final-handoff-index.json` file that summarizes several saved reports:

- `3d-evaluation-local.json`
- `npc-evaluation-local.json`
- `final-acceptance-local.json`
- `final-demo-launch-local.json`
- `ios-deploy-runbook-local.json`

If one of those source reports is old, the final handoff index can still look
ready. On the iPhone this creates a false sense of readiness: the operator sees
a fresh rehearsal wrapper, while the evidence inside the indexed local lane may
come from before the current code revision.

P0.116 adds a read-only source freshness layer to the final handoff index and
surfaces that layer through `/v1/final-demo-launch` and the iPhone final launch
summary.

## Approaches Considered

1. Only keep the P0.115 rehearsal freshness guard. This is simple, but it
   misses stale source reports inside the handoff index.

2. Add freshness checks directly to `final_handoff_index.source_reports`. This
   keeps the report self-contained, blocks the local rehearsal lane when any
   required saved source is stale, and lets downstream rehearsal/final-launch
   payloads show the exact stale source.

3. Add a separate top-level "all evidence freshness" endpoint. This might be
   useful later, but it adds another report surface and makes the iPhone panel
   reconcile multiple freshness concepts.

P0.116 uses approach 2.

## Design

`services/backend/src/myth_forge_api/final_handoff_index.py` will add a
`freshness` object to each saved local source report. The shape matches the
existing final acceptance and rehearsal freshness payload:

```json
{
  "status": "fresh",
  "classification": "fresh_report",
  "checked_against": "git_head",
  "source_modified_at": "2026-06-07T12:05:00Z",
  "current_revision": "abc1234",
  "current_revision_committed_at": "2026-06-07T12:00:00Z"
}
```

Freshness rules:

- missing source file: `status=unknown`, `classification=source_missing`
- git metadata unavailable: `status=unknown`, `classification=git_unavailable`
- source file mtime older than current git HEAD commit time:
  `status=stale`, `classification=stale_report`
- source file mtime equal to or newer than current git HEAD commit time:
  `status=fresh`, `classification=fresh_report`

For required local source reports, stale evidence changes the source report
`status` to `blocked` and uses `classification=stale_report`. The local
rehearsal lane then becomes blocked through the existing lane status aggregator.
The final handoff index receives a new `freshness_summary`:

```json
{
  "fresh": 5,
  "stale": 0,
  "unknown": 1
}
```

The operator sequence and command list stay unchanged. The next action remains
the existing local rerun command:

```text
make final-rehearsal-local
```

`ios_device_launch_rehearsal_readiness` will not re-check each local source on
disk. It will copy a bounded freshness summary from the saved
`final_handoff_index` step when present in the saved rehearsal sequence. This
keeps the rehearsal readiness endpoint read-only and avoids duplicating source
inspection logic.

The iOS model will decode the new optional freshness fields on rehearsal
sequence rows:

- `freshness_status`
- `freshness_classification`
- `freshness_summary`

`FinalLaunchMobileSummary.launchRehearsalRows` will show stale source evidence
near the existing rehearsal freshness row:

- `Source freshness: 5 fresh, 1 stale, 0 unknown.`
- `final_handoff_index: blocked | make final-handoff-index | stale_report`

The existing SwiftUI `FinalLaunchStatusView` already renders
`launchRehearsalRows`, so no new visual component is needed.

## Safety

The backend may run read-only git metadata commands:

- `git rev-parse --short HEAD`
- `git log -1 --format=%ct HEAD`

The feature must not run handoff/rehearsal commands, write ignored reports,
apply resources, write backend `.env`, write iOS deploy config, call
Meshy/OpenAI/Treatstock, start servers, run deploy preflight, run Xcode, touch
signing/keychain state, accept Apple licenses, or mutate global machine state.

Sanitizers must continue to remove provider secrets, bearer tokens, raw media
URIs, payment links, and absolute local paths from all new fields.

## Tests

Backend tests should prove:

- missing local handoff sources include `freshness.status == "unknown"` and
  `classification == "source_missing"`
- fresh local source reports keep `source_reports[*].status == "ready"`
- stale required local source reports become `blocked` with
  `classification == "stale_report"`
- `freshness_summary` counts fresh, stale, and unknown source reports
- `/v1/final-demo-launch` embeds the final handoff source freshness in the
  rehearsal readiness sequence after saved rehearsal JSON is written
- source gates require final handoff source freshness coverage

Swift tests should prove:

- `FinalDemoLaunchReport` decodes rehearsal sequence source freshness fields
- `FinalLaunchMobileSummary.launchRehearsalRows` displays stale source freshness
- unsafe freshness strings are redacted from the encoded mobile summary

Visual evidence should show a 390px-wide iPhone final launch status surface
where the Launch Rehearsal section includes the source freshness count and stale
source action.

## Non-Goals

- Do not add a new endpoint.
- Do not change live provider behavior.
- Do not call live providers.
- Do not run or configure Xcode.
- Do not write provider keys, backend `.env`, or iOS deploy config.
- Do not require the iPhone app to execute Mac-side commands.
