# P0.606 Final Showcase Live Consent Next Action

## Context

`final-showcase-readiness` already marks the live 3D and NPC capability rows with
`requires_live_provider_consent` and `completion_requires_live_provider_consent`.
Its top-level `first_blocker` keeps those fields, but the top-level `next_action`
can drop them. This makes the iPhone-visible final launch handoff less explicit when
the next step is a live Meshy/OpenAI action that needs cost consent.

## Decision

The final showcase top-level `next_action` should copy live-provider consent metadata
from the selected first blocker. This keeps the report machine-readable while leaving
capability ordering, command choice, validation commands, and provider execution
unchanged.

## Acceptance

- When `game_asset_3d_generation` is the first blocker, top-level `next_action`
  includes `requires_live_provider_consent: true`.
- The same `next_action` includes `completion_requires_live_provider_consent: true`.
- Existing capability row `next_action` metadata remains unchanged.
- No Meshy, OpenAI, print-provider, Xcode signing, or global machine action is run.
