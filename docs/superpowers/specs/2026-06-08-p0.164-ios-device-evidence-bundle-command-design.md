# P0.164 iOS Device Evidence Bundle Command Design

## Context

P0.163 made Xcode build evidence explicit with
`make mobile-xcode-build-evidence`. The lower-level
`ios_device_evidence_bundle` report can now consume that saved Xcode evidence,
but it still has no first-class CLI or Make entry point. The only way to inspect
it is through Python internals or reports that embed it indirectly.

For final iOS demo acceptance, the operator needs a direct report that says:

- whether the backend device server, mobile deploy preflight, Xcode build gate,
  and iOS launch rehearsal are ready;
- which saved report proves each slot;
- which local/global action is still required;
- whether the Xcode slot is blocked by installation, license, or project build.

## Goal

Expose the iOS device evidence bundle as a first-class local acceptance artifact
with CLI, Make, and `final-local-report-refresh` integration.

## Non-Goals

- Do not run Xcode, signing, keychain, or `xcode-select`.
- Do not run live providers or external print services.
- Do not mutate `.env`, `final-resources.env`, or iOS deploy config.
- Do not replace the underlying device preflight, Xcode evidence, or launch
  rehearsal commands.

## Options Considered

### Option A: Keep bundle internal only

This avoids code changes but leaves the most useful device acceptance report
hidden behind embedded final demo launch JSON. It makes final acceptance harder
to inspect and repeat.

### Option B: Add CLI/Make only

This gives a direct operator command but keeps `final-local-report-refresh` from
writing the bundle as part of its local evidence set. The operator would need to
remember one more separate command.

### Option C: Add CLI/Make and include it in final refresh

This is the selected approach. Add:

```bash
make ios-device-evidence-bundle
```

and wire `final-local-report-refresh` to write:

```text
services/backend/.local/ios-device-evidence-bundle.json
```

The refresh integration only reads saved reports; it does not execute device,
Xcode, signing, or live provider commands.

## Architecture

Add a CLI subcommand:

```bash
cd services/backend
uv run python -m myth_forge_api.cli ios-device-evidence-bundle \
  --repo-root ../.. \
  --output .local/ios-device-evidence-bundle.json
```

Add a Make target:

```bash
make ios-device-evidence-bundle
```

Add a `final-local-report-refresh` step after
`ios_device_launch_rehearsal` and before `live_provider_evidence`:

```text
ios_device_evidence_bundle -> services/backend/.local/ios-device-evidence-bundle.json
```

The step calls `build_ios_device_evidence_bundle_report(repo_root=repo_root)` and
therefore stays read-only apart from writing its own `.local` report.

## Safety

The new command and refresh step must keep the existing bundle safety contract:

- `commands_run: false`
- `provider_calls: false`
- `live_provider_calls: false`
- `global_mutation: false`
- `xcode_or_signing: false`
- `keychain_writes: false`
- `writes_backend_env: false`
- `writes_ios_deploy_config: false`
- `describes_global_actions: true`

## Acceptance

- `myth_forge_api.cli ios-device-evidence-bundle` writes a JSON report and exits
  `0` when all slots are ready, otherwise `2`.
- `make ios-device-evidence-bundle` writes
  `services/backend/.local/ios-device-evidence-bundle.json`.
- `make final-local-report-refresh` writes the same report as part of the local
  evidence set.
- The bundle uses `services/backend/.local/mobile-xcode-build-evidence.json`
  when present.
- Existing final refresh, final demo launch, backend lint/test, Swift checks,
  and visual regression remain valid.
