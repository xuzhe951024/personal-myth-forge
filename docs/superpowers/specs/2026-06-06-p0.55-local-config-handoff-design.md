# P0.55 Local Config Handoff Design

## Product Intent

P0.54 made `make mobile-deploy-preflight` strict enough to catch missing iOS
signing values and unreachable backend health before a physical iPhone demo.
The remaining handoff gap is operator setup: the project tells the user what
values are needed, but still expects manual editing of an ignored xcconfig file.

P0.55 adds a project-local config writer that creates or updates
`apps/mobile/ios/Config/Deployment.local.xcconfig` from explicit values. This
keeps the final acceptance path ready for user-provided resources without
mutating global Xcode settings, touching the keychain, signing the app, running
a device, or calling live Meshy/OpenAI/print providers.

## Behavior

Add `apps/mobile/ios/scripts/write_deploy_local_config.sh`.

The script writes only these keys:

- `DEVELOPMENT_TEAM`
- `PRODUCT_BUNDLE_IDENTIFIER`
- `PMF_BACKEND_BASE_URL`

Inputs come from environment variables by default:

```sh
DEVELOPMENT_TEAM=ABCDE12345 \
PRODUCT_BUNDLE_IDENTIFIER=com.example.personalmythforge \
PMF_BACKEND_BASE_URL=http://192.168.1.10:8080 \
apps/mobile/ios/scripts/write_deploy_local_config.sh
```

It also supports equivalent flags so the Make target and direct shell use are
ergonomic:

```sh
apps/mobile/ios/scripts/write_deploy_local_config.sh \
  --team ABCDE12345 \
  --bundle-id com.example.personalmythforge \
  --backend-url http://192.168.1.10:8080
```

The script fails without writing when:

- any required value is missing.
- `PMF_BACKEND_BASE_URL` is loopback or empty.
- the local config file is tracked by git.
- an unknown flag is passed.

On success it writes the ignored local xcconfig using a temporary file and
atomic move. Output is concise:

```text
Deployment.local.xcconfig written.
Bundle: com.example.personalmythforge
Backend: http://192.168.1.10:8080
Next: make mobile-deploy-preflight
```

The file contains no provider secrets. The script must not call `sudo`,
`xcode-select`, `xcodebuild -license`, `security`, `codesign`, Meshy, OpenAI, or
printing services.

## Make Target

Add:

```sh
make mobile-write-deploy-config
```

The target invokes the script. Users can supply values through env vars:

```sh
DEVELOPMENT_TEAM=ABCDE12345 \
PRODUCT_BUNDLE_IDENTIFIER=com.example.personalmythforge \
PMF_BACKEND_BASE_URL=http://192.168.1.10:8080 \
make mobile-write-deploy-config
```

This target prepares local config only. It does not validate backend health;
the existing `make mobile-deploy-preflight` remains the validation command.

## Acceptance

Add pytest coverage that runs the real script inside a temporary git repo:

- missing values exits `2` and writes no local config.
- loopback backend exits `2` and writes no local config.
- unknown flag exits `2` and writes no local config.
- tracked `Deployment.local.xcconfig` exits `1` and refuses to overwrite it.
- env input writes the expected three-key xcconfig.
- flag input overrides env input and writes the expected values.
- running the writer, then running `deploy_preflight.sh` against a local
  test HTTP server returning `{"status":"ok"}`, passes.

Extend Swift project source checks to prove the writer exists, the Make target
exists, the required keys are present, tracked-local-config protection exists,
and banned global mutation/provider commands are absent.

## Docs and Evidence

Update README and mobile README with the new two-step local handoff:

1. `make mobile-write-deploy-config`
2. `make mobile-deploy-preflight`

Create static visual evidence showing the handoff lane:

- user-owned values enter the writer.
- ignored local xcconfig is produced.
- deploy preflight validates local config and backend `/health`.
- global machine state and live providers remain untouched.

## Non-Goals

- No Apple developer portal, provisioning profile, certificate, keychain, or
  Xcode signing automation.
- No simulator/device run.
- No backend `.env` writer in this slice.
- No provider key storage.
- No live Meshy/OpenAI/Treatstock/Sculpteo calls.
