# P0.581 Closure Optional Action Pruning Design

## Context

After refreshing local device evidence with `backend-device-demo` running, the final showcase moved from iOS evidence blocked to live provider partial. The closure packet still surfaced optional resource rows, such as `OPENAI_API_BASE_URL` and optional print provider keys, as top-level operator actions.

Those optional rows are useful inside the resource section, but they are not required operator work once the section is ready. Keeping them above live provider consent makes the final handoff look less executable.

## Decision

`final_launch_closure_packet._operator_actions()` should skip actions whose status is `optional`, the same way it already skips `ready`.

The report still preserves optional rows inside `sections_by_id.resource_inputs.actions`. Only the top-level operator queue is pruned.

## Acceptance

- Optional resource rows do not produce `make final-resources-preflight`.
- Optional print rows do not produce the backend-auto print readiness action.
- Live provider consent remains the first operator action when it is the remaining blocker.
- Existing blocked, missing, partial, live, and manual actions still produce operator guidance.
