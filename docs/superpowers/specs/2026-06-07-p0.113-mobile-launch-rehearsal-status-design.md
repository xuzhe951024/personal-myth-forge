# P0.113 Mobile Launch Rehearsal Status Design

## Context

P0.112 added `make ios-device-launch-rehearsal`, a safe Mac-side wrapper that
refreshes the ignored JSON evidence set before the physical iPhone operator
pass. The report is useful, but it only lives in
`services/backend/.local/ios-device-launch-rehearsal.json` and the iPhone Final
Launch panel cannot show whether that rehearsal was run recently or why it is
blocked.

P0.113 exposes that saved rehearsal result through the existing read-only final
launch surface. The goal is not to make the phone run Mac commands. The goal is
to make the phone display enough sanitized evidence for the operator to confirm
that the safe Mac-side rehearsal was refreshed before continuing to deploy and
live-provider acceptance.

## Approaches Considered

1. Add a new `/v1/ios-device-launch-rehearsal` endpoint.
   This is direct, but it adds another mobile fetch path and splits final launch
   readiness across multiple app calls.

2. Embed the rehearsal readiness in `/v1/final-demo-launch`.
   This follows the existing pattern for final acceptance, 3D evaluation, NPC
   evaluation, resource handoff, and iOS deploy runbook readiness. It keeps the
   iPhone Final Launch panel as the single operator-facing launch digest.

3. Only document that Mac operators should inspect the JSON manually.
   This preserves code simplicity but weakens the final iPhone demo because the
   phone cannot show whether the launch evidence is current.

P0.113 uses approach 2.

## Backend Design

Add a focused readiness module,
`myth_forge_api.ios_device_launch_rehearsal_readiness`, that reads
`services/backend/.local/ios-device-launch-rehearsal.json` without running any
commands. The module returns a sanitized report with:

- `kind`: `ios_device_launch_rehearsal_readiness_report`
- `status`: `missing`, `ready`, `partial`, or `blocked`
- `source_file`: stable relative path plus existence flag
- `summary`: ready/missing/blocked/partial/manual/live counts copied from the
  saved report when present
- `sequence`: compact per-step status rows from the saved rehearsal report
- `operator_actions`: short next actions, preferring saved actions and falling
  back to `run make ios-device-launch-rehearsal`
- `commands`: the safe refresh command and report path
- `safety`: all report-only safety flags, including no provider calls, no global
  mutation, no secret/raw media/payment/local-path leakage

`build_final_demo_launch_report` will embed this readiness as
`ios_device_launch_rehearsal_readiness`. The existing final launch command list
will include `make ios-device-launch-rehearsal` so both Mac and iPhone surfaces
point to the same refresh command.

This backend work is read-only. It does not call providers, apply secrets, start
servers, run deploy preflight, run Xcode, write config, touch signing/keychain,
or mutate global state.

## Mobile Design

Add Codable Swift models for `IOSDeviceLaunchRehearsalReadinessReport`, its
summary, source file, sequence rows, and safety block. Extend
`FinalDemoLaunchReport` with an optional
`iosDeviceLaunchRehearsalReadiness` field.

`FinalLaunchMobileSummaryBuilder` will add `launchRehearsalRows`:

- missing: "Run iOS device launch rehearsal to refresh final evidence."
- blocked/partial: show up to three blocking sequence rows or operator actions
- ready: show ready counts and confirm the safe rehearsal evidence exists
- always sanitize commands, local paths, keys, raw media, and payment links

`FinalLaunchStatusView` will render those rows under a new compact
`Launch Rehearsal` section. It should reuse the existing grouped-line style and
avoid adding new interaction patterns.

## Testing

Backend tests:

- missing saved rehearsal produces a `missing` readiness report with no leaks
- saved blocked/partial rehearsal is embedded into final demo launch
- final demo launch commands include `make ios-device-launch-rehearsal`

Swift tests:

- decode final launch payloads containing rehearsal readiness
- build ready and blocked `launchRehearsalRows`
- redact unsafe report text from rehearsal rows

Source gates:

- `ios_showcase_acceptance` must require backend readiness integration, Swift
  models, summary rows, UI section, and contract tests.
- `PersonalMythForgeMobileProjectChecks` must check the same source-level
  contract for no-global-Xcode verification.

Visual regression:

- Add a small mobile-width HTML evidence card showing the new Launch Rehearsal
  group in blocked and ready states.

## Non-Goals

- Do not run `make ios-device-launch-rehearsal` from the backend endpoint or the
  iPhone app.
- Do not add a new mobile endpoint.
- Do not perform live Meshy/OpenAI/Treatstock calls.
- Do not create provider keys or write ignored config files.
- Do not run Xcode, sign an app, touch keychain, accept Apple licenses, or modify
  global developer settings.
