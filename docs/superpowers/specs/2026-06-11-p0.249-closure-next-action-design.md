# P0.249 Closure Packet Next Action Design

## Context

`final-launch-closure-packet` is the highest-level handoff packet that groups:

- resource inputs
- safe local writes
- iOS device evidence
- live provider consent
- configured evidence bundle
- final acceptance

It already computes `first_blocker`, but it does not expose a top-level `next_action`. The saved report therefore has enough information to identify the first blocker, but not the same immediate-action shape used by `final-showcase-readiness` and `final-demo-launch-local`.

## Goal

Add `next_action` to `final-launch-closure-packet` so the final handoff packet directly tells the operator the next command/action to run.

## Non-Goals

- Do not change section ordering, status semantics, or blocker selection.
- Do not execute Xcode, signing, Keychain, live providers, print orders, or paid provider calls.
- Do not write provider keys or local resource values into tracked files.
- Do not make blocked sections appear ready.

## Approaches Considered

1. Map `first_blocker` into a top-level `next_action`.
   This is selected because it keeps the existing blocker priority and gives the report a standard action field.

2. Recalculate next action from all sections independently.
   This risks diverging from the existing first-blocker logic.

3. Leave `next_action` null and rely on `first_blocker`.
   This keeps current behavior but leaves the final handoff packet less actionable than lower-level reports.

## Design

Add a helper:

```python
def _next_action(first_blocker: dict[str, Any] | None) -> dict[str, Any] | None:
    ...
```

When `first_blocker` exists, return:

```json
{
  "id": "...",
  "label": "...",
  "status": "...",
  "classification": "...",
  "command": "...",
  "detail": "...",
  "source": "first_blocker",
  "section_id": "...",
  "action_id": "..."
}
```

When there is no blocker, return `null`.

The report builder will compute `first_blocker` once, then pass it into `_next_action`.

## Testing

Add a RED/GREEN backend test that builds a missing-resource closure packet and asserts:

- `first_blocker.id == "resource_inputs"`
- `next_action.id == "resource_inputs"`
- `next_action.command == "make final-resources-preflight"`
- `next_action.detail` includes the Meshy API key handoff detail
- `next_action.source == "first_blocker"`
- `next_action.section_id` and `next_action.action_id` are preserved
- no secret values leak

Focused verification:

```bash

uv run pytest tests/test_final_launch_closure_packet.py::test_final_launch_closure_packet_exposes_next_action_from_first_blocker

```


`make final-local-report-refresh` is expected to exit 2 while external resources and iPhone/Xcode conditions remain missing, but `services/backend/.local/final-launch-closure-packet.json` must include a non-null `next_action`.

## Self-Review

- No placeholders or deferred behavior.
- Scope is one report field and one regression test.
- The design improves final handoff clarity without changing blocked external conditions.
