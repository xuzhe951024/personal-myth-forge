# P0.434 Device Preflight Saved Next Action Design

## Context

P0.430-P0.433 added a backend `device_action_bundle` for
`ios_device_launch_rehearsal_readiness`, preserved saved sequence
`operator_actions`, decoded `saved_next_action` in Swift, and displayed it in
the iPhone Final Launch summary. The iPhone Device Preflight panel still shows
the rehearsal rerun step and blocker detail, but not the saved concrete next
device action that will unblock the physical iPhone path after the rehearsal
refresh.

The final goal is still blocked by external device resources:
`DEVELOPMENT_TEAM`, an iPhone-reachable `PMF_BACKEND_BASE_URL`, real iPhone or
Xcode evidence, and live provider keys/consent. This slice does not resolve
those external gates; it makes the next local operator action visible in the
preflight surface used before the device demo.

## Decision

Extend `DevicePreflightSummaryBuilder` so the
`ios_device_launch_rehearsal_readiness` item appends the first action's
`savedNextAction` when present:

```text
Saved next: DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto
| Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

The saved action is read from:

```swift
readiness.deviceActionBundle?.firstAction.savedNextAction
```

The existing detail sanitizer remains the single redaction boundary because all
Device Preflight items already flow through `item(...)`.

## Scope

Included:

- Add a Swift contract test proving Device Preflight renders the saved next
  action command and detail.
- Update `DevicePreflight.swift` only enough to append this detail for blocked
  or partial launch rehearsal readiness.
- Add source-inspector coverage for the new contract test.
- Update the roadmap entry for P0.434.

Not included:

- No backend schema changes.
- No provider calls, live Meshy/OpenAI/Treatstock checks, or paid actions.
- No Xcode signing, Keychain, global environment edits, or device deployment.
- No change to the Final Launch summary, which already renders saved next
  action.
