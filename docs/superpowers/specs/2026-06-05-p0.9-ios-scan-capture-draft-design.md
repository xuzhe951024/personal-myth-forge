# P0.9 iOS Scan Capture Draft Design

Date: 2026-06-05

## Context

P0.8 produced an Xcode app shell and a locally verified SwiftPM core. The next
gap on the path to an iOS-deployable MVP is object capture. The backend already
accepts scan-oriented capture modes and content types:

- `single_photo`
- `photo_set`
- `manual_upload`
- `arkit_scan`
- `image/jpeg`, `image/png`, `image/heic`, `image/heif`
- `model/gltf-binary`, `model/vnd.usdz+zip`, `application/octet-stream`

The iOS core currently only builds image upload parts, so an iOS scan bundle
cannot be represented or uploaded safely.

## Goal

Add a mobile-side capture draft layer that can model and validate single-photo,
multi-photo, and future ARKit scan uploads before they are sent to
`POST /v1/object-captures`.

## Non-Goals

- No real camera, PhotosPicker, ARKit, RealityKit, LiDAR, or Object Capture API
  runtime integration.
- No simulator/device build on this machine.
- No provider keys or secrets in the app.
- No backend storage rewrite; the current backend capture contract already
  supports the content types P0.9 needs.

## Chosen Approach

P0.9 stays in SwiftPM-verifiable code and app source:

- Add strongly typed capture modes and capture media draft values in
  `PersonalMythForgeMobileCore`.
- Add `CaptureDraft`, a value object that turns UI form state plus selected media
  into `ObjectCaptureMetadata` and `[CaptureUpload]`.
- Add validation rules that mirror the backend contract and provide local user
  errors before network calls:
  - `single_photo`: exactly one image.
  - `photo_set`: two to twelve images.
  - `manual_upload`: one to twelve supported files.
  - `arkit_scan`: at least one scan asset and up to twelve total files.
- Extend `PersonalMythForgeAPIClient` safe filename/content-type handling to
  support image and scan asset uploads.
- Update the SwiftUI capture form to surface the selected capture mode and scan
  readiness state as source-level app UI.
- Add a static mobile storyboard and Browser screenshot for visual regression
  until full Xcode/simulator verification is available.

## Data Model

```swift
public enum CaptureMode: String, Codable, Equatable, CaseIterable {
    case singlePhoto = "single_photo"
    case photoSet = "photo_set"
    case manualUpload = "manual_upload"
    case arkitScan = "arkit_scan"
}

public enum CaptureMediaKind: String, Codable, Equatable {
    case image
    case scanAsset
}

public struct CaptureMediaDraft: Equatable {
    public let originalFilename: String
    public let contentType: String
    public let data: Data
    public let kind: CaptureMediaKind
}

public struct CaptureDraft: Equatable {
    public let label: String
    public let materialsText: String
    public let visualNotes: String
    public let source: String
    public let mode: CaptureMode
    public let media: [CaptureMediaDraft]

    public func validatedUploadPayload() throws -> CaptureUploadPayload
}
```

`CaptureUploadPayload` contains backend-ready `ObjectCaptureMetadata` and
`[CaptureUpload]`.

## Error Handling

New validation errors stay in the core layer:

```swift
public enum CaptureDraftValidationError: Error, Equatable {
    case missingLabel
    case unsupportedContentType(String)
    case invalidMediaCount(CaptureMode, Int)
    case missingScanAsset
}
```

The API client still throws `ForgeFlowError.encodingFailed` for unsafe upload
payloads at the network boundary.

## Visual Regression

Add a P0.9 static storyboard that shows:

- capture mode segmented control shape
- single photo, photo set, and ARKit scan states
- scan asset readiness text
- upload summary bound to backend capture modes
- no horizontal overflow at `390x844`

This is storyboard evidence, not simulator evidence.

## Acceptance Criteria

- Swift contract tests cover valid `single_photo`, `photo_set`, and `arkit_scan`
  capture drafts.
- Swift contract tests reject missing labels, wrong media counts, unsupported
  content types, and ARKit drafts without scan assets.
- API client upload builder supports GLB, USDZ, octet-stream scan assets, and
  still sanitizes filenames.
- Project checks still pass.
- Backend lint and tests still pass.
- Mobile docs explain that P0.9 is a capture draft layer, not real ARKit runtime.
- Browser visual evidence exists for the P0.9 capture/scan storyboard.
