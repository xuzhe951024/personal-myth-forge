# P0.198 Resource Requirements First Blocker Design

## Purpose

`final_local_report_refresh` can infer that `MESHY_API_KEY` is the first missing resource, but `final_resource_requirements_report` itself does not expose a structured `first_blocker`. That makes upstream reports reimplement the same inference and weakens the final launch blocker chain.

## Scope

- Add top-level `first_blocker` to `final_resource_requirements_report`.
- Use the first required requirement whose status is `missing` or `blocked`.
- Include id, label, status, classification, command, detail, domain, destination, and validation_command.
- Keep secret values out of the report; expose only the requirement id and existing metadata.
- Add backend tests and iOS showcase source gates.

## Acceptance

- Missing `MESHY_API_KEY` produces a first blocker with command `provide MESHY_API_KEY in final-resources.env`.
- Ready reports produce `first_blocker: null`.
- Selecting `PRINT_PROVIDER=treatstock` with no Treatstock key produces a print-provider blocker.
- Existing safety metadata remains unchanged.
