# P0.248 Refresh Check Detail Summary Design

## Context

`final-local-report-refresh` promotes each child report into an operator-facing step row. For reports that expose a `checks` list, the current hint logic uses only the first non-ready check's detail.

For `mobile-deploy-preflight-evidence`, this means the refresh report can show only:

```text
Missing DEVELOPMENT_TEAM
```

while the same report also contains another blocker:

```text
PMF_BACKEND_BASE_URL must be iPhone-reachable
```

The final iOS demo operator needs both values before a physical-device rehearsal can progress.

## Goal

Make `final-local-report-refresh` summarize all non-ready check details for `checks`-based child reports while preserving the existing command selection.

## Non-Goals

- Do not change report status semantics.
- Do not run mobile deploy preflight, Xcode, signing, Keychain, providers, or paid actions.
- Do not alter `first_blocker` or `next_action` handling.
- Do not add provider secrets or local filesystem paths to reports.

## Approaches Considered

1. Join all non-ready check details in `_blocker_hint` when a report has checks.
   This is selected because it keeps the existing report shape and gives the operator the full missing-input list.

2. Special-case `mobile_deploy_preflight_evidence`.
   This solves the immediate issue but duplicates report-specific logic in a generic summarizer.

3. Leave the first check only and rely on lower-level JSON.
   This keeps current behavior but makes the summary row incomplete.

## Design

When `_blocker_hint(report)` reaches the `checks` fallback:

- collect all checks whose status is not `passed` or `ready`
- use the first blocked check as the command/classification source, preserving current command behavior
- set `detail` to the semicolon-joined details from all non-ready checks
- deduplicate repeated details
- fall back to the first check's detail when no other details exist

Expected refresh step shape:

```json
{
  "id": "mobile_deploy_preflight_evidence",
  "status": "blocked",
  "command": "provide DEVELOPMENT_TEAM in Deployment.local.xcconfig",
  "detail": "Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable"
}
```

## Testing

Add a RED/GREEN backend test with an injected child report that has two blocked checks. Assert the promoted step detail includes both blocker details in order, while the command still comes from the first operator action.

## Self-Review

- No placeholders or deferred behavior.
- Scope is one generic checks-detail summarization rule and one regression test.
- The change improves operator clarity without executing blocked external actions.
