# P0.98 Mobile Capture Generation Receipt Design

## Context

The iPhone flow can capture photos, guided scan images, or ARKit scan packages,
upload them to `/v1/object-captures`, create a myth session from the capture,
and receive generation provenance from the backend. The app already shows media
selection before forging and generation provenance after the artifact appears,
but it does not show a compact receipt that ties the physical capture to the
generated 3D asset.

For a physical-device demo, the operator needs one phone-visible answer to:
"Did this scan/photo actually become the 3D generation input, and what route did
the backend use?"

## Design

Add a mobile-core summary type:

- `CaptureGenerationReceipt`
- `CaptureGenerationReceiptBuilder`

The builder consumes:

- optional `ObjectCapture` from `ForgeFlowState.capture`
- optional `MythSession` from the ready flow
- current `CaptureGenerationReadiness`

It derives a safe, read-only receipt with:

- status: waiting, ready, or needs attention
- title and detail
- rows for capture id, mode, media role counts, generated asset provider, input
  mode, source image/asset counts, and selected image count
- privacy notes confirming no raw media, local capture URIs, local paths,
  provider secrets, or payment links are exposed

`ForgeRootView` will build the receipt and render it between `CaptureFormView`
and `ArtifactSummaryView`, so it appears immediately after the capture/form
workflow and before the 3D preview.

## Receipt Rules

- Missing capture: `waiting`, with the current readiness title/detail.
- Capture present and session missing: `waiting`, showing capture id, mode, and
  media counts while session creation is still in progress.
- Capture and session present with provenance: `ready`, showing provider,
  `input_mode`, route, source counts, and selected images.
- Capture and session present but provenance missing: `needsAttention`, because
  the app cannot prove which capture inputs were used for 3D generation.
- Any capture or generation text containing secrets, absolute paths,
  `local-capture://`, raw/private context markers, or checkout/payment links is
  redacted.

## UI

Add `CaptureGenerationReceiptView` in the SwiftUI app source.

The view follows the existing compact panel style:

- title row with badge (`WAIT`, `READY`, `CHECK`)
- title/detail text
- rows as small secondary text
- privacy notes at the bottom

It should not include raw filenames beyond safe counts and should not expose
media URIs. It does not run commands or mutate provider/deploy config.

## Tests

Swift contract tests first:

- waits before capture and carries readiness detail
- shows uploaded capture receipt while session is still being created
- shows ready capture-to-generation receipt for guided scan provenance
- marks missing provenance as needs attention
- redacts unsafe capture/provenance text

Swift project checks and backend iOS showcase acceptance will require the new
builder, view, root wiring, and contract tests.

## Visual Regression

Add a 390x844 static visual showing:

- capture generation receipt card in `READY` state
- capture id, guided scan mode, image count, provider, route, and selected image
  count
- no raw `local-capture://` URI, local path, or provider key text

## Non-Goals

- No backend endpoint or payload shape changes
- No new provider adapters
- No live Meshy/OpenAI/Treatstock calls
- No Unity changes
- No writes to `.env`, deploy xcconfig, signing, keychain, Xcode, or global
  system state
