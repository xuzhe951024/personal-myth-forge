# P0.383 Final Local Refresh Step Next Actions Design

## Context

`final-local-report-refresh` is the local operator report that regenerates and
summarizes the evidence chain used by final launch. Each step already promotes
the most useful child `next_action` into flattened fields such as `command`,
`detail`, and `validation_command`.

The flattened fields are useful for plain-text operator actions, but they lose
the structured shape that newer reports now expose. In the current ignored
local report, `steps_by_id.*.next_action` is `null` even when the step command
and validation command clearly came from a child report next action.

## Decision

Add optional `next_action` to non-ready refresh step rows.

- A step with status `blocked`, `partial`, or `failed` gets a `next_action`.
- The action mirrors the step `id`, `label`, `status`, `classification`,
  `command`, `detail`, `output`, and `step_id`.
- `validation_command` is included when present.
- Ready steps omit `next_action`.
- Existing top-level `first_blocker`, top-level `next_action`,
  `showcase_next_action`, and `operator_actions` behavior remains unchanged.

## Data Flow

```text
child report first_blocker / next_action
  -> final-local-report-refresh step command/detail/validation
  -> final-local-report-refresh steps[].next_action
  -> final-local-report-refresh steps_by_id.*.next_action
```

## Safety

This phase only adds structure to a sanitized local report. It does not run
live providers beyond the existing local refresh behavior, does not write
secrets, does not perform Xcode signing, and does not place print orders.

## Testing

Focused backend tests cover:

- child `next_action` promoted into `step.next_action`
- special final resource apply preview override mirrored into `step.next_action`
- ready steps omitting `next_action`
