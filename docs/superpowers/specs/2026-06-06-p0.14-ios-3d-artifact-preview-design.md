# P0.14 iOS 3D Artifact Preview Design

## Problem

The mobile app can now capture media, call the backend, and receive a generated
3D asset reference, but the artifact is still rendered as text metadata. The
final demo needs a visible generated object in a mobile scene. P0.14 adds the
first iOS 3D artifact preview shell so the app has a real 3D surface wired to
`MythSession.generatedAsset`.

The local environment still cannot run simulator or device builds because only
Command Line Tools are selected. P0.14 therefore stays within the SwiftPM app
compile gate and static visual evidence. It does not claim real device rendering
or remote GLB/USDZ download.

## Design

Add a core `ArtifactPreviewState` model that converts a `MythSession` into
display-ready preview state:

- title from the myth seed
- provider and format labels from `GeneratedAsset`
- URI for diagnostics and handoff
- whether the format is a SceneKit-loadable local scene candidate
- status text that distinguishes local scene-ready assets, remote assets, and
  unsupported or missing formats

Add a SwiftUI `Artifact3DPreviewView` that imports SceneKit and displays a
`SceneView` with a generated preview scene. The scene uses a simple pedestal and
myth-object proxy geometry so the UI has a real 3D surface while the app still
waits for a later download/import step to load provider GLB/USDZ files directly.
The panel also shows the asset URI, provider, format, and preview status.

`ArtifactSummaryView` becomes the owner of this 3D preview: it keeps the myth
title and resonance text, then embeds `Artifact3DPreviewView(session:)` instead
of only showing `GeneratedAsset.uri` as caption text.

## Supported Asset Semantics

The state model treats `usdz`, `usd`, `scn`, and `dae` local file URLs as
SceneKit-loadable candidates. It treats `glb` and `gltf` as generated 3D assets
that still need a later iOS import/conversion path. This is deliberate: Meshy
can produce GLB, but this iteration should not pretend the app has already
implemented robust GLB runtime import.

Remote `http` and `https` URIs are classified as remote assets. The app displays
the generated object proxy scene and status text until a later iteration adds
authenticated download/cache/import behavior.

## Verification

P0.14 verification adds:

```bash
swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests
swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks
swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck
```

The existing backend lint/test gates still run. Visual evidence is a static
mobile storyboard showing the 3D preview panel, SceneKit gate, generated asset
metadata, and a no-simulator notice.

## Non-Goals

- Do not run `xcode-select`, simulator, device, archive, signing, or deployment
  commands.
- Do not download provider assets to the phone.
- Do not implement real GLB runtime import or conversion.
- Do not replace the eventual Unity or richer village scene decision.
- Do not move provider API keys into the mobile app.
