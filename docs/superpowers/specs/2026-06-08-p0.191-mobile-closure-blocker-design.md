# P0.191 Mobile Closure Blocker Design

## Problem

P0.190 added a top-level `first_blocker` to the backend
`final_launch_closure_packet_report`, but the iOS model still ignores that field.
The mobile Final Launch panel can show closure packet section fallbacks, yet it
does not directly surface the canonical packet blocker that operators now use
for the next required action.

## Scope

P0.191 carries the closure packet blocker through the mobile contract:

- Decode `final_launch_closure_packet.first_blocker` into Swift.
- Display the decoded blocker near the top of `closurePacketRows`.
- Preserve the existing section fallback, configured evidence bundle row,
  operator action row, consent row, and safety row.
- Sanitize blocker command/detail/classification before rendering.
- Add source gates so the mobile showcase acceptance requires the field and row
  builder.

## Non-Goals

- Do not change backend closure packet generation.
- Do not run provider calls, Xcode signing, keychain writes, or global commands.
- Do not mark final launch ready without real resource, provider, and device
  evidence.
- Do not redesign the Final Launch panel layout.

## Data Shape

The Swift model uses the existing JSON decoder with
`.convertFromSnakeCase`, so backend JSON:

```json
{
  "first_blocker": {
    "id": "resource_inputs",
    "label": "Resource inputs",
    "status": "blocked",
    "classification": "missing_required_value",
    "command": "make final-resources-preflight",
    "detail": "Backend-only secret for live Meshy 3D generation.",
    "section_id": "resource_inputs",
    "action_id": "provide_MESHY_API_KEY"
  }
}
```

maps to `FinalLaunchClosurePacketBlocker? firstBlocker`.

The mobile row format is:

```text
First blocker: resource_inputs blocked missing_required_value | provide_MESHY_API_KEY | make final-resources-preflight | Backend-only secret for live Meshy 3D generation.
```

## Acceptance Criteria

- `FinalDemoLaunchReport` decoding preserves closure packet `firstBlocker`.
- `FinalLaunchMobileSummaryBuilder` includes a first blocker row before closure
  section fallback rows when the blocker exists.
- Unsafe blocker command/detail text is redacted by the existing sanitizer.
- Existing closure packet rows continue to show configured evidence bundle,
  cost consent, and safety state.
- Backend mobile final launch readiness acceptance requires the Swift model
  field and mobile row builder tokens.
- iOS project checks and Swift contract tests pass.

## Self-Review

- Placeholder scan: no TBD or deferred requirements remain.
- Scope check: one bounded mobile contract and summary iteration; no backend
  behavior or provider state changes.
- Ambiguity check: `firstBlocker` is optional; missing blockers keep current
  fallback behavior.
