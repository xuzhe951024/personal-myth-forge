# P0.403 iOS Runbook Operator Actions Design

## Context

The final showcase first blocker is still the iOS deployable path. The
operator-facing final showcase and final demo reports now expose concrete
commands, but the source `ios-deploy-runbook-local` report still includes older
fallback actions:

```text
run make final-resource-apply-preview before applying resources
unblock final_resources_preflight: make final-resources-preflight
unblock preview_final_resource_apply: make final-resource-apply-preview
unblock apply_final_resources: make final-apply-resources
unblock mobile_deploy_preflight: make mobile-deploy-preflight
```

Those strings are weaker than the concrete child actions already available from
resource preflight, apply preview, final acceptance, and deploy config slots.
The biggest risk is `unblock apply_final_resources: make final-apply-resources`,
which can imply that applying resources is the next action even when the preview
and required values are still blocked.

## Goal

Make `ios-deploy-runbook-local` operator actions show concrete, safe handoff
commands for the iPhone deployment path without exposing stale `unblock ...`
fallbacks or blocked apply commands.

## Scope

- Change only `myth_forge_api.ios_deploy_runbook` and its tests.
- Preserve `command_sequence` as the full ordered runbook, including
  `make final-apply-resources`.
- Preserve concrete child actions from final resources, apply preview, final
  acceptance, 3D/NPC evaluation, and deploy config slots.
- Preserve the manual Xcode build gate action.
- Do not write ignored resource files, deploy configs, provider keys, Xcode
  state, or print order state.

## Design

1. Change the `final_resource_apply_preview` input slot action from a sentence
   to the canonical command:

   ```text
   make final-resource-apply-preview
   ```

2. Stop adding synthetic `unblock <step>: <command>` operator actions for
   blocked command-sequence rows. These rows are still present in
   `command_sequence`, but top-level operator actions should come from the
   concrete source reports and input slots that explain what is missing.

3. Keep the manual Xcode build gate action because it represents a real external
   Mac/Xcode step that cannot be completed by local report generation.

## Tests

Update `services/backend/tests/test_ios_deploy_runbook.py`:

- The missing-inputs test should expect `make final-resource-apply-preview` and
  should reject the old sentence-shaped preview action.
- Add a regression test proving `operator_actions` do not include:
  - `unblock final_resources_preflight: make final-resources-preflight`
  - `unblock preview_final_resource_apply: make final-resource-apply-preview`
  - `unblock apply_final_resources: make final-apply-resources`
  - `unblock mobile_deploy_preflight: make mobile-deploy-preflight`
  - any operator action containing `make final-apply-resources` while preview is
    blocked

Focused verification:

```bash
cd services/backend && uv run pytest tests/test_ios_deploy_runbook.py -q
```

Report verification:

```bash
make final-local-report-refresh
jq '.steps_by_id.ios_deploy_runbook_local.report_operator_actions' \
  services/backend/.local/final-local-report-refresh.json
```

Expected: the report still blocks on missing external values, but the iOS
runbook action list no longer exposes stale `unblock ...` fallbacks.
