# P0.519 External Ledger Concrete Next Action Design

## Context

After P0.518, the main final-showcase, final-local-report-refresh, final-demo-launch, final-handoff-index, and iOS rehearsal reports expose the current iOS device blocker as a complete copy-pasteable command chain:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

`final-external-action-ledger` still exposes that same action differently at the top level:

- `first_blocker.command = DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto`
- `next_action.command = DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto`
- `validation_command = make mobile-deploy-preflight`

Its `operator_actions[0]` is already the complete command chain. The top-level fields should match that handoff quality so the external action ledger can be used directly during final launch work.

## Goal

Promote the final external action ledger top-level device blocker and next action to a validation-aware command while keeping `validation_command` available as structured metadata.

## Non-Goals

- Do not mark the ledger ready without real device evidence.
- Do not write Apple signing data, Keychain entries, provider keys, or ignored local resource files.
- Do not call Xcode, Meshy, OpenAI, Treatstock, or any paid/live provider.
- Do not reorder provider or print handoff actions.

## Behavior

When `final-external-action-ledger` promotes the first device action from the iOS deploy runbook and that action has:

```text
command = DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto
validation_command = make mobile-deploy-preflight
```

then the report should expose:

```text
first_blocker.command = DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
next_action.command = DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
validation_command = make mobile-deploy-preflight
```

If a command already contains the validation suffix, it must not be duplicated.

## Testing

Add a focused regression test in `services/backend/tests/test_final_external_action_ledger.py` that asserts the top-level `first_blocker.command` and `next_action.command` are the same validation-aware command already shown as the first operator action.

Verification commands:

```bash
cd services/backend && uv run pytest tests/test_final_external_action_ledger.py::test_external_action_ledger_top_level_device_blocker_promotes_validation_aware_command -q
cd services/backend && uv run pytest tests/test_final_external_action_ledger.py -q
make backend-lint
make backend-test
make final-external-action-ledger
```

The refreshed `.local/final-external-action-ledger.json` should remain `blocked`, but top-level `first_blocker.command` and `next_action.command` should use the complete command chain.
