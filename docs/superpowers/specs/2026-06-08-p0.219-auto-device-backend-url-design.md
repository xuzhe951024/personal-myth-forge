# P0.219 Auto Device Backend URL Design

## Context

The C-path final showcase is still blocked on real iOS device evidence. The existing handoff already has `make backend-device-demo`, `mobile-write-deploy-config`, and `mobile-deploy-preflight`, but the operator must manually discover the Mac LAN IP and type a full `PMF_BACKEND_BASE_URL`. That manual step is easy to get wrong and is one of the last pieces before an iPhone can prove `/health`.

## Goal

Allow the iOS deploy config writer to resolve a LAN backend URL when `PMF_BACKEND_BASE_URL=auto`, and expose a Make target for the common physical-device demo path.

## Non-Goals

- Do not start the backend server.
- Do not run iOS deploy preflight, Xcode, signing, keychain, Apple license, Meshy, OpenAI, or Treatstock.
- Do not write provider secrets, backend `.env`, or global machine state.
- Do not mark final showcase readiness as ready without actual device evidence.

## Behavior

`apps/mobile/ios/scripts/write_deploy_local_config.sh` accepts:

```bash
PMF_BACKEND_BASE_URL=auto make mobile-write-deploy-config
```

or:

```bash
make mobile-write-deploy-config-auto
```

When `auto` is used, the script resolves a non-loopback IPv4 host and writes:

```text
PMF_BACKEND_BASE_URL = http://<host>:8080
```

The port defaults to `8080` and can be overridden with `PMF_BACKEND_PORT`. Tests can force deterministic resolution with `PMF_BACKEND_AUTO_HOST`, which is treated as a host/IP only, not a secret. After resolution, the existing single-line, final launch mode, untracked config, and loopback validation still run.

## Safety

The script only writes `apps/mobile/ios/Config/Deployment.local.xcconfig`, which is ignored and guarded against git tracking. Automatic host discovery does not call providers, start servers, change Xcode settings, touch keychain, or accept Apple licenses. If auto resolution cannot find a non-loopback host, the script exits `2` before writing config.
