# P0.72 Mobile Final Launch Status Design

## Context

The iPhone app already fetches `/v1/final-demo-launch` and uses it for a compact
Device Preflight row. That row is useful for readiness, but it hides the launch
phases, operator checklist, command list, and final resources preflight detail
that an operator needs during a physical device demo.

## Decision

Build a read-only Mobile Final Launch Status panel in the iOS app.

Recommended approach:

- Add a small core summary layer that maps `FinalDemoLaunchReport` into
  display-safe rows and action snippets.
- Add a SwiftUI `FinalLaunchStatusView` under Device Preflight.
- Keep `/v1/final-demo-launch` as the source of truth. Do not add a new backend
  endpoint in this iteration.

Rejected alternatives:

- Expand the existing Device Preflight rows. This keeps the top panel too dense
  and makes commands hard to scan.
- Add a backend mobile-only summary endpoint. The backend report is already
  sanitized and stable enough; another endpoint would add surface area without
  changing demo capability.

## User Experience

The panel shows:

- Overall launch state and mode.
- Phase counts from the backend report.
- The first few launch phases with status, command, and note summary.
- Operator actions from final resources preflight when present.
- A compact command checklist for local handoff.
- Error and loading states when the backend report is unavailable.

The view is designed for iPhone width. It is not a marketing page; it is an
operator surface for checking what to run next.

## Data Flow

`ForgeRootView` already loads:

```text
PersonalMythForgeAPIClient.getFinalDemoLaunch(mode: "local")
```

P0.72 adds:

```text
FinalDemoLaunchReport
  -> FinalLaunchMobileSummaryBuilder.build(report:error:)
  -> FinalLaunchStatusView
```

The summary builder has no network access and no file access.

## Safety

The iOS app does not read `.env`, `final-resources.env`, or provider keys. It
only renders the backend report it already fetches. Every display string from
the report passes through the same mobile redaction rules used by Device
Preflight.

The view must not:

- show provider secrets;
- show local absolute paths;
- show checkout/payment URLs;
- run commands;
- mutate Xcode, signing, keychain, or global developer settings.

## Testing

Core contract tests cover:

- missing report becomes a waiting summary;
- partial report shows phase counts, commands, and final resource actions;
- unsafe strings are redacted;
- ready report produces a ready summary.

Project checks cover:

- `FinalLaunchMobileSummary` core source exists;
- `FinalLaunchStatusView.swift` exists and is included in Xcode sources;
- `ForgeRootView` wires the new view from `finalDemoLaunch`;
- source acceptance requires the new mobile final launch status contract.

Visual evidence is a static 390x844 iPhone screenshot showing the new panel with
partial final launch status and missing final resources.
