# P0.327 Final Handoff Index Next Action Design

## Context

The final iPhone showcase objective depends on a Mac-side operator sequence: refresh local rehearsal evidence, review configured resources, start a LAN backend, run mobile deploy preflight, and only then use live provider or Xcode actions with explicit consent. `final_handoff_index` is the top-level report that joins those lanes, but it currently exposes only `first_blocker`. Consumers must infer the next action from `operator_actions` or lane details.

Current `.local` evidence shows:

- `final_handoff_index` is blocked.
- `first_blocker.id == "local_rehearsal"`.
- `next_action` is absent.
- Downstream final launch and device evidence reports still need concrete handoff commands to avoid vague "refresh index" loops.

## Goal

Expose a top-level `next_action` on `final_handoff_index_report` so Mac-side and iPhone-visible handoff flows can show the concrete next step from the unified handoff index without parsing nested lanes.

## Non-Goals

- Do not run Xcode, signing, Keychain, real iPhone deploy, live providers, Treatstock, or paid provider actions.
- Do not write API keys, `.env`, or iOS deployment config.
- Do not change lane ordering, lane status semantics, operator sequence semantics, or final showcase status.
- Do not mark the global goal complete.

## Backend Behavior

`build_final_handoff_index_report()` will include:

- `first_blocker`: existing field, unchanged.
- `next_action`: `None` when `first_blocker` is `None`; otherwise a copy of `first_blocker` with `source: "first_blocker"` and `validation_command`.

`validation_command` will be a Make command when the blocker command starts with `make `. For local rehearsal blockers this is `make final-rehearsal-local`. For configured preflight it is `make final-configured-preflight`. For device deploy it is `make mobile-deploy-preflight`.

The report remains sanitized by the existing sanitizer before it is returned or written.

## Mobile Behavior

Swift decoding will accept optional `firstBlocker` and `nextAction` on the final handoff index model if that model exists in the final launch/device payload path. If the mobile path only receives summarized handoff rows, no new screen is required; backend rows will still expose the action through existing report-derived handoff text.

## Testing

Backend tests will first fail because `next_action` is absent. Tests will assert:

- Missing local reports produce `next_action.id == "local_rehearsal"`.
- `next_action.source == "first_blocker"`.
- `next_action.validation_command == "make final-rehearsal-local"`.
- Ready handoff index reports keep `next_action is None`.
- Saved local blocker detail still produces the concrete `provide DEVELOPMENT_TEAM...` operator action while the index-level validation command remains `make final-rehearsal-local`.

Swift tests will be added only if current mobile models decode the full final handoff index payload. If they do not, backend contract tests and existing mobile summary rows are sufficient for this slice.

## Safety

This phase only changes read-only report JSON shape and tests. It does not run shell writers, mutate ignored env/config files, call external providers, or touch global machine state.
