# P0.513 Final Local Report Refresh Backend URL Priority Design

## Context

`final-showcase-readiness` and `final-demo-launch-local` now guide the local iPhone showcase through the correct device chain:

1. write the ignored iOS deploy config with `DEVELOPMENT_TEAM`
2. start `backend-device-demo`
3. set `PMF_BACKEND_BASE_URL` to an iPhone-reachable LAN URL

Fresh `final-local-report-refresh` evidence still lists provider handoff immediately after `backend-device-demo`, leaving the backend URL action behind provider and print actions. This makes the top-level local refresh report less actionable for the final iPhone showcase.

## Decision

Update `final_local_report_refresh` operator action prioritization so that when a backend-device-demo action is present, a LAN backend URL action is promoted immediately after it and before provider or print handoffs.

Desired top ordering:

1. concrete deploy writer or next action
2. backend-device-demo handoff
3. iPhone-reachable backend URL handoff
4. provider handoff
5. guarded print quote handoff
6. remaining blocked actions

If no backend-device-demo action exists, preserve the current ordering behavior.

## Scope

In scope:

- Add a focused final local report refresh regression test.
- Update `_prioritize_final_local_report_operator_actions`.
- Refresh local evidence after merge.

Out of scope:

- Real device checks.
- Xcode signing.
- Meshy/OpenAI/Treatstock calls or cost-incurring provider actions.
- Changing external ledger or closure packet ordering in this phase.

## Validation

- The new focused test must fail before implementation.
- The focused test and `tests/test_final_local_report_refresh.py` must pass after implementation.
- `make backend-lint` and `make backend-test` must pass before merge.
- `make final-local-report-refresh` must show deploy writer, backend-device-demo, and backend URL as the first three operator actions.
