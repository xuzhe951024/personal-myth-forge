# P0.384 Refresh Step Next Action Command Design

## Context

P0.383 added structured `next_action` objects to non-ready
`final-local-report-refresh` step rows. That made the next local handoff easier
to inspect, but it also exposed a shape problem: some child reports encode an
operator command plus a human detail in a single string separated by ` | `.

For example, the iOS device launch rehearsal step can surface:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

That full text is useful in plain operator action lists, but the structured
`next_action.command` field should stay copy-pasteable. The missing-value
explanation belongs in `next_action.detail`.

## Decision

Normalize structured refresh-step next action commands by splitting off a
trailing ` | ...` detail suffix before assigning `next_action.command`.

- Apply this only while building structured `next_action` objects.
- Keep the step `detail` unchanged, so the user still sees the missing-value
  explanation.
- Preserve `validation_command` exactly when a step provides it.
- Leave top-level status, first blocker selection, showcase next action
  selection, and operator action ordering unchanged.

## Safety

This phase is report-shape only. It does not invoke live providers, write
secrets, change Xcode signing state, place print orders, or mutate global
machine configuration.

## Testing

Focused backend tests will cover a blocked step whose command contains a
detail suffix. The expected structured `next_action.command` is the executable
command without ` | ...`, while `next_action.detail` preserves the full detail
text and `validation_command` remains available.
