# P0.314 iOS Device Evidence Next Action Design

## Goal

Make the iOS device evidence bundle expose the current blocking slot and next action as structured report fields that the iOS final launch summary can decode and display.

## Context

The final showcase remains blocked on external iPhone deployment inputs: Apple team configuration, an iPhone-reachable backend URL, and local Xcode/license evidence. The backend already records `evidence_slots` and `operator_actions`, but the iOS device evidence bundle does not expose top-level `first_blocker` or `next_action`. Operators and the iOS summary must infer the immediate blocker from arrays.

## Scope

This iteration is local-only. It does not run Xcode, write signing config, touch Keychain, call providers, place orders, write API keys, or run paid services.

## Behavior

When the bundle is blocked, select the first required evidence slot whose status is not `ready` as `first_blocker`. The order is the existing evidence slot order:

1. `backend_device_server`
2. `mobile_deploy_preflight`
3. `xcode_build_gate`
4. `ios_device_launch_rehearsal`

`next_action` is a sanitized copy of that blocker with `source: "first_blocker"`. It keeps the blocker command and detail so the iOS final launch summary can show the immediate local action without parsing `operator_actions`.

When the bundle is ready, both `first_blocker` and `next_action` are `null`.

## iOS Contract

Swift decoding adds `IOSDeviceEvidenceBundleFirstBlocker` and `IOSDeviceEvidenceBundleNextAction` models. `IOSDeviceEvidenceBundleReport` includes optional `firstBlocker` and `nextAction`. `FinalLaunchMobileSummary` displays compact rows for both fields when present, with the same redaction path used for the rest of the report.

## Testing

Backend tests prove blocked and ready report shapes. Swift contract tests prove final launch payload decoding and summary text include the new fields.
