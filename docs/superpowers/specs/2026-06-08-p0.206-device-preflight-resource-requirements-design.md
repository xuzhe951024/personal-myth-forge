# P0.206 Device Preflight Resource Requirements Design

## Purpose

The backend final launch report exposes `final_resource_requirements.first_blocker`, and the mobile final launch summary already renders it. `DevicePreflightSummaryBuilder` currently starts resource guidance at the fill guide and apply preview, so the iPhone preflight skips the earliest resource handoff question: which required value should be provided first.

## Scope

- Add a `final_resource_requirements` device preflight item labeled `Resource Requirements`.
- Show waiting when the final launch report or resource requirements report is missing.
- Show blocked when `FinalResourceRequirementsReport.firstBlocker` is present, including the command, detail, destination, domain, and validation command.
- Show ready when the resource requirements report status is `ready`.
- Include the item in the required ready set before fill guide and apply preview.
- Reuse the existing device preflight sanitizer; do not read secrets, write files, execute shell commands, or call providers.
- Add Swift contract tests, ProjectChecks gates, and backend mobile readiness source requirements.

## Acceptance

- `DevicePreflightSummaryBuilder` returns a `final_resource_requirements` item.
- Blocked reports surface `provide MESHY_API_KEY in final-resources.env`, destination, and `make final-resources-preflight`.
- Ready reports mark the item ready and include total/ready/missing/required/secret counts.
- Encoded preflight output redacts provider keys, bearer tokens, local paths, local capture URLs, checkout text, and `api_key` snippets from requirement first blocker fields.
- Mobile final launch readiness requires the item, label, source, contract tests, and required-ready set entry.
