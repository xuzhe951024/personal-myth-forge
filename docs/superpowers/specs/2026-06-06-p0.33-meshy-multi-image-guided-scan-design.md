# P0.33 Meshy Multi-Image Guided Scan Design

## Context

Guided scan intake already accepts 2-12 iPhone images, prepares HEIC/HEIF as
provider-safe JPEGs, and passes all prepared reference images into
`ThreeDGenerationRequest.source_images`. The remaining backend gap is inside the
Meshy adapter: it currently picks the first supported source image and calls
single `image-to-3d`, which discards most of the guided scan signal.

Meshy's official Multi-Image to 3D API now accepts `image_urls` with 1-4
public URLs or data URIs for the same object from different angles, and can
return `glb` and `usdz` model URLs. P0.33 should use that endpoint when the
capture contains multiple provider-supported reference images.

Source: https://docs.meshy.ai/en/api/multi-image-to-3d

## Goal

Route Meshy-backed guided scans with at least two supported prepared images
through Multi-Image to 3D, cap provider input to four images, preserve existing
single-image/text fallbacks, and expose local no-key evidence that multi-image
scan input survived the backend pipeline.

## Non-Goals

- No live Meshy calls in automated tests.
- No API key creation or storage.
- No global environment mutation.
- No iOS device installation or Apple SDK license changes.
- No client-side photogrammetry reconstruction.
- No public media hosting; provider input continues to use backend-only data
  URIs created from local capture storage.
- No print repair or fulfillment changes.

## Recommended Approach

Update `MeshyThreeDProvider.generate_game_asset` routing:

1. collect supported source images whose prepared content type is JPEG or PNG
2. if count is at least two, call `/openapi/v1/multi-image-to-3d`
3. include up to four `image_urls`, preserving capture order
4. request `target_formats=["glb", "usdz"]`, `should_texture=true`, and
   `enable_pbr=true`
5. poll the same endpoint path with the returned task id
6. map `model_urls.glb` and optional `model_urls.usdz` through the existing
   generated asset variant contract

Fallback behavior stays stable:

- exactly one supported image: existing `/openapi/v1/image-to-3d`
- no supported image: existing text-to-3D preview/refine path
- unsupported images before supported images are ignored

## Provider Input Observability

The local stub should make scan source usage visible without leaking raw media:

- keep `provider input summary: source_images=N; source_assets=M`
- add `selected_source_images=K` when provider-specific selection is known
  only for Meshy tests, not required in public API yet

The generated asset report must still not contain data URIs, raw bytes, file
paths, or local capture media contents.

## Components

`services/backend/src/myth_forge_api/providers/three_d.py`

- Add `MESHY_MULTI_IMAGE_TO_3D_MAX_IMAGES = 4`.
- Add `_multi_image_to_3d_path = "/openapi/v1/multi-image-to-3d"`.
- Add `_meshy_supported_source_images(...)` returning all supported JPEG/PNG
  prepared images.
- Route 2+ supported images to `_generate_multi_image_asset(...)`.
- Keep `_generate_image_asset(...)` unchanged for one supported image.

`services/backend/tests/test_meshy_provider.py`

- Add tests proving multi-image requests use `/multi-image-to-3d`.
- Assert exactly four images are sent when more than four supported images are
  available.
- Assert unsupported images are ignored and do not force text fallback.
- Assert existing single-image and text-to-3D tests still pass.

`services/backend/tests/test_capture_api.py`

- Add or extend a from-capture test to prove guided scan uploads forward
  multiple prepared source images to the injected 3D provider without leaking
  raw media in the response.

## Error Handling

- A failed multi-image Meshy task raises `MeshyProviderError` through the same
  sanitized API path as text/image generation.
- Missing `model_urls.glb` remains a provider error.
- If only unsupported image content types are present, provider falls back to
  text-to-3D. In normal capture flow, HEIC/HEIF is prepared to JPEG before
  reaching the provider, so this fallback mainly protects malformed direct
  provider usage and future capture modes.

## Testing

Backend tests will cover:

- multi-image Meshy task creation and polling with two images
- cap to four image URLs when five supported images are supplied
- single supported image still uses image-to-3D
- no source images still uses text-to-3D preview/refine
- capture API forwards multiple guided scan source images into the provider
- no raw data URI, local media bytes, or local path appears in API responses

Regression will run:

## Acceptance Criteria

- Meshy gets `/openapi/v1/multi-image-to-3d` when a guided scan has at least two
  prepared JPEG/PNG reference images.
- Meshy sends no more than four image URLs to the multi-image endpoint.
- One supported image still uses Image-to-3D.
- No supported images still use Text-to-3D.
- Guided scan from-capture API evidence proves multiple images reach the 3D
  provider without exposing raw media or local paths.
- Existing provider readiness, final acceptance, NPC agent, mobile compile, and
  deploy gate behavior remain intact.
