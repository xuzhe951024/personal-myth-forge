# P0.334 iOS Deploy Writer Base Bundle Design

## Context

The final showcase is still blocked on physical iPhone deployment evidence. The
current preflight correctly reads `PRODUCT_BUNDLE_IDENTIFIER` from the committed
`apps/mobile/ios/Config/Deployment.xcconfig`, but the helper writer
`apps/mobile/ios/scripts/write_deploy_local_config.sh` still requires the
operator to provide the bundle id explicitly.

That makes the final handoff noisier than the actual blocker. The project
already has a default bundle id, while the remaining machine-specific values are
`DEVELOPMENT_TEAM` and an iPhone-reachable `PMF_BACKEND_BASE_URL`.

## Decision

Teach `write_deploy_local_config.sh` to use the committed deployment defaults
when writing the ignored local config:

- read `PRODUCT_BUNDLE_IDENTIFIER` from `Deployment.xcconfig` when the operator
  does not provide it by flag or environment;
- keep `DEVELOPMENT_TEAM` explicit because the committed default is intentionally
  empty;
- keep `PMF_BACKEND_BASE_URL` explicit or `auto` because the committed default is
  loopback and is not valid for a physical iPhone;
- keep command-line flags and environment variables as higher precedence than
  committed defaults.

## Safety

This does not run Xcode, sign an app, mutate global state, touch Keychain, call
providers, or write secrets. The writer still writes only the ignored
`apps/mobile/ios/Config/Deployment.local.xcconfig`, still refuses tracked local
config files, and still rejects loopback backend URLs.

## Acceptance

- `DEVELOPMENT_TEAM=ABCDE12345 PMF_BACKEND_BASE_URL=http://192.168.1.10:8080 make mobile-write-deploy-config`
  can write a local config without also passing `PRODUCT_BUNDLE_IDENTIFIER`.
- Existing explicit env/flag bundle id overrides keep working.
- Existing preflight and evidence tests continue to pass.
