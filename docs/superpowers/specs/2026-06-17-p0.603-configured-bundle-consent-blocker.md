# P0.603 Configured Bundle Consent Blocker

## Context

After the iPhone/backend preflight is ready, `final-configured-evidence-plan` moves to
live provider consent in the intended order: configured 3D evaluation, configured NPC
evaluation, then configured final acceptance. `configured-live-evidence-bundle` already
uses that order in `operator_actions`, but its `current_blocker` can still point at the
saved live evidence report's `final_acceptance_configured` blocker first.

## Decision

When the configured plan status is `consent_required`, the configured live evidence
bundle should use the configured plan first blocker as its current/first blocker before
falling back to stale or missing live provider evidence.

## Acceptance

- The bundle current/first blocker is `three_d_evaluation_configured` when configured
  provider evidence has not been run and the plan requires live provider consent.
- The blocker command includes `PMF_ALLOW_LIVE_PROVIDER_CALLS=1` and keeps
  `make final-configured-evidence-plan` as its validation command.
- The first operator action includes the same command plus rerun validation.
- Existing missing-resource, configured-plan-blocked, and ready evidence behavior
  remains unchanged.
- No live provider calls, API key writes, print orders, or Xcode/global actions are
  introduced.
