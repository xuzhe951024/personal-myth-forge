# P0.326 Final External Action Ledger Next Action Design

## Context

The final showcase goal is blocked by external operator actions: missing provider keys, iOS deploy values, Xcode/license state, live provider consent, and Treatstock quote evidence. Most final reports already expose a top-level `first_blocker` and `next_action`, but `final_external_action_ledger` still requires readers to infer the next step from `operator_actions` or nested action groups.

Current local evidence shows:

- `final-showcase-readiness` is blocked by iOS deploy evidence.
- `final-local-report-refresh` includes `final_external_action_ledger` as blocked.
- The ledger has five action groups and a sanitized operator action list, but no top-level machine-readable next action.

## Goal

Expose a sanitized, mobile-visible top-level `first_blocker` and `next_action` on `final_external_action_ledger` so the final handoff path has a direct, actionable next step without reading nested group internals.

## Non-Goals

- Do not run live Meshy, OpenAI, Treatstock, Xcode, signing, Keychain, or global machine actions.
- Do not write provider secrets or local deploy values.
- Do not change the ordering or semantics of existing action groups beyond deriving the top-level blocker.
- Do not mark the overall project goal complete.

## Backend Behavior

`build_final_external_action_ledger_report()` will derive `first_blocker` from the existing ordered action groups after actions are built and before sanitization.

Selection priority:

1. `missing` or `blocked` actions that require user input.
2. Any `missing` or `blocked` action.
3. `manual` actions that require user confirmation.
4. `live` actions that require cost consent.
5. `manual` actions.
6. `live` actions.
7. No blocker when all actions are ready.

The derived blocker will include:

- `id`
- `label`
- `status`
- `classification`
- `command`
- `detail`
- `group_id`
- `group_label`
- optional `destination`
- optional `validation_command`

`validation_command` uses the action command when the action command is a `make ...` validation target. For missing resource inputs, it uses the existing action command (`make final-resources-preflight` or `make mobile-deploy-preflight`) so the operator gets an immediately rerunnable check.

`next_action` will copy `first_blocker`, add `source: "first_blocker"`, and keep the same command and validation command. It is omitted or `null` when the ledger is ready.

## Mobile Behavior

Swift decoding will accept optional `firstBlocker` and `nextAction` on `FinalExternalActionLedgerReport`.

The Final Launch `External Actions` summary will add:

- `Next action: <id> <status> | <command> | <detail>`
- `First blocker: <id> <status> | <group> | <command>`

Rows must continue using existing sanitization so secrets, local paths, raw private content, and payment links remain redacted.

## Testing

Backend tests will first fail because `first_blocker` and `next_action` are absent. They will then assert:

- missing local resources produce `provide_MESHY_API_KEY` as `first_blocker`;
- `next_action.source == "first_blocker"`;
- iOS deploy resource actions keep `make mobile-deploy-preflight` as validation;
- ready resources still route to manual or live actions without leaking secrets.

Swift contract tests will first fail because `FinalExternalActionLedgerReport` lacks the new fields. They will then assert:

- final demo payload decodes `firstBlocker` and `nextAction`;
- mobile summary rows include `Next action` and `First blocker`;
- unsafe text remains redacted.

## Safety

This phase is read-only with respect to real resources. The implementation does not invoke provider APIs, mutate global machine state, write ignored env/config files, or perform Xcode signing. All generated reports remain sanitized through the existing report sanitizer and Swift summary sanitizer.
