# P0.85 NPC Action Gate Design

Date: 2026-06-06

## Problem

The iPhone NPC panel shows agent mode and tick summaries, but the primary NPC actions still only check whether a session exists and whether another tick/autonomy call is running. When NPC provider setup is missing, the app can still present `Advance Village` and `Run Autonomy` as available. That is inconsistent with the final demo goal: NPC actions should be visibly driven by the AI Agent/provider readiness contract.

## Goal

Add a mobile-core NPC action gate that controls `Advance Village` and `Run Autonomy` from the same state the operator sees: myth session, NPC Agent mode, tick summary, and current loading flags.

## Non-Goals

- Do not add new backend endpoints.
- Do not change NPC tick or autonomy provider behavior.
- Do not require OpenAI for local demo mode.
- Do not remove the existing NPC Agent tick summary panel.
- Do not run live provider calls or global Xcode/signing commands.

## Options Considered

1. Keep conditions inside `NPCTickView`.
   - Simple, but the button policy stays hard to test and can drift from provider readiness.
2. Reuse `NPCAgentTickSummary` as the button gate.
   - Avoids a new type, but tick summary is about state reporting, not action eligibility.
3. Add `NPCAgentActionGate`.
   - Slightly more code, but gives a small testable policy object and matches the P0.84 Forge gate pattern.

Chosen approach: option 3.

## Model

Add `NPCAgentActionGate` and `NPCAgentActionGateBuilder` in mobile core. The builder consumes:

- `session`
- `NPCAgentModeSummary`
- `NPCAgentTickSummary`
- `isAdvancingTick`
- `isRunningAutonomy`

The gate returns:

- `canAdvanceVillage`
- `canRunAutonomy`
- `advanceTitle`
- `autonomyTitle`
- `detail`
- `disabledReason`

Priority order:

1. No session: both disabled, reason `session_required`.
2. Action in progress: both disabled, reason `npc_action_running`.
3. NPC setup missing: both disabled, reason `npc_setup_required`.
4. Tick summary needs attention: both disabled, reason `npc_recovery_required`.
5. Ready local demo or OpenAI mode: both enabled.

All detail copy is redacted with the same safety stance as the existing NPC summaries: no provider secrets, raw local paths, local capture URIs, authorization text, or payment URLs.

## UI Behavior

`ForgeRootView` builds `npcAgentActionGate` beside `npcAgentModeSummary` and `npcAgentTickSummary`, then passes it into `NPCTickView`.

`NPCTickView`:

- disables `Advance Village` with `!actionGate.canAdvanceVillage`
- disables `Run Autonomy` with `!actionGate.canRunAutonomy`
- renders `actionGate.detail` below the buttons
- keeps existing progress views and error rendering

Local demo remains available when session exists and NPC mode is `localDemo`. OpenAI mode is available when provider readiness says OpenAI is ready. Missing `OPENAI_API_KEY` blocks the action with visible next-step copy.

## Tests

Swift core contract tests cover:

- no session disables both NPC actions
- local demo session enables both actions
- missing OpenAI provider setup disables both actions and exposes `OPENAI_API_KEY`
- running autonomy disables both actions
- unsafe detail text is redacted

Source acceptance gates cover:

- `NPCAgentActionGateBuilder`
- `canRunAutonomy`
- `disabledReason`
- `NPCTickView` uses `actionGate.canRunAutonomy`, `actionGate.canAdvanceVillage`, and `actionGate.detail`
- `ForgeRootView` builds and passes `npcAgentActionGate`

## Visual Regression

Add a static 390x844 verification page showing an NPC panel in setup-needed mode:

- `NPC Agent Mode` missing `OPENAI_API_KEY`
- `Next Village Tick` shows `CHECK`
- `Advance Village` and `Run Autonomy` appear disabled
- the gate detail explains the missing setup
- no text overlap or right-edge clipping

## Safety

P0.85 is source/local only. It must not call Meshy, OpenAI, Treatstock, Sculpteo, Xcode signing, Apple license commands, or any global state mutation.
