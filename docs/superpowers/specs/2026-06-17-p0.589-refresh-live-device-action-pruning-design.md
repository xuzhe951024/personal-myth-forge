# P0.589 Final Local Refresh Live Device Action Pruning Design

## Goal

Keep `final_local_report_refresh.operator_actions` focused on the current live-provider blocker when the first action is the explicit Meshy/OpenAI cost-consent command.

## Context

After the local handoff reports are ready, the final showcase and final demo launch reports correctly surface one top-level action:

```text
PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence
```

`final-local-report-refresh` can still import a `backend-device-demo` fallback from nested launch reports even when live-provider consent is already the first and only actionable blocker. That makes the final operator checklist look like it has two parallel first steps.

## Design

Keep device handoff behavior unchanged for normal iPhone/device blockers.

Add a narrow final pruning rule in `myth_forge_api.final_local_report_refresh`:

- if the first operator action is the guarded live-provider consent command
- then remove later device handoff fallbacks such as `backend-device-demo`
- otherwise preserve existing backend device demo priority and LAN backend URL behavior

This keeps local/device recovery paths available when they are actually the blocker, while preventing live-provider acceptance from being mixed with unrelated device fallback actions.

## Scope

In scope:

- final-local-report-refresh operator action pruning
- focused RED/GREEN coverage for live consent plus backend-device-demo fallback
- full `test_final_local_report_refresh.py` regression
- roadmap entry for P0.589

Out of scope:

- live Meshy/OpenAI calls
- iOS signing, Xcode, or physical device execution
- changing final showcase readiness capability classification
- changing normal device-first operator actions

## Verification

This phase is valid when:

- RED proves live consent plus nested backend-device-demo currently emits both actions
- GREEN proves the final action list keeps only the live consent command in that scenario
- existing device blocker priority tests still pass
- refreshed `.local/final-local-report-refresh.json` no longer includes `backend-device-demo` when live consent is first
