# P0.126 Showcase Apply Preview Ledger Design

## Problem

P0.125 added `final_resource_apply_preview`, but the top-level final showcase
readiness ledger still only records `final_resources_preflight` and
`resource_handoff` evidence. That leaves the final C-path summary one step
behind the actual operator flow: the safest path is now
`requirements -> preflight -> apply preview -> apply resources`.

The user wants to provide API/key resources at the end in one integrated pass.
The final showcase readiness report should therefore expose the dry-run apply
preview as first-class evidence before any resource writer is run.

## Goals

- Add `final_resource_apply_preview` evidence to
  `final_showcase_readiness_report.evidence`.
- Make the `provider_key_handoff` capability reference the apply preview status
  and command.
- Keep the report read-only: no provider calls, no shell writer scripts, no
  `.env` write, no iOS local config write, no Xcode/signing mutation.
- Preserve existing final showcase readiness status semantics:
  - missing or blocked apply preview keeps provider/key handoff blocked
  - ready final resources + ready apply preview + ready resource handoff can
    progress to partial until live provider evidence is ready
  - ready live provider evidence then makes provider/key handoff ready
- Keep mobile compatibility by only changing existing capability fields that
  the iPhone already decodes: `command`, `detail`, and `evidence`.

## Non-Goals

- Do not run `make final-apply-resources`.
- Do not create or edit `services/backend/.env`.
- Do not create or edit `apps/mobile/ios/Config/Deployment.local.xcconfig`.
- Do not call Meshy, OpenAI, Treatstock, Sculpteo, Xcode, keychain, or signing
  tools.
- Do not change final resource values or require real keys.

## Data Flow

```text
final_resource_requirements
final_resources_preflight
final_resource_apply_preview
resource_handoff
live_provider_evidence
        |
        v
final_showcase_readiness.provider_key_handoff
```

`final_resource_apply_preview` is built in-process by
`build_final_resource_apply_preview_report(repo_root=...)`. Its sanitized
summary is embedded under:

```json
{
  "evidence": {
    "final_resource_apply_preview": {
      "kind": "final_resource_apply_preview_report",
      "status": "missing",
      "summary": {
        "write_targets": 2,
        "missing": 5
      }
    }
  }
}
```

The provider/key handoff capability evidence list should include:

```text
final_resource_apply_preview:<status>
```

When not ready, its command should be:

```bash
make final-resource-apply-preview
```

This makes the next safe operator action explicit before any apply command.
