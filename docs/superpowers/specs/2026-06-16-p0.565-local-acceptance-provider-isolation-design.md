# P0.565 Local Acceptance Provider Isolation Design

## Problem

`make final-acceptance-local` must be a no-live-provider acceptance path. With real provider values present in the local ignored backend env, the local generated asset handoff checks can still inherit `THREE_D_PROVIDER=meshy` and `NPC_PROVIDER=openai` through the FastAPI app factory calls. That can turn a safe local acceptance run into a live Meshy/OpenAI path.

## Goal

Keep local final acceptance deterministic and no-cost even when backend `.env` or process env contains real provider configuration.

## Approach

The local acceptance app checks will run the FastAPI app with a temporary local provider patch:

- `build_three_d_provider()` returns the local 3D provider.
- `build_npc_director()` returns the local NPC director.
- `build_npc_tick_runtime()` returns the local NPC tick runtime.
- `build_print_provider()` returns the local print provider.
- `build_myth_session_store()` uses a temporary local store so acceptance requests do not depend on the developer's persisted sessions.

`capture_scene_handoff_acceptance` already patches capture storage; it will share the same local provider patch while preserving its capture-store override.

## Safety

The patch is scoped to the acceptance run and restored in `finally`. Reports still redact raw media, local paths, auth tokens, and provider secrets. The local checks must pass even when env variables point to configured live providers without making live provider calls.

## Tests

Add focused tests that set provider env values to live/configured-looking settings and verify:

- `run_local_asset_handoff_acceptance()` still succeeds with `generated_asset_provider == "local_stub"`.
- `run_capture_scene_handoff_acceptance()` still succeeds with local served asset URLs and no provider secret leakage.
