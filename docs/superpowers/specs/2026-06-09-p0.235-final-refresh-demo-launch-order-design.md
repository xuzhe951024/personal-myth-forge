# P0.235 Final Refresh Demo Launch Order Design

## Context

P0.234 makes `ios-device-launch-rehearsal.json` sequence rows include blocker
`detail`. A safe `final-local-report-refresh` run now writes that file with
detail, but it still writes `final-demo-launch-local.json` earlier in the same
run. Therefore the final demo launch report can embed stale rehearsal readiness
without detail until the operator runs refresh a second time.

## Goal

Make one safe local refresh pass write a self-consistent final demo launch
report. `final_demo_launch_local` and `final_demo_launch_configured` should run
after the reports they aggregate, including iOS launch rehearsal, evidence
bundle, configured evidence bundle, print fulfillment readiness, final showcase
readiness, and closure packet inputs.

## Scope

This only changes refresh step ordering. It does not change report generation
logic, provider calls, resource application, Xcode behavior, signing, or mobile
UI code.

## Architecture

Move the two final demo launch refresh steps near the end of `_default_steps`,
after the source reports that final demo launch embeds. Keep final showcase and
closure packet ordering intact, then write final demo launch reports from the
fresh source files produced in the same run.

## Testing

Use TDD:

1. Add a failing test that runs `run_final_local_report_refresh` once and reads
   `final-demo-launch-local.json`.
2. Assert its embedded `ios_device_launch_rehearsal_readiness.sequence[0]`
   contains `detail` from the same refresh.
3. Assert `ios_device_launch_rehearsal` appears before `final_demo_launch_local`
   and `final_demo_launch_configured` in step order.
4. Move the steps.
5. Verify focused refresh tests, backend tests, mobile compatibility, and diff
   hygiene.
