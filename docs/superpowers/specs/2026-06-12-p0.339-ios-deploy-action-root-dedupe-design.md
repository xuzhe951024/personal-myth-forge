# P0.339 iOS Deploy Action Root Dedupe Design

## Context

P0.335-P0.338 made the project-local iOS deploy writer the preferred physical-device handoff:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

The final readiness reports can still show the same writer twice in one top-level `operator_actions` list: once as a bare command and once with a detail suffix such as:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

That duplication is not a new required step. It is the same command surfaced from two evidence paths.

## Goal

When the iOS deploy writer appears multiple times with the same command root, show it once in each top-level operator action list.

## Non-Goals

- Do not remove backend URL actions.
- Do not remove final-resources.env actions for API keys, print configuration, Team ID, bundle ID, or backend URL.
- Do not collapse unrelated commands that happen to share a validation command.
- Do not write ignored deploy config, run Xcode, touch Keychain, call providers, or place print orders.

## Design

Extend `prefer_project_local_ios_deploy_handoff_actions()` in `operator_actions.py`.

The helper already detects whether the concrete iOS deploy writer exists in an action list and, when it does, removes older generic/manual deploy-config actions. It should also collapse duplicate entries whose command root is exactly the writer validated action:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

Comparison uses the command root before any ` | detail` suffix. If two writer actions share that root:

- keep the first writer position in list order;
- prefer a detail-bearing writer action over a bare writer action;
- keep the first detail-bearing writer if multiple detail variants exist.

Existing callers in final showcase readiness, iOS device launch rehearsal readiness, and the full iOS device launch rehearsal report already pass through this helper, so the change remains shared and narrow.

## Acceptance Criteria

- `final-showcase-readiness.json.operator_actions` includes at most one iOS deploy writer action.
- `ios-device-launch-rehearsal.json.operator_actions` includes at most one iOS deploy writer action.
- If a detail-bearing writer action is present, it is the retained writer action.
- Old manual/generic deploy-config suppression from P0.337-P0.338 still works.
- Focused readiness tests, lint, diff checks, and backend tests pass.
