# P0.443 Print Readiness Validation Action Design

## Goal

Make `print-fulfillment-readiness` emit the same validation-aware operator handoff that higher-level final reports already surface for configured Treatstock quote evidence.

## Context

The final iOS showcase remains blocked by external resources and physical-device evidence. One partial capability is print fulfillment: local print proof is ready, but configured Treatstock quote evidence requires explicit cost consent and a sanitized `.local` evidence file.

`build_print_fulfillment_readiness_report()` already exposes `next_action.validation_command = "make print-fulfillment-readiness"`. Its `operator_actions`, however, currently include the quote handoff command and then a separate bare `make print-fulfillment-readiness`. That is less useful than the final reports' chained action because the operator has to infer that the validation command belongs after the quote handoff.

## Behavior

When `configured_treatstock_quote` is the first missing or blocked print readiness check, `operator_actions` should include one copy-pasteable action:

```text
after explicit Treatstock cost consent, save a sanitized services/backend/.local/print-quote-configured.json from POST /v1/print-quotes; rerun make print-fulfillment-readiness
```

The report must keep `next_action` unchanged, including `requires_cost_consent`.

When other checks are not ready, their commands may still be listed. The readiness report should avoid appending a standalone validation command when it has already chained validation onto an actionable blocker.

## Safety

This change is read-only reporting behavior. It must not call Treatstock, write secrets, create quotes, trigger payment, write Xcode config, mutate global state, or move any API key into the repository.

## Tests

Add a focused test for the partial local-proof state where the configured Treatstock quote is missing. The test should assert:

- the first operator action is the quote handoff chained with `; rerun make print-fulfillment-readiness`
- the standalone `make print-fulfillment-readiness` action is not emitted for that state
- existing `next_action` semantics remain intact
