# P0.150 Configured Evidence Plan Design

## Problem

The project now has separate reports for final resource filling, apply preview,
configured preflight, configured live provider evidence, and final launch. This
is enough for an engineer, but it is still too scattered for the final handoff:
after the user provides API keys and iOS deploy values, they need one ordered
plan that answers:

- what to run first
- which commands are read-only
- which commands write ignored repo-local config
- which commands can contact live providers and spend credits
- which evidence files prove the configured Meshy/OpenAI/iOS path

## Goal

Add a read-only configured evidence plan report that turns the existing final
resource and live evidence reports into one ordered operator sequence.

## Non-Goals

- Do not execute commands from the plan.
- Do not call Meshy, OpenAI, Treatstock, or any live provider.
- Do not write backend `.env`.
- Do not write iOS deploy config.
- Do not run Xcode, signing, keychain, or device install commands.
- Do not expose provider secrets, raw private context, raw media, payment links,
  local absolute paths, or private LAN URLs.

## Report Shape

Add `final_configured_evidence_plan.py` with
`build_final_configured_evidence_plan_report(repo_root=None, settings=None,
allow_live_provider_calls=False)`.

The report includes:

- `kind`: `final_configured_evidence_plan_report`
- `status`: `blocked`, `consent_required`, `ready_to_run`, or `ready`
- `summary`: counts for steps, blocked steps, live steps, cost steps,
  local-write steps, ready evidence, and consent-required steps
- `steps`: ordered evidence plan steps
- `steps_by_id`: keyed step lookup
- `source_reports`: bounded summaries of:
  - final resource fill guide
  - final resource apply preview
  - final configured preflight
  - live provider evidence
- `operator_actions`: concise next actions
- `commands`: ordered commands
- `live_call_policy`
- `safety`

## Step Ordering

The plan is intentionally linear:

1. `final_resource_fill_guide`: read-only list of required values.
2. `final_resource_apply_preview`: read-only preview of repo-local writes.
3. `final_apply_resources`: writes only ignored backend/iOS local config.
4. `final_configured_preflight`: read-only configured readiness check.
5. `provider_handoff`: read-only real-provider configuration handoff.
6. `three_d_evaluation_configured`: live Meshy 3D evidence, cost risk.
7. `npc_evaluation_configured`: live OpenAI NPC Agent evidence, cost risk.
8. `final_acceptance_configured`: live configured acceptance, cost risk.
9. `final_demo_launch_configured`: read-only configured launch report.
10. `live_provider_evidence`: read-only aggregate evidence check.

## Status Rules

Before resources are ready, resource and downstream steps are `blocked`.

When resources and configured preflight are ready:

- non-live steps become `ready_to_run` unless their output evidence is already
  ready
- live/cost steps become `consent_required` unless
  `allow_live_provider_calls=True`
- with `allow_live_provider_calls=True`, live/cost steps become `ready_to_run`
  unless their evidence is already ready

If all configured evidence slots are ready, the report status is `ready`.
If no required step is blocked but live/cost steps still need consent, the report
status is `consent_required`. If consent has been granted but evidence files
still need to be produced, the report status is `ready_to_run`. Otherwise it is
`blocked`.

## CLI and Make

Add:

```bash
cd services/backend && uv run python -m myth_forge_api.cli final-configured-evidence-plan --repo-root ../.. --output .local/final-configured-evidence-plan.json
```

Add Make target:

```bash
make final-configured-evidence-plan
```

The default command is read-only and does not include live consent. A CLI flag
`--allow-live-provider-calls` only changes the plan status for live steps; it
does not execute provider calls.

## Refresh Integration

Add the report to `final-local-report-refresh` so the final local evidence
bundle contains the configured evidence plan next to the fill guide and live
provider evidence report.

## Testing

Use TDD:

- missing resources produce a blocked configured evidence plan without command
  execution or secret leaks
- ready resource/preflight fixtures produce consent-required live steps by
  default
- `allow_live_provider_calls=True` changes live steps to ready-to-run without
  executing them
- ready configured evidence fixtures produce a ready plan
- CLI writes JSON
- Make dry-run exposes the target
- final local report refresh writes the configured evidence plan JSON
