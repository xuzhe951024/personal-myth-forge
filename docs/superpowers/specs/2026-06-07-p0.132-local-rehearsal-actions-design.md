# P0.132 Local Rehearsal Actions Design

## Context

P0.131 made top-level iOS device launch rehearsal actions state-aware. After a
safe refresh, the report no longer tells the operator to rerun
`make ios-device-launch-rehearsal`. It now routes blocked saved reports to their
nested `operator_actions` when available.

One gap remains: the `final_rehearsal_local` step is a synthetic aggregate of
local reports. It is blocked when local final acceptance or the iOS deploy
runbook is blocked, but it currently has no nested `operator_actions`. The
top-level report therefore falls back to:

```text
review final_rehearsal_local: make final-rehearsal-local
```

That is accurate but too coarse. The local source reports already contain
clearer actions, especially `ios_deploy_runbook_local`, which lists iOS deploy
config, resource preview, and Xcode gate actions.

## Decision

Teach `final_rehearsal_local` to aggregate nested actions from its blocked local
source reports.

Rules:

- If any local source report is missing, keep refresh actions for the missing
  source commands.
- If a local source report is blocked and has nested `operator_actions`, expose
  them as `<source id>: <action>`.
- If a local source report is blocked and lacks nested actions, fall back to
  `review <source id>: <source command>`.
- Limit the aggregate to a compact bounded list so the final iPhone handoff
  remains scan-friendly.
- Keep the existing safety guarantees: report building does not run commands,
  call providers, run Xcode, write deploy config, touch keychain, or mutate
  global state.

## Data Flow

`build_ios_device_launch_rehearsal_report` already builds `local_sources` from
the saved local report JSON files, then wraps them in `_final_rehearsal_local`.
P0.132 will add an `operator_actions` field to the synthetic
`final_rehearsal_local` step by reading the already-sanitized source summaries.
The existing top-level `_operator_actions` function will then naturally prefix
those actions with `final_rehearsal_local:`.

## Tests

The RED test creates a local rehearsal set where `ios-deploy-runbook-local.json`
is blocked and contains nested operator actions. It asserts that:

- `final_rehearsal_local` has nested `operator_actions`;
- the top-level `operator_actions` include a concrete local source action; and
- the top-level `operator_actions` do not fall back to
  `review final_rehearsal_local: make final-rehearsal-local`.

Before implementation, the test fails because `final_rehearsal_local` has no
nested actions.

The GREEN checks run:

```bash
cd services/backend && uv run pytest tests/test_ios_device_launch_rehearsal.py -q
make ios-device-launch-rehearsal
make final-demo-launch
```

The refreshed JSON should keep blocked status while exposing concrete local
actions for the iPhone Final Launch panel and CLI operator handoff.
