# P0.570 Rehearsal Cycle Break Design

## Context

The current final local handoff can form a report cycle:

`final_showcase_readiness -> ios_device_launch_rehearsal_readiness -> final_rehearsal_local -> final_demo_launch_local -> final_showcase_readiness`

When `mobile-deploy-preflight-evidence` and `mobile-xcode-build-evidence` are already ready, the cycle still keeps `ios-device-launch-rehearsal-readiness` blocked by an instruction to rerun the same rehearsal chain. This weakens the final iPhone handoff because the next action no longer points to the remaining concrete device evidence.

## Design

`ios_device_launch_rehearsal_readiness` should treat saved sequence rows that only restate the rehearsal/showcase loop as self-referential. A self-referential row is either:

- a non-ready `final_demo_launch_local` row whose detail or operator action points back to `final_showcase_readiness` or `ios-device-launch-rehearsal`; or
- a non-ready `final_rehearsal_local` row whose detail or operator action routes through `final_demo_launch_local` and then back to `final_showcase_readiness` or `ios-device-launch-rehearsal`.

The readiness report should:

- Preserve stale, missing, malformed, and explicit report-file blockers.
- Keep all saved sequence rows visible for audit.
- Choose the first non-ready non-self-referential row as `first_blocker`.
- Build `device_action_bundle.first_action` from the same preferred row.
- Fall back to the existing first non-ready row if every row is self-referential.
- Continue preserving provider, print, and Xcode actions when they are the only actionable blockers.

## Safety

This phase only changes local report interpretation. It does not run live providers, write secrets, change Xcode signing, write Keychain items, or place print orders.

## Testing

Add regression tests that write saved iOS device launch rehearsal reports with:

- a blocked `final_demo_launch_local` row that points back to `ios-device-launch-rehearsal`;
- a blocked `final_rehearsal_local` row that points through `final_demo_launch_local` and back to `ios-device-launch-rehearsal`;
- a later blocked `final_handoff_index` row with a concrete backend/mobile preflight device action.

The readiness report should select `final_handoff_index` as the first blocker and expose the concrete device action as the next action.
