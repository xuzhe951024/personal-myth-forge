# P0.286 Detail Backend Action Validation Design

## Goal

Ensure detail-suffixed `backend-device-demo` operator actions keep the `make mobile-deploy-preflight` validation handoff when final local readiness reports aggregate saved iOS device evidence.

## Context

The final iPhone showcase objective is still blocked on real external evidence: `MESHY_API_KEY`, `OPENAI_API_KEY`, Treatstock configuration, `DEVELOPMENT_TEAM`, an iPhone-reachable backend URL, and physical iPhone/Xcode proof. Local readiness reports therefore need crisp next actions so the operator can safely unblock the external steps without guessing.

Recent iterations normalized backend-device-demo and iOS deploy config actions into concrete commands with validation suffixes. The saved `final-local-report-refresh` payload still includes one variant:

```text
start backend-device-demo before device checks: make backend-device-demo | PMF_BACKEND_BASE_URL must be iPhone-reachable
```

That action carries useful detail, but it omits the validation suffix because ` | detail` is appended after the action root and prevents the shared suffix matcher from recognizing the root.

## Design

Normalize operator actions as two logical parts:

- the command portion before a ` | ` detail separator
- the optional detail suffix that should be preserved verbatim

`normalize_operator_action()` should normalize only the command portion, then reattach the detail. This keeps existing behavior for plain actions and colon-prefixed saved actions while allowing detail-suffixed backend-device-demo actions to become:

```text
start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight | PMF_BACKEND_BASE_URL must be iPhone-reachable
```

The same split should apply inside `add_mobile_deploy_validation_command()` so direct callers get identical behavior.

## Scope

In scope:

- shared operator action normalization
- focused tests for detail-suffixed backend-device-demo actions
- a final local report refresh test that proves the aggregated operator action is validation-aware
- roadmap entry for P0.286

Out of scope:

- starting the backend-device-demo server
- editing real `.local` secrets or iOS deploy values
- running live Meshy/OpenAI/Treatstock calls
- Xcode signing, Keychain, or global machine changes

## Verification

The iteration is valid when:

- a focused test fails before implementation for the detail-suffixed action
- the focused tests pass after implementation
- `make final-local-report-refresh` still exits blocked for expected external reasons but no longer emits the unvalidated detail-suffixed backend-device-demo action
- backend lint and backend tests pass
