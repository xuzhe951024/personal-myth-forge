# P0.398 Showcase iOS Destination Actions Design

## Context

The final iOS showcase remains blocked by real iOS deploy evidence and provider credentials. The top-level readiness reports therefore need to direct the operator to the right ignored local files.

P0.397 made `final-demo-launch` project iOS deploy requirements to `Deployment.local.xcconfig` and `make mobile-deploy-preflight`. `final-showcase-readiness` still surfaces some iOS deploy resource actions as:

```text
provide DEVELOPMENT_TEAM in final-resources.env; rerun make final-resources-preflight
provide PRODUCT_BUNDLE_IDENTIFIER in final-resources.env; rerun make final-resources-preflight
```

Those actions are misleading for the final iPhone deploy path. The iOS values are consumed by the ignored iOS deployment config and validated by mobile deploy preflight.

## Goal

Make `final-showcase-readiness.operator_actions` rewrite iOS deploy resource actions to:

- `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`
- `provide PRODUCT_BUNDLE_IDENTIFIER in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`
- `provide PMF_BACKEND_BASE_URL in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`

Backend provider key actions must remain in `final-resources.env`.

## Scope

- Change only `myth_forge_api.final_showcase_readiness` report action projection.
- Do not change `final_resources_preflight`, `final_resource_requirements`, resource writers, provider adapters, Xcode, signing, or live provider behavior.
- Keep existing provider/print handoff prioritization and dedupe behavior.
- Keep the first blocker and capability rows unchanged unless they already use the corrected iOS child action.

## Design

Add a destination-aware rewrite inside final showcase operator action normalization:

1. Normalize each nested report action as today.
2. Before final dedupe/prioritization, rewrite known iOS deploy resource roots from `final-resources.env` to `Deployment.local.xcconfig`.
3. Use `make mobile-deploy-preflight` as the validation command for those rewritten iOS actions.
4. Preserve backend provider secrets and print/provider handoff actions unchanged.
5. Let existing `prefer_project_local_ios_deploy_handoff_actions()` continue to prefer the deploy writer when present.

The rewrite is deliberately local to showcase readiness because lower-level resource preflight still describes the unified final resources input file. Showcase readiness is the final operator-facing report and should point to the concrete iPhone deploy handoff.
