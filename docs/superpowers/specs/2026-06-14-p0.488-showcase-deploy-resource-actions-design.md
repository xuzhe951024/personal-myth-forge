P0.488 Final Showcase Deploy Resource Actions Design

Goal

Make final showcase readiness surface all blocked iPhone deploy check/resource handoff actions that are already present in saved mobile deploy preflight evidence.

Context

The final iPhone demo is blocked on local iOS deploy resources. `mobile-deploy-preflight-evidence.json` currently records blocked checks for both:

- `development_team`: missing Apple Team ID.
- `backend_base_url`: `PMF_BACKEND_BASE_URL` must be iPhone-reachable.

The top-level `final-showcase-readiness.json` starts with the correct writer action:

`DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight`

However, it does not consistently include the direct backend URL action:

`set PMF_BACKEND_BASE_URL to an iPhone-reachable LAN URL; rerun make mobile-deploy-preflight`

That makes the final operator handoff less complete for the physical iPhone path.

Non-Goals

- Do not run `mobile-deploy-preflight`, Xcode, signing, Keychain, or real device checks.
- Do not write `Deployment.local.xcconfig` or infer a real Apple Team ID or LAN URL.
- Do not change final showcase status, first blocker, next action priority, or provider/print handoff semantics.
- Do not add new resource requirements; only surface evidence that saved reports already contain.

Design

Extend final showcase operator action aggregation to read deploy-related `checks` and `resources` from the saved mobile deploy preflight evidence report. For each blocked row, convert known IDs into the same copy-pasteable actions already used elsewhere:

- `development_team` -> `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`
- `product_bundle_identifier` -> `provide PRODUCT_BUNDLE_IDENTIFIER in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`
- `backend_base_url` -> `set PMF_BACKEND_BASE_URL to an iPhone-reachable LAN URL; rerun make mobile-deploy-preflight`

These actions should flow through existing final showcase dedupe and writer preference logic, so the writer action remains first while the backend URL action remains available.

Acceptance Criteria

- When saved mobile deploy preflight evidence has blocked `backend_base_url` in `checks` or `resources`, final showcase operator actions include the backend LAN URL action.
- The writer action for `DEVELOPMENT_TEAM` remains first when it is the saved next action.
- Existing final showcase provider, print, Xcode, self-action filtering, and validation-aware behavior remains unchanged.
- Safe local evidence remains blocked on external iOS/provider resources and does not run external actions.

Safety

This phase changes report text selection only. Verification may write ignored `.local` JSON reports through safe wrappers. It must not touch real provider keys, launch Xcode signing, mutate global Mac state, call live providers, or place print orders.
