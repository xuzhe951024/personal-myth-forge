# P0.183 Configured Acceptance Handoff Command Design

## Context

P0.182 added the manual Make entrypoint:

```text
make final-acceptance-configured
```

The wrapper behind that target runs the configured final acceptance command with
`--require-real-core` and `--allow-live-provider-calls`, then writes
`services/backend/.local/final-acceptance-configured.json`.

Several final handoff reports still display the older long raw CLI command.
That is confusing for final acceptance because the operator now has two visible
ways to run the same live-provider step. The product should present one
canonical command while preserving consent and cost-review warnings.

## Goal

Make every backend operator-facing report use `make final-acceptance-configured`
as the configured final acceptance command, while keeping the actual live
provider flags inside the wrapper created by P0.182.

## Design

Create a small shared module:

```text
services/backend/src/myth_forge_api/configured_acceptance_command.py
```

It defines:

```python
CONFIGURED_FINAL_ACCEPTANCE_COMMAND = "make final-acceptance-configured"
CONFIGURED_FINAL_ACCEPTANCE_OUTPUT = (
    "services/backend/.local/final-acceptance-configured.json"
)
CONFIGURED_FINAL_ACCEPTANCE_COST_REVIEW_ACTION = (
    "run make final-acceptance-configured only after live provider cost review "
    "and --allow-live-provider-calls consent"
)
```

Reports will import the shared command instead of spelling out the raw CLI. The
underlying Make target and shell wrapper remain the only place that owns the
full provider-mode CLI invocation.

## Affected Reports

Update command surfaces in:

- `final_configured_preflight`
- `final_demo_launch`
- `final_operator_handoff`
- `final_handoff_index`
- `final_external_action_ledger`
- `ios_deploy_runbook`
- `ios_device_launch_certificate`
- `live_provider_evidence`
- `resource_handoff`

The configured final acceptance evidence path remains unchanged. Evidence files
are still read from:

```text
services/backend/.local/final-acceptance-configured.json
```

## Safety

This slice does not run real configured final acceptance. It only changes report
command text and tests. The live call consent boundary remains:

- Reports say the operator must review cost and consent first.
- `make final-acceptance-configured` is not a dependency of
  `final-rehearsal-local`.
- The wrapper still requires `--allow-live-provider-calls` internally.

## Non-Goals

- Do not change Meshy/OpenAI/provider behavior.
- Do not run live provider calls.
- Do not change iOS signing or Xcode behavior.
- Do not change evidence file paths or JSON kinds.
- Do not remove local final acceptance.

## Acceptance

- Backend source files no longer contain the raw fragment
  `final-acceptance --profile quick --provider-mode configured`.
- The configured acceptance wrapper remains the allowed owner of the full raw
  CLI.
- Built final reports expose `make final-acceptance-configured` for configured
  final acceptance.
- Operator actions mention `make final-acceptance-configured` and preserve live
  provider cost/consent wording.
- Focused report tests pass.
- Backend lint/tests, Swift mobile checks, and safe local rehearsal still pass.

## Self-Review

- Placeholder scan: no TBD/TODO/open placeholders.
- Safety check: live-provider execution remains manual and wrapper-gated.
- Scope check: one command-handoff slice; no provider, mobile UI, Xcode, or

- Ambiguity check: report commands use Make target; evidence paths stay as file
