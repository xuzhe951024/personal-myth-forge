# P0.510 iOS Rehearsal Backend Priority Design

## Goal

Make the iOS device launch rehearsal report tell an operator to start `backend-device-demo` immediately after the deploy-config writer, before provider, print, Xcode, or provider-key handoff actions.

## Current Behavior

`services/backend/.local/ios-device-launch-rehearsal.json` is blocked as expected, but its top-level `operator_actions` currently place provider, print, and Xcode actions before the backend device server action:

1. `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight`
2. provider handoff
3. print handoff
4. Xcode license action
5. `start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight`

That conflicts with the physical iPhone path. Once the deploy writer is the first actionable blocker, the next local step should be starting the backend LAN server so `make mobile-deploy-preflight` can test an iPhone-reachable `/health`.

## Desired Behavior

When rehearsal operator actions begin with the concrete deploy writer handoff, order them as:

1. deploy writer or deploy preflight handoff
2. validated `backend-device-demo` handoff
3. iPhone-reachable backend URL handoff, when present
4. provider and print handoffs
5. remaining actions such as Xcode, provider keys, and configured evidence

If the first action is not the concrete deploy writer handoff, keep the current conservative ordering: provider and print handoffs may still move ahead as before, but all other actions keep their existing relative order so missing-report and non-device blocked states do not incorrectly promote backend startup.

## Scope

In scope:

- Add a focused regression test to `services/backend/tests/test_ios_device_launch_rehearsal.py`.
- Update `services/backend/src/myth_forge_api/ios_device_launch_rehearsal.py` sorting logic.
- Add a P0.510 roadmap entry.

Out of scope:

- Real Xcode signing, Keychain writes, provider calls, print quotes, or paid actions.
- Changing provider/key requirements.
- Changing the first blocker, which remains `DEVELOPMENT_TEAM` until the operator supplies device values.
