# P0.409 Final Showcase First Device Action Design

## Context

The final iPhone showcase readiness report now exposes concrete top-level `first_blocker` and `next_action` commands for `ios_deployable`. When saved mobile deploy preflight evidence contains a child `next_action`, the top-level blocker correctly points at the concrete deploy-config writer, for example `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto`.

The same report's `device_action_bundle.first_action` still selects the first blocked action by static action order. In current local evidence, that makes the bundle first action `make backend-device-demo` while the top-level iOS blocker tells the operator to write deploy config first. This is confusing for the final handoff even though both actions are individually valid.

## Goal

Make `final-showcase-readiness.json` expose one consistent first iPhone action: if mobile deploy preflight evidence provides a concrete child `next_action`, `device_action_bundle.first_action` should prefer that action over the static backend-start action.

## Scope

- Update final showcase readiness only.
- Keep the existing `actions` array order stable for UI rows.
- Keep top-level blocker and next action behavior unchanged.
- Preserve no-mutation safety flags and no-provider-call behavior.
- Add focused tests for child `next_action` promotion and existing ready-evidence behavior.
- Update the roadmap with P0.409.

## Non-Goals

- Do not run backend, Xcode, signing, Keychain, provider, iPhone, or paid-print commands.
- Do not write iOS deploy config or backend env files.
- Do not change mobile UI models in this iteration.
- Do not change final resource/key blocker ordering.
- Do not embed secrets, local absolute paths, raw media, or payment links.

## Design

The bundle builder will continue to derive four stable actions from saved device evidence:

- `start_backend_device_demo`
- `run_mobile_deploy_preflight`
- `resolve_xcode_build_gate`
- `run_ios_device_launch_rehearsal`

Only `first_action` selection changes. `_first_device_action` will prefer the blocked `run_mobile_deploy_preflight` action when it carries a child `next_action.command`; it will return a copy of that action with its top-level `command` and `validation_command` replaced by the child command and validation command. This keeps the action source, evidence detail, evidence status, and blocks intact while making the first action copy-pasteable and aligned with the top-level blocker. The backend-start row can still display the same evidence detail, but it must not consume the preflight child action as the bundle's first action.

If no child `next_action` is available, the existing fallback remains: first non-ready action, or the first action if the bundle is non-empty.
