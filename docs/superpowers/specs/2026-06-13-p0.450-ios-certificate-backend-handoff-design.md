# P0.450 iOS Certificate Backend Handoff Design

## Context

The final showcase ledger is still blocked on physical iPhone deploy evidence. Recent phases made the backend-device-demo handoff visible in final demo launch and iOS rehearsal reports, but the iOS device launch certificate still drops that handoff from its top-level `operator_actions`.

The saved `final-handoff-index.json` includes:

```text
start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight
```

The saved `ios-device-launch-certificate.json` exposes backend startup in nested device actions and command metadata, but its top-level `operator_actions` are capped at six and prioritize provider/print/iOS config actions first. The validated backend startup step is filtered out before the final operator checklist.

## Goal

Preserve the validated backend-device-demo handoff in the iOS device launch certificate's top-level `operator_actions` so a final physical-iPhone operator sequence has the LAN backend startup step visible before the user runs mobile deploy preflight.

## Approaches Considered

1. Increase the top-level action limit from six to seven.
   - Pro: minimal code.
   - Con: treats the symptom as a list-size problem and can still hide backend startup if more actions are added ahead of it.

2. Classify backend-device-demo as a certificate priority handoff.
   - Pro: matches the device launch purpose and keeps the visible checklist stable under list truncation.
   - Con: requires a small predicate and priority update.

3. Rebuild certificate actions from `device_action_bundle`.
   - Pro: canonical device action source.
   - Con: larger refactor and risks changing existing provider/print action ordering.

Recommended approach: option 2. It is narrow, testable, and keeps existing provider/print ordering while adding the missing physical-device handoff.

## Design

Add a backend-device-demo handoff classifier in `ios_device_launch_certificate.py`. The classifier should match normalized actions containing `backend-device-demo`, including the canonical validated action produced by `BACKEND_DEVICE_DEMO_VALIDATED_ACTION`.

Use that classifier in two places:

- `_final_handoff_priority_actions()` should select backend-device-demo actions from `final_handoff_index` and `final_demo_launch`, instead of dropping them as non-IOS actions.
- `_prioritize_certificate_operator_actions()` should keep backend-device-demo actions in the priority group after provider and print actions, before lower-value remaining actions are truncated.
- If both the complete preflight-validated backend handoff and the bare `start backend-device-demo` action are present, the certificate should output only the complete handoff in top-level `operator_actions`.

The existing first action behavior remains unchanged: if a deploy writer action is the first blocker, it stays first. Provider and print handoff actions stay ahead of the backend action so current configured-resource visibility is preserved.

## Testing

Add a regression test to `tests/test_ios_device_launch_certificate.py` where the injected final handoff index contains the validated backend-device-demo handoff and the final demo launch contributes the deploy action. The test should fail before implementation because the top-level certificate actions do not include the backend startup handoff, then pass after the classifier and prioritization update.

Focused verification:

```bash
cd services/backend
uv run pytest tests/test_ios_device_launch_certificate.py::test_ios_device_launch_certificate_preserves_backend_device_demo_handoff -q
uv run pytest tests/test_ios_device_launch_certificate.py::test_ios_device_launch_certificate_preserves_backend_device_demo_handoff tests/test_ios_device_launch_certificate.py::test_ios_device_launch_certificate_uses_final_demo_child_action_when_handoff_has_none tests/test_ios_device_launch_certificate.py::test_ios_device_launch_certificate_promotes_final_handoff_provider_and_print_actions -q
```

Final safe verification:

```bash
make backend-lint
git diff --check
```

No real backend server, Xcode signing, provider key, live provider call, or print ordering action is performed in this phase.
