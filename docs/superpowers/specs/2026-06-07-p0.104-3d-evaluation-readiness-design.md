# P0.104 3D Evaluation Readiness Design

## Product Intent

Before an iPhone demo can honestly claim real 3D generation readiness, the operator needs a visible answer to one question: did the saved 3D provider evaluation actually run, cover the expected prompt and scan modes, and finish without failures? The repo already has `evaluate-3d`; P0.104 makes its saved output part of final launch readiness and the mobile final launch status.

## Scope

P0.104 adds a read-only `three_d_evaluation_readiness` report. It reads a saved local JSON file at `services/backend/.local/3d-evaluation-local.json`, validates that it is a `three_d_evaluation_report`, summarizes total/succeeded/failed cases, and checks coverage for text prompt input, image-backed input, scene-loadable variants, and generated variant roles.

The readiness builder never runs `evaluate-3d`, never calls Meshy, never opens media, and never exposes API keys, raw media, local paths, payment links, or raw private context. It only turns saved evaluation output into a safe final-launch status.

## Backend Contract

Add `myth_forge_api.three_d_evaluation_readiness` with:

- `DEFAULT_THREE_D_EVALUATION_PATH = services/backend/.local/3d-evaluation-local.json`
- `LOCAL_THREE_D_EVALUATION_COMMAND` pointing at `evaluate-3d --provider local --suite default-v0`
- `build_three_d_evaluation_readiness_report(...)`

Statuses:

- `missing`: no saved evaluation report exists. Operator action tells the user how to run `evaluate-3d`.
- `ready`: saved report is valid, has at least one success, zero failures, text-prompt coverage, scene-loadable cases, and generated variant roles.
- `blocked`: saved report is unreadable, wrong kind, empty, failed, or lacks required 3D coverage.

The final launch report embeds `three_d_evaluation_readiness`. The final operator handoff gets a `three_d_evaluation` step after local final acceptance and before NPC evaluation, because 3D quality readiness is a core product demo gate.

## Mobile Contract

Add Swift Codable models for `ThreeDEvaluationReadinessReport`, then add `threeDEvaluationReadiness` to `FinalDemoLaunchReport`. `FinalLaunchMobileSummary` gains `threeDEvaluationRows`, displayed by `FinalLaunchStatusView` under a `3D Evaluation` section.

Ready rows show case count, text/image input coverage, scene-loadable cases, and variant-role coverage. Missing and blocked rows give operator-readable next actions and sanitized blocker details.

## Acceptance

Backend tests cover missing, ready, invalid JSON, failed/redacted report, final launch embedding, and final operator handoff step status. Swift contract tests cover decoding from final launch payload, ready summary rows, blocked summary rows, and redaction. ProjectChecks and iOS showcase acceptance add source gates for the new models, summary rows, view section, and contract test names.

## Non-Goals

P0.104 does not run Meshy, score visual quality automatically, inspect GLB geometry, store generated media, modify `evaluate-3d` case generation, add Unity integration, place print orders, or resolve Apple signing/iOS deploy blockers.
