# P0.571 Rehearsal Ready Action Design

## Context

After `make backend-device-demo`, `make mobile-deploy-preflight`, and
`make ios-device-launch-rehearsal` succeed, the saved iOS launch rehearsal can
still remain `partial` because live Meshy/OpenAI evidence has not been approved
or generated. That state is valid for local mode: local device rehearsal is no
longer blocked by backend reachability, but the final goal still needs live
provider consent and configured evidence.

The current readiness chain misreads this state. `ios-device-launch-rehearsal`
has no `first_blocker`, yet still exposes a fallback `operator_actions` entry:

```text
start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight
```

Downstream reports treat that fallback as an actionable blocker, so
`ios-device-evidence-bundle`, `final-showcase-readiness`, and
`final-launch-closure-packet` keep telling the operator to repeat a device
backend step that already passed.

## Design

`ios_device_launch_rehearsal_readiness` should distinguish two cases:

- A saved rehearsal report is actually blocked or missing. Existing blocker and
  operator-action routing should remain unchanged.
- A saved rehearsal report is `partial` with no `first_blocker` and no
  `next_action`. This means the local rehearsal evidence is acceptable for the
  device lane, and the remaining work belongs to later live provider or manual
  evidence lanes.

For the second case, readiness should:

- Treat the saved rehearsal row as `ready` for device-lane aggregation.
- Preserve the original saved status in a `saved_status` field for audit.
- Drop fallback backend-device-demo operator actions from the readiness output.
- Keep provider consent actions in upstream reports where they are the actual
  remaining work.
- Continue returning `blocked` when a saved rehearsal has a real blocker, stale
  file, missing file, malformed JSON, or explicit non-device failure.

## Safety

This phase changes local report interpretation only. It does not run live
provider calls, write secrets, change Xcode signing, write Keychain items,
accept Apple licenses, or place print orders.

## Testing

Add regression coverage for a saved `ios-device-launch-rehearsal.json` report
with `status=partial`, no `first_blocker`, no `next_action`, all saved sequence
rows non-blocked, and only the fallback backend-device-demo operator action.
The readiness report should become `ready`, expose no next action, and keep the
saved partial status visible for audit.
