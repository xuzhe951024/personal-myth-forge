# P0.349 Apply Preview Blocker No Writer Design

## Context

`final-resource-apply-preview.json` is a read-only handoff report. It shows what
`make final-apply-resources` would write without touching `services/backend/.env`
or `apps/mobile/ios/Config/Deployment.local.xcconfig`.

The report currently keeps the direct writer command in blocked
`first_blocker.command` and `next_action.command` values:

```text
make final-apply-resources
```

That is too easy to misread in the final launch handoff. While provider keys or
iOS deploy values are missing, the actionable path should remain initialization,
value fill, preflight, and preview. Direct apply is valid only after preview is
ready and explicitly reviewed.

## Goal

Make blocked or missing final resource apply-preview blockers avoid direct writer
commands. The report may still document the writer in target metadata and the
command list, but `first_blocker.command` and `next_action.command` must point to
a non-writing action until the preview is ready.

## Non-Goals

- Do not remove `make final-apply-resources` from the static `commands` list; the
  report still documents the eventual writer.
- Do not change `final-apply-resources` behavior.
- Do not execute resource writers, backend `.env` writes, iOS deploy config
  writes, Xcode signing/builds, live provider calls, or print orders.
- Do not change final resources parsing, key requirements, or secret redaction.

## Design

Add apply-preview blocker command derivation:

- If the resources file is missing, use `make final-resource-init`.
- If a target is blocked by a missing required slot, use
  `provide <KEY> in final-resources.env`.
- If `PMF_BACKEND_BASE_URL` is blocked, use
  `set PMF_BACKEND_BASE_URL to an iPhone-reachable LAN URL`.
- If `PRODUCT_BUNDLE_IDENTIFIER` is blocked, use
  `set PRODUCT_BUNDLE_IDENTIFIER to a unique app bundle id`.
- Otherwise, use `make final-resource-apply-preview` as the safe non-writing
  recheck.

`first_blocker` keeps its target-oriented identity, destination, writer, and
`blocked_by` list so downstream reports still know which output is blocked. Only
the actionable command changes away from direct apply.

## Acceptance Criteria

- Missing-resource apply-preview reports use `make final-resource-init` for
  `first_blocker.command` and `next_action.command`.
- Existing-resource reports with missing Meshy/OpenAI values use a value-fill
  command instead of `make final-apply-resources`.
- Existing-resource reports with a loopback backend URL use the iPhone-reachable
  backend URL action instead of `make final-apply-resources`.
- Ready apply-preview behavior is unchanged: `first_blocker` and `next_action`
  are `null`, and operator actions may still say to review preview before direct
  apply.
- Focused apply-preview tests, operator action tests, lint, diff checks, and the
  refreshed local apply-preview report pass.
