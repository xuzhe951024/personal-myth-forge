# P0.230 Operator Action Make Normalization Design

## Context

P0.229 normalized configured handoff report command fields to Make targets. The current final showcase readiness report can still promote operator actions from older saved nested evidence, such as an iOS rehearsal report, that contain raw `uv run python -m myth_forge_api.cli final-demo-launch --mode configured ...` commands. That creates a mixed final handoff checklist even after the current report builders are fixed.

## Goal

Normalize promoted operator action strings so final handoff lists consistently show Make targets:

- Raw configured final demo launch CLI -> `make final-demo-launch-configured`
- Raw local final demo launch CLI -> `make final-demo-launch-local`
- Raw provider handoff CLI -> `make provider-handoff`

The normalization must preserve source prefixes such as `final_handoff_index: run ...` so the final showcase still shows where an action came from.

## Scope

This iteration only changes operator action string normalization and its tests. It does not run live providers, write resource files, alter iOS signing, or change report status logic.

## Architecture

Extend `myth_forge_api.operator_actions.normalize_operator_action` with command rewrite rules. Keep the function deterministic and string-only because it is used when aggregating actions from multiple saved reports.

Then add tests at two levels:

- Unit tests for direct action normalization.
- Final showcase readiness tests proving nested legacy actions are normalized when promoted into the top-level operator checklist.

## Testing

Use TDD:

1. Add failing tests for legacy command normalization.
2. Implement the minimal string rewrite logic.
3. Verify focused backend tests, Swift project checks, full backend lint/test, and refreshed read-only reports.

## Self-Review

- No placeholders remain.
- Scope is limited to operator-facing strings.
- Safety posture is unchanged: no commands are executed by the reports.
