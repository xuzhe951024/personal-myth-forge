# P0.170 Configured Bundle Showcase Gate Design

## Decision

Implement option C as a narrow final acceptance hardening pass. The iOS final
showcase and live provider consent surface must treat the configured live
evidence bundle as the canonical readiness bundle for configured acceptance.

## Current Gap

P0.169 decodes `configuredLiveEvidenceBundle`, renders
`configuredEvidenceBundleRows`, and shows a dedicated "Configured Evidence
Bundle" group in `FinalLaunchStatusView`. The higher-level handoff surfaces
still have two gaps:

- `FinalShowcaseSummary.provider_handoff` only considers live evidence,
  apply preview, fill guide, and resource handoff rows. A blocked configured
  bundle can be invisible in the top-level showcase stage.
- `LiveProviderConsentSummary.canRunConfiguredAcceptance` only requires provider
  readiness, final resources, resource handoff, and live evidence. It can report
  configured acceptance as runnable even when the configured bundle itself is not
  ready.

## Considered Approaches

1. Add a separate top-level showcase stage for the configured bundle.
   This is explicit, but it adds another stage to a screen that is already dense.
2. Fold configured bundle rows into the existing provider handoff stage and live
   consent gate.
   This keeps the UI stable and makes the existing handoff stage represent the
   real configured-provider acceptance state.
3. Only tighten backend source acceptance.
   This prevents future regressions in code shape, but it does not change the
   iPhone behavior.

Use approach 2, with approach 3 as a source-level guard. Do not add new screens
or broad visual changes.

## Scope

- `FinalShowcaseSummary` includes `configuredEvidenceBundleRows` in provider
  handoff candidate rows.
- If live evidence is ready but the configured bundle is blocked, the provider
  handoff stage becomes `needsAttention` and shows the bundle blocker/action.
- If both live evidence and the configured bundle are ready, provider handoff
  remains ready.
- `LiveProviderConsentSummary` adds a "Configured bundle" row from
  `FinalDemoLaunchReport.configuredLiveEvidenceBundle`.
- `canRunConfiguredAcceptance` is true only when the configured bundle status is
  ready, along with existing provider/resource/live-evidence checks.
- Backend source acceptance requires the new Swift contract tests and source
  references so the final iOS acceptance gate catches regressions.

## Privacy And Safety

All user-facing text continues through the existing Swift sanitizers. The new
rows must not expose provider secrets, absolute local paths, raw capture URIs,
checkout URLs, bearer tokens, or API key values. The iPhone remains read-only:
it summarizes whether Mac-side configured acceptance can be run, but it does not
call live providers directly.

## Tests

Add Swift contract tests first:

- provider handoff blocks on a blocked configured bundle even when live evidence
  is ready.
- provider handoff redacts unsafe configured bundle blocker text.
- live provider consent shows a configured bundle row.
- live provider consent blocks `canRunConfiguredAcceptance` when the bundle is
  blocked.

Add backend acceptance tests first:

- `ios_showcase_acceptance` requires the new provider handoff bundle test.
- `ios_showcase_acceptance` requires the live consent configured bundle row/test.

Verification after implementation:

- `cd services/backend && uv run pytest tests/test_ios_showcase_acceptance.py -q`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests`
- `make backend-lint`
- `swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck`

## Out Of Scope

- No Meshy, OpenAI, or print provider live call changes.
- No new key storage behavior.
- No new iOS screen or visual redesign.
- No Unity integration.
