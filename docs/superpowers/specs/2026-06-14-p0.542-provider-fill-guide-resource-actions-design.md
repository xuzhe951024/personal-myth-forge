# P0.542 Provider Fill Guide Resource Actions Design

## Context

The final showcase reports already have a safe ignored handoff for provider
setup:

```text
make final-resource-fill-guide; rerun make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence
```

This chain is better than individual bare actions such as
`provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight`
because it starts with the generated fill guide, keeps the secret destination
inside ignored local files, previews the apply step, and then refreshes provider
evidence.

Current saved reports still surface the bare Meshy/OpenAI key actions in several
top-level operator action lists and in the `provider_key_handoff` capability.
Those actions are not unsafe, but they are less executable than the canonical
fill-guide chain and compete with the current preferred handoff.

## Goal

Make final provider/key handoff rows and top-level operator action lists prefer
the canonical fill-guide provider chain whenever provider resources are missing,
while preserving detailed missing-resource evidence inside resource-specific
reports.

## Non-Goals

- Do not read or write real provider keys.
- Do not run Meshy, OpenAI, Treatstock, Xcode, signing, or any paid/provider
  command.
- Do not remove detailed resource missing rows from
  `final-resources-preflight`, `final-resource-requirements`, or
  `final-resource-fill-guide`.
- Do not change the current first blocker priority for physical iPhone
  deployment.

## Design

Add a shared operator action helper that treats backend provider key actions
for `MESHY_API_KEY`, `OPENAI_API_KEY`, and configured print-provider keys as
subsumed by the canonical provider live handoff chain when that chain is already
present. iOS deploy values stay separate because they unblock a different device
path and already have a dedicated deploy writer handoff.

Update the `provider_key_handoff` capability so that missing local provider
resources point to the canonical provider chain instead of copying the child
`final_resources` missing-key command. The capability will keep
`validation_command: make live-provider-evidence`, matching the final chain's
last proof target, and the evidence list will still show which underlying reports
are blocked.

Aggregation reports that combine many child action lists should run the shared
preference helper after normalization and before prioritization. This keeps
`final-showcase-readiness`, `final-local-report-refresh`,
`final-external-action-ledger`, and `final-launch-closure-packet` aligned.

## Acceptance Criteria

- `provider_key_handoff.command` is the canonical fill-guide provider chain when
  final resources are missing.
- Top-level final handoff operator action lists do not include bare
  `provide MESHY_API_KEY in final-resources.env` or
  `provide OPENAI_API_KEY in final-resources.env` when the canonical chain is
  present.
- Resource-specific reports still retain missing-resource detail and validation
  commands.
- The current no-key local state still writes blocked `.local` readiness evidence
  without returning failing Make exit codes for safe wrapper targets.
