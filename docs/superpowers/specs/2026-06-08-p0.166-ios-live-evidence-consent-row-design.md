# P0.166 iOS Live Evidence Consent Row Design

## Context

P0.165 made `provider-handoff.json` a first-class saved evidence report and made
`live-provider-evidence.json` explain when core real providers are not ready for
live evidence. The iPhone already decodes `live_provider_evidence` inside the
final launch payload, but the `Live Provider Consent` panel only summarizes
provider readiness, live-call policy, final resources, and resource handoff.

That leaves a demo gap: the operator can see that live provider consent is
blocked, but the clearest evidence-specific blocker is one panel away in the
larger final launch digest. For the iOS-first showcase, the consent panel should
surface the live evidence gate directly.

## User-Facing Goal

On iPhone, the `Live Provider Consent` panel shows a dedicated `Live evidence`
row that summarizes the saved live provider evidence state and, when blocked or
missing, includes the first blocker detail such as:

`provider_handoff blocked: Core real providers are not ready for live evidence.`

Provider keys, local paths, raw media URIs, authorization headers, and checkout
URLs must remain redacted in the mobile summary.

## Approaches Considered

1. **Recommended: add a live evidence row to `LiveProviderConsentSummaryBuilder`.**
   This keeps the consent panel's data source in the mobile core summary layer,
   reuses already-decoded `FinalDemoLaunchReport.liveProviderEvidence`, and lets
   the existing SwiftUI view render it without layout changes.

2. **Add a second visual block to `LiveProviderConsentView`.**
   This would provide more layout control, but it duplicates status formatting
   already handled by summary rows and increases UI code surface.

3. **Leave the consent panel unchanged and rely on Final Launch details.**
   This preserves current code but makes the live provider gate less visible
   during the final iPhone demo. It is weaker for operator acceptance.

## Design

`LiveProviderConsentSummaryBuilder.build(...)` will append a `live_evidence`
row whenever a final launch report is available. The row will be derived from
`FinalDemoLaunchReport.liveProviderEvidence`.

If no live evidence report is present, the row status is `waiting` and the
detail says the report has not loaded. If the report is `ready`, the row status
is `ready` and the detail summarizes ready evidence counts. If the report is
`blocked`, `missing`, or `partial`, the row status maps through the existing
consent status helper and includes the first blocker when available. The detail
uses only sanitized fields already present in the final launch payload:
evidence status, summary counts, first blocker id/status/classification/detail,
and no secrets.

The existing `LiveProviderConsentView` will render the new row through the
current `ForEach(summary.rows)` loop. No separate UI component is needed.

## Data Flow

1. Backend `GET /v1/final-demo-launch` returns `live_provider_evidence`.
2. `PersonalMythForgeAPIClient` decodes it into `FinalDemoLaunchReport`.
3. `ForgeRootView` passes the final launch report to
   `LiveProviderConsentSummaryBuilder`.
4. The summary builder appends `Live evidence` to `summary.rows`.
5. `LiveProviderConsentView` renders the row on the phone.

## Error Handling

- Missing final launch report still shows the existing waiting rows.
- Missing `liveProviderEvidence` inside a loaded final launch report shows a
  waiting `Live evidence` row.
- Unsafe text is redacted through the existing summary sanitizer.
- The row does not execute commands, call providers, mutate config, or write
  local reports.

## Testing

- Add Swift contract coverage proving the consent summary shows:
  - a ready live evidence row,
  - a blocked provider handoff first blocker detail,
  - redaction of unsafe blocker text.
- Add source/acceptance coverage so the mobile readiness acceptance suite knows
  the consent summary includes live evidence.
- Run Swift contract tests, Swift project checks, Swift build, backend focused
  acceptance tests, backend lint, and visual regression local evidence.

## Non-Goals

- No new backend endpoint.
- No live provider calls.
- No Xcode signing, device build, or global developer directory mutation.
- No change to provider key storage policy.
