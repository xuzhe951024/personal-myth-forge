# P0.45 iPhone Backend Handoff Design

## Context

The mobile app already reads `PMFBackendBaseURL` from Xcode config and the
deploy preflight blocks loopback URLs for iPhone deployment. The backend run
commands, however, still center on `make backend-dev`, which runs Uvicorn on the
default localhost interface. That is fine for browser/local Mac testing, but a
real iPhone needs the backend listening on a LAN-reachable interface and the app
configured with the Mac's LAN URL.

P0.45 closes that handoff gap without mutating global state, accepting Apple
licenses, starting long-running servers during tests, or claiming real device
validation.

## Goal

Make the iPhone backend launch path explicit and include it in quick final
acceptance as a source-level handoff gate.

## Non-Goals

- Do not run Xcode, simulators, or device installs.
- Do not accept Apple SDK licenses or change `xcode-select`.
- Do not start a long-running backend server during automated acceptance.
- Do not infer or write a developer's LAN IP into tracked config.
- Do not call Meshy/OpenAI or print providers.

## Behavior

Add a Make target:

```bash
make backend-device-demo
```

It runs the same FastAPI app as `backend-dev`, but binds to `0.0.0.0:8080` so an
iPhone on the same LAN can reach it when macOS firewall/network permissions
allow the connection.

Add a source-only backend acceptance module:

```python
run_ios_backend_handoff_acceptance(repo_root=None)
```

The report checks that:

- `Makefile` exposes `backend-device-demo`.
- The target launches `myth_forge_api.main:app` with `--host 0.0.0.0` and
  `--port 8080`.
- `Deployment.local.xcconfig.example` uses a LAN-style backend URL.
- `deploy_preflight.sh` rejects loopback backend URLs.
- `Info.plist` reads `$(PMF_BACKEND_BASE_URL)`.
- `AppConfiguration.swift` reads `PMFBackendBaseURL`.
- README/mobile README explain the device backend handoff.

The report includes relative evidence strings only. It does not include local
paths, keys, raw media, or provider data.

Wire the new acceptance check into `final-acceptance --profile quick` after
`ios_showcase_acceptance`. On the default machine, the quick summary should move
from `passed=5, blocked=2` to `passed=6, blocked=2`; the two blockers remain
local deploy config and Apple SDK/Xcode license.

## Operator Flow

Local browser/backend demo remains:

```bash
make backend-dev
```

iPhone demo backend becomes:

```bash
make backend-device-demo
```

Then copy the example local Xcode config and set the iPhone-reachable URL:

```xcconfig
PMF_BACKEND_BASE_URL = http://<mac-lan-ip>:8080
```

Run:

```bash
make mobile-deploy-preflight
make mobile-xcode-build
```

The first command validates project-local config. The second remains blocked on
this machine until the Apple SDK license state is handled outside the repo.

## Tests

Backend tests:

- New iOS backend handoff acceptance passes on a complete fixture.
- It fails with sanitized missing evidence when `backend-device-demo` is absent.
- Final acceptance includes the new check in quick and full profiles.
- Quick profile summary updates to six passing source/in-process checks and two
  known local blockers.

Docs/source tests:

- Resource handoff command recommendations include `make backend-device-demo`.
- Existing final acceptance and CLI reports remain sanitized.
