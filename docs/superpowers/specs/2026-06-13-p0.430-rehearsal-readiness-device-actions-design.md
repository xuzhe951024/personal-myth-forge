# P0.430 Rehearsal Readiness Device Actions Design

## Context

The final iOS deploy blocker is now mostly a physical-device evidence problem. `ios-device-launch-rehearsal-readiness.json` already exposes `first_blocker` and `next_action`, but it does not expose a `device_action_bundle`. Other final launch reports use device action bundles to give the iPhone UI structured, ordered manual actions with safety metadata.

## Decision

Add a `device_action_bundle` to `ios_device_launch_rehearsal_readiness_report`. The bundle will be derived from the sanitized readiness `sequence`, not from new commands or provider calls. The iOS model will decode the bundle and the Final Launch summary will show the bundle rows so the physical iPhone rehearsal path is visible on device.

## Behavior

- Missing or invalid saved rehearsal reports still return a bundle with one manual action to run `make ios-device-launch-rehearsal`.
- Saved rehearsal sequence rows become device actions.
- The first non-ready sequence row becomes `first_action`.
- Device actions include `id`, `label`, `status`, `classification`, `command`, `detail`, `manual`, provider/global/Xcode safety flags, and `validation_command`.
- Non-ready actions include a nested `next_action` with `source: "device_action_bundle"`.
- If all actions are ready, `first_action` falls back to the first action so the existing mobile bundle model can decode the report.
- iOS Final Launch summary appends the rehearsal readiness device action rows after sequence/operator action rows.
- Existing `first_blocker`, `next_action`, `operator_actions`, and freshness behavior stays unchanged.

## Safety

The builder remains read-only. It only parses ignored `.local` JSON files and sanitizes paths, raw media references, payment links, provider secrets, and private context markers already handled by the module. It does not run shell commands, touch Xcode, write env files, write deploy config, call providers, or place print orders.

## Tests

Backend tests will verify that a blocked saved rehearsal readiness report includes a device action bundle whose first action is the local rehearsal blocker, whose command and validation command are actionable, and whose safety metadata remains false for external side effects. Swift contract tests will verify decode and Final Launch summary display.
