# P0.391 iOS Runbook Resource Blocker Design

## Context

The final showcase is currently blocked on the iOS deployable path. The iOS
deploy runbook correctly includes final resource checks, but its first blocker
always says `run make final-resource-init` for the `final_resources_env` slot.
That is correct when the ignored resource file is missing, but misleading when
`services/backend/.local/final-resources.env` already exists and specific values
such as `MESHY_API_KEY` are missing.

## Goal

Make the iOS deploy runbook surface the concrete final-resource blocker when
the resource file exists but is incomplete. This keeps the iOS deploy path
actionable without applying resources, writing secrets, touching signing, or
calling live providers.

## Behavior

- If `final-resources.env` is missing, keep the existing first action:
  `run make final-resource-init`.
- If `final-resources.env` exists but final resources are blocked, use the child
  preflight blocker command and validation command, for example:
  `provide MESHY_API_KEY in final-resources.env` plus
  `make final-resources-preflight`.
- Preserve the runbook slot id `final_resources_env` so existing consumers do
  not lose the iOS runbook grouping.
- Add sanitized child blocker fields to the slot and first blocker:
  `source_blocker_id`, `source_blocker_label`, `validation_command`, and a
  specific detail.
- Keep reports read-only and sanitized. Do not include secret values, absolute
  temporary paths, raw private context, media payloads, payment links, or live
  provider output.

## Testing

Add a focused regression where `final-resources.env` exists with only optional
defaults. The runbook should remain blocked, keep `final_resources_env` as the
first blocker id, and expose `provide MESHY_API_KEY in final-resources.env` as
the command with `make final-resources-preflight` as validation.

Existing missing-file tests keep proving `run make final-resource-init`.
