# P0.22 NPC Agent Ticks Design

## Context

P0.17 introduced NPC agent traces on the initial myth session. The backend can
produce a structured trace per NPC, and the mobile app can render those traces.
That proves the first interpretation pass, but the target demo needs NPCs that
appear self-directed after the artifact arrives. Today there is no contract for
asking the NPC runtime what happens next.

## Approaches Considered

### Recommended: stateless next-tick API

Add a `POST /v1/npc-ticks` endpoint that accepts the current `MythSession`, a
tick number, and optional recent visible events. The backend produces one new
agent trace and reaction per NPC, then passes those reactions through the
existing world arbitrator. This creates a visible loop without introducing
database persistence, background workers, or Unity dependencies.

### Alternative: session persistence plus tick history

The backend could store sessions and tick history server-side. That is closer
to the eventual product, but it introduces storage, migration, and replay
questions before the mobile vertical slice needs them.

### Alternative: mobile-only simulated ticks

The iOS app could synthesize follow-up NPC actions locally. This would be easy
to display, but it moves agent behavior away from the backend provider boundary
and would not exercise the eventual OpenAI/local NPC runtime interface.

## Design

Add a small NPC tick runtime beside the existing NPC director. The tick runtime
takes a `MythSession`, `tick_index`, and `recent_events`. It returns an
`NPCAgentTick` with:

- `session_id`
- `tick_index`
- `agent_runtime`
- `npc_agent_traces`
- `npc_reactions`
- `world_resolution`

The initial implementation uses a deterministic local runtime. It derives each
NPC's next belief from the previous session state and recent events, proposes a
single visible action per NPC, then uses the existing `LocalWorldArbitrator` to
accept or reject actions. This keeps behavior reproducible for tests while
making the demo feel like NPCs are continuing to deliberate and act.

The OpenAI path remains a backend-only extension point. P0.22 does not add live
OpenAI calls for ticks, but it keeps the output shape identical to the existing
agent trace and world resolution models so a future `OpenAINPCTickRuntime` can
slot in behind the same API without mobile changes.

## API Contract

Request:

```http
POST /v1/npc-ticks
```

```json
{
  "session": { "...": "existing MythSession payload" },
  "tick_index": 1,
  "recent_events": ["Mara knelt near the relic."]
}
```

Response:

```json
{
  "session_id": "myth_...",
  "tick_index": 1,
  "agent_runtime": "local_tick_runtime",
  "npc_agent_traces": [],
  "npc_reactions": [],
  "world_resolution": {}
}
```

The endpoint does not accept raw personal data. It only receives the approved
capsule-derived myth session payload and optional short event summaries.

## Mobile Surface

The Swift mobile core adds `NPCAgentTick`, `NPCAgentTickRequest`, and
`PersonalMythForgeAPIClient.createNPCAgentTick(...)`. The app adds a compact
`NPCTickView` below the existing village response. It can show the latest tick
result, accepted/rejected actions, and visible world changes. P0.22 keeps this
as an explicit user-triggered demo action rather than a timer, because timer
loops would complicate test determinism and simulator/device debugging.

## Testing

Backend tests:

- local tick runtime returns one trace and reaction per NPC
- world arbitration rejects unsafe tick actions
- `/v1/npc-ticks` returns the tick contract from a valid myth session payload
- API errors are sanitized and do not echo data URIs, secrets, or raw personal
  details

Mobile tests:

- Swift decodes `NPCAgentTick`
- API client posts `POST /v1/npc-ticks` with snake_case body keys
- HTTP errors from tick calls use the existing sanitized error path
- project checks require the new app view and Xcode source membership

Visual regression:

- static mobile-width evidence page for the NPC tick panel showing runtime,
  NPC intentions, accepted/rejected actions, and visible world changes

## Non-Goals

P0.22 does not add persistent server-side sessions, background autonomous loops,
Unity character movement execution, voice NPCs, live OpenAI tick generation,
or push notifications.
