# P0.259 Final Refresh Showcase Next Action Design

## Context

`final-local-report-refresh.json` is the safe Mac-side report that refreshes all ignored `.local` evidence before the final iPhone showcase. Its top-level `first_blocker` and `next_action` correctly point to the first missing final resource, currently `MESHY_API_KEY`.

The current global goal also needs the next iPhone showcase action to stay visible. `final-showcase-readiness.json` already exposes the concrete device action:

- `command`: `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig`
- `validation_command`: `make mobile-deploy-preflight`
- `detail`: includes both missing `DEVELOPMENT_TEAM` and iPhone-reachable backend URL requirements

That action is buried inside the `final_showcase_readiness` step when reading the aggregate refresh report.

## Goal

Add a sanitized top-level `showcase_next_action` field to `final-local-report-refresh.json` so the auto-iteration loop can see the most relevant final iPhone showcase action without changing the report's resource-first blocker semantics.

## Selected Approach

Use an additive field derived from `steps_by_id["final_showcase_readiness"].blocker_hint`.

This keeps:

- `first_blocker`: resource-first final handoff blocker
- `next_action`: action derived from `first_blocker`
- `showcase_next_action`: current final showcase blocker action

Alternatives considered:

1. Replace top-level `next_action` with showcase readiness.
   This would hide the first missing key and break the existing resource handoff contract.
2. Reorder refresh steps so showcase readiness comes first.
   This would distort the refresh pipeline and make provider/key prerequisites harder to see.
3. Add `showcase_next_action`.
   This is selected because it is explicit, backwards compatible, and matches the goal loop's need for the nearest iPhone showcase action.

## Behavior

When the `final_showcase_readiness` step is blocked and has a promoted blocker hint, `showcase_next_action` is:

```json
{
  "id": "final_showcase_readiness",
  "label": "Final showcase readiness",
  "status": "blocked",
  "classification": "ios_deploy_evidence",
  "command": "provide DEVELOPMENT_TEAM in Deployment.local.xcconfig",
  "detail": "iOS deploy runbook and device launch rehearsal must both be ready. | Next device action: provide DEVELOPMENT_TEAM in Deployment.local.xcconfig | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable",
  "source": "final_showcase_readiness",
  "output": "services/backend/.local/final-showcase-readiness.json",
  "step_id": "final_showcase_readiness",
  "validation_command": "make mobile-deploy-preflight"
}
```

If the step is ready, missing, failed, or has no useful action, the field is `null`.

## Safety

This change does not run provider calls, Xcode, signing, Keychain, global environment writes, real quote/order flows, or real iPhone deploy commands. It only changes the shape of the ignored `.local` aggregate report and its tests.

## Verification

Use TDD:

1. Add a failing backend test asserting `showcase_next_action` exists on the blocked refresh fixture and includes the concrete mobile preflight validation command.
2. Implement the smallest helper needed to derive the field from the final showcase step.
3. Run the focused refresh tests, backend lint, backend tests, `git diff --check`, and a safe `make final-local-report-refresh` smoke. The smoke may exit `2` while external resources and iOS device evidence remain missing, but the saved report must include `showcase_next_action`.
