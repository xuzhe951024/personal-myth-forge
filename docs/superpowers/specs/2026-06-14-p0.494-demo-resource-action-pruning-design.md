# P0.494 Demo Resource Action Pruning Design

## Context

`final-demo-launch-local.json` is one of the operator-facing launch reports for the final iPhone handoff. It already exposes concrete resource input actions such as:

- `provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight`
- `provide OPENAI_API_KEY in final-resources.env; rerun make final-resources-preflight`
- `provide PRODUCT_BUNDLE_IDENTIFIER in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`

The same top-level `operator_actions` list still includes the bare validation action `make final-resource-requirements`. That command is useful in `commands` and nested resource reports, but it is low information in the top-level next-action checklist once the exact missing resource actions are already present.

## Options Considered

1. Filter only `final-demo-launch-local.operator_actions`.
   - Pro: focused on the current report.
   - Con: duplicates normalization behavior if configured mode later needs the same cleanup.

2. Filter inside shared `operator_actions.normalize_operator_action()`.
   - Pro: global.
   - Con: too broad; callers may still want `make final-resource-requirements` as a command inventory item.

3. Filter inside `final_demo_launch._dedupe_operator_actions()`.
   - Pro: affects final demo launch top-level checklists for both local and configured modes while keeping command inventories and nested reports intact.
   - Con: requires focused tests to ensure sparse reports can still keep the validation action.

Chosen approach: option 3.

## Behavior

`final_demo_launch._dedupe_operator_actions()` should remove the top-level `make final-resource-requirements` action when the same list contains at least one specific resource input action that reruns `make final-resources-preflight`.

The report should still preserve:

- `make final-resource-requirements` in `commands`
- nested `final_resource_requirements.validation_commands`
- the bare action if no specific `provide ...; rerun make final-resources-preflight` action exists

## Safety

This is report-only. It does not write `.env`, modify iOS deployment config, call Meshy/OpenAI/Treatstock, run Xcode, touch Keychain, or place print orders.

## Verification

Focused tests must prove:

- Local final demo launch top-level `operator_actions` no longer contains `make final-resource-requirements` when specific resource handoff actions are present.
- `_dedupe_operator_actions()` still keeps `make final-resource-requirements` when no specific final resource preflight action exists.
- `commands` and nested resource requirement fields continue to expose the validation target.
- Focused `test_final_demo_launch.py`, related `test_operator_actions.py`, lint, and refreshed local evidence succeed.
