# P0.44 Live Provider Consent Gate Design

## Context

`demo-acceptance --provider-mode configured` and `final-acceptance --provider-mode
configured` are the handoff commands used after backend-only provider keys are
configured. Today, once Meshy/OpenAI keys are present and readiness passes, those
commands can build live providers and make real external calls. That is useful for
the final operator-run validation, but unsafe as the default behavior because it
can spend credits or hit third-party systems during routine local acceptance.

The project already has two relevant safety principles:

- Provider secrets stay on the backend and are never returned to mobile clients.
- Local demo and final acceptance should be runnable without external providers.

P0.44 adds a third principle:

- Real provider calls require an explicit CLI/runtime consent flag.

## Goal

Prevent accidental live Meshy/OpenAI provider calls from demo/final acceptance
when `provider_mode="configured"` and real provider keys are available, unless
the caller explicitly opts in.

## Non-Goals

- Do not change `generate-asset`, `generate-batch`, or `evaluate-3d`; those are
  direct generation commands where provider choice is the explicit action.
- Do not call live Meshy/OpenAI in automated tests.
- Do not move provider keys to mobile or include secret values in reports.
- Do not solve Apple signing, Apple SDK license acceptance, or real iPhone
  validation.

## Behavior

Add `allow_live_provider_calls: bool = False` to:

- `run_demo_acceptance(...)`
- `run_final_acceptance(...)`
- CLI `demo-acceptance --allow-live-provider-calls`
- CLI `final-acceptance --allow-live-provider-calls`

Default behavior:

- `provider_mode="local"` still forces local `three_d_provider` and
  `npc_provider`, so it never requires consent.
- `provider_mode="configured"` inspects selected core providers before building
  providers.
- If configured mode selects live core providers and readiness is otherwise good,
  but `allow_live_provider_calls` is false, `demo-acceptance` exits `2` before
  building providers.
- `final-acceptance` passes the consent value into `demo-acceptance` and classifies
  demo exit `2` as a blocked provider configuration state.

The report should make the block actionable without exposing secrets:

```json
{
  "allow_live_provider_calls": false,
  "live_provider_consent_required": true,
  "live_provider_kinds": ["three_d", "npc"],
  "selected_providers": {
    "three_d": "meshy",
    "npc": "openai",
    "capture_storage": "local"
  },
  "error": "Live provider calls require --allow-live-provider-calls."
}
```

Readiness failures still win over consent failures. If `--require-real-core` is
set and keys are missing, the report remains the existing missing-env block and
no providers are built.

## Provider Scope

For P0.44, the consent gate protects the provider kinds that the demo acceptance
flow can contact directly:

- `three_d`
- `npc`

`capture_storage` remains listed in core readiness, but the current demo
acceptance path does not contact remote capture storage. It can be added to the
live-call gate later if the acceptance flow starts uploading or fetching capture
media through a remote storage provider.

## Tests

Backend acceptance:

- Configured mode with Meshy/OpenAI keys and `require_real_core=True` exits `2`
  before provider builders are called when live calls are not allowed.
- The blocked report is sanitized, includes selected providers, and instructs
  the operator to pass `--allow-live-provider-calls`.
- The same configured settings proceed when `allow_live_provider_calls=True`; the
  test uses monkeypatched local/fake providers so no network is touched.

Final acceptance:

- The top-level final report includes `allow_live_provider_calls`.
- The final runner passes the value through to the demo acceptance runner.
- A configured ready-but-no-consent demo block is classified as blocked provider
  configuration, not a test failure.

CLI:

- `demo-acceptance --allow-live-provider-calls` passes `True` into
  `run_demo_acceptance`.
- `final-acceptance --allow-live-provider-calls` passes `True` into
  `run_final_acceptance`.
- Existing defaults remain false.

Docs:

- Update README/mobile README/resource handoff command text so final live-provider
  validation examples include `--allow-live-provider-calls` when real calls are
  intended.

## Acceptance

- `make backend-lint`
- `make backend-test`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests`
- `swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck`
- `resource-handoff` still exits `2` for missing external resources and does not
  leak secrets or local paths.
- `final-acceptance --profile quick --provider-mode local` still exits `2` only
  for known local iOS deployment blockers.
- A configured provider-mode command with ready Meshy/OpenAI settings but without
  consent exits `2` before any provider builders are called.
