# P0.245 Showcase iOS Preflight Detail Design

## Context

The lower-level iOS device evidence reports now expose concrete blockers:

- `backend_device_server`: `PMF_BACKEND_BASE_URL must be iPhone-reachable`
- `mobile_deploy_preflight`: `Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable`
- `final-handoff-index`: local rehearsal blocker with the same concrete mobile preflight detail

`final-showcase-readiness` still builds the top-level `ios_deployable` blocker from the first device action. That first action is `make backend-device-demo`, so the top-level readiness detail only mentions the backend URL and hides the missing `DEVELOPMENT_TEAM`.

## Goal

Make `final-showcase-readiness.first_blocker.detail` and `next_action.detail` prefer the most complete blocked iOS preflight action detail when available.

## Non-Goals

- Do not change capability IDs, ordering, or status semantics.
- Do not add mobile UI fields.
- Do not run Xcode signing, Keychain, mobile deploy preflight, live providers, or paid provider calls.
- Do not hide the backend action or remove the device action bundle.

## Approaches Considered

1. Prefer the blocked `run_mobile_deploy_preflight` action when it has evidence detail.
   This is selected. It surfaces the operator-relevant config blocker while preserving existing device action bundle order.

2. Concatenate every blocked iOS action detail into the top-level blocker.
   This is too noisy for the main readiness ledger and can duplicate lower-level evidence.

3. Change the device action bundle ordering so mobile preflight comes first.
   This would distort the actual action sequence. The backend server still needs to start before preflight.

## Design

`_first_device_action_hint` will continue returning a single `Next device action` string. It will first look for a blocked `run_mobile_deploy_preflight` action with a non-missing `evidence_detail`. If present, it uses that action's command and detail. Otherwise it falls back to the current first-action behavior.

The resulting top-level iOS blocker should include:

```text
Next device action: make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

The report sanitizer remains unchanged.
