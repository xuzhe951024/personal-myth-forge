P0.487 Final Showcase Self Action Filter Design

Goal

Remove self-referential `make final-showcase-readiness` entries from final showcase readiness `operator_actions` while keeping the command available in the report's `commands` list.

Context

`final-showcase-readiness.json` is the top-level operator handoff for the final iPhone showcase. It currently ends its `operator_actions` with:

`make final-showcase-readiness`

That command only regenerates the same report. It is useful as a reference command in `commands`, but it is not a blocker-clearing operator action. Keeping it in `operator_actions` makes the final handoff list less actionable after the concrete device, provider, print, and Xcode actions have already been listed.

Non-Goals

- Do not remove `make final-showcase-readiness` from the `commands` list.
- Do not change readiness status, capability classification, first blocker, or next action semantics.
- Do not run live providers, Xcode, signing, device checks, Keychain writes, real provider calls, print quotes, or resource writers.
- Do not broaden action normalization beyond this single self-action.

Design

Add a small final-showcase-specific filter for self-referential operator actions. `_operator_actions()` can continue to append the self refresh command as a candidate, but `_filter_showcase_operator_actions()` should drop exact self-action entries before prioritization and limiting.

This preserves the current aggregation pipeline:

1. Gather next action, child report actions, final rehearsal refresh, capability actions, and live provider hints.
2. Dedupe and normalize.
3. Filter non-actionable showcase-only entries.
4. Prioritize provider and print handoffs.
5. Add validation commands and enforce the action limit.

Acceptance Criteria

- `build_final_showcase_readiness_report()` no longer emits `make final-showcase-readiness` in `operator_actions`.
- The same report still includes `make final-showcase-readiness` in `commands`.
- Existing provider, print, deploy writer, final rehearsal, and validation-aware operator action behavior remains unchanged.
- Safe local readiness evidence remains blocked on external iOS/provider resources and does not run external actions.

Safety

This phase only changes report text selection. Verification may write ignored `.local` JSON reports through safe Make wrappers. It must not touch real provider keys, launch Xcode signing, mutate global Mac state, call live providers, or place print orders.
