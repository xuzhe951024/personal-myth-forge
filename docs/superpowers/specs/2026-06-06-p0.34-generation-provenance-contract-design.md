# P0.34 Generation Provenance Contract Design

## Context

P0.33 routes Meshy guided scans with multiple prepared source images through
Multi-Image to 3D, but the public `GeneratedAsset` response still only exposes
provider, format, URI, prompt, moderation status, and variants. During final
real-device/key acceptance, that is not enough to prove which generation route
was used or whether the iPhone guided scan materially affected the generated
asset.

The next useful contract is safe generation provenance: counts, selected mode,
and provider route, without raw images, data URIs, local paths, or secrets.

## Goal

Add optional `generation_provenance` metadata to generated assets, decode it in
the iOS core, render it in the artifact summary, and include it in acceptance
reports so operators can verify scan-to-3D behavior without seeing private
media.

## Non-Goals

- No live Meshy/OpenAI calls.
- No API key creation or storage.
- No global environment mutation.
- No raw media URLs, data URIs, filesystem paths, or capture media bytes in
  public API responses.
- No change to generated asset download/cache behavior.
- No print fulfillment or print repair changes.

## Proposed Contract

Add a backend model:

```python
class GeneratedAssetProvenance(BaseModel):
    input_mode: Literal["unknown", "text_prompt", "single_image", "multi_image"]
    provider_route: str | None = None
    source_image_count: int = Field(default=0, ge=0)
    selected_source_image_count: int = Field(default=0, ge=0)
    source_asset_count: int = Field(default=0, ge=0)
    max_source_images: int | None = Field(default=None, ge=1)
    selection_reason: str
    raw_sources_included: bool = False
```

Add to `GeneratedAsset`:

```python
generation_provenance: GeneratedAssetProvenance | None = None
```

`generation_provenance` is optional so older fixtures and stored sessions remain
decodable. New providers should populate it.

## Provider Behavior

`LocalThreeDProvider`

- `text_prompt` when no source images are present
- `single_image` when one source image is present
- `multi_image` when two or more source images are present
- `provider_route="local_stub"`
- selected count equals source image count
- raw sources are never included

`MeshyThreeDProvider`

- `text_prompt` for text-to-3D preview/refine
- `single_image` for image-to-3D
- `multi_image` for multi-image-to-3D
- `provider_route` is the Meshy API path used:
  - `/openapi/v2/text-to-3d`
  - `/openapi/v1/image-to-3d`
  - `/openapi/v1/multi-image-to-3d`
- `source_image_count` is the number of prepared provider-supported JPEG/PNG
  images available to Meshy
- `selected_source_image_count` is the number actually sent
- `max_source_images=4` for multi-image mode
- `selection_reason` explains filtering/capping in safe terms only

## Mobile Behavior

Swift adds:

```swift
public struct GeneratedAssetProvenance: Codable, Equatable, Sendable
```

`GeneratedAsset.generationProvenance` decodes as optional. Older responses
without the field remain valid.

`ArtifactSummaryView` or `Artifact3DPreviewView` renders compact provenance:

- mode
- selected image count versus available image count
- provider route
- raw sources included: no

The UI must not show long raw provider payloads, local file paths, data URIs, or
secrets.

## Acceptance Reports

`demo-acceptance` should add generated asset provenance to the session summary.
`final-acceptance` already nests the demo report and sanitizes recursively; no
new sanitizer is needed beyond tests proving data URI/path absence.

Example report fragment:

```json
{
  "generated_asset_provider": "local_stub",
  "generated_asset_format": "glb",
  "generation_provenance": {
    "input_mode": "multi_image",
    "provider_route": "local_stub",
    "source_image_count": 3,
    "selected_source_image_count": 3,
    "source_asset_count": 0,
    "max_source_images": null,
    "selection_reason": "Local provider records all prepared source images.",
    "raw_sources_included": false
  }
}
```

## Error Handling

- If a provider cannot determine provenance, it may set
  `input_mode="unknown"` with `raw_sources_included=false`.
- Counts must never be negative.
- Pydantic and Swift decoders must tolerate missing provenance.
- API error sanitization remains unchanged.

## Testing

Backend tests will cover:

- local provider provenance for text, single-image, and multi-image requests
- Meshy provenance for text, image, and multi-image routes
- Meshy multi-image selected count capped at four
- API response includes provenance without data URI or local path leakage
- demo acceptance report includes provenance

Swift tests will cover:

- decoding `generation_provenance`
- older fixture without provenance still decodes
- app source compile with provenance UI

Regression will run backend lint/test, SwiftPM project checks, Swift core
contract tests, app compile check, final acceptance quick, known local iOS
deploy/Xcode gates, screenshot evidence, and `git diff --check`.

## Acceptance Criteria

- New generated assets include safe provenance metadata.
- iOS decodes and displays provenance while remaining compatible with older
  responses.
- Acceptance reports surface provenance.
- No report or API response exposes raw media, data URI, local path, or secret.
- Existing Meshy multi-image routing, provider readiness, final acceptance, NPC
  agent, and mobile compile behavior remain intact.
