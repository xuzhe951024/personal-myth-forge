# P0.353 Final Demo Launch iOS Blocker Command Design

## Goal

Make `final-demo-launch-local` expose the concrete iOS deploy handoff command on its top-level `first_blocker` when saved mobile preflight evidence already provides that action.

## Context

The final showcase readiness report now promotes the iOS deploy config handoff onto both `first_blocker` and `next_action`. The local final demo launch report still keeps `first_blocker.command` at the broader `make mobile-deploy-preflight` phase command, while `next_action.command` already points to the concrete handoff such as:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto
```

The final iPhone demo operator path is easier to follow when the top-level blocker and next action agree on the actual next input, with `make mobile-deploy-preflight` retained as the validation command.

## Design

When `final-demo-launch-local` selects the `mobile_deploy_preflight` phase as its first blocker and the nested final showcase readiness report has a concrete `next_action`, the blocker should copy:

- `command`
- `validation_command`

from that concrete action.

The blocker should keep its existing label, id, status, classification, source, source id, and detail. Its detail already includes saved preflight evidence, so this phase only fixes the command contract. If no concrete final showcase action exists, the blocker remains unchanged and continues to use `make mobile-deploy-preflight`.

`next_action` remains derived from `first_blocker`, so after this change it mirrors the concrete command without needing a second projection path.

## Safety

This is a read-only report contract change. It does not write `.env`, write deploy config, call Meshy/OpenAI/Treatstock, run Xcode, sign a build, touch Keychain, or place any print order. Verification may refresh ignored `.local` JSON evidence and should still report blocked until real local resources are provided.
