# P0.506 External Ledger Backend Priority Design

## Context

The final iOS showcase remains blocked by local device evidence, not by repo code generation. Recent phases moved the local backend device demo handoff near the top of `final-showcase-readiness`, `final-demo-launch-local`, and `final-local-report-refresh`.

`final-external-action-ledger` still emits the backend-device-demo handoff after provider keys, live provider actions, print quote consent, and Xcode actions. That makes the external handoff less executable for the current first blocker because the operator should start the LAN backend before rerunning physical iPhone checks.

## Decision

Promote the canonical backend demo action in `final-external-action-ledger.operator_actions` so it appears directly after the promoted iOS deploy writer action and before provider/key/print handoffs:

1. `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight`
2. `start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight`
3. provider/key/print/global follow-up actions

This keeps the action groups unchanged. The ledger still records resource inputs, safe local writes, live provider costs, global machine actions, and device runtime actions in their existing structured sections. Only the flat operator handoff list gets a final priority pass.

## Alternatives Considered

1. Reorder `action_groups` so device runtime actions come before resources. This would make the flat list correct, but it would change report semantics and risk breaking consumers that rely on the existing group order.
2. Special-case the backend action in `_device_runtime_actions`. This would not fix the flat list because group flattening still happens after resource/provider/global sections.
3. Add a small operator action priority pass. This is the narrowest change and matches the recent final demo/local refresh fixes.

## Behavior

- If the canonical backend-device-demo validated action is present and the deploy writer action is present, backend-device-demo is inserted immediately after the deploy writer.
- If the deploy writer action is absent, backend-device-demo keeps its existing relative order.
- Duplicate backend-device-demo actions are removed through existing dedupe behavior.
- Provider handoff, print quote, Xcode license, and iPhone-reachable backend URL normalization remain unchanged.
- No shell command, provider call, global machine mutation, signing action, or key read/write is introduced.

## Tests

Add a regression test in `services/backend/tests/test_final_external_action_ledger.py` that builds a ledger from an empty local repo and asserts:

- deploy writer remains the first operator action,
- backend-device-demo is present,
- backend-device-demo appears before `provide MESHY_API_KEY`, `provide OPENAI_API_KEY`, provider handoff, guarded print quote, and Xcode license actions,
- backend-device-demo is directly after deploy writer.

The existing focused external ledger/operator action suite remains the primary regression surface for this phase.

## Self-Review

- Scope is limited to one report's flat operator action order.
- No provider, print, Xcode, Keychain, signing, or global-state action is executed.
- The design preserves current structured report groups and safety fields.
- Requirements are concrete and testable with local deterministic report generation.
