# P0.329 Final Configured Preflight Next Action Design

## Context

`final_configured_preflight_report` is the Mac-side configured launch packet. It
aggregates final resources, provider readiness, resource handoff, configured
final launch, and configured iOS deploy runbook evidence before any live
provider or device acceptance run.

The child reports already expose concrete next actions, but configured preflight
only exposes a long `operator_actions` list at the top level. That weakens the
final handoff path because parent reports and operators must infer the first
configured blocker instead of reading a stable field.

## Decision

Add top-level `first_blocker` and `next_action` to
`final_configured_preflight_report`.

The component priority is unchanged and follows the existing status gate:

1. `final_resources_preflight`
2. `provider_handoff`
3. `resource_handoff`
4. `configured_final_launch`
5. `configured_ios_deploy_runbook`

For each blocked or missing component, the wrapper blocker includes:

- component id, label, status, and classification;
- command to run or provide the immediate missing value;
- detail from the child blocker/action when available;
- source report kind and source id for traceability;
- validation command, preferring the child validation command.

`next_action` mirrors `first_blocker` with `source: "first_blocker"`. Ready
configured preflight reports return `null` for both fields.

## Safety

This is read-only report shaping. It does not:

- read or write real provider keys;
- call Meshy, OpenAI, Treatstock, or print ordering APIs;
- write `.env`, xcconfig, Keychain, Xcode signing, or global machine state.

## Acceptance

- Focused backend tests prove blocked configured preflight exposes a concrete
  `first_blocker`/`next_action` sourced from final resources.
- Ready configured preflight returns `null` for both fields.
- The CLI/Make report still exits `2` locally while missing resources, but the
  ignored `.local/final-configured-preflight.json` contains the new action
  fields.
