# P0.321 Final Resource Item Actions Design

## Context

The final resources preflight report now exposes top-level `first_blocker` and `next_action`, but individual `items[]` still omit actionable handoff fields. The current ignored local report proves this gap: `MESHY_API_KEY` is top-level actionable, while the item list has `command` and `validation_command` as absent/null when inspected.

This keeps the iPhone Final Launch resource checklist less useful than the Mac-side operator actions because each missing or blocked resource row cannot explain exactly what to do and how to validate it.

## Decision

Add per-item action metadata to the final resources preflight report for resource items that require operator attention:

- required missing values get `command: "provide <ID> in final-resources.env"`
- blocked backend URL values get `command: "set PMF_BACKEND_BASE_URL to an iPhone-reachable LAN URL"`
- blocked bundle id values get `command: "set PRODUCT_BUNDLE_IDENTIFIER to a unique app bundle id"`
- blocked provider/mode values get their existing canonical fix commands
- every actionable item gets `detail` and `validation_command: "make final-resources-preflight"`
- ready and optional items do not get action fields unless they already carry apply-time handoff notes

The top-level `first_blocker` and `next_action` continue to use the same action derivation, so this is a contract enrichment rather than a behavior fork.

## iOS Handoff

`FinalResourcesPreflightItem` will decode optional `command`, `detail`, and `validation_command` fields. `FinalLaunchMobileSummary` will append these values to resource checklist rows for missing or blocked items so the iPhone launch panel can show concrete handoff instructions per resource slot.

The UI summary must remain sanitized and must not expose raw secret values, local absolute paths, or provider credentials.

## Non-Goals

- Do not write `.env` files, iOS deploy config, Keychain values, or global settings.
- Do not call Meshy, OpenAI, Treatstock, Xcode signing, or any paid provider.
- Do not change the required resource list or the external blocker order.
- Do not mark final resources as ready without user-provided values.

## Acceptance

- Backend tests prove missing and blocked final resource items carry `command`, `detail`, and `validation_command`.
- Backend tests prove ready/optional items do not gain unnecessary command fields.
- Swift contract tests prove iOS decodes the new fields and shows them in resource checklist rows.
- `make final-resources-preflight` still exits with code 2 in the current no-key local state and writes per-item action fields without leaking secrets.
