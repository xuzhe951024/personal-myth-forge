# P0.16 3D Asset Variant Contract Design

## Goal

Make the generated 3D asset contract explicit enough for iOS to prefer a
SceneKit-loadable asset variant while preserving the backend-owned provider key
boundary. The first target is a primary GLB game asset plus an optional USDZ
iOS scene asset variant.

## Context

P0.15 taught the iOS app to download and cache backend-approved asset URIs. It
also kept the GLB path honest: GLB can be cached, but SceneKit still needs a
conversion/import path before it can be shown as the real generated relic.

Meshy currently supports `usdz` as a requested output format for Text to 3D,
Image to 3D, and Multi-image to 3D tasks, and exposes generated file URLs in
`model_urls`. Its Convert API also supports `usdz` output. That makes a small
contract iteration worthwhile before runtime conversion work: the backend should
be able to return multiple asset variants, and iOS should choose the scene-ready
variant when one is present.

## Product Behavior

The backend still returns `generated_asset` as the backward-compatible primary
asset:

- `format`: usually `glb`
- `uri`: the primary game asset URI
- `prompt`: the generation prompt
- `moderation_status`: provider or local review status

It now also returns `generated_asset.variants`, a list of concrete alternate
files derived from the same generation task. Each variant declares:

- `role`: the consumer role, starting with `game_asset` and `ios_scene_asset`
- `format`: the file format, such as `glb` or `usdz`
- `uri`: the downloadable or local URI
- `is_scene_loadable`: whether the mobile preview can hand it directly to
  SceneKit

The iOS core model accepts missing `variants` as an empty list so older backend
responses still decode. When variants are present, the asset preparer selects
the first `ios_scene_asset` with `is_scene_loadable == true` before falling back
to the primary generated asset.

## Backend Design

Add a `GeneratedAssetVariant` Pydantic model and attach
`variants: list[GeneratedAssetVariant]` to `GeneratedAsset`.

`LocalThreeDProvider` returns:

- primary GLB asset: `local://generated-assets/{session_id}.glb`
- iOS scene variant: `local://generated-assets/{session_id}.usdz`

`MeshyThreeDProvider` requests `glb` and `usdz` target formats for text and
image generation. When Meshy returns `model_urls.glb` and `model_urls.usdz`, the
primary asset remains the GLB and the USDZ becomes the `ios_scene_asset` variant.
If USDZ is absent, the backend still returns the GLB-only asset rather than
failing a successful generation.

## Mobile Design

Add `GeneratedAssetVariant` to the Swift core models and make
`GeneratedAsset.variants` default to `[]` during decoding.

Update `ArtifactAssetPreparer` to prepare a scene-preferred reference:

1. First use a scene-loadable `ios_scene_asset` variant when available.
2. Otherwise use the primary `generated_asset`.

This keeps all provider credentials server-side. Mobile only receives reviewed
asset URIs and format metadata.

## Out Of Scope

- Unity village import.
- Real iOS simulator or physical-device deployment.
- Runtime GLB import.
- Backend GLB-to-USDZ conversion for providers that do not return USDZ.
- Print asset repair or third-party print quote flow.
- Meshy API key creation or secret management.

## Testing And Evidence

- Backend tests cover `GeneratedAsset.variants`, local provider variants, Meshy
  `target_formats`, and extraction of `model_urls.usdz`.
- Swift core contract tests cover decoding responses with variants, decoding
  old responses without variants, and preferring the USDZ variant during asset
  preparation.
- Swift project checks require the new model and scene variant selection path.
- Static Browser evidence shows a 390x844 iOS-first handoff screen where GLB is
  the primary asset and USDZ is the selected SceneKit variant.
