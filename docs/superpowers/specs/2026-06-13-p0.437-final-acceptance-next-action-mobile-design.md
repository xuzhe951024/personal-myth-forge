# P0.437 Final Acceptance Next Action Mobile Design

## Context

After refreshing safe local evidence for the current head, local visual
regression is ready and print fulfillment is partial only because configured
Treatstock quote evidence requires explicit cost consent. The remaining local
functional blocker is iOS deploy validation:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
```

Backend final demo launch already embeds this copy-pasteable action under
`final_acceptance_readiness.blockers[].next_action`, but the iOS model only
decodes the blocker command/detail fields. The Final Launch acceptance section
therefore shows the generic validation command while dropping the safer saved
writer-plus-rerun handoff.

## Decision

Add optional `nextAction` support to the mobile `FinalAcceptanceBlocker` model
and render it in Final Launch acceptance rows. The primary blocker command stays
unchanged; the next action is appended as a separate `next ...` segment when
present.

## Scope

Included:

- Decode `final_acceptance_readiness.blockers[].next_action` into iOS models.
- Display the nested next action in blocked Final Acceptance rows.
- Add Swift contract coverage for decoding and summary text.
- Update the roadmap.

Not included:

- No backend provider calls, real Meshy/OpenAI/Treatstock evidence, Xcode
  signing, Keychain writes, or global Apple SDK license actions.
- No change to backend blocker ordering or command generation.
- No new backend behavior unless verification reveals the report is not already
  emitting `next_action`.
