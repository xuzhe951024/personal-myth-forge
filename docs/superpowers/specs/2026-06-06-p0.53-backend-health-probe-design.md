# P0.53 Backend Health Probe Design

## Product Intent

P0.52 shows whether the configured backend URL looks suitable for an iPhone
demo, but it still cannot prove from inside the mobile app that the backend is
actually reachable. P0.53 adds an operator-triggered backend health probe to the
iOS showcase surface so the final demo can fail early on LAN/firewall/server
issues before capture, 3D generation, NPC autonomy, or print quote actions.

This moves the app closer to a physical iPhone deployment without requiring
local signing values, Apple SDK license acceptance, simulator/device execution,
or provider keys.

## UX

The existing `Device Preflight` panel remains the home for device readiness.
It gains a compact `Check` button in the header. The button calls the configured
`PMFBackendBaseURL` `/health` endpoint once per tap and never runs in a
background loop.

The backend row states become:

- `blocked` when the configured host is loopback (`127.0.0.1`, `localhost`,
  or `::1`), because a physical iPhone cannot reach it.
- `waiting` when the URL is not loopback but the operator has not checked
  backend health yet.
- `waiting` while the manual check is in flight.
- `ready` when `/health` returns `{"status":"ok"}`.
- `blocked` when the request fails, the HTTP response is not 2xx, or the
  response does not decode to `status=ok`.

The panel continues to show provider readiness, local demo readiness, and saved
NPC history exactly as P0.52 does. Provider keys, checkout links, raw media, and
local paths must not appear in the health probe text.

## Architecture

Add pure Swift mobile-core types:

- `BackendHealthResponse`: decoded from backend `/health`.
- `BackendHealthProbeStatus`: `notChecked`, `checking`, `reachable`,
  `unreachable`.
- `BackendHealthProbe`: status plus sanitized detail.

`PersonalMythForgeAPIClient` gains `getBackendHealth()`, which sends a `GET`
request to `/health` and decodes `BackendHealthResponse`. Existing transport
injection makes this contract testable without network access.

`DevicePreflightSummaryBuilder.build(...)` gains a `backendHealthProbe`
argument. The backend URL row uses both URL shape and health probe status.
Loopback remains blocked even if a Mac/simulator can reach it, because P0.53 is
about physical iPhone readiness.

`ForgeRootView` owns:

- `@State private var backendHealthProbe: BackendHealthProbe?`
- `@State private var isCheckingBackendHealth = false`

It passes those values into `DevicePreflightSummaryBuilder` and gives
`DevicePreflightView` a `checkBackend` action. The action calls
`apiClient.getBackendHealth()`, maps success to `.reachable`, and maps all
failures to `.unreachable` with a short sanitized message.

## Testing

Swift core contract tests prove:

- `getBackendHealth()` builds a `GET /health` request.
- `getBackendHealth()` decodes `status=ok`.
- non-ok health status becomes a preflight blocked backend row.
- unchecked non-loopback URL becomes a waiting backend row.
- checking state becomes a waiting backend row.
- reachable health makes a non-loopback backend row ready.
- unreachable health redacts unsafe error details.
- loopback URL remains blocked even with reachable health.

Project checks prove:

- `ForgeRootView` wires `getBackendHealth`, `backendHealthProbe`,
  `isCheckingBackendHealth`, and `checkBackendHealth`.
- `DevicePreflightView` exposes the `Check` action.
- `DevicePreflight.swift` contains the backend health probe types.

Backend source acceptance adds `backend_health_probe` to the iOS showcase gate,
raising the internal source summary from `13/13` to `14/14`.

Visual evidence is a static 390x844 page showing the preflight panel with the
new health check button and backend row states. It is source evidence, not a
simulator or device screenshot.

## Non-Goals

- No new backend endpoint; reuse existing `/health`.
- No automatic polling or retry loop.
- No simulator/device execution.
- No Xcode global developer directory, signing, or license mutation.
- No provider API calls or provider key creation.
- No network call during repository acceptance tests.
