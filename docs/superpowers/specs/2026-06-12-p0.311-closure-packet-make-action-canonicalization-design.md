# P0.311 Closure Packet Make Action Canonicalization Design

## Goal

Normalize final launch closure packet top-level make actions so operators see canonical make targets instead of `run make ...` wrappers.

## Context

The final iOS showcase is still blocked by external provider keys, iOS deployment values, Xcode/device evidence, and live provider consent. The closure packet is the final operator-facing bundle that ties those prerequisites together. Its structured section/action rows already carry concrete `command` values, but the top-level `operator_actions` currently wraps ordinary make targets as `run {command}`.

That creates residual actions such as:

- `run make final-resources-preflight`
- `run make print-fulfillment-readiness`
- `run make ios-device-launch-rehearsal`

The rest of the final handoff flow has been moving toward bare make targets for operator actions. The closure packet should follow that convention while preserving explicit wording for cost consent and global/manual confirmation actions.

## Approach

Add a small helper inside `final_launch_closure_packet.py`:

- if the action command starts with `make `, emit the command unchanged.
- otherwise emit `run {command}` as before.

Keep existing behavior for:

- `provide_*` actions, which still include destination plus validation handoff.
- `requires_cost_consent`, which still says to approve live provider cost first.
- `requires_user_confirmation`, which still says to confirm global/manual action first.

## Testing

Use TDD in `tests/test_final_launch_closure_packet.py`:

- Missing-resource closure packets should include bare make targets such as `make final-resources-preflight`, `make print-fulfillment-readiness`, and `make ios-device-launch-rehearsal`.
- The same top-level operator action list should not contain values beginning with `run make `.
- Existing provide action validation and destination behavior remains unchanged.

Refresh the ignored `services/backend/.local/final-launch-closure-packet.json` report and verify it remains blocked while top-level make actions are canonical.

## Non-Goals

- Do not call real providers or live acceptance commands.
- Do not run Xcode, write signing state, access Keychain, or place print orders.
- Do not change section/action command fields, readiness classification, or blocker ordering.
