# P0.121 Showcase Visual Regression Index Design

## Context

P0.120 added a live provider evidence manifest and surfaced it through the final launch packet and iPhone final launch panel. The project now has broad source, backend, SwiftPM, rehearsal, provider-readiness, and handoff gates. The remaining visual regression gap is that `make visual-regression` still checks only one static iPhone artifact: the P0.118 SceneKit load proof.

That is too narrow for the current showcase. The iPhone demo now spans object capture, guided scanning, 3D generation receipts, SceneKit load states, NPC Agent ticks, print handoff, live provider consent, and final launch readiness. Those visual artifacts already exist as checked-in static HTML plus 390x844 PNG evidence, but the default visual regression report does not protect them.

## Goal

Make `make visual-regression` validate a curated full-showcase set of static iPhone evidence artifacts so the final handoff has one no-key visual gate covering the core mobile demo path.

## Options Considered

### Option A: Keep A Single Artifact And Rely On Source Acceptance

This is the lowest change, but it leaves visual regressions in most of the mobile showcase invisible to the final handoff. Source acceptance proves strings and files exist; it does not prove the rendered 390x844 evidence artifacts are present, correctly sized, and free of unsafe text.

### Option B: Curate A Fixed Default Showcase Artifact Set

This adds a fixed list of existing static artifacts to `DEFAULT_VISUAL_ARTIFACTS`. The current checker already validates HTML presence, PNG presence, dimensions, required text, notes, and unsafe text. Extending the spec list is simple, deterministic, no-key, and aligned with the project's existing static evidence pattern. This is the recommended option.

### Option C: Discover Every Verification Artifact Automatically

This catches more files, but it makes the default gate noisy and unstable. Many older artifacts are historical evidence for narrow implementation steps rather than current final showcase surfaces. Auto-discovery would also make required-text expectations weak because each file needs artifact-specific assertions.

## Chosen Design

Implement Option B.

`DEFAULT_VISUAL_ARTIFACTS` will become a curated sequence of ten final-showcase artifacts:

- `p0.12_ios_device_media_input`: scan/photo/file input bridge.
- `p0.19_guided_scan_entry`: Object Capture guided scan entry.
- `p0.98_capture_generation_receipt`: capture-to-3D receipt.
- `p0.103_generation_result_receipt`: post-forge 3D/NPC/print proof.
- `p0.118_scene_load_proof`: SceneKit load/conversion/failure proof.
- `p0.82_npc_agent_tick_summary`: AI Agent NPC tick and world-resolution proof.
- `p0.101_print_fulfillment_receipt`: print handoff approval proof.
- `p0.100_live_provider_consent`: live provider consent proof.
- `p0.112_ios_device_launch_rehearsal`: safe iPhone launch rehearsal proof.
- `p0.119_visual_regression_handoff`: final launch visual evidence handoff proof.

Each artifact uses the existing `VisualArtifactSpec` fields:

- HTML path under `docs/superpowers/verification/`.
- PNG path under `docs/superpowers/verification/assets/`.
- dimensions `390x844`.
- a small, artifact-specific set of required text tokens.
- the matching regression note path when it exists.

No screenshot generation is added in this iteration. P0.121 only indexes and validates existing checked-in evidence.

## Data Flow

`make visual-regression` already calls:

```bash
cd services/backend && uv run python -m myth_forge_api.cli visual-regression --repo-root ../..
```

The CLI calls `check_visual_artifacts()` with `DEFAULT_VISUAL_ARTIFACTS`. After P0.121, the returned report should contain ten artifacts and a summary of `{"passed": 10, "failed": 0}` on the current checkout.

`make visual-regression-local` writes the same expanded report to:

```text
services/backend/.local/visual-regression-local.json
```

Downstream readiness reports continue to read that saved JSON. The report remains sanitized and path-relative.

## Safety

The visual regression index is read-only:

- no live provider calls
- no provider keys
- no raw capture media
- no backend or iOS config writes
- no Xcode, signing, simulator, or device install
- no global machine mutation

The existing unsafe-text matcher continues to block provider secrets, bearer tokens, data URIs, local capture URIs, absolute user paths, file URLs, and payment/checkout links in HTML or notes.

## Testing

Backend tests will cover:

- the default visual artifact list has ten curated artifacts in the intended order.
- every default artifact uses 390x844 dimensions.
- key required text tokens cover scan input, guided scan, capture-to-3D, 3D generation, SceneKit load proof, NPC Agent, print fulfillment, live provider consent, iOS launch rehearsal, and final launch visual regression.
- the default checker passes against the checked-in evidence files.
- the CLI writes a report with ten passed artifacts for a complete fixture.

Existing tests continue to cover:

- PNG dimension parsing.
- missing PNG failure.
- wrong-dimension failure.
- missing required text failure.
- unsafe HTML/notes redaction.

## Documentation

`README.md` will document that `make visual-regression` now covers the full-showcase visual index rather than only P0.118. `docs/iteration-roadmap.md` will add P0.121.

## Out Of Scope

- Rendering new screenshots.
- Adding Playwright or browser automation.
- Running an iOS simulator or installing on a device.
- Running Xcode, signing, or accepting Apple SDK licenses.
- Calling Meshy, OpenAI, Treatstock, or any other live provider.
- Replacing source acceptance or SwiftPM compile checks.

## Acceptance

P0.121 is accepted when:

- `make visual-regression` reports ten passed artifacts.
- `make visual-regression-local` writes a ten-artifact report.
- visual regression readiness consumes the expanded saved report without changing its safety semantics.
- backend tests, backend lint, SwiftPM contract tests, SwiftPM app compile check, final rehearsal, and iOS device launch rehearsal still pass.
