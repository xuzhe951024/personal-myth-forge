# P0.351 Closure Packet Concrete Resource Action Design

## Goal

Make the final launch closure packet's top-level `first_blocker` and `next_action` tell the operator the concrete resource input to provide, starting with `MESHY_API_KEY`, while preserving the existing validation command that proves the input was accepted.

## Context

`final-external-action-ledger` already reports the first missing resource as `provide MESHY_API_KEY in final-resources.env` with `make final-resources-preflight` as the validation command. The closure packet still surfaces `make final-resources-preflight` as the top-level blocker command, even though its `operator_actions` include the more actionable `provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight` instruction.

This creates a small handoff mismatch at the final goal boundary. The final iOS showcase remains blocked by real keys and local device evidence, so the top-level closure packet should identify the next human input instead of only naming the command that will fail until that input exists.

## Design

For resource actions with all of these properties:

- `id` starts with `provide_`
- `requires_user_input` is true
- `status` is `missing` or `blocked`

the closure packet top-level `first_blocker.command` should be the concrete input command:

```text
provide MESHY_API_KEY in final-resources.env
```

The underlying section action keeps its existing command, such as `make final-resources-preflight`, because that command remains the validation and refresh mechanism for the resource section. The top-level blocker adds `validation_command` with that command so an operator or mobile-visible summary can show both the input and the verification step without guessing.

`next_action` continues to be derived from `first_blocker` and mirrors the same concrete `command`, `validation_command`, `section_id`, and `action_id`. `operator_actions` remains unchanged and still includes the combined instruction with the rerun step.

## Safety

This phase does not write secrets, call Meshy/OpenAI/Treatstock, run Xcode signing, mutate Keychain, or perform paid actions. The only local writer used during verification is the existing ignored report refresh target, and it is expected to remain blocked when real resources are absent.
