# P0.63 Final Demo Launch Profile Design

## Product Intent

The project now has the pieces needed for the final mobile-first showcase:
capture, Meshy 3D generation, OpenAI NPC agent ticks, Treatstock print quote
handoff, iOS deployment config writers, and final acceptance gates. The remaining
operator gap is orchestration. When the user provides API keys and iOS values at
the end, there should be one report that says which local files to write, which
commands to run, which steps may spend provider credits, and what still blocks an
iPhone deployment.

P0.63 adds a final demo launch profile report. It is not a daemon and does not
start Xcode, call providers, change signing state, write secrets, or mutate
global machine configuration. It turns the existing resource handoff and final
acceptance commands into a single, source-tested launch lane.

## Chosen Approach

Add a backend CLI command:

```bash
cd services/backend
uv run python -m myth_forge_api.cli final-demo-launch \
  --mode configured \
  --repo-root ../.. \
  --output /tmp/personal-myth-forge-final-demo-launch.json
```

Modes:

- `local`: no-key dry-run lane for local deterministic providers.
- `configured`: final resource lane for Meshy, OpenAI, optional Treatstock, LAN
  iPhone backend URL, signing values, and final acceptance commands.

The command builds a static JSON report from current settings and repo-local
config files. It reuses `build_resource_handoff_report()` instead of duplicating
resource parsing. The report exits `0` when the selected mode has no missing or
blocked resources; it exits `2` when the selected mode is blocked by missing
provider keys, loopback iOS backend URL, missing signing values, or other local
resource gaps.

## Report Shape

Top-level fields:

- `kind: "final_demo_launch_report"`
- `mode: "local" | "configured"`
- `overall_status: "ready" | "blocked" | "partial"`
- `summary`: counts by `ready`, `missing`, `blocked`, `manual`, `optional`
- `resource_report`: sanitized nested resource handoff report
- `launch_phases`: ordered phase list for resource writing, backend serving,
  readiness, acceptance, iOS preflight, and Xcode build gate
- `operator_checklist`: concise ordered checklist for the user
- `commands`: copyable commands for local/configured launch
- `live_call_policy`: explicit statement that live calls require the configured
  acceptance command and `--allow-live-provider-calls`
- `safety`: secret/path/global-mutation flags

Each `launch_phases` item includes:

- `id`
- `label`
- `status`
- `required_for`
- `command`
- `notes`

## Resource Rules

Local mode:

- Does not require Meshy/OpenAI/Treatstock keys.
- Shows backend/iOS setup commands but treats provider-key gaps as optional.
- Still surfaces iOS deploy blockers because a physical phone demo needs a LAN
  backend URL and signing values.

Configured mode:

- Requires `THREE_D_PROVIDER=meshy` and `MESHY_API_KEY`.
- Requires `NPC_PROVIDER=openai` and `OPENAI_API_KEY`.
- Treatstock is real-ready when `PRINT_PROVIDER=treatstock` and
  `TREATSTOCK_API_KEY` are configured, but the launch profile keeps print order
  placement out of scope.
- Requires project-local iOS deploy config with non-loopback
  `PMF_BACKEND_BASE_URL`, `DEVELOPMENT_TEAM`, and `PRODUCT_BUNDLE_IDENTIFIER`.
- Keeps Apple SDK license as a manual external gate and never runs license
  acceptance.

## Safety

The launch profile may name environment variables, local destination filenames,
provider names, and commands. It must not include:

- API key values
- bearer tokens
- raw media or base64 data
- local absolute paths
- payment or checkout URLs

It must not execute:

- Meshy, OpenAI, Treatstock, or Sculpteo calls
- Xcode build commands
- Apple license commands
- `sudo`, `xcode-select`, `security`, `codesign`, or keychain operations

## Testing

Backend tests cover:

- `final-demo-launch --mode local` returns a report with local commands and no
  live provider calls by default.
- `final-demo-launch --mode configured` blocks when Meshy/OpenAI/iOS resources
  are missing.
- Configured report marks real provider resources ready when settings and
  project-local iOS deploy config are present.
- Treatstock wording is updated to reflect the implemented quote adapter.
- Reports never leak supplied secret values or local absolute paths.
- Resource template acceptance recognizes the final demo launch command/Make
  target as part of the final handoff surface.

Regression remains:

- `make backend-lint`
- `make backend-test`
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks`
- quick final acceptance, expected blocked only by local iOS deploy/Xcode
  environment where applicable

## Non-Goals

- Do not start backend servers.
- Do not call live providers.
- Do not run Xcode or deploy to a device.
- Do not repair or order print assets.
- Do not write `.env` or `Deployment.local.xcconfig`; existing writer scripts own
  those operations.
