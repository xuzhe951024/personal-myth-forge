# P0.173 Closure Configured Bundle Section Design

## Context

P0.171 made `final_showcase_readiness` depend on the saved
`services/backend/.local/configured-live-evidence-bundle.json` report. P0.172
made the safe local refresh command generate that saved bundle before final
showcase readiness.

The final launch closure packet still has only a broad `live_provider_consent`
section and a final acceptance section. It does not expose the configured live
evidence bundle as a first-class closure section, and its command list does not
mention `make configured-live-evidence-bundle`. That weakens the final handoff:
an operator can read the closure packet without seeing the exact bundle gate that
final showcase readiness now requires.

## Goal

Add a required `configured_evidence_bundle` section to the final launch closure
packet so the last operator packet explicitly proves or blocks the configured
Meshy/OpenAI/live-evidence bundle before final acceptance.

## Design

`final_launch_closure_packet.py` will load the saved
`services/backend/.local/configured-live-evidence-bundle.json` report directly.
If the file is missing, unreadable, malformed, or has the wrong kind, the closure
packet will synthesize a blocked/missing report stub with command
`make configured-live-evidence-bundle`.

The new section will sit after `live_provider_consent` and before
`final_acceptance`. It will contain a single required action:
`configured_live_evidence_bundle`. The action status comes from the saved bundle
status, its command comes from `current_blocker.command` when available or
`make configured-live-evidence-bundle` otherwise, and its detail comes from the
bundle blocker, operator actions, or a stable fallback.

The section is required because final showcase readiness now gates provider/key
handoff on the configured bundle. This means the closure packet can only become
ready when the configured bundle is ready.

The packet remains read-only. It does not run the configured bundle builder,
write `.local` files, call Meshy/OpenAI/Treatstock, apply env or iOS config
files, run Xcode, touch signing/keychain state, or mutate global state.

## Alternatives Considered

1. Read the saved bundle report and add a required closure section.
   This is selected because it matches the P0.171 evidence boundary and gives the
   final operator packet explicit visibility into the gate.

2. Derive the bundle state only from
   `final_showcase_readiness.evidence.configured_live_evidence_bundle`.
   Rejected because the final readiness evidence summary does not include the
   bundle blocker details and operator actions needed for a useful closure
   section.

3. Call `build_configured_live_evidence_bundle_report` from the closure packet.
   Rejected because the configured bundle builder can traverse configured
   preflight/final demo launch paths that already include closure/readiness
   reports. The closure packet should read saved evidence, not generate upstream
   final evidence.

## Acceptance

- `final_launch_closure_packet_report.sections` includes
  `configured_evidence_bundle` between `live_provider_consent` and
  `final_acceptance`.
- `sections_by_id.configured_evidence_bundle` has command
  `make configured-live-evidence-bundle` and a first action with id
  `configured_live_evidence_bundle`.
- Missing bundle evidence blocks the closure packet and adds a runnable operator
  action for `make configured-live-evidence-bundle`.
- Ready bundle evidence makes the configured bundle section ready.
- `source_reports.configured_live_evidence_bundle` summarizes the saved bundle
  report.
- `commands` includes `make configured-live-evidence-bundle` after
  `make live-provider-evidence` and before `make final-showcase-readiness`.
- The packet sanitizer removes unsafe bundle details such as provider secrets,
  local paths, file URLs, bearer tokens, and payment field names.
- Existing closure packet, final demo launch, and final showcase readiness tests
  pass.

## Test Strategy

Use TDD in `services/backend/tests/test_final_launch_closure_packet.py`:

- Extend the missing-actions test to require the new section order, blocked
  first action, operator action, command list entry, and source report summary.
- Extend the ready-sections test to write a ready configured bundle and assert
  the new section becomes ready.
- Extend the redaction test to write unsafe configured bundle details and assert
  the closure packet sanitizer removes them.

Then update `services/backend/src/myth_forge_api/final_launch_closure_packet.py`
with the smallest loader, section, command, and source-report changes needed to
pass those tests.

## Self-Review

- Placeholder scan: no open placeholders.
- Scope check: one backend closure packet evidence section; no iOS UI, Makefile,
  provider adapter, or live-provider behavior changes.
- Safety check: the closure packet reads a saved ignored report and keeps all
  safety flags read-only.
- Ambiguity check: section id, action id, command, report path, section order,
  and redaction expectations are explicit.
