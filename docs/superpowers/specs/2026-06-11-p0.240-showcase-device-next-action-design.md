# P0.240 Showcase Device Next Action Design

## Context

`final-demo-launch-local` now surfaces the concrete mobile deploy preflight blockers at the top level. `final-showcase-readiness` still exposes a generic `first_blocker` and `next_action` for `ios_deployable`, while the specific device action is only available deeper in `device_action_bundle.first_action`.

This leaves two different operator entry points: one says to fix `DEVELOPMENT_TEAM` and the iPhone-reachable backend URL; the other says only that the iOS runbook and rehearsal must be ready. The final iPhone showcase should make the same next physical-device step visible from both entry points.

## Goal

When `final-showcase-readiness` has saved mobile preflight evidence, enrich the top-level `first_blocker.detail` and `next_action.detail` with the `device_action_bundle.first_action` command and evidence detail.

## Non-Goals

- Do not change the top-level blocker identity from `ios_deployable`; it is still the capability blocking the final goal.
- Do not change configured provider behavior, live provider calls, Xcode signing, Keychain, or global Mac state.
- Do not run Xcode, write deploy config, call Meshy/OpenAI/Treatstock, or place orders.
- Do not require mobile model changes; the existing detail string already appears in the iPhone Final Launch summary.

## Approaches Considered

1. Enrich the existing top-level `ios_deployable` detail with the first device action.
   This is selected. It keeps the capability contract stable while making the displayed row actionable.

2. Replace `next_action` with `start_backend_device_demo`.
   This would be more direct, but it changes the existing `next_action.id` semantics from capability-level to action-level and would ripple into Swift contract expectations.

3. Add a new top-level `device_next_action` field.
   This is additive, but the mobile UI already renders `next_action.detail`; a new field would require extra mobile wiring to be visible.

## Design

`build_final_showcase_readiness_report` will compute the device action bundle once before building the report dictionary. It will pass the bundle into `_first_blocker` and `_next_action`.

If the selected first blocker is `ios_deployable`, and `device_action_bundle.first_action` has a non-missing `evidence_status`, the blocker detail will append:

```text
Next device action: <command> | <evidence_detail>
```

The same enriched detail flows into `next_action`. Missing evidence keeps the existing generic detail to avoid noisy placeholders in source-only tests.

- Assert `first_blocker.id` and `next_action.id` remain `ios_deployable`.
- Assert both details include `Next device action: make backend-device-demo` and `PMF_BACKEND_BASE_URL must be iPhone-reachable`.
- Assert the details do not include provider-key blockers such as `MESHY_API_KEY`.

## Self-Review

- No placeholders or deferred requirements.
- Scope is one report contract and one focused backend test.
- The behavior reuses already-sanitized device action evidence and does not introduce external calls.
