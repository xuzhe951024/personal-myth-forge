# P0.623 Showcase Demo Fallback Pruning Spec

## Context

`final-showcase-readiness` is the mobile-visible readiness source for the final iPhone demo. When the final showcase is blocked on configured live provider evidence, its top-level next action should point to the most specific guarded provider consent command.

The report currently keeps this fallback action after the 3D/NPC consent gates:

```text
make final-demo-launch-configured; rerun make final-configured-evidence-plan
```

That command is useful when no more specific live provider consent action is available, but it is redundant when the showcase already presents capability-specific Meshy/OpenAI consent gates.

## Requirements

- Preserve `make final-demo-launch-configured` in non-live-consent nested handoff contexts.
- When a live provider consent gate is present, treat configured demo-launch commands as provider fallback actions.
- Continue preserving print quote consent actions and other non-provider actions.
- Do not execute live provider calls or mutate global developer state.

## Acceptance

- A focused test proves configured demo-launch fallback is pruned only when live consent is present.
- Existing final showcase readiness tests still pass.
- Safe `.local/final-showcase-readiness.json` refresh no longer lists the configured demo-launch fallback while blocked on 3D live provider consent.
