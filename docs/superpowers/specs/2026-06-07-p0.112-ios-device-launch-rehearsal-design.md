# P0.112 iOS Device Launch Rehearsal Design

## Context

P0.111 added `make ios-device-launch-certificate`, a read-only Mac-side packet
for starting the physical iPhone demo sequence. It intentionally does not run
the upstream commands that refresh local rehearsal reports, configured
preflight, or the final handoff index. That keeps the certificate safe, but it
still leaves final validation spread across several commands.

P0.112 adds a project-root rehearsal wrapper that refreshes the safe local and
read-only reports in the expected order, accepts report-written exit code `2`,
and writes one ignored rehearsal summary for the operator.

## User-Facing Contract

Add:

```bash
make ios-device-launch-rehearsal
```

The target writes:

```text
services/backend/.local/ios-device-launch-rehearsal.json
```

The target returns exit `0` when every command in the rehearsal sequence exits
`0` or `2` and the summary report is written. Exit `2` from individual report
commands means "blocked report written" and should not stop the wrapper. Exit
`1` or any other exit code means an unusable command failure and must stop the
wrapper.

## Rehearsal Sequence

The wrapper runs these commands from the project root:

1. `make final-rehearsal-local`
2. `cd services/backend && uv run python -m myth_forge_api.cli final-configured-preflight --repo-root ../.. --output .local/final-configured-preflight.json`
3. `cd services/backend && uv run python -m myth_forge_api.cli final-handoff-index --repo-root ../.. --output .local/final-handoff-index.json`
4. `cd services/backend && uv run python -m myth_forge_api.cli ios-device-launch-certificate --repo-root ../.. --output .local/ios-device-launch-certificate.json`
5. `cd services/backend && uv run python -m myth_forge_api.cli ios-device-launch-rehearsal --repo-root ../.. --output .local/ios-device-launch-rehearsal.json`

The first four commands may write blocked reports. The final command is
read-only and summarizes saved JSON reports.

## Report Shape

Top-level fields:

- `kind`: `ios_device_launch_rehearsal_report`
- `status`: `ready`, `partial`, or `blocked`
- `summary`: counts by command/report status
- `sequence`: ordered command report sources
- `final_handoff_index`: saved report status
- `configured_preflight`: saved report status
- `ios_device_launch_certificate`: saved report status
- `operator_actions`: next safe commands
- `commands`: flat command list
- `safety`: command/writer contract

`status = blocked` when any saved required report is missing, unreadable, or
blocked. `status = partial` when required saved reports exist and no required
report is blocked, but manual/live gates remain. `status = ready` when required
saved reports are ready.

## Safety

The wrapper may run local/no-key or read-only commands and write ignored JSON
reports under `services/backend/.local/`. It must not call live providers, apply
secrets, write backend env, write iOS deploy config, start servers, run Xcode,
touch signing/keychain, accept Apple licenses, or mutate global machine state.

The summary report must redact provider key-like strings, bearer tokens, raw
media URIs, payment links, local file URLs, and absolute repo/home paths.

## Acceptance

- Unit tests cover missing reports, ready/partial saved reports, CLI output, and
  sanitization.
- Shell script tests cover accepting exit `2` and failing unusable exit `1`.
- Makefile dry-run verifies command order.
- Resource template acceptance and iOS showcase acceptance require the script,
  CLI, Make target, output path, and safety strings.
- README, mobile README, roadmap, and visual evidence describe how to run and
  inspect the rehearsal.
