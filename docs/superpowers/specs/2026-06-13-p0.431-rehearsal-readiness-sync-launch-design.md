# P0.431 Rehearsal Readiness Sync Launch Design

## Context

P0.430 added `device_action_bundle` to `ios-device-launch-rehearsal-readiness.json` and taught the iPhone Final Launch model to decode it. The standalone `make ios-device-launch-rehearsal-readiness` target still writes only the readiness report. That leaves `final-demo-launch-local.json`, the iPhone-visible launch report, stale until another command regenerates it.

## Decision

Replace the direct Makefile command with a small wrapper script. The wrapper writes `ios-device-launch-rehearsal-readiness.json`, accepts exit codes `0` and `2` as usable local reports, then regenerates `final-demo-launch-local.json` so the iPhone launch payload embeds the latest readiness report.

## Behavior

- `make ios-device-launch-rehearsal-readiness` runs `services/backend/scripts/write_ios_device_launch_rehearsal_readiness.sh`.
- The wrapper writes `services/backend/.local/ios-device-launch-rehearsal-readiness.json`.
- If readiness exits `0` or `2`, the wrapper runs `final-demo-launch --mode local` and writes `services/backend/.local/final-demo-launch-local.json`.
- If final launch sync exits `0` or `2`, the wrapper exits `0`.
- Any other readiness or sync exit code fails the wrapper.

## Safety

The wrapper only writes ignored `.local` JSON reports. It does not write backend env files, iOS deploy config, Xcode project state, Keychain, provider resources, or print orders. It does not call live providers by default.

## Tests

Script tests will assert accepted blocked readiness reports sync final demo launch afterward. Makefile dry-run tests will assert the target uses the wrapper instead of directly calling the CLI.
