# P0.91 NPC Evaluation Readiness Design

## Purpose

P0.90 added a local `evaluate-npc` command, but the final iPhone launch surface
does not yet show whether that report exists or is ready. P0.91 makes NPC Agent
quality readiness visible in the same final-launch path as resources, final
acceptance, and operator handoff.

This iteration remains read-only. It does not run `evaluate-npc`, call OpenAI,
call Meshy, start servers, build Xcode projects, sign apps, or mutate global
machine state.

## Scope

In scope:

- A backend `npc_agent_evaluation_readiness` report that reads the saved local
  NPC evaluation JSON.
- Integration into `/v1/final-demo-launch`.
- Swift models for decoding the readiness report.
- iPhone `Final Launch Status` rows that show missing, blocked, or ready NPC
  Agent evaluation state.
- Source acceptance and visual regression coverage.

Out of scope:

- Running NPC evaluation automatically from an API endpoint.
- Adding an iPhone button that executes Mac-side commands.
- Live OpenAI quality evaluation.
- Reworking the existing final operator handoff sequence.

## Backend Behavior

The backend reads:

```text
services/backend/.local/npc-evaluation-local.json
```

Missing file:

- `status`: `missing`
- exit code: `2`
- operator action: run local `evaluate-npc` and write the expected file.

Ready file:

- `kind` must be `npc_agent_evaluation_report`
- `failed` must be `0`
- `succeeded` must be positive
- coverage must include at least one expected NPC set, one trace set, and one
  world-resolution step
- `status`: `ready`
- exit code: `0`

Blocked file:

- invalid JSON, wrong shape, wrong kind, failed cases, empty success count, or
  missing coverage becomes `blocked`
- blockers include safe `id`, `classification`, `detail`, and `command`

The report includes:

- `kind`: `npc_agent_evaluation_readiness_report`
- `status`
- `source_file`
- `summary`
- `coverage`
- `blockers`
- `operator_actions`
- `safety`

It sanitizes provider secrets, raw private markers, media payloads, local paths,
payment URLs, and checkout fields.

## Final Demo Launch

`build_final_demo_launch_report` embeds the readiness report as top-level
`npc_agent_evaluation_readiness`.

This field is informational. It does not change final launch exit codes or phase
counts yet because local demo launch can still proceed before a saved NPC
evaluation report exists. The operator sees the missing or blocked state on the
iPhone and can run the Mac-side command before final review.

## Mobile Behavior

Swift adds `NPCAgentEvaluationReadinessReport` and related summary/blocker models.
`FinalDemoLaunchReport` decodes the optional
`npc_agent_evaluation_readiness` field.

`FinalLaunchMobileSummary` gains `npcEvaluationRows`:

- missing: `Run local NPC Agent evaluation to create the readiness report.`
- ready: `NPC Agent evaluation ready: 6 cases, 12 ticks.`
- blocked: first blocker row with command and safe detail

`FinalLaunchStatusView` renders a new `NPC Evaluation` group between
`Acceptance` and `Next`.

## Safety

The mobile summary reuses existing redaction. Backend readiness also sanitizes
all strings before returning the report. Tests include a fake bearer token and
fake raw private marker to prove redaction.

## Testing

Backend:

- missing file readiness
- ready report readiness
- invalid JSON / wrong kind / failed cases readiness
- final demo launch embeds the readiness field
- source acceptance requires backend and mobile wiring

Swift:

- decode readiness payload
- summary rows for missing, ready, and blocked
- unsafe strings are redacted before display
- project/source checks require the model, summary builder, view section, and
  contract tests

Visual:

- a static 390x844 final launch panel snapshot showing the new NPC Evaluation
  section with ready coverage.

## Acceptance Criteria

- A saved local NPC evaluation report is visible through `/v1/final-demo-launch`.
- The iPhone final launch status can show whether NPC Agent quality review is
  missing, blocked, or ready.
- No API endpoint or UI path executes local commands or live providers.
- Existing final acceptance still reports only the known local iOS/Xcode
  blockers on this machine.
