# P0.284 Backend Device Action Validation Design

## Context

The final iOS showcase remains blocked by device evidence: Apple development
team configuration, an iPhone-reachable backend URL, Xcode license/build proof,
and real device health checks. Local reports already identify those blockers,
but some operator actions still say `start backend-device-demo and rerun mobile
deploy preflight` without the concrete make target or validation command.

The final handoff should consistently tell the operator which local command to
start and which validation command proves the next gate.

## Goal

Normalize backend-device-demo operator actions so final handoff reports prefer:

```text
start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight
```

This makes the iPhone backend action explicit without starting the backend,
touching Xcode, or making provider calls.

## Scope

- Only local report action text changes.
- No backend server process is started.
- No iOS deploy, signing, Keychain, global Xcode state, or real device action is
  executed.
- No Meshy/OpenAI/Treatstock/Sculpteo calls are made.
- Existing source report safety and secret redaction remain unchanged.

## Approach

Use the existing shared `operator_actions` helper as the normalization point:

- Add a canonical backend device demo action constant.
- Normalize legacy backend-device-demo phrases to that canonical action.
- Teach mobile deploy validation to append `make mobile-deploy-preflight` to the
  canonical backend device action.
- Use the shared validation-aware helper from source reports that currently emit
  the legacy phrase directly.

Rejected alternatives:

- Hand-edit only `final-showcase-readiness`. That leaves child reports and final
  local refresh inconsistent.
- Replace the action with only `make backend-device-demo`. That loses the paired
  validation command the operator needs after starting the server.

## Testing

- RED test in `test_operator_actions.py` proves the legacy backend-device-demo
  phrase normalizes to the canonical command plus validation.
- RED tests in mobile deploy preflight and final acceptance/readiness reports
  prove source reports no longer emit the legacy phrase.
- Refresh `.local/final-local-report-refresh.json` and
  `.local/final-showcase-readiness.json`; both may stay blocked, but their
  operator actions must include the canonical backend device action and omit the
  legacy phrase.
- Run backend lint, backend tests, and `git diff --check` before merging.
