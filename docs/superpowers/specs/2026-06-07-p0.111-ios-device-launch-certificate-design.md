# P0.111 iOS Device Launch Certificate Design

## Context

The project now has separate readiness surfaces for final handoff, configured
provider preflight, local rehearsal, iOS deploy runbook, and mobile deploy
preflight. That is enough for an operator who already knows the repo, but the
final C-path goal needs a single Mac-side certificate that answers:

- Is this checkout ready to start the iPhone demo sequence?
- Which final mode will the iPhone use?
- Are signing, bundle id, LAN backend URL, local rehearsal, configured
  preflight, and live-provider consent accounted for?
- What exact commands should the operator run next?

P0.111 adds that certificate as a read-only report. It does not start uvicorn,
does not call Meshy/OpenAI/Treatstock/Sculpteo, does not write `.env` or
`Deployment.local.xcconfig`, and does not invoke Xcode signing.

## User-Facing Contract

Add:

```bash
make ios-device-launch-certificate
```

The target writes:

```text
services/backend/.local/ios-device-launch-certificate.json
```

The report returns exit `0` only when the device launch packet is ready enough
to begin the operator-run iPhone sequence. It returns exit `2` when required
handoff inputs are missing or blocked.

## Report Shape

Top-level fields:

- `kind`: `ios_device_launch_certificate_report`
- `status`: `ready` or `blocked`
- `mode`: `local` or `configured`
- `certificate`: sanitized signing/backend/final-mode facts
- `device_gates`: required/manual/live gate list
- `final_handoff_index`: embedded status and lane summary
- `ios_deploy_runbook`: embedded mode/status and deploy slot summary
- `final_demo_launch`: embedded mode/status and launch summary
- `operator_sequence`: exact next command order
- `commands`: flat command list
- `safety`: read-only/no-provider/no-Xcode/no-secrets contract

## Status Logic

`status = ready` when:

- `final_handoff_index.status == ready`
- iOS deploy config has:
  - non-empty `DEVELOPMENT_TEAM`
  - non-empty `PRODUCT_BUNDLE_IDENTIFIER`
  - non-loopback `PMF_BACKEND_BASE_URL`
  - `PMF_FINAL_LAUNCH_MODE` of `local` or `configured`
- `ios_deploy_runbook.status != blocked`
- `final_demo_launch.overall_status != blocked`

Manual and live gates remain visible but do not block this certificate:

- `backend_device_server`: manual
- `mobile_deploy_preflight`: operator-run, reflected from runbook slots
- `xcode_build_gate`: manual
- `configured_final_acceptance`: live and requires explicit consent

## Privacy And Safety

The report must redact provider key-like strings, bearer tokens, raw media URIs,
payment URLs, local file URLs, and absolute repo/home paths. It must not include
raw backend environment values or local signing values; it should only expose
configured/readiness booleans and source file paths.

Safety flags must explicitly state:

```json
{
  "commands_run": false,
  "provider_calls": false,
  "writes_local_config": false,
  "writes_backend_env": false,
  "writes_ios_deploy_config": false,
  "global_mutation": false,
  "xcode_or_signing": false,
  "keychain_writes": false,
  "provider_secrets_in_report": false,
  "raw_media_in_report": false,
  "payment_links_in_report": false,
  "local_paths_in_report": false
}
```

## Acceptance

- Unit tests cover missing/blocked and ready configured fixtures.
- CLI test covers `ios-device-launch-certificate --output`.
- Makefile exposes the target and writes the ignored `.local` JSON path.
- Resource template acceptance includes this report builder in source gates.
- iOS ProjectChecks source gate includes the new Make/CLI/output/safety strings.
- Docs explain how to run and read the certificate.
- A mobile-sized visual regression artifact documents the operator experience.
