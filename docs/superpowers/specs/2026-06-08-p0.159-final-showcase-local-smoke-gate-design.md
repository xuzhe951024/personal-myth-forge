# P0.159 Final Showcase Local Smoke Gate Design

## Context

P0.158 added a no-key `local_showcase_smoke_report` that runs the complete local HTTP loop:

- guided scan upload
- myth session creation from capture
- generated GLB and iOS-loadable DAE download
- NPC autonomy run with two backend-owned ticks
- print quote creation
- downloadable 3MF print candidate

That proof is embedded in `final_demo_launch_report` and visible in the iPhone Final Launch Status panel. The remaining gap is that `final_showcase_readiness_report`, the final C-path objective ledger, does not yet require or summarize this proof. A final readiness report can therefore look at source gates, saved evaluation reports, and visual readiness without explicitly proving the actual iPhone-facing local loop still works.

## Decision

Add a required `local_showcase_smoke` capability to `final_showcase_readiness_report`.

The capability is backed by `build_local_showcase_smoke_report()`. It runs in temporary storage, uses only local providers, and does not start uvicorn, call live providers, write repo media, mutate global state, expose raw media, expose payment links, or include provider secrets.

The capability is placed after `capture_scanning` and before `game_asset_3d_generation` because it is the integrated bridge between source-level capture support and standalone 3D/NPC/print evaluation readiness.

## Alternatives Considered

1. **Use saved `.local/local-showcase-smoke.json` only.**
   This would avoid rerunning the smoke during readiness, but it would be stale-prone and weaker than the current P0.158 design, which already proves the loop cheaply with temporary storage.

2. **Keep local smoke only in `final_demo_launch_report`.**
   This preserves the existing UI but leaves `final_showcase_readiness_report` incomplete as the objective ledger.

3. **Recommended: run and summarize local smoke in final showcase readiness.**
   This keeps the ledger self-contained and turns the integrated local loop into a first-class gate without needing keys or device access.

## Backend Behavior

`build_final_showcase_readiness_report()` will:

- import `build_local_showcase_smoke_report`
- run it once during report construction
- pass the report into `_capabilities()`
- add `local_showcase_smoke` to `CAPABILITY_ORDER`
- include a top-level evidence summary under `evidence.local_showcase_smoke`
- include the smoke report status and summary counts in capability evidence

Capability statuses:

- `ready`: smoke status is `succeeded`, summary has `failed == 0`, `http_steps >= 6`, `npc_ticks >= 2`, `downloads >= 3`, and safety flags prove no provider calls, global mutation, repo media writes, raw media, payment links, or local paths.
- `blocked`: any smoke failure, missing report shape, insufficient count, or unsafe safety flag.

The command for the capability is `make local-showcase-smoke`, so an operator can reproduce the standalone proof and write `services/backend/.local/local-showcase-smoke.json`.

## Mobile Behavior

`FinalShowcaseSummary` will gain a compact digest row for the local smoke capability. The row should show:

- ready state: `Local smoke ready: HTTP 6, NPC ticks 2, downloads 3.`
- blocked state: first local smoke blocker or command
- sanitization of unsafe source text

This is distinct from P0.158's `FinalLaunchMobileSummary.localShowcaseSmokeRows`: Final Launch shows the raw launch report section, while Final Showcase summarizes objective readiness.

## Acceptance Gates

Update source acceptance so both final showcase gates verify local smoke:

- `final_showcase_readiness.py` imports and uses `build_local_showcase_smoke_report`
- `final_showcase_readiness_report` includes `local_showcase_smoke`
- `FinalShowcaseSummary` exposes local smoke rows
- Swift contract tests decode and render the new final showcase digest
- `ios_showcase_acceptance` checks the backend and mobile wiring
- visual regression includes an iPhone-sized final showcase view with the local smoke row

## Safety

This iteration must not:

- call Meshy, OpenAI, Treatstock, or any live provider
- start a persistent backend server
- write `.env`, Xcode config, keychain, signing settings, or global system config
- include raw capture media, raw personal context, provider secrets, local file paths, or payment links in reports
- claim physical-device validation or real provider validation
