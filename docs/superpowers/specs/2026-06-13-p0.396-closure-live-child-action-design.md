# P0.396 Closure Packet Live Child Action Design

## Context

`final-external-action-ledger` now preserves concrete child actions for live provider and print handoff rows. For example, live provider evidence can tell the operator to run:

```text
make final-resource-apply-preview; rerun make live-provider-evidence
```

and print fulfillment can tell the operator to save a sanitized configured quote before rerunning print readiness.

`final-launch-closure-packet` consumes the ledger action groups, but its compacted action rows drop the ledger `operator_action` field. The packet then reconstructs generic top-level actions such as `approve live provider cost before make live-provider-evidence`. That makes the highest-level closure packet less actionable than the underlying ledger.

## Goal

Make `final-launch-closure-packet.operator_actions` prefer concrete ledger `operator_action` values for live provider, configured print quote, and manual Xcode/global rows while preserving existing fallbacks.

## Scope

- Preserve `operator_action` when ledger actions are compacted into closure packet actions.
- Prefer `operator_action` in closure packet top-level `operator_actions` when an action requires cost consent or manual confirmation.
- Continue using existing generic action text when no concrete operator action is available.
- Keep resource input formatting unchanged.
- Keep status, first blocker, next action, section ordering, and safety flags unchanged.

## Out of Scope

- No Meshy/OpenAI/Treatstock calls.
- No `.env` or iOS deployment config writes.
- No Xcode build, signing, Keychain, Apple license, or global machine mutation.
- No changes to provider readiness semantics.
- No changes to mobile runtime UI.

## Design

`final_external_action_ledger._action()` already emits `operator_action` only when it has a safer or more specific operator-facing command. `final_launch_closure_packet._compact_ledger_action()` should carry that optional field into the closure action row.

`final_launch_closure_packet._operator_actions()` should use this order:

1. Resource input rows still produce `provide <slot> in <file>; rerun <validation command>`.
2. Cost-consent rows use `operator_action` when present.
3. Manual-confirmation rows use `operator_action` when present.
4. Rows without a concrete `operator_action` fall back to the existing generic text.
5. Deduplication and apply-preview-before-apply filtering remain unchanged.

This keeps the report read-only and deterministic while making the final closure packet mirror the current best handoff commands.

## Tests

Add a regression test in `services/backend/tests/test_final_launch_closure_packet.py` that builds a closure packet from a fixture where live provider evidence exposes a child `next_action` and print fulfillment exposes configured quote handoff. The test must verify:

- `operator_actions` contains `make final-resource-apply-preview; rerun make live-provider-evidence`.
- `operator_actions` contains the Treatstock configured quote handoff action.
- Generic fallbacks such as `approve live provider cost before make live-provider-evidence` and `approve live provider cost before make print-fulfillment-readiness` are absent.

Focused verification:

```bash
cd services/backend && uv run pytest tests/test_final_launch_closure_packet.py
```

Broader verification:

```bash
make backend-test
make final-local-report-refresh
```
