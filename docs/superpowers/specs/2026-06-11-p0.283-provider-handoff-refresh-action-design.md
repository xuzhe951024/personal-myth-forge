# P0.283 Provider Handoff Refresh Action Design

## Context

`final-local-report-refresh` is the safe local command that regenerates ignored
handoff reports without calling live providers, mutating global Xcode state, or
writing secrets. The current saved report still exposes an operator action like
`review refreshed provider_handoff report` even though the provider handoff
report already publishes executable `next_commands`, including the make target
for refreshing provider handoff evidence.

That wording is not ideal for the final iOS showcase handoff because the user
should see concrete, rerunnable commands wherever the repo already knows them.

## Goal

When a blocked refresh step has no promoted blocker command but its child report
contains `next_commands`, `final-local-report-refresh` should promote the first
safe next command for that step instead of emitting a generic review action.

## Scope

This iteration only changes local report aggregation:

- No real Meshy, OpenAI, Treatstock, or Sculpteo calls.
- No provider key reads beyond existing settings behavior.
- No Xcode signing, Keychain, global environment, or device operations.
- No change to provider readiness semantics.

## Approach

Preferred approach: store a bounded step-local `next_command` while each refresh
step is being run, derive it from the child report's `next_commands`, and use it
only as a fallback when the step has no explicit `command`.

Rejected alternatives:

- Add provider-handoff-specific special cases to `_operator_actions`. This would
  solve the immediate symptom but make future child reports inconsistent.
- Change `provider_handoff` to expose `operator_actions`. That would help one
  report, but refresh already handles many report shapes and should understand
  existing `next_commands`.

## Behavior

- `_run_step()` records `next_command` from the first non-empty string in a child
  report's `next_commands`.
- `_blocker_command()` keeps preferring an explicit blocker/next-action command.
  If none exists, it uses the step-local `next_command`.
- `_operator_actions()` uses the same fallback, then normalizes and dedupes via
  the existing shared operator action normalizer.
- Provider handoff CLI commands inside `next_commands` continue normalizing to
  `make provider-handoff`.

## Testing

- Add a focused RED test that runs a custom blocked child report with only
  `next_commands`, then asserts the step and top-level operator actions contain
  the normalized make target and do not contain `review refreshed ...`.
- Run focused final-local refresh and operator action tests.
- Run `make final-local-report-refresh` to refresh ignored evidence. It may exit
  `2` while external resources and iPhone evidence remain missing, but the saved
  report must no longer contain `review refreshed provider_handoff report`.
- Run backend lint and backend tests before merge.
