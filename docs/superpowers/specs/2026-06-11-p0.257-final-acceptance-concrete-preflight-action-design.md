# P0.257 Final Acceptance Concrete Preflight Action Design

## Goal

When local final acceptance is blocked by the mobile deploy preflight gate, surface the same concrete next action already recorded in `mobile-deploy-preflight-evidence.json`.

## Current Gap

After refreshing local reports, final showcase readiness correctly points its first blocker to `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig`. The `functional_regression` capability still receives a final acceptance readiness blocker whose detail is derived from command output and whose operator action says only `provide iOS deploy config and rerun mobile deploy preflight`.

## Requirements

- Keep final acceptance readiness read-only. It must not run commands or write configs.
- If `mobile_deploy_preflight` is blocked and saved preflight evidence has `next_action`, attach that sanitized action to the readiness blocker.
- Operator actions should use the concrete command plus validation command when available.
- Preserve existing fallback behavior when the evidence file is missing, malformed, or lacks `next_action`.
- Do not expose secrets, local paths, payment links, raw media, or raw personal content.

## Non-Goals

- No iOS signing or Xcode build.
- No provider calls.
- No change to readiness status ordering.
