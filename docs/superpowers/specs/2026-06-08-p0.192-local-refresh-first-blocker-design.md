# P0.192 Local Refresh First Blocker Design

## Problem

`final_local_report_refresh_report` is the safe Mac-side snapshot that refreshes
the local final evidence bundle. It can currently return `status=blocked` while
the top-level `first_blocker` is `null`, so operators must inspect `steps[]` to
find the first actionable report that still needs attention.

## Scope

P0.192 adds a structured top-level `first_blocker` to the local refresh report:

- Choose the first `failed` step before any blocked step.
- If no step failed, choose the first `blocked` step.
- Include step id, label, status, classification, command, detail, output path,
  and the original step id.
- Reuse any nested report `first_blocker`, operator action, missing requirement,
  failed check, or local refresh fallback text without running extra commands.
- Pass the blocker through the existing report sanitizer so local paths, keys,
  bearer tokens, and file URLs remain redacted.
- Update source gates so future local refresh changes cannot remove the field.

## Non-Goals

- Do not change refresh step ordering.
- Do not change ready/blocked/failed status semantics.
- Do not run live providers, device preflight, Xcode, signing, keychain, or
  global commands.
- Do not mark final local refresh ready without real ready evidence.

## Data Shape

For the default blocked run, the report should include:

```json
{
  "id": "final_resource_requirements",
  "label": "Final resource requirements",
  "status": "blocked",
  "classification": "missing_required_value",
  "command": "run make final-resource-init",
  "detail": "Backend-only secret for live Meshy 3D generation.",
  "output": "services/backend/.local/final-resource-requirements.json",
  "step_id": "final_resource_requirements"
}
```

For an unexpected failed extra step, the blocker should use
`classification=step_failed`, `command=fix final-local-report-refresh step <id>`,
and the sanitized exception text as detail.

## Acceptance Criteria

- Default blocked refresh reports expose a non-null top-level `first_blocker`.
- Unexpected failed refresh steps produce a failed `first_blocker` and redact
  unsafe exception text.
- Ready refresh reports return `first_blocker=null`.
- Existing report summary, step order, operator actions, output writes, exit
  codes, and safety flags remain unchanged.
- iOS showcase source acceptance requires `first_blocker` in
  `final_local_report_refresh.py`.

## Self-Review

- Placeholder scan: no TBD or deferred decisions remain.
- Scope check: one backend report contract plus source gate only.
- Ambiguity check: failed steps take precedence over blocked steps because a
  failed refresh indicates the refresh pipeline itself broke.
