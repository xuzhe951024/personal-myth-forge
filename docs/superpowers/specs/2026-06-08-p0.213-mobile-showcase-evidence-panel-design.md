# P0.213 Mobile Showcase Evidence Panel Design

## Goal

Add an iPhone-visible evidence panel that proves the local demo loop has progressed through capture-to-3D provenance, generated result readiness, SceneKit load proof, NPC Agent resolution, and print handoff state.

## Context

The app already has separate receipts for capture generation, generation result, SceneKit load proof, NPC Agent ticks, and print fulfillment. `FinalShowcaseSummary` and `DemoScript` show readiness at a broad level, but they do not give the operator one compact, privacy-safe evidence packet that answers: "Did this phone actually complete the showcase loop?"

## Approach

Create a Swift core summary model and builder:

- `ShowcaseEvidenceSummary`
- `ShowcaseEvidenceItem`
- `ShowcaseEvidenceSummaryBuilder`

The builder consumes existing receipt types and returns ordered items for:

1. Capture-to-3D
2. Generated result
3. SceneKit load proof
4. NPC Agent
5. Print handoff

The summary is read-only and does not trigger provider, file, Xcode, or keychain operations. It only redacts and classifies state that is already present in memory.

## iOS UI

Add a compact SwiftUI `ShowcaseEvidenceView` that matches the existing working-panel style used by `FinalShowcaseSummaryView`: headline, status badge, ordered rows, and privacy notes. Insert it near the top of `ForgeRootView`, after `FinalShowcaseSummaryView`, so the operator can inspect demo evidence before lower-level panels.

`Artifact3DPreviewView` will expose an optional callback:

```swift
onSceneLoadProofChange: @MainActor @escaping (ArtifactSceneLoadProof) -> Void
```

The root view stores the latest proof and passes it to `ShowcaseEvidenceSummaryBuilder`. This keeps the actual SceneKit proof owned by the existing preview view while making the evidence visible elsewhere.

## Status Rules

The summary status is:

- `waiting` when no myth session has produced enough evidence yet.
- `ready` when capture-to-3D is ready, generation result can present, SceneKit proof is loaded, NPC Agent summary is ready with at least three saved ticks, and print handoff is ready or explicitly not required for the local demo.
- `needsAttention` when any required evidence is blocked, missing, failed, or unsafe.

Print remains a demo handoff, not an order placement flow. A missing quote is classified as `waiting` until the NPC loop is complete, then `needsAttention` only if print fulfillment is blocked or waiting on required approval.

## Privacy

The builder must redact:

- API-key-like values
- bearer tokens and authorization strings
- local file paths
- local capture URIs
- checkout/payment links
- raw private context markers

The evidence panel must not display raw capture media, provider keys, local absolute paths, or checkout URLs.

## Testing

Use TDD:

- Swift contract tests for waiting, ready, needs-attention, and redaction behavior.
- Swift ProjectChecks source gates for the new model, view, root wiring, SceneKit callback, and contract test names.
- Swift compile-check for the iOS app source.

No backend tests are required because this iteration does not change backend behavior.

## Self-Review

- Scope is limited to iOS core/app wiring and source gates.
- No new provider calls, no global config writes, no external secrets.
- UI matches the existing compact SwiftUI panel pattern instead of introducing a new visual language.
- The design does not claim real provider readiness; it reports current in-memory demo evidence only.
