# P0.61 NPC Agent Provider Acceptance Design

## Purpose

The backend already contains OpenAI-backed NPC director and tick runtimes, and
the mobile app can display NPC traces and tick updates. The current final
acceptance path proves these pieces indirectly through `provider_handoff`,
`demo_acceptance`, and `ios_showcase_acceptance`, but it does not expose a
dedicated gate that answers the product question: "Are NPCs wired to an AI Agent
provider path with backend-only keys and structured output?"

P0.61 adds that missing source/config acceptance gate.

## Scope

This is a source-only and configuration-only acceptance iteration. It must not
call OpenAI, create provider keys, mutate global shell state, install on an
iPhone, or change NPC gameplay behavior.

In scope:

- Add a backend `npc_agent_provider_acceptance` runner.
- Verify OpenAI NPC director structured output, stateless tick runtime,
  provider factory wiring, provider readiness capabilities, resource handoff
  language, final acceptance nesting, and docs.
- Include the new gate in quick and full final acceptance reports.
- Keep reports sanitized: no provider secrets, raw media payloads, local paths,
  or payment links.

Out of scope:

- Running a live OpenAI request.
- Replacing the local deterministic NPC fallback.
- Persisting server-side tick history.
- Adding Unity movement execution or voice NPCs.
- Adding new mobile UI.

## Requirements

`npc_agent_provider_acceptance` must pass when these source facts are present:

- `services/backend/src/myth_forge_api/providers/npc.py` contains
  `OpenAINPCDirector`, `OpenAINPCOutput`, `text_format=OpenAINPCOutput`,
  `agent_runtime="openai_structured_runtime"`, and
  `OPENAI_API_KEY is required for NPC generation.`
- `services/backend/src/myth_forge_api/providers/npc_ticks.py` contains
  `OpenAINPCTickRuntime`, `OpenAINPCTickOutput`,
  `runtime_name = "openai_tick_structured_runtime"`,
  `text_format=OpenAINPCTickOutput`, and
  `OPENAI_API_KEY is required for NPC tick generation.`
- `services/backend/src/myth_forge_api/providers/factory.py` routes
  `npc_provider == "openai"` to both `OpenAINPCDirector.from_settings` and
  `OpenAINPCTickRuntime.from_settings`.
- `services/backend/src/myth_forge_api/providers/readiness.py` reports
  `structured_agent_traces`, `structured_agent_ticks`, and missing
  `OPENAI_API_KEY` when `NPC_PROVIDER=openai` lacks a key.
- `services/backend/src/myth_forge_api/resource_handoff.py` includes
  `NPC_PROVIDER`, `OPENAI_API_KEY`, `AI Agent NPC traces and ticks`, and
  backend-only key notes.
- `README.md` documents `NPC_PROVIDER=openai`, `OPENAI_API_KEY`,
  `OpenAINPCDirector`, `OpenAINPCTickRuntime`, and that mobile never receives
  provider keys.
- `apps/mobile/README.md` documents the OpenAI NPC tick provider and the
  backend-only key boundary.

The report must include:

- `kind: "npc_agent_provider_acceptance_report"`.
- `status: "succeeded"` only when every requirement passes.
- `summary` with passed and failed counts.
- `required_features` with stable ids for each source/config contract.
- `safety` booleans proving there were no provider calls and no secret/path/raw
  media leakage in the report.

Final acceptance must include a top-level in-process check with id
`npc_agent_provider_acceptance` after `demo_acceptance` and before
`capture_3d_acceptance`.

## Architecture

The runner follows the existing source-only acceptance pattern used by
`ios_showcase_acceptance`: define declarative `FeatureRequirement` entries,
scan repository-relative files for required substrings, return a sanitized JSON
report, and expose an `InlineCheckResult`-compatible dataclass.

`final_acceptance` receives an optional runner injection for tests and defaults
to `run_npc_agent_provider_acceptance(repo_root=selected_repo_root)`. This keeps
final acceptance orchestration testable without shelling out or calling
providers.

## Testing

Use TDD:

1. Add source-fixture tests for the new acceptance runner.
2. Watch them fail because the runner does not exist.
3. Implement the minimal runner until those tests pass.
4. Add final acceptance tests that expect the new check id and updated summary
   counts.
5. Watch them fail because final acceptance does not invoke the runner.
6. Wire the runner into final acceptance and re-run focused tests.
7. Update docs and static verification evidence.
8. Run backend and Swift regressions.

## Acceptance

P0.61 is complete when:

- `cd services/backend && uv run pytest tests/test_npc_agent_provider_acceptance.py tests/test_final_acceptance.py -q` passes.
- `make backend-lint` passes.
- `make backend-test` passes.
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks`
  passes.
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests`
  passes.
- `swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck`
  passes.
- Quick final acceptance still exits `2` only for known local iOS/Xcode blockers,
  with the new `npc_agent_provider_acceptance` check passed and no failed
  checks.
