# P0.507 Closure Packet Backend Priority Design

## Context

The final iOS showcase is still blocked by physical iPhone deploy evidence and provider credentials. Recent phases aligned the top-level handoff order for showcase readiness, local demo launch, local report refresh, and the external action ledger so operators see:

1. write or repair iOS deploy config,
2. start `backend-device-demo`,
3. continue provider, print, and global machine handoffs.

`final-launch-closure-packet` still emits `backend-device-demo` after provider keys, iPhone backend URL, Xcode, and live provider actions in current local evidence. In an empty repo fixture it also emits `backend-device-demo` after required resource keys. That weakens the closure packet as the final operator-facing handoff because the first blocker is iPhone deploy evidence, and the LAN backend must be started before rerunning the physical device checks.

## Decision

Add a final ordering pass to `final_launch_closure_packet._operator_actions`:

- keep the first concrete mobile deploy handoff as the first action,
- move `start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight` immediately after that first mobile deploy handoff,
- keep provider key, provider evidence, print quote, and Xcode/global actions after the local device runtime pair.

The anchor is intentionally general: any non-backend action whose operator root reruns `make mobile-deploy-preflight` can anchor the backend demo. This covers both current forms:

- `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight`
- `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`

## Alternatives Considered

1. Reorder closure sections so device evidence comes before resource inputs. This would change report semantics and risk breaking section consumers.
2. Special-case only the deploy writer action. This would fix current `.local` evidence but leave the empty-repo fixture and manual deploy config flow inconsistent.
3. Add a narrow final operator action priority helper. This preserves structured sections and fixes both writer and non-writer mobile deploy handoff forms.

## Behavior

- If the canonical backend demo action is absent, the action list is unchanged.
- If no mobile deploy handoff anchor exists, the action list is unchanged.
- If backend demo is already immediately after the first mobile deploy handoff, the action list is unchanged.
- The helper only reorders existing action strings; it does not create commands, read secrets, write local resources, call providers, run Xcode, or mutate global machine state.
- Existing provider handoff, guarded print quote, iPhone-reachable backend URL, and deploy writer normalization remain unchanged.

## Tests

Add a regression test in `services/backend/tests/test_final_launch_closure_packet.py` using the empty repo fixture. It should assert:

- the first action remains the specific mobile deploy handoff,
- the second action is the canonical backend-device-demo handoff,
- backend-device-demo appears before Meshy/OpenAI key handoffs, provider handoff, print quote, and Xcode license actions.

## Self-Review

- Scope is limited to one flat operator action order in one final handoff packet.
- Structured sections, safety fields, provider guards, and ignored local evidence semantics stay intact.
- Requirements are concrete and covered by local deterministic tests.
