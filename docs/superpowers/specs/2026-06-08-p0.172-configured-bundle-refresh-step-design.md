# P0.172 Configured Bundle Refresh Step Design

## Context

P0.171 made `final_showcase_readiness` depend on the saved
`services/backend/.local/configured-live-evidence-bundle.json` report. That is
the right boundary for the readiness ledger: it should read evidence, not
recursively build upstream final reports.

The remaining gap is that `final-local-report-refresh` still refreshes
`final_configured_evidence_plan`, `live_provider_evidence`, and
`final_showcase_readiness`, but it does not write
`configured-live-evidence-bundle.json`. A safe full local refresh can therefore
finish with `final_showcase_readiness.provider_key_handoff` blocked or partial
only because the configured bundle report was never generated.

## Goal

Add the configured live evidence bundle as a first-class safe refresh step before
`final_showcase_readiness` so the integrated local evidence refresh produces the
same saved report that the final readiness ledger now requires.

## Design

The refresh runner will import `build_configured_live_evidence_bundle_report` and
add a `configured_live_evidence_bundle` step that writes
`services/backend/.local/configured-live-evidence-bundle.json`.

The step will run after `live_provider_evidence` and before
`print_fulfillment_readiness`, `final_external_action_ledger`, and
`final_showcase_readiness`. This order ensures the bundle sees the latest saved
live evidence manifest and final showcase sees the latest saved bundle.

The builder must be called with local settings and default
`allow_live_provider_calls=False`. It remains report-only: it reads existing
ignored evidence files and does not call Meshy, OpenAI, Treatstock, Xcode,
signing/keychain APIs, backend env writers, or iOS deploy config writers.

## Alternatives Considered

1. Add the bundle step to `final-local-report-refresh`.
   This is the selected approach because refresh already owns ignored report
   generation and can keep final readiness read-only.

2. Make `final_showcase_readiness` build the configured bundle directly.
   Rejected because P0.171 showed this creates recursive final-report
   dependencies through configured preflight and final demo launch.

3. Leave the refresh chain unchanged and require the operator to run
   `make configured-live-evidence-bundle` manually every time.
   Rejected because it weakens the "single safe evidence refresh" path and makes
   final readiness easier to misinterpret after a refresh.

## Acceptance

- `run_final_local_report_refresh` includes a
  `configured_live_evidence_bundle` step.
- The step writes
  `services/backend/.local/configured-live-evidence-bundle.json`.
- The step order is:
  `live_provider_evidence` before `configured_live_evidence_bundle`, and
  `configured_live_evidence_bundle` before `final_showcase_readiness`.
- The saved final showcase readiness report reads
  `evidence.configured_live_evidence_bundle` from the refreshed bundle report.
- The refresh safety flags remain unchanged: no live provider calls, no global
  mutation, no Xcode/signing/keychain actions, no backend env writes, and no iOS
  deploy config writes.
- Existing final local refresh tests and final showcase readiness tests pass.

## Test Strategy

Use TDD in `services/backend/tests/test_final_local_report_refresh.py`:

- Extend the safe report writing test to require the new output file and step.
- Extend the ordering test to assert the configured bundle step sits between
  `live_provider_evidence` and `final_showcase_readiness`.
- Assert the final showcase readiness evidence summary includes the refreshed
  configured bundle report kind.
- Run focused refresh tests, adjacent final showcase readiness tests, lint, and
  the backend regression before merging.

## Self-Review

- Placeholder scan: no open placeholders.
- Scope check: one backend refresh-sequence change, no iOS UI or provider API
  changes.
- Safety check: the selected builder defaults to no live calls and writes only
  ignored `.local` evidence through the existing refresh writer.
- Ambiguity check: the report path, step id, order, and verification commands
  are explicit.
