# P0.276 Source Showcase iOS Actions Design

## Context

P0.275 added shared mobile preflight validation for iOS deploy configuration
actions, then wired it into several source and aggregate reports. Current
readiness evidence still shows two remaining top-level gaps:

- `final-resource-apply-preview` can emit bare
  `set PMF_BACKEND_BASE_URL to an iPhone-reachable LAN URL`.
- `final-showcase-readiness` can promote bare nested iOS deploy actions into the
  final showcase ledger.

Both surfaces are part of the operator path toward the physical iPhone demo.
After changing `Deployment.local.xcconfig`, the safe validation command remains
`make mobile-deploy-preflight`.

## Goal

`final-resource-apply-preview` and `final-showcase-readiness` should reuse the
shared mobile deploy validation helper so known iOS deploy config operator
actions include `; rerun make mobile-deploy-preflight`.

## Non-Goals

- Do not add new iOS deploy action roots.
- Do not change readiness status, first blocker, next action, or capability
  calculations.
- Do not alter historical `.local/final-demo-launch-p0.*.json` snapshots.
- Do not write env/config files, run writer scripts, call live providers, run
  Xcode/signing, touch Keychain, or place print orders.

## Design

Import `add_mobile_deploy_validation_command()` from
`myth_forge_api.operator_actions` in:

- `final_resource_apply_preview.py`
- `final_showcase_readiness.py`

For `final_resource_apply_preview`, compose the mobile helper with the existing
final-resource helper before dedupe. For `final_showcase_readiness`, update the
existing `_validation_aware_operator_action()` path so every promoted nested
action is normalized, mobile-validation-aware, and final-resource-validation
aware.

This keeps one canonical mobile preflight validation rule while preserving all
existing report status and blocker semantics.

## Acceptance

- RED/GREEN test proves `final-resource-apply-preview.operator_actions` has no
  known bare iOS deploy config action.
- RED/GREEN test proves `final-showcase-readiness.operator_actions` has no
  known bare promoted iOS deploy config action.
- Existing final-resource validation behavior still passes.
- Refreshed `.local/final-resource-apply-preview.json` and
  `.local/final-showcase-readiness.json` keep expected blocked statuses while
  top-level iOS deploy config actions include
  `; rerun make mobile-deploy-preflight`.
- Safety flags remain read-only with no provider calls, env/config writes,
  global mutation, Xcode/signing, or Keychain writes.
