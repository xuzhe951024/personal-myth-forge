# P0.319 Final Acceptance Next Action Design

## Context

`final-acceptance-local.json` is the current functional-regression evidence source for the final iOS showcase path. It records blocked command checks and operator actions, but it does not expose a top-level `first_blocker` or `next_action`.

The final readiness chain already expects those fields on newer reports. Adding them to final acceptance makes the functional-regression blocker directly actionable without requiring readers to inspect every check.

## Scope

Add top-level fields to final acceptance reports:

- `status`: alias of `overall_status` for consistency with other readiness reports.
- `first_blocker`: first blocked or failed check, converted into an operator-facing blocker.
- `next_action`: `first_blocker` plus `source: "first_blocker"`.

The report will keep existing `overall_status`, `summary`, `operator_actions`, and `checks` fields unchanged.

## Blocker Selection

The first check whose status is `blocked` or `failed` becomes the blocker.

Known mappings:

- `mobile_deploy_preflight` with local deploy config issues asks for `DEVELOPMENT_TEAM` in `Deployment.local.xcconfig` and validates with `make mobile-deploy-preflight`.
- `mobile_deploy_preflight` with backend health issues asks to start `make backend-device-demo` and validates with `make mobile-deploy-preflight`.
- `mobile_xcode_build` asks the operator to resolve the Xcode license/build gate outside Codex and validates with `make mobile-xcode-build-evidence`.
- Provider configuration blockers reuse a child report `first_blocker` or `next_action` when available; otherwise they point to `make provider-handoff`.
- Failed checks point to the underlying check command when present.

When all checks pass, `first_blocker` and `next_action` are `null`.

## Safety Behavior

This is metadata-only. It does not execute additional commands, write env files, mutate global state, call live providers, place print orders, or modify Xcode signing.

All blocker fields pass through the existing final acceptance sanitizer so absolute paths, bearer tokens, raw media, local capture URIs, and payment links are redacted.

## Acceptance Criteria

- Local final acceptance reports expose `status`, `first_blocker`, and `next_action`.
- Known local deploy blockers point to concrete validation commands.
- Provider configuration blockers can reuse child provider handoff next-action metadata.
- Passed final acceptance reports expose `first_blocker: null` and `next_action: null`.
- Existing final acceptance consumers remain backward-compatible.
- Focused final acceptance tests, lint, SwiftPM gates, refreshed local report, and full backend tests pass.

## Self Review

- Scope is limited to report metadata.
- No new external provider, device, Xcode, or payment action is introduced.
- The design preserves existing final acceptance fields.
- Blocker order remains deterministic because it follows the existing check order.
