# P0.394 Final Local Refresh Live Child Action Design

## Context

`final-demo-launch-local` now exposes the concrete live provider child action:

```text
make final-resource-apply-preview; rerun make live-provider-evidence
```

The higher-level `final-local-report-refresh` report still promotes the
provider handoff step action:

```text
make final-resource-apply-preview; rerun make provider-handoff
```

That action is useful, but it is less aligned with the final demo launch path.
The refresh report is the operator's broad local evidence refresh view, so it
should carry the same concrete live evidence handoff that the final launch
packet exposes.

## Goal

Make `final-local-report-refresh` promote final demo launch handoff actions,
including the concrete live provider child action and print quote handoff, into
its top-level operator actions.

## Behavior

- Keep the existing first blocker action from `final_resource_requirements`.
- Keep the existing showcase/iOS action near the top.
- Promote provider and print handoff actions from
  `final_demo_launch_local.operator_actions`.
- If both a provider step action and a final demo launch live provider child
  action are present, the final demo launch child action should appear in the
  top-level refresh actions.
- Preserve the existing provider step action when no final demo launch child
  action is available.
- Do not run live providers, apply resources, write secrets, run Xcode signing,
  write Keychain entries, or create print orders.

## Testing

Add a unit-level regression for `_operator_actions()` where a synthetic
`final_demo_launch_local` step carries:

```text
make final-resource-apply-preview; rerun make live-provider-evidence
```

The returned top-level actions should include that action before ordinary
blocked-step noise while still retaining the first two blocker actions.
