# P0.605 Top-Level Live Consent Metadata

## Context

P0.604 made the top-level final handoff reports preserve the concrete configured 3D
live-provider action. The command is now correct, but the top-level `first_blocker`
and `next_action` payloads can still omit the structured flags that explain the
operator action needs live provider and cost consent.

## Decision

When a section action or ledger action is already marked as a live provider or
cost-consent action, copy that metadata into the top-level blocker and next-action
objects. This keeps the operator handoff machine-readable without changing command
ordering, provider behavior, or report status.

## Acceptance

- `final-launch-closure-packet` first blocker and next action include
  `requires_cost_consent: true` and `live_provider_call: true` for configured 3D
  evidence consent actions.
- `final-external-action-ledger` first blocker and next action include the same flags
  for the live provider cost action.
- The implementation only propagates existing action metadata; it does not infer or
  create live-provider state for ordinary local blockers.
- No Meshy, OpenAI, print-provider, Xcode signing, or global machine action is run.
