# P0.19 Guided Scan Entry Design

## Problem

The mobile app can already upload photo sets, manual files, and ARKit scan files,
but the scan workflow still begins with "choose a file." That is not enough for
the iOS-first demo: users should see a native scanning entry point that starts a
guided capture flow on the phone, even if final device validation is still
blocked by the local Apple SDK license gate.

Apple's Object Capture API is a better next step than hand-rolled LiDAR mesh
export for this slice. It is designed for object capture, drives an
`ObjectCaptureSession` through `ObjectCaptureView`, stores captured images in a
directory, and can later feed photogrammetry or external image-to-3D providers.
The current backend and Meshy spike already understand image reference inputs,
so P0.19 should connect guided scan output to the existing media upload path.

## Design

Add a new capture mode named `guided_scan`. It means "a guided Object Capture
photo scan" rather than a prebuilt mesh file. The backend accepts and stores this
mode; generation receives the captured photos as source images just like
`photo_set`.

Add a Swift core importer that converts captured Object Capture image files into
`CaptureMediaDraft` image media. It enforces the current backend limits:

- supported image content types: JPEG, PNG, HEIC, HEIF
- minimum images: 2
- maximum images: 12
- maximum bytes per image: `CaptureDraft.maxFileBytes`

Add a SwiftUI scan entry to the app:

- `CaptureFormView` shows `Guided scan` as a capture mode.
- Its controls include `Start Guided Scan` and a fallback `Choose Photos`.
- `GuidedScanCaptureView` uses `ObjectCaptureSession` and `ObjectCaptureView`
  behind `#if os(iOS) && canImport(RealityKit)`.
- The macOS SwiftPM compile path renders a small unsupported-state fallback, so
  the existing no-Xcode compile gate still works.
- On capture completion, `ForgeRootView` reads the scan image directory,
  converts the photos into `CaptureMediaDraft` values, and stores them in
  `CaptureMediaSelection(mode: .guidedScan)`.

The checked-in Xcode project should include the new app source file so the
P0.18 `make mobile-xcode-build` gate covers it once the user accepts the Apple
SDK license.

```bash
swift run --package-path apps/mobile/ios PersonalMythForgeMobileCoreContractTests
swift build --package-path apps/mobile/ios --product PersonalMythForgeMobileAppCompileCheck
make backend-lint

git diff --check HEAD
```

P0.19 should also run:

```bash
make mobile-xcode-build
```

On the current machine this is still expected to fail at the Apple SDK license
gate until the user accepts the license in Terminal. That result should be
recorded as an environment gate, not as a source failure.

## Non-Goals

- Do not run `xcode-select` or accept the Apple SDK license.
- Do not claim simulator, device, signing, archive, or TestFlight readiness.
- Do not produce a local USDZ/GLB mesh from Object Capture in this iteration.
- Do not add on-device photogrammetry or Meshy multi-image provider changes.
- Do not move OpenAI, Meshy, or print provider keys into the mobile app.
