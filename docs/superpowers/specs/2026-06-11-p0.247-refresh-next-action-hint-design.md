# P0.247 Refresh Next Action Hint Design

## Context

`final-showcase-readiness` now returns a concrete immediate device command in `next_action.command`:

```text
make mobile-deploy-preflight
```

The parent `final-local-report-refresh` report still promotes each child report's `first_blocker.command` into the step summary. For `final_showcase_readiness`, this means the refresh step displays the higher-level gate `make ios-device-launch-rehearsal` even though the child report's `next_action.command` correctly points to `make mobile-deploy-preflight`.

## Goal

Make `final-local-report-refresh` step summaries prefer a child report's `next_action` when present, so the refresh report shows the immediate operator command.

## Non-Goals

- Do not change report status semantics or step ordering.
- Do not change any provider, Xcode, signing, Keychain, or iPhone execution behavior.
- Do not make blocked external conditions look ready.
- Do not remove `first_blocker` from any child report.

## Approaches Considered

1. Prefer `next_action` before `first_blocker` for every child report in `_blocker_hint`.
   This is selected because `final-local-report-refresh` is an operator-facing command ledger; its step `command` should tell the operator what to do next.

2. Special-case only `final_showcase_readiness`.
   This would solve the current mismatch but encode report-specific behavior in a generic helper.

3. Keep using `first_blocker` and rely on the detail text.
   This preserves current shape but leaves the command column misleading.

## Design

Update `_blocker_hint(report)` to inspect `report["next_action"]` first. When it is a mapping, return `_hint_from_mapping(next_action)`. Existing fallback behavior stays unchanged:

1. `next_action`
2. `first_blocker`
3. blocked requirements
4. non-ready checks
5. first operator action

The saved `final-local-report-refresh.json` should then include:

```json
{
  "id": "final_showcase_readiness",
  "status": "blocked",
  "command": "make mobile-deploy-preflight",
  "detail": "iOS deploy runbook and device launch rehearsal must both be ready. | Next device action: make mobile-deploy-preflight | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable"
}
```

## Testing

Add a focused unit-style backend test using an injected `extra_steps` child report with both `first_blocker` and `next_action`. Assert the promoted refresh step uses `next_action.command` and `next_action.detail`, not the higher-level `first_blocker.command`.

Focused verification:

```bash

uv run pytest tests/test_final_local_report_refresh.py
```

```bash

```

`make final-local-report-refresh` is expected to exit 2 while external resources and iPhone/Xcode inputs remain missing, but the saved `final-local-report-refresh.json` must show `steps_by_id.final_showcase_readiness.command` as `make mobile-deploy-preflight`.

## Self-Review

- No placeholders or deferred behavior.
- Scope is one generic hint selection rule and one regression test.
- The design keeps all external actions blocked until explicit user-provided resources exist.
