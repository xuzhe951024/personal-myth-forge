# P0.505 Final Local Refresh Backend Priority Design

## Context

The local final demo launch and final showcase readiness reports now prioritize the iPhone device path by listing the deploy writer first and the backend-device-demo handoff second. The canonical `final-local-report-refresh.json` still lists provider, print, and key handoffs before the same backend-device-demo action.

For the final iPhone showcase, this report is the operator-facing refresh packet. It should keep the first blocker first, then surface the local backend-device-demo handoff before provider/key work.

## Scope

This iteration only changes `final-local-report-refresh` operator-action ordering. It does not start backend servers, write iOS deploy config, accept Xcode licenses, call Meshy/OpenAI/Treatstock, or place print orders.

## Design

`final_local_report_refresh._prioritize_final_local_report_operator_actions()` should keep the existing first-two action behavior when no backend-device-demo handoff is present. When backend-device-demo is present, it should preserve the first concrete action, then promote backend-device-demo before provider and print handoffs. Provider and print actions remain prioritized ahead of unrelated fallback actions.

Canonical promoted action:

`start backend-device-demo before device checks: make backend-device-demo; rerun make mobile-deploy-preflight`

Existing deploy writer, provider chain preference, guarded print quote preference, LAN backend URL preference, and duplicate-root pruning remain unchanged.

## Testing

Add a regression test that feeds `_operator_actions()` a deploy writer, provider chain, print quote, key handoff, and backend-device-demo handoff in the current problematic order. The test must fail before implementation because backend-device-demo is currently ordered after provider/print/key actions.
