# P0.58 ARKit Showcase Acceptance Design

## Purpose

P0.57 added the SwiftPM ARKit scan package bridge, but the top-level iOS
showcase acceptance report still does not list that bridge as a required source
feature. P0.58 closes that gap so final acceptance explicitly proves the
post-device-scan ARKit/Object Capture package path exists before the project is
handed off for real iPhone deployment.

## Scope

This iteration is source-only acceptance coverage. It does not add new app UI,
does not run Xcode, does not require an iPhone, and does not call Meshy/OpenAI.

In scope:

- Add `arkit_scan_package` to `ios_showcase_acceptance`.
- Require evidence from the mobile core builder, app bridge, and Swift contract
  tests.
- Update final acceptance test fixtures from 14 to 15 iOS showcase source
  features.
- Update docs and visual regression evidence.

Out of scope:

- Live ARKit/Object Capture model export on device.
- Real provider-key generation.
- Signing, simulator, or physical iPhone execution.

## Requirements

`ios_showcase_acceptance` must require these source facts:

- `apps/mobile/ios/Sources/PersonalMythForgeMobileCore/ARKitScanPackageBuilder.swift`
  contains `ARKitScanPackageBuilder`, `maximumReferenceImages = 11`, and
  `CaptureMediaSelection(mode: .arkitScan`.
- `apps/mobile/ios/App/ForgeRootView.swift` contains `arkitScanPackageSelection`
  and `ARKitScanPackageBuilder.selection`.
- `apps/mobile/ios/Sources/PersonalMythForgeMobileCoreContractTests/main.swift`
  contains `testARKitScanPackageBuilderBuildsReadySelection`.

When all source evidence is present, the iOS showcase report summary must become
`passed=15, failed=0`. If the ARKit scan package builder file is missing, the
report must fail only the new `arkit_scan_package` feature and must not leak
absolute local paths or secrets.

## Architecture

The acceptance runner keeps its current simple source-requirement model. P0.58
adds one `FeatureRequirement` after `guided_scan`, because the scan package
bridge is the next layer between device-side capture and upload-ready
`arkit_scan` media selection.

`final_acceptance` already treats `ios_showcase_acceptance` as one top-level
check. No orchestrator behavior changes are required beyond updating the test
stubs that model a passing iOS showcase report.

## Testing

Use TDD:

1. Update `test_ios_showcase_acceptance_passes_complete_fixture` to expect the
   new feature and `passed=15`.
2. Add a missing-builder failure test that expects `passed=14, failed=1` and
   sanitized output.
3. Update final acceptance test stubs to report `passed=15`.
4. Run the focused backend tests until green.
5. Run full backend and SwiftPM smoke gates.

## Acceptance

P0.58 is complete when:

- `cd services/backend && uv run pytest tests/test_ios_showcase_acceptance.py tests/test_final_acceptance.py -q` passes.
- `make backend-test` passes.
- `swift run --package-path apps/mobile/ios PersonalMythForgeMobileProjectChecks` passes.
- Quick final acceptance still exits `2` with the same external blockers and
  `failed=0`, while the nested iOS showcase summary is `passed=15, failed=0`.
