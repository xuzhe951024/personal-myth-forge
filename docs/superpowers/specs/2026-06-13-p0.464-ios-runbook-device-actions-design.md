# P0.464 iOS Runbook Device-First Actions Design

## Context

The final showcase readiness report now correctly treats physical iPhone deployability as the first blocker in local mode. The saved local iOS deploy runbook still lists provider key actions first in `operator_actions`, even when its own `first_blocker` is the missing local deploy config. That creates a misleading operator path for the final iPhone demo.

## Decision

For `mode="local"`, `ios_deploy_runbook.operator_actions` must start with the concrete iOS deploy config action for the current required device slot before provider key actions. The first action should match the same device blocker family as `first_blocker`, then retain final resource, provider, evaluation, and Xcode actions after it.

Configured mode remains provider-resource-first because live provider evidence is required for the configured acceptance path.

## Scope

- Add a focused regression test proving local runbook actions start with the deploy config blocker before Meshy/OpenAI key actions.
- Implement action ordering inside the iOS deploy runbook only.
- Preserve validation suffixes from existing operator action helpers.
- Preserve the existing device action bundle contract.

## Out Of Scope

- No live provider calls.
- No Xcode signing or global Xcode state changes.
- No changes to key storage or final resource apply behavior.
- No external print order or paid quote action.

## Acceptance

- The new focused test fails before implementation and passes after implementation.
- Existing iOS deploy runbook tests pass.
- Focused final showcase/local report/closure packet tests still pass.
- `make backend-lint` and `git diff --check` pass.
