# P0.492 Showcase Provider Capability Chain Design

## Context

`final-showcase-readiness.json` already surfaces the complete provider handoff
operator action:

```text
make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence
```

However, the structured `game_asset_3d_generation.next_action` and
`ai_agent_npc.next_action` still expose:

```text
command: make final-resource-apply-preview
validation_command: make live-provider-evidence
```

That skips the required `make provider-handoff` step for clients that read the
structured capability rows instead of the flattened operator action list.

## Design

When live provider evidence reports a child blocker with
`source_blocker_validation_command`, keep the command and validation split:

- `command`: `make final-resource-apply-preview; rerun make provider-handoff`
- `validation_command`: `make live-provider-evidence`

The top-level operator action builder then renders the full executable chain:

```text
make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence
```

This keeps the capability next action structured while ensuring no reader skips
provider-handoff.

## Acceptance

- A RED/GREEN backend test proves the 3D and NPC capability rows include
  `make provider-handoff` before `make live-provider-evidence`.
- The flattened `operator_actions` list contains the complete provider chain
  once and does not contain the weaker apply-preview-to-live-only chain.
- Existing final showcase and shared operator-action tests pass.
- `make final-showcase-readiness` refreshes ignored `.local` evidence with the
  same external blockers and updated provider capability command.

## Safety

This phase only changes report generation. It does not call Meshy, OpenAI, print
providers, Xcode, signing, Keychain, or any paid/global external action.
