# P0.399 Certificate Child Actions Design

## Context

The final iPhone showcase is still blocked by real iOS deployment evidence and
external provider credentials. The operator-facing reports need to keep the next
manual action concrete and file-specific.

`final-handoff-index` already promotes concrete child actions such as:

```text
DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight
make final-resource-apply-preview; rerun make live-provider-evidence
after explicit Treatstock cost consent, save a sanitized services/backend/.local/print-quote-configured.json from POST /v1/print-quotes; rerun make print-fulfillment-readiness
```

`ios-device-launch-certificate` still falls back to generic actions when its
`final_handoff_index` gate is blocked:

```text
run make final-handoff-index
provide iOS deploy config and rerun mobile deploy preflight
```

That creates an extra interpretation step for the operator even though the child
report already knows the next concrete iPhone action.

## Goal

Make `ios-device-launch-certificate` promote concrete child actions from the
embedded `final_handoff_index` report for both `first_blocker.command` and
top-level `operator_actions`.

## Scope

- Change only `myth_forge_api.ios_device_launch_certificate` and its tests.
- Preserve provider and print handoff priority.
- Preserve the generic `make final-handoff-index` fallback when the child report
  has no concrete actions.
- Do not change final resource writers, Xcode/signing behavior, provider calls,
  print quote calls, or `.local` input files.

## Design

Extend the certificate report's final-handoff action selection:

1. Read `final_handoff_index.operator_actions` and
   `final_demo_launch.operator_actions`.
2. Select concrete iOS deploy, provider handoff, and print handoff actions.
3. Normalize selected actions through the existing operator-action normalizer so
   old `provide iOS deploy config` strings become
   `provide iOS deploy config in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`.
4. Prefer the project-local iOS deploy writer when present.
5. Keep `backend-device-demo` actions on their own backend server gate, even
   when they mention `make mobile-deploy-preflight`.
6. Do not treat a bare `make mobile-deploy-preflight` validation command as a
   concrete child handoff action.
7. Prefer iOS deploy child actions ahead of provider and print actions for the
   certificate's first blocker.
8. Use the first selected child action as the final handoff gate action.
9. Fall back to `run make final-handoff-index` only when no child action exists.

The certificate remains read-only and sanitized. It does not execute the child
action, mutate Xcode state, write secrets, or call live providers.

  action but `final-demo-launch` does.
- Assert `first_blocker.command` uses that concrete child action.
- Assert `operator_actions[0]` is the concrete child action rather than
  `run make final-handoff-index`.
- Assert provider and print child actions remain present and ordered before the
