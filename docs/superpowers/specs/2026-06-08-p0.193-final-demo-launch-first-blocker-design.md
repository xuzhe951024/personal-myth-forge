# P0.193 Final Demo Launch First Blocker Design

## Purpose

The final mobile launch receipt needs one canonical "first thing to fix" instead of deriving it from several nested reports on-device. The backend final demo launch report should expose that decision as `first_blocker`, and iOS should decode and display it directly.

## Scope

- Add top-level `first_blocker` to `final_demo_launch_report`.
- Prefer the launch phase blocker because it is the operator-facing execution path for the final demo launch.
- Fall back to existing nested report blockers only when no launch phase needs attention.
- Decode the new field in Swift and use it for the receipt row.
- Keep all reports read-only, sanitized, and free of provider secrets, payment links, raw media, and local paths.

## Backend Contract

`first_blocker` is either `null` or an object with:

- `id`: stable blocker id.
- `label`: display label.
- `status`: original status such as `blocked`, `missing`, `manual`, or `partial`.
- `classification`: machine-readable reason when available.
- `command`: operator command to run next.
- `detail`: short human-readable detail.
- `source`: where the blocker came from, such as `final_demo_launch_phase` or `final_showcase_readiness`.
- `source_id`: source object id.

The first pass selects the earliest launch phase with status priority `failed`, `blocked`, then `missing`. Phase blockers use `source = "final_demo_launch_phase"`, `source_id = phase.id`, and a detail derived from the phase purpose and first note. If no launch phase blocks, the report checks nested top-level blockers from final acceptance readiness, final showcase readiness, final launch closure packet, live provider evidence, print fulfillment readiness, configured bundle, and operator handoff. Manual or partial launch phases are considered only after nested blockers, because `xcode_build_gate` is always manual and should not hide a more specific readiness blocker.

## iOS Contract

Add `FinalDemoLaunchFirstBlocker` and `FinalDemoLaunchReport.firstBlocker`. `FinalLaunchMobileSummaryBuilder.firstBlockerReceiptRow` should prefer `report.firstBlocker`; the existing nested inference remains only as backward compatibility for older backend payloads.

## Testing

- Backend test proves configured reports include a top-level phase blocker without leaking secrets or local paths.
- Backend test proves a ready phase set falls back to nested final showcase readiness `first_blocker`.
- Swift contract test proves decoding and receipt rows use the top-level blocker.
- iOS source acceptance requires backend, Swift model, and mobile summary support for this field.
