# P0.267 Final Demo Resource Validation Design

## Goal

Make `final-demo-launch` resource handoff actions preserve validation rerun commands so downstream final refresh reports do not show both validation-aware and bare resource prompts.

## Context

The final iOS showcase still depends on operator-provided resources such as `MESHY_API_KEY`, `OPENAI_API_KEY`, `DEVELOPMENT_TEAM`, and an iPhone-reachable backend URL. Recent phases made `final_resource_requirements` and `final_local_report_refresh` preserve validation commands, but `final-demo-launch-local.operator_actions` still emits bare strings like `provide MESHY_API_KEY in final-resources.env`. Those strings are then pulled into the final local refresh checklist and compete with the improved action.

## Design

Normalize final demo launch operator checklist entries before de-duplication:

- Prefer the validation-aware action already emitted by `final_resource_requirements.operator_actions`.
- Drop bare resource prompts when a validation-aware version for the same command is present.
- Preserve existing mobile preflight next action priority.
- Preserve existing safety behavior: no provider calls, no key writes, no Xcode/signing, no Keychain changes.
- Keep report schemas unchanged.

## Test Plan

Extend final demo launch tests so:

- `final-demo-launch-local.operator_actions` includes `provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight`.
- The first eight final demo launch actions do not include the bare `provide MESHY_API_KEY in final-resources.env` string.
- `final-local-report-refresh.operator_actions` also stops surfacing that bare action in its first eight actions after consuming the final demo launch report.

The tests should fail before implementation because final demo launch currently forwards bare resource prompts.

- Scope is one handoff-cleanup behavior.
- The change improves the final external-resource handoff without mutating resources or using live providers.
