# P0.262 Final Demo Operator Actions Design

## Goal

Expose concrete top-level `operator_actions` on `final-demo-launch` reports so the iPhone Final Launch summary can show the same immediate operator action already available in `next_action`.

## Context

`final-demo-launch-local.json` currently exposes a concrete `next_action`, for example:

`provide DEVELOPMENT_TEAM in Deployment.local.xcconfig`

with `validation_command`:

`make mobile-deploy-preflight`

The report still serializes `operator_actions` as `null` or omits it, while mobile summary code already prefers `report.operatorActions.first` when present. This means the iPhone-facing summary cannot consistently show the immediate Mac-side handoff action from the top-level launch report.

## Options Considered

1. **Add `operator_actions` to `final-demo-launch` directly.** This is the preferred path because it fixes the report closest to the mobile contract and keeps downstream aggregators simpler.
2. **Only teach mobile to synthesize actions from `next_action`.** This would help the app, but it duplicates backend command formatting in Swift and does not help other handoff reports.
3. **Only rely on `operator_checklist`.** This field is broader and less ordered around the immediate blocker; it is useful but not precise enough as the first next action.

## Design

`build_final_demo_launch_report()` will compute `next_action` once, then derive `operator_actions` from it and from the existing `operator_checklist`.

Rules:

- If `next_action.command` is present, it becomes the first operator action.
- If `next_action.validation_command` is present, append `; rerun <validation_command>`.
- Add existing `operator_checklist` entries after the concrete action.
- Deduplicate actions while preserving order.
- Cap the list to a small bounded set for mobile display and report readability.
- Do not include secrets, local absolute paths, payment links, live-provider calls, or global side effects.

## Test Plan

Extend the existing local final demo launch preflight evidence test to assert:

- `operator_actions[0]` equals `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig; rerun make mobile-deploy-preflight`.
- The report still preserves `next_action` and `operator_checklist`.
- No `MESHY_API_KEY` appears in the top-level operator action for the local iOS blocker.

## Self-Review

- No placeholders.
- Scope is limited to a top-level launch report contract.
- The change advances the iOS display/handoff path without touching real provider keys or Xcode signing.
