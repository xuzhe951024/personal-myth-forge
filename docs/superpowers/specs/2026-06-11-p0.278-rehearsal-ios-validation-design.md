# P0.278 Rehearsal iOS Validation Design

## Context

P0.275 through P0.277 made most final-launch reports retain
`make mobile-deploy-preflight` for known iOS deploy configuration actions. The
iOS device launch rehearsal path still has two aggregation helpers that only
apply final-resource validation:

- `ios_device_launch_rehearsal.py`
- `ios_device_launch_rehearsal_readiness.py`

Those helpers normalize nested saved report actions, but they do not add mobile
preflight validation for actions such as:

- `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig`
- `provide PRODUCT_BUNDLE_IDENTIFIER in Deployment.local.xcconfig`
- `set PMF_BACKEND_BASE_URL to an iPhone-reachable LAN URL`

This leaves the final physical-iPhone rehearsal lane less explicit than the
other launch ledgers.

## Goal

The iOS device launch rehearsal report and its readiness reader should apply
the same mobile deploy validation helper used by the rest of the final-launch
report stack.

## Non-Goals

- Do not change `final-resources.env` resource input semantics.
- Do not change readiness status, source freshness, first blocker, or sequence
  ordering.
- Do not run writer scripts, edit ignored env/config files, call live
  providers, trigger Xcode signing, touch Keychain, or place print orders.
- Do not refresh physical-device evidence beyond regenerated ignored reports.

## Design

Import `add_mobile_deploy_validation_command()` in both rehearsal modules and
compose it inside each `_validation_aware_operator_action()`:

1. Normalize legacy command text.
2. Add mobile deploy validation for known iOS deploy config action roots.
3. Add final resource validation for backend/final resource action roots.

This keeps one canonical mobile validation rule and preserves existing
final-resource behavior.

## Acceptance

- RED/GREEN test proves live rehearsal aggregation adds
  `; rerun make mobile-deploy-preflight` for nested saved iOS deploy config
  actions.
- RED/GREEN test proves rehearsal readiness adds the same validation suffix
  when reading saved operator actions.
- Existing final resource action tests still pass and keep
  `; rerun make final-resources-preflight`.
- Refreshed `.local/ios-device-launch-rehearsal.json` remains blocked by real
  missing device/provider inputs, but any nested iOS deploy config action roots
  carry the mobile preflight validation command.
- Safety flags continue to show no provider calls, no env/config writes, no
  global mutation, no Xcode/signing, and no Keychain writes from the report
  builders.
