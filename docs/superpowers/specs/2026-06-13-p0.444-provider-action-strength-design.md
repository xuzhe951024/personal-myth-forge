# P0.444 Provider Action Strength Design

## Goal

Make the final showcase operator action list keep the strongest provider handoff chain and drop weaker duplicate provider actions.

## Context

`final-showcase-readiness` is still blocked by physical iPhone deployment evidence and missing live-provider resources. The report already includes a complete provider handoff sequence:

```text
make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence
```

The same list can still include weaker forms of the same workflow:

```text
make final-resource-apply-preview; rerun make live-provider-evidence
make provider-handoff
```

That makes the final operator handoff harder to follow even though the full chain is present.

## Behavior

When final showcase action aggregation sees the complete provider chain, it should keep that chain and remove weaker provider duplicates that would skip the provider handoff validation step or ask the operator to run provider handoff as a separate unvalidated command.

This applies inside `final_showcase_readiness._dedupe_operator_actions()` so downstream reports that consume final showcase actions inherit the cleaner action list.

The first iOS deploy action must remain first, print handoff actions must still be retained, and non-provider actions must keep their existing order.

## Safety

This is read-only report shaping. It must not call Meshy, OpenAI, Treatstock, Xcode, Keychain, or any live provider. It must not write `.env`, local deploy config, or payment/order artifacts.

## Tests

Add a focused unit test that passes a list containing:

- the complete provider chain
- the weaker apply-preview-to-live-evidence action
- bare `make provider-handoff`
- a print quote handoff action

The expected output keeps only the complete provider chain and the print quote handoff action.
