# P0.86 Final Launch Demo Script Design

Date: 2026-06-06
Branch: `codex/p0.86-final-launch-demo-script`

## Context

P0.85 made NPC controls provider-aware, but the iPhone `Demo Script` still treats
the final `resources` row as complete once local provider readiness is available.
That is useful for the no-key local loop, but it hides the stricter final launch
state already exposed by `FinalLaunchMobileSummary`: final resources preflight,
local/configured acceptance readiness, operator handoff, deploy preflight, and
Xcode build blockers.

The next step toward an iOS-deployable final showcase is to make the operator
script consume that final launch summary. The app should keep the local loop
usable, but it should not say the full demo is ready when final launch remains
blocked by deployment config or local Apple SDK state.

## Approaches Considered

### A. Add a final launch step to Demo Script

Append a `final_launch` step when a `FinalLaunchMobileSummary` is available.
The step maps `.ready` to complete, `.waiting` to waiting, and `.blocked` to
blocked. The script's `nextAction` becomes final-launch aware after capture,
forge, scene, NPC autonomy, print quote, and provider resources are complete.

This keeps existing local-loop behavior intact for older call sites while making
the real iPhone root view stricter because it already has a final launch summary.

### B. Replace the existing resources step

Make `resources` mean final launch status instead of provider readiness. This is
too broad because it loses the useful distinction between backend provider
readiness and final Mac-side launch blockers.

### C. Keep Demo Script unchanged and rely on Final Launch Status panel

This preserves all current behavior but leaves the first actionable script panel
able to say the loop is ready while the adjacent launch panel says blocked. That
is a bad operator experience during a live iPhone demo.

## Decision

Use approach A.

`DemoScriptBuilder.build` gets an optional `finalLaunchSummary:
FinalLaunchMobileSummary? = nil` argument. When nil, existing callers keep the
six current steps. When present, the builder appends:

- `id`: `final_launch`
- `label`: `Final Launch`
- `status`: `.complete`, `.waiting`, or `.blocked` based on
  `FinalLaunchMobileSummary.overallStatus`
- `detail`: a sanitized short detail from the final launch summary

`ForgeRootView.demoScript` passes the existing `finalLaunchMobileSummary`, so the
actual iPhone app always shows the final launch row.

`ShowcaseAutopilotPlanner.plan` inspects the optional `final_launch` step after
print quote and provider readiness. If the step is waiting or blocked, the
autopilot button is not executable and explains the final launch state. If the
step is complete, the final plan remains complete.

## Data Flow

```text
/v1/final-demo-launch
  -> FinalDemoLaunchReport
  -> FinalLaunchMobileSummaryBuilder
  -> FinalLaunchMobileSummary
  -> DemoScriptBuilder(finalLaunchSummary:)
  -> DemoScript.steps["final_launch"]
  -> ShowcaseAutopilotPlanner
  -> DemoScriptView button/detail
```

The mobile app remains read-only for final launch commands. It displays status
and operator next actions; it does not run deploy, Xcode, provider, or resource
apply commands.

## Redaction and Safety

Final launch details must use the same conservative redaction posture as the
existing demo script and final launch summary. Encoded `DemoScript` and
`ShowcaseAutopilotPlan` must not expose:

- OpenAI-style `sk-` tokens
- bearer tokens
- raw API key strings
- absolute local paths
- file URLs
- local capture URIs
- checkout or payment URLs

## Test Plan

- Add Swift core contract tests proving:
  - a blocked final launch summary appends a blocked `final_launch` step
  - a ready final launch summary allows the full script to complete
  - unsafe final launch detail is redacted in encoded script output
  - autopilot blocks on a blocked `final_launch` step
  - autopilot completes when `final_launch` is complete
- Update Swift project source checks for root wiring and source evidence.
- Update backend iOS showcase source acceptance so final Demo Script wiring is
  part of quick final acceptance.
- Add static 390x844 visual evidence showing the Demo Script with a final launch
  blocked row and disabled autopilot state.

## Out of Scope

- No new backend endpoints.
- No provider calls.
- No changes to Meshy, OpenAI, Treatstock, Sculpteo, or iOS signing config.
- No Xcode license, `DEVELOPMENT_TEAM`, or global system changes.
