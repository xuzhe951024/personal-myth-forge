# P0.12 iOS Device Media Input Design

## Context

P0.10 wired the iOS app shell to `ForgeFlowService`, but `ForgeRootView` still
uses in-memory sample media. P0.11 completed the backend/provider path that can
use real uploaded reference images for 3D generation. The next gap toward an
iOS-deployable demo is letting the app collect real device media while keeping
provider API keys and external services on the backend.

## Goal

Replace sample-media capture actions in the iOS app shell with real PhotosPicker
and fileImporter input paths that produce `CaptureMediaDraft` values for the
existing forge flow.

## Non-Goals

- Full Xcode simulator, archive, or device deployment on this machine.
- Global `xcode-select` or developer directory changes.
- Live camera capture, ARKit mesh capture runtime, LiDAR reconstruction, or
  RealityKit rendering.
- Provider API-key entry in the mobile app.
- Changing backend capture or 3D provider contracts.

## Product Behavior

The first screen remains the forge workflow, not a landing page. The capture
area keeps the existing quiet form layout and adds real input controls:

- `single_photo`: choose one photo from Photos.
- `photo_set`: choose two to twelve photos from Photos.
- `manual_upload`: import one to twelve supported files.
- `arkit_scan`: import a scan/model file and optionally add reference photos.

The app shows selected-media count and file names before upload. `Forge Myth`
continues to call `ForgeFlowService.forge(draft:context:)`, using selected real
media instead of sample bytes.

## Architecture

Add a small core model:

```swift
public struct CaptureMediaSelection
```

It stores the current `CaptureMode` and `[CaptureMediaDraft]`, computes selected
image/scan counts, exposes a summary, validates mode-level readiness, and builds
the existing `CaptureDraft`. This keeps capture state testable in SwiftPM
without importing SwiftUI, PhotosUI, or UniformTypeIdentifiers into the core
module.

Update App-layer SwiftUI:

- `CaptureFormView` receives media summary text and uses PhotosPicker for photo
  modes.
- `ForgeRootView` owns `CaptureMediaSelection`, PhotosPicker selection state,
  fileImporter state, and async media loading.
- Photos are read with `PhotosPickerItem.loadTransferable(type: Data.self)`.
- Imported files are read with security-scoped URL access when available.
- Content types are inferred in the App layer from `UTType`, then normalized by
  the existing `CaptureDraft` validation before upload.

## Error Handling

- Failed photo or file reads update a short `captureInputError` string in the
  form instead of calling the backend.
- Unsupported or oversized media is still rejected by `CaptureDraft` validation
  before network upload.
- Switching capture mode clears selected media, preventing stale photo/file
  combinations from being uploaded under the wrong mode.
- Mobile source checks continue to reject committed provider secrets.

## Testing

SwiftPM contract tests cover:

- `CaptureMediaSelection` summary and readiness for single photo, photo set,
  manual upload, and ARKit scan selections.
- Building a `CaptureDraft` from selected media preserves existing upload
  validation behavior.
- Switching capture mode clears selected media.

Project source checks cover:

- `ForgeRootView` no longer contains sample media payloads.
- App source uses `PhotosPicker`, `fileImporter`, `loadTransferable`,
  `UTType`, and `CaptureMediaSelection`.
- App source does not contain provider API keys or bearer tokens.

Visual evidence shows the mobile capture input surface with real-device input
controls and selected media state. It must explicitly state that this is
source-level/no-simulator evidence and not a live device capture or ARKit mesh
run.

## Acceptance Criteria

- Swift project checks pass.
- Swift mobile core contract tests pass.
- Backend lint and tests remain green.
- `ForgeRootView` no longer falls back to sample media for capture actions.
- The iOS app source has PhotosPicker and fileImporter input paths that build
  `CaptureMediaDraft` media for the existing forge flow.
- Docs identify the remaining device gaps: live camera, ARKit mesh capture,
  simulator/device deployment, and iOS 3D rendering.
