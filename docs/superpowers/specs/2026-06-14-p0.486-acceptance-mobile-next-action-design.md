P0.486 Final Acceptance Readiness Mobile Next Action Design

Goal

Make final acceptance readiness use saved mobile deploy preflight next actions when the local acceptance report is blocked by iOS deploy preflight.

Context

`final-acceptance-local.json` can be generated during safe local refresh without running live device checks. In that mode the mobile deploy preflight check may be classified as `blocked_by_local_ios_backend_health`, and final acceptance readiness currently falls back to:

`start backend-device-demo and rerun mobile deploy preflight`

However, the project also saves `mobile-deploy-preflight-evidence.json`, whose `next_action` often contains the more concrete first device action:

`DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight`

Using the saved next action makes the functional regression handoff match final showcase readiness and avoids telling the operator to start backend services before missing local deploy config is written.

Non-Goals

- Do not run `mobile-deploy-preflight`, Xcode, signing, Keychain, or real device checks.
- Do not write `Deployment.local.xcconfig` or set a real Apple Team ID.
- Do not change final acceptance pass/fail semantics.
- Do not remove the backend-device-demo fallback when saved mobile preflight evidence is absent.

Design

`final_acceptance_readiness._enrich_mobile_preflight_blocker()` already attaches saved mobile preflight `next_action` to mobile deploy blockers. Update `_operator_actions()` so all `mobile_deploy_preflight` blockers first try `_mobile_preflight_operator_action(blocker)`.

If a saved next action exists, use it regardless of whether the classification is deploy config or backend health. If no saved next action exists and the classification is `blocked_by_local_ios_backend_health`, keep the existing backend-device-demo fallback.

Acceptance Criteria

- A blocked mobile deploy preflight readiness report with saved writer `next_action` produces the writer operator action.
- The old backend-device-demo fallback is not used when saved writer evidence exists.
- The backend-device-demo fallback remains when no saved mobile preflight next action exists.
- Existing stale report, malformed report, ready report, and deploy-config enrichment behavior continue to pass.
- Safe local evidence remains blocked on external iOS conditions and does not run device/global actions.

Safety

This phase changes report action selection only. Verification may write ignored `.local` JSON through safe report builders. It must not call live providers, run Xcode signing, write Keychain, place print orders, or write real iOS deploy config.
