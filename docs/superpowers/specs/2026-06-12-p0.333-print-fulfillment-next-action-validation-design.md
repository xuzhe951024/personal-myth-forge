# P0.333 Print Fulfillment Next Action Validation Design

## Context

`print_fulfillment_readiness_report` is part of the final objective because the
demo must derive a printable asset and hand off a third-party quote path. The
report already exposes a partial blocker when the configured Treatstock quote is
missing, but its `next_action` lacks two pieces of machine-readable handoff
metadata:

- the validation command that proves the quote handoff is present afterward
- an explicit cost-consent flag for the live Treatstock quote step

The command text mentions explicit Treatstock cost consent, but downstream
summaries and operators should not need to parse that sentence.

## Decision

Extend `print_fulfillment_readiness_report.next_action`:

- Always include `validation_command: "make print-fulfillment-readiness"` when
  there is a blocker.
- Include `requires_cost_consent: true` when the action is the configured
  Treatstock quote handoff.
- Preserve `requires_cost_consent: false` for other print blockers.
- Keep ready reports at `next_action: null`.

This does not create, quote, or order anything. It only makes the existing
handoff safer and more explicit.

## Safety

This phase does not call Treatstock, write provider keys, create checkout links,
place orders, write deploy config, run Xcode, or mutate global state. The report
continues to redact local paths, secrets, and payment links.

## Acceptance

- Missing configured Treatstock quote actions expose validation and cost-consent
  metadata.
- Unsafe configured quote blockers expose validation metadata without claiming
  that another live quote should be requested.
- Ready configured quote reports keep `next_action` as `null`.
- The final local refresh operator action still includes
  `make print-fulfillment-readiness`.
