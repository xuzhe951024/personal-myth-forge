# P0.36 Media-Backed 3D Evaluation Design

## Context

P0.35 added a canonical `default-v0` text-only 3D provider evaluation suite.
That suite is useful for prompt quality, but it does not exercise the scan-like
provider inputs that matter for the mobile-first product direction. The backend
already has a `ThreeDGenerationRequest.source_images` contract, Meshy routing
for single-image and multi-image generation, and safe generation provenance.

P0.36 adds a media-backed evaluation suite that can prove the evaluation runner
passes source images to providers, captures single-image and multi-image
provenance, and writes a safe report without raw media.

## Approach Options

### Option A: Built-In Guided Scan Smoke Suite

Add `guided-scan-smoke-v0` as a built-in suite with three synthetic media cases:

- one PNG source image -> expected `single_image`
- two PNG source images -> expected `multi_image`
- four PNG source images -> expected `multi_image`

Each image is a tiny valid PNG encoded as a provider-facing data URI inside the
suite definition. The report includes source image counts and source roles, but
never returns the data URI.

This is deterministic, no-key, and directly tests the source image path.

### Option B: External Media Manifest

Add a CLI manifest format that points to user-provided local image files. This
is closer to real iPhone validation, but it needs path handling, media
preparation, validation, and careful path redaction. It is the right next step
after the runner can already handle source image cases.

### Option C: Reuse Capture Store Fixtures

Drive evaluation through `POST /v1/object-captures` fixtures and
`/v1/myth-sessions/from-capture`. This exercises more of the product loop but
mixes API acceptance with provider evaluation and makes the report harder to
keep focused.

## Decision

Use Option A for P0.36. It creates a stable media-backed smoke suite while
keeping live provider calls opt-in. P0.37 can add external media manifests for
real iPhone images once this source-image evaluation path is established.

## Feature Summary

P0.36 adds:

- `ThreeDEvaluationSourceImage` metadata in the evaluation module
- optional `source_images` on `ThreeDEvaluationCase`
- built-in suite `guided-scan-smoke-v0`
- CLI support:

```bash
cd services/backend
uv run python -m myth_forge_api.cli evaluate-3d \
  --provider local \
  --suite guided-scan-smoke-v0 \
  --output /tmp/personal-myth-forge-guided-scan-eval.json
```

- report fields per case:
  - `source_image_count`
  - `source_image_roles`
  - existing `generation_provenance`
  - no raw `data:image` payloads
- coverage still comes from generated asset provenance:
  - `single_image`: 1
  - `multi_image`: 2

## Built-In Suite

`guided-scan-smoke-v0` has three cases:

1. `scan_01_single_reference_key`
   - category: `guided_scan_smoke`
   - source image roles: `front`
   - expected input mode: `single_image`
   - review focus: single reference image routes to Image-to-3D

2. `scan_02_two_angle_key`
   - category: `guided_scan_smoke`
   - source image roles: `front`, `side`
   - expected input mode: `multi_image`
   - review focus: two guided scan references route to Multi-Image-to-3D

3. `scan_03_four_angle_key`
   - category: `guided_scan_smoke`
   - source image roles: `front`, `side`, `top`, `detail`
   - expected input mode: `multi_image`
   - review focus: four guided scan references preserve provider max-count

The data URIs are synthetic PNGs with no private user data. They exist only to
exercise provider input shape and routing. They are not quality benchmarks.

## Report Shape

For the local provider, the suite should produce:

```json
{
  "kind": "three_d_evaluation_report",
  "suite": "guided-scan-smoke-v0",
  "provider": "local",
  "total_cases": 3,
  "succeeded": 3,
  "failed": 0,
  "coverage": {
    "input_modes": {
      "text_prompt": 0,
      "single_image": 1,
      "multi_image": 2,
      "unknown": 0
    },
    "variant_roles": {
      "game_asset": 3,
      "ios_scene_asset": 3
    },
    "scene_loadable_cases": 3
  },
  "cases": [
    {
      "case_id": "scan_01_single_reference_key",
      "source_image_count": 1,
      "source_image_roles": ["front"],
      "generation_provenance": {
        "input_mode": "single_image",
        "source_image_count": 1,
        "selected_source_image_count": 1,
        "raw_sources_included": false
      }
    }
  ]
}
```

The report must not contain:

- `data:image`
- local filesystem paths
- provider secrets
- bearer tokens

## Testing Strategy

Backend tests:

- `evaluate-3d --suite guided-scan-smoke-v0` writes three successful cases.
- coverage includes `single_image == 1` and `multi_image == 2`.
- case reports expose source image counts and roles, not data URIs.
- a recording fake provider receives source image counts `[1, 2, 4]`.
- existing `default-v0` and custom prompt suite tests remain green.
- provider errors from media cases are still sanitized.

Regression:

- `make backend-lint`
- `make backend-test`
- SwiftPM project/contract/app compile checks remain green.
- local guided scan suite smoke writes a three-case report.
- final acceptance quick keeps known local blocker classification.

Visual evidence:

- static 390x844 evidence page for `guided-scan-smoke-v0`
- screenshot rendered with installed Google Chrome through Playwright MCP

## Non-Goals

- No external user media manifest in P0.36.
- No real iPhone image quality scoring.
- No automatic Meshy run without `MESHY_API_KEY`.
- No change to Meshy provider routing; this iteration only proves the
  evaluation runner can exercise existing source-image routing.
