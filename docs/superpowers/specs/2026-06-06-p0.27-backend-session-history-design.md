# P0.27 Backend Session History Design

## Context

P0.26 added a local iOS demo snapshot so a generated myth session can be restored
after app relaunch. The backend still treats myth sessions and NPC ticks as
stateless responses:

- `POST /v1/myth-sessions`
- `POST /v1/myth-sessions/from-capture`
- `POST /v1/npc-ticks`

For a fuller minimum demo, NPC activity should have a backend-owned continuity
contract. The app should be able to ask the backend what happened in the current
myth run without relying only on a client-side snapshot.

## Decision

Add a local backend myth session history store. This is a development/demo store,
not an account system and not durable production memory.

Chosen approach:
- Store session history as JSON files under a repo-local ignored directory.
- Save each generated `MythSession`.
- Append each generated `NPCAgentTick`.
- Add read endpoints for session history.
- Add Swift mobile models and API client methods so the app can consume the
  history contract in a later UI iteration.

Alternatives considered:
- In-memory store: simpler, but loses history on backend restart and does not
  help demo continuity.
- SQLite/Postgres: closer to production, but premature before auth, user IDs,
  retention policy, and deployment target are defined.
- Extend P0.26 client snapshot only: useful for relaunch, but it does not prove
  backend-owned NPC world continuity.

## Backend Contract

New models:

- `MythSessionHistory`
  - `session_id`
  - `session`
  - `npc_ticks`
  - `updated_at`

New endpoints:

```http
GET /v1/myth-sessions/{session_id}
GET /v1/myth-sessions/{session_id}/history
```

Behavior:
- Creating a direct or from-capture myth session saves an initial history record.
- Creating a tick appends it to the matching session history when present.
- If a tick request references a session not yet in the store, the backend saves
  the supplied session first, then appends the tick. This preserves backwards
  compatibility for clients that still send a full session body.
- Session IDs are validated with a strict pattern.
- Unknown session IDs return `404`.
- History returns ticks ordered by `tick_index` and bounded to the latest 24.

## Storage

Create `LocalMythSessionStore`:

- Default root: `services/backend/.local/myth-sessions`
- Env override: `MYTH_SESSION_STORAGE_DIR`
- Record path: `<root>/<session_id>/history.json`
- Atomic write through a temporary file and rename.

The store persists only sanitized structured API objects:

- `MythSession`
- `NPCAgentTick`
- `updated_at`

It does not persist raw uploaded media bytes, provider source image data URIs,
provider keys, bearer tokens, or original personal source documents.

## Mobile Contract

Add Swift models:

- `MythSessionHistory`

Add API client methods:

- `getMythSession(sessionId:)`
- `getMythSessionHistory(sessionId:)`

The app UI does not need a visible history screen in P0.27. This iteration only
adds the backend contract and mobile decode/client path so a follow-up can wire
server history into the P0.26 restored demo banner.

## Testing

Backend tests:
- `LocalMythSessionStore` saves, loads, appends, sorts, bounds, and clears.
- Store rejects invalid session IDs/path traversal.
- Store JSON does not leak data URIs, bearer tokens, or test secret strings.
- Direct and from-capture session creation save history.
- `POST /v1/npc-ticks` appends tick history.
- `GET /v1/myth-sessions/{session_id}` and `/history` return the saved record.
- Unknown session returns `404`.

Mobile tests:
- Swift decodes `MythSessionHistory`.
- API client builds GET requests for session and history.
- HTTP error body sanitization still covers these endpoints.

Visual regression:
- Add a 390x844 static evidence page showing backend-owned session history,
  latest 24 tick cap, mobile client contract, and secret/raw-media exclusions.

## Non-Goals

- No user accounts or cloud sync.
- No production database.
- No server-side autonomous background tick loop.
- No long-term memory beyond a bounded local demo history.
- No raw personal data, raw media, provider key, or data URI persistence.
- No print fulfillment.
