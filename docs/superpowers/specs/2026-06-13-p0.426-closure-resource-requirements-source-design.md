# P0.426 Closure Resource Requirements Source Design

## Goal

Carry the canonical final resource requirements report into the final launch closure packet so the closure evidence can trace missing API keys, iOS deploy values, and provider handoff blockers through a structured action bundle.

## Current State

`final_resource_requirements` now exposes a safe `device_action_bundle`, and `final_local_report_refresh` preserves it in the step row. `final_launch_closure_packet` still sources resource evidence from the fill guide and external action ledger only, so the closure packet cannot directly show the canonical requirement rows or their new bundle.

## Approach

Build `final_resource_requirements` inside `build_final_launch_closure_packet_report()`, include `make final-resource-requirements` in the closure command list, and add `source_reports.final_resource_requirements` using the existing `_source_summary()` helper. The helper already preserves child `device_action_bundle` fields and runs through the closure sanitizer.

## Safety

This only adds sanitized report metadata to an existing read-only closure packet. It does not write `.env`, write iOS deploy config, call Meshy/OpenAI/Treatstock, run Xcode, sign builds, or place print orders.

## Testing

Add a focused closure packet contract test that asserts `source_reports.final_resource_requirements.device_action_bundle` exists, starts with `MESHY_API_KEY`, includes the resource validation command, has no live provider calls, and does not leak temp paths or synthetic secrets.
