# P0.73 Mobile Context Capsule Review Design

## Context

The iPhone app already sends a compact `ContextCapsule` with `currentTheme` and
`desiredTone` when forging a myth session. That matches the privacy direction of
using user-approved summaries instead of raw private sources, but the app does
not yet make approval visible or enforce it before `Forge Myth`.

## Decision

Add an explicit mobile `Context Capsule Review` step.

Recommended approach:

- Add a Swift core `ContextCapsuleReviewBuilder` that summarizes the capsule
  status and sanitizes display strings.
- Add a SwiftUI `ContextCapsuleReviewView` with an `Approve Capsule` toggle.
- Disable and guard `Forge Myth` until both media is ready and the capsule is
  approved.
- Reset approval when the theme or tone changes.

Rejected alternatives:

- Add a Personal Context connector now. That would expand scope into source
  access and privacy review before the iOS demo needs it.
- Send an `approved` field to the backend. The current backend contract already
  accepts only the summary capsule; this iteration enforces approval at the
  mobile demo layer without changing API compatibility.

## User Experience

The capture form still contains the two summary fields:

- `Current theme`
- `Desired tone`

Below those fields, the app shows a compact review panel with:

- capsule readiness status;
- sanitized theme and tone summary;
- privacy notes stating that no raw email, chat, calendar, files, or provider
  keys are included;
- an `Approve Capsule` toggle.

Changing either text field clears approval. `Forge Myth` is disabled until the
operator approves the displayed capsule.

## Data Flow

```text
currentTheme + desiredTone + isContextCapsuleApproved
  -> ContextCapsuleReviewBuilder.build(...)
  -> ContextCapsuleReviewView
  -> CaptureFormView disables Forge Myth unless media ready and approved
  -> ForgeRootView.forgeMyth guard
  -> ContextCapsule(currentTheme, desiredTone)
```

The review builder has no network access and no file access.

## Safety

The app still does not read raw personal data. The only personal context fields
used by the forge flow are the manually entered summary strings. All review
display strings are redacted for secrets, local paths, file URLs, payment links,
and local capture URIs.

The view must not:

- request access to email, calendar, chat, files, or contacts;
- display raw private source bodies;
- read provider keys or `.env` files;
- call Meshy, OpenAI, Treatstock, or any personal data provider.

## Testing

Core contract tests cover:

- missing theme or tone waits for review;
- unapproved complete capsule waits for approval;
- approved complete capsule is ready;
- unsafe strings are redacted.

Project checks cover:

- `ContextCapsuleReview.swift` exists in core;
- `ContextCapsuleReviewView.swift` exists and is included in Xcode sources;
- `CaptureFormView` renders the review and disables forge when unapproved;
- `ForgeRootView` owns approval state, resets it on edits, and guards
  `forgeMyth`;
- source acceptance requires the new privacy-preserving context capsule review.

Visual evidence is a static 390x844 iPhone screenshot showing an approved
context capsule review above the Forge action.
