# P0.243 Device Action Slot Detail Design

## Context

`ios-device-evidence-bundle` now has concrete `evidence_slots[].detail` values for backend reachability, mobile deploy preflight, Xcode, and launch rehearsal. The top-level `operator_actions` list still uses generic command-only strings:

```text
start backend-device-demo before device checks: make backend-device-demo
run make mobile-deploy-preflight after backend is running
run Xcode build gate manually on the Mac: make mobile-xcode-build
run make ios-device-launch-rehearsal
```

The iPhone final-launch/operator handoff surfaces these actions as a concise action list. Without the slot detail attached, the list hides the concrete blocker that was already proven in the same report.

## Goal

Make `ios-device-evidence-bundle.operator_actions` include the relevant non-ready slot detail so the device handoff action list is immediately actionable.

## Non-Goals

- Do not add or rename report fields.
- Do not change `evidence_slots` ordering or status semantics.
- Do not add Swift model or UI changes.
- Do not run Xcode signing, Keychain, mobile deploy preflight, live providers, or paid provider calls.

## Approaches Considered

1. Append `| <slot.detail>` to each non-ready slot action.
   This is selected. It preserves existing action prefixes and command text while exposing the already-sanitized blocker detail.

2. Add a new `operator_action_detail` field to each slot.
   This would be explicit but requires downstream readers to learn a new field. The current `detail` field already exists and is sanitized.

3. Make only the first operator action detailed.
   This is lower value because the iPhone handoff list often shows several device actions at once, including Xcode and rehearsal.

## Design

`_operator_actions(status, slots)` will continue returning `["iOS device evidence is ready"]` when all slots are ready.

For each non-ready slot, it will use the current command-oriented base action and append the slot detail:

```text
run make mobile-deploy-preflight after backend is running | Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable
```

If a slot has no detail, the action remains the existing base string. The whole report still passes through `_sanitize_report`, so secrets, local paths, payment URLs, and raw media remain redacted.

## Testing

Add a backend RED/GREEN test:

- Arrange blocked final acceptance, blocked saved mobile preflight evidence, blocked saved Xcode evidence, and blocked saved launch rehearsal readiness.
- Build `ios-device-evidence-bundle`.
- Assert operator actions include:
  - backend action plus `PMF_BACKEND_BASE_URL must be iPhone-reachable`
  - mobile preflight action plus `Missing DEVELOPMENT_TEAM; PMF_BACKEND_BASE_URL must be iPhone-reachable`
  - Xcode action plus `Apple SDK license agreement is not accepted.`
  - rehearsal action plus the saved rehearsal blocker detail
- Assert no `MESHY_API_KEY` appears.


`make ios-device-evidence-bundle` is expected to exit 2 while external device requirements are still missing, but it must write `services/backend/.local/ios-device-evidence-bundle.json` with detailed operator actions.

## Self-Review

- No placeholders or deferred decisions.
- Scope is one backend report contract and one focused test.
- The design only improves existing handoff text; it does not trigger external or paid actions.
