# P0.433 Mobile Saved Next Action Design

## Context

P0.432 added `saved_next_action` to `ios_device_launch_rehearsal_readiness.device_action_bundle.first_action`. The backend now keeps the immediate stale-report rerun command and separately exposes the concrete command that will unblock the next device step after rerun, such as `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto`.

The iOS core currently decodes `FinalShowcaseDeviceAction.nextAction`, but it does not decode `saved_next_action`. Final Launch rows therefore show the stale rerun and generic detail, but not the cleaner saved next device command that P0.432 added for the iPhone handoff.

## Decision

Extend the Swift `FinalShowcaseDeviceAction` model with an optional `savedNextAction: FinalShowcaseReadinessNextAction?`, mapped from `saved_next_action`. Then update `FinalLaunchMobileSummaryBuilder.showcaseDeviceActionRow` to append a `saved next ...` segment when the field is present.

## Behavior

- Existing `next_action` behavior remains unchanged.
- `saved_next_action` is optional and backward compatible with older reports.
- Final Launch device action rows show `saved next <command> | <detail>` after normal `next ...` content and before raw operator action rows.
- All displayed text still passes through the existing `sanitize` path.
- No backend reports, provider calls, Xcode signing, Keychain entries, or ignored resource files are changed.

## Tests

Add Swift contract tests that decode `saved_next_action` from the final demo launch payload and assert `launchRehearsalRows` includes the saved next command and detail.
