# P0.256 Final Showcase Blocker Concrete Action Design

## Goal

Make final showcase readiness use the saved concrete mobile deploy child action in the blocker detail, matching the top-level `next_action.command`.

## Current Gap

P0.254 and P0.255 promote `provide DEVELOPMENT_TEAM in Deployment.local.xcconfig` into top-level next-action fields, but `final-showcase-readiness.json.first_blocker.detail` still says `Next device action: make mobile-deploy-preflight`. The first blocker remains accurate, but the operator-facing detail is less actionable than the adjacent next action.

## Requirements

- Keep `first_blocker.id == "ios_deployable"` and `first_blocker.command == "make ios-device-launch-rehearsal"`.
- When mobile preflight evidence includes a child `next_action.command`, `first_blocker.detail` says `Next device action: <child command>`.
- Keep existing fallback behavior when no child next action exists.
- Do not run Xcode, provider calls, signing, Keychain writes, or paid actions.

## Non-Goals

- No changes to readiness status ordering.
- No changes to resource/key requirements.
- No iOS UI changes; the existing summary consumes this backend detail.
