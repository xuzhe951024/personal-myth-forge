# P0.184 Mobile Configured Acceptance Command Row Design

## Goal

The iPhone Final Launch panel must surface `make final-acceptance-configured` whenever a configured launch report contains it, even when that command is not one of the first four report commands.

## Current Behavior

`FinalLaunchMobileSummaryBuilder` builds `commandRows` with `report.commands.prefix(4).map(sanitize)`. Backend handoff reports now publish the safe wrapper command `make final-acceptance-configured`, but configured launch reports can include that command after earlier setup commands. The mobile UI can therefore hide the exact configured acceptance entry the operator needs at the final handoff.

## Scope

- Keep the `Commands` section capped at four rows.
- Preserve local-mode ordering unless the configured acceptance command is present.
- In configured mode, promote the exact `make final-acceptance-configured` command to the first row.
- Avoid duplicates when the promoted command already appears in the first four rows.
- Keep all rows sanitized.
- Do not run live provider calls or real configured acceptance.

## Acceptance

- Swift core contract test proves configured mode promotes the command from beyond the fourth item.
- Existing partial/local final launch summary behavior remains intact.
- ProjectChecks guard the source-level contract so future edits cannot return to a raw `prefix(4)` command list.
- App compile and safe rehearsal continue to pass.
