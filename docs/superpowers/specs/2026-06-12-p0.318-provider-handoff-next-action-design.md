# P0.318 Provider Handoff Next Action Design

## Context

The configured evidence bundle now points to `make provider-handoff` when real Meshy/OpenAI provider evidence is not ready. The provider handoff report itself still requires readers to inspect `providers`, `missing_env`, and `next_commands` to understand the next action.

For the final iOS showcase goal, the provider handoff report should directly tell the operator what to do next while preserving the rule that API keys and provider secrets never enter the repository.

## Scope

Add top-level `first_blocker`, `next_action`, `operator_actions`, and `safety` fields to `provider_handoff_report`.

The new fields will:

- Be `null` for `first_blocker` and `next_action` when core real providers are ready.
- Prefer missing backend env values as blockers when a real provider is selected but not configured.
- Otherwise identify the first core provider still running as a local stub.
- Point operators to ignored `final-resources.env` handoff and safe validation commands.
- Keep existing `providers`, `missing_env`, and `next_commands` fields intact for backward compatibility.

## Out Of Scope

- No provider key generation.
- No real Meshy, OpenAI, Treatstock, or Sculpteo calls.
- No writes to `.env`, `.xcconfig`, Keychain, or global machine state.
- No changes to provider readiness calculation.
- No changes to which providers are selected by default.

## Blocker Selection

Provider handoff reports choose blockers in this order:

1. If `core_real_ready` is true, no blocker.
2. If any `missing_env` values exist, the first sorted missing env becomes the blocker.
3. Otherwise, the first core provider whose `is_real_provider_ready` is false becomes the blocker.

The `next_action` mirrors `first_blocker` and adds `"source": "first_blocker"`.

## Safety Behavior

The report remains metadata-only. `safety` records that no provider calls, writes, Keychain changes, Xcode signing, raw media exposure, or payment links are produced by this handoff report.

## Acceptance Criteria

- Default local provider handoff reports include a blocker for the first non-real core provider.
- Meshy/OpenAI selected without keys produces a missing env blocker with validation guidance.
- Ready Meshy/OpenAI configuration produces `first_blocker: null` and `next_action: null`.
- CLI-written provider handoff JSON contains the new fields.
- Existing reports remain backward-compatible.
- Focused provider handoff tests, configured preflight tests, lint, diff check, Swift compile gates, and full backend tests pass.

## Self Review

- The design is scoped to handoff visibility, not provider execution.
- The design preserves backend-only secrets and ignored local resource files.
- Blocker ordering is deterministic.
- No requirement depends on live provider access.
