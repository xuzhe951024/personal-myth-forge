# P0.493 Closure Specific Actions Design

## Context

The final iOS showcase is still blocked by external device and provider inputs. The saved `final-launch-closure-packet.json` already includes specific operator handoffs such as:

- `DEVELOPMENT_TEAM=YOUR_TEAM_ID make mobile-write-deploy-config-auto; rerun make mobile-deploy-preflight`
- `provide MESHY_API_KEY in final-resources.env; rerun make final-resources-preflight`
- `make final-resource-apply-preview; rerun make provider-handoff; rerun make live-provider-evidence`
- `PMF_ALLOW_PRINT_PROVIDER_CALLS=1 make print-quote-configured; rerun make print-fulfillment-readiness`

It still also surfaces bare validation targets like `make final-resources-preflight` and `make ios-device-launch-rehearsal` in top-level `operator_actions`. Those commands are valid validation commands, but they are weaker as first-run handoff steps because they do not say which missing input or prerequisite must be supplied first.

## Decision

Keep bare validation targets available in `commands`, section `command`, and `validation_command` fields. Tighten only the top-level `operator_actions` checklist so it prefers specific, copy-pasteable handoff actions over bare validation commands when those specific actions already exist.

The closure packet should:

- Drop top-level `make final-resources-preflight` when resource-specific actions ending in `rerun make final-resources-preflight` are present.
- Drop top-level `make ios-device-launch-rehearsal` when device-specific actions ending in `rerun make mobile-deploy-preflight`, backend startup handoff, or Xcode evidence handoff are present.
- When the first blocker is still the iOS device chain, promote the first concrete device action ahead of provider/resource actions so the top-level checklist remains aligned with the blocker.
- Preserve the bare validation targets if no more specific action exists, so sparse or degraded reports still tell the operator how to refresh evidence.
- Leave `commands` unchanged because it is the complete closure packet command inventory rather than the next-action checklist.

## Safety

This phase is report-only. It does not run provider calls, write ignored env files, write iOS deploy config, invoke Xcode signing, or perform print fulfillment. The generated report continues to block until real device/provider evidence is supplied by the user.

## Verification

Focused tests must prove:

- A missing-resource fixture no longer emits bare `make final-resources-preflight` at top level when validated `provide ...; rerun make final-resources-preflight` actions are present.
- The same fixture no longer emits bare `make ios-device-launch-rehearsal` at top level when concrete device handoff actions are present.
- A device-blocked fixture keeps `next_action.command == "make ios-device-launch-rehearsal"` but starts `operator_actions` with a concrete device preflight action.
- The full closure packet focused test suite passes.
- Backend lint passes.
