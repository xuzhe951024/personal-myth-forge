# P0.76 Final Acceptance Readiness Design

## Context

P0.75 made the iPhone app show whether NPC behavior is local or OpenAI-backed.
The remaining quick final acceptance blockers are still local deployment gates:
`mobile_deploy_preflight` and `mobile_xcode_build`. The app already renders a
read-only `Final Launch Status` from `/v1/final-demo-launch`, but that report
does not expose the concrete final acceptance check classifications from a
recent `final-acceptance` run.

The next useful slice is to make the final acceptance blockers visible in the
same mobile launch surface without letting the iPhone trigger Xcode, signing,
provider calls, or global machine changes.

## Decision

Add a read-only `final_acceptance_readiness` report. The backend reads a saved
final acceptance JSON file from the ignored local path
`services/backend/.local/final-acceptance-local.json`. It never runs
`final-acceptance`, `xcodebuild`, deployment scripts, or provider calls during
API handling.

`/v1/final-demo-launch` will embed the readiness report so the existing mobile
fetch path stays simple. The report has three states:

- `missing`: no saved final acceptance JSON exists yet.
- `blocked`: a saved report exists with one or more blocked or failed checks.
- `ready`: a saved report exists with no blocked or failed checks.

## Backend Shape

Create `myth_forge_api.final_acceptance_readiness`.

The report includes:

- `kind`: `final_acceptance_readiness_report`
- `status`: `missing`, `blocked`, or `ready`
- `source_file`: repo-relative path plus existence flag
- `summary`: `passed`, `blocked`, `failed`, `skipped`
- `blockers`: compact rows with `id`, `label`, `status`, `classification`,
  `command`, and `detail`
- `operator_actions`: next steps such as running local final acceptance,
  fixing a failed check, configuring deploy preflight, or handling Xcode build
  gate externally
- `safety`: booleans proving no API-triggered commands, provider calls, global
  mutation, secrets, raw media, payment links, or local paths are exposed

The parser accepts only dictionary-shaped JSON. Invalid JSON or unexpected
shape is `blocked` with a safe parse action. All detail text is sanitized.

## iOS Shape

Extend mobile models with:

- `FinalAcceptanceReadinessReport`
- `FinalAcceptanceReadinessSummary`
- `FinalAcceptanceBlocker`
- `FinalAcceptanceSourceFile`

Extend `FinalDemoLaunchReport` with optional `finalAcceptanceReadiness`.

Extend `FinalLaunchMobileSummary` with `acceptanceRows: [String]`. The builder
maps readiness as:

- missing report -> run local final acceptance command
- ready report -> `Final acceptance ready.`
- blocked report -> first few blocker rows with status/classification/command

`FinalLaunchStatusView` renders an `Acceptance` section below `Resources` and
above `Commands`.

## Testing

Backend TDD:

- missing readiness file returns `missing`, exit 2, safe action
- blocked saved final acceptance returns blocked blocker rows and safe actions
- ready saved final acceptance returns `ready`, exit 0
- malformed saved file returns blocked without leaking file paths or secrets
- `/v1/final-demo-launch` embeds `final_acceptance_readiness`

iOS TDD:

- decode `final_acceptance_readiness` from final launch payload
- mobile summary waits on missing readiness
- mobile summary lists blocked acceptance gates
- mobile summary marks ready readiness
- unsafe readiness text is redacted

Acceptance and visual evidence:

- iOS source acceptance requires the new model, summary field, view section,
  and backend embedding.
- 390x844 static evidence shows missing, blocked, and ready final acceptance
  states without unsafe strings.

## Non-Goals

- Do not run final acceptance from an API request.
- Do not run Xcode, signing, simulator, device deployment, or Apple license
  commands.
- Do not store provider keys in the app.
- Do not add a mobile button that executes Mac-side commands.
