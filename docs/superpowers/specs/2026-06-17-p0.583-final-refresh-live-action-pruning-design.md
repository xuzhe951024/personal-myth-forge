# P0.583 Final Refresh Live Action Pruning Design

## Context

After local iOS/device evidence is ready, `final-local-report-refresh.json` can correctly identify the configured live provider evidence plan as the first blocker and expose the guarded command:

```text
PMF_ALLOW_LIVE_PROVIDER_CALLS=1 make final-acceptance-configured; rerun make live-provider-evidence
```

The same report can still include older provider fallback actions in top-level `operator_actions`, such as a provider handoff chain or a bare `make live-provider-evidence`. That gives the final operator multiple live-provider entry points and weakens the consent boundary.

## Decision

Keep the first blocker and safe report refresh behavior unchanged. Add one operator-action preference rule:

- If the guarded live provider consent action is present, keep it as the live provider entry point.
- Drop provider handoff and bare live provider evidence fallback actions from the same top-level action list.
- Preserve unrelated device, configured preflight, print, and local validation actions.

## Acceptance

- A consent-required configured evidence next action prunes provider fallback actions.
- The guarded live provider command remains in `operator_actions`.
- Existing final-local report refresh tests remain green.
- Report generation still performs no live provider calls, Xcode signing, keychain writes, or global mutations.
