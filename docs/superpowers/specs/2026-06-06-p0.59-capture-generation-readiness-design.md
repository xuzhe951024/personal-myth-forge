# P0.59 Capture Generation Readiness Design

## Purpose

The iOS app currently shows scan readiness with static copy: `2+ photos` for
guided scan and `mesh + reference` for ARKit scan. That is too weak for the
final mobile demo because it does not explain how the selected capture will feed
3D generation, whether Meshy can use the input, or whether the app is only ready
for the local deterministic provider path.

P0.59 adds a tested mobile-core readiness builder that converts the current
`CaptureMediaSelection` plus backend provider readiness into a source-safe 3D
generation route summary. `CaptureFormView` will display that summary for scan
modes.

## Scope

In scope:

- Add `CaptureGenerationReadinessBuilder` to the SwiftPM mobile core.
- Distinguish `single_image`, `multi_image`, and `scan_asset` 3D generation
  routes from the selected capture media.
- Show guided scans as multi-image-ready when at least 2 photos are selected.
- Show ARKit scan packages as scan-asset-ready when a scan asset is present.
- Surface the difference between local demo provider readiness and real Meshy
  readiness without exposing secrets or local paths.
- Wire `ForgeRootView` to pass the readiness summary into `CaptureFormView`.
- Add SwiftPM contract tests and source gates.

Out of scope:

- Live device photogrammetry/reconstruction.
- New backend endpoints.
- New provider API calls.
- Xcode/device/simulator execution.

## Data Model

Add:

- `CaptureGenerationReadinessStatus`: `waiting`, `ready`, `needsAttention`.
- `CaptureGenerationRoute`: `waiting`, `textPrompt`, `singleImage`,
  `multiImage`, `scanAsset`.
- `CaptureGenerationReadiness`: title, detail, status, route, source counts,
  and selected provider source count.
- `CaptureGenerationReadinessBuilder.build(selection:providerReadiness:providerReadinessError:)`.

The builder is pure Foundation and does not depend on SwiftUI or iOS-only
frameworks.

## Route Rules

- No selection or not upload-ready: `waiting`.
- Single photo: `singleImage` with 1 provider source.
- Photo set or guided scan: `multiImage`; provider source count is capped at 4,
  matching the Meshy multi-image route already used by the backend.
- ARKit scan with a scan asset: `scanAsset`; references remain counted in the
  detail.
- Manual upload chooses `scanAsset` if a scan asset exists, otherwise
  `multiImage` for 2+ images, otherwise `singleImage` for 1 image.

Provider readiness detail:

- Real 3D provider ready: say real 3D provider is ready.
- Local 3D provider ready only: say local demo route is ready and Meshy is still
  needed for live 3D.
- Missing 3D provider configuration: `needsAttention` with missing env names.
- Provider readiness error: `needsAttention`, sanitized.

## App Flow

`ForgeRootView` computes:

```swift
CaptureGenerationReadinessBuilder.build(
    selection: mediaSelection,
    providerReadiness: providerReadiness,
    providerReadinessError: providerReadinessError
)
```

It passes `title` and `detail` into `CaptureFormView`. The existing scan
readiness row remains visible only for `.guidedScan` and `.arkitScan`, but its
content becomes the builder output instead of static copy.

## Testing

SwiftPM contract tests must cover:

- Guided scan with one photo waits for more photos.
- Guided scan with 6 photos is `multiImage`, selected provider source count 4,
  and local demo provider detail mentions Meshy.
- ARKit scan package with one scan asset and references is `scanAsset`.
- Missing 3D provider env marks readiness as `needsAttention`.
- Provider errors are redacted.

Project checks must read `CaptureGenerationReadiness.swift`, require
`CaptureGenerationReadinessBuilder`, and verify `ForgeRootView` /
`CaptureFormView` source wiring.

## Acceptance

P0.59 is complete when:

- Swift mobile core contract tests pass.
- Swift mobile project checks pass.
- Swift app compile gate passes.
- Backend tests still pass.
- Static visual evidence shows the new generation readiness summary in the
  mobile capture flow.
