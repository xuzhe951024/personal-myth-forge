# P0.316 Print Fulfillment Next Action Design

## Context

The final showcase goal still needs third-party print fulfillment handoff evidence. The current `print_fulfillment_readiness_report` already classifies local print proof, iPhone source coverage, visual proof, resource handoff, and configured Treatstock quote evidence. It exposes `first_blocker`, but it does not expose a top-level `next_action` for operators or the iPhone Final Launch summary.

This differs from the newer iOS deploy and final launch reports, where `next_action` gives the operator a concrete command without having to inspect nested checks.

## Scope

Add a top-level `next_action` to `print_fulfillment_readiness_report`.

The field will:

- Be `null` when print fulfillment readiness is `ready`.
- Mirror the sanitized `first_blocker` when readiness is not ready.
- Add `"source": "first_blocker"` so downstream summaries know the action came from the selected blocker.
- Preserve the blocker command and detail, including the configured Treatstock quote handoff wording.

Also update the iOS Swift models and final launch summary so the iPhone can decode and display the print fulfillment next action.

## Out Of Scope

- No real Treatstock API call.
- No checkout, payment link, or order creation.
- No provider key reads beyond existing ignored local readiness paths.
- No Xcode signing, Keychain writes, or global machine mutation.
- No changes to readiness status ordering or which check becomes the first blocker.

## Data Flow

`build_print_fulfillment_readiness_report()` builds the same checks as before, computes `first_blocker`, then computes `next_action` from that blocker:

```text
checks -> first_blocker -> next_action
```

Swift decodes the new optional field:

```text
print_fulfillment_readiness.next_action
  -> PrintFulfillmentReadinessNextAction?
  -> FinalLaunchMobileSummary.printFulfillmentReadinessRows
```

## Error And Safety Behavior

Sanitization remains centralized in `_sanitize_report()`, so the new field receives the same secret, local path, payment link, and private context redaction as the rest of the report.

If a report does not include `next_action`, Swift decoding remains backward-compatible because the field is optional.

## Acceptance Criteria

- Backend report includes `next_action` for blocked or partial print fulfillment readiness.
- Backend report sets `next_action` to `None` when configured Treatstock quote evidence is ready.
- Swift contract tests decode `nextAction` from final launch payloads.
- iPhone Final Launch summary rows show `Next action: ...` for print fulfillment readiness.
- Existing redaction tests cover the new row.
- Focused backend and Swift contract tests pass.

## Self Review

- No placeholder requirements remain.
- The design is limited to print fulfillment readiness handoff visibility.
- The field shape matches established `next_action` patterns from other final launch reports.
- Real provider cost actions remain behind explicit operator consent and are not executed by this change.
