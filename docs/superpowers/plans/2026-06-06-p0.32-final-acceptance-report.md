# P0.32 Final Acceptance Report Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> superpowers:subagent-driven-development (recommended) or
> superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add one final acceptance CLI command that runs local/configured demo
handoff checks, classifies known provider/iOS/Xcode blockers, and writes a
sanitized JSON report for final API/key/device acceptance.

**Architecture:** Add `myth_forge_api.final_acceptance` as a small orchestrator
around in-process provider/demo checks plus subprocess-based mobile/regression
gates. Wire it into `myth_forge_api.cli final-acceptance`.

**Tech Stack:** Python 3.10, pytest, ruff, existing FastAPI backend package,
existing Make/SwiftPM/Xcode gates.

## Acceptance Criteria

- `final-acceptance --profile quick --provider-mode local` writes one sanitized
  JSON report covering provider handoff, demo acceptance, iOS deploy preflight,
  and Xcode build gate.
- Known local environment blockers are reported as `blocked` with CLI exit `2`,
  not as product failures.
- `final-acceptance --profile full --provider-mode local` includes backend and
  Swift regression checks.
- `final-acceptance --provider-mode configured --require-real-core` exits `2`
  before live provider calls until real Meshy/OpenAI keys are supplied.
- Existing P0.31 acceptance, provider handoff, 3D generation, NPC tick, mobile,
  and deploy gates remain intact.
