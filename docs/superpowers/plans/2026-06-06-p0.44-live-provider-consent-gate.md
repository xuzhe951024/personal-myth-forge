# P0.44 Live Provider Consent Gate Plan

## Goal

Add an explicit consent gate before demo/final acceptance can contact configured
live Meshy/OpenAI providers. Defaults must remain local/no-cost and reports must
stay sanitized.

## Files

- Modify `services/backend/src/myth_forge_api/acceptance.py`
- Modify `services/backend/src/myth_forge_api/final_acceptance.py`
- Modify `services/backend/src/myth_forge_api/cli.py`
- Modify `services/backend/src/myth_forge_api/resource_handoff.py`
- Modify `services/backend/tests/test_acceptance.py`
- Modify `services/backend/tests/test_final_acceptance.py`
- Modify `services/backend/tests/test_cli.py`
- Modify `README.md`
- Modify `apps/mobile/README.md`
- Add `docs/superpowers/verification/2026-06-06-p0.44-live-provider-consent-gate-regression.md`

## TDD Steps

### 1. Acceptance Red Tests

Add tests in `test_acceptance.py`:

- Configured Meshy/OpenAI with dummy keys, `require_real_core=True`, and default
  `allow_live_provider_calls=False` exits `2` before provider builders are called.
- The report includes:
  - `allow_live_provider_calls: false`
  - `live_provider_consent_required: true`
  - `live_provider_kinds: ["three_d", "npc"]`
  - selected providers for `three_d`, `npc`, and `capture_storage`
  - no key leakage
- Configured Meshy/OpenAI with dummy keys and `allow_live_provider_calls=True`
  reaches provider builders. Monkeypatch builders to local/fake providers to
  avoid network.

Expected: tests fail because the new argument/report fields do not exist.

### 2. Final Acceptance Red Tests

Add/update tests in `test_final_acceptance.py`:

- Top-level final report exposes `allow_live_provider_calls`.
- Default value passed to the demo runner is false.
- Explicit true value is passed through.
- A demo acceptance exit `2` caused by no live-provider consent is classified as
  blocked provider configuration.

Expected: tests fail because `run_final_acceptance` lacks the argument and does
not pass it through.

### 3. CLI Red Tests

Update `test_cli.py`:

- Default `demo-acceptance` and `final-acceptance` calls pass
  `allow_live_provider_calls=False`.
- `--allow-live-provider-calls` passes true for both commands.

Expected: parser rejects the flag and call dictionaries are missing the field.

### 4. Implementation

In `acceptance.py`:

- Add `allow_live_provider_calls=False` to `run_demo_acceptance`.
- Include the value in the base report.
- Detect selected live-call providers before building providers.
- Keep the existing `require_real_core` missing-env block first.
- If live providers are selected and consent is false, return exit `2` with a
  sanitized not-ready report.

In `final_acceptance.py`:

- Add `allow_live_provider_calls=False` to `run_final_acceptance`.
- Include it in the top-level report.
- Pass it to `selected_demo_acceptance_runner`.
- Keep exit `2` from demo acceptance classified as blocked provider
  configuration.

In `cli.py`:

- Add `--allow-live-provider-calls` to both acceptance subcommands.
- Pass the bool into helper functions and runners.

In `resource_handoff.py`, `README.md`, and `apps/mobile/README.md`:

- Update configured-mode final validation examples so intentional live-provider
  validation uses `--allow-live-provider-calls`.
- Add wording that leaving the flag off is a safe dry handoff check that blocks
  before contacting live providers.
