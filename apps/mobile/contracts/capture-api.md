# Capture API Contract

P0.6 mobile clients use two required calls and one optional lookup call:

1. `POST /v1/object-captures`
2. `GET /v1/object-captures/{capture_id}`
3. `POST /v1/myth-sessions/from-capture`

Capture ids are generated by the backend and match `cap_[0-9a-f]{16}`. Clients
should treat any other shape as invalid and should not invent ids except in local
fixtures.

## Upload Object Capture

Request: `multipart/form-data`

- `metadata_json`: JSON matching `apps/mobile/fixtures/object-capture-metadata.json`
- `files`: one or more image or scan files

Accepted local content types:

- `image/jpeg`
- `image/png`
- `image/heic`
- `image/heif`
- `model/gltf-binary`
- `model/vnd.usdz+zip`
- `application/octet-stream`

Supported `capture_mode` values:

- `single_photo`: exactly 1 image.
- `photo_set`: 2-12 images.
- `guided_scan`: 2-12 images captured or imported from the iOS Object Capture
  guided scan entry.
- `manual_upload`: 1-12 accepted files.
- `arkit_scan`: 1-12 accepted files with at least one scan/model asset.

Local development limits:

- max files: 12
- max bytes per file: 10 MiB
- local storage: `CAPTURE_STORAGE_DIR`, defaulting to `services/backend/.local/captures`

Example:

```bash
curl -X POST http://127.0.0.1:8080/v1/object-captures \
  -F 'metadata_json={"label":"old brass key","materials":["metal","brass"],"source":"phone_capture","capture_mode":"single_photo","visual_notes":"worn teeth"}' \
  -F 'files=@/path/to/key.jpg;type=image/jpeg'
```

The response is an `ObjectCapture` manifest. It contains `local-capture://` URIs in
local development and never returns raw file bytes.

Example response:

```json
{
  "capture_id": "cap_ba02a3816bd145b4",
  "status": "ready",
  "source": "phone_capture",
  "capture_mode": "single_photo",
  "object_observation": {
    "label": "old brass key",
    "materials": ["metal", "brass"],
    "source": "phone_capture",
    "visual_notes": "worn teeth"
  },
  "media_items": [
    {
      "media_id": "media_0",
      "role": "reference_image",
      "content_type": "image/jpeg",
      "byte_size": 9,
      "uri": "local-capture://cap_ba02a3816bd145b4/media_0.jpg",
      "moderation_status": "needs_review"
    }
  ],
  "created_at": "2026-06-05T00:00:00+00:00"
}
```

Error status guide:

- `422`: malformed JSON, metadata validation failure, or no files
- `413`: too many files or a file over the local size limit
- `415`: unsupported content type

### Guided Scan Notes

`guided_scan` is a photo-set contract. The iOS app may collect the photos through
RealityKit `ObjectCaptureSession`, but P0.19 uploads the captured images rather
than a local reconstructed mesh or USDZ. The backend passes guided scan images to
the 3D provider contract as `source_images`.

JPEG, PNG, HEIC, and HEIF are accepted by the upload API. The Meshy adapter only
uses JPEG/PNG as Image-to-3D input today; mixed captures choose the first
JPEG/PNG source image, while HEIC/HEIF-only captures safely fall back to
text-to-3D until a transcoding step exists.

## Lookup Object Capture

```bash
curl http://127.0.0.1:8080/v1/object-captures/cap_ba02a3816bd145b4
```

Returns the same `ObjectCapture` manifest shape as upload.

Error status guide:

- `404`: well-formed id, but no manifest exists
- `422`: invalid capture id shape

## Create Myth From Capture

```bash
curl -X POST http://127.0.0.1:8080/v1/myth-sessions/from-capture \
  -H 'Content-Type: application/json' \
  -d @apps/mobile/fixtures/myth-session-from-capture.json
```

Request body:

```json
{
  "capture_id": "cap_0000000000000000",
  "context_capsule": {
    "current_theme": "deadline pressure",
    "desired_tone": "tender, strange",
    "recent_milestone": "finished a difficult project draft"
  }
}
```

Unity should render `generated_asset.uri` when it is a remote GLB. Local stub URIs
are metadata-only and should use a placeholder artifact.

Error status guide:

- `404`: well-formed id, but no manifest exists
- `422`: invalid request body or capture id shape
- `502`: configured AI or 3D provider failed before producing a session
